## Context

A `/design-crit` of the `/frameworks` browse view (the `index.html` the skill emits — generated by `plugins/pmos-learnkit/skills/frameworks/scripts/build-library.mjs`) surfaced one headline complaint ("the page keeps reloading") plus six UX/quality findings. Live reproduction in a real browser proved the page **never literally reloads** (one navigation, zero idle DOM mutations, zero network requests after load); the "reload" perception is a **flash + scroll-snap on every interaction**, caused by the emitted client JS tearing down and rebuilding the entire result DOM (`groupsEl.innerHTML = …`, up to 272 inline SVGs / ~13,200 nodes) on every keystroke and every card click.

All fixes live in **one file** — `scripts/build-library.mjs` — which both emits the browse-view client JS and carries its own inline behavioral test assertions (e.g. line ~608 asserts the current scroll-save/restore hack). This is a **single-skill epic** (`route: skill`) → the singleton-epic wrap (D18): one build story carries all findings.

Design contract: `docs/pmos/features/2026-06-13_frameworks-browse-ux-fixes/02_design.html`.

## Findings (from the design crit, verbatim severity)

- **F1 [high] Perceived reload.** `render()` does a full `groupsEl.innerHTML=html` teardown+rebuild on every interaction; `openReader()`/`closeReader()` call full `render()` just to toggle one selection class; search is un-debounced; narrowing the result set collapses document height so the browser clamps scroll to the top. The compound flash + scroll-snap reads as a reload.
- **F2 [high] Detailed view eager-renders all 272 SVG thumbnails** (~13k DOM nodes) on every render — the perf root cause behind F1's flash.
- **F3 [high] Reader sidebar weak keyboard/focus (WCAG 2.2).** `Escape` closes filter dropdowns but not the reader; opening the reader does not move focus into it; closing does not return focus to the triggering item.
- **F4 [medium] "Share" / deep-link half-wired.** Deep links (`#id`) open the reader on load, but `openReader()` never updates `location.hash`, so Share copies prose, not a link to the framework.
- **F5 [medium] No clear affordance for the text query.** Applied-chip bar covers facet filters only; no one-click clear of the search text / all filters.
- **F6 [low] Subtitle hardcodes "272"** while the live count uses `DATA.length` — they drift if the corpus changes.
- **F7 [low] Mobile reader placement.** At ≤720px the reader renders after the full list in DOM order; opening a framework drops content far below the fold with no scroll-into-view.
