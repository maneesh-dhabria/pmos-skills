## Context

From maintainer feedback (2026-06-13): `/plan`'s final-verification task (`TN`) is a **smoke / does-it-render checklist** — it confirms the implementation works *mechanically* (lint, tests, renders, no console errors) but never **exercises the deliverable on a realistic end-to-end task to assess its utility**. The maintainer repeatedly ends up specifying the dogfooding task *post-deployment* ("use the new `/research` skill, have an independent LLM judge score the report, fix the gaps, repeat") when it should have been authored into the **verification step** up front, with both objective and subjective eval criteria, and a feedback loop that turns failures into fix tasks until satisfied.

This epic makes a **load-bearing dogfooding task** a first-class verification-time step: a new shared substrate `_shared/dogfooding.md`, a `/plan` change that emits the task (mandatory T2/T3) + asks the user to approve its scenario+criteria, and a `/verify` change that gates on its verdict and reconciles the iterate loop. `/feature-sdlc` inherits the behaviour free.

Design contract (the cross-skill coherence doc the stories cite by anchor): `docs/pmos/features/2026-06-13_dogfooding-verification/02_design.html`.

### Maintainer decisions captured at define (2026-06-13, AskUserQuestion grill)

- **D0 — Scope:** substrate + `/plan` + `/verify`; `/feature-sdlc` inherits free. Chosen over `/plan`-only (gate too soft) and over also changing `/feature-sdlc` (orchestrator phase redundant — children carry it).
- **D1 — Tier policy:** T2 + T3 mandatory, T1 optional (recommended-skip).
- **D2 — Iterate loop:** auto-iterate, cap 2, then ask; net-worse guard; non-interactive accepts-residuals-and-surfaces. Mirrors the skill-eval remediation loop.
- **D3 — Dual criteria:** every dogfood task declares BOTH objective (measurable) and subjective (blind LLM-judge rubric) criteria.
- **D4 — Independent judge:** a fresh, blind subagent returns a structured verdict, makes no edits; never the producing context.
- **D5 — Additive:** distinct from and additive to the existing TN smoke/render checklist; both run.
- **D6 — Placement & loop home:** "TN−1: Dogfood / Utility Verification" immediately before TN; `/execute` runs it; the fix loop reuses `/execute`'s existing discovered-work routing — no `/execute` change.
- **D7 — Approval gate:** `/plan` asks the user to approve/edit the dogfood scenario + criteria at plan time.
- **D8 — Archetype catalog:** skill→LLM-judge, frontend→Playwright-friction, CLI→real-invocation, data→golden-sample, API→scenario; bespoke when none fit.
- **D9 — Leanness (§K):** the substrate holds the detail; `/plan` and `/verify` cite it and state only deltas.

## Acceptance Criteria

- [ ] **Substrate** `plugins/pmos-toolkit/skills/_shared/dogfooding.md` exists and is the canonical home (§K) for: dogfood task anatomy, the objective+subjective eval-criteria framework (D3), the archetype catalog (D8), the independent-judge contract (D4), the iterate-until-satisfied protocol with cap 2 + net-worse guard + non-interactive posture (D2), the tier policy (D1), and the plan-time approval gate (D7).
- [ ] **`/plan` emits the task:** `plan/reference/plan-templates.md` carries a `TN−1: Dogfood / Utility Verification` task template (immediately before TN, citing the substrate); the plan body's Tier Gates require it for T2/T3 (offer for T1); a `/plan` phase selects an archetype, drafts the scenario + dual criteria, and runs an `AskUserQuestion` to approve/edit (with a `(Recommended)` option).
- [ ] **`/plan` review enforces it:** a structural review check fails any T2/T3 plan whose dogfood task is missing, not load-bearing, or lacks BOTH objective and subjective criteria + a named independent judge; exit criteria updated.
- [ ] **TN−1 serializes into `tasks.yaml`** as a real task `/execute` runs; the iterate loop reuses discovered-work routing (no `/execute` change — D6).
- [ ] **`/verify` gates on it:** a Phase-7 hard gate "Dogfood verdict" (peer of browser-evidence) makes a satisfied dogfood verdict load-bearing for T2/3 — an unsatisfied/missing verdict caps the run at `PASS-WITH-GAPS`/`FAIL` with gaps enumerated; non-skippable for T2/3.
- [ ] **`/verify` reconciles residuals:** `accepted_residuals[]` from the iterate loop are reconciled like skill-eval residuals (still-failing accepted → `KNOWN / accepted`, loud, non-blocking; newly-failing objective gate → blocks).
- [ ] **Both skills stay lean (D9):** `/plan` and `/verify` cite `_shared/dogfooding.md` and state only per-skill deltas; no duplicated contract prose.
- [ ] **Conventions:** both stories pass `skill-eval.md` (floor 43/47) + repo `CLAUDE.md` (canonical path, NI inline block byte-identical, `(Recommended)`/`defer-only` on every `AskUserQuestion`, §H–§L); release prerequisites under the spec/plan `## Release prerequisites` only, never as `/execute` wave tasks; repo hygiene lints green (flags-vs-hints, phase-refs, non-interactive-inline, audit-recommended).
- [ ] **Single plugin (D17):** all changes land in pmos-toolkit.
