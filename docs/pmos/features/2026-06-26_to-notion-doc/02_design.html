Design — /to-notion-doc

Epic 260626-4sm · Plugin pmos-utilities · Route skill · Status defining · Date 2026-06-26

This is the cross-cutting design contract for the single-story epic. Grounded in research/notion-api-findings.md (verified Notion-docs research). Two architectural forks resolved with the maintainer during this define run are marked resolved.

1. Problem & charter fit

A PM has a finished local document (.md / .html / .txt) and wants it as a faithful Notion page. Driving it through the raw Notion MCP by hand is high-friction: heading-style and visual-style preferences are re-litigated every time; images silently fail (the MCP cannot upload local files); and tables frequently render with dropped rows/columns. /to-notion-doc is the purpose-built converter that designs that friction out.

Charter: pmos-utilities — a standalone document-format converter, sibling to /converter (JSON↔YAML, HTML↔MD, PDF↔MD). It neither ships a feature nor teaches a topic; it converts one artifact into another. resolved — maintainer chose pmos-utilities

2. Skill surface

  • Invocation: /to-notion-doc <path-to-source> [--parent <notion-page-url-or-id>] [--into <existing-page-url-or-id>] [--update-mode rewrite|archive|in-place] [--style minimal|expressive] [--headings toggle|normal] [--non-interactive]. Bare /to-notion-doc <path> is the happy path (create-new under a chosen parent); every flag has a natural-language form and a remembered default. --into targets an existing page (§ D11).
  • Inputs: a single local source file. Format inferred from extension; HTML and Markdown are parsed structurally, .txt is treated as paragraphs (blank-line separated) with light heuristics (ALL-CAPS / underlined lines → headings).
  • Output: a new Notion page under a user-chosen parent, plus an in-chat conversion report (block counts, ambiguities resolved, images handled, verification result) and — when images fell to the stub ladder — a local ./to-notion-doc-assets/<slug>/ folder beside the trigger dir.
  • Dependencies: the Notion MCP (page create + append-block-children + search). Optionally a Notion integration token for the File Upload API (images only — see §6). No npm dependencies; parsing is done in-skill / with a zero-dep parser script.

3. Control flow

Six phases, mirroring the maintainer's stated process:

  1. Resolve preferences (Phase 0). Read .pmos/settings.yaml :: to_notion_doc (heading style, visual style). Present-and-valid → confirm-against-last-run in one prompt (mirrors /feature-sdlc Phase 0e soft-gate-defaults). Absent → ask both, then persist. Flags override. resolved — prefs live in .pmos/settings.yaml
  2. Parse + block-map (Phase 1). Parse the source into a normalized block tree, then map each node to its best-match Notion block (§5). Collect an ambiguity queue of nodes with no faithful mapping (rich media: inline SVG/diagrams, embeds, cards, iframes, footnotes, complex nested layout).
  3. Resolve ambiguities + image strategy (Phase 2). For each ambiguity-queue entry, one AskUserQuestion offering the realistic conversions (e.g. an inline SVG → "upload as image" / "bookmark the source" / "skip with a placeholder callout"). If the doc has images, ask once whether a Notion File Upload API token is configured (§6); if not, fall back to the local-extract stub.
  4. Create-or-update + write blocks (Phase 3). Create-new: make the page under the parent. Update (--into, D11): resolve the target and the update mode (rewrite / archive-under-toggle-H1 / in-place); the archive mode wraps existing top-level blocks under a new collapsed heading_1 "Archive" first. Then write blocks in chunked, resumable batches (§7): ≤100 blocks/request, ≤2 nesting levels/request, create-then-append for deep nesting and large tables. Track a write cursor so a mid-write crash resumes.
  5. Verify completeness (Phase 4). Re-fetch the page's blocks and reconcile against the source block tree: every source block accounted for (mapped / explicitly-stubbed / user-skipped), none dropped (§8).
  6. Verify integrity (Phase 5). No orphaned blocks, no ragged tables (every row's cell count == table_width), no empty-on-error blocks, valid code languages, valid colors. Emit the report.

6. Image ladder (the core friction)

The official Notion MCP cannot upload local images (no file-upload tool; icon/cover accept external URLs only). So:

  1. If an integration token is configured (maintainer-confirmed at runtime; read from NOTION_TOKEN env or .pmos/settings.yaml :: to_notion_doc.notion_token_env): upload each local image via the File Upload API — POST /v1/file_uploads → POST /v1/file_uploads/{id}/send (multipart, key file) → append an image block {type:"file_upload", id}. Attach within the 1-hour window. Single-part ≤20 MB (≤5 MiB on a free workspace — downscale or warn past that).
  2. External URL already present in the source (![](https://…)): pass through as an external image (must be public HTTPS; Notion hotlinks, never re-hosts).
  3. Otherwise — local-extract stub (the always-available rung): copy the image to ./to-notion-doc-assets/<slug>/<name>, emit a callout ("🖼 Image: <relative-path> — <alt/name>") followed by an empty image placeholder block for the user to drag-drop fill in Notion. Verification (§8) counts these as accounted-for.

Google Drive URL trick is researched but rejected (Google throttles hot-linking; brittle).

7. Large-document chunking & resumable write

  • Hard API limits: 100 blocks per append request; 2 nesting levels per request.
  • Write algorithm: create the page with the first batch (≤100 top-level blocks), then PATCH /blocks/{id}/children in ≤100-block batches. For blocks needing depth >2 (nested toggles, columns, deep lists), emit to the limit, capture returned ids, append children to them.
  • Tables > ~99 rows: create with the header + first rows, append the rest in ≤100-row batches.
  • Resumability: persist a write cursor (page id + last source-block index written) to a run-state file; a re-run with the same source + page resumes from the cursor instead of re-appending (idempotency = source-block index). Bounds the blast radius of a Notion API hiccup on a big doc.

8. Verification passes

Completeness (Phase 4): re-fetch page blocks; build a reconciliation table source-block → disposition ∈ {mapped, stubbed, user-skipped}; fail loudly if any source block is unaccounted-for. Report the count.

Integrity (Phase 5): assert no orphaned blocks (no child whose intended parent is missing); every table_row.cells.length == table_width; no accidentally-empty blocks where source had content; all code.language in the enum; all colors in the 19-value enum. Any failure → surface in the report with the offending source location; offer a targeted re-write of just those blocks.

Both modes produce identical structure; expressive only adds icon/color/callout decoration. minimal is the default for the first run.

10. Preference persistence

Under .pmos/settings.yaml:

to_notion_doc:
  heading_style: toggle | normal      # D5
  visual_style: minimal | expressive  # D6
  last_parent: <page-id>               # D8 — suggestion only, never auto-applied
  notion_token_env: NOTION_TOKEN       # D2 — name of the env var holding the upload token (optional)
  updated: <YYYY-MM-DD>

First run: ask heading + visual style, write the block. Subsequent runs: one confirm-against-last-run prompt (Confirm all / Edit). Flags override for the run without rewriting settings unless the user opts to save.
