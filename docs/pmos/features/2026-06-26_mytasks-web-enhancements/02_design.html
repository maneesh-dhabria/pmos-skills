Design — /mytasks web UI enhancements

Epic 260626-a8a · Plugin pmos-toolkit · Route skill · Status defining · Date 2026-06-26

The cross-story design contract for raising the /mytasks web UI (shipped epic 260613-5av, currently pmos-toolkit/v2.93.0) to an "inline-everything" Todoist-class app, plus a new /mytasks import command. Grounded in a read of the current web layer — scripts/webapp/{app.js,app.css,index.html}, scripts/serve.js, scripts/lib.js, schema.md — and the sibling /people store.

1. Problem

The terminal verbs are full-featured, but the web app lags behind in the small interactions that make a task app pleasant, and it has no affordance for several capabilities the CLI already supports. From hands-on use, the gaps are:

  • Projects, labels, and people can't be created from the sidebar without leaving the app; there is no people surface at all.
  • Editing a task means opening the side panel — there is no inline title edit, and no @/ #/+ autocomplete while typing.
  • Subtasks show as separate top-level rows instead of nesting under their parent; opening a subtask gives no parent context.
  • The always-grey LNO flag left of the checkbox is unexplained noise; the checkbox is tiny; completed tasks aren't struck through.
  • No way to add a task below the list (only the bottom bar); no way to set type, recurrence, or a check-in note; Due/Start/Project/People are free-text rather than pickers.
  • No bulk capture — pasting an outline of work must be typed task-by-task.

2. Non-goals

  • No new persistent task fields. schema.md is unchanged (I2). Everything this epic exposes (type, importance, recur, parent, project, labels, people) already exists.
  • No "IC / Coordinate / Groom" type model (D1) — the 6-value type enum stays; the UI just makes it editable.
  • No dependencies / build step / CDN (I4).
  • No change to terminal behavior of record — the CLI keeps deriving projects/labels from task files; the registry (§4) only adds sidebar visibility, never hides.

4. Architecture & invariants

The web layer stays a thin, stateless view over markdown files (design of 260613-5av, unchanged):

  • I1 — files of record, stateless server. Every endpoint re-reads the store per request; writes go through lib.js atomic temp-then-rename. No server-side task state.
  • I2 — terminal ↔ web parity, additive only. Web mutations map to the same field edits the CLI makes; the task file shape never changes. New surfaces (registry, people CRUD) are additive.
  • I3 — optimistic concurrency. Task mutations keep expected_version; the 409 + reload-banner path is untouched.
  • I4 — zero-dependency. Native browser elements only — <input type=date> for pickers, custom div/datalist dropdowns for autocomplete. No npm/CDN.
  • I5 — import is agent-driven (not a server endpoint): the model parses + confirms; writes reuse lib.js + mint-id.mjs.

Registry (D5)

~/.pmos/tasks/registry.json = { "projects": [], "labels": [] }, created on first write. /api/projects / /api/labels POST add a slug-normalized, deduped entry. /api/meta returns { projects, labels } = the sorted, deduped union of registry entries and values derived from task frontmatter. A project that exists only via a task still appears (back-compat); an empty registry-only project also appears. The terminal never reads registry.json — it derives from task files as today, so the registry can only make the web show more, never less.

People over the web (D6)

/api/people reads/writes ~/.pmos/people/{handle}.md using the same frontmatter parse/serialize as lib.js (the people record shape is in /people schema.md: handle, name, optional designation/role/team/ email/aliases). Handle derivation reuses the /people lookup.md kebab rule. After any write the server regenerates ~/.pmos/people/INDEX.md so the CLI sees a consistent index. A person created in the web is fully usable from /people and vice-versa.

7. /mytasks import (story 260626-j9v)

A new import subcommand (a #routing entry + an #import phase in SKILL.md). Agent-driven (I5):

  1. Take the text inline (pasted after the command; prompt if absent and interactive).
  2. Parse structure-first (D3): a leading #project line → container; indentation depth → subtask of the nearest shallower line; bullet/checkbox markers → task lines; +label/@handle tokens and trailing dates → fields. Structure always wins on conflict; only the unresolved remainder is AI-inferred.
  3. Confirm the tree — print an indented preview (project → task → subtask with inferred labels/people/dates) and resolve real ambiguities with AskUserQuestion (carrying a (Recommended) default). Nothing is written until confirmed. Under --non-interactive the canonical block AUTO-PICKs the recommended reading and records ambiguities as open questions.
  4. Write — mint ids (mint-id.mjs), set parent for subtasks, apply project/labels/people/due, register new projects/labels, regenerate INDEX.md once. People tokens resolve via /people find; unresolved tokens are flagged, never invented.
  5. Report — list created tasks/subtasks (id + project + inferred fields), new projects/labels, and any unresolved @handle.
