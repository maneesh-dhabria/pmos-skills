Requirements — /backlog web viewer

Epic: 260613-p3c  ·  Route: skill  ·  Plugin: pmos-toolkit  ·  Tier: 2  ·  Date: 2026-06-13

Problem

The backlog is a repo-resident set of markdown files (backlog/items/*.md) plus a regenerated INDEX.md and a claims/ lock directory. Today the only ways to see the backlog are the terminal verbs (/backlog, groom, next, releases, list) and reading the raw markdown. For understanding status across the whole backlog at a glance — which epics are in flight, how far each is, what's ready to build vs. ship — a flat terminal dump is hard to scan. The user wants a visual, browsable view of the entire backlog and its status.

Solution direction

Add a read-only web viewer as a new web verb on the existing /backlog skill. /backlog web starts a lightweight local server that serves a single self-contained HTML file; the page reads the live backlog (parsed fresh on each request) and renders it for browsing. The viewer never mutates the backlog — all writes stay with the existing terminal verbs.

In scope

  • A web verb routed from /backlog that launches the viewer and opens the browser.
  • A zero-dependency Node server (loopback, ephemeral port) that, per request, parses backlog/items/*.md + reads claim locks and serves a derived JSON model; serves the single-file HTML viewer at GET /.
  • Single-file HTML viewer (inline CSS + JS, no build step, no external CDN) rendering three things the user asked for:
    • Epic → story tree with progress (status badges, done/total per epic).
    • Three release queues (groom / next / releases) mirroring the dashboard semantics.
    • Filters by status, route, and plugin (and grouping the whole backlog by them).
  • Live read: a refresh in the browser reflects the current on-disk backlog state.

Out of scope (this epic)

  • Any mutation — no claim/unclaim, status edits, item creation, or refine from the web UI. Viewer only.
  • Dependency graph view — explicitly deferred (user dropped it from the view set). A simple "blocked-by" hint inside a story row is acceptable if free; a graph is not built.
  • Workstream / multi-repo aggregation (the viewer reads the current repo's backlog/ only).
  • Authentication, remote access, or binding beyond 127.0.0.1.

Primary user & journey

The repo maintainer driving the three-loop backlog. Journey: runs /backlog web → browser opens → scans the epic→story tree to see overall state → flips to the three-queue view to see what needs grooming / what's ready to build / what's release-ready → filters by plugin or status to narrow → leaves it open, refreshing as work progresses.

Epic-level acceptance

  • /backlog web starts a local server and opens a browser to a working single-file viewer; Ctrl-C stops it.
  • The viewer shows every epic and story with accurate status and per-epic progress, derived using the same rules as the terminal verbs (schema.md status machines, #groom/#next/#releases membership).
  • A page refresh after an on-disk change (e.g. a new /backlog add) shows the change.
  • No request to the server mutates any file under backlog/.
  • Conforms to skill-patterns.md §A–§L and the repo's CLAUDE.md skill-authoring conventions (the standing acceptance criteria).
