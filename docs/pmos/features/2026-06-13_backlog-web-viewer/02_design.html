Design — /backlog web viewer

Epic: 260613-p3c  ·  Route: skill (this doc is the design_doc: coherence contract — there is no epic /spec)  ·  Plugin: pmos-toolkit  ·  Tier: 2  ·  Date: 2026-06-13

Single in-scope skill: /backlog. One story (260613-14b) builds the whole viewer — the web verb, the server, the HTML viewer, and tests ship together (one /execute, one /skill-eval). The G3 default of one-story-per-skill holds because the parts are not independently skill-eval'able.

Decisions

D1 — New web verb on /backlog, not a new skill. The viewer reads the same backlog/ store the skill already owns; a separate skill would duplicate the "where is the backlog" resolution and split the charter. Add one row to the #routing table and one #web handler section. (User-confirmed.)
D2 — Live read on each request, served by an own zero-dep Node script. The viewer must reflect on-disk state on refresh, so the static game-launcher (serve.js, serves exactly one file) is insufficient. A new script backlog/scripts/serve-web.mjs borrows the launcher conventions (loopback 127.0.0.1, ephemeral free port, default-browser auto-open, run-until-Ctrl-C, --no-open test seam) but adds a data route that parses the backlog per request. Node stdlib only (node:http/fs/path/child_process) — no dependencies. (User-confirmed: live read.)
D3 — Read-only. No mutation endpoints exist. Only GET / (the HTML) and GET /api/backlog (the JSON). No POST/PUT//save. The server opens no write handles to backlog/. (User: "Only looking to build a viewer.")
D4 — Single-file HTML viewer, inline everything, no network. One backlog/web/viewer.html with inline <style> + <script>; vanilla JS, no Tailwind/CDN/build. It fetch()es /api/backlog (same-origin localhost) and renders client-side. "Single file that gets served" is honoured; the only runtime input is the local JSON.
D5 — Derivation lives server-side, reusing the skill's existing rules. Epic rollups (done/total, in-flight), queue membership (groom/next/releases), and story readiness are derived in serve-web.mjs from the same logic the terminal verbs specify (schema.md status machines; SKILL.md#groom/#next/#releases; readiness = planned + deps all done, D21). The viewer renders; it does not re-derive backlog semantics. One home for the rules.
D6 — Views: epic→story tree + progress, three queues, filters (status/route/plugin). Dependency graph deferred. A story row MAY show a plain "blocked by #id" line when its ## Notes records it (free from existing data); no graph is built. (User-selected view set.)
D7 — Frontmatter parsing is shared, not reinvented. serve-web.mjs parses item frontmatter with the same minimal YAML-frontmatter reader the backlog scripts already use; if a reusable parser helper does not exist, add a tiny local one in backlog/scripts/ and have both the (existing) index path and the web path call it. Malformed files are skipped with a warning (mirrors #rebuild-index), never abort the server.

Server contract

Entry: node backlog/scripts/serve-web.mjs [--no-open] [--port N], run from the repo root (resolve <repo> via git rev-parse --show-toplevel, fallback cwd).

RouteMethodResponse
/GET, HEADbacklog/web/viewer.html (text/html)
/api/backlogGETDerived JSON model (below), parsed fresh from disk on every call
anything elseany404
  • Binds 127.0.0.1 on an ephemeral free port (or --port if given and free); prints Backlog viewer ready at http://127.0.0.1:<port>/; auto-opens the browser (open/xdg-open/start), degrading to a printed URL when headless or --no-open.
  • Node prerequisite error contract: if node is absent the web verb reports it and stops (no silent fallback).
  • Empty/absent backlog → /api/backlog returns empty arrays; the viewer shows an empty-state message.

/api/backlog JSON shape

{
  "generated_at": "<ISO>",
  "repo": "<basename>",
  "epics": [
    { "id","title","status","route","plugin","type","priority",
      "progress": { "done": N, "total": M, "blocked": K },   // non-wontfix stories
      "rollup": "inbox|defining|in-flight|all-stories-done|released",
      "stories": [
        { "id","title","status","route","type","priority","score",
          "claimed_by", "claim_stale", "dependencies": [ids],
          "blocked_by": "<note text or null>" }
      ] }
  ],
  "queues": {
    "groom":    { "needs_definition": [epicId], "needs_grooming": [storyId],
                  "blocked": [storyId], "stale_claims": [storyId] },
    "next":     { "pick": storyId | null },
    "releases": { "release_ready": [epicId], "in_flight": [epicId], "blocked": [epicId] }
  },
  "facets": { "statuses": [...], "routes": [...], "plugins": [...], "types": [...] }  // for filter UI
}

All derived fields (progress, rollup, queues, claim_stale) computed server-side per D5. claim_stale = claim lock older than the 4h TTL (read via the existing claim-lock status path). plugin derived from labels/route as #releases already does.

Viewer contract

backlog/web/viewer.html — three primary views, switchable via a top tab/segment control:

  • Tree (default): each epic as a collapsible row — id, title, status badge, route/plugin chips, and a done/total progress bar; expand to list its stories with per-story status badges and a blocked-by line when present.
  • Queues: three columns/lanes — Groom (needs-definition epics, needs-grooming + blocked stories, stale claims), Next (the single machine pick), Releases (release-ready / in-flight / blocked epics). Each row shows the copy-ready next command as text (display only — no execution).
  • A persistent filter bar (applies across views): multi-select status, route, plugin (built from facets); a clear-filters control; a result count.
  • Status badges use a small fixed colour map (e.g. inbox/draft grey, defining/ready/planned blue, in-progress amber, done/released green, blocked red, wontfix muted-strike).
  • A Refresh button re-fetches /api/backlog (live read). Page also fetches on load.
  • No write controls anywhere in the UI (read-only contract, D3).
  • Self-contained: opening viewer.html from file:// shows a graceful "start me with /backlog web" message when the fetch fails (no server).

Conformance & cross-skill invariants

  • Canonical path: all new files live under plugins/pmos-toolkit/skills/backlog/ (CLAUDE.md §Canonical skill path). No new skill dir.
  • skill-patterns.md §H gates: the routing decision is deterministic (verb match); derivation is script-side arithmetic (rollups, counts) — never model-judged at runtime.
  • §I flags: web is a verb, not a flag; --no-open/--port are server-script args (machine-coupled, headless determinism) and belong in the script, surfaced in the #web body. Update argument-hint to list web.
  • Non-interactive (W14): the web verb issues no prompts (like the machine verbs) — nothing to defer; the skill already carries the canonical block.
  • §E scripts: server + tests under scripts/ + tests/; bash/node portability per CLAUDE.md (stdlib only).
  • Read-only safety: loopback bind only; no 0.0.0.0; no write syscalls to backlog/ on any path.

Risks / open points (resolve at build)

  • Whether a reusable frontmatter parser already exists in backlog/scripts/ (mint-id/claim-lock don't parse items) — if not, D7 adds a tiny one; rebuild-index logic currently lives in SKILL prose, so the server implements parsing directly against schema.md.
  • Queue/readiness derivation must match the terminal verbs exactly — build verifies the web JSON against /backlog next --json / releases --json for the same repo state.
  • Live smoke at build: start the server, drive the served viewer in a browser (Playwright), assert tree + queues + filter all render against a real backlog — the load-bearing dogfood (a served-UI skill).
