T0 — the seniority question

Three candidate shapes, to be put to the maintainer:

  • Flat (188). One descriptor set per dimension; seniority is already expressed through weights and through which archetypes are run. Cheapest, and consistent with how the corpus works today — but "solid, some prompting OK" genuinely does mean something different for an APM and a Director.
  • Parameterized (188 + a rider). One descriptor set, plus a short seniority-sensitive clause in the skeleton contract stating how the bar shifts with level. Keeps the corpus at 188 while acknowledging the real variance; the rider is one fact in one home rather than 940 duplicated ones.
  • Per-seniority (~1,128). Full variants. Correct in principle, but it is a different-sized story and should be re-split rather than absorbed silently.

Whatever is chosen, AC1's count is restated to match before authoring begins.

Rationale for B

level-ladder.md already varies competency weights across the six levels, so the bar already shifts with seniority — through the weighting, not through the prose. B names that explicitly in one place instead of duplicating ~240 near-identical work-history descriptors that would drift apart on the first edit (skill-patterns.md §K: one fact, one home). C stays available later as its own story if calibration data shows the flat descriptors mis-fire at the APM and Director ends of the ladder.

T2–T5 are sequential rather than parallel on purpose: the descriptors must read as one calibrated voice across archetypes, and each batch is written against the running vocabulary the previous batch established. Batching by archetype (not by level) keeps each task's diff reviewable and each archetype internally consistent.

Authoring rules (from the contract, applied here)

  • Behavioural and observable in a transcript — what the candidate did, not what they are. "Named the tradeoff and picked a side" ✓; "strategic thinker" ✗.
  • The pass line permits prompting. "unprompted" / "up front" / "led with it" / "without being asked" appear at the top level only. This is the exact defect the epic exists to close; baking it into a level-3 descriptor would ship the bug into the corpus permanently.
  • Derived, not invented. Each descriptor traces to that archetype's existing strong/weak markers and probe intent. This is a re-expression of the corpus's own standard at four resolutions, not a new standard.
  • Distinguishable adjacent levels. If a reader cannot tell 2 from 3 by observable behaviour, the descriptor is not doing its job — the whole point is that the scorer can quote it and check the evidence against it.
  • Purely additive. No dimension added, removed, renamed, or reweighted.

Risks & mitigations

  • Authoring the ceiling into the pass line — at corpus scale. The epic's central defect, replicated 47 times and frozen into the shipped corpus. T6's grep for "unprompted / up front / led with it / without being asked" must hit only the highest data-v of each dimension; a hit anywhere else is a blocking failure, not a nit.
  • Silent drift in the frozen fields. 8 files of hand-editing is where a stray weight or dimension-id change hides. T6 asserts the git diff contains data-level attribute insertions and nothing else — the strongest available proof of AC5.
  • Scope explosion via T0. If per-seniority wins, do not absorb ~1,128 descriptors into this story — re-split and say so. T0's output is a recorded decision either way.
  • Voice drift across batches. Four authoring tasks over 47 dimensions can produce four different registers. T1's source table fixes the vocabulary up front and T2–T5 run sequentially against it.
  • Partial archetypes. All-or-none is per dimension, so a half-finished archetype can still pass the validator dimension-by-dimension while leaving the file incomplete. T6 counts dimensions per file against the expected table (5/5/5/5/5/5/5/12 = 47), not just validator exit codes.
