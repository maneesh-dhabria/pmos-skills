Design — compact-mode-setting

Epic route: skill  |  Triage source: 0c_feedback_triage.html

This design introduces a repo-level compact_mode setting that lets users who have configured an autocompact window skip the manual /compact instruction at every pipeline checkpoint. Two surfaces are affected: the /feature-sdlc compact checkpoint and the /execute phase-boundary HALT.

The Problem

Both pipeline checkpoints currently require the user to manually run /compact when they want to compact their context window:

  1. reference/compact-checkpoint.md — surfaces "Pause to /compact, then resume" before each heavy phase in /feature-sdlc. User picks Pause → types /compact → types /feature-sdlc --resume.
  2. _shared/phase-boundary-handler.md — emits "Run /compact to clear context, then re-invoke /execute --resume" at every passing phase boundary.

Claude Code supports configuring an autocompact window that compacts context automatically when the threshold is reached. When this is active, the manual /compact instruction is unnecessary — the user only needs to re-invoke with --resume.

The Setting

Cross-skill invariant (I1): Both stories read compact_mode from .pmos/settings.yaml. The field is absent from first-run setup — users add it manually when they configure autocompact. Both surfaces treat absent = manual for backward compatibility. 
# .pmos/settings.yaml — new field
compact_mode: manual   # or: auto  (values are case-sensitive; only lowercase 'manual' and 'auto' are valid)
Case-sensitivity: Values are case-sensitive. Only lowercase manual and auto are valid. Auto, AUTO, Manual etc. are treated as unrecognised and fall through to manual with a log warning.

Field contract

FieldTypeDefaultValuesScope
compact_mode string manual (absent field or absent file) manual | auto (lowercase only) repo-level only; no CLI flag, no workstream override
File-absent case: If .pmos/settings.yaml does not exist, settings is treated as an empty dict; compact_mode defaults to manual. The reading pattern below applies regardless of whether the file is present.
D1. Default is manual (absent field = manual). This preserves backward compatibility — existing runs behave identically without any settings change.
D2. Repo-level only — no --compact-mode CLI flag. The autocompact window is a persistent user environment preference, not a per-run choice. A settings field is the right granularity.
D3. Not prompted at first-run setup. compact_mode is an advanced setting that only matters once the user has configured an autocompact window. Adding it to the first-run prompt would confuse new users. Users who want auto add the field manually.

Behavioral Contract per Surface

Surface 1 — reference/compact-checkpoint.md (owned by story S1)

compact_modeBehavior at checkpoint
manual (default) Current behavior unchanged: fire AskUserQuestion with Continue (Recommended) / Pause to /compact, then resume / Continue without compacting.
auto Skip the AskUserQuestion entirely. Emit the log line as a standalone chat message before invoking the heavy phase: compact_mode: auto — checkpoint at <phase-label> skipped; autocompact active (where <phase-label> is the caller-supplied phase name, e.g. execute or verify). Then proceed immediately.
D4. In auto mode, skip the checkpoint entirely (no prompt at all). The "Continue" option is the Recommended default even in manual mode — in auto mode, there is no decision to surface; just log and proceed.
D4a. The log line is emitted as a standalone chat message before invoking the heavy phase, so it is visible at the phase-transition boundary rather than buried in phase output. The <phase-label> token is a human-readable name supplied by the caller (e.g., execute, verify, wireframes). Concrete example: compact_mode: auto — checkpoint at execute skipped; autocompact active.
Tier-1 gate unchanged: Tier-1 runs already skip the checkpoint entirely in both modes — the compact_mode reading is only reached at Tier 2+.

Surface 2 — _shared/phase-boundary-handler.md (owned by story S2)

compact_modeHALT_FOR_COMPACT message
manual (default) Current message unchanged: "Phase N verified green. Run /compact to clear context, then re-invoke /execute --resume to continue with phase N+1."
auto "Phase N verified green. Re-invoke /execute --resume to continue with phase N+1 (autocompact handles context)."
D5. Still hard-stop in auto mode. The HALT serves two purposes: (a) clean stopping point for phase discipline and (b) /compact instruction. In auto mode (b) is unnecessary, but (a) still applies — phase boundaries exist to control context growth and the user still needs to re-invoke. The RETURN HALT_FOR_COMPACT return value is unchanged; only the message string differs.
D5a. compact_mode: auto is a trust-based declaration — the skill trusts that autocompact is configured and active; it does not verify it at runtime. The failure mode is documented in phase-boundary-handler.md: if autocompact is not actually configured, context growth is unmanaged and the user is responsible for ensuring the autocompact window is set up before setting compact_mode: auto.

Reading the Setting

Cross-skill invariant (I2): Both skills that consume these reference files (/feature-sdlc and /execute) already read .pmos/settings.yaml at Phase 0. The compact_mode value is read from the already-loaded settings dict — no second file read. Both reference files describe the conditional logic; callers apply it inline.

Reading pattern (applies to both reference files):

compact_mode = settings.get("compact_mode", "manual")
# valid: "manual" | "auto" (lowercase only; case-sensitive)
# invalid / unrecognised → treat as "manual"; emit visible chat warning:
#   compact_mode: unrecognised value '<v>'; treating as manual
# file absent → settings is empty dict; compact_mode resolves to "manual"

Call convention: Each reference file (compact-checkpoint.md, phase-boundary-handler.md) includes a "Reading compact_mode" section that callers implement inline using the already-resolved value from Phase 0 settings. The reference files do not issue their own file reads. The caller reads .pmos/settings.yaml once at Phase 0 and passes the resolved compact_mode value forward.

Log channel: Both the auto-mode checkpoint log line and the unrecognised-value warning are emitted to chat as visible output (same channel as other Phase 0 diagnostic lines like mode: interactive (source: …)), not to a file or stderr only.

S1 Acceptance Criteria

  1. reference/compact-checkpoint.md documents the compact_mode field (values, default, where to read it from).
  2. In auto mode: the "When the checkpoint fires" section describes skipping the AskUserQuestion and logging the canonical line.
  3. In manual mode: existing behavior is unchanged (same AskUserQuestion, same Pause contract).
  4. Tier-1 bypass logic is unchanged (still skips in both modes).
  5. Invalid compact_mode values fall through to manual with a visible chat warning.
  6. The log line for auto mode is emitted as a standalone chat message before the heavy phase is invoked. Format: compact_mode: auto — checkpoint at <phase-label> skipped; autocompact active (e.g. … at execute skipped …).
  7. reference/compact-checkpoint.md documents the compact_mode field inline (values, default, call convention, log channel).

S2 Acceptance Criteria

  1. _shared/phase-boundary-handler.md documents the compact_mode field inline (values, default, call convention, log channel) — independent of S1's addendum.
  2. In auto mode: the HALT_FOR_COMPACT message changes per the behavioral contract table above — no /compact instruction.
  3. In manual mode: existing HALT message is unchanged.
  4. The HALT (hard-stop) behavior is preserved in both modes — only the message string differs.
  5. The RETURN value type (HALT_FOR_COMPACT) is unchanged.
  6. Invalid compact_mode values fall through to manual with a visible chat warning.
  7. Documents the trust-based nature of auto mode: skill does not verify autocompact is configured; failure mode documented.

Settings Schema Addendum

Both stories must add a documentation note for compact_mode wherever .pmos/settings.yaml fields are described. The canonical location is _shared/pipeline-setup.md Section A (first-run setup) — add a comment block after the existing fields describing the new field, its values, and when to set it to auto.

D6. Both S1 and S2 add their own minimal settings note for compact_mode — scoped to the context of each file (checkpoint vs. phase boundary). This avoids a cross-story dependency while still giving each reference file's reader the full field contract inline.
