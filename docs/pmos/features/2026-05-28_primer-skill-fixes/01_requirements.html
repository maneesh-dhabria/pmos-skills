Problem

The pmos-learnkit:primer skill in v0.1.0 produces under-sourced (~7 sources), under-sized (~3K words) primers on senior-PM topics that warrant 20+ sources and 5K+ words. Five independent gaps surfaced in a side-by-side comparison against a hand-curated reference (40+ sources, ~40K words):

  1. Practitioner-blind research — Phase 2 query frames are topic-only (<topic> overview, <topic> best practices); canonical voices on a topic are never named or searched for.
  2. Order-of-magnitude wrong source floor — fixed at 4 across all topics; passes thin primers on rich topics.
  3. No richness check at intake — vagueness heuristic only catches under-specified topics, not topics too narrow to support multiple frameworks / decision guides / case studies.
  4. No worked-examples instruction — the curator-lens framing prompt never directs the writer to surface named-company examples, so primers come out as abstract synthesis.
  5. No depth control — the user cannot ask for a deeper or briefer primer; word target is implicit at ~3K.

Constraint: keep /primer's identity intact — single-pass, ramp-up-before-meeting, citation-clean. Changes are in-phase instruction tightening only. Anything that would push /primer toward /rewrite's multi-pass architecture is out of scope.

Proposed solution direction

Five focused edits across two phases and two reference files:

  1. Phase 2 source discovery → two-step pattern (practitioners-and-books first, then queries per-practitioner / per-book free entry).
  2. Source floor → depth-tier-scaled (brief=6 / standard=10 / deep=15).
  3. Phase 1 intake → richness classifier added next to the vagueness heuristic (rich / thin / narrow-by-design).
  4. Phase 4 curator-lens prompt → worked-examples instruction; Phase 5 reviewer emits informational examples_per_h2_distribution field (no new check_id, no hard-block).
  5. New --depth brief|standard|deep flag + .pmos/settings.yaml :: default_primer_depth + first-run prompt. Word targets and source floors derive from the resolved depth tier.

User journeys

J1 — Senior PM rams up on a broad topic before a strategy review

User runs /primer "monetization and pricing for PMs". First-run prompt asks default depth → user picks standard. Richness check returns rich → proceed. Phase 2 names 8 practitioners (Gurley, Campbell, Berman, …) and 4 canonical books; runs per-practitioner + per-book-free-entry queries; lands 22 usable sources (above the standard-tier floor of 10). Phase 4 draft surfaces ≥1 worked example per H2 where available. User opens the primer, sees per-camp framing with named voices, walks into the review oriented.

J2 — PM wants a quick brief on a narrow topic

User runs /primer "feature flag rollout strategies" --depth brief. Richness check returns narrow-by-design → proceed with flag (outline phase notes no decision guide warranted). Phase 2 hits the brief-tier floor (6 sources) on a single retry. Ships ~2.5K words in <10 min, citation-clean.

J3 — PM tries an over-narrow topic

User runs /primer "user authentication" meaning something specific. Richness check returns thin with 3 broader reframings (e.g., "auth tradeoffs in B2B SaaS"). User picks a reframing → proceeds. Or user keeps as-is → logged to OQ buffer, run continues thin.

J4 — Returning user, second invocation in same project

.pmos/settings.yaml already has default_primer_depth: standard. No first-run prompt fires. User runs /primer "growth loops" --depth deep → CLI flag overrides settings. Phase 2 must hit floor=15 → may trigger one retry; if still under, source-floor gate surfaces (Abort / thin-disclosure).

Functional requirements (per-fix)

Fix 1 — Practitioner + book query generation (Phase 2)

  • FR-1.1 — Phase 2 MUST run a practitioner+book naming step BEFORE URL generation. Prompt: list 6–10 named practitioners/authors/institutional voices canonically associated with the topic; plus 3–5 canonical books/courses/textbooks. Each item carries a one-line "why canonical" note.
  • FR-1.2 — Naming output MUST persist into a new sources.json[].practitioner_index field (schema additive; existing consumers unaffected).
  • FR-1.3 — Query generation MUST run two strands in parallel:
    • Primary strand: per named practitioner → query <practitioner> <topic>. Per named book/course → query <book title> free entry point and <book author> <topic> (target: First Round Review, Lenny, podcast transcripts, author blogs, publisher excerpts).
    • Secondary strand: existing topic-frame queries (<topic> overview, etc.) run but are demoted — their sources count toward the floor only after the primary strand settles.
  • FR-1.4 — Books and paid courses MUST be cited via free entry point URL (e.g., First Round Review for Monetizing Innovation). The book/course is attributed in prose ("Ramanujam, Monetizing Innovation (Wiley, 2016)") with the free entry as the <a href>. No paid landing-page URLs in citations.
  • FR-1.5 — Robustness mitigation (from feedback OQ): if NO query for a named practitioner resolves to a usable source, the practitioner is silently dropped and logged to the OQ buffer. Prevents hallucinated-practitioner pollution.

Fix 3 — Source floor scales with depth tier

  • FR-3.1 — The Phase-2 source-floor gate threshold MUST derive from the resolved depth tier: brief=6, standard=10, deep=15.
  • FR-3.2 — The existing retry-once-then-gate flow (reference/source-floor.md §"Retry-once protocol") is UNCHANGED — only the threshold moves.
  • FR-3.3 — The existing 3-option gate (Abort / Continue with thin-source disclosure / Retry alternate frame) is reused verbatim; banner text substitutes the per-tier floor value.

Fix 5 — Topic-richness check at Phase 1 intake

  • FR-5.1 — Phase 1 MUST run a topic-richness LLM step IMMEDIATELY AFTER the existing vagueness check, returning one of rich / thin / narrow-by-design.
  • FR-5.2 — Prompt: "Is this topic broad enough to support (a) at least one named framework or model, (b) a meaningful decision-guide for practitioners, (c) at least one worked example or case study?"
  • FR-5.3 — On rich → proceed. On narrow-by-design → proceed; the outline phase notes the topic doesn't warrant a decision guide. On thin → surface a single AskUserQuestion offering 2–3 LLM-generated broader reframings + Keep as-is + Abort.
  • FR-5.4 — Under --autonomous the deferred default is Keep as-is, logged to OQ buffer with reason topic-thin.
  • FR-5.5 — This replaces the original-proposal Phase-3 topic-type classifier. NO frameworks/concept/methodology/process taxonomy is added — the outline phase remains shape-neutral.

Fix 6 — Worked-examples drafting instruction (Phase 4)

  • FR-6.1 — reference/curator-lens.md §"Phase-4 framing prompt" MUST instruct: "For each H2 section, try to surface ≥1 worked example — a named company, named product, named incident, or a clearly-labeled hypothetical scenario. Skip when no canonical example exists (do not invent)."
  • FR-6.2 — Phase 5 reviewer output schema MUST add an INFORMATIONAL field examples_per_h2_distribution — an array reporting the count of named/hypothetical examples per H2. NO new check_id. NO pass/fail. Surfaced as a report-only signal at the write gate.
  • FR-6.3 — No hard-block on examples missing. The "try but don't force, then report" pattern prevents the worst-case failure mode (invented examples on topics that genuinely don't have them).

Fix 8 — Three-tier depth + settings persistence

  • FR-8.1 — Three depth tiers:
    • brief: word target 2,000–3,000; source floor 6.
    • standard: word target 4,000–6,000; source floor 10.
    • deep: word target 7,000–10,000; source floor 15.
  • FR-8.2 — Resolution precedence: --depth CLI flag > .pmos/settings.yaml :: default_primer_depth > first-run prompt.
  • FR-8.3 — First /primer invocation in a project with no setting present: Phase 0.5 MUST surface a new AskUserQuestion "Preferred default depth for primers in this project?" with three tier options + descriptions. Recommended = standard.
  • FR-8.4 — On answer, MUST write default_primer_depth: <tier> to .pmos/settings.yaml.
  • FR-8.5 — Subsequent runs: read silently from settings; no prompt. --depth flag still overrides per-run.
  • FR-8.6 — --depth accepts brief / standard / deep. Unknown value → platform-aware error per Phase-1 conventions, exit 64.
  • FR-8.7 — Phase-4 word target floor/ceiling come from the resolved tier table; no continuous formula. Sizing precedence: tier table is authoritative.
  • FR-8.8 — argument-hint frontmatter MUST list --depth alongside existing flags.
  • FR-8.9 — .pmos/settings.yaml schema doc in _shared/pipeline-setup.md MUST document the new default_primer_depth field.

Open questions for /spec
