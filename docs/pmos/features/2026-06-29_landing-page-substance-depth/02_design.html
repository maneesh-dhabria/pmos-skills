Why this epic

A single /landing-page run shipped a clean, valid, self-contained page on the first pass — every deterministic gate (structure, single-CTA, mobile-overflow, asset-fidelity) went green. But the build was shallow on substance: it collapsed a 5-plugin marketplace into one "product builder" persona, undersold learnkit, used an unrelated framework SVG to stand in for the delivery pipeline, and told (named skills, listed features) where it should have shown. The user pushed back hard, twice, forcing a deep rebuild (parallel research agents, per-plugin personas, real artifact screenshots) and a third pass that added live interactive embeds.

The through-line: the skill's gates verify form (does a hero exist, is there one CTA, does it overflow on mobile) but never substance (does the hero convey each product's distinct value, is a claim shown or merely asserted, does a captioned visual depict what it claims). The five findings are all substance-depth gaps — two blockers, two friction, one nit. Every fix lands in the /landing-page skill: SKILL.md plus three of its bundled references (multi-product.md, section-scaffolds.md, copy-gates.md). No substrate change.

Scope & shape

This is a singleton skill epic (D18 wrap, same shape as 260629-pd2 / bm9 / 9ne): every fix edits the same skill — plugins/pmos-toolkit/skills/landing-page/{SKILL.md, reference/*} — and the FRs interlock (the Phase-2 claim-mapping feeds the Phase-6 hard gate; the new ladder rung feeds both), so the D24 litmus (a task in one story depending on a task in another) fails any multi-story split. One story (260629-t8z, route: skill). No _shared/ change, no script change, no new flag — every fix is a doc/contract edit inside the skill's own files, honouring CLAUDE.md §K one-fact-one-home (the rule lands in its canonical reference home; SKILL.md cites it, never restates).

Decisions (maintainer-approved, this define run)

IDDecisionProvenance
D1 Per-product persona derivation overrides the global 1–2 persona cap; one primary persona per product. When product_count: multi, the global "select 1–2 personas" rule is overridden: each detected product gets one primary persona (with its own jargon tolerance) and its own signature moments, written as a distinct brief slice. Phase 1 emits one brief slice per product before Phase 2 may run — a drafting precondition, not a post-hoc note. This kills the global-collapse failure without the 5×3 = 15-persona explosion a full triad-per-product would cause. F1 [blocker]; grilled via AskUserQuestion (chose "one primary persona per product" over "full triad per product" / "capped by product count").
D2 Phase 2 maps every feature/value claim to a show-surface before drafting is permitted. The existing Phase-2 map covers only the brief's signature moments; it is widened so that every page-level feature or value claim the structure intends to make is mapped to a concrete show-surface (embed / interactive snippet / video / carousel / annotated or plain screenshot — best-available first) before the draft is allowed. A claim with no available surface is either demoted (cut from the page) or explicitly flagged as tell-only — drafting cannot start with unmapped claims. F2 [blocker]; adopted (the Phase-2 half of the fix).
D3 do>show>tell becomes a HARD Phase-6 gate — asset-gated binary-presence, §H-clean. The copy-gates.md#show-ratio check is promoted from advisory to a hard gate: a page that makes feature claims AND has at least one capturable/embeddable show-surface available MUST show at least one claimed feature — a page that tells every signature moment in prose while showing none does not emit. The hardness rides a binary presence condition (≥1 show-surface present for a claimed feature), never an arithmetic ratio or token count (§H: deterministic = hard, arithmetic = script — so no counting). When no asset can exist for any claim, the gate is N/A and logged, so it never over-blocks a genuinely show-less page. F2 [blocker]; grilled via AskUserQuestion (chose "hard binary-presence gate, asset-gated" over "keep advisory, escalate to blocking prompt").
D4 Embed-the-real-artifact is the top "do" rung of the do>show>tell ladder. section-scaffolds.md#governing-principles gains "embed the real self-contained artifact (iframe, page-folder-relative)" as a first-class show-surface — the strongest proof for a repo whose value is producing real openable artifacts (iframe a real primer, embed a playable game). It ships with the load-on-open pattern (<details> + a data-src that swaps into the iframe src on first open) so the embed does not bloat initial paint or scroll. It honours the skill's self-contained constraint (D3 of the page): the embedded artifact is page-folder-relative and opens from file:// — never a remote URL/CDN. F3 [friction]; adopted.
D5 Phase 6 gains a judgment layer: two new checks in copy-gates.md. (a) Value-coverage — HARD structural for multi-product: a multi-product suite/overview hero fold MUST name each detected product with a distinct value line; a product missing from the hero blocks emit (structural presence — deterministic). (b) Asset-claim-match — ADVISORY judgment: each embedded visual must depict the thing its caption names (the Hooked-model-SVG-as-pipeline mismatch); this needs vision/judgment, so per §H it stays advisory — surfaced loudly in the reviewer pass, never silently, but does not block. F4 [friction]; grilled via AskUserQuestion (chose "hard structural for multi-product value-coverage; asset-claim-match stays advisory").
D6 Dev-tool hero subhead must name the buyer and the category explicitly. A clarity-rule addition in copy-gates.md: when product_type is a dev tool, the hero subhead must name who it's for (the buyer/persona) and what category it is in the headline region — the disambiguator may not be deferred to body copy (which is exactly where it landed on the weak first pass). Enforced in Phase 6 as a presence check on the subhead. F5 [nit]; adopted.

Non-goals

  • No change to the six bundled visual styles, the style-token sets, or the hero archetypes.
  • No promotion of any new check into a shared cross-skill substrate — every gate is scoped to /landing-page's own reference/copy-gates.md.
  • No automatic fan-out of the multi-product hub principle (still gated/chosen at Phase 2); FR-1 changes persona depth, not the suite/hub/single-focus choice.
  • No new flag; argument-hint stays as-is. The hard gates are contract text, not CLI surface.
  • No arithmetic show-ratio threshold — INV-3 forbids it; the gate is binary-presence only.
