1. Problem & positioning

A PM or founder has a product — a repo, a doc, or a description — and wants a high-converting landing page, not a generic AI-stamped hero-with-three-cards. The toolkit can author many artifacts but none produces a marketing landing page grounded in established conversion craft.

SkillProducesGap vs. this
/wireframesmid-fi internal UI screensapp screens, not a persuasive public marketing page
/prototypeclickable app journeysproduct UX, not a conversion-optimized sales page
/artifacttemplated docs (PRD, etc.)internal docs, no hero/CTA/conversion craft
/logoa single brand markone asset, not a full page
/diagramone SVGa visual, not a page

The missing primitive: a guided generator that turns product context into a single, portable, self-contained HTML landing page — opinionated about section order, hero craft, copy, and visual style — through a workflow that keeps the user in control at the three highest-leverage decisions (section structure, hero fold, visual style). The governing filter (Julian Shapiro) is baked in: Purchase Rate = Desire − (Labor + Confusion) — every section must add desire or remove labor/confusion, or it is cut.

The skill is a guided, gated pipeline. The three approval gates (structure, hero, style) are the maintainer's stated priorities; the rest is automated.

3. Section taxonomy (default scaffold)

Default order = Julian's 7-element backbone, expanded to a fuller CRO taxonomy. Each row is filtered by the governing equation. The proposer adapts the default by product-type before presenting it.

#SectionPurposeEssential / optional
1NavbarMinimal orient + top-right CTA (strip links on a 1:1 campaign page)Essential
2HeroSay exactly what you sell + primary CTA, above the foldAlways
3Social-proof stripBorrow credibility immediately (logos / "trusted by")If any proof exists
4Problem (PAS / JTBD)Name the pain in the visitor's wordsHeavy consumer/info; light dev-tool
5Solution / How-it-worksFrame the product as the obvious fix (3-step / demo)Essential
6Features as objection-handlingheader + paragraph (handles an objection) + image; each ties back to the hero's value propEssential
7Deeper social proofQuantified testimonials (name + role + company + photo)Essential at higher price
8Objection / FAQKill last doubts; "Is this for me?"Essential for paid/considered
9PricingClear offer + guaranteePaid only; omit for waitlist
10Repeat / final CTARestate the offer at peak intentEssential
11FooterTrust, legal, secondary linksEssential

Product-type variants (the scaffold the proposer starts from):

  • B2B SaaS — heavy logo bars, quantified ROI testimonials, security/trust, demo CTA.
  • Consumer app — lead with felt problem + outcome; lighter copy; lifestyle imagery; app-store badges.
  • Dev tool — compress the Problem section (devs know it); product-as-demo, code samples, dark mode.
  • Info-product / course (the growth.design archetype) — long copy, curriculum modules, instructor authority, money-back guarantee, "Is this course for me?".

Copy-length rule: free offer → short copy; paid → longer copy, effectiveness rising with price.

4. Hero fold

The hero is the page's single most important decision, so Phase 3 always explores it with real renders (D2), not descriptions. Elements (all above the fold): headline · subhead · primary CTA · optional secondary CTA · hero visual · trust strip. Four selectable archetypes:

ArchetypeShapeBest for
A. Benefit-led + product shotBold outcome headline + crisp UI screenshot/GIFB2B SaaS, productivity
B. Outcome + live demo"Product is the demo" framing + demo frameDev tools, complex products
C. Bold statement + illustrationProvocative one-liner + custom illustrationInfo-product, brand-led, consumer
D. Social-proof-forwardHeadline + immediate user-count / logo wall / ratingCategory leaders, waitlists

Hero rules (enforced regardless of archetype): headline passes Julian's litmus — "if the visitor reads only this text, will they know exactly what you sell?"; subhead ≤ 2 sentences; exactly one visually-isolated primary CTA (von Restorff); CTA copy is first-person + outcome-led ("Start free", "Book my demo" — never "Submit"/"Learn more"); the visual shows the product in action, never abstract stock imagery.

Phase-3 mechanism: emit working/hero-options.html in the page folder (a working artifact, not the final output) containing 2–3 fully-styled hero renders stacked for side-by-side comparison, each labelled with its archetype. The user picks an option (and may request copy/layout tweaks); the chosen hero is carried verbatim into Phase 5.

5. Visual style system (D1)

Two coupled deliverables: a bundled offline swatch gallery the user previews, and a theme-token set per style the generator binds at draft time. No full sample pages (rejected — they go stale and bias content); no description-only picker (rejected — no live preview).

The six styles

1. Clean minimal SaaSWhite + 1 accent, airyGeometric sans (Inter)LowCrisp product UILinear, Vercel, Notion
2. Dark developer toolNear-black + neon accentSans + mono codeMedium, bentoCode, live demosVercel, Supabase
3. Bold playful illustrationBright warm, high-contrastFriendly sans, big displayMediumCustom illustrationgrowth.design, Duolingo
4. Editorial / typographicCream + serifSerif display + sans bodyLow, breathing roomType-led, minimal imgPremium editorial
5. Warm consumer lifestyleSoft warm, photographicRounded humanist sansMediumReal-people photographyConsumer / wellness / D2C
6. Enterprise trustNavy/blue, restrainedCorporate sansHigherLogo walls, certs, badgesEnterprise B2B

Theme tokens. Each style is a named token set — CSS-variable values for palette (--bg, --fg, --accent, --muted, surfaces), type (display/body font stacks, scale, weight), spacing/density, radius, shadow, and an imagery directive (what the visual slots should contain). Stored as a frozen, validated reference data file (§10). The generator emits inline CSS from the chosen token set (§6) — themes are pure data, so a new style is added by adding a token set + a gallery swatch, never by editing the SKILL body.

style-gallery.html. One bundled, self-contained, offline HTML file rendering all six styles as labelled hero+section swatches (not full pages) from their token sets, so what the user previews is exactly what the generator will bind. The skill opens it (launcher / local server convention) at Phase 4. Design bias from the research: favor restraint + a point of view over commodity effects (shaders, bento, WebGL are now baseline) — the gallery shows tasteful, distinctive swatches.

6. Output contract

Per-page folder (D8) — each landing page gets its own directory so the brief, extracted reference material, and working artifacts live beside the final page and survive for later iteration:

{docs_path}/landing-page/<YYYY-MM-DD>-<title-slug>/
  brief.md            # Phase 1 persisted design brief (product + research + decisions)
  index.html          # the final self-contained landing page (the deliverable)
  reference/          # extracted source material (README excerpts, copy, asset notes, fetched refs)
  working/
    hero-options.html # Phase 3 throwaway hero renders (kept for traceability)
  • Final page format: a single self-contained index.html (D3) — inline <style> (CSS variables = the bound theme tokens) + vanilla JS only; no CDN, no build step; opens offline from file://. Matches the gamekit / single-file-artifact convention.
  • Responsive + accessible: mobile-first, semantic landmarks, alt text, prefers-color-scheme respected where the style allows, focus-visible CTAs, contrast-safe palettes (the token sets are pre-checked).
  • Placeholders, honestly marked: where the brief lacks a real asset (logo, screenshot, testimonial), emit a clearly-labelled placeholder (never invented fake metrics or fake customer quotes — §7 forbids unfalsifiable/fabricated proof).
  • Inline pmos-comments block + <meta name="pmos:skill" content="landing-page"> per the repo HTML-artifact convention, so the page is annotatable.

7. Copy & conversion gates (pre-emit self-review)

Phase 6 runs these deterministically-described checks and revises the draft in place before surfacing it:

  • Julian's headline litmus — "read only this, know exactly what's sold?"; no slogans; inject specificity.
  • Harry Dry's 3 tests on every headline/claim — Can I visualize it? Can I falsify it (numbers/dates)? Can nobody else say this?
  • Julian's 6-criteria review — Conversion · Interest (1–10) · Clarity · Expansion · Brevity · Disbelief.
  • One isolated primary CTA (von Restorff); 1:1 attention ratio on campaign pages (strip distractions).
  • Clarity > cleverness, always; benefits not self-congratulation; no wall of text; handle only major objections.
  • Psychology levers placed by section — social proof, authority, reciprocity, scarcity (only if true), commitment, liking, unity.
  • Anti-pattern avoid-list — corporate slogans, vague phrasing, bloated header, abstract stock imagery, vague testimonials, low-contrast CTA, wrong copy-length for price.

Visual self-check (D10). After the text gates, render index.html headless (Playwright), screenshot desktop + mobile into working/, and run a reviewer pass that confirms the hero/CTA/sections/style render correctly and look right (no overflow, contrast OK, CTA visible above the fold), fixing issues in place — a bounded ≤2-iteration loop, same pattern as /wireframes and /prototype. Graceful degradation: if no headless browser is available, fall back to the text gates + a structural HTML check and log the skipped visual pass (never silently drop it).

These also become the skill-eval acceptance surface: the skill must cite and apply these gates, and a generated page must structurally contain a hero with a single primary CTA, the approved sections in order, and the bound style's tokens.

8. Research & design brief (Phase 1)

Phase 1 is a real understanding step, not a one-shot extraction — it researches the context and persists a brief the rest of the run (and future re-runs) build on. It accepts, in priority of richness: a local repo path (read README, package manifests, docs, existing marketing copy) · a GitHub URL (fetch README + about) · a doc (PRD, one-pager) · a free-form description. Optionally, with the user's go-ahead, it fetches 1–2 reference/competitor pages the user names to inform positioning and style.

It writes brief.md into the page folder (§6) and copies any extracted source material into reference/. The brief captures: {name, what_it_is, audience, core_value_prop, differentiators[], product_type, proof_assets[], primary_action, secondary_action, tone, recommended_hero_archetype, recommended_style} — with each field cited to where it came from (repo file, doc, or "asked"). The brief is then confirmed with the user (correcting the brief is far cheaper than correcting the page); missing fields are asked, never invented (D6). The product_type drives the §3 scaffold variant and the §4/§5 archetype/style recommendations (D5). Persisting the brief makes the page folder self-describing and lets a later run regenerate or restyle without re-researching from scratch.

12. Stories

Story A — bundled style system + reference substrate (no SKILL.md). The independently-shippable, self-tested half: the 6 theme-token sets, the offline style-gallery.html, the section-scaffold / hero-archetype / copy-gates reference files, and a selftest. Produces a working, previewable style gallery from data with no LLM present. route: skill, plugin pmos-toolkit, no dependencies.

Story B — the /landing-page SKILL.md. The six-phase guided workflow (research+brief → propose-structure → hero-explore → style-pick → draft → self-review), the per-page folder output contract (D8), opt-in Phase-1 web research (D9), the Phase-6 visual self-check — render + screenshot + reviewer ≤2-loop with text fallback (D10), the non-interactive contract, and skill-eval. Cites design_doc: anchors #workflow, #hero-fold, #style-system, #output, #copy-gates, #ingestion. route: skill, plugin pmos-toolkit, depends on Story A.

13. Out of scope (v1)

  • Deploying/hosting the page (the user takes the HTML file where they want).
  • A/B test variants or analytics wiring.
  • Real image generation / logo design (use /logo; v1 emits labelled placeholders).
  • Multi-page sites or routing — one landing page per run.
  • CMS/React/framework output — single self-contained HTML only (D3).
