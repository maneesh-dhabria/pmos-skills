All shaping decisions are settled and binding; this doc frames them as requirements — it does not reopen them.

Problem

The shaped problem is trustworthy cross-source coverage, not "generate a brief." A PM's inbound arrives across email, team chat, calendar, and doc-collaboration comments; no single tool shows a trustworthy picture of what arrived and what it demands. Today the user either pays a manual tool-by-tool morning sweep (time + tab-thrash) or risks catching commitments late; and items caught in the sweep don't reliably become tracked tasks (discussed ≠ tracked). Three co-equal pains: fear-of-missing, triage time-cost, and the capture gap. The ★ sub-problem is staying trustworthy in week 4 — silent heuristic rot, source breakage, or a false sense of coverage is worse than no tool, because the user stops sweeping manually the moment they trust the view.

Falsifiable success signal: the brief is opened every workday and the manual all-tool sweep observably stops. Counter-evidence: the brief is read and the tools are still swept.

User & jobs

  • Who: an IC PM (or any knowledge worker with a multi-channel inbound mix) running Claude Code in a live session with MCP connectors authenticated for their declared sources.
  • Job 1 — see everything: "When I start my day, show me one trustworthy view of everything that arrived or is still open across my sources, ranked by what matters to me, so I don't sweep tools by hand."
  • Job 2 — act on it once: "Let me clear the view in one pass — track these as tasks, archive/resolve those at the source, leave the rest — with one confirmation."
  • Job 3 — keep it mine: "Let the ranking learn my rules from my corrections, and show me exactly what was and wasn't covered, so I can keep trusting it."

Functional requirements

IDRequirement
FR-1Declared sources. The user declares each source (kind, how to read it, relative priority, dismiss affordance) as configuration under ~/.pmos/morning-brief/sources.yaml. The skill assumes nothing about any user's channel mix or priorities; priority is user-indicated, never inferred. First run with no config enters guided setup (/morning-brief sources).
FR-2Sweep. Bare /morning-brief sweeps every declared source via its live connector: items new since the last-run cursor plus carryover items still unresolved in their source (unarchived / unreplied / unresolved comment) and not yet tracked. An unreachable source never fails the run silently — it lands in the coverage manifest.
FR-3Categorize + rank. Each item is categorized against the user's rules (seeded from a small generic GTD-4D taxonomy: do / delegate-reply / defer-track / drop-FYI) and ranked into prominence tiers. Heuristics decide prominence only, never inclusion.
FR-4Show-everything brief. One static HTML brief is emitted per run to ~/.pmos/morning-brief/briefs/: prominence tiers at top, FYI tier collapsed to expandable counts, and every in-window item present at least as a one-line FYI row — recall is structural.
FR-5Coverage manifest. Every brief carries a per-run manifest: per-source swept/failed status, item counts, the cursor window, and items no heuristic matched. Never a silent gap.
FR-6/mytasks lane. The brief includes a read-only lane from /mytasks (due / overdue / check-ins / waiting-on). /mytasks remains the sole system of record for tasks.
FR-7Batch confirm. The run ends in one editable proposed action set — create these /mytasks tasks (deduped against existing items), perform these source-native dismissals (archive email / resolve comment), leave these — reviewed in chat and confirmed once. Nothing executes unconfirmed.
FR-8Derive, don't store. No per-item triage state is persisted. Re-running derives the view from live source state. Only the last-run cursor (cursor.yaml) plus user config (sources.yaml, rules.md) persist.
FR-9Observe + correct rules. User corrections at the confirm step ("this should have been top-tier") become candidate rules; the user approves them into rules.md (human-readable, hand-editable). Cold start with the seed taxonomy only — no priors seeded from /people or history.
FR-10Rules verb. /morning-brief rules shows the current rule file and supports add / edit / retire — the same observe-and-correct file the confirm step appends to.

Constraints & non-functional

  • Privacy (hard): all state and every emitted brief live under ~/.pmos/morning-brief/. Work-comms content is never written inside any git repo. LLM processing of work content on every run is an explicitly accepted premise — stated, not implicit.
  • On-demand, live session: runs when invoked in an authed interactive session; no cron dependency. Headless runs without authed connectors degrade loudly via the manifest.
  • Inform + recommend only: the user confirms and triggers every action. Source-native dismissal gives the skill confirm-gated write access to work systems — always user-confirmed, and DEFER (destructive) under --non-interactive.
  • Zero new npm dependencies (repo norm); deterministic work (cursor I/O, manifest counts, brief rendering) in scripts per §H; judgment work (categorization, ranking, rule synthesis) stays LLM.
  • Non-interactive contract: sweep + brief emit run unattended (Recommended → AUTO-PICK); batch confirm and dismissals DEFER; W14 block inlined byte-identical.

Out of scope (v1)

  • A served interactive brief (serve.js web UI with in-page confirm) — the v1 brief is static HTML; the action set is confirmed in chat.
  • Scheduled / cron runs and push delivery.
  • Slack and Google-Drive-comment sources in the bundled examples — no Slack MCP in the reference environment and Drive MCP lacks comment threads; the abstraction supports them the day a connector exists, and the manifest names them as declared-but-unreachable if configured.
  • Numeric speed/clear-time targets — adoption (sweep stops) is the only success measure.
  • An ordered day-plan / calendar-blocking framing — rejected at shape.
