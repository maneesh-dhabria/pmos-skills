Design — /morning-brief

pmos-toolkit · epic 260702-kck · route: skill · the cross-cutting coherence contract for stories 260702-b6q and 260702-ww7. Companion to 01_requirements.html; grounded in the grill-hardened shape brief. Stories cite this doc by anchor.

1. Overview

/morning-brief turns a user-declared set of inbound sources into one trustworthy, ranked, show-everything morning view, and lets the user clear it in one confirmed pass. It is a pure derived view: sources and /mytasks are the only state stores; the skill persists only user configuration, personal rules, and a last-run cursor. It is the repo's first connector-reader skill — the source abstraction (§4) is the load-bearing generic layer.

bare /morning-brief          sweep → categorize+rank → brief HTML + manifest → batch confirm → act → cursor
/morning-brief sources       declare / edit sources + priorities (first-run guided setup)
/morning-brief rules         view / add / edit / retire personal categorization rules

2. Invariants

  • INV-1 — Derive, don't store. No per-item triage ledger, ever. The view is recomputed from live source state + /mytasks each run; the only run-state written is cursor.yaml. An item leaves the brief only by being resolved (acted on in its source, or tracked), never by being seen.
  • INV-2 — Show everything, structurally. Every in-window item appears in the brief at least as a one-line FYI row. Rules and ranking set prominence, never inclusion. A judgment miss must be detectable as "ranked too low", not invisible.
  • INV-3 — User-declared, never assumed. Source presence AND relative priority are user configuration. The skill ships zero assumptions about anyone's channel mix; bundled examples are examples, not defaults.
  • INV-4 — Privacy residency. All config, state, and emitted briefs live under ~/.pmos/morning-brief/. Work-comms content is never written inside any git repo. LLM processing of work content is an accepted, stated premise.
  • INV-5 — Confirm-gated actions. The skill informs and recommends; every mutation (task creation, source-native dismissal) executes only from the user-confirmed batch action set. Under --non-interactive, mutations DEFER (destructive) — never auto-executed.
  • INV-6 — /mytasks sole system of record. The brief's task lane is read-only; task creation goes through /mytasks' own data contract with dedupe. No parallel task store.

4. Source abstraction

A source is anything with a connector the session can call. The skill defines a per-kind read contract and resolves concrete MCP tools at run time (ToolSearch by the declared connector hint) — it never hardcodes a tool inventory:

KindRead contract (per run window)Native dismiss
emailthreads new-in-window + threads still unarchived/unreplied (carryover)archive / label
calendartoday + tomorrow's events; new invites needing responserespond to invite
doc-commentscomment threads mentioning/assigned to the user, new + still-unresolvedresolve / reply
chatmentions + DMs new-in-window (declared only when a connector exists)varies / none
customuser-described read instruction interpreted at run timenone unless declared
  • Unreachable ≠ silent: a declared source whose connector is missing, unauthed, or erroring is reported in the manifest (§7) as failed: <reason>; the run continues with the remaining sources. Kinds with no dismiss affordance stay visible at low prominence (INV-1 — nothing to resolve them with except tracking or source action elsewhere).
  • Reference environment note: Gmail / Google Calendar / Notion MCP exist here; Slack does not, and Drive MCP lacks comment threads — exactly why presence is declared per user, not assumed (INV-3).

All flags NL-first per §I; no machine-coupled flags in v1 beyond --non-interactive (inherited contract).

6. The run pipeline (bare verb)

  1. Sweep each declared source per its §4 read contract, window = cursor → now (first run: first_window_days back, D9 — stated in the brief header). Carryover reaches back to carryover_horizon_days; older unresolved items are counted, not rendered (D10). Collect the /mytasks read-only lane (due / overdue / check-ins / waiting-on) from the tasks store.
  2. Categorize every item against rules.md (LLM judgment; the rules file is the prompt's rubric). Items no rule matches are categorized by the seed taxonomy AND flagged no-rule-matched for the manifest.
  3. Rank into three prominence tiers using category + user-indicated source priority: Needs you today · Worth knowing · FYI (collapsed to per-source counts, expandable). Every item lands in exactly one tier (INV-2).
  4. Emit the brief (§8) + manifest (§7); print the absolute path.
  5. Batch confirm (§9): one editable proposed action set, one confirmation.
  6. Act on the confirmed set: create tasks via the /mytasks contract (dedupe first), perform source-native dismissals via each source's connector. Report per-action success/failure verbatim.
  7. Correct: offer to capture any user re-tiering/re-categorization from the confirm step as candidate rules → approved ones append to rules.md.
  8. Cursor: write cursor.yaml atomically. Steps 5–7 skipped/deferred under non-interactive (§10) — the cursor still advances only when the sweep completed cleanly for all reachable sources.

7. Coverage manifest

A fixed section at the top of every brief — the week-4 trust mechanism. Per declared source: swept ✓ / failed ✗ (reason), item counts (new / carryover / shown-by-tier), the exact cursor window, the list of no-rule-matched items, and per source "N older items beyond the carryover horizon" (D10 — the cap is visible, never silent). Deterministic counts are script-computed (§H), never LLM-estimated. A run with any failed source says so in the brief header line, not only in the manifest body.

8. Brief artifact

  • Self-contained static HTML at ~/.pmos/morning-brief/briefs/YYYY-MM-DD.html; inline CSS; zero external requests; no repo substrate payload (it lives outside the repo, INV-4 — the pmos-comments overlay contract does not apply to home-dir artifacts).
  • Layout: header (date, window, one-line coverage status) → manifest (§7) → Needs you today → Worth knowing → /mytasks lane → FYI (per-source <details> collapsed counts). Each item row: source badge, one-line summary, category, why-this-tier (the rule that fired, or no-rule-matched), and a deep link to the item in its source when the connector provides one.
  • The proposed action set is printed in the brief for reference but reviewed and confirmed in chat (v1 decision D4) — the HTML has no write path.

9. Batch confirm + /mytasks interop

  • One proposal, one confirm: the skill proposes {create-task[], dismiss[], leave[]} as a numbered editable list — mirror of /mytasks import's preview→confirm shape. The user edits by number ("move 3 to dismiss", "retitle 5"), then confirms once. No per-item approval chain.
  • Task creation: through /mytasks' own data contract (its quick-add grammar / lib in plugins/pmos-toolkit/skills/mytasks/scripts/) so minted items are byte-compatible; each created task carries a source link in its body. INV-6.
  • Dedupe rule: before proposing a create, match against open /mytasks items by source link (exact) then by title similarity (judgment); matches downgrade the proposal to "already tracked — shown in lane".
  • Dismissals: executed via the source's declared native action (§4). Per-item failure is reported and leaves the item for the next run (INV-1 — the source still holds it, so it derives back in as carryover).

10. Non-interactive contract

  • Sweep → categorize → rank → emit (steps 1–4) run unattended; structured prompts with a Recommended option AUTO-PICK per the W14 block (inlined byte-identical in SKILL.md).
  • The batch confirm is tagged <!-- defer-only: destructive --> — the action set is written into the brief + OQ flush and nothing executes. Rule-correction capture likewise defers.
  • First-run setup (sources) under non-interactive DEFERs (free-form) — a source declaration cannot be auto-picked.

11. §H / §I / §L notes

  • §H: deterministic work in scripts — cursor read/write (atomic), sources.yaml parse/validate, manifest count assembly, brief HTML rendering from a JSON model, dedupe-by-source-link exact match. LLM keeps judgment: categorization, tiering, summarization, title-similarity dedupe, rule synthesis. Scripts zero-dependency Node, each with --selftest.
  • §I: NL-first surface; argument-hint carries only the verbs + --non-interactive.
  • §L: per-source sweeps may dispatch parallel subagents (sonnet — bounded read+extract per source, parent validates); categorization/ranking stays in the parent (inherit) because it needs the whole picture and the user's rules.

12. Decisions

IDDecisionRationale
D1Verb surface = bare run + sources + rules.Smallest surface covering setup and rule stewardship; no standalone dismiss verb — dismissal lives only in the confirmed batch (INV-5), and re-runs re-derive anything left (INV-1).
D2Data layout = sources.yaml / rules.md / cursor.yaml / briefs/ under ~/.pmos/morning-brief/.Rules as prose markdown for hand-editing; cursor isolated so config never churns; follows /mytasks//people home-dir residency.
D3Rules live in a dedicated rules.md, not ~/.pmos/learnings.md. Personal triage rules are user content, not agent-behavior learnings; different lifecycle and ownership.
D4v1 brief is static; batch confirm happens in chat.No server/write-path plumbing in v1; matches the /mytasks import confirm shape the user already knows. A served brief is a natural later epic.
D5Connector resolution is runtime ToolSearch by declared hint.First-of-kind connector-reader: hardcoding tool names would rot; the declared hint + kind contract keeps the abstraction generic (INV-3).
D6Cursor advances only after a clean sweep of reachable sources.A crashed run must not eat items; failed sources don't block the cursor (they're carried in the manifest and re-swept by derive-on-read anyway).
D7Same-day briefs never overwrite (suffix -2…).A rerun after actions is a new derived view; the morning's record stays.
D9First-run window is set at guided setup (first_window_days, default 7), stated in the brief header.Grill follow-up: no cursor on run 1; a fixed 24h misses weekends, an unbounded carryover-only first sweep is unpredictable in cost.
D10Carryover horizon (carryover_horizon_days, default 14): older unresolved items drop from FYI rows but are counted per-source in the manifest.Grill follow-up: unbounded carryover makes old noise accumulate — the exact week-4 trust killer; the manifest count keeps the gap detectable, preserving INV-2's spirit (visible, not invisible).
D82 stories, sequential (b6q core coverage → ww7 action lane).Both edit one SKILL.md → dependency, not parallel; S1 alone is independently verifiable (brief + manifest with no mutations), S2 adds the mutation lane behind the confirm.
