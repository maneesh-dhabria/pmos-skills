Design — /logo skill refinements (epic 260617-n7a · route: skill · pmos-toolkit)

This epic refines the /logos skill (built in 2026-06-13_logos-skill, story 0613-36f) from one retro run, and renames it /logos → /logo. The skill decomposes a brief into N named logo needs, extracts a style-profile, authors 2–3 self-contained SVG variants per need conditioned on a bundled theme, gates each through a deterministic SVG-metrics script plus a renderer-backed vision check, and emits a self-contained showcase. Today its biggest gap is upstream of generation: it picks the idea and the visual style silently and never seeks the user's input — no research, no clarification, no approval — so the first time the user sees a direction is when eight finished SVGs already exist. The bulk of this epic adds a concept- and style-exploration phase with explicit approval in front of generation; the rest are smaller correctness/UX fixes from the retro.

Product decisions on the genuinely open questions (exploration shape; clarification scope; rename scope; aspect-gate fix) were resolved with the maintainer up front via structured questions — see D1–D8. The exploration reframe (D2/D3) came directly from the maintainer's insight that a "theme" today is only a rendering style, never the concept the mark expresses.

Functional requirements

FR1 — Rename to /logo

  • FR1.1 — The skill directory is plugins/pmos-toolkit/skills/logo/ (via git mv, preserving history) and frontmatter name: logo (a-name-matches-dir).
  • FR1.2 — Every user-facing and internal reference to the command is /logo (announce line, prose, anti-patterns, spec-lineage).
  • FR1.3 — Output artifacts rename: showcase logo.html; sidecar <run-slug>.logo.json; cache dir ~/.pmos/logo-cache/; default out dir {docs_path}/logo/<run-slug>/; the showcase carries <meta name="pmos:skill" content="logo">; the learnings header is ## /logo.
  • FR1.4 — Repo-wide sweep: update live /logos / skills/logos references (root README.md, plugins/pmos-toolkit/README.md skill row, any sibling-skill citations). Historical references (changelog entries, other features' spec-lineage) are left as-is.
  • FR1.5 — No /logos back-compat alias is created.
  • FR1.6 — No dangling reference remains: a repo grep for live /logos\b, skills/logos, logos-cache, and content="logos" returns only intentional-historical hits.

FR2 — Clarification gate

  • FR2.1 — A deterministic "thin-brief" check decides whether to clarify: a brief is thin when it lacks an explicit usage context AND lacks any color/vibe signal (from text or --ref). Rich briefs skip the gate (logged, not silent).
  • FR2.2 — When thin, fire ONE consolidated AskUserQuestion covering: primary usage context (favicon / app-nav / marketing/brand), color or vibe leanings + any hard constraints, and must-work-in-one-color (yes/no). Each sub-choice carries a Recommended default.
  • FR2.3 — Answers are merged into the style-profile and threaded into the concept + style proposals (FR3/FR4).
  • FR2.4 — Under --non-interactive, the gate defers per the canonical block (questions recorded to the OQ buffer) and best-fit defaults proceed — no deadlock.

FR3 — Concept exploration & approval

  • FR3.1 — After decomposition, the skill researches the subject (text brief + any --ref signals + clarify answers) and proposes 2–3 distinct concept directions. Each concept names: the idea/metaphor, what it depicts, the symbolism, and the subject's conventional colors (e.g. solitaire → card-red + felt-green).
  • FR3.2 — The concepts must be genuinely different ideas (not three renderings of one) — the same divergence discipline the generation phase already requires of variants.
  • FR3.3 — Interactive: one AskUserQuestion to approve a concept (or two), edit, or re-explore; the message is self-contained (lists each concept + its metaphor). Non-interactive: AUTO-PICK the best-fit concept, recording the choice + the alternatives weighed to the logo.json sidecar and OQ buffer.
  • FR3.4 — The approved concept(s) are recorded as the binding direction for generation; the concept is never silently invented inside Phase 4.

FR4 — Style exploration & approval

  • FR4.1 — Alongside the concept proposals, the skill proposes 2–3 candidate styles drawn from the 6 bundled themes, each chosen to serve the leading concept, with a one-line rationale.
  • FR4.2 — Each style proposal states its color ceiling and flags when the leading concept's natural palette exceeds it (D7), recommending a better-fit style (e.g. suit-red ⇒ prefer gradient-duotone over flat-minimal).
  • FR4.3 — Interactive: a separate approval (the second axis) selects the style; an explicit --theme flag still overrides and is surfaced as the pre-selected option rather than skipping the approval. Non-interactive: AUTO-PICK the best-fit style + record rationale.
  • FR4.4 — The approved style governs the per-variant technique/palette in generation exactly as --theme does today.

FR5 — Generation consumes the approved direction

  • FR5.1 — The generation phase authors variants within the approved concept(s) × approved style — diverging on structural dimensions, not re-deciding the idea.
  • FR5.2 — Per-need mark type is passed to the evaluator's deterministic gate (FR6) so lockups are checked with the right bound.

FR6 — Mark-type-aware aspect gate

  • FR6.1 — svg-metrics.mjs accepts --mark-type <type> (and --mark-type=…). Square-context types keep [0.8, 1.25]; lockup types (combination, emblem, wordmark) use [0.8, 4.0].
  • FR6.2 — Absent the flag, the bound is the current [0.8, 1.25] (back-compat — existing callers and the svg-metrics.test.mjs default cases are unaffected).
  • FR6.3 — An unknown --mark-type value is a usage error (exit 64) with the valid list, mirroring the --theme/--renderer enum handling.
  • FR6.4 — The icon-only deliverable of a lockup need is gated as a square icon (it must work at favicon/nav size), not as a lockup.
  • FR6.5 — eval/code-metrics.md documents the new bound table; the hard-fail id stays viewbox-not-square.

FR7 — Learnings approval rule inlined

  • FR7.1 — The learnings phase text states the rule directly: surface the exact proposed bullet(s) in chat, get y/n/edit approval, and never silently append to ~/.pmos/learnings.md — independent of opening _shared/learnings-capture.md (which it still cites for format).
  • FR7.2 — Under --non-interactive, the proposed bullets are recorded to the OQ buffer rather than auto-written (no silent disk-append).

Invariants (coherence contract)

  • Inv-1 ($0 authoring): SVGs are authored in-session; no paid generation API, ever (unchanged core constraint, shared with /diagram).
  • Inv-2 (renderer hard-gate): the Phase 0 renderer detection (Playwright → rsvg-convert → cairosvg) stays a hard gate — refuse to run with none present.
  • Inv-3 (metrics are script-only): color counts, stroke ratios, and the new mark-type aspect bound are script arithmetic (§H); the model judges vision only.
  • Inv-4 (rename completeness): the rename is complete and consistent — no dangling live /logos/skills/logos reference; name: logo in skills/logo/ (a-name-matches-dir holds).
  • Inv-5 (svg-metrics back-compat): absent --mark-type, behavior is byte-identical to today; pre-existing svg-metrics.test.mjs cases pass unchanged (new cases are additive).
  • Inv-6 (non-interactive contract): every new gate (clarify, concept, style) carries a Recommended option or a defer-only tag; the inline non-interactive block stays byte-identical to _shared/non-interactive.md; audit-recommended.sh + lint-non-interactive-inline.sh stay green.
  • Inv-7 (skill conformance): the skill conforms to skill-patterns.md §A–§L and the host CLAUDE.md (canonical path plugins/pmos-toolkit/skills/logo, no release-prereq tasks in the plan). lint-phase-refs.sh + lint-flags-vs-hints.sh stay green across the phase renumber.

Open questions / deferred

  • Per-need concepts (deferred): the maintainer chose run-level two-axis approval; proposing/approving a concept separately per decomposed need was the runner-up. Revisit if multi-need briefs feel under-served.
  • New themes (out of scope, D7): a richer multi-color theme for inherently colorful subjects is not authored here — the fix is better selection among the existing six. A future epic could add one.
  • Lockup aspect ceiling: 4.0 is the proposed wide bound; the implementer may settle in 3.0–4.0 based on what real wordmark/combination lockups need during the build self-check.
  • Thin-brief threshold: the exact "thin vs rich" rule (FR2.1) is a heuristic; the implementer tunes it so rich briefs don't get nagged and bare briefs always clarify.
