1. Problem & positioning

A PM finishes a PRD/strategy doc and wants the review a seasoned product leader would give before exec sign-off: where is this weak, what's missing entirely, and which load-bearing claims won't survive scrutiny? The toolkit critiques slices of this but never the whole product-doc as an opinionated standalone peer.

SkillCritiquesGap vs. this

/grilla committed artifact, liveneeds a human in the loop; no written scorecard

The missing primitive: a standalone, opinionated, scannable verdict-by-axis scorecard with a forced-ranking of the weakest claims, as a written artifact the author can act on without a live session. Envisioned arc (D5): /artifact (generate) → /artifact-critique (review) → /artifact refine (rewrite to findings) → loop, eventually orchestrated by a future /artifact-sdlc. Must also run standalone on any doc path the user passes.

2. The reverse-engineered framework (the substance to replicate)

2.1 Fixed 10-axis rubric (the coverage surface)

Every doc is scored on the same 10 axes, in this order, regardless of type:

Customer · Solution · Scope · Metrics · Pricing · Strategy · GTM · Stage · AI · Risks

The list is fixed precisely so omissions surface — an axis the author never addressed gets an ABSENT verdict (the highest-value output).

2.2 Verdict vocabulary — single ordinal scale + reason tag (locked)

VerdictMeaning
STRONGpresent, concrete, evidenced — a first-class, freely-given verdict
MIXEDpartially there; strengths + real gaps
WEAKgestured at, not concrete/quantified
ABSENTexpected for this doc-type but not addressed (a real defect)
N/Anot applicable to this doc-type — but stated, never silently skipped

The sharp shape-language ("output, not outcome"; "wedge, not a moat") lives in the free-text reason, keeping the at-a-glance scorecard a clean ordinal. ABSENT vs N/A is decided deterministically by the doc-type applicability map (§4.3), never ad hoc.

2.3 Critique structure (identical skeleton every run)

  1. Opening framing — names the doc-type and commits: "On a careful read, this is a [type]… I'll push hardest on [up to three things]."
  2. Verdict by Axis — the scannable scorecard (10 × verdict + one-line summary).
  3. Per-axis deep-dives — each header is itself a verdict; pulls a verbatim block-quote from the source, interrogates it, closes with a prescriptive "What I'd want to see:" ask (a concrete artifact).
  4. Weakest Claims (0–3) — ranked load-bearing assertions + bulleted follow-up questions. May return fewer/none; never padded to a quota.
  5. Bottom line — synthesis; credits strengths, names the 3 must-dos.

2.4 Cross-cutting heuristics (the reasoning spine — doc-type-agnostic)

These recur on every axis and are the real intelligence — extracted as heuristics.md (Inv-1):

  • Assertion vs. evidence — "asserted, not demonstrated." Demands verbatim user quotes / incident data / ticket counts, not category descriptions.
  • Hypothesis falsifiability — every solution needs a stated if/then with a named mechanism. "A list of features is not a hypothesis."
  • Outcome vs. output metrics — every metric needs baseline + target + timeframe + counter-metric + kill/scale/graduate threshold.
  • Durable vs. current advantage — "Affordable pricing is a wedge, not a moat."
  • Stage-fit — scope/rigor must match lifecycle stage.
  • AI as a risk surface — any AI feature needs a Behavior Contract (GOOD/BAD/REJECT), fallback/kill-switch, eval metrics, red-team list.
  • Pre-mortem — "6 weeks post-launch and it failed — what happened?" Generates missing failure modes.
  • Alternatives-considered; Scope IN/OUT/CUT discipline; multi-sided completeness (every affected party); no-burial (load-bearing content in the body, not an annexure).

4. Resolved define-time open items (brief §6 → concrete contracts)

4.2 axes.md — per-axis checks (the spine mapped onto axes)

axes.md declares, for each of the 10 axes: a one-line scope, the concrete checks (which §2.4 heuristics apply), and the "what I'd want to see" template. Illustrative (authored fully in story fbd):

AxisCore check(s)
Customerevidence-vs-assertion; verbatim user quotes / segment sizing — not a category description; multi-sided completeness
Solutionfalsifiable hypothesis w/ named mechanism; alternatives-considered; "a list of features is not a hypothesis"
Scopeexplicit IN / OUT / CUT lists; v1 cut decisions; stage-fit
Metricsoutcome-vs-output; baseline + target + timeframe + counter-metric + kill/scale/graduate threshold
Pricingdurable-vs-current (wedge vs moat); willingness-to-pay evidence; (build-vs-buy + opportunity cost for internal tools)

GTMchannel/motion named; first-customer path; not "we'll figure it out"

AIBehavior Contract (GOOD/BAD/REJECT); fallback/kill-switch; eval metrics; red-team list; "a wrong answer is a learning harm, not a UX bug"
Riskspre-mortem present; failure modes + mitigations; no-burial of financial/risk content

4.3 doc-types.md — applicability map (drives ABSENT vs N/A)

Per doc-type, each axis is E (expected — missing ⇒ ABSENT), N/A (not applicable — missing ⇒ stated N/A), or C (conditional — a rule promotes it to E or N/A). Hybrids take the union (any axis E in any detected type is E), so nothing expected gets a free N/A.

Conditional rules: ¹ Pricing → N/A for internal tools / platforms (no external price), reframed as build-vs-buy + opportunity cost under Strategy. ² AI → E iff the doc proposes an AI/LLM feature; else N/A. The map text doubles as the author-facing explanation ("PRDs are expected to address Risks"). The skill auto-detects the type and declares it in the opening line (user-correctable interactively / recorded as an assumption non-interactively); a <Work in Progress>/placeholder section is treated as a decision-blocking gap, not absence.

4.4 Deterministic eval script (hard gate, 100% pass)

A Node script (skill-patterns §H: deterministic = script, never LLM-counted) takes the extracted source text + the parsed findings JSON and asserts:

CheckAssertion
E-schemafindings JSON parses + conforms to pmos-critique-findings/v1
E-axes-completeall 10 axes present, in fixed order, each verdict ∈ the ordinal set
E-applicable-consistencyapplicable=false ⇔ verdict=N/A; the applicable set matches the resolved doc-type map (union for hybrids)
E-quote-lenevery non-null quote ≥40 chars
E-quote-in-sourceevery quote, whitespace-normalized, is a verbatim substring of the whitespace-normalized source text (the §6 verification mechanism)
E-gap-namedevery ABSENT/WEAK axis has a non-empty reason
E-weakest-rankedweakest_claims length 0–3; ranks unique 1..n; each has a source-verified quote ≥40 chars
E-openingopening.pushing_hardest_on has 1–3 entries

Exits 0 (all pass) / 1 (≥1 fail, listing them) / 2 (script error → the skill surfaces, never silently passes). Advisory reviewer (separate agent, per _shared/reviewer-protocol.md, ≤2 loops): judges grounding quality, fairness (credits strengths before attacking), voice adherence, and flags manufactured/nitpick findings — non-blocking, surfaced.

4.5 Corpus, few-shot exemplars & fixtures

Define decision: vendor anonymized samples. The 5-doc Coach corpus (~/Downloads/critique/) is confidential internal product docs — none is committed. Story fbd creates a small set of anonymized/redacted mini-samples (doc-excerpt + paired critique-output JSON) under docs/pmos/features/2026-06-24_artifact-critique/corpus-samples/, covering ≥2 doc-types and an AI-feature case, so the eval-script fixtures and the few-shot voice exemplars are self-contained and reproducible without the author's local machine. Few-shot voice exemplars are short curated critique lines (≤~12), drawn from the anonymized samples — never the "Gokul Rajaram" attribution (D5/§6 of the brief; voice = unnamed seasoned-product-leader persona).

5. Skill shape — phases

Charter: Give a product document the opinionated, axis-by-axis review a seasoned product leader would — a scannable verdict scorecard, quote-grounded per-axis critique, and a forced-ranking of the weakest claims — as a written artifact, then hand off to /artifact to rewrite.

  1. Load context — pipeline-setup, resolve output_format/--format, read learnings.
  2. Ingest — resolve the doc (path/paste/Notion-export traversal), extract text + section structure, read embedded diagrams/images (D6); note unreadable visuals (Inv-5). Full doc in context; map-reduce verbatim-quote gathering only past the real limit; never silently truncate.
  3. Doc-type detection — classify (PRD/strategy/POV/roadmap/hybrid), declare in the opening line (user-correctable / assumption in NI); resolve the applicable-axis set from doc-types.md (union for hybrids).
  4. Axis scoring — single-pass reviewer — one agent reads the whole doc and scores all axes against axes.md checks + heuristics.md spine → ordinal verdict + reason. Per-axis fan-out deferred as a latency optimization only (keeps cross-axis reasoning intact).
  5. Per-axis deep-dive — verbatim quote → interrogation → "What I'd want to see:".
  6. Weakest Claims (0–3) — ranked + follow-ups; may return fewer/none; never padded.
  7. Synthesize — opening framing ("push hardest on …") + bottom-line.
  8. Emit — output_format-respecting artifact (HTML primary via html-authoring; "Copy markdown" affordance) with the embedded structured-findings block (§4.1); advisory hand-off prose. Run the two-tier quality gate (§4.4) before finalizing.

6. Voice (locked)

Replicate the opinionated senior-operator register via an unnamed "seasoned product leader" persona, held by an explicit voice rubric (take a position; no hedging; credit strengths before attacking; ground every critique in a quote; ventriloquize the executive reader — "The CPO should ask…") plus a few curated few-shot exemplar lines. Do not copy the "Gokul Rajaram" attribution; use the pmos wordmark/footer. The advisory reviewer scores voice adherence to resist drift toward generic-helpful/hedged output.

7. Cross-skill invariants (cite by anchor)

  • Inv-1 — one rubric, one home. The 10 axes, the §2.4 spine, the verdict scale, and the applicability map live in _shared/critique-rubric/ and nowhere else; the skill cites, never restates or forks (CLAUDE.md §K).
  • Inv-2 — coverage-of-absence is load-bearing. Every applicable axis gets a verdict each run; missing+expected ⇒ ABSENT, missing+not-applicable ⇒ stated N/A; the applicable set is resolved deterministically from doc-types.md.
  • Inv-3 — quote grounding. Every deep-dive + weakest-claim is grounded in a verbatim ≥40-char quote verified present in the source by the deterministic gate (never an LLM string-match — CLAUDE.md §H).
  • Inv-4 — no padding. Weakest-claims 0–3, never padded; STRONG is freely given.
  • Inv-5 — honest about limits. Inferred-but-unverifiable ⇒ "not visible in this doc"; unreadable visual named, its content not scored ABSENT.
  • Inv-6 — dangling-cite guard. Nothing references _shared/critique-rubric/ except /artifact-critique until the /artifact adoption story lands. (Backstop: /complete-dev --epic route:skill coherence [J] gate surfaces dangling cites at release.)
