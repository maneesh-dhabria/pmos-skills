Requirements — primer-inline-diagrams

Tier: 2 · Mode: skill-feedback · Target: plugins/pmos-learnkit/skills/primer/SKILL.md + reference/rubric.md

Standing acceptance criteria: the revised /primer must conform to reference/skill-patterns.md §A–§F (frontmatter, description triggering, structure & progressive disclosure, body content, scripts/tooling, platform-conditional frontmatter).

Problem

/primer Phase 4 (Draft) explicitly excludes diagrams (“Non-goals: No diagram-pass”). Primer topics frequently have structural shapes — loops, comparisons, hierarchies, sequences — better conveyed visually. Pure prose forces the reader to assemble the mental model from text alone, raising friction for topics where one well-placed diagram saves a paragraph.

Solution direction

Permit (and recommend) inline SVG diagrams during the single-pass Phase 4 draft — not a separate diagram-pass, not mandatory per section. Diagrams ship inline in the HTML artifact (single-file primer contract preserved). Reviewer rubric R10 gains an informational diagrams_per_h2_distribution field, parallel to the existing examples_per_h2_distribution — the write gate surfaces a note when diagram density is unusually low for the topic, but never blocks.

Functional requirements

  • FR-D01 — Phase 4 (Draft) MUST include an instruction directing the drafter to embed inline SVG diagrams within sections where a visual aids comprehension (concept loops, comparisons across camps, hierarchical relationships, sequence/flow). The instruction frames diagrams as a drafting affordance, not a per-section requirement.
  • FR-D02 — Diagrams MUST be inline SVG (no <img> with external src, no <object>, no JS-rendered canvas). The single-file primer contract is preserved.
  • FR-D03 — Diagrams MUST carry an accessible <title> child element (and optional <desc>) so screen readers and the no-CSS render path remain usable.
  • FR-D04 — Phase 4 §Non-goals MUST be updated: “No diagram-pass” → “No separate diagram-pass; diagrams ship inline during the single-pass draft when a visual aids comprehension.”
  • FR-D05 — Anti-patterns MUST gain a new entry: do NOT force a diagram into every section / use diagrams as decoration / use diagrams to displace citation density. The drafter picks diagrams where they earn their place.
  • FR-D06 — reference/rubric.md R10 MUST grow an informational field diagrams_per_h2_distribution: [{h2_id, h2_title, count}] — counts inline <svg> elements per H2 section. Informational only; does NOT affect the R10 verdict.
  • FR-D07 — The Phase 5 write gate’s informational-note logic SHOULD additionally prepend a one-line note when diagrams_per_h2_distribution reports zero diagrams across all H2s AND the topic appears diagram-amenable (heuristic deferred to spec).
  • FR-D08 — Diagram styling MUST inherit the page’s prose CSS (no inline style attributes beyond what the diagram itself requires for layout); diagrams should use the substrate’s neutral palette where practical.
  • FR-D09 — The Worked example MUST be updated to show 1–2 inline diagrams (e.g., a percentage-rollout flow diagram, or a build-vs-buy comparison) so the new behavior is anchored in the example.
  • FR-D10 — Diagram instructions MUST cross-reference the existing /diagram skill’s style conventions where practical, but the drafter does NOT invoke /diagram as a subagent (Phase 4 runs inline per spec D2).

Non-functional requirements

  • NFR-D01 — SKILL.md remains < 800 lines after edits (current ~430 lines; new diagram instruction + worked-example refresh estimated +50 lines).
  • NFR-D02 — Reference file reference/rubric.md remains < 300 lines (current ~108).
  • NFR-D03 — No new dependencies (no new scripts, no new npm packages, no new external services).

Out of scope

  • A separate diagram-pass Phase between Outline and Draft — rejected: increases turn count and conflicts with the single-pass-draft architectural choice.
  • Mandating diagrams per section — rejected: produces forced/decorative diagrams in topics where visuals don’t aid comprehension.
  • Invoking /diagram as a subagent during Phase 4 — rejected per spec D2 (Phase 4 runs inline; no subagent dispatch outside Phase 5 reviewer).
  • Raster diagrams or external SVG files — rejected: violates single-file primer contract.
  • A new reviewer rubric check that hard-fails on missing diagrams — rejected: diagrams are taste-tier; informational-note at write gate is sufficient.

User journey

  1. User runs /primer feature flagging at scale.
  2. Phases 0–3 run unchanged; outline approved.
  3. Phase 4 draft now produces ~4,800 words AND ~2 inline SVG diagrams: a percentage-rollout flow under H2 “How the smart teams are using it” and a build-vs-buy comparison matrix under H2 “Tradeoffs we considered against”.
  4. Phase 5 reviewer scores; R10 informational fields report diagrams_per_h2_distribution = 2 H2s with diagrams / 10 H2s with zero. Write gate AUTO-PICKs Approve (no zero-density note triggered since 2 of 10 is acceptable).
  5. Artifact ships as a single HTML file containing inline SVGs.
