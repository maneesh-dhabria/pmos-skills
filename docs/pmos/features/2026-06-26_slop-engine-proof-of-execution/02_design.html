Design — slop-engine proof-of-execution

Epic 260626-804 · Plugin pmos-toolkit · Route skill · Status defining · Date 2026-06-26

The cross-skill design contract for a two-story epic that closes a skip-by-assertion hole in the deterministic slop-engine consumers. Grounded in a verified read of the engine and all five candidate consumer skills (see §3 scope verification). Source: a /reflect retro on /design-crit (run against a live local game, v2.93.0).

1. Problem

/design-crit Phase 3.5 runs a deterministic slop pre-pass — it injects the vendored engine (_shared/slop-engine/browser.js) via assets/slop-prepass.mjs, scans the live DOM, and emits machine findings that complement the LLM critique. In the retro run the model silently skipped the pre-pass and rationalized the skip with an unverified claim — "the vendored engine bundle is not wired into this standalone run." The engine was in fact present; when the helper was finally run (after the user challenged the skip) it executed cleanly and produced 40 findings, including a ~187-char line-length readability tell the LLM pass had missed.

The root cause is structural, not a one-off lapse: Phase 3.5's graceful-degradation text (Inv-5) is permissive — "the engine lane is purely additive… proceeds exactly as today" — and nowhere requires the model to actually run the helper and surface its output before claiming the lane was unavailable. "I skipped it" can therefore masquerade as "it degraded gracefully."

2. Charter fit

pmos-toolkit — both target skills (/design-crit, /verify) ship a feature's quality gates. This epic does not add capability; it makes an existing deterministic gate honest. No engine code changes — only the consuming skills' instructions for proving the lane ran.

4. Cross-skill invariants

Each story cites these by id; they are the coherence contract across the two skills.

  • I1 — One engine, two lanes (unchanged). _shared/slop-engine/ is the single engine. /design-crit uses the browser bundle; /verify uses the Node detector. Neither story edits engine code — both edit only how the consuming skill proves its lane ran.
  • I2 — Proof-of-execution, not proof-of-findings. The gate requires evidence the helper ran (its literal stdout/stderr line and the JSON artifact it writes), never that it found anything. Zero findings is a valid PASS. The target is the skip claim: "skipped/unavailable" may be recorded only when the helper actually emitted its skip-note (exit 0 with skipped:true) or exited non-zero — never by narrative assertion.
  • I3 — Graceful degradation is preserved, just earned. Inv-5 (the additive lane never rolls back or alters the critique/verify flow) survives in both skills. The only change: degradation is earned by the helper's own output, not claimed by the model.
  • I4 — The three floor-citing skills stay out of scope (verified — §3). Recorded so a future reader does not "harden" a skill that runs no helper.
  • I5 — The NI-block nit is declined. The retro's proposal to factor the inlined non-interactive block into a shared include is declined: it conflicts with the codified CLAUDE.md invariant — the block is deliberately inlined byte-identical in every skill, hand-maintained, and lint-enforced (tools/lint-non-interactive-inline.sh), and a body-rewriter was explicitly rejected as "riskier than the re-paste tax it saves." No story. maintainer-confirmed skip 2026-06-26

5. What the helpers already emit (the gate's raw material)

No helper changes are needed — both already print enough to gate on. The fix is to require the consuming skill to surface it.

assets/slop-prepass.mjs (design-crit)

  • Success (stdout): [slop-prepass] N deterministic finding(s), M overlay node(s) → <outFile>
  • Skip (stderr): [slop-prepass] slop-engine unavailable — skipping deterministic pre-pass: <reason>
  • Artifact: {out_dir}/slop-findings.json — { generated, source, engine: <resolved browser.js path>, overlaysRendered, skipped?, reason?, findings:[…] }. The resolved engine path lives in engine; the count is findings.length.
  • Exit codes: 0 ok (incl. graceful engine-absent skip) · 1 args · 3 Playwright missing (dependency error, distinct from the soft skip).

scripts/slop-gate.mjs (verify)

  • Run (stderr): [slop-gate] … GATE FIRES … / … no quality blocker; slop advisory only
  • Skip (stderr): [slop-gate] slop gate skipped — engine/parser unavailable: <reason>
  • Artifact (stdout JSON): { ran, skipped, source, engine, counts:{quality,slop,blockers}, … }
  • Exit codes: 2 blocker present (gate fires) · 0 no blocker · 64 usage. The exit code already drives the verdict; only the skip branch is narrative.

6. Story A — /design-crit hardening (260626-pgh, route: skill)

One /execute run on one skill. Four edits, all instruction-level (plus a small capture-path addition and report-rendering change); the engine and slop-prepass.mjs are untouched.

A1 — Blocker: Phase 3.5 proof-of-execution hard gate (I2)

  • Phase 3.5 becomes a hard gate: the model MUST run node {skill_dir}/assets/slop-prepass.mjs … once per captured target, and MUST surface the helper's own evidence line verbatim in chat before Phase 4 may proceed — the success line (carrying the finding count + output path) or the stderr skip-note.
  • Phase 4 MUST read {out_dir}/slop-findings.json produced this run; the Phase 6 report's slop section is populated from that JSON (engine path + findings.length, or the JSON's skipped+reason) — never from narrative.
  • Rewrite the Inv-5 paragraph: graceful degradation is claimable only when the helper emitted its skip-note (skipped:true) or exited non-zero. Add an explicit anti-pattern: "Do NOT assert the engine is absent / not-wired-in without running the helper — the helper resolves it at _shared/slop-engine/browser.js and records the resolved path in slop-findings.json :: engine; if you believe it is absent, run it and show the skip-note."

A2 — Friction: stateful-app / SPA capture path (Phase 3)

  • Add a sanctioned "stateful app / SPA / game" capture path alongside the existing files / journey-config / crawl modes. A journey there is a chain of in-canvas interactions (navigate → click → screenshot), driven by an interactive driver (Playwright MCP, or a click-stepped Playwright script) rather than a static file list or crawlable route set.
  • Sanction a storage-reset step for first-run / cold-open capture (e.g. indexedDB.deleteDatabase / localStorage.clear before the cold-open screenshot).
  • Composition rule: the A1 pre-pass still runs in SPA mode — against the live URL after the interactions reach each state — so the proof-of-execution gate is not bypassed by the new capture path.

A3 — Friction: --report-only mode (the "just give me the crit" path)

  • Add --report-only (NL: "just give me the crit" / "go through the flows and share your critique"): skips the per-finding AskUserQuestion disposition loop (Fix/Skip/Defer) but still runs the slop pre-pass (A1), writes the report, and emits the mandated "<N> surfaced, <M> unsurfaced" chat line. The most common ask becomes a sanctioned mode, not an off-spec deviation.
  • Per skill-patterns.md §I: a behavior-mode flag with headless-determinism coupling → it is a contract flag and is listed in argument-hint; the natural-language form is honored too.

A4 — Friction: markdown fallback when the HTML-authoring substrate is unresolvable

  • Output stays HTML-first. But from a standalone plugin-cache invocation the _shared/html-authoring/ substrate may be unreachable. Sanction a markdown fallback keyed strictly off substrate-unreachability: attempt HTML via the substrate; if its path cannot be resolved/read, write a self-contained markdown report and log a single stderr note naming the unresolved substrate path.
  • This is not a return of the retired md/both output preference — it is a degradation guard mirroring the engine's own Inv-5 shape (earned by a real resolution failure, surfaced loudly). Design tension recorded: the cleaner long-term fix is making the substrate reliably resolvable from the cache; the md fallback is the pragmatic v1 guard and should carry a comment pointing at that follow-up.

7. Story B — /verify slop-gate skip-branch proof (260626-4mc, route: skill)

One /execute run on one skill; a small, instruction-only edit mirroring A1 but lighter (the verdict is already exit-code-driven — only the skip branch needs hardening).

  • The Slop-gate phase MUST surface the runner's literal output line before recording the verdict — the run-summary ([slop-gate] … GATE FIRES … / … no quality blocker …) or the skip-note ([slop-gate] slop gate skipped — … : <reason>).
  • The "ran vs skipped" determination is tied to the JSON report's ran/skipped field and the exit code — never to narrative. A skip is recorded only when the runner emitted its skip-note (skipped:true) or exited in the skip path.
  • Rewrite the verify Inv-5 wording the same way as A1: degradation earned, not asserted. The exit-code-driven PASS condition (zero unfixed [Blocker] quality faults whenever the gate ran) is unchanged.

  • Conforms to feature-sdlc/reference/skill-patterns.md §A–§L (§H: the proof-of-execution gate is a hard gate, not advisory; §I: --report-only is a hinted contract flag with an NL form; §J: stable {#slug} phase anchors preserved; §K: the helper-output contract has one home — §5 here / the helper source — cited, not restated, in the SKILL).
  • Honors host CLAUDE.md: canonical skill paths plugins/pmos-toolkit/skills/{design-crit,verify}/SKILL.md; non-interactive inline block byte-identical to skills/_shared/non-interactive.md (I5 — not refactored); every AskUserQuestion keeps a (Recommended) option or a defer-only tag.
  • Engine code (_shared/slop-engine/) and both helper scripts are unchanged — only the consuming SKILL.md instructions (and design-crit's capture-path + report-rendering text) change. Existing helper tests stay green.
  • Release prerequisites (version bump, changelog, README row, manifest sync) are NOT plan tasks — listed under "Release prerequisites" for /complete-dev at epic release.
