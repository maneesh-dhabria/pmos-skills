Approach

  • Foundation (T1): the state-schema.md build phases[] shape — the reconcile entry + per-inner-phase status the Phase 0b cursor reads, plus the new story-state fields. Everything else cites it.

  • Consumer (T4): feature-sdlc/SKILL.md #build-mode step 0 — the reconcile algorithm + driver-identity contract + decision log.
  • Proof (T5, T6): claim-lock + reconcile-selection tests; skill-eval + repo lints; zero-crash pick path unregressed.

Risks & decisions carried from design

Claim-freshness is the wrong liveness signal. A just-crashed story has a fresh claim. T2/T4 must key resume on holder ownership (D2/D3), never on freshness alone — else self-resume regresses to the 4h wait.
Don't corrupt a correct result. T1+T4 must let a post-verify-PASS crash finalize via write-back without re-running /verify (D6). The per-inner-phase status in phases[] is what makes this precise.
Forward-progress reset, not raw attempt count. T4's poison guard resets resume_attempts on new commits (D4) so a flaky-but-progressing story is not falsely abandoned — the exact API-flake trigger this epic exists for.
claim-lock.js header invariant. The file documents "PID-liveness is NOT a reclaim trigger." T2 extends (own-holder identity is a new trigger alongside TTL) — it must not contradict the TTL-for-foreign-holders rule. Keep zero-dep; keep --selftest green.

Final verification checklist

  • claim-lock.js --selftest passes; new own-holder + foreign-holder cases asserted.
  • Reconcile-selection test: own/stale-claim in-progress story chosen; fresh-foreign-claim skipped.
  • /backlog set rejects resume_attempts + last-progress marker.
  • skill-eval-check.sh + lint-phase-refs.sh + lint-flags-vs-hints.sh + non-interactive lint + audit-recommended all green.
  • With zero in-progress stories, build --next picks byte-identically to pre-change behaviour.
