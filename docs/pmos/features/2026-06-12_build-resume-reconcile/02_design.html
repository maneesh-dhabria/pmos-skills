Epic 0612-w4e — Build-loop resume-first reconcile-in-flight

Define-loop design contract · route: skill · pmos-toolkit · 2026-06-12 · self-heal stories that crash mid-build under an unattended /loop

The problem (observed gap)

/feature-sdlc build --next runs one story per iteration: pick → claim → worktree → /execute → /verify → write-back → STOP, driven repeatedly by a single unattended /loop. Two writes land before any cleanup runs:

  • the claim lock (backlog/claims/<id>.lock, O_EXCL) + claimed_by: — removed only by the final write-back's /backlog unclaim;
  • the story status flip planned → in-progress (stamped by /execute via --backlog <id>).

An API error (or any crash) during /execute never reaches write-back, leaving the story at status: in-progress, claim held, partial work in feat/<id>, no write-back — neither done nor blocked.

The rot. build --next calls /backlog next --status planned + unclaimed. The crashed story fails both filters (it is in-progress AND claimed), so the picker skips it forever. Even after the 4h claim TTL frees the lock, the status is still in-progress, so --next never re-picks it. It surfaces in /backlog groom as a stale claim only after 4h, and never auto-recovers. The loop marches on to new work while the crashed story is silently abandoned.

Coherence invariants

  • Zero behavior change when nothing crashed. A clean backlog (no in-progress stories) must pick exactly as today — reconcile is a no-op fall-through.
  • One story per iteration, unchanged. Reconcile resumes one story and STOPs, or falls through to one planned pick. The bounded-iteration contract (#build-mode) is preserved.
  • Never head-of-line-block on a poison story. A story that crashes every time must eventually exit the auto-loop (→ blocked) so other ready work proceeds.
  • Never corrupt a correct result. Resuming a story whose work is already complete-and-verified must not re-run a gate that could flip it to blocked on flake.
  • Single-driver model. One /loop at a time; a concurrent manual build --story <id> is the only other executor we must not stomp. Multi-driver/cron concurrency (heartbeat) is explicitly out of scope — deferred.

Design — the reconcile-in-flight step

Add a step 0 reconcile-in-flight at the top of #build-mode, before pick. It runs every iteration: resume a resumable in-progress story (then STOP), else fall through to today's planned pick. Decisions below; all five were grilled (standard depth, 2026-06-12) and landed on these answers.

D1 — reconcile-first, fall-through-clean. Step 0 queries status: in-progress stories (D22 order). None → fall straight through to the existing --next planned pick (zero-change path). One resumable → resume it and STOP. The bounded one-story-per-iteration contract is untouched.
D2 — claim-guarded resumability (NOT pure "in-progress = crashed"). A story at in-progress is resumable iff its claim lock is ABSENT, STALE (>TTL), or held by MY OWN driver id. A fresh claim held by a different holder means another executor (a concurrent manual build --story <id>) is actively on it → skip, do not resume. This is what keeps single-driver signal A safe against the one realistic concurrent case.

Rationale: claim freshness alone is the wrong liveness signal — a story that crashed 2 min ago has a 2-min-old (fresh) claim, so "skip if fresh" would defeat fast self-resume. Ownership is the discriminator.
D3 — stable per-loop driver identity. The build driver claims with a stable --holder string across loop ticks (e.g. loop:<root-session-id> or a loop-run uuid), so tick N+1 recognizes tick N's abandoned claim as "mine" and resumes immediately — no waiting out the 4h TTL. claim-lock.js gains a holder-ownership comparison (acquire may reclaim a same-holder lock without the staleness wait); backlog claim/next thread the holder through.
D4 — poison guard with forward-progress reset. Track resume_attempts AND a last-progress marker (last completed task index or the commit sha at attempt start). On resume, if new work was committed since the prior attempt, reset resume_attempts to 0. The cap (default 2) fires only on consecutive unproductive resumes (crash with zero new progress) — the true stuck/poison signal. A flaky API that still advances the story each tick is never falsely abandoned. (Mirrors the skill-eval ≤2-iteration cap philosophy.)
D5 — cap → blocked, with a diagnosable note. On hitting the cap: set status: blocked, unclaim, fall through to a fresh planned pick (never head-of-line-block). The crash killed the process before it could record the error, so the blocked note captures the observable facts: resume_attempts, last completed task / commit sha, in-flight phase (from state.yaml current_phase), timestamps — honest that the raw error is unrecoverable post-hoc; points the human at the worktree + git reflog.
D6 — re-entry via the Phase 0b cursor, not a blind restart. Resume reuses the existing Phase 0b resume cursor: jump to the first inner phases[] entry not yet complete. A crash after /verify wrote PASS but before write-back resumes straight to write-back (idempotent finalize) — it must NOT re-run /verify (a flaky re-run, or a moved main, could wrongly flip a correct result to blocked). Requires the build inner pipeline's phases[] to expose per-inner-phase status (execute / skill-eval / verify / write-back) so the cursor re-enters precisely.
D7 — skill-managed fields; reconcile scope. resume_attempts and the last-progress marker are skill-managed backlog fields — rejected by /backlog set as direct edits (same posture as id/created/updated). reconcile-in-flight only ever considers in-progress stories — it never touches blocked/done/released; a resume-capped blocked story correctly exits the auto-loop until human grooming.

Explicitly out of scope

  • Heartbeat / multi-driver liveness. Distinguishing "ongoing" from "stuck" across concurrent drivers needs a positive heartbeat (executor touches heartbeat_at per wave). Deferred — single-driver is the committed model; the claim-ownership guard (D2/D3) covers the one concurrent manual---story case.
  • Recovering the raw crash error. Not possible post-hoc (the crash killed the process); D5 records observable facts instead.
  • Reviving blocked stories automatically. A resume-capped story stays blocked until human grooming (/backlog set status=planned).

Epic acceptance criteria

AC1 — build --next gains a reconcile-in-flight step 0 that resumes a resumable in-progress story before picking new work; a clean backlog picks exactly as today (D1).
AC2 — resumability is claim-guarded: resume iff claim absent/stale/own-holder; a fresh foreign-held claim is skipped (D2).
AC3 — the build driver claims with a stable per-loop holder id; claim-lock.js reclaims a same-holder lock without the TTL wait; foreign-holder TTL semantics preserved (D3).
AC4 — resume_attempts + last-progress marker tracked; counter resets on forward progress; cap (2) fires only on consecutive unproductive resumes (D4).
AC5 — cap → blocked + unclaim + diagnosable note (attempts, last task/sha, phase, timestamps); loop is never head-of-line-blocked (D5).
AC6 — resume re-enters via the Phase 0b cursor; a post-verify-PASS crash finalizes (write-back) without re-running /verify; build phases[] exposes per-inner-phase status (D6).
AC7 — resume_attempts + marker are skill-managed (rejected by /backlog set); reconcile touches only in-progress stories (D7).
AC8 — skill-eval green; backlog + claim-lock tests updated to cover holder-ownership + reconcile selection; no regression to the zero-crash pick path.
