Problem

Every pmos-emitted HTML artifact today loads styling and interactivity from a sibling assets/ directory. When opened outside a local web server — Claude Code desktop preview, Mail.app, Dropbox preview, a copy saved to disk, an attachment received in chat — the sibling fetches fail and the artifact renders unstyled with no comments overlay. The user sees a broken document and has to remember to launch via a helper script just to make the file readable.

Who experiences this?

The plugin author (sole user today), plus any stakeholder who receives a pmos artifact by link, attachment, or shared folder. The friction lands hardest when an artifact has just been emitted and the author opens it via the IDE preview pane to sanity-check the output — the most natural first interaction is the one that breaks.

Why now?

The /comments overlay (v2.56.0, 2026-05-25) added a sidecar persistence model and a launcher trio that depend on local-http hosting. That work made the existing sibling-fetch problem worse: the artifact is no longer just unstyled under file://, it also silently loses every comment thread because the FSA write path is Chrome-only and localStorage fallbacks don't survive across machines. The sidecar+artifact pair also introduced a drift hook (T25) that refuses commits where the two halves are out of sync — a recurring footgun. Now is the right moment to collapse the sidecar into the artifact and let one file carry everything.

Goals & Non-Goals

Goals

  • Open-anywhere fidelity — opening any pmos artifact under file:// (desktop preview, email, Dropbox, archived copy) shows the toolbar, properly-styled prose, and existing comment threads in read-only form — measured by visual parity against http://localhost render across the 14 emit surfaces.
  • Single-file portability — one .html file is the complete artifact; no required siblings — measured by every docs/pmos/ tree containing zero .comments.json files after migration.
  • Lossless write path — launching via comments-open.* still provides full read/write commenting against the same single file — measured by the existing manual smoke matrix (macOS/Linux/Windows × Chrome/Safari/Firefox) passing without the FSA fallback or localStorage buffer.
  • Substrate shrinkage — retire the sidecar format, the drift hook, the FSA write path, the localStorage fallback, and diff-match-patch.js — measured by deletion of those files from authoring source + the comments-bundle-size gate moving downward.
  • One-touchpoint fanout — the 14 emit surfaces inherit the change through a single template-renderer substitution; no per-surface edits to ride this through — measured by the _shared/html-authoring/template.html + asset-substrate being the only authoring-source bytes changed for the cross-cutting behavior.

Non-Goals (explicit scope cuts)

  • NOT shipping a build pipeline / bundler step (variant #10) — because template-substitution is a 10-line renderer change; a build step is needless indirection.
  • NOT dual-emitting the artifact (variant #12) — because doubling disk for parity with the legacy shape has no upside once the standalone shape is the only shape.
  • NOT migrating already-published / already-shared artifacts that live on someone else's disk — because there is one human user today; only this repo's docs/pmos/** is the migration corpus.
  • NOT keeping the FSA write path in any form — because it's Chrome-only, requires a user-gesture file picker per save, and dies in Safari/Firefox; http://localhost via the existing launcher trio is uniform.
  • NOT preserving the Copy-Markdown button after turndown is removed — because there is no in-domain signal that the button is used; users who want Markdown can re-emit with output_format=both.
  • NOT resolving assumption #1 (does Claude desktop preview honor inline <style>/<script>?) as a precondition — because the chosen architecture is invariant under both outcomes (worst case: file:// is unstyled-read-only and http://localhost via launcher is the always-functional path); the question becomes a non-blocking validation item for /verify.

User Experience Analysis

Motivation

  • Job to be done: Open an artifact I just emitted (or one a stakeholder shared) and have it work — read it with styling, see existing comments, optionally add my own — without thinking about whether the assets are next to the file.
  • Importance / urgency: Daily friction for the author every time the IDE desktop preview pane is the first place they look at an emit. Higher latent friction for any future stakeholder receiving an artifact by attachment.
  • Alternatives: (a) remember to comments-open.sh <artifact> every time even just to read; (b) screenshot the styled version and ship that; (c) just live with the unstyled view. None preserve commenting; (a) is a per-open cognitive tax; (b)/(c) lose the artifact's affordances entirely.

Satisfaction Signals

  • Author re-opens a freshly emitted artifact in the IDE preview pane and the styled view is the first thing they see — no "oh right, I have to launch the helper".
  • A shared artifact opens with full styling in the recipient's mail client or download folder. They can read and link to a section. They never see an unstyled wall of HTML.
  • A comment thread from three weeks ago is still visible without depending on a sidecar file that may or may not have made it into the shared bundle.
  • The author commits a comment-bearing artifact without thinking about the drift hook — because there is no drift hook to think about.

Solution Direction

Make every emitted artifact fully self-contained:

  1. Inline at emit time. The substrate's template.html grows three new tokens: {{inline_css}} (concatenated style.css + comments.css in a single <style>), {{inline_js}} (concatenated viewer.js + slimmed comments.js in a single <script>), and {{inline_comments_json}} (seeded {"schema":1,"version":0,"threads":[]}). The external <link> and <script src=> references for those files are removed. The substrate renderer reads each authoring-source file once at emit and pastes its contents into the template.
  2. Delimited comments block. Comments persist as an inline JSON payload inside the artifact, bracketed by explicit start/end markers. The server has an unambiguous slot to rewrite without parsing the surrounding HTML, and a future schema bump can find the block by marker alone.
  3. Read path under file://. The page renders fully styled. The viewer code reads the inline JSON block and surfaces existing threads in read-only mode (compose UI hidden), with the read-only state detected by the runtime so the artifact never offers a compose action it can't actually fulfil.
  4. Write path under http://localhost. The existing comments-open.* trio still launches serve.js in the same one-artifact-per-port shape as today. serve.js gains a comment-save endpoint that crash-safely replaces the inline JSON block inside the artifact. The client posts the full updated JSON plus an expected_version for optimistic concurrency; on a version mismatch the server rejects the save and the client prompts the user to reload.
  5. Re-emit preserves comments. When a pmos skill regenerates an existing artifact (e.g., /spec --update), the renderer first reads the existing file's <script id="pmos-comments"> block and writes it back into the new emission. The existing quote-hash anchor resolver in comments.js ports forward: matched → preserve; near-match → status: needs-review; lost → status: orphaned with a visible bottom-of-page panel.
  6. Retire the sidecar substrate. Delete the .comments.json sidecar format; the pre-commit drift hook in scripts/install-comments-hooks.sh; the FSA write path; the localStorage fallback; diff-match-patch.js from authoring source. Migrate every existing *.comments.json into its sibling *.html's embedded block via an idempotent scripts/migrate-sidecars-to-inline.sh, then delete the sidecars.
  7. Drop turndown + Copy-Markdown. Remove the turndown.umd.js + turndown-plugin-gfm.umd.js + LICENSE.turndown.txt assets and the data-pmos-action="copy-md" button from the template's toolbar. Users who want Markdown emit with --format=both; the existing html-to-md.js handles that path unchanged.
  8. Fan-out via the single template. All 14 emit surfaces consume the same _shared/html-authoring/template.html. The change is one template edit + renderer-side inlining glue; no per-surface code lands.

Per-artifact size budget: the inlined CSS (style ~15KB + comments ~6KB) plus JS (viewer ~13KB + slimmed comments ~15KB) lands ~50KB heavier than today's reference-only HTML. Turndown (~80KB) and DMP (~80KB) are not inlined; both are deleted outright. A typical /spec artifact moves from ~50KB to ~100KB. Roughly 2.5MB across the current ~50 artifacts in this repo — acceptable per the brief.

User Journeys

Primary Journey (Happy Path) — Author opens a freshly emitted artifact

  1. Author runs /spec (or any of the 14 surfaces). The skill writes 02_spec.html via the substrate.
  2. Author opens the file via the IDE's Cmd-click → "Reveal in Preview" pane.
  3. The pane renders the artifact fully styled — toolbar visible, prose laid out, section anchors live. The comments-overlay is dormant (no http://localhost reachable) but the UI affordance hint indicates "Read-only — launch comments-open.sh to comment".
  4. Author scans the doc, decides it's good. Done.

Alternate Journeys

Stakeholder reviews via mail / Dropbox link.

  1. Author shares the 02_spec.html by attaching or linking it.
  2. Recipient opens it from Mail.app / Downloads / a Dropbox preview URL.
  3. Page renders fully styled. Recipient reads it, optionally copies a section link by anchor. They cannot comment from this entry point (no localhost), but the artifact is fully legible — the read-only goal is met.

Author launches the commenter to leave annotations.

  1. Author double-clicks assets/comments-open.command (or runs comments-open.sh <artifact> from a terminal) — the existing launcher trio, no changes to the invocation contract.
  2. serve.js picks a free port, opens the default browser to http://localhost:PORT/<artifact>.
  3. Page loads with the comments overlay active. Existing threads (read from the inline JSON block at page load) are visible; compose UI is enabled.
  4. Author selects text, clicks 💬, writes a thread. On submit, the client POSTs the updated JSON + expected_version to /save; serve.js rewrites the file atomically; the client re-reads the persisted JSON to confirm.
  5. Author closes the browser; serve.js exits when the process is killed (existing lifecycle).

Author re-emits an artifact that has accumulated comments.

  1. Author runs /spec --update on a folder where 02_spec.html already has 4 comment threads.
  2. The substrate renderer reads the existing file's <script id="pmos-comments"> block before overwriting.
  3. The renderer emits the new HTML, splicing the preserved JSON block back into the new file between the sentinels.
  4. The quote-hash anchor resolver runs at next page-load: matched threads remain; near-match threads display a status: needs-review badge; orphaned threads (the underlying quote text was removed in the rewrite) surface in a bottom-of-page "Orphaned comments" panel.

Error Journeys

Two tabs editing the same artifact.

  1. Tab A and Tab B both load artifact.html at version v=5.
  2. Tab A submits a thread; serve.js writes v=6.
  3. Tab B submits a thread with expected_version=5.
  4. Server returns 409 Conflict with the current version + JSON. Tab B's client surfaces a "Document changed since you opened it — reload to merge" banner and disables further submits until reload.

Server crashes mid-write.

  1. serve.js writes artifact.html.tmp with the updated JSON block.
  2. Process dies before rename(2).
  3. On next launch, startup scans for orphaned .tmp siblings in the served directory. If found: surface in stderr ("Orphan tmp file: artifact.html.tmp from previous crash; inspect and delete manually"). No automatic recovery — the user owns the decision.

Embedded </script> in a comment body.

  1. User pastes "</script> HA HA" into a comment body.
  2. The client-side serializer encodes < inside JSON string values as < before embedding the JSON into the <script> block — so the literal token </script> never appears in the rendered HTML.
  3. On read, the standard JSON.parse on the <script> content restores the original text. The user's comment displays correctly.

Inline-script-stripping email client.

  1. Recipient's mail client (or sanitizer) strips all <script> tags.
  2. Page loses both the viewer JS and the comments JSON.
  3. The page still renders styled (CSS is in <style>, untouched), so the document is legible.
  4. No comments visible. Acceptable: comments-in-email was never on the table.

Design Decisions

#DecisionOptions ConsideredRationale
D1 Inline CSS + JS via template-substitution at emit time (a) inline via template substitution; (b) post-emit bundler step (variant #10); (c) data-URI assets; (d) dual-emit (variant #12); (e) inline-CSS-only fallback (variants #4/#5) (a) is a 10-line glue change against the single shared template. (b) adds build-pipeline indirection for no benefit. (c) is functionally equivalent to (a) but harder to debug. (d) doubles disk and keeps the legacy shape supported indefinitely — no upside once standalone is the only shape. (e) remains as a planned fallback only if assumption #1 fails for the JS path during /verify.
D2 Comments persist as a delimited inline JSON block inside the artifact (a) inline JSON block with explicit start/end delimiters; (b) keep sidecar; (c) inline JSON block without explicit delimiters; (d) attributes on body element (a) is the only option that satisfies single-file-portability. Explicit delimiters give the server an unambiguous slot to rewrite, independent of how the rest of the HTML changes. (b) is what we're trying to leave. (c) makes server-side rewriting brittle once other inline scripts join the page. (d) caps comment payload at attribute-length limits in practice.
D3 serve.js stays one-artifact-per-port; gains a comment-save endpoint (a) one-artifact-per-port (today's shape); (b) persistent daemon serving docs/pmos/ root (a) is the lower-risk path. Matches existing launcher contract; the save endpoint is a minimal incremental addition to a static-file server we already trust. (b) is friendlier UX but adds daemon lifecycle (start/stop/PID file) and cross-artifact routing — a separate, larger feature; not now.
D4 Saves are crash-safe: never leave the artifact in a half-written state (a) crash-safe atomic replace; (b) direct in-place write; (c) write a sidecar copy then swap it over the artifact (a) means a crashed save either leaves the previous version intact OR leaves a clearly-named orphan file the user can inspect — never a corrupted artifact. (b) risks corruption with no recovery path now that the sidecar is gone. (c) is what we're leaving — and reintroduces the sidecar surface we just retired.
D5 Optimistic concurrency via version field; 409 on mismatch; user manually reloads to merge (a) version + 409; (b) BroadcastChannel single-writer-tab lock; (c) automatic merge (a) is the simplest correct option for a single-user tool. (b) is a UX upgrade we can take later if multi-tab editing becomes routine. (c) is incorrect — comment ordering is user-meaningful, not auto-mergeable without conflict.
D6 Drop diff-match-patch.js from authoring source (a) delete outright; (b) keep for advanced near-match scoring in the resolver; (c) move DMP logic server-side into serve.js (a) is correct because the only reason DMP was vendored was to reconcile sidecar drift against artifact text — with no sidecar, there is no drift surface to score. The existing quote-hash anchor resolver in comments.js is exact-match by hash; near-match needs only substring containment + character-window scoring, which doesn't need DMP. (c) is contingency: if the inline-resolver miss rate measurably degrades on the calibration corpus, port a minimal substring scorer into serve.js rather than re-importing DMP.
D7 Drop turndown + Copy-Markdown button outright (a) remove button + asset; (b) flip output_format default to both so MD sidecar is always present; (c) keep turndown client-side (a) wins on minimalism — no user signal that the button is used, and re-emit with --format=both covers the rare case. (b) adds ~30KB/artifact of MD that mostly nobody reads. (c) keeps ~80KB/artifact of dead JS.
D8 Migration is automated and idempotent; no per-artifact human review (a) automated idempotent script; (b) per-artifact interactive confirm; (c) leave existing sidecars; new artifacts only inline (a) — single user, low artifact count (~50), trivial mechanical transform (read sidecar JSON → inject between sentinels → delete sidecar). Spot-check 3–5 in dry-run; commit the whole transform in one go. (b) is overkill. (c) leaves a permanent split-brain in the codebase.
D9 Assumption #1 (Claude desktop preview inline-style/script support) deferred to /verify (a) treat as precondition spike; (b) defer to /verify as non-blocking validation (b) — the chosen architecture is invariant under both outcomes. If inline style works but inline script doesn't (the "Inline-CSS-only" fallback in variants #4/#5), comments overlay silently no-ops under file:// while styling still works; that's a graceful degradation, not a re-architecture. The full architecture spec lands now; /verify validates the bonus.
D10 Per-artifact size bloat (~50KB) accepted; .gitattributes marks docs/pmos/**/*.html as linguist-generated + -diff (a) accept bytes + tame diff noise; (b) reject the inlining premise on cost grounds; (c) accept bytes but keep diffs full (a) — 50 artifacts × ~50KB ≈ 2.5MB across the repo, one-time cost the user has signed off on. Tagging as linguist-generated hides them from GitHub language stats and -diff collapses them in PR views so the asset noise doesn't drown content edits.
