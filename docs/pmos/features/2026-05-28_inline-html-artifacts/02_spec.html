1. Problem Statement

Pmos-emitted HTML artifacts depend on a sibling assets/ directory for styling and interactivity. Under file:// contexts (Claude Code desktop preview, Mail.app, Dropbox preview, archived copies), the sibling fetches fail and artifacts render unstyled with no comments overlay. The /comments overlay (v2.56.0) compounded the problem with sidecar persistence that's Chrome-only (FSA) or localStorage-bound. Primary success metric: all 14 pmos emit-surfaces render fully-styled under file:// and remain fully-interactive under http://localhost via the existing comments-open.* launcher trio, with zero .comments.json sidecar files anywhere in docs/pmos/ after migration.

3. Non-Goals

  • NOT shipping a build pipeline / post-emit bundler — template substitution suffices.
  • NOT dual-emitting artifacts in legacy + standalone shapes — standalone is the only shape going forward.
  • NOT migrating off-repo / already-shared artifacts that live on other users' disks.
  • NOT keeping the FSA write path; http://localhost is the only write path.
  • NOT preserving the Copy-Markdown button after turndown is removed; users wanting Markdown re-emit with --format=both.
  • NOT specifically adapting inlining behavior to /diagram outputs — HTML wrapper inherits the template change incidentally; standalone .svg is out of scope.
  • NOT instrumenting an automated anchor-resolver regression gate after DMP removal — manual /verify review only (risk accepted).
  • NOT blocking on assumption #1 (Claude desktop preview inline-style/script support) — non-blocking validation at /verify.

5. User Personas & Journeys

5.1 Primary persona — plugin author

Daily IDE-preview reader; opens artifacts via desktop preview pane after every emit. Wants the preview to "just work". Launches comments-open.* only when intentionally annotating. Sometimes shares artifacts by file attachment or Dropbox link with non-technical stakeholders.

5.2 Primary journey — author emits and previews

  1. Author runs /spec (or any of the 14 surfaces).
  2. Skill writes 02_spec.html via the substrate renderer; template tokens {{inline_css}} / {{inline_js}} / {{inline_comments_json}} are substituted; output is a single self-contained file plus an assets/ sibling (kept for the comments-open launcher trio only — not loaded by the artifact).
  3. Author opens artifact in IDE desktop preview.
  4. Page loads under file://; viewer.js (inlined) reads the inline JSON; protocol short-circuits to read-only mode; threads render (if any); compose UI hidden with hint "Launch comments-open to comment".
  5. Author scans, closes. Done.

5.3 Secondary journey — stakeholder receives by attachment

  1. Author emails the .html attachment.
  2. Recipient opens from mail / downloads / Dropbox preview.
  3. Page renders styled. Threads visible (script-permitting host); section-link buttons functional.
  4. Recipient cannot compose (no localhost); read-only is honest.

5.4 Comment write journey

  1. Author runs comments-open.command (or .sh / .bat).
  2. Launcher spawns serve.js on a free port; opens browser to http://localhost:PORT/<artifact>.
  3. Page loads; protocol is http → HEAD probe to /save → 204 (or 200) → read-write mode.
  4. Author selects text, clicks 💬, writes a thread.
  5. Client POSTs {schema:1, version:N, generated_at:'ISO', threads:[...]} + expected_version: N to /save.
  6. Server reads artifact, validates expected_version against current version in the file, atomically writes new file with version: N+1 and fresh generated_at. Returns 200 { version: N+1, generated_at: 'ISO' }.
  7. Client updates in-memory state.

5.5 Re-emit with pre-existing comments

  1. Author runs /spec --update on a folder where 02_spec.html has 4 threads at version=12.
  2. Substrate renderer reads existing file, extracts the inline JSON block between sentinels, parses it.
  3. Renderer emits new HTML with the preserved threads, increments version to 13, updates generated_at.
  4. Atomic write replaces the old file.
  5. If a browser tab is open at version 12 and tries to save → 409 with current version 13 → client banner prompts reload → user reloads, sees the new artifact content with their threads intact.

7. Functional Requirements

7.1 Substrate (template + renderer)

IDRequirement
FR-01_shared/html-authoring/template.html gains three new token slots: {{inline_css}} in <head> (single <style> block), {{inline_js}} in <body> end (single <script defer> block), {{inline_comments_json}} in <body> end (bracketed by sentinel comments per FR-04).
FR-02Existing external <link rel="stylesheet" href="...style.css"> + <link ...comments.css> + <script defer src="...viewer.js"> + <script defer src="...comments.js"> references in template.html are REMOVED.
FR-03The substrate renderer (the code path each emit-skill calls to instantiate template.html) reads style.css + comments.css from the authoring source, concatenates them, substitutes into {{inline_css}}. Same for viewer.js + slimmed comments.js → {{inline_js}}. Files are read once per emit; no build-time bundling step.
FR-04The {{inline_comments_json}} token expands to exactly: <!-- pmos-comments:start -->\n<script id="pmos-comments" type="application/json">\n{JSON-PAYLOAD}\n</script>\n<!-- pmos-comments:end --> where JSON-PAYLOAD is a UTF-8 JSON string with < escaped as < inside string values (defeats embedded </script> sequences).
FR-05Comments JSON schema (v1): { "schema": 1, "version": <integer ≥ 0>, "generated_at": "<ISO-8601>", "threads": [<thread>...] }. <thread> structure ports forward from the current sidecar schema unchanged. Seeded value on fresh emit: {schema:1, version:0, generated_at:"<emit-time-ISO>", threads:[]}.
FR-06Re-emit MUST preserve the threads array from the existing artifact's inline block (read between sentinels, JSON.parse), MUST increment version by 1, MUST stamp a fresh generated_at, MUST write back into the new emission's {{inline_comments_json}} slot.
FR-07When no existing artifact is present (fresh emit), the renderer seeds version: 0.

7.2 serve.js extensions

IDRequirement
FR-08serve.js gains a HEAD /save handler that returns 204 No Content (no body). Used by the client to detect read-write mode under non-file:// protocols.
FR-09serve.js gains a POST /save handler. Request body: JSON { expected_version: N, payload: { schema, version, generated_at, threads } } (Content-Type: application/json). Response on success: 200 { version: N+1, generated_at: "ISO" }. Response on mismatch: 409 { current_version: M, current_generated_at: "ISO" }. Response on payload validation failure: 400 { error: "schema-validation-failed", details: "..." }.
FR-10POST /save algorithm: (1) read current artifact file; (2) extract inline JSON block between sentinels; (3) parse, compare version against request's expected_version; (4a) on mismatch return 409 with current version; (4b) on match: build new JSON payload with version+1 + fresh generated_at + request's threads; (5) write replacement block content (regex-replace between sentinels) to <artifact>.html.tmp; (6) fs.rename(tmpPath, artifactPath); (7) return 200 with new version.
FR-11serve.js startup: scan the serving directory for any *.html.tmp sibling files. For each, log to stderr: orphan .tmp from previous run: <path> — inspect and delete manually. No auto-delete, no auto-recovery, no in-page UI surface (per D14).
FR-12serve.js keeps its existing one-artifact-per-port shape and lifecycle. No persistent daemon mode in this feature.

7.3 Client-side viewer.js + comments.js

IDRequirement
FR-13On page load, comments.js MUST read the inline JSON block via JSON.parse(document.getElementById('pmos-comments').textContent) and hydrate the in-memory thread store. Does NOT fetch a sidecar.
FR-14Read-only detection algorithm: (a) if window.location.protocol === 'file:' → read-only, no probe issued; (b) else issue HEAD /save with a 500ms timeout (AbortController after 500ms); (c) on 2xx response → read-write; (d) on 4xx/5xx/network-error/timeout → read-only.
FR-15In read-only mode: compose UI (selection trigger, thread-create dialog) MUST be hidden; existing threads still render with their badges; a passive footer hint reads "Read-only — launch comments-open to add comments".
FR-16In read-write mode: full compose UI active. On thread submit, client POSTs to /save with expected_version set to the version it loaded. On 200, client updates in-memory version + generated_at. On 409, client surfaces a dismissible banner at the top of the page: This document was updated since you opened it (current version: M). Reload to merge.. Compose remains disabled until reload.
FR-17JSON-string escape: any user-entered text destined for the inline JSON MUST encode < as < in JSON.stringify wrappers before being written to the script tag (handled by FR-04's server-side and FR-16's client-side serializers symmetrically).
FR-18The FSA write path in current comments.js (showSaveFilePicker branches) is DELETED. The localStorage fallback is DELETED. Comments.js is shrunk accordingly.

7.4 Migration

IDRequirement

9. API Contracts

9.1 HEAD /save (read-write probe)

HEAD /save

Request body: none.

Response: 204 No Content. No body.

Errors: none (any error response means "not read-write"). Client interprets any non-2xx as read-only.

9.2 POST /save (write thread payload)

POST /save
Content-Type: application/json

Request body:

{
  "expected_version": 5,
  "payload": {
    "schema": 1,
    "version": 5,
    "generated_at": "2026-05-28T12:34:56.789Z",
    "threads": [ /* full thread list, not a diff */ ]
  }
}

Response (200) — success:

{
  "version": 6,
  "generated_at": "2026-05-28T12:35:00.123Z"
}

Response (409) — version mismatch:

{
  "error": "version-conflict",
  "current_version": 7,
  "current_generated_at": "2026-05-28T12:34:50.000Z"
}

Response (400) — payload validation:

{
  "error": "schema-validation-failed",
  "details": "<reason>"
}

Response (500) — server error (e.g., disk full, rename failed):

{
  "error": "internal",
  "details": "<sanitized message>"
}

9.3 Closed error set

POST /save error responses use exactly these error values: version-conflict, schema-validation-failed, internal. Any other server error → 500 with error: "internal".

10. Database Design

N/A. This feature has no database. The artifact file IS the persistence layer. The "schema" lives in JSON inside each artifact's inline block (FR-05).

15. Rollout Strategy

15.1 Cutover order (single PR, ordered commits)

  1. Template + renderer change — modify template.html with the 3 new tokens; update the substrate renderer to inline CSS/JS/seed-JSON. (No emit-skill code changes; they all consume the template by reference.)
  2. serve.js extension — add HEAD /save + POST /save handlers + atomic-rename helper + startup orphan-.tmp scan.
  3. comments.js slim — delete FSA + localStorage branches; switch read path to inline JSON via document.getElementById('pmos-comments'); implement the FR-14 detection algorithm + the FR-16 banner.
  4. Asset deletions — remove DMP + turndown + turndown-plugin-gfm + LICENSE.turndown.txt; replace DMP call sites in 4 skills + comments resolver with exact-id + substring-contains.
  5. Template toolbar change — remove Copy-MD button; remove handler.
  6. Coverage gate update — modify scripts/check-comments-coverage.sh per FR-28.
  7. Drift hook uninstall + migration — at /complete-dev Phase 8: (a) remove scripts/install-comments-hooks.sh + any existing local .git/hooks/pre-commit drift hook; (b) dry-run migration on docs/pmos/; (c) spot-check 3-5 outputs; (d) run real migration; (e) commit the migrated artifacts.
  8. .gitattributes — add docs/pmos/**/*.html linguist-generated=true -diff.
  9. CLAUDE.md "Inline doc comments" section — update to reflect the new persistence model (single-file artifact; serve.js write path; FSA/sidecar/drift-hook retired).

15.2 Rollback

Single PR; rollback = git revert. The migration's commit (step 7e) can be reverted independently to restore sidecars from history. Pre-feature artifacts in docs/pmos/ are restored from git show <pre-cutover-sha>:<path> if needed. No data loss is possible because every state lives in git.
