1. Problem Statement

The agent-skills repo hosts a single plugin (pmos-toolkit) and its marketplace manifest. A second plugin (pmos-learnkit) is pre-scaffold; more pmos-* plugins are anticipated. The current topology gives no place for a second plugin without coupling it to the toolkit's identity or building parallel publication mechanisms. After two /grill passes, the chosen direction is a single repo named pmos-skills with namespaced tags, peer-copy _shared/, and a generalized /complete-dev. This spec defines the exact files, code paths, hook implementations, and cutover sequence.

Primary success metric: /plugin marketplace add maneesh-dhabria/pmos-skills in a fresh Claude Code instance exposes both pmos-toolkit and pmos-learnkit in the install picker, within one release of pmos-learnkit/v0.1.0.

3. Non-Goals

  • NOT giving each plugin its own GitHub repo, stars, issues, or SEO — explicitly traded away on 2026-05-22 for tooling simplicity (D1 rationale).
  • NOT building scripts/release-plugin.sh — generalized /complete-dev + drift hook + sync-shared.sh + pre-push hook do the entire job (D6).
  • NOT using git subtree split / git filter-repo / git filter-branch — single-repo trivializes cross-history concerns.
  • NOT extracting _shared/ into its own pmos-shared plugin — peer copies + drift hook is the chosen mechanism (D9). Revisit when N ≥ 4 plugins OR _shared/ surface > 30% of any one plugin (OQ3).
  • NOT supporting cross-plugin _shared/ divergence at HEAD — drift hook hard-fails on commit.
  • NOT publishing a public migration banner on the archived old repo — archive + private makes it impossible (Q1/D13).
  • NOT supporting hotfix-on-prior-line as a delivered-to-users mechanism (D10). Forward-only.
  • NOT enforcing the drift check at pre-push or in CI — pre-commit only (D16/Q4).
  • NOT building /complete-dev with explicit knowledge of repo-specific remotes — second/third remotes are configured at the git layer (D19).
  • NOT deferring Codex marketplace — same-repo dual-manifest ships day one (D4).
  • NOT publishing per-plugin GitLab mirrors (D5).
  • NOT building a sandbox throwaway plugin or unit-test suite for the sync/drift tooling — solo-maintainer pragmatism.
  • NOT preserving the pre-cutover monorepo tag history under the new naming convention — existing tags kept as-is (D7/Q5).
  • NOT documenting an explicit rollback procedure for a failed pmos-toolkit/v2.50.0 cutover release — fix forward; accept any window of brokenness (DevOps Q2; per solo-maintainer pragmatism).

7.3 scripts/sync-shared.sh

IDRequirement
FR-20Script is bash, target ≤ 30 LOC, idempotent. Path: scripts/sync-shared.sh.
FR-21Usage: scripts/sync-shared.sh --from=<plugin>. <plugin> is a directory name under plugins/ with a skills/_shared/ subtree.
FR-22Behavior: for every plugins/*/ other than --from, run mkdir -p plugins/<other>/skills/_shared/ (idempotent; ensures destination exists for first-run bootstrap per J2), then rsync -a --delete plugins/<from>/skills/_shared/ plugins/<other>/skills/_shared/. Source-of-truth direction is one-way: the --from plugin's _shared/ overwrites every peer. The --delete flag is destructive on the destination — files present in the destination but absent from the source are removed. This is the documented invariant; commit work-in-progress before running sync.
FR-23--dry-run flag prints proposed rsync commands to stdout without mutating filesystem. --help prints usage and exits 0.
FR-24Exit codes: 0 = success (or dry-run); 64 = bad usage (unknown plugin, missing --from); 1 = rsync failure.
FR-25Single-plugin case (only one entry under plugins/): script is a no-op, exits 0. Prints sync-shared.sh: only one plugin, nothing to sync. to stderr.

7.4 Pre-commit Drift Hook

IDRequirement
FR-30Hook is bash, target ≤ 30 LOC, lives at .githooks/pre-commit. Note: core.hooksPath is already configured (per Subagent A §5) so no extra git config setup is needed by Maneesh; external contributors get the instruction in CONTRIBUTING.md (NEW).
FR-31On every git commit: iterate plugins/*/skills/_shared/ directories. If exactly one exists, exit 0 (no peers to compare). If two or more, pick the alphabetically-first as baseline and run diff -rq baseline other for each other peer.
FR-32On any difference: exit non-zero with stderr output: 
_shared/ drift detected:
  plugins/<baseline>/skills/_shared/  →  plugins/<other>/skills/_shared/
    <file> differs
    ...
Resolve with: scripts/sync-shared.sh --from=<source>
The <source> suggestion is the plugin whose _shared/ contains files the others don't (heuristic: most additions vs deletions); if heuristic is ambiguous, suggest --from=<baseline>.
FR-33The hook MUST NOT run on commits that don't touch _shared/ paths — short-circuit on git diff --cached --name-only if no _shared/ entries appear. (Performance: solo-maintainer commits trivially fast.)
FR-34Hook is bypassable via git commit --no-verify; this is an accepted risk per D16. No additional bypass guard.

FR-40For every directory under plugins/*/, validate the 3-way version match: marketplace.json[<plugin>].version == plugins/<plugin>/.claude-plugin/plugin.json version == plugins/<plugin>/.codex-plugin/plugin.json version. Where marketplace.json is BOTH .claude-plugin/marketplace.json AND .codex-plugin/marketplace.json. On any mismatch: reject the push with the named plugin + the three observed versions.

(Solo-maintainer + secondary external-contributor profile both run macOS or Linux.)
NFR-05IdempotenceBoth sync-shared.sh AND both hooks are idempotent. Running any of them twice consecutively with no other changes is a no-op.
NFR-06Backwards compatibilityExisting skill substrate references via ${CLAUDE_PLUGIN_ROOT}/skills/_shared/ (32 references per Subagent A §7) MUST continue working unchanged. Zero skill code edits required for the peer-copy model.
NFR-07ObservabilityEvery hook + script emits a single-line summary to stderr on completion (success or failure) per repo convention. /complete-dev retains its existing NFR-07 chat log line.
NFR-08Atomic writesThe cutover release commit (FR-75) MUST include all CLAUDE.md edits, the new ## Release policy section, the marketplace manifest changes, AND the generalized /complete-dev SKILL.md changes — as ONE logical change. Not split across multiple commits on main.

  4. Merge feature branch to main on origin (GitHub pmos-skills).
  5. Cut pmos-toolkit/v2.50.0 (FR-83): /complete-dev --plugin pmos-toolkit from main. The new code path runs for the first time.

  8. Archive + privatize old repo (FR-86): gh repo archive + gh repo edit --visibility private.
  9. Done. End-users on cached v2.49.0 from the old marketplace continue read-only. New installs flow through the new marketplace.
