1. Overview

Migrate the single-plugin pmos-toolkit repo to a multi-plugin pmos-skills monorepo so a second plugin (pmos-learnkit) and future pmos-* plugins can be installed via one marketplace URL. The work spans repo-level tooling (≤100 LOC of new bash: scripts/sync-shared.sh, .githooks/pre-commit, .githooks/pre-push extension), marketplace manifests for both Claude Code and Codex, generalization of /complete-dev SKILL.md (14 templated path refs + per-plugin routing), CLAUDE.md policy edits, and the one-shot cutover release (pmos-toolkit/v2.50.0) that dogfoods the new code path. The cutover is monolithic by design: all code changes ship in one commit (NFR-08), the new /complete-dev code runs at-release for the first time (FR-61), and the old GitHub repo is archived+private after verification (FR-86).

Done when: /plugin marketplace add maneesh-dhabria/pmos-skills in a fresh Claude Code instance shows pmos-toolkit v2.50.0 in the install picker; old repo returns archived=true, private=true from gh api; all 18 shell-assertion tests under tests/scripts/assert_*.sh (the new ones from this plan) exit 0; git ls-remote origin pmos-toolkit/v2.50.0 returns the same SHA on all three configured remotes; the pre-push hook rejects a deliberate 3-way-version mismatch and accepts a clean one.

Done-when walkthrough: from a freshly-cloned pmos-skills repo on macOS, running bash tests/scripts/assert_marketplace_json_schema.sh through bash tests/scripts/assert_no_hardcoded_pmos_toolkit_path_in_complete_dev.sh prints PASS: on each of the 18 named scripts. Opening a fresh Claude Code session and running /plugin marketplace add maneesh-dhabria/pmos-skills causes the marketplace picker to list pmos-toolkit with version 2.50.0; selecting Install completes and the plugin's commands (e.g., /pmos-toolkit:plan) appear in the prompt completer. From the maintainer's terminal, gh api repos/maneesh-dhabria/pmos-toolkit -q '.archived, .private' returns two lines: true and true. Running git ls-remote origin pmos-toolkit/v2.50.0 | awk '{print $1}' returns a 40-char SHA equal to the output of the same command against work-mirror and gitlab-mirror. Finally, staging echo drift >> plugins/pmos-toolkit/skills/_shared/test-drift.txt + git commit -m test fails with stderr containing _shared/ drift detected and a suggested sync-shared.sh --from=... command.

  • The 4 currently-configured remotes (verified via git remote -v): origin = GitLab, github = GitHub personal (maneesh-dhabria/pmos-toolkit), github-work = GitHub work, all = bulk-fanout (GitLab push + 2 GitHub pushes).   • Existing .githooks/ contains pre-push + README.md. The repo-local core.hooksPath is already configured for Maneesh's clone (per Subagent A §5). T3 adds pre-commit alongside.
  • Per FR-43, pre-existing tags (v2.x, pmos-toolkit-v2.x, pmos-toolkit-2.x) are NOT retroactively validated — T4's tag-format check applies only to tags being newly pushed (i.e., not already present on the remote). 3.4 Stack signals

  • No package.json, Gemfile, go.mod, pyproject.toml, Cargo.toml, docker-compose.yml, Makefile, or Dockerfile at the repo root. This is a markdown + bash + JSON skill-loader repository — no application stack to detect.
