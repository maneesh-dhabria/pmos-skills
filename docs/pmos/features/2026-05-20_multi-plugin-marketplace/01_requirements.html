Problem

The agent-skills repo currently hosts a single plugin (pmos-toolkit) and its marketplace manifest in one GitHub repo (maneesh-dhabria/pmos-toolkit). A second plugin (pmos-learnkit) is about to be scaffolded as an independent product, and more pmos-* plugins are anticipated. The current topology gives Maneesh no place to add a second plugin without either coupling it to the toolkit's identity or building parallel publication mechanisms. Two prior approaches were considered and discarded:

  1. Full per-plugin standalone repos (docs/migration-separate-repos.md) — forfeits the cross-plugin _shared/ atomic-refactor ergonomics the toolkit depends on.
  2. Umbrella + per-plugin standalone repos + monorepo-local + subtree-split-on-release (this doc's earlier revision) — preserves substrate ergonomics and gives each plugin its own GitHub presence, but requires ~300 LOC of release tooling. On 2026-05-22 the user concluded the per-plugin GitHub identity was not worth the tooling weight; marketplace-level branding under a single repo is enough discovery surface for solo-maintained pmos-* plugins.

The second /grill pass (2026-05-22 PM) on the single-repo design then surfaced six additional load-bearing decisions, principally: (a) marketplace.json[plugin].version is the canonical update signal CC reads (not git tags), so hotfix-on-prior-line doesn't reach end-users without a maintenance-branch detour we won't pay for; (b) /complete-dev is hardcoded to plugins/pmos-toolkit/ in 10+ places and must be generalized for multi-plugin awareness; (c) CLAUDE.md cutover touches five sections, not one.

Who experiences this?

  • Primary: Maneesh (sole maintainer) — running releases, doing cross-plugin substrate refactors, deciding repo topology.
  • Secondary: end-user plugin consumers — discovering, installing, and updating plugins via Claude Code's marketplace add mechanism.
  • Tertiary: future plugin authors (initially Maneesh in a different hat; potentially external contributors later) — adding new pmos-* plugins under plugins/.

Why now?

pmos-learnkit is at the pre-scaffold moment. Setting topology before its first commit is far cheaper than reorganising after the second plugin already has a release history. The 2026-05-22 simplification additionally collapses an estimated multi-day release-tooling build into ~60 LOC of helper scripts + a generalized /complete-dev.

Goals & Non-Goals

Goals

  • End-users install all plugins via one marketplace add — measured by: /plugin marketplace add maneesh-dhabria/pmos-skills exposes both pmos-toolkit and pmos-learnkit in the install picker.
  • Per-plugin release cadence is fully independent within the single repo — measured by: tagging and releasing pmos-learnkit/v0.1.0 does not modify any file under plugins/pmos-toolkit/ or any pmos-toolkit/v* tag.
  • marketplace.json[plugin].version is the canonical update signal CC consumes — measured by: this field is bumped on every plugin release and the pre-push hook enforces 3-way version match (marketplace.json + .claude-plugin/plugin.json + .codex-plugin/plugin.json) per plugin.
  • Cross-plugin substrate refactors remain atomic on monorepo main — measured by: a substrate edit lands as a single feature branch containing the canonical edit + a sync delta; never an N-way manual coordination.
  • _shared/ peer copies are byte-identical at HEAD — measured by: pre-commit drift-check hook hard-fails on any difference between the N peer copies; main branch is provably drift-free.
  • Skill references to _shared/ are strictly scoped to plugin-local copies, never to a repo-level shared directory — measured by: no plugins/_shared/ directory exists in the repo at any commit. Verified in /grill L1: skills already reference _shared/ via the plugin-relative ${CLAUDE_PLUGIN_ROOT}/skills/_shared/... token, so peer copies work transparently at install time with zero skill code changes.
  • /complete-dev is generalized to read a plugins: list from host CLAUDE.md and route version-bumps + tags + remote-pushes per plugin via a --plugin <name> flag (auto-detect from diff path-prefix when unambiguous; refuse-on-ambiguity otherwise; smart-detect for substrate-only commits prompts "ride which plugin?") — measured by: /complete-dev --plugin pmos-toolkit and /complete-dev --plugin pmos-learnkit work end-to-end on the same monorepo without code-path branching at the skill level beyond reading CLAUDE.md.
  • Cutover from the existing maneesh-dhabria/pmos-toolkit standalone repo is forward-only and the old repo is non-discoverable — measured by: old repo archived AND made private (no public banner); existing cached installs at v2.49.0 continue functioning read-only with no update path until users manually re-add the new marketplace.

Non-Goals (explicit scope cuts)

  • NOT giving each plugin its own GitHub repo / own stars / own issues / own SEO — explicitly traded away on 2026-05-22 for tooling simplicity. Marketplace-level branding under pmos-skills is sufficient discovery for solo-maintained plugins. Revisit only if a plugin's audience grows to a scale where dedicated-repo discovery becomes load-bearing.
  • NOT building scripts/release-plugin.sh (subtree-split / cross-repo push tooling) — no second repo to push to. The generalized /complete-dev + the pre-commit drift hook + sync-shared.sh + the pre-push hook do the entire job.
  • NOT building an umbrella marketplace pointing at separate plugin repos — the marketplace IS the repo.
  • NOT using git subtree split / git filter-repo / git filter-branch — single-repo trivializes cross-history concerns.
  • NOT extracting _shared/ into its own standalone plugin (pmos-shared) — peer copies + drift hook is the chosen mechanism. Revisit when N ≥ 4 plugins OR _shared/ surface exceeds ~30% of any one plugin's skill content (OQ3).
  • NOT supporting cross-plugin _shared/ divergence at HEAD — drift hook hard-fails on commit; main is always drift-free.
  • NOT publishing a public migration banner on the archived old repo — Q1 closed with archive + private; no banner visible. Migration comms via new repo README only (Q13 closed).
  • NOT supporting hotfix-on-prior-line as a delivered-to-users mechanism — Q2 closed with forward-only. Hotfixes are subsumed into the next forward release. A tag like pmos-toolkit/v2.49.1 remains technically creatable for power-users who manually git checkout, but the marketplace install path will not surface it.
  • NOT enforcing the drift check at pre-push or in CI — Q4 closed with pre-commit only. Solo-maintainer pragmatism: trust the hook installed locally; --no-verify bypass is a known accepted risk.
  • NOT building /complete-dev with explicit knowledge of repo-specific remotes like the work-mirror — OQ10 closed: work-mirror is configured as a second git remote (git remote add work-mirror …); /complete-dev's existing push-to-configured-remotes behavior covers it transparently. The fact that this repo has two remotes is documented in CLAUDE.md ## Release policy. This preserves /complete-dev's generic-skill posture for repo-specific concerns.
  • NOT deferring Codex marketplace support — same-repo dual-manifest (.claude-plugin/marketplace.json + .codex-plugin/marketplace.json) ships day one. Both reference plugins via relative source which both schemas already support.
  • NOT publishing per-plugin GitLab mirrors — GitLab remains the canonical-source monorepo backup; per-plugin discovery isn't a GitLab use case.
  • NOT building a sandbox throwaway plugin or unit-test suite for the sync/drift tooling — solo-maintainer pragmatism: visual inspection of --dry-run output for sync-shared.sh + first-real-use-is-test.
  • NOT preserving the pre-cutover monorepo tag history under the new naming convention — Q5 closed: existing tags (mixed v2.x, pmos-toolkit-v2.x, pmos-toolkit-2.x) are kept as-is; the new <plugin>/v<semver> convention applies only to tags created from pmos-toolkit/v2.50.0 onward.

User Experience Analysis

Motivation

  • Job to be done (Maneesh): ship multiple independent plugin products from one workspace without paying ongoing release-coordination tax or losing cross-plugin refactor ergonomics, AND without building / maintaining ~300 LOC of cross-repo release tooling.
  • Importance/Urgency: blocking pre-scaffold for pmos-learnkit; every additional pmos-toolkit release before this lands is one more release that would need its history re-stitched if topology changed.
  • Alternatives: (a) full separate repos (forfeits substrate ergonomics); (b) umbrella + subtree-split (chose-then-rejected for tooling weight); (c) single repo + namespaced tags + peer _shared/ (chosen 2026-05-22 PM); (d) status quo (blocks pmos-learnkit). The 2026-05-22 pivot trades per-plugin GitHub identity for ~85% reduction in tooling LOC.

Satisfaction Signals

  • Maneesh ships pmos-learnkit/v0.1.0 in the same day as an unrelated pmos-toolkit/v2.51.0 — single repo, two tags, two pushes (to both remotes each), no coupling.
  • A _shared/html-authoring substrate edit happens once + one sync-shared.sh run + one commit; pre-commit hook passes; PR merges. Both plugins' next releases pick up the change.
  • An end-user discovers pmos-skills via direct GitHub search and adds both plugins from one marketplace command. (Some pre-cutover pmos-toolkit users won't migrate — accepted attrition per Q1/Q13.)
  • No release-tooling LOC count grows over the project's life beyond the initial ~60 LOC; only the canonical CLAUDE.md ## Release policy section grows.

Solution Direction

The chosen topology — confirmed in the 2026-05-22 pivot + second-grill — has four structural pieces and four tooling pieces.

Structural

  1. Single repo at github.com/maneesh-dhabria/pmos-skills holds the entire marketplace: .claude-plugin/marketplace.json + .codex-plugin/marketplace.json at root, plus all plugins under plugins/<name>/. The marketplace's name field changes from "pmos-toolkit" → "pmos-skills" at cutover (D17). Install URL: /plugin marketplace add maneesh-dhabria/pmos-skills. Branch protection on main with admin-bypass enabled.
  2. Per-plugin tree at plugins/<name>/ is fully self-contained: .claude-plugin/plugin.json + .codex-plugin/plugin.json + skills/ + agents/ + tools/ + scripts/ + tests/ + skills/_shared/ (a real directory, byte-identical to its peers in other plugins). No symlinks; no cross-plugin path resolution at runtime — skills already use the plugin-relative ${CLAUDE_PLUGIN_ROOT} token (verified /grill L1).
  3. _shared/ peer-copy invariant. No plugins/_shared/ directory exists at the repo level. Each plugin owns a real plugins/<name>/skills/_shared/. The N copies are kept byte-identical at HEAD by the pre-commit drift-check hook; they may temporarily diverge in a feature-branch working tree but must converge before commit lands on main.
  4. Existing maneesh-dhabria/pmos-toolkit standalone repo is archived AND made private at cutover (Q1). Old URL returns 404 publicly; no public migration banner possible. Existing cached end-user installs at v2.49.0 continue functioning read-only; no update path until users manually re-add the new marketplace. New repo's README is the sole migration breadcrumb (Q13).
  5. Two git remotes on the local pmos-skills monorepo: origin (personal: github.com/maneesh-dhabria/pmos-skills) + work-mirror (work: github.com-work:maneesh-dh/pmos-skills). /complete-dev's existing push-to-configured-remotes behavior covers both transparently — no code change needed to /complete-dev for the dual-remote case. GitLab remote remains the canonical-source backup (third remote). CLAUDE.md ## Release policy documents this two-remote topology as a repo-specific fact (OQ10).

Tooling

  1. scripts/sync-shared.sh (bash, target ≤ 30 LOC). Usage: scripts/sync-shared.sh --from=<plugin>. Behavior: rsync -a --delete plugins/<source>/skills/_shared/ plugins/<other>/skills/_shared/ for every plugin other than the source. Optional --dry-run prints proposed actions; mutates nothing. Idempotent.
  2. Pre-commit drift-check hook (bash, target ≤ 30 LOC; installed as .githooks/pre-commit; user runs git config core.hooksPath .githooks once after clone). Iterates plugins/*/skills/_shared/, picks the alphabetically-first peer as comparison baseline, runs diff -rq against each other peer. On any difference: fail with output naming divergent paths and the exact sync-shared.sh command. Single-layer enforcement (no pre-push, no CI) per Q4.
  3. Pre-push hook extension (extends existing .githooks/pre-push — note: .githooks/ directory doesn't yet exist in this repo; /execute creates it). Iterates plugins/*/ for: (a) 3-way version match per plugin — marketplace.json[plugin].version == plugins/<plugin>/.claude-plugin/plugin.json version == .codex-plugin/plugin.json version; (b) namespaced tag format on push (^<plugin>/v\d+\.\d+\.\d+$); (c) tag version matches all 3 manifest version fields for that plugin (rejects tag-without-bump bypass). Existing pre-cutover tags are not retroactively validated (Q5).
  4. Generalized /complete-dev (Q3 — flips the "NOT modifying /complete-dev" Non-Goal). Edits: (a) replace 10+ hardcoded plugins/pmos-toolkit/ references with plugins/${plugin}/ templated by the resolved --plugin argument; (b) add --plugin <name> flag handling; (c) add CLAUDE.md ## Release policy plugins: list reader; (d) add auto-detect-from-diff path-prefix logic with refuse-on-ambiguity for cross-plugin code commits; (e) add smart-detect for substrate-only multi-plugin commits → "ride which plugin's next release?" prompt; (f) bump marketplace.json[plugin].version alongside plugin.json + .codex-plugin/plugin.json (3-way). The cutover release — pmos-toolkit/v2.50.0 — is the LAST release done via the old hardcoded flow. First release done via new code: pmos-toolkit/v2.50.0 itself (chicken-and-egg accepted: dogfood at release time).

CLAUDE.md cutover edits

The current CLAUDE.md has 4 sections that reference single-plugin plugins/pmos-toolkit/ paths. The cutover generalizes all 4 + adds the new ## Release policy section (D18):

  • ## Canonical skill path (pmos-toolkit) → generalize to "each plugin loads from plugins/<plugin>/skills/<skill-name>/SKILL.md"; section title drops the parenthetical.
  • ## Skill-authoring conventions → generalize path references; the SDLC sub-bullet on /feature-sdlc skill still applies.
  • ## Plugin manifest version sync → expand to 3-way (marketplace.json + .claude-plugin + .codex-plugin) AND per-plugin (iterates plugins/*/). Lists the plugin set (initially pmos-toolkit, pmos-learnkit).
  • ## Release entry point → unchanged in spirit (/complete-dev is canonical); reference to --plugin <name> flag added.
  • ## Release policy (NEW) — encodes: plugin set; namespaced tag convention <plugin>/v<semver>; --plugin <name> requirement; auto-detect-from-diff with smart-detect rules; substrate-only-commit handling; drift-hook + sync-shared.sh contract; dual-remote push policy (origin + work-mirror); archival posture of the old standalone repo; tag-history-preserved-as-is.

User Journeys

J1 — Primary: Maneesh ships pmos-toolkit/v2.50.0 (the cutover release)

  1. Feature branch in pmos-skills repo. Commits land normally; cross-plugin substrate edits covered by J4. The pre-commit drift hook runs on every commit attempt.
  2. Merged into local main. Maneesh invokes /complete-dev --plugin pmos-toolkit (or omits the flag and lets auto-detect resolve from diff path-prefix if the diff is exclusively under plugins/pmos-toolkit/).
  3. Generalized /complete-dev reads CLAUDE.md ## Release policy, confirms the plugin set, then bumps in lockstep: plugins/pmos-toolkit/.claude-plugin/plugin.json 2.49.0 → 2.50.0, .codex-plugin/plugin.json 2.49.0 → 2.50.0, AND .claude-plugin/marketplace.json [pmos-toolkit].version → 2.50.0 + .codex-plugin/marketplace.json [pmos-toolkit].version → 2.50.0. Updates plugins/pmos-toolkit/CHANGELOG.md. Tags locally as pmos-toolkit/v2.50.0.
  4. Pre-push hook validates: (a) namespaced tag format, (b) 3-way version match per plugin, (c) tag-version-equals-manifest-version. Hook also fires drift-check defensively (idempotent — pre-commit already ran).
  5. Push to origin AND work-mirror (and GitLab as canonical-source backup). /complete-dev's existing push-to-configured-remotes behavior handles both transparently.
  6. End-users on /plugin marketplace add maneesh-dhabria/pmos-skills see pmos-toolkit v2.50.0 on their next CC client refresh (the marketplace.json[pmos-toolkit].version bump is what CC reads).
  7. Cutover note: v2.50.0 is the LAST release done via the old hardcoded /complete-dev codepath. v2.50.0 itself ships the generalized code; subsequent releases (pmos-toolkit/v2.51.0, pmos-learnkit/v0.1.0) use the new flow.

J2 — First-time: Maneesh ships pmos-learnkit/v0.1.0

  1. Scaffolds plugins/pmos-learnkit/ on a feature branch: skill dirs, both plugin.json manifests at 0.1.0, README (with Codex install snippet per OQ7), CHANGELOG starter. Adds empty skills/_shared/.
  2. Runs scripts/sync-shared.sh --from=pmos-toolkit to populate plugins/pmos-learnkit/skills/_shared/ from the canonical peer. Drift hook now passes.
  3. Updates root .claude-plugin/marketplace.json AND .codex-plugin/marketplace.json to add the pmos-learnkit entry: {"name": "pmos-learnkit", "source": "./plugins/pmos-learnkit", "version": "0.1.0", ...} in both.
  4. Updates CLAUDE.md ## Release policy plugins: list to include pmos-learnkit.
  5. Normal /complete-dev --plugin pmos-learnkit (using the generalized code shipped in v2.50.0): tags pmos-learnkit/v0.1.0, pushes to both remotes.
  6. Verifies in a fresh CC instance: /plugin marketplace add maneesh-dhabria/pmos-skills lists both plugins; pmos-learnkit installable at v0.1.0.

J3 — End-user installs pmos-skills

  1. End-user runs /plugin marketplace add maneesh-dhabria/pmos-skills in CC.
  2. CC fetches .claude-plugin/marketplace.json at HEAD of main.
  3. The picker shows both plugins with descriptions, homepages, categories from the manifest.
  4. End-user picks pmos-learnkit. CC resolves {"source": "./plugins/pmos-learnkit"}, fetches that directory, installs.
  5. Updates: CC re-reads marketplace.json[pmos-learnkit].version on next refresh. When it differs from the cached install version, CC offers the update.

J4 — Cross-plugin substrate refactor

  1. On a feature branch, Maneesh edits plugins/pmos-toolkit/skills/_shared/html-authoring/conventions.md.
  2. Tries to commit. Pre-commit hook detects drift between plugins/pmos-toolkit/skills/_shared/ and plugins/pmos-learnkit/skills/_shared/; fails with: 
    _shared/ drift detected:
  plugins/pmos-toolkit/skills/_shared/  →  plugins/pmos-learnkit/skills/_shared/
    html-authoring/conventions.md differs
Resolve with: scripts/sync-shared.sh --from=pmos-toolkit
  3. Maneesh runs scripts/sync-shared.sh --from=pmos-toolkit. Stages the resulting changes. Re-commits. Hook now passes.
  4. PR contains the substrate edit + sync delta as one logical change. Lands on main atomically.
  5. Next plugin release (either) carries the substrate change. If only the substrate changed (no plugin-private code), /complete-dev's smart-detect prompts "substrate-only change detected across pmos-toolkit + pmos-learnkit — ride which plugin's next release?" — user picks.

J5 — Error: forgot to run sync-shared.sh

  1. Edits _shared/ in one plugin. git add + git commit.
  2. Pre-commit hook fails with the drift message + exact sync-shared.sh command.
  3. Maneesh runs the sync. Re-stages. Commit passes.

J6 — Error: ambiguous commit set for /complete-dev

  1. Maneesh accidentally bundles real code edits across plugins/pmos-toolkit/ and plugins/pmos-learnkit/ in one merge (not just _shared/ — actual divergent code).
  2. Invokes /complete-dev without --plugin. Smart-detect classifies: "Diff spans plugins/pmos-toolkit/ AND plugins/pmos-learnkit/ with non-_shared changes in each. Refusing to auto-bump. Either pass --plugin <name> explicitly (and re-invoke once per plugin), or split the merge into per-plugin commits first."
  3. Maneesh picks one path. (Substrate-only-multi-plugin doesn't reach this refuse-branch — it hits the J4 smart-detect prompt instead.)
