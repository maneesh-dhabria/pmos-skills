Problem

The agent-skills repo currently hosts a single plugin (pmos-toolkit) and its marketplace manifest in one GitHub repo (maneesh-dhabria/pmos-toolkit). A second plugin (pmos-learnkit) is about to be scaffolded as an independent product, and more pmos-* plugins are anticipated. The current topology gives the second plugin no GitHub presence of its own (no stars/issues/SEO under its own name), conflates the toolkit's identity with the marketplace identity, and provides no per-plugin release independence. The original migration plan (docs/migration-separate-repos.md) proposed full separate repos per plugin, but that would forfeit the cross-plugin _shared/ refactor ergonomics the toolkit already depends on heavily.

Who experiences this?

  • Primary: Maneesh (sole maintainer) — running releases, doing cross-plugin substrate refactors, deciding repo topology.
  • Secondary: end-user plugin consumers — discovering, installing, and updating plugins via Claude Code's marketplace add mechanism.
  • Tertiary: future plugin authors (initially Maneesh in a different hat; potentially external contributors later) — scaffolding new pmos-* plugins under the umbrella.

Why now?

pmos-learnkit is at the pre-scaffold moment. Setting topology before its first commit is far cheaper than reorganising after the second plugin already has a release history. The ideation (2026-05-20) confirmed Claude Code's marketplace.json supports external git sources with ref/sha pinning, unblocking the umbrella pattern that was previously assumed unavailable.

Goals & Non-Goals

Goals

  • Each plugin (pmos-toolkit, pmos-learnkit, future pmos-*) is independently discoverable on GitHub by its own name — measured by: each plugin has its own GitHub repo with own stars, issues, README, and SEO surface.
  • Per-plugin release cadence is fully independent — measured by: tagging and releasing pmos-learnkit v0.1.0 does not modify any file or tag in pmos-toolkit's standalone repo.
  • Cross-plugin substrate refactors (e.g., changes to plugins/pmos-toolkit/skills/_shared/ or any future shared module) remain atomic — measured by: a substrate edit lands as a single commit on the monorepo's local main, then propagates to both plugins' standalone repos on next release each without manual sync.
  • End-users install all plugins via one umbrella marketplace add — measured by: /plugin marketplace add maneesh-dhabria/pmos-agent-skills exposes both pmos-toolkit and pmos-learnkit in the install picker.
  • The release tooling (script + /complete-dev extension) is honest and reversible — measured by: a --dry-run mode exists and is run before every first-time release of a new plugin, and the local monorepo is preserved unchanged so the topology can be unwound by abandoning the umbrella alone.

Non-Goals (explicit scope cuts)

  • NOT extracting _shared/ into its own standalone plugin/package in this iteration — because the pmos-learnkit consumption level is unknown until learnkit is real; revisit once consumption exceeds ~30% of _shared/ surface.
  • NOT shipping a Codex umbrella marketplace in this iteration — because Codex's external-source schema is not documented at parity with Claude Code; per-plugin Codex install remains the norm. Revisit when Codex documents the schema.
  • NOT creating per-plugin GitLab mirrors — because GitLab is solely the monorepo's backup/canonical-source remote; per-plugin GitHub repos serve discovery, not mirroring.
  • NOT migrating any existing pmos-toolkit GitHub stars / forks / issues forward — because the standalone pmos-toolkit repo already exists and continues; only its content source changes (from full-monorepo to subtree-split).
  • NOT building CI for the umbrella repo in this iteration — because the umbrella is near-static (changes only on add-plugin or ref-pin update); a manual gh api sanity check at edit time is sufficient until volume justifies CI.
  • NOT supporting git-subdir source type in the umbrella manifest — because github + standalone repos already give per-plugin discovery; git-subdir would re-point at the monorepo and undo the discovery axis.
  • NOT modifying /complete-dev (the pmos-toolkit skill) — because /complete-dev is a generic skill that already honors host-repo CLAUDE.md policies. Multi-plugin release scoping for this repo lives in CLAUDE.md ## Release policy; /complete-dev reads it as policy. Any generic /complete-dev extension (config-file scoping, etc.) is a separate skill-level change out of this feature's scope (per /grill Q8).
  • NOT publishing umbrella tags / GitHub releases on the umbrella repo — because the umbrella is rolling commits-on-main; each commit IS the meaningful state update (per /grill L2).
  • NOT installing pre-push hooks on standalone plugin repos — because GitHub branch protection is the sole guard against direct human pushes (standalone repos are publication targets, not dev surfaces) (per /grill L1).
  • NOT preserving pre-cutover historical tags (v2.49.0 and older) on the standalone pmos-toolkit repo after the first subtree-split cutover — because forward-only is simpler; pre-cutover tags remain resolvable in the monorepo (per /grill L4).
  • NOT building a sandbox throwaway plugin or unit-test suite for the release script — because solo-maintainer pragmatism: --dry-run against pmos-toolkit + first-real-use-is-test (per /grill Q7).

User Experience Analysis

Motivation

  • Job to be done (Maneesh): ship multiple independent plugin products from one workspace without paying ongoing release-coordination tax or losing cross-plugin refactor ergonomics.
  • Importance/Urgency: blocking pre-scaffold for pmos-learnkit; every release of pmos-toolkit done before this lands is a release that would need its history re-stitched if we later changed topology. Cost compounds.
  • Alternatives: (a) full separate repos (forfeits _shared/ atomic refactors), (b) namespaced tags in one repo (forfeits per-plugin discovery), (c) status quo (blocks pmos-learnkit's product identity). All worse than the umbrella + subtree-split shape per the ideation's decision table.

Satisfaction Signals

  • Maneesh ships pmos-learnkit v0.1.0 and immediately ships an unrelated pmos-toolkit v2.51.0 the next day — both releases feel independent end-to-end (no coupled bumps, no shared changelog edits, no cross-plugin tag prefixes).
  • A _shared/html-authoring substrate edit happens once on the local monorepo and propagates to both standalone repos on next release each — no "where do I make this change?" hesitation.
  • An end-user discovers pmos-learnkit via direct GitHub search (not via pmos-toolkit's README) within the first month of public visibility.

Solution Direction

The chosen topology — confirmed in the ideation artifact's decision table — has three structural pieces and two tooling pieces.

Structural

  1. Umbrella repo at github.com/maneesh-dhabria/pmos-skills (name finalised per /grill Q9) holds only .claude-plugin/marketplace.json + README + (eventually) cross-plugin docs. Install URL: /plugin marketplace add maneesh-dhabria/pmos-skills.
  2. Per-plugin standalone GitHub repos — maneesh-dhabria/pmos-toolkit (already exists; cutover via /grill Q6/L4), maneesh-dhabria/pmos-learnkit (new), maneesh-dhabria/pmos-<future> as they're created. Each is the discovery surface for one product: README, stars, issues, releases. Branch protection on main with admin-bypass enabled (per /grill L3).
  3. Work-account mirror repos — github.com-work:maneesh-dh/<plugin> kept in sync by the release script. Umbrella references personal account only (per /grill Q4).
  4. Local monorepo at agent-skills/ stays as-is structurally — single .git, plugins under plugins/<name>/. Cross-plugin substrate refactors remain atomic single-commit operations.
  5. Shared substrate relocation — plugins/pmos-toolkit/skills/_shared/ moves up to plugins/_shared/ (top-level, sibling to each plugin). The release script vendors a copy of plugins/_shared/ into each plugin's tree immediately before subtree-split, so standalone repos are self-contained (per /grill Q3). Each release pulls the current substrate vintage; per-release substrate-vintage divergence between plugins is accepted (per /grill L6).

Tooling

  1. Release script (scripts/release-plugin.sh, bash, target ~250 LOC) wraps the full publication flow per plugin per release. Responsibilities:
    • Vendor plugins/_shared/ into plugins/<name>/_shared/ (or equivalent path; /spec resolves).
    • git subtree split --prefix=plugins/<name> -b release-<name> against current monorepo HEAD or a passed-in --from-branch <ref> (for hotfix flows per /grill Q1).
    • Validate remote URLs match expected patterns before pushing (personal + work mirror).
    • Push release-<name> to personal standalone repo as main; push the unprefixed tag v<semver>.
    • Push the same to the work-mirror remote.
    • Auto-bump the umbrella manifest's ref for this plugin to the just-pushed tag, commit, push umbrella (per /grill Q5).
    • --dry-run mandatory: prints every would-be action; mutates nothing.
    • Idempotent retry: re-running after partial failure is safe; already-successful pushes no-op (per /grill L5).
  2. Pre-push hook extension (.githooks/pre-push, monorepo only):
    • Iterate plugins/*/ and validate per-plugin manifest version-sync (extends existing single-plugin hook).
    • Validate namespaced tag format ^(?P<plugin>[a-z0-9-]+)/v\d+\.\d+\.\d+$ on push; reject unprefixed v* tags.
    • Active bypass guard — when pushing a tag <plugin>/v<ver>, verify git ls-remote <plugin> refs/tags/v<ver> returns a sha matching what the standalone repo would have at that tag. On miss, reject with tag bypasses release flow — run scripts/release-plugin.sh first (per /grill Q2).
  3. CLAUDE.md ## Release policy section in THIS repo — encodes the multi-plugin release rules /complete-dev reads as host-repo policy (per /grill Q8): the plugin set, the namespaced-tag-in-monorepo / unprefixed-in-standalone convention (D7), the --plugin <name> requirement, the shared-only-commit policy, the cutover audit checklist. No /complete-dev code changes are part of this feature — see Non-Goals.

Codex (deferred)

The umbrella ships with a .claude-plugin/marketplace.json only. A future iteration adds .codex-plugin/marketplace.json when Codex documents external-source schema at parity. Until then, Codex users install each plugin's .codex-plugin/plugin.json from its standalone repo individually (Codex's current norm).

User Journeys

J1 — Primary: Maneesh ships pmos-toolkit v2.50.0

  1. Maneesh works on a feature branch in the local monorepo (agent-skills/). Commits land normally; cross-plugin substrate edits (if any) land as single commits on plugins/_shared/.
  2. Branch merged into local main. Maneesh invokes /complete-dev --plugin pmos-toolkit (or relies on its host-CLAUDE.md-driven detection if commits clearly map to one plugin).
  3. /complete-dev (following CLAUDE.md ## Release policy) bumps plugins/pmos-toolkit/.claude-plugin/plugin.json + .codex-plugin/plugin.json from 2.49.0 to 2.50.0, updates plugins/pmos-toolkit/CHANGELOG.md, and tags locally as pmos-toolkit/v2.50.0. The pre-push hook validates the tag format on its way out.
  4. /complete-dev's publish phase invokes scripts/release-plugin.sh pmos-toolkit 2.50.0. The script: (a) vendors plugins/_shared/ into the pmos-toolkit tree, (b) git subtree split --prefix=plugins/pmos-toolkit -b release-pmos-toolkit, (c) validates personal + work-mirror remote URLs, (d) pushes release-pmos-toolkit → maneesh-dhabria/pmos-toolkit:main and tag v2.50.0, (e) pushes the same to the work-mirror, (f) bumps the umbrella manifest's pmos-toolkit ref to v2.50.0, commits, pushes umbrella.
  5. The pre-push hook's bypass guard sees the standalone repo now has v2.50.0 at the expected sha; allows the monorepo tag push to proceed.
  6. Monorepo pushes to GitLab as usual (canonical source-of-truth backup).
  7. End-users on the umbrella marketplace see pmos-toolkit v2.50.0 next time their CC client refreshes the manifest (umbrella now pins to v2.50.0; rollback = umbrella commit reverting the ref to v2.49.0).

J2 — First-time: Maneesh ships pmos-learnkit v0.1.0

  1. Maneesh creates two GitHub repos via gh repo create: personal (github.com/maneesh-dhabria/pmos-learnkit) and work mirror (github.com-work:maneesh-dh/pmos-learnkit). Branch protection on personal's main with admin-bypass enabled.
  2. Scaffolds plugins/pmos-learnkit/ in the monorepo (skill dirs, both plugin.json manifests at 0.1.0, README, CHANGELOG starter). Updates CLAUDE.md ## Release policy to include pmos-learnkit in the plugin set.
  3. Adds two git remotes in the monorepo: git remote add pmos-learnkit git@github.com:maneesh-dhabria/pmos-learnkit.git and git remote add pmos-learnkit-work git@github.com-work:maneesh-dh/pmos-learnkit.git.
  4. Edits the umbrella repo's .claude-plugin/marketplace.json to add the pmos-learnkit entry with {"source": "github", "repo": "maneesh-dhabria/pmos-learnkit", "ref": "v0.1.0"}. (Initial ref is set to the version about to ship; umbrella commit is preceded by the release-script push.)
  5. Runs scripts/release-plugin.sh pmos-learnkit 0.1.0 --dry-run first to inspect the would-be-pushed branch. Output enumerates: vendored substrate paths, subtree-split commit count, personal/work push targets, umbrella ref bump (from absent to v0.1.0). Maneesh confirms.
  6. Runs scripts/release-plugin.sh pmos-learnkit 0.1.0 for real. Script pushes to both standalone repos and commits/pushes the umbrella ref bump.
  7. Verifies end-to-end with /plugin marketplace add maneesh-dhabria/pmos-skills in a sandbox CC instance — both plugins listed; pmos-learnkit installable at v0.1.0.

J2.5 — Hotfix: Maneesh ships pmos-toolkit v2.49.1 a week after v2.50.0 already shipped

  1. A security fix is needed on the v2.49.0 line; monorepo main has moved past v2.50.0 with unrelated work.
  2. Maneesh creates a hotfix branch in the monorepo: git checkout -b hotfix/pmos-toolkit-v2.49.1 pmos-toolkit/v2.49.0. Cherry-picks the fix commit (or hand-codes it) on this branch. Bumps pmos-toolkit's manifests to 2.49.1; updates CHANGELOG with a hotfix entry.
  3. Tags locally as pmos-toolkit/v2.49.1.
  4. Runs scripts/release-plugin.sh pmos-toolkit 2.49.1 --from-branch hotfix/pmos-toolkit-v2.49.1 --dry-run. Script subtree-splits the hotfix branch (not main) and reports: "Would push to a release branch release-2.49.x on the standalone repo (not main, since this is older than main's current tag), plus tag v2.49.1."
  5. Runs for real. Standalone repo gets the hotfix on a maintenance branch + the tag v2.49.1. main on the standalone repo is unchanged (still at v2.50.0 content).
  6. Umbrella manifest can optionally pin to v2.49.1 for users who haven't moved to v2.50.0 yet (rare; usually just publishes the hotfix tag and lets v2.50.0 users continue). Most common path: don't touch the umbrella; v2.49.1 is reachable for anyone who explicitly pins.
  7. Hotfix is merged forward into monorepo main as a separate commit later (so v2.51.0 includes the fix in its line as well).

J3 — End-user installs the umbrella

  1. End-user runs /plugin marketplace add maneesh-dhabria/pmos-agent-skills in Claude Code.
  2. CC fetches the umbrella's .claude-plugin/marketplace.json from GitHub.
  3. The picker shows both plugins with descriptions, homepages, and categories pulled from the manifest.
  4. End-user picks pmos-learnkit. CC resolves {"source": "github", "repo": "maneesh-dhabria/pmos-learnkit", "ref": "main"}, clones at HEAD of main, installs.
  5. End-user uses the plugin. Updates come either automatically (when the umbrella tracks main) or on next umbrella refresh (when pinned).

J4 — Alternate: cross-plugin substrate refactor

  1. Maneesh edits plugins/pmos-toolkit/skills/_shared/html-authoring/conventions.md on a feature branch in the monorepo.
  2. Both pmos-toolkit and (future) pmos-learnkit consume this file. The edit is single-commit on the monorepo — same as today.
  3. Maneesh decides to release the substrate change as part of pmos-toolkit v2.51.0 first (because pmos-toolkit's release cadence is faster). /complete-dev --plugin pmos-toolkit as in J1. The pmos-toolkit standalone repo gets the updated _shared/ via the subtree-split.
  4. pmos-learnkit's next release (whenever) picks up the substrate change automatically as part of its own subtree-split.
  5. No "sync these N repos manually" step is ever required.

J5 — Error: release script remote mismatch

  1. Maneesh runs scripts/release-plugin.sh pmos-learnkit 0.1.0 without having added the pmos-learnkit remote (or after accidentally pointing it at the wrong URL).
  2. The script's pre-push validation step runs git remote get-url pmos-learnkit and verifies the URL matches the expected pattern git@github.com:maneesh-dhabria/pmos-learnkit(\.git)?.
  3. On mismatch: prints "Remote pmos-learnkit URL is <actual>; expected git@github.com:maneesh-dhabria/pmos-learnkit.git. Fix with git remote set-url and retry." Exits non-zero. No subtree-split or push happens.
  4. Maneesh fixes the remote, re-runs. Idempotent (the script is safe to re-invoke).

J6 — Error: ambiguous commit set for /complete-dev

  1. Maneesh accidentally bundles edits across plugins/pmos-toolkit/ and plugins/pmos-learnkit/ in one merge.
  2. Invokes /complete-dev without --plugin. The skill's auto-detect reports "Diff spans plugins/pmos-toolkit/, plugins/pmos-learnkit/. Refusing to auto-bump; pass --plugin <name> explicitly, or split the merge into per-plugin commits first."
  3. Maneesh picks one path: either re-runs with explicit --plugin pmos-toolkit (and again with --plugin pmos-learnkit), or rewrites history to split the commits.
