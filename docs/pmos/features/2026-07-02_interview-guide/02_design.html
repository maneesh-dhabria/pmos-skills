/interview-guide — author interview guides before the round

Epic design = the cross-story coherence contract. Two stories cite this by anchor. Sibling to /interview-feedback: this authors the guide before the interview; /interview-feedback scores a candidate after. They share one guidelines substrate.

One-line charter. Given a role and the competencies a round must assess, /interview-guide produces three grounded, self-contained HTML artifacts a hiring team can use immediately: (a) an Interviewer Reference (how to run and probe the round, what strong/average/poor looks like, common mistakes, a nudge ladder), (b) a Scoring Sheet (competency-aligned dimensions, scales, green/red flags, overall hire criteria), and — for case rounds only — (c) a candidate-facing Case Document (1–2 pages). The already-researched 7-archetype PM-round corpus that currently lives inside /interview-feedback is the grounded canon; this epic first lifts it into a shared managerkit substrate both skills consume.

0Context & why this is an epic, not a monolith

The 7-archetype guidelines corpus (recruiter-screen, product-sense, analytical, technical, behavioral, case-study, case-presentation), the reference-skeleton.html / scorecard-skeleton.html authoring contracts, and the interviewer-effectiveness.html rubric were researched and shipped inside plugins/pmos-managerkit/skills/interview-feedback/reference/ (epic 260616-9bt). The user's brief explicitly asks to "abstract that to a shared reference" so a second skill can build on it. That extraction touches an existing, shipped skill (retrofit + re-verify), so it is independently shippable substrate and gets its own story that the new-skill story depends on — the exact define-route-skill split rule (substrate is its own story; dependents list it in dependencies:, claim-time dep-merge makes it present in the consumer's build before its skill-eval).

Bootstrap note (CLAUDE.md "sync-shared intersection-only" gap). pmos-managerkit has no skills/_shared/ directory yet. This is the first shared substrate in the plugin, so Story 1 creates it by hand (a move, not a cross-plugin sync). No sync-shared.sh path applies — this substrate is managerkit-internal, consumed by two sibling skills in the same plugin.

3Output shapes (grounded in the porter gold-standard)

3a — Interviewer Reference (all rounds)

Sections: the one-line key / what the round tests · what a strong vs average vs poor answer looks like (per competency) · how to open and time the round · a probing / nudge ladder (softest nudge that unsticks the candidate; the gap between soft and hard nudge is itself signal) · common interviewer mistakes (leading, over-talking, anchoring) · calibration guidance. Modeled on v3-bidding-case-interviewer-brief.html + the corpus interviewer-reference.html.

3b — Scoring Sheet (all rounds)

Competency-aligned dimensions (from the round's stated competencies, D4/D7), each with a scale, weight, green/red flags, and a notes field; an overall hire/no-hire criterion tying dimension scores to a bar. Carries the scorecard-skeleton anchors (INV-2). Modeled on v3-bidding-case-scoring-sheet.html.

3c — Case Document (case rounds only, D9)

Candidate-facing, 1–2 pages: the business context, the brief/prompt, any data table, constraints, and what's expected of the candidate. Authored from the user's supplied business context (D7/D11). The matching interviewer reference-solution (3a) and rubric (3b) are authored together so the "what strong looks like" markers map to the actual case. Modeled on v3-bidding-case-question.html.
