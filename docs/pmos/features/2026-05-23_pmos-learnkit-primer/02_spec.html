1. Problem Statement

PM-adjacent practitioners ramping on an unfamiliar topic — before a meeting, scope, or doc review — have no fast tool that produces a verified-source, reviewer-vetted, persistent teaching artifact. /primer fills that gap as the upstream skill of a new pmos-learnkit plugin. Primary success metric: < 6 min wall-clock from invocation to written HTML artifact on the first 3 real runs, with 100% of cited URLs resolving against the persisted sources.json sidecar.

2. Goals

3. Non-Goals

  • NOT a source-eval / author-trust YAML registry in v1 — load-bearing quality comes from prompt-level curator-lens framing (A1 hypothesis); registry deferred to a v2 sibling /source-eval skill.
  • NOT --depth shallow/deep — ship --depth mid only.
  • NOT diagram-pass, persona-pass, blind-primer phase, learnings-as-phase, or worked-examples phase — explicitly NOT porting /rewrite's full machinery.
  • NOT topic-or-URL auto-detect, --sources BYO-sources, or topic-tree recursion in v1.
  • NOT validating A1 pre-ship; NOT staging an A1 mid-build kill criterion (per D-A1).
  • NOT pinned-tag resume — /primer is single-session; harness handles compact-then-resume.
  • NOT flashcard / quiz / retrieval-practice outputs — those are sibling skills consuming /primer.
  • NOT spawning subagents in Phases 1–4 — only Phase 5 reviewer (per D2).
  • NOT introducing a plugins/pmos-toolkit/skills/_shared/ change that breaks existing pmos-toolkit skills — substrate parameterization (per D-ATTRIB) MUST default to "pmos-toolkit" so existing skills are byte-stable.

7. Functional Requirements

7.1 FR-A — skill-patterns.md §A: Frontmatter (derived requirements)

IDRequirement
FR-A1The skill ships at canonical path plugins/pmos-learnkit/skills/primer/SKILL.md.
FR-A2Frontmatter name: primer (lowercase-hyphenated; matches directory name; ≤64 chars).
FR-A3Frontmatter description: <string> — ≤1024 chars, third-person, contains both a one-clause WHAT and an explicit WHEN, ≥5 user-spoken trigger phrases (see FR-B3); no embedded numbered/bulleted step list.
FR-A4Frontmatter includes user-invocable: true (Claude Code slash-command requirement).
FR-A5Frontmatter includes argument-hint: string enumerating every flag the body parses: <topic> [--audience <senior-pms|all-pms>] [--autonomous] [--format <html|md|both>] [--non-interactive] [--interactive].
FR-A6The name primer reads as a noun-of-action (the artifact produced), permitted per §A's resolved rule (verb or gerund or action-noun) — the failure case is bare-noun-phrase like pdf-utilities.

7.2 FR-B — skill-patterns.md §B: Description & triggering

IDRequirement
FR-B1Description includes a one-clause WHAT: "Generate a verified-source, audience-shaped HTML primer on a topic, with a reviewer-vetted artifact at a deterministic path." (or near-equivalent).
FR-B2Description includes explicit WHEN/trigger conditions: ramp-up before meeting/scope/doc-review; PM topic the user does not yet own.
FR-B3Description contains ≥5 user-spoken trigger phrases. Minimum set: "write me a primer on X", "ramp me up on Y", "generate a primer", "I need to learn about Z before a meeting", "/primer", "create a teachable artifact on this topic".
FR-B4Description is written in third person ("Use when…", not "I will…").
FR-B5Description contains NO numbered/bulleted step list, NO embedded mini-workflow. (Workflow lives in the body.)
FR-B6Description trigger conditions are phrased assertively ("Use this whenever…"), not hedged ("This skill can optionally help with…"). Anti-under-triggering.

7.8 FR-1..N — /primer behavior (per requirements)

IDRequirement
FR-1Invocation. CLI shape: /primer <topic> [--audience <senior-pms|all-pms>] [--autonomous] [--format html|md|both] [--non-interactive] [--interactive]. Last flag wins on conflict. --non-interactive/--interactive follow the canonical non-interactive-block precedence (cli > parent marker > settings.default_mode > builtin "interactive").
FR-2Phase 0: Pipeline setup. Read .pmos/settings.yaml for docs_path, output_format, default_mode. Read ~/.pmos/learnings.md §/primer if present. Resolve output_format per the canonical FR-12 precedence (cli --format > settings > default "html"). Print to stderr: output_format: <v> (source: <cli|settings|default>) and mode: <m> (source: <…>).
FR-3Phase 0.5: Consolidated confirm. Read .pmos/primer.lastrun.yaml if present. Single AskUserQuestion seeded from lastrun values (audience, output_format) lets the user one-touch-confirm or edit. --autonomous skips this gate.
FR-4Phase 1: Intake. Derive <slug> from topic via _shared slug rules (lowercase, hyphenate, ≤64 chars). Canonical artifact path: {docs_path}/primer/{YYYY-MM-DD}_<slug>.html. If --audience missing AND not in lastrun, AskUserQuestion (Recommended: senior-pms). If --audience value is unknown (e.g., eng-leads), reject with platform-aware error listing v1 set + suggestion "use --audience all-pms as the broadest preset" (per requirements error journey #8).
FR-4.1Topic vagueness heuristic. If topic is <3 tokens AND no clear noun phrase, AskUserQuestion with 3 candidate refinements + "Other" escape (per error journey #1). --autonomous deferred-default: proceed with the topic as given (logged in OQ buffer).
FR-4.2Artifact path collision. If {date}_{slug}.html exists, AskUserQuestion: Overwrite (git-snapshot first) / Append -2 suffix (Recommended) / Abort.
FR-5Phase 2: Research — try-both strategy. Dispatch in parallel: (a) context7 resolve-library-id with the topic verbatim; (b) WebFetch on 6–10 candidate URLs derived from the topic (seed via tool's domain-discovery). For each fetched URL, verify it resolves and capture a 1-line takeaway. Persist sources.json sidecar.
FR-5.1≥3-source short-circuit. If either strand (context7 OR WebFetch) yields ≥3 usable resolving sources BEFORE the other completes, prefer that strand and cancel/drop the other's pending fetches (best-effort cancellation; on the host platform that means simply not awaiting). This bounds Phase 2 latency.
FR-5.2Source floor = 4. If merged Phase-2 output yields <4 verified resolving sources, surface the source-floor gate (Abort (Recommended) / Proceed with thin-source disclosure / Retry with alternate query frame). On Retry, run one alternate query reframe; if still <4, re-surface the gate (no auto-retry loop).
FR-5.3sources.json schema. One JSON file at {date}_{slug}.sources.json sibling to the HTML: { "topic": "...", "fetched_at": "<ISO-8601>", "audience": "...", "sources": [ { "url": "...", "title": "...", "takeaway": "<1-line>", "source": "<web|context7>", "resolved_at": "<ISO-8601>" }, ... ] }.
FR-5.4WebFetch unavailable. If no WebFetch tool present (platform adaptation), Phase 2 surfaces "WebFetch unavailable. Abort / Continue with no sources (NOT recommended)." If continued, artifact is marked "uncited draft" in footer banner; reviewer rubric flags it; trust-tier cites-real-urls resolves to N/A-fail (counts as a trust fail unless explicitly overridden).
FR-6Phase 3: Outline. Render a 10–14 H2 section outline biased by the audience preset's required-sections list (per requirements §Audience preset shape). Inline the outline in chat for user review.
FR-6.1Phase-3 interactive gate. AskUserQuestion: Approve outline (Recommended) / Edit / Re-prompt with feedback / Abort. On Edit, accept free-form reply and re-render. On Re-prompt, accept guidance text and re-generate from scratch. --autonomous skips the gate.
FR-7Phase 4: Draft. Single-pass draft against the approved outline, inlining citations as <a href="{url-from-sources.json}"> tags. Target ~3000 words for --depth mid. No diagram-pass, no persona-pass, no blind-primer. Curator-lens framing applied via reference/curator-lens.md prompt content.
FR-7.1Citation discipline. Every <a href> URL in the draft MUST appear in sources.json. No novel URLs introduced. (Enforced as trust-tier reviewer check cites-real-urls.)
FR-8Phase 5: Eval — reviewer subagent. Dispatch Task tool with a fresh subagent prompt: read the draft + sources.json + the 10-check rubric from reference/rubric.md; return a JSON array of {check_id, verdict: pass|fail, evidence, quote}. The reviewer makes NO edits — scores and reports only. (Mirrors /skill-eval D10 separation.)
FR-8.1Reviewer URL verification authority. Reviewer cross-checks every URL in the draft against sources.json; reviewer does NOT re-fetch URLs (per requirements D15 / OQ-6 resolution). Phase 2 owns URL resolution.
FR-8.2Reviewer-output validation. After the reviewer returns, /primer validates the returned set: (a) check_id set MUST equal the 10 IDs declared in reference/rubric.md — any miss/extra is a hard fail with the precise diff. (b) For every fail verdict, the quote field MUST be a ≥40-char verbatim substring of the draft (or of sources.json for plagiarism check) — a fail with empty/non-matching quote is treated as pass. Pattern mirrors /skill-eval FR-43/44.
FR-8.3Auto-apply iteration 1. If iteration-1 verdict has any fail, /primer authors a fix patch inline (NOT the reviewer — separation of writer and reviewer) and re-runs the reviewer ONCE. Iteration cap = 1.
FR-8.4Trust-tier hard-block. After iteration 1, if ANY trust-tier check is still failing (cites-real-urls, no-plagiarism, no-hand-wavy-claims, sections-json-ids-match), /primer hard-blocks the write per FR-RECOVERY below.
FR-8.5Taste-tier accept-as-known-risk. If after iteration 1 only taste-tier checks are failing, /primer surfaces the residuals (AskUserQuestion: Accept as known risk and write (Recommended) / Iterate manually / Abort). On Accept, the artifact ships with a "Known residuals" footer note listing the failed taste-tier check IDs.
FR-8.6Phase-5 interactive gate. Before the final write (when not hard-blocked), AskUserQuestion: Approve and write (Recommended) / Iterate manually / Abort. --autonomous skips this gate (residuals still apply per FR-8.5; trust-tier still hard-blocks per FR-8.4).
FR-9Phase 5 → write. On approve, atomic temp-then-rename for the trio: {date}_{slug}.html.tmp, {date}_{slug}.sections.json.tmp, {date}_{slug}.sources.json.tmp. After all three temp-writes succeed, rename all three in sequence; any rename failure aborts and removes all temps. (No partial states.) 7.9 FR-RUBRIC — the 10 reviewer-rubric checks (per OQ-2 resolution)

Lives in plugins/pmos-learnkit/skills/primer/reference/rubric.md. The reviewer subagent is prompted with this file inlined as the rubric source-of-truth.

#check_idtierPass-criterion (binary)
R1cites-real-urlstrustEvery <a href> URL in the primer body is present (exact match, after normalization — trailing-slash + fragment-stripped) in sources.json[].url. No novel URLs.
R2no-plagiarismtrustNo verbatim span ≥40 contiguous chars from any sources.json[].takeaway appears outside a <blockquote> or <pre>/<code> block. (Code samples and explicit blockquotes are allowed verbatim.)
R3no-hand-wavy-claimstrustNo orphan "should/might/could" sentence without a cited antecedent or self-derivation. Every quantitative claim (number, percentage, "most", "many", "studies show") has either a <a href> citation OR a self-derivation in-text.
R4covers-groundtasteEvery approved Phase-3 H2 section title appears as an <h2> in the draft (case-insensitive, normalized).
R5right-reading-leveltasteFor --audience senior-pms: no inline definitions of common PM concepts (a-b-test, NPS, churn, retention, activation). For --audience all-pms: every term of art on first use carries a 1-sentence inline definition.
R6structurally-completetasteDocument has exactly one <h1>, ≥10 <h2> sections, opening paragraph <120 words, closing section matches audience preset's "closing shape" (senior: open debates; all: concrete actions).
R7sections-json-ids-matchtrust (per D-RUBRIC)sections.json[].id set equals the on-page <h2>/<h3> id= attribute set, in document order. (Composability contract for /flashcards, /quiz consumers.)
R8audience-vocabtasteSection selection matches audience preset's required-section list (per requirements §Audience preset shape). senior-pms: includes "Tradeoffs we considered against", "Metrics calibration", "Open debates". all-pms: includes "Common pitfalls", "What to do on day 1", "Questions to ask your team".
R9flags-uncertaintytasteWhere the cited sources disagree, or where evidence is thin (3 sources or fewer agreeing), the primer says so — at least one phrase like "the evidence is mixed", "sources disagree", "data is thin" or near-equivalent OR an explicit "Open debates" section flags the disagreement.
R10primer-shapedtasteThe document teaches a topic — it does NOT argue a thesis. No thesis statement in the opening (no "this article argues that…", "the answer is…", "X is the right approach"). No call-to-action closing for a specific position.

15. Rollout Strategy

  1. Substrate change first. Land FR-14 (template.html parameterization) in pmos-toolkit, bump pmos-toolkit to a patch release (e.g. 2.51.1), confirm byte-stable attribution on existing artifacts.
  2. Scaffold pmos-learnkit plugin. Create plugins/pmos-learnkit/.claude-plugin/plugin.json + .codex-plugin/plugin.json at v0.1.0. Update both marketplace.json files to add the pmos-learnkit entry.
  3. Sync substrate. Run scripts/sync-shared.sh --from=pmos-toolkit. Verify pre-commit drift hook is green.
  4. Implement /primer. Per /plan tasks (T1..T_N from the plan doc). Single-skill scope; no script bundles.
  5. Skill-eval gate. /feature-sdlc Phase 6a runs the rubric. Trust-tier checks must pass; taste-tier residuals require explicit accept-as-known-risk.
  6. Verify. /verify runs the manual smoke triple (§14.2) + deterministic skill-eval + audit-recommended.
  7. /complete-dev releases. Phase 8 of /feature-sdlc writes the README row, the changelog entry, the ~/.pmos/learnings.md ## /primer header bootstrap, tags pmos-learnkit/v0.1.0, and pushes to both configured remotes (origin + gitlab-mirror).
  8. Rollback plan. The new plugin is additive — removal is git rm -r plugins/pmos-learnkit/ + revert the four manifest entries + revert the substrate change. No data migrations to reverse. Cached installs of pmos-toolkit are unaffected.
  9. Post-ship monitoring. Watch for: (a) the first 3 runs hitting < 6 min (G1 calibration); (b) reviewer rubric catching ≥1 issue per run on average over 10 runs (NFR calibration — if the rubric never fires, it's miscalibrated); (c) user external-share signal within 2 weeks (G3 / A1 validation early indicator).
  10. Post-v1 A1 retrospective. After 3 successful runs, the 2-hour side-by-side spike runs (D-A1). If A1 disconfirms in practice, v2 ports a minimal /source-eval sibling skill (per requirements Non-Goal #6).
