Standing acceptance criteria: the produced skill must conform to reference/skill-patterns.md §A–§F (frontmatter, description & triggering, structure & progressive disclosure, body & content, scripts & tooling, platform-conditional frontmatter). /spec will turn each applicable § into one or more FRs; /skill-eval grades the produced skill against the binary rubric in reference/skill-eval.md.

Problem

A PM (or PM-adjacent practitioner) who needs to ramp on an unfamiliar topic — before a meeting, a feature scope, or a doc review — has no fast tool that produces a verified-source, reviewer-vetted, persistent teaching artifact. Direct LLM prompts ("write me a primer on X") produce ungrounded summaries: no URL resolution, no source persistence, no quality pass, no audience targeting, no artifact to return to. The existing /rewrite skill in pmos-content handles the source-grounded rewriting case (URL in) with a 9-phase machinery, but there is no peer for the greenfield "topic in" case, and porting /rewrite's full apparatus (author YAML registry, diagram-pass, persona-pass, pinned-tag resume) is disproportionate.

v1 differentiation claim — "verified-source primer with a reviewer pass": what /primer delivers over a direct LLM prompt is (a) a sources.json sidecar where every cited URL has been WebFetch-resolved and the takeaway captured at fetch time, (b) a binary reviewer rubric with split trust-tier and taste-tier semantics, (c) an audience preset that drives concrete differences in section selection, background depth, and closing shape, and (d) a persistent addressable HTML artifact at a deterministic path. The brief's larger "curator-lens primer" framing is a stretch goal contingent on the post-v1 A1 validation (see OQ-1 and Non-Goal #5) — v1 ships and is marketed on the smaller, defensible claim.

Who experiences this?

  • Primary persona: Maneesh (the author) and PM-adjacent practitioners — PMs scoping a feature in unfamiliar territory, EMs joining a domain mid-flight, founders evaluating a new market.
  • Context: Inside Claude Code with the pmos-skills marketplace installed; pre-meeting / pre-scoping / pre-doc-review.
  • Job to be done: "When I want to ramp on a PM topic I don't yet own, I want a primer that synthesizes vetted sources into a teachable artifact, so I can move from 'I've heard of this' to 'I can lead a conversation about it' in one read."

Why now?

  • pmos-skills already ships a mature pipeline (requirements → spec → plan → execute → verify) and a substrate (_shared/html-authoring, interactive-prompts, pipeline-setup, non-interactive). The marginal cost of a second plugin reusing that substrate is now low.
  • /rewrite's curator-lens patterns are proven in pmos-content (v0.15.0). The lessons transfer; the machinery does not need to.
  • Every other learnkit skill the user has sketched (/flashcards, /quiz, /source-eval, /topic-tree) needs content to operate over. /primer is the upstream dependency — it has to exist before the rest of pmos-learnkit can.

Goals & Non-Goals

Goals are observable user outcomes; engineering acceptance criteria belong in /spec.

Goals

  • G1 — One-command topic-to-primer. User runs /primer <topic> and gets a teachable, source-grounded HTML artifact in under 6 minutes wall-clock for the default depth — measured by an end-to-end timer on the first 3 real runs.
  • G2 — Source-grounded, not hallucinated. Every citation in the primer is a real, resolving URL captured in a sidecar sources.json — measured by the reviewer subagent's binary "cites real URLs" check passing on every run.
  • G3 — Audience-shaped on three concrete axes. Output differs measurably between the senior-pms and all-pms presets along three axes — section selection, background depth, and closing shape (see "Audience preset shape" subsection below) — not as a one-line prompt-suffix slider. Measured by reviewer-rubric checks tied to each axis, and by user post-run "would-share-externally" signal across the first 5 runs.
  • G4 — Substrate-native. /primer reuses the pmos-skills _shared/ substrate (html-authoring, interactive-prompts, pipeline-setup, non-interactive) — measured by SKILL.md staying under 500 lines and the artifact being /polish-compatible out of the box.
  • G5 — Composable upstream of the rest of pmos-learnkit. The primer's HTML carries a stable sections.json with addressable section IDs so future /flashcards, /quiz, and /source-eval skills can consume /primer output unchanged — measured by sections.json being present, parseable, and IDs matching the on-page <h2>/<h3> ids.

Non-Goals (explicit scope cuts)

  • NOT shipping a source-eval / author-trust YAML registry in v1 — because (a) the load-bearing quality comes from prompt-level curator-lens framing per the brief's hypothesis, and (b) maintaining a registry has ongoing cost that's only justified if v1 proves the framing alone is insufficient. Deferred to /source-eval as a v2 sibling skill.
  • NOT supporting --depth shallow or --depth deep in v1 — because three modes triples the testing surface and the brief's premortem flags this as the most likely scope-creep failure. Ship --depth mid (~3000 words) only; promote to flag when there's real demand.
  • NOT including a diagram-pass, persona-pass, blind-primer phase, learnings-as-phase, or worked-examples phase — because /rewrite's full machinery is what we are explicitly NOT porting. Each is a v2+ backlog candidate gated on real usage signal.
  • NOT supporting topic-or-URL auto-detect, --sources BYO-sources, or topic-tree recursion in v1 — because each adds an intake branch that doubles edge-case surface. Single shape: /primer <topic>.
  • NOT validating the A1 curator-lens hypothesis pre-ship — because the user has explicitly chosen to defer the side-by-side spike until after v1 exists. This is a logged accepted risk (see Open Questions OQ-1).
  • NOT staging an A1 mid-build kill criterion — explicitly chosen during /grill. v1 ships end-to-end on the unvalidated curator-lens hypothesis with no hand-prototype gate in /spec, /plan, or /execute. Implication: roughly two weeks of build effort commits before any quality signal. v2 scope responds to post-ship readership / share signal and the post-v1 A1 retrospective; if A1 disconfirms in practice, v2 ports a minimal source-eval reference doc as a sibling /source-eval skill (per the brief's idea variant #13). This Non-Goal makes the deliberate "no kill criterion" stance explicit so /spec and downstream skills do not silently introduce one.
  • NOT including pinned-tag resume semantics — because /primer is a single-session skill (no multi-day work); resume after compact is handled by Claude Code's harness, not by /primer's own state file.
  • NOT shipping flashcard / quiz / retrieval-practice outputs — because those are sibling skills (/flashcards, /quiz) that consume /primer's artifact. Goal G5 future-proofs the consumption contract.

User Experience Analysis

Motivation

  • Job to be done: Ramp on a PM topic the user does not yet own, fast enough to participate substantively in a conversation about it within the next hour or two.
  • Importance / urgency: High — the typical trigger is a same-day or next-day meeting / scope / review where bluffing is costly and reading 5 long articles is too slow.
  • Alternatives: (a) ChatGPT / claude.ai direct prompt — no source verification, no persistence; (b) Reading 3–5 articles oneself — 60–90 minutes, no synthesis; (c) /rewrite in pmos-content — requires a known seed URL, not a topic name; (d) Asking a colleague — high latency, low repeatability. /primer beats (a) on rigor, (b) on speed, (c) on entry shape, (d) on availability.

Satisfaction Signals

  • The user reads the primer end-to-end on first run without skipping the gates ("if I trusted it, I'd skim; gating means I'm reading").
  • The user shares the primer URL or copied content externally (Slack / DM / Notion) within two weeks of generation.
  • The user runs /primer a second and third time on different topics without grimacing at cost or output quality.
  • The user requests a sibling pmos-learnkit skill (/flashcards <primer>, /quiz <primer>) — the "appetite" signal that /primer earned its place.

Solution Direction

Composite finalist from the ideate brief (★): a 5-phase, single-SKILL.md, substrate-reusing pipeline with no source-eval registry, interactive-by-default, --depth mid only, HTML-primary output.

Invocation shape

/primer <topic> [--audience <senior-pms|all-pms>] [--autonomous] [--format html|md|both]
  • Interactive-by-default; --autonomous inverts gates (matches the pmos-toolkit non-interactive contract).
  • If a required field is unspecified, a single consolidated AskUserQuestion at intake collects everything in one pass.
  • --depth, --sources, topic-or-URL auto-detect, topic-tree recursion are NOT v1.

Five-phase pipeline (user-observable behaviour)

  1. Phase 1 — Intake. Resolve topic, audience, output_format from .pmos/settings.yaml + CLI flags. Single consolidated AskUserQuestion for any missing field. Derive slug + canonical artifact path {docs_path}/primer/{YYYY-MM-DD}_<slug>.html. Echo decisions back; one-touch confirm.
  2. Phase 2 — Research. WebFetch on 6–10 candidate sources for the topic (NOT WebSearch alone). For each, verify the URL resolves and capture a 1-line takeaway. Persist sources.json sidecar. Prefer context7 for library/SDK-shaped topics. Source floor (per Q5 grill disposition): 4 verified sources. If Phase 2 yields fewer than 4 resolving sources, surface a gate before proceeding to Phase 3 — options: abort (Recommended) / proceed with thin-source disclosure banner in the artifact footer / retry research with an alternate query frame. No subagents in v1.
  3. Phase 3 — Outline. Render a 10–14 H2 section outline for --depth mid, biased by audience presets and the curator-lens reference doc. Interactive gate (default): user can edit / cut / reorder / re-prompt before drafting. --autonomous mode skips this gate and proceeds.
  4. Phase 4 — Draft. Single-pass drafting against the approved outline, inlining citations to sources.json URLs. ~3000 words for --depth mid. No diagram-pass, no persona-pass.
  5. Phase 5 — Eval. Reviewer subagent (per the _shared/ reviewer pattern) reads the draft against a ~10-check binary rubric, with checks tagged trust-tier or taste-tier (per Q4 grill disposition):
    • Trust-tier (hard-block on failure): (b) cites real URLs — verification authority per Q6: reviewer cross-checks every URL in the draft against sources.json; the draft must not introduce novel URLs not in sources.json. Reviewer does NOT re-fetch — Phase 2 owns URL resolution. (i) no plagiarism / direct copy from sources. (c) no hand-wavy claims (no orphan "should/might", every quantitative claim has a number, no LLM-confabulation patterns).
    • Taste-tier (accept-as-known-risk on failure): (a) covers ground, (d) right reading level for audience, (e) structurally complete, (f) sections.json IDs match on-page IDs, (g) audience-appropriate vocabulary per the three-axis preset, (h) flags uncertainty where appropriate, (j) primer-shaped not article-shaped.
    Auto-apply fixes; max 1 iteration. After iteration: if any trust-tier check still fails, hard-block the write and trigger the recovery flow (see Q7). If only taste-tier checks fail, surface as accepted residuals and proceed to write. Interactive gate (default): confirm before final write. --autonomous mode skips this gate (residuals still apply).

Audience preset shape (per Q3 grill disposition)

The audience preset must drive measurable, observable differences in the primer — not act as a one-line system-prompt-suffix slider. v1 commits to three concrete axes:

Axissenior-pmsall-pms
Section selectionIncludes "Tradeoffs we considered against", "Metrics calibration / what to measure", and "Open debates in this area" as required sections.Includes "Common pitfalls", "What to do on day 1", and "Questions to ask your team" as required sections. The deeper-cut sections are optional.
Background depthTerms of art used directly — assumes the reader has built or scoped 2+ products in adjacent space. No inline definitions for common PM concepts.First use of any term of art carries a 1-line inline definition. Common PM concepts get a sentence of grounding before being used.
Closing shapeEnds with open debates, current arguments, what to monitor. The implicit audience already knows what to do; they need the frontier.Ends with concrete actions — checklist of "first 5 things to do when you land in this area", questions to ask, calibration heuristics.

/spec turns each axis into one or more reviewer-rubric checks (e.g., taste-tier check: "section list matches the audience preset's required-section list"). If a future audience preset is added in v2 (e.g., exec, new-to-pm), it must define values on these same three axes — the axes are the contract, not the preset names.

Output shape (user-observable)

  • Primary: {docs_path}/primer/{YYYY-MM-DD}_<slug>.html rendered through the _shared/html-authoring substrate (template.html, style.css, viewer.js, copy-MD button, addressable section IDs).
  • Companion: sections.json (per substrate convention — stable IDs, on-page anchors).
  • Companion: sources.json (URL + 1-line takeaway per source, for human spot-check and future /source-eval consumption).
  • Optional sidecar: {slug}.md via output_format=both (read-only, generated through html-to-md.js).
  • Asset substrate: assets/* copied from _shared/html-authoring/assets/ on first write to {docs_path}/primer/.
  • Attribution: footer reads "Created using pmos-learnkit" (NOT pmos-toolkit) — requires a small substrate parameterization or a writer override (see OQ-3).
  • Polish-compatible: /polish {primer-path} works without modification.
  • Hard-block recovery artifact (per Q7 grill disposition): when the reviewer hard-blocks the write on a trust-tier failure, /primer does NOT discard work. Phase 2's sources.json is persisted to the canonical path. The Phase 4 draft is written to {docs_path}/primer/{date}_{slug}.draft.html with: (a) a prominent banner reading "REJECTED BY REVIEWER — DO NOT TRUST" at the top of the body, (b) a "Failing checks" section listing each trust-tier check that failed with the reviewer's verdict text, (c) the draft body so the user can read what would have shipped. The user can then decide: manually patch the draft (re-run reviewer on the patched draft via --from-draft v2 flag, NOT v1 scope), rerun /primer fresh (sources.json may be reused as cache — v2 optimization), or discard. v1 v1 ships only the persisted artifact + manual patch path; --from-draft is a v2 backlog item.

Plugin and substrate strategy

  • Plugin home: plugins/pmos-learnkit/ in the same pmos-skills marketplace repo as pmos-toolkit.
  • Marketplace registration: both .claude-plugin/marketplace.json and .codex-plugin/marketplace.json get a new entry for pmos-learnkit at version 0.1.0. Pre-commit drift hook enforces the four-file version sync per CLAUDE.md ## Plugin manifest version sync.
  • Manifests: plugins/pmos-learnkit/.claude-plugin/plugin.json + plugins/pmos-learnkit/.codex-plugin/plugin.json at 0.1.0, mirrored content per the existing pmos-toolkit pair.
  • _shared/ strategy: pmos-learnkit carries its own plugins/pmos-learnkit/skills/_shared/ directory, kept in step with pmos-toolkit's by scripts/sync-shared.sh --from=pmos-toolkit. This is the existing convention (confirmed by code inspection — sync-shared.sh exists, supports multi-peer rsync, and is the only sanctioned mutation path per CLAUDE.md ## Release policy / Drift hook contract). Cross-plugin references are NOT supported by the plugin loader.
  • Settings: /primer reads the shared .pmos/settings.yaml at the repo root — docs_path, output_format, default_mode. No learnkit-scoped sub-key.
  • Repeat-run memory file: .pmos/primer.lastrun.yaml stores the user's last-used audience, output_format, and topic-shape hints to seed the Phase 0.5 consolidated-confirm prompt on the next run. This file is per-developer and gitignored (matching the .pmos/complete-dev.lastrun.yaml pattern introduced in pmos-toolkit v2.41.0). Not a pipeline state file — purely an interactive-UX optimization. v1's .gitignore must include .pmos/*.lastrun.yaml (or the wildcard equivalent) if not already present.

User Journeys

Primary Journey (Happy Path) — interactive, mid depth, default audience

  1. User runs /primer "feature flagging at scale" in Claude Code.
  2. System prints "Using primer to research and draft a teachable artifact on 'feature flagging at scale'." Phase 1 intake: detects audience and output_format unspecified, fires a single consolidated AskUserQuestion asking for audience (default senior-pms) and output_format (default from settings.yaml). User accepts defaults.
  3. System echoes resolved decisions and the canonical artifact path docs/pmos/primer/2026-05-23_feature-flagging-at-scale.html. One-touch confirm.
  4. Phase 2 research: system runs WebFetch on 6–10 candidate URLs for the topic, prints each as "Fetched: <url> — <takeaway>" or "Skipped: <url> — <reason>". sources.json sidecar populated.
  5. Phase 3 outline: system renders a 12-section outline biased to senior-PM audience. Prints the outline inline. Interactive gate: "Approve outline / Edit / Re-prompt / Abort." User picks Approve.
  6. Phase 4 draft: system drafts ~3000 words inline against the approved outline, citing sources.json URLs.
  7. Phase 5 eval: reviewer subagent reads the draft against the 10-check rubric. Returns a verdict per check. Any failures trigger an auto-apply fix iteration (max 1). System prints the rubric verdict table inline. Interactive gate: "Approve and write / Iterate manually / Abort." User picks Approve and write.
  8. System atomically writes 2026-05-23_feature-flagging-at-scale.html + sections.json + sources.json via temp-then-rename. Asset substrate copied from _shared/html-authoring/assets/ to docs/pmos/primer/assets/. Footer reads "Created using pmos-learnkit".
  9. System prints the final artifact path and a one-line summary. Total elapsed time displayed.

Alternate Journey — autonomous mode

  1. User runs /primer "PLG metrics" --audience all-pms --autonomous.
  2. System echoes the full intake decisions in one line; no intake AskUserQuestion (all fields supplied).
  3. Phases 2–4 run sequentially without gates. Phase 5 eval reviewer runs; if it returns no failures, system writes the artifact and exits. If it returns failures, auto-apply runs once; if failures remain, system writes the artifact AND surfaces a "non-blocking residuals" note in the chat summary (mirrors the /complete-dev "known risk" pattern).
  4. Final artifact path printed; total elapsed time displayed.

Alternate Journey — repeat run with lastrun memory

  1. User has previously run /primer with --audience senior-pms.
  2. User runs /primer "OKRs vs KPIs" with no flags.
  3. Phase 0.5 consolidated confirm (per /complete-dev v2.41.0 pattern) reads .pmos/primer.lastrun.yaml and surfaces the prior audience as the default in one AskUserQuestion. User one-touch confirms.
  4. Pipeline proceeds as in the primary journey from Phase 2 onward.

Error Journeys

  1. Topic too vague. User runs /primer "growth". Intake heuristic detects the topic is < 3 tokens and ambiguous. System surfaces an AskUserQuestion with 3 candidate refinements ("growth loops", "PLG growth", "B2B sales-led growth") + an "Other" escape. User picks one or aborts.
  2. WebFetch returns junk / paywalled / fabricated sources — source floor triggered. Phase 2 attempts 10 URLs, 7 return non-content (paywall, 404, redirect to home). Floor = 4 verified sources is breached. System surfaces a Phase 2 gate: "Only 3 verified sources found (floor = 4). Abort (Recommended) / Proceed with thin-source disclosure (artifact gets a 'thin source base' footer banner) / Retry research with alternate query frame." If user retries, system tries an alternate frame. If user aborts, no artifact is written but sources.json is persisted to a discoverable location (e.g., .pmos/primer-cache/{date}_{slug}.sources.json) for diagnostic.
  3. Reviewer subagent fails the rubric after auto-apply iteration — trust-tier failure. Phase 5 eval iteration 1 still has a failing trust-tier check (e.g. "cites real URLs" or "no plagiarism"). System hard-blocks the write, persists sources.json, writes the draft to {date}_{slug}.draft.html with the REJECTED banner + failing-checks list (per the recovery contract). User sees: "Trust-tier check '<name>' failed after auto-apply iteration. Hard-block: no primer artifact written. Draft preserved at <path>.draft.html for review. Manually patch and rerun, or rerun fresh."
  4. Reviewer subagent fails the rubric — taste-tier only. Iteration 1 leaves only taste-tier failures (e.g. "right reading level", "primer-shaped not article-shaped"). System surfaces: "N taste-tier checks failing. Accept as known risk and write (Recommended) / Iterate manually / Abort." On Accept, the artifact is written with a "Known residuals" footer note listing the residual checks. Mirrors /skill-eval's residual pattern for non-trust-dimension checks.
  5. Artifact path collision. User runs /primer twice on the same topic same day. System detects {date}_{slug}.html exists. Surfaces: "Overwrite (will git-snapshot first) / Append -2 suffix (Recommended) / Abort."
  6. WebFetch unavailable / network down. Phase 2 cannot fetch any source. System surfaces: "WebFetch unavailable. Abort / Continue with no sources (NOT recommended)." If user continues, the artifact is marked clearly as "uncited draft" in the footer and reviewer rubric flags it.
  7. Topic is library-shaped (e.g. "tRPC", "Pandas"). Intake heuristic detects library-shape (single proper-noun token, known package). System routes Phase 2 research preferentially through context7 + WebFetch (not WebFetch alone).
  8. --audience value unknown. User passes --audience eng-leads (not in v1 set). System rejects with a clear error listing the v1 set + "use --audience all-pms as the broadest preset."

(b) ★ Parameterize attribution per writer-plugin.Plugin-level provenance is meaningful for users browsing artifacts. Mechanism (substrate parameter vs writer override) is a /spec concern; OQ-3 captures the unresolved mechanism choice.
D11Reviewer subagent rubric is binary with ~10 checks; auto-apply max 1 iteration; checks split into trust-tier (hard-block on residual) and taste-tier (accept-as-known-risk on residual).(a) Scalar 0–10 score per check. (b) Uniform binary; all residuals accept-as-known-risk (matches /skill-eval). (c) ★ Binary per check; trust/taste split; trust residuals hard-block, taste residuals accept.Per Q4 grill disposition. A primer is a teaching artifact the user reads and trusts; shipping with a residual "cites real URLs" failure (trust dimension) would invalidate the v1 differentiation claim. /skill-eval's uniform accept-as-known-risk pattern is wrong for this artifact category. Trust = cites real URLs / no plagiarism / no hand-wavy claims. Taste = reading level / structure / vocab. The exact 10 checks + their tier assignments are a /spec concern (OQ-2).
D12Single-session skill; no own state file. Compact resume relies on lastrun.yaml + harness.(a) Pinned-tag resume (matches /rewrite). (b) ★ Single-session; lastrun.yaml for repeat-run defaults only./rewrite's pinned-tag resume is for multi-day work; /primer is a single-session ramp-up (target <6 min). The harness's compact-then-resume + /primer's idempotent intake (driven by lastrun.yaml in Phase 0.5) covers the realistic recovery case.
D13WebFetch (not WebSearch alone) for source acquisition; context7 preferred for library/SDK topics.(a) WebSearch + WebFetch combo. (b) ★ WebFetch with URL-resolution verify; context7 for library-shaped topics.The brief's premortem #2 (research hallucinates citations) is High likelihood without a registry. WebFetch forces verification; WebSearch alone returns titles that may be hallucinated. context7 is the right source for library docs (per its own MCP instruction).
D14Phase 2 source floor = 4 verified resolving sources. Below the floor, surface a gate.(a) No floor; let the reviewer rubric absorb thin sourcing. (b) Floor = 3 (matches the original error-journey wording). (c) ★ Floor = 4; abort by default, override with thin-source disclosure or alt-query retry.Per Q5 grill disposition. The v1 differentiation claim is "verified-source primer"; below 4 sources the claim doesn't hold and the artifact becomes barely better than a Wikipedia summary. 4 is the tightest floor that doesn't over-exclude topics with genuinely sparse high-quality coverage (some niche PM topics have only ~5 good sources). Below 4: hard gate (abort / disclosure / retry) rather than silent thin-source ship.
D15Reviewer URL verification authority = cross-check against sources.json (NOT independent re-fetch, NOT just well-formedness).(a) Reviewer independently re-fetches every URL. (b) Reviewer checks URL well-formedness only. (c) ★ Reviewer cross-checks: every URL in draft must appear in sources.json; no novel URLs introduced in drafting.Per Q6 grill disposition. Phase 2 already owns URL resolution and writes sources.json — that's where verification happens. Reviewer's job is consistency-check: the draft cannot invent URLs. (a) doubles network cost and adds drift risk if Phase 2's verification was correct. (b) is theater. Making this explicit at the requirements level prevents /spec from defaulting to (b) by oversight.
D16Hard-block recovery = persist sources.json + {slug}.draft.html with REJECTED banner + failing-checks list.(a) Discard all work on hard-block (cheap, lose information). (b) Persist sources.json only; discard draft (medium). (c) ★ Persist sources.json + draft with rejection banner + failing-checks list. (d) Add a --override flag that downgrades hard-block to known-risk for one run (v2 backlog).Per Q7 grill disposition. Phase 2–4 cost 4–5 minutes; discarding on a single trust-tier residual is excessive. The draft.html with prominent rejection banner lets the user read what would have shipped, see exactly which check failed, and decide patch / rerun / abandon. Matches git --no-verify's "user is informed, can choose" pattern. v2 backlog: --from-draft rerun flag and --override per-run escape hatch.
