/landing-page enhancements

Epic 260626-7s4 · route: skill · pmos-toolkit · defined 2026-06-26

This epic revises the existing /landing-page skill in response to grounded feedback from a real run (the poker-coach landing page). The skill is sound in its bones — a six-phase, reference-grounded, self-contained-HTML generator — but the run exposed structural gaps in what it shows, who it speaks to, how it renders choices, and how assets and media are handled. This document is the cross-cutting design contract the two stories cite by anchor; it does not restate the skill body or its reference files (one fact, one home).

Skill under revision: plugins/pmos-toolkit/skills/landing-page/SKILL.md + reference/{section-scaffolds,hero-archetypes,copy-gates,style-tokens}.*. Original design lineage: docs/pmos/features/2026-06-24_landing-page/02_design.html.

2. Decisions

D1 — do>show>tell is a governing principle, not a single gate

Elevate do > show > tell to a stated principle alongside the existing governing equation (Purchase Rate = Desire − (Labor + Confusion)). The brief gains a "signature moments to demonstrate" field (the 2–4 product moments worth showing — e.g. pot-equity readout, coaching card, opponent-type picker). Phase 2 maps each to a show-surface (screenshot / annotated shot / carousel / video / interactive snippet). Phase 6 adds a show-ratio check: flag sections that tell a benefit a captured asset could show. This is advisory (judgment), not a hard arithmetic gate.

D2 — Persona narrowing to 1–2 personas, with jargon calibration

The brief's loose "target audience" becomes an explicit 1–2 persona selection (interactive pick; non-interactive picks the product-type default and logs an OQ). Each persona carries a jargon tolerance (novice / fluent). When any selected persona is novice, the copy gates reject undefined domain jargon (equity, GTO) unless inline-defined or replaced with plain language. Personas live in the brief; the rule lives in copy-gates.md.

D3 — New section taxonomy rows + dedup pass

Add to section-scaffolds.md: a "Who this is for / not for" row (qualifies the visitor, removes labor/confusion) and an explicit note that the below-hero proof strip is omitted when the hero caption already carries those proof values (no double-counting). Phase 2 runs a coherence pass: no section restates the hero's value prop or duplicates proof.

D4 — ≥3 structure variants at Phase 2

Phase 2 stops proposing a single ordered list. It proposes ≥3 distinct variants — varying section order, framing (problem-led vs outcome-led vs proof-led), and copy length — each one-line-summarised, with a recommended default. The gate becomes a pick among variants (AUTO-PICK recommended non-interactively). Reuses the Phase-3 rendered-comparison muscle where cheap.

D5 — Logo phase (new Phase 1.5), optional

After the brief, detect an existing logo/brand mark in the researched assets. If present → bind it. If absent → offer to run /logo (gated; recommended = "use a simple wordmark", so it AUTO-PICKs to a no-op-safe default non-interactively rather than spawning a heavy skill unattended). /logo output (SVG + favicon + monochrome) is bound in Phase 5 and the footer.

D6 — Media-strategy gate (new Phase 4.5)

A new gate asks the show-format per signature moment / for the page overall: static device-framed image · carousel · video. Video path uses Playwright's recordVideo context option to capture a real product flow, then ffmpeg to trim / compress / generate a poster + (optionally) a muted autoplay-loop <video> or animated WebP. Graceful degrade (no ffmpeg / no headless browser): video → animated GIF/WebP if Playwright present, else a captioned static shot + logged skip. Recommended default = static device-framed (cheapest, always works), so non-interactive runs never block on capture. The capability detection + degrade ladder lives in a new reference/media-strategy.md.

D7 — Asset fidelity + mobile as a hard visual dimension

copy-gates.md#visual-self-check (and the draft rules) gain: preserve native aspect ratio (capture/store intrinsic dimensions; render with object-fit: contain/cover inside fixed frames — never stretch), device frames for product shots, and mobile-appropriate assets (portrait crop / different image where a wide desktop shot is illegible on a phone). The visual self-check's mobile screenshot becomes a hard pass dimension: layout must not overflow, CTA must stay visible, images must not skew or clip. Mobile failure blocks emit (within the existing ≤2-iteration fix loop) rather than being informational.

D8 — Live style preview at Phase 4

Phase 4 no longer just points at the static style-gallery.html. It renders the chosen hero (from Phase 3) in 2–3 candidate styles as a side-by-side comparison artifact (working/style-options.html), the same "show, don't ask" mechanism Phase 3 uses for heroes. The gallery stays as the full reference; the preview is the decision surface.

D9 — Multi-product organizing principle

Phase 1 detects whether the input is a single product or a multi-product repo (monorepo with N user-facing apps; a plugin with N user-facing skills). When multi-product, Phase 1 proposes an organizing principle and the user picks: (a) one page, one section per product (a suite/overview page), (b) a product-index/hub linking to per-product pages, or (c) focus on one product (pick which). Each chosen product gets its own brief slice (persona, signature moments, sections). The detection heuristic + the three organizing principles live in a new reference/multi-product.md. Non-interactive default = focus on the single most-prominent product, OQ logged.

D10 — Shared design-authoring substrate

Adopt the house-style / responsive guidance the other visual skills honour. Phase 5 + Phase 6 cite the host repo's DESIGN.md (when present) and the shared responsive/contrast norms used by /wireframes · /prototype · /design-crit, rather than re-deriving layout rules. Concretely: read DESIGN.md if present and bias style/token choice + spacing toward it; cite the shared responsive checklist in the visual self-check. (Spec resolves exactly which shared file/section is the canonical citation.)

D11 — Footer attribution

Every emitted index.html (and per-product page) bakes a small footer line: "Built with pmos-toolkit" linking to the repo ({{repo_url}} convention, per the repo's existing wordmark/footer token work). Additive to the existing footer section; does not displace product trust/legal links.

3. Constraints (single-skill invariants)

  • C1 — Self-contained output is preserved. Every fix must keep the page a single file://-openable HTML with inline CSS + vanilla JS and no CDN (D3 of the original design). Video/carousel assets are embedded (data: URI) or stored inside the page folder and referenced relatively — never a remote <script src> or hotlink.
  • C2 — One fact, one home. New rules land in reference files (media-strategy.md, multi-product.md, and edits to section-scaffolds.md / hero-archetypes.md / copy-gates.md); the SKILL.md body cites them. No rule is duplicated in the body.
  • C3 — No fabricated proof. The do>show>tell push must not invent demo content. "Show" assets come from the real product (captured screenshots/recordings) or are labelled placeholders (original D6). A video of "an actual hand + feedback" is captured from the running product, not mocked.
  • C4 — Gates stay non-interactive-safe. Every new gate (media-strategy 4.5, logo 1.5, structure-variants, multi-product, persona) carries a (Recommended) option that AUTO-PICKs a no-heavy-op-safe default under --non-interactive, per the W14 contract; new heavy ops (Playwright recording, /logo spawn) never run unattended without an explicit recommended-safe default.
  • C5 — skill-patterns §A–§L + skill-eval. The revised skill must still pass reference/skill-patterns.md §A–§L and the binary skill-eval.md rubric — argument-hint vs body flags lint, phase-ref/slug lint, non-interactive inline block, audit-recommended.
