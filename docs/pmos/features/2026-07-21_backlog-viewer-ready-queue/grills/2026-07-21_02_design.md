## Resolved

- **Field shape → `queues.next.queue`** (sibling of `pick`), not a top-level
  `queues.ready_queue`. Nesting makes `pick === queue[0]` structural rather than
  documentary; a peer-level array reads as a second, independent derivation of
  readiness — the drift §K exists to prevent. **This contradicts the capture's
  written AC**, which named `queues.ready_queue`; the AC is amended in the epic
  record and the supersession is noted in the design's Open section. → D1

- **Fallback scope → both dashboards list the queue.** The inline `#dashboard`
  render (SKILL.md step 3) is the same view without a browser, and it is the path
  every `--non-interactive` define/build tick takes. Leaving it single-pick would
  make the two renderings of one dashboard disagree, with the headless one — the
  one unattended drivers read — being the degraded copy. Accepted cost: a few extra
  lines per tick. `#next` stays untouched (it answers a different question). → D7, FR-8

- **Stale-claimed ready rows → shown, chipped `stale claim`.** Readiness rule and
  D22 order untouched. → D10, FR-10

- **Lane density → no cap, every row keeps its copy-ready command.** The head row
  carries `/feature-sdlc build --next`, the rest `/feature-sdlc build --story <id>`.
  The entire value of showing the queue is that row 2 is buildable in parallel *now*;
  a row you cannot copy-paste restates the problem instead of solving it. A cap would
  be the defect wearing a smaller hat. Height is the accepted cost. → D3, D5

## Gaps surfaced

- **Stale-claim rows occupy two lanes.** The readiness filter admits
  `has_lock && stale` stories, so such a story is simultaneously a build candidate
  and a Needs-you → Stale claims row. Invisible today because only the pick renders;
  uncapped it becomes a visible contradiction. Not anticipated by the capture.
  → new D10 + FR-10.

- **`pick` was to be computed a second time off `candidates`.** Two reads of the same
  sorted array is exactly how INV-2 would eventually drift. Collapsed to one
  expression: `const queue = candidates.map(c => c.id); const next = { queue, pick: queue[0] ?? null };`
  → D9, making INV-2 structural rather than test-only.

- **Test fixture already encodes the ordering case.** `tests/serve-web.test.mjs`
  already sets up S2 (should) and S6 (could) as the two candidates and asserts
  `pick === 'S2'`. The new ordering assertion is `queue === ['S2','S6']` against that
  same fixture — no new fixture work. → FR-9.

## Confirmed, not re-litigated

- Read-only viewer (INV-5) — no claim/build action is added to the new rows.
