/playbook → evolution-only mode

Source: docs/pmos/playbooks/SKILL-FEEDBACK-for-playbook-author.md (real run on poker-coach, plugin v0.27.0) + three maintainer guideline overrides. Worked target: poker-coach/docs/pmos/playbooks/2026-06-17_poker-coach-evolution/index.html.

Summary

/playbook is rebuilt around a single idea: trace how a product (or one skill) evolved, and how the AI-SDLC pipeline shaped the decisions along the way — structured by inflection points, not a release log. The per-problem case-study mode (FR-50/51, clustering, ranked propose/pick) is removed entirely to keep the skill simple. Evolution becomes the default and only mode; there is no subcommand for the common case.

Three maintainer overrides drive the shape and win over the source feedback wherever they conflict (the feedback proposed "keep both modes"; the maintainer chose evolution-only):

  1. Evolution is the default, always. In a Claude Code / Codex skills-marketplace repo, prompt the user to choose a target (a specific skill or the whole repo). Otherwise, target the repo's own evolution with no prompt.
  2. Case-study mode is removed. Simpler, easier to understand.
  3. No day/session window. Mine everything related to the repo/skill; the milestone spine comes from committed artifacts, not a user-supplied window.

Context — the run that prompted this

A real /playbook run produced two accurate per-problem case studies, but the maintainer's verdict was they "will not be useful for PMs": a single problem thread's takeaway is entangled with the domain (poker), so little transfers. The reusable asset is the evolution arc and the pipeline's influence on it. The maintainer discarded the case studies and hand-built a project-evolution playbook (the worked target above) — milestones anchored on inflection points, a "Where the pipeline mattered" callout per milestone, a cross-cutting process section, a plain title, a cold-reader intro, and a subtle close. This epic encodes that worked example into the skill.

The mode model

Target resolution (Phase: resolve-target)

After setup, /playbook resolves a single target ∈ {repo, skill:<name>}:

  1. Explicit: --skill <name> (or NL "the <name> skill") → skill:<name>. --repo <path> still sets which repo; it does not pick skill-vs-repo.
  2. Marketplace detection (deterministic): the resolved repo is a skills marketplace iff any of — a .claude-plugin/marketplace.json, a .codex-plugin/marketplace.json, or ≥1 plugins/*/skills/*/SKILL.md — is present.
  3. Marketplace + no explicit target: present an AskUserQuestion listing each installed skill plus "The whole repo / a plugin (Recommended)". The user picks one. (Non-interactive → Recommended = whole repo, logged as OQ.)
  4. Not a marketplace: target = repo, no prompt.
The prompt's Recommended option is whole repo, so the non-interactive auto-pick and the interactive default agree — a marketplace run with no --skill evolves the whole repo unless the user chooses a skill.

Mining strategy — mine everything, two sources

No window. The skill inventories the full committed history and the full attributed session set, then anchors the deep read on the milestone spine:

SourceAuthority forRead at
changelog.md (or repo changelog)milestone spine + named per-version decisionsscout (cheap)
features/<f>/ — 01_requirements, grills/*, 02_spec (D/S/P ids), verify/*problem framing, the decisions that changed direction, what verification caughtscout (inventory) → deep-read (per picked milestone)
git logordering + in-flight/unreleased work ("where it's headed")scout (cheap)
session logs (via the resolver)the raw, un-synthesized author inputs the docs drop — verbatim opening prompt, @-files / pasted context, original phrasing, inflection quotesdeep-read, scoped to each milestone's session window

Net: docs say what was decided; sessions say what the author originally said and fed in. An evolution article needs both — the cost saving from D4 is in not clustering the whole repo's sessions, not in skipping the session read. The split lives in reference/evolution-sources.md.

Skill-scoped (D3): when target = skill:<name>, the spine is filtered to feature folders and changelog entries whose changes touch that skill's files, and the session window is filtered to sessions that invoked or edited that skill.

Scout rewrite

scripts/scout.mjs is rewritten from a session-clustering ranker into a milestone-spine builder:

  1. Inventory changelog + features/* → an ordered list of milestones (version/feature, named decisions, doc paths). For skill mode, filter to the target skill.
  2. Run the retained resolver to attribute all sessions to the repo (across worktrees / merged-deleted), printing the coverage line.
  3. Map each milestone → its originating session(s) by feature-folder slug / branch / time window, so the deep read can recover the verbatim opening prompt per milestone.
  4. Emit a milestone-spine JSON (no scoring, no boundary_confidence, no merge_suggestion). The cheap-scout invariant holds: no raw session bodies are read at scout time — only changelog/feature text + cheap session fields.

The deterministic session-log-format.md field contract and the resolver's three signals are unchanged.

Evolution article schema

Replaces the per-problem section list in article-schema.md. Section order:

  1. "What this is" — cold-reader context: what the product/repo (or skill) is and what the document covers. Mandatory first section, before any takeaway or claim.
  2. One section per milestone, versioned/named (e.g. v0.3 — Playing it exposed the real product), each:
    • anchored on its 1–2 inflection decisions (not every change);
    • carrying the author's verbatim opening prompt / context input as short quoted blocks inside the milestone narrative (not a separate gallery);
    • carrying a "Where the pipeline mattered" callout naming the specific pmos skill (/grill, /spec, /verify, …) and what it changed.
  3. "How the pipeline shaped the whole arc" — cross-cutting reflection on the pipeline's influence.
  4. A short, understated takeaway — a reflective close, not a bulleted billboard.

The self-sufficiency rule (reader has no repo access → embed excerpts/screenshots inline) carries over.

Skill-eval & release prerequisites

The rewritten skill must conform to plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§L and the host CLAUDE.md conventions (canonical skill path, manifest version-sync, release entry point). The non-interactive inline block must stay byte-identical to skills/_shared/non-interactive.md; every AskUserQuestion needs a (Recommended) option or a defer-only tag. Version bump, changelog, README row, and manifest sync are release prerequisites for /complete-dev — not /execute tasks.

Out of scope / non-goals

  • Per-problem case-study mode (removed) and its clustering/ranking.
  • Day/session windowing.
  • Multi-repo "how I use AI" generalization.
  • Auto-posting or auto-marking content safe (the share gate is inviolable).
  • Markdown/Substack/LinkedIn export variants (still deferred).

Story split

This epic touches exactly one skill, and every change is interdependent (the SKILL.md phases reference the new scout contract and the new schema; the scout rewrite and the schema rewrite are one vertical slice). Splitting would create cross-story task dependencies → fails the D24 litmus. Singleton skill epic = one story (260617-evo), carrying the full rewrite, scored once against skill-eval.md at build.
