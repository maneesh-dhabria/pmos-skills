html-authoring typography & layout refresh — "Editorial Technical"

Bottom line

Every pmos pipeline doc renders through one shared stylesheet. Today that stylesheet has no real typographic hierarchy, too wide a measure, and stark structural devices. This epic replaces it with a serif reading body, a sans heading voice, and a mono structural layer — a calmer, more credible "editorial technical" system that keeps the pmos identity (mono wordmark, burnt-orange accent, [NN] section counters) but is distinctly ours.

Why — the reading experience is the product surface

pmos artifacts are how a stakeholder meets the work: a spec, a design doc, a TL;DR. The current substrate (_shared/html-authoring/assets/style.css) renders all text in a 12–15px band — h2 is 15px, one pixel above body; h2 and h3 are the same size; the doc title lives at 12px in the chrome. The column is 880px wide (~115 chars/line, vs an ideal ~66). Tables and code sit at 12px and read dense. The result is technically correct but flat — it doesn't invite reading.

The user supplied a reference aesthetic they admire (a serif, warm-canvas teaching page) and asked us to evolve our own styling to reflect it — not copy it. This design takes the reading win (serif body, generous line-height, a narrow measure) and makes the structural system unmistakably pmos.

The proposed type system — three voices

One reading face, one heading face, one structural face. The contrast between them is what makes the page read as a considered editorial-technical document rather than either a plain web page or a clone of the reference.

VoiceFaceUsed for
BodySystem serif (Iowan / Palatino / Charter / Georgia)Paragraphs, lists, the lead — the reading experience.
HeadingSystem sans (grotesque)h1 / h2 / h3 — crisp editorial contrast against the serif body.
StructureJetBrains Mono (existing pmos display face)Section numbers, eyebrow/kicker, labels, table headers, wordmark, code.
Zero bundle cost

Body and heading faces are system stacks — nothing downloaded. Only the existing JetBrains Mono display face (already bundled) is non-system, and it is confined to the structural layer. So the refresh adds richer hierarchy with no new network weight.

Design decisions

D1–D7 carry the new aesthetic; D8–D12 fold in the requirements of the superseded epic 260613-ev1 so nothing it specified is lost.

IDDecision
D1Serif reading body at 17px / line-height 1.66 (was 14px sans / 1.55).
D2Sans headings — h1 34px / h2 21px / h3 17px, sentence case (not the reference's all-serif headings).
D3Mono section number above the title (01, 02…), accent-coloured — the [NN] counter is retained and kept on top of the heading, not hung in the margin.
D4Labelled lead — a mono "Bottom line" chip + plain serif lead, replacing the reference's italic-left-rule "mission" treatment.
D5Centered doc, full-bleed chrome — the toolbar/footer bar spans the viewport with centered inner content; the article is centered at the reading measure. No reserved side column.
D6Tables & code break out to the right only — their left edge stays aligned with the prose so the eye keeps a single left margin; they extend rightward to --wide (880px).
D7Inline comments, no white panel — commented text is annotated with a warm tint a shade darker than the canvas + a thin accent underline; the thread renders inline on the same paper. No floating white overlay toolbar. Long threads collapse to a one-line summary that expands on click so they don't push the doc around.
D8Real body <h1> (from ev1 D3) — template.html demotes the toolbar title to a non-heading breadcrumb and emits exactly one <h1> as the first child of the article body; render.js unchanged.
D9AA contrast (from ev1) — no body text uses --pmos-faint (≈2.4:1); text uses --pmos-muted (≥4.5:1). Faint is decoration-only (rules, list markers).
D10Calmer tables (from ev1 D2) — zebra retained, bottom borders only, no internal vertical rules; mono uppercase header row.
D11Dark-mode parity (from ev1) — the warm palette gets a prefers-color-scheme: dark counterpart; all new tokens have dark values.
D12Edit canonical, then sync (from ev1 D4) — change plugins/pmos-toolkit/.../html-authoring/, then scripts/sync-shared.sh --from=pmos-toolkit to pmos-learnkit; each plugin ships on its next release.

Functional requirements

  1. FR-1 — Tokens & faces. Add --pmos-font-serif (system serif stack), --pmos-font-sans (system grotesque), keep --pmos-font-display (mono). Add the new type-scale + warm-palette tokens; preserve token names already consumed elsewhere. Inline code is calm — soft warm-grey on a light tint (#6f5848 on #f1ece2), not the loud accent-strong colour (which read jarring).
  2. FR-2 — Type scale. Body serif 17px / lh 1.66; h1 sans 34px; h2 sans 21px (sentence case, text-transform:none, top rule, mono [NN] above); h3 sans 17px/600 — a perceptible step at each level.
  3. FR-3 — Measure & layout. Article body centered at ~730px (60–75ch); full-bleed toolbar/footer with centered inner content; tables/pre break out to ~880px to the right only (left edge aligned with prose).
  4. FR-4 — Real body H1. template.html emits one <h1> as the body's first child from {{title}}; toolbar title becomes a breadcrumb; render.js untouched; exactly one <h1> per doc.
  5. FR-5 — Lead & eyebrow. Mono uppercase eyebrow/kicker; a labelled lead (mono chip + serif lead). No italic-left-rule mission block.
  6. FR-6 — Tables. 15px; zebra retained; bottom borders only; mono uppercase header; heavier header bottom border.
  7. FR-7 — Contrast. Audit every color: var(--pmos-faint); text contexts switch to --pmos-muted; decoration may keep faint. All body text ≥4.5:1.
  8. FR-8 — Comments (inline). Update the comments overlay styling (comments.css) to the inline, warm-tinted treatment of D7 — annotated text + inline thread on the same paper, no white panel; long threads collapse/expand.
  9. FR-9 — Dark mode. A prefers-color-scheme: dark palette for every new token; verified to render correctly.
  10. FR-10 — Cross-plugin sync. After the canonical edit, sync-shared.sh --from=pmos-toolkit makes pmos-learnkit's copy byte-identical.
  11. FR-11 — No regressions. build_sections_json.js derives correct sections for a doc with a body h1 (h1 excluded); chrome-strip.js unaffected; h2::before counter sequence intact.

Acceptance criteria

  • Rendered scale: body serif 17px, h1 34px sans, h2 21px sans (none uppercase), h3 17px — perceptible step at each level (measured by getComputedStyle).
  • Article column ≈730px; prose line length 60–75ch; tables/pre reach ~880px and stay left-aligned with prose.
  • Exactly one body <h1> per emitted doc; toolbar title is a breadcrumb; render.js diff is empty.
  • No text node resolves to --pmos-faint; all body text ≥4.5:1 (AA).
  • Tables: zebra + bottom borders only; no vertical rules; mono uppercase header.
  • Comments render inline (warm-tinted annotation + inline thread, no white panel); long threads collapse/expand; opening a thread never reflows the document body.
  • Dark mode renders correctly for all new tokens.
  • pmos-learnkit copy byte-identical after sync (diff clean).
  • No regressions: build_sections_json (h1 excluded), chrome-strip, [NN] counter sequence.
  • Load-bearing live dogfood: re-render ≥1 representative pipeline doc and a comment-bearing doc through the substrate, serve over http, screenshot at 1440×900, assert the metrics above by getComputedStyle; independent blind-judge confirms hierarchy is skimmable, tables calmer, and the comment treatment reads cleanly; gaps → fix → re-run (cap 2, then accept-residuals-and-surface).

Supersedes — epic 260613-ev1

This epic supersedes 260613-ev1 ("Pipeline-doc CSS readability refresh") and its story 260613-h9r, both marked superseded. ev1 was a readability tune-up within the existing look; this epic is a fuller aesthetic evolution that subsumes it. Every ev1 requirement is folded in above: type scale (D1/D2), measure (D6/FR-3), real body H1 (D8/FR-4), calmer tables (D10/FR-6), AA contrast (D9/FR-7), dark mode (D11/FR-9), cross-plugin sync (D12/FR-10), no-regression constraints (FR-11), and the load-bearing live dogfood. ev1's --pmos-fs-md2 token is absorbed into the new full type scale.

Scope & files

Substrate-only, route:feature (CSS/template, not a SKILL.md — no skill-eval rubric).

  • plugins/pmos-toolkit/skills/_shared/html-authoring/assets/style.css — the type system, palette, layout, tables.
  • plugins/pmos-toolkit/skills/_shared/html-authoring/template.html — body <h1> + breadcrumb.
  • plugins/pmos-toolkit/skills/_shared/html-authoring/assets/comments.css (+ comments.js if needed) — inline comment treatment.
  • scripts/sync-shared.sh --from=pmos-toolkit — propagate to pmos-learnkit (byte-identical).
Ride which release?

A substrate change touches both plugins. Per the cross-plugin policy, pmos-toolkit and pmos-learnkit each ship the refreshed CSS on their next respective release at Loop 3 — no coupled release.
