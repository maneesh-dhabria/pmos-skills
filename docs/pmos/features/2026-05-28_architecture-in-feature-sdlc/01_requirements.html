Problem

The repo currently has /architecture as a code-side static-analysis lint and /feature-sdlc as the SDLC orchestrator, but the two never talk to each other. Spec-time design proceeds without an architectural lens; verify-time has no structural-regression check beyond what diff-level reviewers happen to notice. The result is that architectural issues surface post-merge during /reflect retros — or never — and accumulate as silent tech debt.

Who experiences this?

Repo maintainer driving /feature-sdlc on Tier-3 features in pmos-toolkit and adjacent multi-module repos. Single user today (Maneesh), but the contract generalises to any /feature-sdlc operator on a repo that has shipped an /architecture principles configuration.

Why now?

  • State schema v5 just shipped; the next additive bump (v6 or per-phase substructure) is the natural integration window.
  • /simulate-spec has proved out the pre-/plan validator pattern — there is a working precedent to mirror.
  • principles.yaml already ships in /architecture, so the marginal cost is mostly orchestration, not new rule-authoring.

Goals & Non-Goals

Goals

  • Surface architectural concerns at design time (Phase 4.5) for Tier-3 features — measured by user-retains-gate signal (operator does not auto-pass --skip after the first 3 Tier-3 runs) and by a false-positive rate below 20% on --from-spec findings.
  • Catch structural regressions at verify time (Phase 7) for Tier-2 and Tier-3 — measured by /architecture finding a delta in changed modules that /verify's existing reviewers miss.
  • Zero added cost on Tier-1 bug fixes — measured by no /architecture invocation appearing in T1 state.yaml runs.
  • Keep added latency at Phase 4.5 under 90s P50 on Tier-3 — measured by timing the gate on a representative sample run.

Non-Goals

  • NOT auditing the entire repo at Phase 7 — scoped to changed modules only (since merge-base) — because whole-repo audits produce noise at PR time.
  • NOT building a parallel rules engine — because principles.yaml + a single LLM judge is the chosen direction and dual-implementation is the explicit failure mode the brief eliminated.
  • NOT making Phase 4.5 hard at any tier — because pre-spec design is exploratory and false positives train users to skip.
  • NOT building --from-spec as a separate skill — because the existing /architecture skill owns "audit against principles" and --from-spec is a new input mode, not a new domain.

User Experience Analysis

Motivation

  • Job: surface architectural violations while they are still cheap to fix (at spec time vs. post-merge).
  • Importance / Urgency: medium — issues today are caught at retro and become tech debt; no incident is driving it, but the cost compounds.
  • Alternatives: do nothing (status quo); manual ad-hoc /architecture invocation; augmenting /verify's existing reviewer prompts.

Solution Direction

Two integration slots, sharing one rule definition and one LLM judge.

Slot 1 — Phase 4.5 (between /spec and /plan)

Runs /architecture --from-spec <spec-path>. The LLM judge reads principles.md + the spec and renders findings. Soft gate at every tier: Recommended-Run on T3, Recommended-Skip on T2, suppressed on T1.

Slot 2 — Phase 7 sub-step (inside /verify)

The existing /architecture static-analysis path runs against changed modules (--since <merge-base>) and produces mechanical evidence. The same LLM judge reads principles + code + that evidence and renders findings. Runs on T2 (scoped) and T3.

Foundation deliverables (same release)

  • New /architecture --from-spec <spec-path> invocation mode. Reads spec HTML/MD, extracts the proposed module list, dependencies, and architectural assertions, dispatches the judge subagent with principles + spec inputs, and emits the HTML+MD+JSON triplet under {docs_path}/architecture/{date}_<slug>_from-spec.{html,md,json}. Fails loud on specs lacking module/dependency structure.
  • Authoring of plugins/pmos-toolkit/skills/architecture/principles.md — prose source of truth (L1 universal + L2 stack-specific + L3 per-repo conventions). Each principle declares which static tools (if any) detect violations.
  • State schema bump (v5 → v6, or additive sub-structure on existing phase entries) to record phase_4_5_arch_spec_findings_path and phase_7_arch_code_findings_path.

User Journeys

Primary (Tier-3, happy path)

  1. Operator runs /feature-sdlc <idea>; pipeline proceeds normally through /requirements → /grill → /creativity → /wireframes → /prototype → /spec.
  2. Phase 4.5 gate fires: "Run /architecture --from-spec against the just-emitted spec? (Recommended)" — operator confirms.
  3. /architecture --from-spec dispatches the judge subagent and returns 3 findings cited to specific principles + spec sections.
  4. Operator either revises the spec (re-runs /spec) or accepts the findings and proceeds to /plan.
  5. Pipeline continues through /plan → /execute.
  6. Phase 7 sub-step (inside /verify): /architecture --since <merge-base> runs against changed modules; static tools + judge return 1 finding (a cycle introduced); the /verify report includes it alongside lint and test results.
  7. Operator addresses the finding before /complete-dev merges.

Alternate (Tier-2)

Phase 4.5 gate suppressed (Recommended-Skip). Phase 7 sub-step runs scoped to changed modules.

Edge (Tier-1)

Both slots skipped entirely. No /architecture invocation. Zero added latency.

Error journeys

  • --from-spec hits a spec lacking module/dep structure: fails loud (exit non-zero); operator sees a structured error pointing to the minimum spec contract; can revise the spec or pass --force-incomplete.
  • Phase 4.5 LLM judge times out or hits rate limit: soft-gate failure dialog (per /feature-sdlc reference/failure-dialog.md); operator picks Retry / Pause / Skip.
  • Principles doc out of sync with YAML rules: judge cites a principle the tools never check → finding metadata is flagged judge_only: true; does not block.

Design Decisions

#DecisionOptions ConsideredRationale
D1Two slots (Phase 4.5 pre-spec + Phase 7 post-impl)(a) Phase 4 alone, (b) Phase 7 alone, (c) Phase 0 baseline only, (d) bookend at all tiers, (e) standalone handoff suggestionBookend catches issues at both moments where action is cheap; tier-gated to avoid T1 cost. Standalone-only loses signal because users do not run optional tools (per ideate variant #7 rejection).
D2One principles document + one LLM judge (not parallel evaluators)(a) Static tools authoritative + LLM supplement, (b) LLM authoritative + tools as evidence (selected), (c) dual independent evaluatorsDual implementations drift; single judge with tools-as-evidence is the simplest contract and eliminates one of the original H/H risks from the brief.
D3Ship /architecture --from-spec in the same release as the integration(a) Backlog the mode, ship orchestration only, (b) ship both together (selected)Without --from-spec, Phase 4.5 is theatre — has no signal when pointed at prose.
D4Tier-gating: T1 skip both, T2 post-only, T3 both(a) Tier-3-only at all slots, (b) all tiers all slots, (c) ternary T1/T2/T3 dispatch (selected)Pre-spec slot is exploratory; T2 does not reliably need it. T1 fixes do not introduce structural change. Ternary is novel (no precedent in /feature-sdlc) — accepted complexity for right-sized cost.
D5Findings doc shape = same HTML+MD+JSON triplet as code-audit mode(a) New simulation-style folder per /simulate-spec, (b) inline into spec doc as sibling section, (c) standalone triplet (selected)Triplet is the established /architecture artifact contract; reuses substrate; per-feature folder gets a sibling under {docs_path}/architecture/ with cross-links.
D6Principles prose lives at plugins/pmos-toolkit/skills/architecture/principles.md (not in YAML comments)(a) Inline as YAML comments, (b) separate .md authoritative + YAML generated, (c) .md prose alongside YAML (selected; both human-authored, YAML is the machine-readable handoff)Prose readability matters for the LLM judge's rule comprehension; YAML stays the script handoff. Drift risk mitigated by "each principle declares its tool" pattern.
D7Phase 4.5 is a soft gate; never blocks at any tier(a) Hard at T3, soft elsewhere, (b) always soft (selected)Spec-time design is exploratory; hard gates here train users to skip. Findings still rolled into /verify's mandatory pass.
D8Run a 1-day pre-build spike against 5 historical specs to test whether the LLM judge reliably applies prose principles to spec prose; no hard kill-switch threshold(a) Skip spike, trust the judge, (b) spike with hard threshold (e.g. ship Slot 2 only if precision <50%), (c) spike informs design but no go/no-go threshold (selected)The brief flagged judge-on-prose as the riskiest assumption (H/H); the spike is cheap insurance, but a rigid precision threshold would over-mechanise a judgment call. Spike findings feed Decisions in /spec (e.g. cap findings, evidence-citation rules), not a build/no-build gate.
D9New principles ship as both a YAML rule entry and a prose entry in principles.md in the same PR(a) Prose lags YAML and is back-filled, (b) YAML auto-generated from prose, (c) prose + YAML co-authored in the same PR (selected; enforced by review convention)Co-authoring is the cheapest way to keep the two halves aligned. A mechanical drift hook (Q5) would harden this but is out of scope for v1 — review convention is the v1 enforcement and the drift hook is the v2 upgrade path.
