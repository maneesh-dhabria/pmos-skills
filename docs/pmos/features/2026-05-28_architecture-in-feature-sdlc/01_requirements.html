Problem

The repo today has /architecture as a code-side static-analysis lint, /spec as the technical-design emitter, and /verify as the post-impl verification orchestrator — but /architecture is invisible to both. /spec proceeds without an architectural lens; /verify has no structural-regression check beyond what diff-level reviewers happen to notice. The result: architectural issues surface post-merge during /reflect retros — or never — and accumulate as silent tech debt. Even operators running /feature-sdlc end-to-end never get arch coverage; standalone /spec users get less.

Who experiences this?

Repo maintainer driving /feature-sdlc on Tier-3 features in pmos-toolkit and adjacent multi-module repos. Single user today (Maneesh), but the contract generalises to any /feature-sdlc operator on a repo that has shipped an /architecture principles configuration.

Why now?

  • The folding precedent is already proved: /simulate-spec folded into /spec at Phase 6.5; /msf-req folded into /requirements at Phase 5.5. Same pattern applies cleanly to /architecture.
  • principles.yaml already ships in /architecture, so the marginal cost is mostly orchestration plumbing inside /spec and /verify — not new rule-authoring.
  • Folding into the child skills (not the orchestrator) means standalone /spec and standalone /verify invocations also benefit, not just /feature-sdlc runs.

Goals & Non-Goals

Goals

  • Surface architectural concerns at spec time (folded into /spec) for Tier-3 features — measured by user-retains-gate signal (operator does not auto-pass --skip on /spec's arch sub-step after the first 3 Tier-3 runs) and by a false-positive rate below 20% on --from-spec findings.
  • Catch structural regressions at verify time (folded into /verify) for Tier-2 and Tier-3 — validated qualitatively by post-launch retro inspection of /verify reports (Goal #2 is aspirational; no day-1 baseline exists for a quantitative delta metric).
  • Zero added cost on Tier-1 bug fixes — measured by no /architecture invocation appearing in T1 runs of either /spec or /verify.
  • Keep added latency on /spec's arch sub-step under 90s P50 on Tier-3 — measured by timing the folded sub-step on a representative sample run.

Non-Goals

  • NOT auditing the entire repo at /verify time — scoped to changed modules only (since merge-base) — because whole-repo audits produce noise at PR time.
  • NOT building a parallel rules engine — because principles.yaml + a single LLM judge is the chosen direction and dual-implementation is the explicit failure mode the brief eliminated.
  • NOT making /spec's arch sub-step hard at any tier — because pre-impl design is exploratory and false positives train users to skip.
  • NOT building --from-spec as a separate skill — because the existing /architecture skill owns "audit against principles" and --from-spec is a new input mode, not a new domain.
  • NOT adding any new top-level phase to /feature-sdlc — orchestrator stays unchanged; integration lives inside /spec and /verify.

Solution Direction

Two integrations, both folded into the child skill, sharing one rule definition and one LLM judge. /feature-sdlc is unchanged — it never invokes /architecture directly.

How the pieces work together

  • Principles document (principles.md + principles.yaml) — the rules themselves, layered L1 (universal) / L2 (stack-specific) / L3 (per-repo). Source of truth for what "good architecture" means.
  • LLM judge subagent — the evaluator. Reads principles + the artifact under review (spec prose, or code + static-tool evidence) and emits findings citing specific rules.
  • Static tools (dependency-cruiser, ruff, cycle-py, AST harness) — evidence gatherers, code-side only. Feed mechanical signal into the judge.

Integration 1 — folded inside /spec (sibling of /simulate-spec)

/spec grows a new folded sub-step (proposed Phase 6.6, parallel to existing Phase 6.5 /simulate-spec). After /spec emits the spec doc, the sub-step invokes /architecture --from-spec <spec-path>; the LLM judge reads principles.md + the spec and writes findings to {docs_path}/architecture/{date}_<slug>_from-spec.{html,md,json}. Soft sub-step at every tier — uses /spec's existing tier-aware gate semantics (Recommended-Run on T3, Recommended-Skip on T2, suppressed on T1, with auto-upgrade T2→T3 when /spec's own module list contains a name not in the repo's current top-level module set).

Integration 2 — folded inside /verify

/verify grows a new folded sub-step that invokes /architecture --since <merge-base> against changed modules; static tools produce mechanical evidence, the same LLM judge reads principles + code + evidence, findings appear in /verify's aggregated report alongside lint / tests / multi-agent code review. Advisory pattern (matches /verify's existing folded-phase-awareness — Phase 4.5 in /verify SKILL.md lines 450–502). Runs on T2 (scoped) and T3.

Foundation deliverables (same release)

  • New /architecture --from-spec <spec-path> invocation mode. Reads spec HTML/MD, extracts the proposed module list, dependencies, and architectural assertions, dispatches the judge subagent with principles + spec inputs, emits the HTML+MD+JSON triplet. Fails loud on specs lacking module/dependency structure (minimum spec contract defined in /spec per Q9 resolution).
  • Authoring of plugins/pmos-toolkit/skills/architecture/principles.md — prose source of truth (L1 universal + L2 stack-specific + L3 per-repo conventions). Each principle declares which static tools (if any) detect violations.
  • Rule-id-set-equality drift hook (per D9): scripts/check-principles-drift.sh in /architecture's skill folder; runs at /complete-dev time on any commit touching principles.{yaml,md}; refuses to merge if rule-ID sets disagree.
  • /spec sub-step plumbing (Phase 6.6) — including the new spec template contract (Q9): required §Modules section with name+deps per module, §Architectural Assertions section, exit-code/error-message shape for --from-spec failures.
  • /verify sub-step plumbing — folded sub-step invocation, results aggregated into the existing report.
  • NO state schema bump and NO new /feature-sdlc phase IDs — the orchestrator is unchanged.

User Journeys

Primary (Tier-3, happy path via /feature-sdlc)

  1. Operator runs /feature-sdlc <idea>; pipeline proceeds through /requirements → /grill → /creativity → /wireframes → /prototype → /spec.
  2. Inside /spec: after the spec doc lands, the new folded sub-step (Phase 6.6) prompts: "Run /architecture --from-spec against this spec? (Recommended)" — operator confirms.
  3. Sub-step dispatches the judge subagent, returns 3 findings cited to specific principles + spec sections; findings appear in /spec's output.
  4. Operator either revises spec (re-runs /spec — the folded arch sub-step internally re-runs against the new spec) or accepts findings and proceeds.
  5. Pipeline continues through /plan → /execute (no change).
  6. Inside /verify: a folded sub-step runs /architecture --since <merge-base> against changed modules; static tools + judge return 1 finding (a cycle introduced); /verify's aggregated report includes it alongside lint and test results.
  7. Operator addresses the finding before /complete-dev merges.

Standalone /spec invocation (Tier-3)

Operator runs /spec directly (not via /feature-sdlc). The folded arch sub-step still fires — same gate prompt, same findings. This was not possible in the orchestrator-slot design.

Alternate (Tier-2)

/spec's arch sub-step suppressed at gate (Recommended-Skip) unless the spec introduces a new top-level module — then auto-upgrade to Run. /verify's arch sub-step runs scoped to changed modules.

Edge (Tier-1)

Both folded sub-steps skipped entirely. No /architecture invocation in either /spec or /verify. Zero added latency.

Error journeys

  • --from-spec hits a spec lacking module/dep structure: fails loud (exit non-zero); /spec's folded sub-step surfaces the structured error pointing to the minimum spec contract; operator can revise the spec or pass --force-incomplete.
  • LLM judge times out / rate-limit: folded sub-step fails advisory (per /spec's and /verify's existing folded-phase failure pattern); appended to phases.spec.folded_phase_failures[] or phases.verify.folded_phase_failures[]; pipeline continues.
  • Principles doc out of sync with YAML rules (rule-ID drift): drift hook refuses the offending merge at /complete-dev; git commit --no-verify bypass works per the established escape-hatch pattern.

D2One principles document + one LLM judge (not parallel evaluators)(a) Static tools authoritative + LLM supplement, (b) LLM authoritative + tools as evidence (selected), (c) dual independent evaluatorsDual implementations drift; single judge with tools-as-evidence is the simplest contract and eliminates one of the original H/H risks from the brief.

T1 fixes do not introduce structural change; T2 gets the post-impl check; T3 gets both. Auto-upgrade carve-out catches T2 specs that introduce new modules.
D5Findings doc shape = same HTML+MD+JSON triplet as code-audit mode(a) New simulation-style folder per /simulate-spec, (b) inline into spec doc as sibling section, (c) standalone triplet (selected)Triplet remains the established /architecture artifact contract; reuses the substrate. Findings written to {docs_path}/architecture/{date}_<slug>_from-spec.{html,md,json} and {docs_path}/architecture/{date}_<slug>_code.{html,md,json}; cross-linked from /spec's and /verify's output but artifact location unchanged.
D6Principles prose lives at plugins/pmos-toolkit/skills/architecture/principles.md (not in YAML comments)(a) Inline as YAML comments, (b) separate .md authoritative + YAML generated, (c) .md prose alongside YAML (selected; both human-authored, YAML is the machine-readable handoff)Prose readability matters for the LLM judge's rule comprehension; YAML stays the script handoff. Drift risk mitigated by "each principle declares its tool" pattern.
D7Both folded sub-steps use their host skill's existing gate semantics (soft, non-blocking; uses /spec's and /verify's existing folded-phase failure patterns)(a) Hard at T3 / soft elsewhere, (b) always soft via host-skill gate (selected)Folded sub-steps inherit failure semantics from the host skill — no new dialog plumbing needed. Findings are advisory; /verify's mandatory pass still gates merge as today.
D8Run a 1-day pre-build spike against 5 historical specs to test whether the LLM judge reliably applies prose principles to spec prose; no hard kill-switch threshold(a) Skip spike, trust the judge, (b) spike with hard threshold (e.g. ship Slot 2 only if precision <50%), (c) spike informs design but no go/no-go threshold (selected)The brief flagged judge-on-prose as the riskiest assumption (H/H); the spike is cheap insurance. Spike outputs map to three concrete /spec design knobs: (1) top-N findings cap, (2) evidence-citation requirement (judge MUST cite a spec section per finding), (3) confidence threshold (only surface findings rated ≥X). /spec wires each knob from spike data; no hard build/no-build threshold on aggregate precision.
D9New principles ship as both a YAML rule entry and a prose entry in principles.md in the same PR(a) Prose lags YAML and is back-filled, (b) YAML auto-generated from prose, (c) prose + YAML co-authored in the same PR (selected; enforced by review convention)Co-authoring is the cheapest way to keep the two halves aligned. A minimum rule-id-set-equality drift hook ships in v1 (~30 lines bash; YAML rule IDs == MD rule headings — catches the loud "added a rule on one side but not the other" failure). Semantic drift (prose why field mismatched with YAML why) is deferred to v2.
