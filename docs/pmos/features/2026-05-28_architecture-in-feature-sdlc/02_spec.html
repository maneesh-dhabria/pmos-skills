1. Problem Statement

/architecture exists as a static-analysis lint but is invisible to /spec (pre-impl design) and /verify (post-impl gate). Architectural issues surface only at /reflect retros — or never — and accumulate as silent tech debt. This spec folds /architecture as a sub-step inside both /spec (new Phase 6.6) and /verify (new folded sub-step), sharing one principles doc and one LLM judge subagent. /feature-sdlc is unchanged. Primary success metric: operator does not auto-pass --skip-arch across the first 3 Tier-3 features post-launch.

2. Goals

#GoalSuccess Metric

G4/spec arch sub-step P50 latency <90s on T3Time the folded sub-step on first 3 T3 runs; log to release notes

3. Non-Goals

  • NOT whole-repo audit at /verify time — scoped to changed modules since merge-base. Whole-repo audits produce PR-time noise.
  • NOT a parallel rules engine — principles.yaml + single LLM judge is the chosen contract; dual implementations drift.
  • NOT making /spec's arch sub-step hard at any tier — pre-impl design is exploratory; false positives train users to skip.
  • NOT building --from-spec as a separate skill — extends existing /architecture skill with a new input mode.
  • NOT adding any new top-level phase to /feature-sdlc — orchestrator stays unchanged.
  • NOT bumping the /feature-sdlc state schema — sub-step state lives inside child-skill state.
  • NOT running the 1-day judge-precision spike before ship — knob defaults set by judgment (see D8); revisit if post-launch FP rate exceeds threshold.

5. User Personas & Journeys

5.1 Primary Persona

Repo maintainer driving /feature-sdlc on Tier-3 features in pmos-toolkit and adjacent multi-module repos. Single user today (Maneesh); contract generalises to any /feature-sdlc operator on a repo with a shipped /architecture principles configuration.

5.2 Journey — Tier-3 happy path via /feature-sdlc

  1. Operator runs /feature-sdlc <idea>; pipeline proceeds normally through requirements → grill → creativity → wireframes → prototype.
  2. Pipeline enters /spec. /spec emits 02_spec.html and the spec contains the new required §Modules + §Architectural Assertions sections (T3 template addition).
  3. /spec Phase 6.6 fires: gate prompt "Run /architecture --from-spec? (Recommended)"; operator confirms.
  4. Sub-step invokes /architecture --from-spec 02_spec.html. The mode (a) parses §Modules + §Architectural Assertions, (b) dispatches the judge subagent with principles.md (merged L1+L2+L3) + the parsed spec, (c) writes findings triplet to docs/pmos/architecture/2026-05-28_<slug>_from-spec.{html,md,json}.
  5. Findings (3 items, top-N=8 cap) appear in /spec's output cross-linked to the triplet.
  6. Operator either revises spec (re-runs /spec; the folded sub-step internally re-runs against the new spec — no orchestrator cascade) or accepts findings and proceeds.
  7. Pipeline continues through /plan → /execute (no change).
  8. /verify enters. Its new folded sub-step invokes /architecture --since <merge-base> against changed modules; static tools (dependency-cruiser, ruff, cycle-py) produce mechanical evidence; the same judge reads principles + code + evidence; findings appear in /verify's aggregated report alongside lint/tests/code-review.
  9. Operator addresses the finding (a newly introduced cycle) before /complete-dev merges.

5.3 Journey — Standalone /spec (T3)

Operator runs /spec directly (not via /feature-sdlc). Phase 6.6 fires identically — same gate, same findings, same triplet. This was not possible in the orchestrator-slot design (closed in requirements reshape).

5.4 Journey — Tier-2 with new-module auto-upgrade

/spec arch sub-step default suppressed at T2 (Recommended=Skip). Auto-upgrade detector (D12) parses spec §Modules; if any module name is absent from git ls-tree HEAD module-set, gate flips to Recommended=Run with log line arch sub-step: T2→T3 auto-upgrade (new module: <name>). /verify arch sub-step runs scoped to changed modules.

5.5 Journey — Tier-1 edge

Both folded sub-steps skipped. No /architecture invocation. Zero added latency. Log line: arch sub-step: skipped (tier 1).

5.6 Error journeys

  • Spec lacks §Modules: --from-spec exits 65 (spec-contract-violation) with structured stderr listing missing sections; /spec's folded sub-step surfaces the error and offers --force-incomplete (judge runs on whatever sections exist, marked degraded in findings JSON).
  • Judge timeout / rate-limit: sub-step fails advisory per host-skill folded-phase pattern; entry appended to state.yaml.phases.spec.folded_phase_failures[] (or verify); pipeline continues.
  • Principles MD↔YAML drift: pre-commit drift hook refuses the commit; bypassable via git commit --no-verify per established escape-hatch.
  • L3 principles.{yaml,md} missing in target repo: use plugin L1+L2 only; advisory log no L3 overrides found; using plugin defaults.
  • No changes since merge-base in /verify: skip cleanly with architecture: no changes since merge-base; skipping; no judge invocation.

7.1 /architecture --from-spec mode

IDRequirement
FR-01/architecture --from-spec <spec-path> accepts a path to 02_spec.html (or .md sidecar); flag --from-spec mutually exclusive with --since, --baseline, --deep.
FR-02Parses spec for: (a) <section id="modules"> with <table> rows of {name, deps[]}, (b) <section id="architectural-assertions"> with bulleted assertions. Parser uses existing build_sections_json.js substrate.
FR-03On missing required section (modules OR architectural-assertions): exit 65 with JSON-shaped stderr per §6.4 sequence. --force-incomplete flag allows degraded run.
FR-04Loads merged principles (L1+L2 from plugin path, L3 from <repo>/.pmos/architecture/principles.{yaml,md}); missing L3 logs no L3 overrides found; using plugin defaults and proceeds.
FR-05Dispatches judge subagent with (a) merged principles.md prose, (b) parsed §Modules JSON, (c) §Architectural Assertions list, (d) the rule-ID set (from YAML); judge runs at temperature 0.

FR-07Applies knobs: --top-n N (default 8) caps findings by severity then confidence; --min-confidence N (default 70) drops below-threshold; --evidence-required (default true) drops findings without valid quote.
FR-08Emits triplet at {docs_path}/architecture/{date}_<slug>_from-spec.{html,md,json} via existing /architecture HTML substrate; JSON conforms to §13 schema.

9. API Contracts

9.1 /architecture --from-spec CLI

/architecture --from-spec <spec-path>
              [--top-n N]             # default 8
              [--min-confidence N]    # default 70 (0-100)
              [--evidence-required]   # default true; --no-evidence-required to disable
              [--force-incomplete]    # allow run on spec missing required sections
              [--out-prefix <path>]   # default docs/pmos/architecture/{date}_{slug}_from-spec
              [--non-interactive]

Exit codes:

  • 0 — success (findings written to triplet, or empty findings).
  • 1 — runtime error (judge dispatch failed, principles load failed).
  • 64 — usage error (missing arg, mutually exclusive flags, file not found).
  • 65 — spec-contract-violation (missing §Modules and/or §Architectural Assertions; structured stderr per §6.4).

13. Architectural Assertions

  • /spec and /verify must NOT directly invoke principles parsing or judge logic — both dispatch /architecture as a subagent. The judge implementation stays in one place (per D2).
  • The judge subagent must NOT edit any artifact under review (per D11 / FR-34). Reports only. Mirrors /skill-eval D10.
  • principles.yaml remains the machine-readable source of truth for rule structure (id, tier, disposition, delegate_to); principles.md is the human-readable rationale layer. They share a rule-ID set, enforced by the drift hook.
  • --from-spec mode must NOT shell out to static tools (dep-cruiser, ruff, cycle-py); spec mode is judge-only by contract. --since mode is the only path that runs static tools.
  • The orchestrator-side findings validation (FR-06) must hard-drop invalid findings (unknown rule_id, missing/fabricated quote) — not surface them with a "judge hallucinated" warning. Silent drop with log line is the contract (mirrors /skill-eval FR-43 quote-validation pattern).
  • /feature-sdlc orchestrator MUST remain unchanged — no new phase IDs in state-schema.md, no new phases[] entries, no new conditional logic. The arch sub-steps are invisible at the orchestrator layer.
  • The orchestrator-side findings validation logic (FR-06: rule_id ∈ set, quote substring-grep, confidence-range) lives inside /architecture (the host of the judge subagent dispatch), NOT inside /spec or /verify. Validation runs in the same process boundary as the dispatch; the calling skills receive only validated findings.

16. Testing & Verification Strategy

16.1 Unit Tests

  • Spec parser: feed 5 fixture spec.html files (canonical, missing §Modules, empty §Modules table, malformed §Modules row, present-but-empty §Architectural Assertions) → assert extracted JSON shape OR exit 65 with correct missing_sections array.
  • Principles loader: feed (L1+L2 plugin only), (L1+L2 + L3 yaml only), (L1+L2 + L3 yaml + L3 md), (L3 collision with L1) fixtures → assert merged ruleset has expected rule-ID set + L3 last-wins.
  • Findings validator (FR-06): feed judge output with (valid finding), (unknown rule_id), (quote not in source), (confidence out of range), (missing quote, evidence-required=true) → assert valid kept, invalid dropped with expected log lines.
  • Knob application (FR-07): feed 15 findings, top-N=8, min-confidence=70, evidence-required=true → assert exactly 8 returned, all confidence ≥70, all with valid quote.
  • Auto-upgrade detector (FR-18): feed (spec with 1 new module + 3 existing), (spec with all existing modules), (empty spec §Modules) → assert upgrade/no-upgrade/no-upgrade.
  • Drift hook (FR-36): shell test — create YAML with rule U001 only, MD with U001 + U999 → hook exits 1 with diff; both with U001 only → hook exits 0; commit not touching either file → hook skips.

16.2 Integration Tests

  • /architecture --from-spec end-to-end: fixture spec → assert triplet written to expected path; JSON validates against §13 schema; HTML viewable.
  • /spec Phase 6.6 gate, three-tier fixtures: T1 fixture → assert no /architecture dispatch, log line emitted; T2 fixture (no new modules) → assert Recommended=Skip; T2 fixture (new module) → assert auto-upgrade fires + Recommended=Run; T3 fixture → assert Recommended=Run.
  • /verify Phase 4.7 sub-step, three-tier fixtures: T1 → skipped; T2 → run scoped; T3 → run full.
  • /spec re-run cascade (FR-23): run /spec → trigger Phase 6.6 → revise spec via Edit → re-run /spec → assert Phase 6.6 re-runs internally, new triplet emitted (or suffix-2 on filename collision).
  • Advisory failure pattern (FR-22, FR-29): mock judge timeout → assert folded_phase_failures[] append + chat warning + /spec or /verify continues to exit.

16.3 End-to-End Tests

  • Single dry-run of /feature-sdlc on a synthetic T3 feature with the new plumbing: assert Phase 6.6 fires inside /spec, Phase 4.7 fires inside /verify, triplets land at expected paths, /feature-sdlc state.yaml unchanged in shape (no new phase IDs).
  • Standalone /spec invocation on the same synthetic T3 spec (not via /feature-sdlc): assert Phase 6.6 fires identically.
  • Drift hook installer: run bash plugins/pmos-toolkit/skills/architecture/scripts/install-arch-hooks.sh → assert .git/hooks/pre-commit includes drift check; commit YAML-only change → hook refuses; commit YAML+MD aligned → hook passes.

17. Rollout Strategy

  • Single-version release via /complete-dev (minor bump — additive feature). All deliverables ship together: --from-spec mode + principles.md + drift hook + /spec Phase 6.6 + /verify Phase 4.7 + spec template additions.
  • No feature flags. The new behaviour is opt-in by tier gating (T1 skip, T2 conditional, T3 default-on) plus per-invocation --skip-folded-arch escape; operator can always opt out per-run.
  • Migration order at release time:
    1. Author principles.md alongside existing principles.yaml (same PR).
    2. Install drift hook via bash scripts/install-arch-hooks.sh.
    3. Implement --from-spec mode in /architecture.
    4. Wire /spec Phase 6.6 + spec template additions.
    5. Wire /verify Phase 4.7.
    6. Run integration tests; verify drift hook self-check passes against authored principles.md.
  • Rollback plan: the feature is purely additive at the host-skill level. Revert is a single git revert of the release commit; existing /architecture, /spec, /verify behavior restored. principles.md becomes orphan but harmless. Drift hook can be uninstalled via git config core.hooksPath reset or hook file removal.
  • Graceful degradation: if /architecture is broken at runtime (judge dispatch fails, principles unparseable), folded sub-steps fail advisory and /spec / /verify continue. /verify's existing hard gates (Phase 7) still apply — arch sub-step does NOT gate merge.
  • Post-launch monitoring: after first 3 T3 features, manually inspect (a) Phase 6.6 timing (NFR-01), (b) operator skip rate (G1), (c) false-positive rate via accept/reject ratio in operator response (NFR-03). If FP >20%, revisit knob defaults (D8 escape clause) — likely tighten min-confidence to 80.
