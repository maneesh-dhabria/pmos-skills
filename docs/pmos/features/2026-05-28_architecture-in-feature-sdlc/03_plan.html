P1 Tracer-bullet T1 wires the full --from-spec pipeline (parse → judge dispatch → validate → emit) against a hardcoded 2-rule principles slice and a single 1-module fixture spec, before authoring the real 18-rule principles.md. (a) Author principles.md first (T1=foundation), (b) wire the pipeline first with a stub principles slice (selected), (c) wire spec parser first and stub judge. Risky unproven integration is the judge subagent + orchestrator-side validation (rule_id ∈ set + quote substring), per /plan Phase 2 step 7. Authoring 18 rules first proves nothing about whether the pipeline works. A 2-rule slice plus a 1-module fixture is the narrowest end-to-end path that exercises temperature-0 dispatch, JSON-array parsing, rule_id validation, quote validation, and triplet emit. If the judge contract is wrong, we discover it in T1 — not after writing 18 rules of prose.
P2 Implement spec parser (T6) and judge-prompt-template authoring (T3) as separate parallel tasks rather than bundling them. (a) One "spec ingestion" task covering both, (b) split into parser + template (selected), (c) split three ways including loader. The parser touches scripts/parse-spec.js and consumes only build_sections_json.js substrate; the template touches reference/judge-prompt-template.md and consumes only the rule list. Non-overlapping file sets allow Wave 3 parallel dispatch (subagent-driven mode benefits from file-conflict-free waves per /execute deterministic planner). Bundling would force serialization with no engineering benefit.
P3 Place scripts/parse-spec.js as a Node script reusing the existing _shared/html-authoring/assets/build_sections_json.js substrate, NOT a fresh HTML parser. (a) Write a fresh DOM walker, (b) shell out to a Python BeautifulSoup script, (c) reuse build_sections_json.js regex+stack walker (selected, ~80 LOC). Per FR-02 spec text: "Parser uses existing build_sections_json.js substrate". Substrate is zero-dep, already tested across 14 surfaces, handles the same HTML shape /spec emits. Reusing it eliminates an entire class of parser-divergence bugs and keeps the implementation aligned with the spec's stated contract.
P4 Author principles.md (T2) and the judge prompt template (T3) in Wave 2 in parallel with the drift hook (T4), gated on T1's tracer-bullet proving the judge contract first. (a) Author principles.md before any pipeline code, (b) author in parallel with prompt template + drift hook (selected), (c) defer principles.md to last. T1's tracer slice (2 rules) defines the principles.md schema empirically before scaling to 18 rules. Once T1 lands, T2/T3/T4 are leaf parallel tasks with non-overlapping files (principles.md vs reference/judge-prompt-template.md vs scripts/check-principles-drift.sh + install-arch-hooks.sh). Drift hook can be authored without depending on the 18-rule MD content — it operates on rule-ID set equality regardless of count.
P5 Wave-5 host-skill integrations (T12 /spec Phase 6.6 + T13 /verify Phase 4.7) dispatched as parallel tasks because they touch disjoint skill directories. (a) Inline tests per task only (already done), (b) inline + Wave-6 integration/E2E suites (selected), (c) one big test task at end.
