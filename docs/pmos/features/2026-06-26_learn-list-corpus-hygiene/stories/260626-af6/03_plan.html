Plan — Story A: Playwright title + content backfill

Story 260626-af6 · epic 260626-j5k · route: skill · plugin: pmos-learnkit · deps: none. Scoped by design D5/D6/D1/D3 + the story ACs.

Task waves

Wave 1: study (T1). Wave 2: build the recovery engine (T2) + drop/report policy (T3). Wave 3: run + re-summarize (T4, T5). Wave 4: validate cross-cutting invariants (T6) + skill conformance (T7).

Key risks

  • Real-browser escalation flakiness. Non-headless fetches (Amazon, hard Cloudflare) are slower and can still fail. Mitigation: bounded retries + URL-slug last resort (D5); never block the whole run on one record.
  • id re-mint on URL canonicalization. Redirect-following can change the canonical URL, which re-mints ref_<sha256(url)> (C4). Mitigation: re-mint deterministically, report id changes, and keep the redirect decision explicit (only adopt canonical when it's a stable 301/308, not a transient).
  • PII in recovered text. Recovered titles/summaries must pass the same T1 scrub (C1). Mitigation: route every recovered field through the importer's scrub before writing.
  • Re-summarization drift. Re-summarizing ungrounded records could lower quality. Mitigation: only flip summary_grounded:true when a real summary is produced from real body text; otherwise leave the record WEAK, unchanged.

Verification

  • Junk-title rate over the full corpus drops to <5% (report before/after counts per category).
  • Conservative-drop honoured: every dropped record is in the reported confirmed-dead list; spot-check 5 that they're genuinely gone.
  • T1 PII scrub gate passes GREEN over the regenerated corpus.
  • /learn-list browse + /primer library viewers render the regenerated corpus (live DOM check); curated-references-match.mjs prefilter still functions on a sample topic.
  • Re-running the importer reproduces the cleaned corpus deterministically (D3).
  • Touched SKILL.md (if any) passes skill-eval.md + the 4 lints.
