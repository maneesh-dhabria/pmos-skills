/learn-list curated-references corpus hygiene

Epic 260626-j5k · route: skill · plugin: pmos-learnkit · two deliverables (title backfill → tag vocabulary). The cross-skill coherence contract the stories cite by anchor.

This is the epic design_doc (the route: skill analogue of a /spec): it names the cross-cutting invariants and shared-substrate shape that the per-story plans must honour. Per-skill implementation detail lives in each story's /plan, not here.

1. Problem

The shipped reference corpus plugins/pmos-learnkit/skills/_shared/topic-research/curated-references.json (1,817 records) backs /learn-list browse and the /primer library viewer, and feeds the IDF prefilter (curated-references-match.mjs) both skills consume. It was imported once, verbatim, from a Notion spike YAML via import-curated-references.mjs, which only scrubs PII and drops dead (404) pages — it never re-fetched titles or normalized tags. Two end-user-facing quality defects result, both confirmed by measurement on 2026-06-26:

  • Junk titles — 611 / 1,817 (34%). The display label is a non-title: 305 empty, 179 "Just a moment..." (Cloudflare), 98 "Amazon.com", 11 "403 Forbidden", 10 "429 Too Many Requests", 8 WAF pages. 54% of these still carry a real grounded summary — only the title is broken.
  • Tag sprawl — 511 distinct tags. 57% appear ≤2 times, 42% exactly once; only ~99 appear ≥10 times. The browse-viewer facet list is unusable at this width, and brand/company tags (amazon, netflix, openai, anthropic, claude, figma, kafka, redis, kubernetes…) add noise without topic value.

Proven feasible (2026-06-26): a read-only headless-Chromium (Playwright) prototype recovered real titles for 45/50 sampled junk records — 90% reachable, 70% clean on first pass — with every miss traced to a known, addressable cause. Body content comes back too, so ungrounded summaries can be re-fetched in the same pass.

3. Decisions

D1 — Playwright is a build-time tool, never a runtime skill dependency

The backfill runs offline at corpus-refresh time (like import-curated-references.mjs), not when a user invokes /learn-list or /primer. The shipped artifact is the cleaned JSON; the recovery script is tooling. This preserves the learnkit zero-runtime-dependency posture (the skills stay offline, no-network, no-account). Chromium is resolved from the already-installed Playwright cache; the script degrades with a clear error if absent rather than adding an npm dependency to the plugin.

D2 — The tag vocabulary lives as reviewable data, not buried in script logic

The closed vocabulary and the synonym→canonical map are authored as data files under the topic-research substrate (e.g. tag-vocabulary.json + tag-synonyms.json), so a maintainer tunes the taxonomy without editing code. import-curated-references.mjs (or a tag-normalize step it calls) reads them and is the single enforcement point. One fact, one home (CLAUDE.md §K).

D3 — Normalize at import time, not as a one-shot mutation of the shipped JSON

Both fixes are wired into the generation path so a future corpus refresh stays clean — the corpus is never hand-edited. Backfill is a pre-pass that feeds cleaned records into the importer; tag normalization is a transform inside the importer. Re-running the importer over a fresh spike export reproduces a clean corpus deterministically.

D4 — Story order af6 → ex8 (backfill before tag-vocab)

ex8 declares a dependency on af6. The backfill (af6) re-summarizes ungrounded records; those re-summaries must be re-tagged under the new closed vocabulary (ex8) in one generation rather than tagged twice. Running tag-vocab first would re-tag stale summaries that the backfill then overwrites. The picker orders af6 first via the dependency.

D5 — Recovery precedence, real-browser escalation, and the "still-junk" guard

Title recovery precedence is og:title → a clean document.title → first <h1>, following redirects to the canonical URL (the prototype found Substack/Technically redirecting to canonical hosts). A recovered value that is itself junk — host-only like medium.com, an error string, or <5 chars — does not overwrite. Instead the record escalates:

  1. Headless fetch with a real UA + networkidle settle (the prototype default).
  2. On a junk/host-only/error result, one cooldown-retry (clears most Cloudflare JS challenges, which the 2.5s prototype wait was too short for).
  3. Still junk → escalate to a real (non-headless) Playwright browser context — the same path the interactive Playwright MCP took when it successfully recovered "The Principles of Product Development Flow — Donald G. Reinertsen" from a page that headless returned as "Amazon.com". Per maintainer decision (2026-06-26), this real-browser escalation is applied to generic/error titles like Amazon.com too, not just empty ones — we try to fetch the genuine title before any fallback.
  4. Only if the real browser also fails does the URL-slug heuristic act as a last-resort title source (Amazon book slugs are close to the title); a slug-derived title is marked so it can be audited.

No fix silently swaps one junk label for another, and no record is dropped merely because the first (headless) attempt failed — the live-fetch evidence used by D6 is the result after this escalation ladder.

D6 — Conservative drop policy: only confirmed-dead, after full Playwright escalation

Per maintainer decision (2026-06-26), the drop is conservative: a record is removed only when the live fetch — after the full D5 escalation ladder including the real-browser attempt — confirms the page is genuinely gone (hard 404, dead domain, "Deployment Paused"). A record that merely resisted a headless title fetch but is alive is kept with its recovered (real-browser or slug) title; it is never dropped for being hard to fetch, and never re-shipped with an error-page label. This extends the importer's existing DEAD policy (#junk-policy) from title/summary-text detection to post-escalation live-fetch evidence. The before/after record count, the drop list, and any slug-fallback titles are reported by the backfill so every deletion and fallback is auditable.

D8 — X/Twitter records use a host-specific recovery branch (meta tags, not rendered DOM)

Per maintainer decision (2026-06-26), 205 of the 1,817 refs are x.com/twitter.com URLs — none trip the strict junk-title regex, but most carry a generic title ("Thread by @user on Thread Reader App", or bare "Thread Reader App") and 149 of the 205 have summary_grounded:false. Verified live (2026-06-26, headless Playwright on 3 tweet URLs): X serves a login wall in the rendered SPA — [data-testid="tweetText"] is null — but the server-rendered meta tags still carry the full content for the unfurl/crawler path. So tweets get a dedicated branch in the D5 ladder with inverted precedence from the article handler:

  • Title ← document.title (carries the quoted tweet text: Shreyas Doshi on X: "Product Management—in 1 tweet…" / X), stripping the trailing " / X" suffix. Not og:title — for tweets that is author-only (Shreyas Doshi (@shreyas) on X), weaker than today's value.
  • Summary grounding ← og:description (the full tweet body, verbatim) — feeds the D5 re-summarize step so the 149 ungrounded tweets can flip summary_grounded:true.
  • Link/image-only tweets (og:description is just a t.co link, no prose — proven on the garrytan case) stay legitimately WEAK; no fabricated summary. Deleted/suspended tweets return a tombstone title and fall to D6's conservative drop.

No login, API key, or browser-use needed — plain headless Playwright on the existing og: path. This is the first ladder branch tried for an x.com/twitter.com host; the generic og:title → document.title → h1 precedence still governs all other hosts.

D9 — Throttling + retry-budget so the backfill doesn't recreate the junk it's fixing

Per maintainer decision (2026-06-26), the backfill issues ~800 live navigations (611 junk-title articles + ~205 tweets) against hosts that rate-limit aggressively — X, Cloudflare, Amazon. An unthrottled burst earns 429/"Just a moment…" responses, which would recreate the exact junk titles this story removes and could even drop alive records under D6 on false-dead evidence. So throttling is a first-class requirement, not a nicety:

  • Per-host serialization + politeness delay. Group fetches by registrable domain; never run two concurrent requests to the same host; insert a randomized delay (e.g. 1.5–4 s jitter) between same-host requests. Distinct hosts may run with bounded global concurrency (e.g. ≤4) so the full run still completes in reasonable wall-clock.
  • 429/503-aware backoff. A 429/503 (or a Cloudflare/X challenge body) triggers exponential backoff with jitter and honours Retry-After when present; the request is retried, not counted as dead. This is the cooldown-retry rung of the D5 ladder made explicit for rate-limit responses.
  • Bounded retry budget feeds D6. A host that still returns a rate-limit/challenge after the backoff budget is exhausted is recorded as rate-limited, not dead — the record is KEPT with its best recovered (or existing) title and flagged for a later re-run, never dropped. Only genuine 404/dead-domain/tombstone evidence drops a record (D6).
  • Idempotent + resumable. Because D3 wires the backfill as a re-runnable import pre-pass, a rate-limited subset can be re-fetched on a second pass without redoing the whole corpus — the report's rate-limited list is the worklist for that re-run.

D7 — Final tag count target ~100–150; nothing outside the closed vocabulary ships

Every shipped tag must be a member of the closed vocabulary. Unknown tags after normalization are reported and gated (a non-zero unknown count is a build-time failure, not a silent pass), so the taxonomy cannot silently re-sprawl on the next refresh.

4. Constraints (cross-cutting invariants every story honours)

  • C1 — PII gate stays GREEN. The T1 PII scrub gate (tests/test_pii_scrub_gate.sh) must pass over the regenerated corpus. Recovered titles/summaries are scrubbed by the same gate before shipping.
  • C2 — Browse viewers stay correct. /learn-list browse and the /primer library viewer (both built on _shared/library-viewer/) must render the regenerated corpus with no regression; facet filters must reflect the new tag set.
  • C3 — IDF prefilter stays functional. curated-references-match.mjs derives tag IDF weights from the corpus; collapsing 511→~120 tags changes those weights. The prefilter must still function (re-check after re-tag; a ubiquitous tag with IDF≤0 must not break matching).
  • C4 — Schema unchanged. The 7-field record shape (url,title,source_type,publication_date,tags,summary,summary_grounded) and the ref_<sha256(url)> id derivation are preserved. Backfill may change title/summary/summary_grounded values and drop records; it never adds fields. A title/URL change that re-canonicalizes the URL re-mints the id (id is sha256(url)) — consumers key on id, so the re-mint must be deterministic and reported.
  • C5 — Skill conformance. Any touched SKILL.md conforms to skill-patterns.md §A–§L and passes skill-eval.md. Most of this epic is substrate + tooling under _shared/topic-research/, not SKILL.md prose — the eval surface is small but non-zero (the --no-curated / coverage notes in /learn-list and /primer must stay accurate).

6. Epic acceptance surface

  • Junk-title rate drops from 34% toward <5%; remaining genuinely-dead pages dropped, not shipped with an error label (af6).
  • Tag count collapses 511 → ~100–150; every shipped tag is in the closed vocabulary; synonym/plural/brand variants eliminated (ex8).
  • Normalization applied at import time so a refresh stays clean (af6 + ex8).
  • C1–C5 hold: PII gate green, browse viewers + IDF prefilter correct, schema unchanged, skill conformance.
