Design — /primer References section

pmos-learnkit · epic 260704-rgt · route: skill · the cross-cutting coherence contract for stories 260704-ytr (generator + forward wiring + guideline) and 260704-3jt (corpus backfill). Stories cite this doc by anchor. Grill-hardened (D1–D10). No epic /spec — this design doc is the skill-epic contract; per-skill spec folds into each story's /plan.

1. Overview

Every /primer is already built on a fully-verified source set persisted as a {date}_{slug}.sources.json sidecar (array of {url, takeaway, topic, tier, paywalled}). Today that evidence base is invisible in the reader-facing artifact — the HTML ends at ## Where this connects — adjacent topics and the sources live only in the sidecar JSON. This epic surfaces that existing data as a ## References section at the bottom of every primer: the full set of sources the primer was built on (all were read + synthesized), with the ones actually cited inline in the prose visually marked.

The change has three faces, all serving one rule — a reader can see, and trust, exactly what a primer was built on:

  • Forward — new primers emit a References section automatically (story 260704-ytr).
  • Backfill — all 61 bundled corpus primers get the section injected from their existing sidecars (story 260704-3jt).
  • Guideline — References becomes a required closing section enforced by a deterministic rubric check, so it never silently regresses (story 260704-ytr).

2. Invariants

  • INV-1 — One generator, two callers (§K one-home). The References HTML is produced by a single deterministic function. The forward write path and the backfill CLI both call it; neither re-implements the rendering. There is exactly one place that decides how a reference renders.
  • INV-2 — Deterministic, not model-authored (§H). The References list is a pure transform of sources.json + the rendered body. The LLM never hand-writes it; a script does. Hand-authoring a References block is an anti-pattern on par with hand-authoring the primer shell.
  • INV-3 — Data already exists; nothing re-fetched. Both forward and backfill read only the primer's own sources.json. No network, no re-verification, no new sourcing. Backfill of the corpus is a pure local re-render.
  • INV-4 — Idempotent injection. Re-running the generator over a primer that already has a References section replaces it in place — never appends a second one. Backfill is safe to re-run; the forward path never double-emits.
  • INV-5 — Show every source, honestly. The section lists all sources.json entries (every one was read + synthesized = "used in creating"), and marks which also appear as an inline citation. It never silently drops a source, and never claims a source was cited that was not.

3. What the References section is

A flat, numbered back-matter list rendered as the final section of the primer body — after ## Where this connects, immediately before </main>. The <h2 id="references">References</h2> heading picks up the primer's existing [NN] section-signature counter automatically (CSS h2::before), so it reads as the last numbered section.

Each entry (D4 — flat numbered [N]):

[1] <a href="{url}">{label}</a>  ·cited·   (T2)
    {takeaway}
[2] <a href="{url}">{label}</a>            (T1)
    {takeaway}

4. Architecture — the shared generator (INV-1)

A new module plugins/pmos-learnkit/skills/primer/scripts/references-section.mjs (zero-dep Node ESM, matching build-library.mjs) exports two functions:

ExportSignatureUsed by
renderReferencesFragment(sources, citedUrls) → htmlString The single source of truth for the section markup. Returns a content-only <h2 id="references">…</h2><ol>… fragment. Pure; no I/O.
injectReferences(primerHtml, sources) → newHtml Backfill + any post-render path. Parses body <a href> URLs → citedUrls, calls renderReferencesFragment, removes any existing #references section (INV-4), inserts the fragment immediately before the closing </main>.

4.1 Forward path (story 260704-ytr)

Phase 5 builds a content-only {{content}} fragment then renders via renderArtifact(). The References fragment is produced by renderReferencesFragment(sources, citedUrls) — where citedUrls are computed from the drafted body — and appended to content as its last block, before renderArtifact() runs (not post-injected). This keeps the single-render-path contract (Anti-pattern #9) intact: the shell still comes only from template.html. The sections.json assembled in the atomic trio must include the references id (its id set must equal the on-page id set — rubric R7).

4.2 Backfill path (story 260704-3jt)

A new CLI scripts/backfill-references.mjs walks data/primers/*.html, reads the sibling *.sources.json, calls injectReferences(html, sources), and atomically rewrites the HTML in place. Idempotent (INV-4): a second run is a no-op diff. Bundled primers ship no sections.json (confirmed — only .html + .sources.json in the corpus), so backfill touches HTML only; there is no sections.json to reconcile for the static corpus.

5. Guideline & enforcement (story 260704-ytr)

  • Required section. reference/audience-presets.md adds ## References to the required closing-sections contract for every preset (senior-pms + all-pms), placed after ## Where this connects.
  • New rubric check R11 (deterministic, trust-tier). Added to reference/rubric.md: (a) a <h2 id="references"> section exists; (b) every sources.json[].url appears in the References list (membership test — no source silently dropped, INV-5); (c) the references id is in sections.json. R11 is a membership/structure test — like R7, it runs deterministically at construction time and does not require re-fetching.
  • SKILL.md. Phase 4 notes References is generated deterministically (not drafted); Phase 5 step 7 appends the fragment + adds the id to sections.json; the worked example and Anti-patterns gain a "don't hand-author References" note mirroring #9.

6. Decisions (grill-hardened)

  • D1 — List scope: all sources, cited-marked. The section lists every sources.json entry (all were read + synthesized), and marks the subset that also appear as inline <a href> citations. Faithful to "used in creating and cited." (Rejected: cited-only bibliography — would hide background sources the primer genuinely rests on.)
  • D2 — Enforcement: required section + rubric check R11. Durable, eval-gated. (Rejected: procedure-only — silently regresses.)
  • D3 — "cited" is computed, not stored. Deterministic: URL-substring match inside body <a href> outside the References section. No new sidecar field.
  • D4 — Format: flat numbered [N], ordered tier-then-appearance. Matches the primer's [NN] signature aesthetic. (Rejected: grouped-by-H2-topic — heavier, and topic slugs don't always map cleanly to final H2 ids.)
  • D5 — Anchor label = cleaned host. Schema has no title; a derived host label is deterministic and identical for forward + backfill. The takeaway (which typically names the source, e.g. "Wikipedia defines…") carries the human attribution. No schema change.
  • D6 — Stable, idempotent output. Ordering + cited-detection are pure functions of inputs, so re-running produces a byte-identical section (INV-4). Backfill re-runs are no-op diffs.
  • D7 — Placement: absolute bottom. After ## Where this connects, before </main>. References is back-matter apparatus, not teaching.
  • D8 — word_count excludes References. The library index / R10 word_count counts teaching prose only; the References apparatus is excluded so the metric stays comparable to pre-epic values. Backfill does not recompute or touch primers-index.json (sources_count already present; word_count unchanged by design).
  • D9 — Two stories, generator → backfill. Story 260704-ytr ships the shared generator + forward wiring + guideline (no deps). Story 260704-3jt backfills the corpus using that generator (deps: 260704-ytr; D9 claim-time merge brings the module into the backfill worktree). Both are one skill (/primer), one plugin (pmos-learnkit), one release unit.
  • D10 — No re-verification, no re-sourcing. Surfacing existing sidecar data is not a research run. Backfill is a pure local transform; no WebFetch, no rubric R1–R3 re-scoring of the corpus.
