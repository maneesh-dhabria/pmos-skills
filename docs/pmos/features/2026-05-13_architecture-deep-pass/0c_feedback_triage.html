F1 — Deepening review pass new-capability large

already_handled: no. The skill's Phase 4 (Evaluate rules) dispatches by delegate_to ∈ {grep, dependency-cruiser, ruff} — all deterministic, file-local. No subagent dispatch exists today; Phase 0 of SKILL.md mentions "rule evaluation is fully sequential in v1 — no subagent dispatch."

recommendation: apply as proposed — new opt-in Phase 4.5 ("Deepening review") gated by --deep, dispatched after the mechanical pass. Reuse vocabulary from /improve-codebase-architecture (module / interface / implementation / depth / seam / adapter / leverage / locality / deletion test) verbatim. Cap candidate set at ~10 modules. Emit deepening_candidates[] with {module, classification, rationale, deletion_test_outcome, cross_module_patterns[], proposed_reshape, affected_files[]}. Implement double-report demotion: if L1 flagged a file with U001 and the deep pass classifies it as "deep, no reshape," demote U001 to info or annotate the report. Emit tools_skipped: ["deep_pass"] if no subagent tool is available — never crash. Rationale: the user's case (book-companion, 935/797/745-LOC files) is exactly the shape this is for; the matching reference skill already exists and gives us the vocabulary; reusing it makes outputs consistent across both tools.

F2 — Auto-detect framework idioms bug small

already_handled: no. U004's check is regex:console\.log|print\(;paths:src/;exclude:tests/,scripts/ — pure regex, no AST, no idiom detection. 218 false positives on a Typer-heavy backend is the maximum-noise outcome.

recommendation: apply. Detect typer/click/fire/argparse imports per file at scan time → file-level exemption for U004 only. Detect migrations/versions/ path (Alembic) → exempt U001, U004, U007. Detect *_pb2.py / *_pb2_grpc.py (protobuf) → exempt all hygiene rules. Detect pydantic BaseModel subclass → relax U001 (schema modules legitimately get long). Surface in JSON as exemptions.idiomatic[] with {rule_id, file, idiom, reason} so the audit doesn't hide what it skipped. Rationale: false-positive floods are the #1 reason users mentally tune out the report — every Typer project produces this exact noise today.

F3 — Demote U007 to opt-in ux-friction small

already_handled: no. U007 ships severity: info but always emits, producing the 710-finding wallpaper.

recommendation: apply. Two-tier handling: (a) default-off — U007 fires only when --include-info-comments is passed OR when L3 explicitly opts in; (b) when it does fire, harden the heuristic to "file > 100 LOC AND no module-level docstring AND no top-of-file comment in the first 5 non-blank lines." Document in l1-rationales.md. Rationale: the rule's intent (legibility for cold readers) is sound; the default-on behavior is what poisons it.

F4 — Monorepo-aware mode bug small

already_handled: partial. SKILL.md Phase 2 step 2 "Stack detection: scans the resolved root for stack markers" — but it scans the root only. A monorepo with backend/pyproject.toml and frontend/package.json below the root produces zero stacks, no L2 rules, silent success.

recommendation: apply. At scan-root resolution time: (1) check for stack markers at root; (2) if absent, walk depth-2 looking for child stack markers; (3) if found, emit one of: stderr warning + record monorepo_detected: ["backend", "frontend"] in JSON + offer --monorepo fan-out (one report per stack with combined index), OR auto-fan when --monorepo is passed. Silent stack-skipping is the worst failure mode — explicit is better. Rationale: silent failure is much worse than noisy failure; the cost of the depth-2 walk is negligible.

F5 — Risk score (churn + coupling) new-capability medium

already_handled: no. findings[] sorts severity-first then by rule_id / file / line (FR-73). All U001 hits look equally bad.

recommendation: apply. Per finding, compute risk_score = w_severity + w_churn(git log --since=<90d> --oneline | wc -l per file) + w_coupling(F6 fanin count). Concrete weights: severity {block:100, warn:10, info:1}; churn capped at 50 (1 point per commit, max 50); coupling capped at 50 (5 points per inbound import, max 50). Emit risk_score per finding. Keep severity-first sort for backward compatibility, but add a --sort risk flag and use it by default in the stderr summary (F12). Rationale: hot files are where bug rate compounds; this is the single biggest prioritization win and costs one shell-out per file.

F6 — Module fanin / fanout new-capability small

already_handled: no.

recommendation: apply. AST walk (Python ast module + a TS regex pass since dep-cruiser already does the heavy lift on TS side) to count: per module — fanin (count of distinct importers) + fanout (count of distinct imports) + public_symbols (top-level def + class count). Emit per-module under module_metrics[]; flag top-5 by fanin × public_symbols as godmodule_candidates[]. Used by F1 (seed list for deep pass) and F5 (coupling weight). Rationale: a deterministic god-module detector that doesn't need the LLM and feeds the LLM pass when run.

F8 — Python import-cycle detector new-capability medium

already_handled: no. dep-cruiser TS001 covers JS/TS only.

recommendation: apply. New PY009 rule (severity block, delegate_to: cycle-py) with a ~50-LOC tools/cycle-py.py — walks ast.parse per file, extracts Import/ImportFrom, resolves relative imports against package roots from pyproject.toml, builds a module graph, runs Tarjan's SCC, emits cycles > 1 node. Treat as L2 (only fires when Python stack is detected). Rationale: Python cycles surface as runtime MissingGreenlet / partial-init errors that are nasty to debug; a 50-LOC walker closes a real gap.
