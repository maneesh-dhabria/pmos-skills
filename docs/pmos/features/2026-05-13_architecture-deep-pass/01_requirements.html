/architecture — deepening pass + hygiene — Requirements

Date: 2026-05-13 · Last updated: 2026-05-13 (post-grill revision) · Status: Draft — revised per grill report · Tier: 3 — Skill feedback (12 findings) · Mode: interactive · Pipeline: /feature-sdlc skill --from-feedback · Target skill: plugins/pmos-toolkit/skills/architecture/ · Target platform: generic · Acceptance criteria reference: plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§F · Grill report: grills/2026-05-13_01_requirements.html

Problem

The /architecture skill is a fast, deterministic rule engine that runs L1 universal (≤15 rules) + L2 stack-specific (TS via dependency-cruiser, Python via ruff) + L3 per-repo overrides against a scan root and emits a JSON report to stdout. In its current shape (v2.48.0) it catches hygiene symptoms — oversized files, print() in src, dead comments, long arg lists — but it cannot surface the architectural issues that actually matter: shallow modules, cross-module seam leaks, dispatch-by-conditional chains, implicit pipelines, responsibility scatter across CLI + service + route. These require reading code, following call graphs, and applying a deletion test — not regex matching.

On a real Python backend (book-companion, ~900 LOC service files), one run produced 586 findings dominated by 218 print() false positives (all legitimate Typer CLI commands) and 710 info-noise findings from U007 missing-purpose-comment wallpaper. The 14 oversized-file flags are real but undifferentiated — every U001 hit looks equally bad. Worse: running the audit at the monorepo root silently skipped both stacks because pyproject.toml and package.json live in subdirs. And the JSON-on-stdout output forces callers to remember a redirect — there's no human-scannable artifact a developer can just open and read.

Who experiences this?

The developer running /architecture audit . on a real repo — typically a senior engineer or tech lead trying to audit a codebase for architectural health before a refactor, during onboarding, or as a periodic check. Concrete persona from the feedback: the maintainer of book-companion (~900 LOC backend Python service files), running the audit to find architectural problems but receiving an unranked sea of false-positive hygiene flags that swamps the real signal — with no human-readable artifact to share or revisit.

Why now?

Three triggers converged this week:

  1. A real user ran /architecture audit . on book-companion and produced the 586-finding report that motivated this feedback. The single run made the noise/signal ratio undeniable.
  2. The user already runs a sibling skill, /improve-codebase-architecture (Matt Pocock-derived), which has the deletion-test vocabulary and the depth/seam concepts. Running them separately works, but the human must mentally join the two outputs. Merging is strictly additive.
  3. The deterministic improvements (idiom exemption, monorepo detection, ruff selectors, cycle detector, dup-signature detector, risk score, diff mode, scaffold-l3, HTML output) are each cheap on their own but cumulatively transform the report from "586 lines of stdout JSON" into "an HTML page categorized into Must/Should/Won't Fix that a tech lead can read on Monday morning." Shipping piecemeal would mean five separate releases with the noise still poisoning each interim run.

Goals & Non-Goals

Goals

  • Surface architectural issues regex cannot see — when --deep is passed, the deepening pass produces at least 3 findings classified leaky or shallow with valid (substring-grep-verified) evidence on book-companion-shaped backends. Discovery-capable: the subagent sees the whole module graph and picks its own targets (seeded with godmodule_candidates[] as a hint, not a constraint) — see Acceptance test.
  • Cut false-positive noise by ≥80% on a Typer-heavy backend — measured by total findings emitted on book-companion's backend/ before vs. after function-scoped idiom exemptions (F2) and U007 default-off (F3) land. Target: from 586 → ≤120 findings on the same code state.
  • Make findings prioritized by actionability, not by rule_id — every finding carries a disposition ∈ {must_fix, should_fix, wont_fix} (F5) replacing the legacy severity axis; within disposition, risk_score blends disposition weight + churn + coupling for ordering.
  • Ship a human-readable artifact — primary output is an HTML report (with MD sidecar and JSON sidecar) saved to {docs_path}/architecture/{YYYY-MM-DD}_<slug>.{html,md,json}, not JSON piped to stdout. Stderr collapses to a one-line summary: Wrote <path>: N Must Fix, M Should Fix, K Won't Fix in T files.
  • Stop silently misreporting monorepos — when scan root has no stack markers but a depth-2 child does, the user is informed and offered --monorepo fan-out (F4); zero silent stack-skips. Fan-out emits one HTML+MD+JSON triplet per stack plus a top-level index.{html,json}.
  • Close the cross-module visibility gap structurally — Python import cycles (F8) and duplicate function signatures (F9) surface as first-class findings via deterministic AST walkers, not requiring the LLM pass.
  • Make the report usable in CI — --since <ref> and --baseline <path> (F10) consume the JSON sidecar, letting CI emit "3 new findings on this PR" instead of "586 findings, ignore." When both flags are passed, they AND-intersect: filter to changed files AND emit only deltas vs baseline.
  • Collapse the L3 activation-energy barrier — --scaffold-l3 (F11) generates a starter .pmos/architecture/principles.yaml seeded from the current run's detected idioms, using the new disposition: key shape.

Non-Goals

  • NOT turning --deep on by default — because the LLM pass is non-deterministic, slow, and the mechanical pass alone delivers most of the value; opt-in keeps CI usage predictable.
  • NOT rewriting /improve-codebase-architecture or absorbing it wholesale — because that skill is owned in a different project namespace; we reuse its vocabulary (module / interface / implementation / depth / seam / adapter / leverage / locality / deletion test) for output consistency only.
  • NOT auto-fanning monorepos without explicit --monorepo — because the user may legitimately want to audit one stack in isolation; we warn loudly instead of silently fanning.
  • NOT writing new tools/ scripts in languages other than POSIX bash + Python 3 (already required prereqs) — because adding a new prereq breaks the install story.
  • NOT preserving back-compat for principles.yaml severity: keys — the rename to disposition: is breaking. Plugin-shipped rules rewrite in this release; user L3 files with legacy keys parse-error on first run with a message naming the exact key rename.
  • NOT writing JSON to stdout — stdout becomes empty (or carries the same one-line summary as stderr for shell-pipeline use). The JSON sidecar at {docs_path}/architecture/{date}_<slug>.json is the canonical machine-readable output.
  • NOT promoting findings to Nygard ADRs — ADR promotion was a workaround for JSON-on-stdout being unreadable. The HTML report is the artifact now. The --no-adr flag is removed.
  • NOT double-reporting — when the deep pass classifies a size-class finding's file (U001 / U002 / PY005 / PY006 only) as classification: deep, deletion_test.outcome: reappears, the mechanical finding is demoted to Won't Fix with deep_pass_cleared: true, not silently dropped. Other rule classes (dup signature, missing docstring, cycles, dead code) are unaffected by deep-pass clearance.
  • NOT crashing when no subagent/LLM is available — tools_skipped: ["deep_pass:no_subagent_tool_use"] is emitted and the run succeeds with mechanical findings only. Platform support for --deep is detected at runtime (subagent dispatch + tool-use probe); skill target stays generic.
  • NOT capping the deep-pass to a fixed candidate count — discovery-capable: the subagent picks its own targets from the full module graph; godmodule_candidates[] top-10 is a seed_hint[], not a hard seed list.
  • NOT retaining the byte-identical determinism guarantee — tools/check-determinism.sh is removed. Users self-check via --baseline against a prior run if they care about regression in findings set.

User Experience Analysis (Motivation / Friction / Satisfaction)

The "user" here is a developer running /architecture against their repo. The feedback was itself an MSF critique — F2/F3 are friction reductions, F5/F6/F11 are motivation enhancements, F1/F8/F9 are satisfaction improvements. F12 (stderr top-5) is dropped — superseded by the HTML report leading with the Must Fix section. This section makes the mapping explicit.

Motivation

  • Job to be done: "Tell me what's architecturally wrong with my codebase, ranked by what's most likely to bite me next month, so I can spend my Monday morning on the right thing — and give me a page I can share with my team."
  • Importance / urgency: High when the user is mid-refactor, onboarding, or evaluating tech debt for a planning cycle. Today's report fails this job — JSON on stdout, every U001 hit equally bad, so the developer either redirects to a file they later can't find or reads nothing (the actual behaviour).
  • Alternatives: (a) Run a paid SaaS like SonarQube / CodeClimate — heavyweight, language-coupled, doesn't follow the user's own principles. (b) Run ruff + dependency-cruiser by hand — what /architecture orchestrates today, but no ranking, no cross-module seams, no deletion test, no shareable artifact. (c) Just read the code — what an experienced engineer does, but expensive and inconsistent. /architecture's advantage is "your principles, your repo, your IDE, your shareable report" — but that advantage is squandered when the report is unreadable.

Satisfaction Signals

  • The developer opens the HTML report and the first section is Must Fix — they can name the top architectural risk in 10 seconds without scrolling.
  • When they run --deep, the report includes DEEP_LEAKY and DEEP_SHALLOW findings under Must Fix / Should Fix respectively, with a deletion-test rationale and a proposed-reshape paragraph — vocabulary they recognise from /improve-codebase-architecture.
  • The Python cycle detector (F8) catches a service → repo → service cycle that previously surfaced only at runtime as MissingGreenlet — surfaces as a Should Fix (warn-equivalent) by default.
  • The dup-signature detector (F9) flags _resolve_audio_units existing in two files — the highest-value finding in the user's example, surfaced by a 30-LOC AST walker.
  • Running on a 50M-LOC monorepo's backend/ subdir produces a meaningful report; running at the monorepo root produces a useful warning, not a silent zero-stack success; --monorepo emits per-stack triplets + a top-level index page.
  • The Won't Fix section is collapsed by default — preserving the audit trail of idiomatic exemptions and deep-pass clearances without crowding the main report.

Solution Direction

Four layers, each independently shippable and additive over the existing skill:

Layer 1 — Signal hygiene (F2, F3, F4, F7)

Wire framework idiom detection into the scan-time file classifier. For each file: detect Typer/Click/Fire/argparse imports, Alembic migrations path, protobuf *_pb2.py, pydantic BaseModel subclass. Then AST-walk to compute function-scoped exemption ranges: only print() inside functions decorated with @app.command / @click.command / @cli.command / @cli.group / @fire.Fire-equivalent get exempted from U004; helper-function prints in the same file still flag. Emit exempt_ranges[] per file under exemptions.idiomatic[]. Demote U007 to opt-in with hardened heuristic. Add depth-2 stack-marker walk with monorepo warning + --monorepo fan-out. Add 4 new ruff rules (PY005–PY008: complexity, branches, magic values, unused args).

Layer 2 — Prioritization + cross-module visibility (F5, F6, F8, F9)

Add a Python AST pass that, per module, counts fanin + fanout + public_symbols, hashes function signatures across files for duplicate detection, and walks the module graph + runs Tarjan's SCC for cycle detection. Compute per-finding risk_score using bounded-additive weights: {must_fix: 1000, should_fix: 100, wont_fix: 1}; churn (capped 50, 1 pt per commit in the configurable churn window, default 90 days; overridable via L3 risk_score.churn_window_days: N); coupling (capped 50, 5 pts per inbound import). Sum cap on churn+coupling is 100, strictly less than the gap between adjacent disposition weights — disposition class always dominates rank; churn/coupling break ties within class. Emit module_metrics[], godmodule_candidates[] (top-10 by fanin × public_symbols, no floor), and a cycles[] block. The HTML "Architecture metrics" appendix renders the top-5 godmodule_candidates; the JSON sidecar carries the top-10.

Layer 3 — Workflow (F10, F11)

--since <ref> filters findings to files in git diff --name-only <ref>...HEAD; --baseline <path> loads a prior JSON sidecar and emits a diff{} block (new / dropped / unchanged finding sets, keyed on (rule_id, file, line) tuple identity — added v2 fields like risk_score are not part of identity). Passing both AND-intersects. --baseline accepts either a single <stack>.json or a monorepo index.json (which requires the current run to be --monorepo; pairs stacks by name). --scaffold-l3 writes a starter .pmos/architecture/principles.yaml with detected idiomatic exemptions pre-populated (using the new disposition: key shape) and refuses to overwrite existing files.

Layer 4 — Deepening pass (F1)

New opt-in Phase 4.5 (between current Phase 4 evaluate-rules and Phase 5 reconcile-exemptions), gated by --deep. Discovery-capable design: the subagent receives the whole module graph + per-module summary stats (fanin/fanout/public_symbols/LOC) + the seed_hint[] (top-10 from godmodule_candidates[] as a starting suggestion, not a constraint) + the vocabulary cheatsheet — and picks its own targets using its tool access (Read/Grep) to pull source for whichever files it chooses. The orchestrator does not pre-bundle source.

The subagent's prompt template (verbatim shape, spec to refine): "Start with the seed_hint modules; expand to other modules in the graph if cross-module patterns lead you there. For each module you classify, return a JSON object below."

{
  "module": "<path>",
  "classification": "deep|shallow|leaky",
  "rationale": "<one line>",
  "deletion_test": { "outcome": "vanishes|reappears", "estimate": "<rationale>" },
  "cross_module_patterns": [
    { "kind": "dto-leak|dispatch-conditional|duplicated-decision|implicit-pipeline|responsibility-scatter",
      "evidence": "<quoted code or file paths>" }
  ],
  "proposed_reshape": "<one paragraph>",
  "affected_files": ["<path1>", "<path2>"]
}

Orchestrator validation contract: (a) each returned module path resolves to a real file in the repo; (b) every cross_module_patterns[].evidence is verbatim-substring-grep-able against the specific file it names; (c) classification ∈ enum. No seed-set check (seed_hint is advisory). Hard-fail on miss → emit warning, skip the deepening pass output for this run (tools_skipped: ["deep_pass:validation_failed"]), mechanical findings preserved.

Output integration: classifications promote to first-class findings — leaky → Must Fix with new rule_id DEEP_LEAKY; shallow → Should Fix with new rule_id DEEP_SHALLOW; deep → no finding (it's the absence of a problem). The subagent's rationale + proposed_reshape become the finding's message field. Raw subagent output is preserved verbatim under JSON-sidecar parent deep_pass{}:

{
  "deep_pass": {
    "dispatched_at": "<ISO-8601>",
    "seed_hint": ["<path1>", "<path2>", ...],
    "candidates": [ /* raw subagent objects, validated */ ],
    "skipped_reason": null | "no_subagent_tool_use" | "validation_failed" | "no_candidates"
  }
}

Reconciliation: when a size-class finding (U001 oversized file, U002 large class, PY005 complexity, PY006 branches — fixed set) on a file classified deep with deletion_test.outcome: reappears is detected, that finding is demoted to Won't Fix with deep_pass_cleared: true. It appears in the HTML's Won't Fix > "Cleared by deep pass" sub-section, audit-trail preserved. Other rule classes (U004 prints, U007 docstring, U011 dup-signature, PY009 cycles, L3 custom rules) are not affected by deep-pass clearance — a deep file can still have a real dup_signature.

Platform support: runtime-detected. Phase 4.5 probes the platform with a no-op subagent dispatch + Read probe; on failure, sets deep_pass.skipped_reason: "no_subagent_tool_use" and proceeds with mechanical pass only.

Output shape (replaces stdout-JSON behaviour)

Phase 6 of the existing SKILL.md (emit report) changes from "JSON to stdout" to "emit HTML+MD+JSON triplet to {docs_path}/architecture/":

  • Path: {docs_path}/architecture/{YYYY-MM-DD}_<slug>.{html,md,json}. {docs_path} resolves via .pmos/settings.yaml (default docs/pmos/). <slug> is derived from the scan-root basename (audit . → root, audit backend/ → backend); --label <slug> overrides.
  • Same-day collision: append -2, -3, ... to the slug component until unique (spec to confirm).
  • HTML: human-readable artifact emitted via the _shared/html-authoring/ substrate (style.css, viewer.js, kebab-case heading IDs, asset cache-bust). Sections: Must Fix (N) → Should Fix (N) → Won't Fix (N) — click to expand (collapsed <details> with three sub-headings: "Cleared by deep pass" / "Idiomatic exemptions" / "L3 explicit exemptions") → Architecture metrics appendix (top-5 godmodule_candidates, cycles[] if non-empty) → Run metadata (config, scan_root, generated_at).
  • MD: sidecar generated via the substrate's html-to-md.js; read-only.
  • JSON: machine-readable canonical for --baseline and --since. Top-level keys: config{}, scanned{}, findings[] (each with disposition, risk_score, rule_id, file, line, message, optional deep_pass_cleared), exemptions.idiomatic[], module_metrics[], godmodule_candidates[] (top-10), cycles[], monorepo_detected[], deep_pass{}, tools_errored[], tools_skipped[], coverage_gaps[], diff{} (when --baseline).
  • Monorepo fan-out: --monorepo emits one triplet per detected stack + a top-level {YYYY-MM-DD}_index.{html,json}. The index.json contains stacks: [{name, json_path, html_path}] with relative paths under the same date-stamped folder.
  • Stderr: one line — Wrote {docs_path}/architecture/{date}_<slug>.html: N Must Fix, M Should Fix, K Won't Fix in T files. Errors and warnings continue to surface on stderr above this line as before.
  • Stdout: empty (or the same one-line summary for shell-pipeline use; spec to decide).

Acceptance test

On book-companion (the user's repo, with summarizer_service.py 935 LOC, job_queue_worker.py 797 LOC, evaluator.py 745 LOC), /architecture audit . --monorepo --deep must:

  1. Detect both backend (Python) and frontend (TS) stacks via depth-2 monorepo walk and emit two per-stack triplets + a top-level index.
  2. Cut total findings count by ≥80% versus a v2.48.0 run on the same code (function-scoped idiom exemptions + U007 default-off).
  3. Emit an HTML report whose first section is Must Fix, ordered by risk_score, with the three large service files surfacing as Must or Should Fix findings.
  4. Under deep_pass.candidates[]: at least 3 findings classified leaky or shallow with valid (substring-grep-verified) evidence. The previously-named patterns (DTO leak in quiz, duplicated _resolve_audio_units, missing BookIngestionPipeline) are illustrative targets the discovery-capable subagent typically surfaces, but not contractually required by name.
  5. The Won't Fix section in the HTML is collapsed by default and carries the idiomatic exemption rows for Typer-decorated functions.

User Journeys

Primary journey — quick local audit on a Typer-heavy backend

  1. Developer runs /architecture audit backend/.
  2. Idiom detection runs at scan time; Typer-decorated functions in backend/cli/ get function-scoped U004 exemption.
  3. Stderr emits: scanned 47 files (1 stack: python); 12 idiomatic exemption ranges applied.
  4. L1/L2 rules evaluate; PY005 (C901) flags two complexity warnings.
  5. Layer 2 AST pass computes module_metrics; top-10 god-module candidates surfaced as seed_hint[] material and HTML-appendix content.
  6. HTML+MD+JSON triplet written to docs/pmos/architecture/2026-05-13_backend.{html,md,json}.
  7. Stderr emits the single summary line. Developer opens the HTML, reads the Must Fix section, opens the top file, makes the fix.

Alternate journey — monorepo audit

  1. Developer runs /architecture audit . at the monorepo root (no stack markers at root).
  2. Depth-2 walk detects backend/pyproject.toml + frontend/package.json.
  3. Stderr warns: No stack markers at scan root; found backend/pyproject.toml, frontend/package.json. Re-run with --monorepo to fan out, or audit each stack directly.
  4. Developer re-runs with --monorepo; two triplets emit (one per stack: {date}_backend.{html,md,json}, {date}_frontend.{html,md,json}) plus a combined {date}_index.{html,json}. Stderr summary lists both triplets.

Alternate journey — CI delta mode

  1. CI runs /architecture audit . --since origin/main --baseline docs/pmos/architecture/2026-05-12_root.json on a PR branch.
  2. Findings filtered to files in git diff --name-only origin/main...HEAD AND intersected with deltas vs baseline.
  3. Stderr emits: Wrote docs/pmos/architecture/2026-05-13_root.html: 3 Must Fix (new), 0 Should Fix (new), 0 Won't Fix (new); 0 dropped; 583 unchanged on base.
  4. CI fails the PR if diff.new contains any must_fix-disposition finding.

Alternate journey — deepening pass on legacy code

  1. Tech lead runs /architecture audit . --deep before a major refactor.
  2. Mechanical pass completes; Layer 2 produces top-10 godmodule_candidates → seed_hint[].
  3. Phase 4.5 probes platform (subagent + Read tool); on Claude Code, success.
  4. Subagent receives the module graph + seed_hint + vocab cheatsheet; picks its own targets; pulls source via Read for inspected files; returns classifications.
  5. Orchestrator validates (path exists, evidence verbatim-grep, classification enum); leaky/shallow classifications promote to DEEP_LEAKY / DEEP_SHALLOW findings under Must / Should Fix. Raw output preserved in deep_pass.candidates[].
  6. U001-flagged files classified deep, reappears demote to Won't Fix > "Cleared by deep pass".
  7. Tech lead opens the HTML, reads the Must Fix section including the DEEP_LEAKY entries with proposed_reshape paragraphs, decides which to action.

Error / edge journeys

  • No subagent or no Read tool access on --deep: emit deep_pass.skipped_reason: "no_subagent_tool_use"; mechanical pass runs to completion; exit 0.
  • Subagent validation fails (evidence not verbatim in source, classification out-of-enum, module path doesn't resolve): emit deep_pass.skipped_reason: "validation_failed" with detail; mechanical findings preserved; exit 0.
  • --scaffold-l3 with existing file: stderr refusing to overwrite existing .pmos/architecture/principles.yaml; exit 64.
  • --baseline against a schema_version: 1 (legacy) report: parse-error — baseline file uses legacy schema_version 1; v1 reports used `severity`, not `disposition`. Regenerate the baseline with the current /architecture or migrate manually.
  • User L3 principles.yaml with legacy severity: keys: parse-error at Phase 1 load — principles.yaml uses legacy 'severity:' key; rename to 'disposition:' (block→must_fix, warn→should_fix, info→wont_fix). No automatic migration; CHANGELOG documents the rename. Exit 64.
  • Cycle detector finds 0 cycles: emit empty cycles: []; no finding rows.
  • Scan root is not a git repo: risk_score's churn weight is 0 (graceful); --since/--baseline not usable (stderr warning + exit 64 if either flag was passed).
  • Same-day re-run produces a slug collision: append -2, -3, ... to slug; spec to confirm exact behaviour.
  • --baseline index.json but current run is not --monorepo: stderr error index baseline requires --monorepo on current run; aborting; exit 64.

Learnings

New learnings this session:

  • Deep grilling exposes premise-level holes. The original doc framed the deep-pass as "classify a fixed set of Layer-2 candidates." Question 1 (token budget) revealed the framing was already weak; the user's clarifying question ("where do candidates come from?") triggered the pivot to discovery-capable. Takeaway: grill leverage is highest when the questioner can re-pose after a user clarification — don't treat the user's pushback as just an answer to the question asked.
  • JSON-on-stdout is a default that survives because nobody questions it. Asking "how do we save the output?" exposed: there was no save mechanism; users were piping stdout with no convention; F10 (--baseline) loads a file users have no documented way to produce. Takeaway: when adding load-side flags (--baseline, --since), audit the save-side first — implicit contracts rot.
  • Workaround features cluster. ADR promotion, F12 stderr top-5, and the determinism check were all workarounds for the JSON-on-stdout / severity-axis era. Replacing the artifact shape collapsed three features at once. Takeaway: when a doc has a load-bearing primitive that's a workaround, audit features that depend on it — many may drop together.
