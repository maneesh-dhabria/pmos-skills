/architecture — deepening pass + hygiene — Requirements

Date: 2026-05-13 · Last updated: 2026-05-13 · Status: Draft · Tier: 3 — Skill feedback (12 findings) · Mode: interactive · Pipeline: /feature-sdlc skill --from-feedback · Target skill: plugins/pmos-toolkit/skills/architecture/ · Target platform: generic · Acceptance criteria reference: plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§F

Problem

The /architecture skill is a fast, deterministic rule engine that runs L1 universal (≤15 rules) + L2 stack-specific (TS via dependency-cruiser, Python via ruff) + L3 per-repo overrides against a scan root and emits a JSON report. In its current shape (v2.48.0) it catches hygiene symptoms — oversized files, print() in src, dead comments, long arg lists — but it cannot surface the architectural issues that actually matter: shallow modules, cross-module seam leaks, dispatch-by-conditional chains, implicit pipelines, responsibility scatter across CLI + service + route. These require reading code, following call graphs, and applying a deletion test — not regex matching.

On a real Python backend (book-companion, ~900 LOC service files), one run produced 586 findings dominated by 218 print() false positives (all legitimate Typer CLI commands) and 710 info-noise findings from U007 missing-purpose-comment wallpaper. The 14 oversized-file flags are real but undifferentiated — every U001 hit looks equally bad. Worse: running the audit at the monorepo root silently skipped both stacks because pyproject.toml and package.json live in subdirs.

Who experiences this?

The developer running /architecture audit . on a real repo — typically a senior engineer or tech lead trying to audit a codebase for architectural health before a refactor, during onboarding, or as a periodic check. Concrete persona from the feedback: the maintainer of book-companion (~900 LOC backend Python service files), running the audit to find architectural problems but receiving an unranked sea of false-positive hygiene flags that swamps the real signal.

Why now?

Three triggers converged this week:

  1. A real user ran /architecture audit . on book-companion and produced the 586-finding report that motivated this feedback. The single run made the noise/signal ratio undeniable.
  2. The user already runs a sibling skill, /improve-codebase-architecture (Matt Pocock-derived, at ~/Desktop/Projects/personal/book-companion/.claude/skills/), which has the deletion-test vocabulary and the depth/seam concepts. Running them separately works, but the human must mentally join the two outputs. The seam list (oversized + high-fanout files) lives in /architecture's report; the depth analysis lives in the other skill. Merging them is strictly additive.
  3. The deterministic improvements (idiom exemption, monorepo detection, ruff selectors, cycle detector, dup-signature detector, risk score, diff mode, scaffold-l3, top-5 stderr) are each cheap on their own but cumulatively transform the report from "586 lines of noise" into "3 ranked risks to fix this week." Shipping them piecemeal would mean five separate releases with the noise still poisoning each interim run.

Goals & Non-Goals

Goals

  • Surface architectural issues regex cannot see — when --deep is passed, the skill identifies at least the route↔service DTO leak, the duplicated function across two callers, and the missing pipeline orchestrator on book-companion's backend — measured by acceptance test (see Acceptance test).
  • Cut false-positive noise by ≥80% on a Typer-heavy backend — measured by total findings emitted on book-companion's backend/ before vs. after idiom exemptions (F2) and U007 default-off (F3) land. Target: from 586 → ≤120 findings on the same code state.
  • Make findings prioritized, not ranked-by-rule-id — every finding carries a risk_score (F5) blending severity + churn + coupling; the stderr summary surfaces the top 5 risks ranked (F12) rather than counts alone.
  • Stop silently misreporting monorepos — when scan root has no stack markers but a depth-2 child does, the user is informed and offered --monorepo fan-out (F4); zero silent stack-skips.
  • Close the cross-module visibility gap structurally — Python import cycles (F8) and duplicate function signatures (F9) surface as first-class findings via deterministic AST walkers, not requiring the LLM pass.
  • Make the report usable in CI — --since <ref> and --baseline <path> (F10) let CI emit "3 new findings on this PR" instead of "586 findings, ignore."
  • Collapse the L3 activation-energy barrier — --scaffold-l3 (F11) generates a starter .pmos/architecture/principles.yaml seeded from the current run's detected idioms, so users get an actionable starter config out of one command instead of hand-authoring YAML against a 586-finding report.
  • Preserve the deterministic-fast guarantee — the mechanical pass (L1/L2/L3 + new deterministic checks F2/F3/F4/F5/F6/F7/F8/F9) remains the baseline; --deep is strictly opt-in and never gates the mechanical pass — measured by the existing tools/check-determinism.sh still passing.

Non-Goals

  • NOT turning --deep on by default — because the LLM pass is non-deterministic, slow, and the mechanical pass alone delivers most of the value; opt-in keeps CI usage predictable.
  • NOT rewriting /improve-codebase-architecture or absorbing it wholesale — because that skill is owned in a different project namespace; we reuse its vocabulary (module / interface / implementation / depth / seam / adapter / leverage / locality / deletion test) for output consistency only.
  • NOT auto-fanning monorepos without explicit --monorepo — because the user may legitimately want to audit one stack in isolation; we warn loudly instead of silently fanning.
  • NOT writing new tools/ scripts in languages other than POSIX bash + Python 3 (already required prereqs) — because adding a new prereq breaks the install story.
  • NOT changing the JSON output's existing top-level shape — schema_version bumps 1 → 2 additively; old fields preserved; old readers still parse.
  • NOT exceeding the 10-candidate cap on the deep pass per run — because the LLM cost scales linearly with candidates and the top-10 by fanin × public_symbols is sufficient signal density.
  • NOT double-reporting — when the deep pass classifies a U001-flagged file as "deep, no reshape," the mechanical U001 finding is demoted to info (with a deep_pass_cleared: true annotation), not silently dropped.
  • NOT crashing when no subagent/LLM is available — tools_skipped: ["deep_pass"] is emitted and the run succeeds.

User Experience Analysis (Motivation / Friction / Satisfaction)

The "user" here is a developer running /architecture against their repo. The feedback was itself an MSF critique — F2/F3/F12 are friction reductions, F5/F6/F11 are motivation enhancements, F1/F8/F9 are satisfaction improvements. This section makes that mapping explicit.

Motivation

  • Job to be done: "Tell me what's architecturally wrong with my codebase, ranked by what's most likely to bite me next month, so I can spend my Monday morning on the right thing."
  • Importance / urgency: High when the user is mid-refactor, onboarding, or evaluating tech debt for a planning cycle. Today's report fails this job — it lists every U001 hit as equally bad, so the developer either reads everything (impossible) or reads nothing (the actual behaviour).
  • Alternatives: (a) Run a paid SaaS like SonarQube / CodeClimate — heavyweight, language-coupled, doesn't follow the user's own principles. (b) Run ruff + dependency-cruiser by hand — what /architecture orchestrates today, but no ranking, no cross-module seams, no deletion test. (c) Just read the code — what an experienced engineer does, but expensive and inconsistent. /architecture's advantage is "your principles, your repo, your IDE" — but that advantage is squandered when the report is unreadable.

Satisfaction Signals

  • The developer can name the top architectural risk in their codebase within 10 seconds of reading stderr — because the top-5 punch-list is the first thing they see.
  • When they run --deep, they get back a classification (deep / shallow / leaky) per oversized module with a deletion-test rationale — vocabulary they recognise from /improve-codebase-architecture.
  • The Python cycle detector (F8) catches a service → repo → service cycle that previously surfaced only at runtime as MissingGreenlet.
  • The dup-signature detector (F9) flags _resolve_audio_units existing in two files — the highest-value finding in the user's example, surfaced by a 30-LOC AST walker.
  • Running on a 50M-LOC monorepo's backend/ subdir produces a meaningful report; running at the monorepo root produces a useful warning, not a silent zero-stack success.

Solution Direction

Three layers, each independently shippable and additive over the existing skill:

Layer 1 — Signal hygiene (F2, F3, F4, F7)

Wire framework idiom detection into the scan-time file classifier. For each file: detect Typer/Click/Fire/argparse imports, Alembic migrations path, protobuf *_pb2.py, pydantic BaseModel subclass — apply rule-specific file-level exemptions and surface under exemptions.idiomatic[]. Demote U007 to opt-in. Add depth-2 stack-marker walk with monorepo warning + --monorepo fan-out. Add 4 new ruff rules (PY005–PY008: complexity, branches, magic values, unused args).

Layer 2 — Prioritization + cross-module visibility (F5, F6, F8, F9, F12)

Add a Python AST pass that, per module, counts fanin + fanout + public_symbols, hashes function signatures across files for duplicate detection, and walks the module graph + runs Tarjan's SCC for cycle detection. Compute per-finding risk_score = severity_weight + churn(git log -90d) + coupling(fanin). Emit module_metrics[], godmodule_candidates[] (top-5 by fanin × public_symbols), and a cycles[] block. Re-rank stderr top-5 by risk_score.

Layer 3 — Workflow (F10, F11)

--since <ref> filters findings to files in git diff --name-only <ref>...HEAD; --baseline <path> loads a prior report and emits a diff{} block (new / dropped / unchanged finding sets). --scaffold-l3 writes a starter .pmos/architecture/principles.yaml with detected idiomatic exemptions pre-populated and refuses to overwrite existing files.

Layer 4 — Deepening pass (F1)

New opt-in Phase 4.5 (between current Phase 4 evaluate-rules and Phase 5 reconcile-exemptions), gated by --deep. Seed list = top-10 modules by Layer 2's fanin × public_symbols product (god-module candidates). Dispatch a single subagent (Task tool or platform equivalent) with: per-candidate file content, declared L1 findings on that file, fanin / fanout / public_symbols, and the /improve-codebase-architecture vocabulary cheatsheet. Subagent returns one JSON object per candidate:

{
  "module": "<path>",
  "classification": "deep|shallow|leaky",
  "rationale": "<one line>",
  "deletion_test": { "outcome": "vanishes|reappears", "estimate": "<rationale>" },
  "cross_module_patterns": [
    { "kind": "dto-leak|dispatch-conditional|duplicated-decision|implicit-pipeline|responsibility-scatter",
      "evidence": "<quoted code or file paths>" }
  ],
  "proposed_reshape": "<one paragraph>",
  "affected_files": ["<path1>", "<path2>"]
}

Orchestrator validates: each returned module is in the seed set; each cross_module_patterns[].evidence is verbatim-substring-grep-able against the named files; classification ∈ enum. Hard-fail on miss → emit warning, skip the deepening pass for this run (tools_skipped: ["deep_pass:validation_failed"]). Pass → emit under top-level deepening_candidates[].

Reconciliation: when a U001-flagged file is classified classification: deep with deletion_test.outcome: reappears, mark the corresponding U001 finding as deep_pass_cleared: true and demote its severity to info. The mechanical finding stays in findings[] (audit-trail), just demoted.

Acceptance test

On book-companion (the user's repo, with summarizer_service.py 935 LOC, job_queue_worker.py 797 LOC, evaluator.py 745 LOC), /architecture audit . --monorepo --deep must:

  1. Detect both backend (Python) and frontend (TS) stacks via depth-2 monorepo walk.
  2. Cut total findings count by ≥80% versus a v2.48.0 run on the same code (idiom exemptions + U007 default-off).
  3. Surface a top-5 stderr punch-list ranked by risk_score, with the three large service files in the top 5.
  4. Surface, under deepening_candidates[]: (a) the route↔service DTO leak in quiz, (b) the duplicated _resolve_audio_units across api/routes/audio.py and services/job_queue_worker.py, (c) the missing BookIngestionPipeline orchestrator (responsibility scatter).
  5. Pass tools/check-determinism.sh (mechanical pass is byte-identical run-to-run; deepening pass is allowed non-deterministic, recorded under a separate determinism boundary).

User Journeys

Primary journey — quick local audit on a Typer-heavy backend

  1. Developer runs /architecture audit backend/.
  2. Idiom detection runs at scan time; Typer files in backend/cli/ auto-exempted from U004.
  3. Stderr emits: scanned 47 files (1 stack: python); 12 idiomatic exemptions applied.
  4. L1/L2 rules evaluate; PY005 (C901) flags two complexity warnings.
  5. Layer 2 AST pass computes module_metrics; top-5 god-module candidates surfaced.
  6. Stderr emits Top 5 risks (by risk_score): ranked punch-list with file + rule + commits/qtr + inbound.
  7. Developer reads the top entry, opens that file, makes the fix.

Alternate journey — monorepo audit

  1. Developer runs /architecture audit . at the monorepo root (no stack markers at root).
  2. Depth-2 walk detects backend/pyproject.toml + frontend/package.json.
  3. Stderr warns: No stack markers at scan root; found backend/pyproject.toml, frontend/package.json. Re-run with --monorepo to fan out, or audit each stack directly.
  4. Developer re-runs with --monorepo; two reports emit (one per stack) plus a combined index.

Alternate journey — CI delta mode

  1. CI runs /architecture audit . --since origin/main on a PR branch.
  2. Findings filtered to files in git diff --name-only origin/main...HEAD.
  3. Stderr emits: 3 new findings on this PR; 0 dropped; 583 unchanged on base.
  4. CI fails the PR if diff.new contains any block-severity finding.

Alternate journey — deepening pass on legacy code

  1. Tech lead runs /architecture audit . --deep before a major refactor.
  2. Mechanical pass completes; Layer 2 produces god-module candidates.
  3. Phase 4.5 dispatches subagent with top-10 candidates.
  4. Subagent returns classifications; orchestrator validates evidence substrings against source.
  5. JSON report carries deepening_candidates[] with classification, deletion-test outcome, cross-module patterns, proposed reshape.
  6. Stderr summary appends: Deepening: 3/10 modules classified leaky — see deepening_candidates[] in report.
  7. Tech lead reads the proposed reshape paragraphs, decides which to action.

Error / edge journeys

  • No subagent tool available on --deep: emit tools_skipped: ["deep_pass:no_subagent_tool"]; mechanical pass runs to completion; exit 0.
  • Subagent validation fails (evidence not verbatim in source, classification out-of-enum, module not in seed set): emit tools_skipped: ["deep_pass:validation_failed"] with detail; mechanical pass output preserved; exit 0.
  • --scaffold-l3 with existing file: stderr refusing to overwrite existing .pmos/architecture/principles.yaml; exit 64.
  • --baseline against a schema_version: 1 report: load it, treat absent risk_score as 0; diff still works.
  • Cycle detector finds 0 cycles: emit empty cycles: []; no finding rows.
  • Scan root is not a git repo: risk_score's churn weight is 0 (graceful); --since/--baseline not usable (stderr warning + exit 64 if flag was passed).

Design Decisions

#DecisionOptions consideredRationale
D1Deepening pass dispatches ONE subagent for all top-10 candidates, not 10 parallel subagents.(a) One subagent, batched candidates. (b) N parallel subagents, one per candidate. (c) Sequential subagents.Batched single subagent: lower coordination cost, deterministic candidate-count cap, shared context across candidates (helps cross-module pattern detection in F1). Parallel would be faster but cross-module patterns require seeing N modules together. Sequential adds latency with no upside.
D2risk_score weights: severity {block:100, warn:10, info:1}, churn capped at 50 (1pt/commit max 50), coupling capped at 50 (5pt/inbound import max 50). Range 1–200.(a) These caps. (b) Unbounded weights. (c) Multiplicative formula.Bounded additive keeps the score interpretable (severity always dominates rank within a class; churn/coupling break ties). Multiplicative would let a single hot file dominate. Unbounded would make scores hard to compare across runs.
D3F9 (duplicate signatures) is U011, severity warn, delegate_to grep (special-cased AST in harness).(a) New U011 grep-class rule. (b) New ruff rule wrapper (not possible — ruff has no cross-file signature check). (c) Separate top-level duplicates[] block.U011 fits the existing rule-emission shape; delegate_to: grep with a per-file AST under the hood matches how U001/U002 already special-case in run-audit.sh. Keeps the report schema flat.
D4F8 (Python cycles) is PY009, severity block, delegate_to cycle-py (new tools/ script).(a) New PY009 with custom delegate. (b) Reuse PY rules via ruff plugin. (c) New top-level cycles[] block.Block-severity matches dependency-cruiser's TS001 cycle rule; consistency across stacks. New delegate_to value documented; new tools/cycle-py.py ~50 LOC.
D5JSON schema bump: schema_version: 1 → 2. Additive only.(a) Bump to 2 additive. (b) Stay at 1, add fields without bump. (c) Bump to 2 with breaking removals.Additive bump signals new fields exist without forcing readers to upgrade. --baseline must accept v1 reports — handled by null-coalescing missing fields. Breaking removals would invalidate existing CI tooling.
D6Idiom exemptions are surfaced under exemptions.idiomatic[], NOT silently dropped from findings.(a) Surface under exemptions.idiomatic. (b) Silently exclude. (c) Emit as info-severity findings with idiom note.Audit principle: every dropped finding is visible. exemptions.idiomatic[] carries {rule_id, file, idiom, reason} so the user can retire an exemption if the idiom assumption is wrong.
D7U007 default behaviour: OFF. Enable via --include-info-comments flag OR L3 opt-in (rules.U007.severity: info in L3 yaml as explicit opt-in).(a) Default-off, opt-in. (b) Default-on, harden heuristic. (c) Remove the rule.The rule's intent (cold-reader legibility) is sound; the default-on behaviour poisons the report. Default-off + harden-when-on (only fire on file > 100 LOC with no module docstring) is the best of both. Removing it loses the legibility signal entirely.
D8Monorepo mode is explicit opt-in via --monorepo. Without the flag, depth-2 child stacks trigger a stderr warning, NOT silent fan-out.(a) Warn-only by default; explicit fan-out. (b) Auto-fan when stacks below root exceed 1. (c) Always require the flag (no warning).The user may legitimately want to audit one subdir; silent fan-out would scan paths they didn't ask for. The warning closes the silent-failure mode; the flag confirms intent.
D9The deepening pass is a new Phase 4.5 in SKILL.md (between current Phase 4 evaluate-rules and Phase 5 reconcile-exemptions). Numbered to preserve all downstream cross-refs.(a) Phase 4.5. (b) Renumber 5→6, 6→7. (c) Append as Phase 7.Per a documented learning (2026-05-13): inserting a new phase mid-pipeline in an existing skill, use a fractional number rather than renumbering. Cheaper, safer, preserves audit-trail.
D10The vocabulary cheatsheet handed to the deep-pass subagent is bundled as reference/deepening-vocabulary.md, derived from /improve-codebase-architecture (Matt Pocock-derived), cited by URL in the new reference file's header.(a) New reference file in /architecture's reference/. (b) Reference the external skill at runtime. (c) Inline the vocabulary in SKILL.md.Bundling keeps the skill self-contained (the external skill is in a different project namespace and may not be installed). Citing the source via URL preserves attribution. Inlining bloats SKILL.md past the soft cap.
D11The deep-pass subagent validation hard-fails on evidence-not-verbatim or classification-out-of-enum; the run continues with mechanical findings intact and tools_skipped[] records the validation failure.(a) Hard-fail with run continuation. (b) Soft-fail (accept and warn). (c) Hard-fail with whole-run abort.Mirrors the FR-44 pattern in /feature-sdlc Phase 6a: the reviewer (here the deep-pass subagent) gets validated by the orchestrator; on miss, the LLM half is skipped but the deterministic half still ships. Abort would lose the mechanical findings — unacceptable.
