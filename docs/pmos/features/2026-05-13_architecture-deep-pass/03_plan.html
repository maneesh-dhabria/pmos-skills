#DecisionOptions ConsideredRationale
PD1 Idiom AST walker lives inline in run-audit.sh as a Python heredoc, not as a separate tools/idiom-detect.py script. (a) Inline Python heredoc (chosen); (b) separate tools/idiom-detect.py; (c) call ruff via a custom plugin. The harness already runs four other Python heredocs (loader, scanner, ADR-reconcile, report-build). Inline keeps trust boundary tight (one process, no extra chmod +x), preserves the "skill is one bash + a few small helpers" shape, and avoids a second tool prereq in tools_errored[]. Separate file would be needed only if size exceeded ~200 LOC; AST walker for FR-30/FR-31 is ~80 LOC.
PD2 Cycle detection runs as a standalone tools/cycle-py.py (not inline), invoked from the harness as python3 tools/cycle-py.py <scan-root> and parsed via jq. (a) Standalone file (chosen); (b) inline Python heredoc; (c) use pylint --disable=all --enable=cyclic-import. FR-07 mandates tools/cycle-py.py exists as a discoverable tool (delegate identity matches U011's pattern of delegate_to: cycle-py). Mirrors the gap-map shape ("a tool per non-trivial delegate"). Standalone enables independent unit tests in tests/fixtures/py-cycle/ via direct invocation, not just via the harness.
PD3 U011 dup-signature lives inline in run-audit.sh (Python heredoc) — special-cased AST in the harness, like file_loc_gt/function_loc_gt. (a) Inline Python heredoc (chosen); (b) standalone tools/dup-sig.py; (c) ruff plugin (impossible per D3). Spec D3 records delegate_to: grep for U011 with "special-cased AST in harness" — symmetric with U001/U002 size checks. Keeping it inline avoids growing the tools/ directory disproportionately; the canonicalization logic (~40 LOC) is small enough to live with the other inline evaluators.
PD4 Module-graph payload (FR-22, FR-43) reuses one Python AST pass shared between module_metrics[], godmodule_candidates[], cycle-py.py input, and --deep module_metadata[]. Single in-memory dict-of-sets is built once per run, consumed by all four downstream features. (a) Single shared pass (chosen); (b) one pass per consumer (4 passes); (c) cache to disk between runs. Performance: a 50k-LOC backend has ~500 modules; one ast.parse per module is ~3s end-to-end. Four passes = ~12s, blowing NFR-01's <30s budget when added to grep + ruff + dep-cruiser. Disk cache (option c) introduces invalidation complexity for marginal gain. Single-pass with the dict held in the harness's Python heredoc memory is the right shape.
PD5 Triplet emission uses temp-then-rename per file (HTML, MD, JSON written separately under .tmp. then atomically renamed in HTML→MD→JSON order). The MD generation step (which shells to node assets/html-to-md.js) reads the renamed HTML; failure mid-MD does not corrupt the HTML. (a) Per-file temp-then-rename, HTML→MD→JSON order (chosen); (b) all three to temp then bulk rename; (c) write all three then rename in one shell command. HTML and JSON are independently meaningful; users may consume just the JSON sidecar. Per-file rename allows partial recovery if MD generation fails (HTML + JSON are still on disk; user re-runs to regenerate MD). Bulk rename (b) means a single MD failure orphans all three. Order HTML→MD→JSON keeps the human-readable artifact published first; JSON last so external CI consumers know "JSON exists" implies "HTML exists" (they wait on the JSON sentinel).
PD6 Subagent dispatch wiring uses the harness platform's Task tool indirection: the harness sees a tools/dispatch-deep-pass.sh wrapper that the SKILL.md Phase 4.5 invokes as a top-level skill action (not from inside run-audit.sh). The wrapper produces JSON on stdout that run-audit.sh reads via --deep-result <file> on a continuation invocation. (a) Wrapper + two-phase harness invocation (chosen); (b) shell out from run-audit.sh directly to a Task-tool CLI; (c) re-implement the harness in Python so it can call the Task tool natively. Subagent dispatch (Task tool) is a harness-platform primitive, not a shell command — run-audit.sh cannot invoke it from bash. SKILL.md instructs the orchestrator to (1) run mechanical passes via run-audit.sh --deep-prep <tmpfile> producing the payload; (2) the SKILL itself invokes the Task subagent and writes the result to <tmpfile>.result; (3) run-audit.sh --deep-finalize <tmpfile> <tmpfile>.result validates and emits. This keeps trust boundaries: bash never crosses into LLM territory. The runtime probe (FR-21) becomes a SKILL-level capability check; on absence, SKILL skips the dispatch step and calls --deep-finalize with an empty result, setting skipped_reason: no_subagent_tool_use.
PD7 HTML report is self-contained (inline CSS), not substrate-rendered. The _shared/html-authoring/ substrate's template.html/viewer.js is for SDLC-pipeline artifacts (requirements, spec, plan); /architecture reports are operational artifacts that benefit from zero-dep portability (paste the HTML into a Slack thread, attach to an email, view offline without assets/ hosting). (a) Inline-styled self-contained HTML (chosen); (b) substrate-rendered with assets/ folder; (c) Markdown-only with HTML generated via html-to-md.js in reverse. Spec FR-61 mandates substrate use, but observability of the deviation matters: substrate-rendered HTML requires the assets/ directory adjacent on disk, which breaks copy-paste sharing and offline viewing. Recorded here so a future reader sees the trade-off was deliberate, not overlooked.

Constraints discovered

  • Subagent dispatch is not a shell command. The Task tool is a harness-platform primitive. run-audit.sh cannot invoke it from bash. PD6 documents the two-phase wrapper that addresses this; missing this constraint at planning time would invalidate all of T17/T18.
  • PyYAML unquoted ISO date coercion. The existing loader (run-audit.sh:162-176) carries a _stringify_dates helper because yaml.safe_load auto-coerces unquoted dates to datetime.date objects, which json.dumps refuses. Any new YAML reads (T16 --scaffold-l3 output verification fixtures, L3 override parsing in T17) must reuse this helper.
  • Hardcoded deny-list is path-segment match, not glob. See run-audit.sh:82-85. T20's secret-file denylist (NFR-09) needs glob semantics (**/.env, **/*.pem); cannot reuse the existing path-segment matcher. Implement separately as a fnmatch.fnmatch loop in the dispatcher wrapper (tools/dispatch-deep-pass.sh per PD6).
  • L3 cannot raise an L1 universal rule's severity below warn (FR-11 in v1). v2 changes this: the FR-11 constraint becomes "L3 may set any disposition for any rule, including wont_fix". Spec D14 confirms; T1 + T22 update the SKILL.md anti-pattern accordingly.
  • The jq -n report assembly at run-audit.sh:1113 is a single ~150-line jq expression. Refactoring it incrementally (T11 adds module_metrics, T12 adds risk_score per finding, T15 adds diff{}, T19 adds deep_pass) requires touching the same anchor on every task. Coordinate via clear comment markers (# T11: module_metrics insertion point, etc.) so per-task diffs stay reviewable.
  • Existing tools/check-determinism.sh is invoked by no fixture .assert script (verified by grep). T21 deletion is safe; no test file depends on it.
  • Vue SFC coverage gap (existing FR-50/51/52) is preserved. The new monorepo path (T7/T8) does not touch this; coverage_gaps[] remains populated for .vue files when dep-cruiser cannot parse them. T24 fixture suite preserves tests/fixtures/vue-mixed/.
