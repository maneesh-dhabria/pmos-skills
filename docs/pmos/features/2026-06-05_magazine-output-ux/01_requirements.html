Problem

A magazine issue today is a single trustworthy-but-undifferentiated grid of text cards. The eye has no pre-attentive cues (newsletter vs podcast look identical), no sense of catch-up cost, no progress tracking, and no reading-shape choice. Long issues read like a wall, so the user skims less than they could and is tempted back to app-hopping. The fix is visual hierarchy + lightweight reading affordances — without breaking the digest's core promise: a self-contained file that works offline from file://.

Target user & JTBD

Who: the magazine owner clearing a feed backlog in a dedicated catch-up session, opening the latest issue from disk (often offline).

JTBD: skim and move through an issue pleasantly and fast, finishing it and trusting what I read — instead of bouncing between apps.

Resolved decisions

  • D1 — Layout default: Grid is the default view; Grid/Carousel/Listicle all available at any width; the chosen view persists in localStorage. Grid is the no-JS fallback.
  • D2 — Filter bar: dropdowns replace the checkbox/chip wall (feed multi, tag multi, type, read-state) + a live result counter; date range retained.
  • D3 — Visual intensity: subtle/restrained — thin left-border accent + small feed dot for type/source; soft pastel tag tints; minimal chrome. Skim speed beats decoration.
  • D4 — Read-state storage: localStorage keyed by issue_date + guid; device-local is accepted (per the brief's open question).
  • D5 — Scope: Wave 1 only (8 render-only variants). Waves 2–3 deferred.

Functional requirements

  • FR-1 (View-switcher): the issue offers Grid / Carousel / Listicle via a header control. Carousel shows one card at a time with ◀▶ buttons and arrow-key nav. The selected view persists across reloads (localStorage: mag:view). Grid renders with no JS.
  • FR-2 (Read-state): every card carries a stable data-guid. The user can mark a card read (a per-card control and the m key); read cards are visually de-emphasized. A header counter shows "N of M left"; a "hide read" toggle removes read cards from view. State persists (localStorage, keyed by issue date + guid).
  • FR-3 (Keyboard skim): j/k move the focused card next/prev, o opens the focused card's link, f focuses the filter, m toggles read on the focused card. In Carousel, ←/→ navigate.
  • FR-4 (Type color-coding): each card carries data-type; newsletter and podcast get distinct subtle treatments (left-border accent). A small per-feed dot uses a deterministic hue derived from the feed name (stable across issues).
  • FR-5 (Catch-up budget): the header shows total skim time (sum of newsletter reading minutes) and total audio time (sum of podcast lengths), parsed from reading_time. Each portion is omitted gracefully when no data is present.
  • FR-6 (Consistent tag colors): each tag chip gets a deterministic pastel color (hash of the tag → hue), identical for the same tag across all issues.
  • FR-7 (Uniform cards): bullet lists show the first 3 by default with a "show more / show less" toggle, keeping grid card heights even. With JS off, all bullets show.
  • FR-8 (Dropdown filter + counter): the filter bar uses dropdowns — feed (multi), tag (multi), type, read/unread — plus the existing date range, and a live "showing X of Y" counter. Filtering preserves the current semantics: AND across dimensions, OR within a multi-select.

Non-functional requirements

  • NFR-1 (Offline/self-contained): the issue remains ONE self-contained HTML file with inline CSS+JS, zero npm deps, working from file:// with no server (preserves the existing NFR-3). No external requests, no remote assets.
  • NFR-2 (Progressive enhancement): with JavaScript disabled, the grid renders, all cards + bullets are visible, and read/listen links work. All new affordances are enhancements layered on top.
  • NFR-3 (Tests): render-issue.js --selftest stays green; existing assertions are preserved or updated in lockstep with markup changes, and a new assertion is added per FR.
  • NFR-4 (No upstream contract change): the items.json input shape, the pipeline, fetch, and state.json are untouched. Library index unaffected except where it shares CSS.
  • NFR-5 (Restraint): the cumulative visual treatment must not slow skimming; default to subtle.

Out of scope (later waves)

Thumbnails/hero images (9), background transcription cron (15), per-post diagrams (10), front-page hero (2), why-it-matters hook (11), pull-quotes (12), in-issue TOC + search (14), OPML-import discoverability. Each is deferred per the ideation brief's roadmap.

Acceptance criteria

  • node render-issue.js --selftest exits 0 with assertions covering FR-1…FR-8.
  • Opening a rendered issue from file:// shows a working view-switcher (persists on reload), per-card mark-read with a decrementing counter + hide-read, dropdown filters with a live counter, a catch-up budget line, type/tag/feed color cues, and collapsed bullets with show-more.
  • With JS disabled, the grid + all bullets + read links still work.
  • No external network requests; file opens with no server.
