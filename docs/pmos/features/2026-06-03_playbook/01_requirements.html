Problem

A PM who works through real problems with AI accumulates a rich, instructive history — the prompts they started with, how they refined ideas, the trade-offs they decided — but that history is buried in Claude Code session logs that are scattered, voluminous, and unreadable. Today the only way to teach a peer "how I used AI to solve X" is to re-narrate from memory, which loses the exact prompts, the rejected alternatives, and the decision trail — precisely the parts a peer needs to replicate the approach. There is no tool that turns one's own AI-assisted work into focused, shareable, self-sufficient case studies.

Who experiences this?

Author (primary user): an AI-native PM who has solved real problems across many repos/sessions and wants to teach peers — and build a reputation as an AI-native operator — without manually trawling logs.

Reader (beneficiary): a fellow PM, often non-coding, who wants concrete, replicable examples of AI applied to PM work (research, concept notes, specs, surveys, features) — not abstract "how I use AI" advice.

Why now?

The author has built up months of substantive AI-assisted work across ~45 repos and thousands of sessions (validated: 5,343 session files). The teaching value is real and perishable — and the logs already contain everything needed (prompts, decisions, generated artifacts), so the gap is purely synthesis, not data capture.

Goals & Non-Goals

Goals

  • From a single repo, produce one or more focused, self-sufficient case-study articles, each anchored to one specific problem — measured by: a peer PM can read one article and replicate the approach on their own problem without access to the author's repo.
  • Surface the teaching payload a reader actually needs — the verbatim starting prompts, how ideas were refined, research done, and the load-bearing decisions & trade-offs — measured by: every published article contains ≥3 real prompts and ≥1 decision table with the actual alternatives weighed.
  • Find the whole story, not a fraction — including work done in worktrees that were later merged and deleted — measured by: the coverage line reports sessions recovered via worktree/sibling/merge-history, not just the literal repo dir.
  • Stay tractable and fast on large repos — measured by: a focused article is producible in <10 min from one command without the author hand-curating logs.
  • Be safe to share — measured by: no article reaches "shareable" state without the author signing off a per-item review of flagged sensitive content (proper nouns, metrics, screenshots).

Non-Goals

  • NOT a generalized cross-repo "how I use AI" playbook — because the teachable value is the concrete decision trail behind one specific problem; generalization produces the bland output this skill exists to avoid.
  • NOT multi-repo in v1 (--repo a,b,c) — because v1's per-repo resolver already unifies one repo's sibling/worktree sprawl, and a true cross-repo story is rare; deferred post-v1.
  • NOT a raw session-log viewer/exporter — because the value is selection + synthesis + a shareable narrative, not a transcript dump.
  • NOT an auto-publisher — because the author is the mandatory human gate for what gets shared; the skill produces a draft + a review checklist, never posts.
  • NOT a teacher of external topics (that is /primer / /learn-list) — /playbook is inward-facing on the author's own process.

User Experience Analysis

Motivation

  • Job to be done: "Turn the AI-assisted work I just finished into a few shareable articles that teach peers how I did it."
  • Importance/Urgency: high for reputation-building and peer enablement; the alternative (write-up from memory) is effortful and lossy, so it usually doesn't happen at all.
  • Alternatives: writing from memory (lossy); generic AI-tips threads (not replicable); doing nothing (the status quo).

"Will it even find the work? I did most of it in worktrees that are now merged/gone."Sessions scatter across sibling/nested worktree dirs; merged worktrees are deleted from disk.Multi-signal session resolution + a visible coverage line reporting what was found and how.

Satisfaction Signals

  • The author recognizes their own work in the proposed candidates and thinks "yes, that's a good story."
  • The draft reads like something they'd be proud to share, with their real prompts and decisions intact.
  • The review checklist gives confidence that nothing sensitive slips through.

Solution Direction

/playbook [--repo <path>] [--days N | --sessions N | --since <date>] [--include-headless] — a repo-scoped skill that runs a propose-then-build flow:

  1. Scope & discover. Resolve all of a repo's session history (including worktree and merged-then-deleted work) within an additive time/count window (default: last 30 days), and filter to interactive sessions by default.
  2. Propose. Cheaply cluster the sessions into problem threads and present a ranked shortlist of candidate playbooks, each with a one-line "why this is teachable." The author picks which to build (and may merge/split).
  3. Build. For each chosen thread, deep-read only that cluster and synthesize a focused, self-sufficient article (verbatim prompts, refinement arc, research, decision ledger, skills-used strip, embedded artifact excerpts + screenshots), plus a tweet/tweet-thread.
  4. Gate. Run a detect-and-flag anonymizer pass and emit a REVIEW-BEFORE-SHARING.md; the author signs off before anything is shareable.
  5. Emit. Write each playbook to its own dated folder under {docs_path}/playbooks/.

The user-observable contract is "point it at a repo, pick from a good shortlist, get a shareable draft + a safety checklist." The mechanics of resolution, clustering, and extraction are specified in /spec.

User Journeys

Primary Journey (happy path)

  1. Author runs /playbook from inside a repo (or --repo <path>), optionally with a window flag.
  2. Skill prints a coverage line — e.g. "found 4 session dirs (2 via merged worktrees), 41 interactive sessions (12 headless dropped), 6 candidate threads" — and asks the author to confirm before any heavy read.
  3. Skill presents a ranked shortlist (~5) of candidate playbooks, each with a "why teachable" line.
  4. Author picks one or more (and may merge or split candidates).
  5. Skill deep-reads the chosen thread(s) and synthesizes a focused article + tweet/thread per pick, with screenshots of visual artifacts embedded.
  6. Skill runs the anonymizer pass and writes REVIEW-BEFORE-SHARING.md listing flagged proper nouns, metrics, and every captured screenshot.
  7. Author reviews the checklist, edits/redacts as needed, and signs off; the playbook folder is theirs to publish.

Alternate Journeys

  • Headless-was-the-story: author passes --include-headless to re-admit subprocess sessions for a rare case where an automated pipeline they built is the teachable story.
  • Narrow window: author passes --since / --days / --sessions to scope to a specific body of work; the dry-run count lets them widen/narrow before committing.
  • Pick-and-defer: author builds one candidate now and re-runs later for others; each playbook is an independent dated folder.

Error Journeys

  • Ambiguous attribution: a sibling/branch can't be confidently tied to the repo → skill surfaces the ambiguous attribution for the author to confirm or drop rather than guessing.
  • No teachable threads: the window yields only thin candidates (below the instructiveness floor) → skill reports "nothing teachable in this window" and suggests widening, rather than proposing hollow playbooks.
  • Overlapping candidates: a feature branch + its follow-up appear as two candidates for one story → skill offers to merge them.

Design Decisions

All resolved in the deep grill (12 decisions); carried here as the binding direction for /spec.

#DecisionOptions consideredRationale
D1Playbook unit = session-thread (not folder)(a) session-thread, (b) folder/doc, (c) branchThe process is the teaching payload; docs/folders are supporting evidence.
D2Thread boundary = branch-then-topic fallback(a) branch-then-topic, (b) pure temporal windows, (c) user-drawsHandles both worktree work (branch) and main-branch work (topic+time); avoids manual burden.
D3Multi-signal session resolution(a) path-prefix only, (b) multi-signalPath-prefix misses sibling + merged-deleted worktrees (validated: 10/29 poker-coach sessions). Multi-signal recovers them.
D4Interactive-only by default; --include-headless escape(a) interactive-only default, (b) keep+flag all, (c) hard-filterHeadless/subprocess runs inflate volume (book-companion 3,839→handful) and aren't teachable; escape covers the rare exception.
D5Instructiveness floor suppresses thin candidates(a) composite score + hard floor, (b) LLM-judge each, (c) no gateFloor (≥1 non-trivial decision AND ≥1 skill/tool beyond /exit) kills 1-session noise without killing short-but-dense threads.
D6Decision extraction = prose-first, AskUserQuestion as anchors(a) prose-first, (b) AskUserQuestion-primary, (c) regexAskUserQuestion is sparse (8/44 sessions); pushbacks live in free prose.
D7Self-sufficiency = embed everything inline(a) embed all, (b) narrative+prompts only, (c) link outReader has no repo access; replicability needs inline prompts, decisions, excerpts, screenshots.
D8Anonymization = detect + forced human gate(a) detect+gate, (b) aggressive auto-redact, (c) per-repo allowlistAuto-redact guts concreteness; allowlist is burdensome. Detect-and-flag preserves value; author is the gate.
D9Full Playwright screenshot pipeline in v1(a) screenshots v1, (b) text-first defer, (c) neverAuthor chose vivid visual artifacts; leak surface handled by D10.
D10Screenshots gated by author manual verification(a) manual verify, (b) OCR-flag, (c) pre-marked-public onlyImages can't be text-flagged; author eyeballs each via the review checklist (consistent with D8).
D11Default window = last 30 days + mandatory dry-run confirm(a) 30d+confirm, (b) require explicit window, (c) all historyLow friction; no silent unbounded read.
D12Home plugin = pmos-learnkit(a) pmos-learnkit, (b) pmos-toolkitGrouped with /primer//learn-list by output-audience; coupling to toolkit conventions is light (string/path).
D13v1 output formats = HTML article + tweet/thread(a) HTML+tweet only, (b) +markdown export, (c) +LinkedIn variantHTML is already self-sufficient; tweet-thread covers succinct sharing. Markdown/Substack + LinkedIn variants deferred post-v1 to keep scope tight (resolves OQ1).
D14Clustering UX = auto-suggest merges; confirm only low-confidence boundaries(a) auto-suggest + low-confidence confirm, (b) always manual merge/split, (c) fully automaticAuto-detects overlapping threads (branch + follow-up) and suggests a merge the author accepts/rejects; thread-boundary confirm fires only below a topic-similarity confidence threshold — confident clusters are never re-litigated (resolves OQ2+OQ3).
D15Screenshot target = thread-referenced artifacts; graceful degrade(a) thread-referenced + degrade, (b) all repo artifacts, (c) author-selected onlyCapture visual artifacts referenced in the chosen thread (author may add more); when Playwright/browser is unavailable, degrade to embedded text excerpt + a "screenshot unavailable" note rather than hard-failing (resolves OQ4).
D16Denylist seeding = ship empty; one-time first-run prompt(a) empty + first-run prompt, (b) empty no prompt, (c) auto-seed from git metadata~/.pmos/playbook/sensitive.yaml ships empty; first run offers a one-time optional prompt to seed known client/repo/strategy names, stored for reuse. Balances zero-config with a useful head start (resolves OQ5).
