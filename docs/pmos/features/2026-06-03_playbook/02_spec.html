Tier 3 · pmos-learnkit · feat/playbook

Overview

Date: 2026-06-03 · Status: Draft · Tier: 3 · Source: 01_requirements.html (D1–D16) + grill gap (ambiguous-attribution).

/playbook is a standalone pmos-learnkit skill that mines the author's own interactive Claude Code session history for one repo and synthesizes focused, self-sufficient case-study articles. It is a propose-then-build pipeline: resolve → filter → scout/cluster → propose → deep-read → synthesize → anonymize/gate → emit. It does NOT load workstream context and does NOT feed the requirements→spec→plan pipeline (sibling-shaped with /learn-list, /primer).

This spec turns reference/skill-patterns.md §A–§F into concrete FRs (FR-90..FR-96) so the produced skill is testable against the authoring standard, and resolves the grill's ambiguous-attribution gap (FR-22).

FR — CLI & invocation

FRRequirement
FR-01Invocation: /playbook [--repo <path>] [--days N | --sessions N | --since <date>] [--include-headless] [--format <html|md|both>] [--non-interactive|--interactive]. --repo defaults to cwd. Scope flags are additive; if none given, default window = last 30 days (D11).
FR-02Standalone utility: MUST NOT load workstream context; MUST NOT write into docs/pmos/features/. Announce-at-start line per skill-patterns.
FR-03Resolve output_format via .pmos/settings.yaml default html; --format overrides. Print the standard stderr line.
FR-04Non-interactive mode honored per the canonical _shared/non-interactive.md block; the dry-run confirm and all merge/boundary prompts defer to their Recommended option (auto-pick) and log deferrals.

FR — Session resolution (multi-signal)

FRRequirement
FR-10Session-log root discovery: enumerate ~/.claude/projects/ and ~/.claude-personal/projects/ (both; either may be absent). Each child dir name is a path with /→- encoding; decode via the recorded cwd field inside the sessions (authoritative), falling back to dir-name decode when no record carries cwd.
FR-11Attribute a session dir to --repo when ANY signal fires: (a) nested-prefix — decoded cwd starts with <repo>/.claude/worktrees/ OR equals/starts-with <repo>/; (b) sibling-token-strip — cwd is a sibling dir <repo>-<slug> whose name, stripping trailing -token segments, resolves to an existing dir == <repo>; (c) branch-in-merge-history — the session's gitBranch appears in git -C <repo> log --merges or reflog (recovers merged-and-deleted worktrees).
FR-12Use realpath canonicalization for all path comparisons. A session dir matched by ≥1 signal is attributed once (dedupe).
FR-22Ambiguous-attribution (grill gap): when a candidate session matches ONLY by sibling-token-strip OR ONLY by branch-in-merge-history AND the same gitBranch/sibling-name could plausibly belong to another repo (e.g. a generic branch name like feat/ux-fix present in multiple repos' histories), mark it ambiguous and surface a confirm (interactive) / record-and-skip (non-interactive). NEVER silently attribute an ambiguous session. Confident matches (nested-prefix, or sibling+branch agree) are never prompted.
FR-13Coverage line (printed before any heavy read): found <N> session dirs (<M> via worktree/sibling/merged), <K> interactive sessions (<H> headless dropped), <C> candidate threads. Provenance per dir available on request/in verbose.

FR — Interactive-only filter

FRRequirement
FR-20A session is interactive iff it contains ≥1 record of type: permission-mode (or type: mode). Headless/subprocess sessions (SDK / claude -p) lack these. This is the primary discriminator.
FR-21Corroborating secondary signals (used only to rescue edge cases near the boundary, never to override a clear permission-mode presence): very short session (≤ small line count) + templated system-style opening + absence of any <command-name> tag → treat as headless. userType:external is NOT a discriminator (present on all sessions).
FR-23Default behavior drops headless sessions; --include-headless re-admits them. The dropped count is reported in the coverage line (FR-13). If 0 interactive sessions remain, emit "0 interactive (N headless dropped); re-run with --include-headless if intended" and exit cleanly.

FR — Scout, clustering & proposal

FRRequirement
FR-30Deterministic script scout (scripts/scout.mjs): reads only cheap fields per in-window interactive session — aiTitle, first genuine human prompt (first type:user message that is not a compaction-injected summary nor a <command-name> wrapper), <command-name> tags, gitBranch, file mtime. Emits a compact ranked-candidate JSON to stdout. The LLM consumes ONLY this JSON at scout time (never raw logs).
FR-31Thread boundary (branch-then-topic): group sessions by non-HEAD gitBranch first; for gitBranch==HEAD (or null) sessions, cluster by topic-similarity (over aiTitle + first-prompt tokens) + temporal contiguity (a new thread starts on topic shift AND a gap > threshold). Merge adjacent same-topic sessions (compaction continuations).
FR-32Instructiveness floor (D5): a candidate thread qualifies only if it has ≥1 non-trivial decision signal (an AskUserQuestion call OR a detected free-prose pushback/redirect) AND ≥1 skill/tool used beyond /exit. Below-floor threads are suppressed from the proposal (logged in verbose, not shown).
FR-33Scoring & ranking: score = f(decision density, distinct skills used, thread span, presence of a struggle→resolution arc). Present the ranked top ~5 qualifying threads with a one-line "why teachable" each.
FR-34Merge/split UX (D14): auto-detect overlapping threads (a non-HEAD branch + a temporally-adjacent same-topic HEAD thread) and SUGGEST a merge the author accepts/rejects at the pick step. Thread-boundary confirm prompts fire ONLY when topic-similarity confidence is below threshold; confident clusters are never re-litigated.
FR-35Dry-run confirm (D11): after the coverage line + ranked proposal, require an explicit confirm (interactive) before deep-read. Author picks which candidate(s) to build.

FR — Deep-read & extraction

FRRequirement
FR-40Deep-read touches ONLY the picked thread's sessions + the repo docs referenced therein. Use subagent fan-out (one per chosen playbook) with a strict output contract when the platform has subagents; degrade to sequential single-agent otherwise.
FR-41Prose-first extraction (D6): semantically extract decision points — the choice, the alternatives weighed, the rationale, and the author's pushbacks/redirects — from the full thread prose. Use AskUserQuestion tool_use blocks as high-confidence structured anchors (verbatim option text) that boost but do not bound the extraction.
FR-42Extract the starting prompt = the first genuine human message of the earliest session in the thread (skip compaction summaries + <command-name> wrappers), verbatim.
FR-43Extract the skills-used provenance: ordered list of <command-name> invocations + notable tools, each with a one-line "why this here."
FR-44Mine the repo's generated docs (docs/pmos/** + other referenced docs) for interesting load-bearing choices/snippets to embed as artifact excerpts.

FR — Article synthesis & outputs

FRRequirement
FR-50Self-sufficient article (D7) per reference/article-schema.md, sections: TL;DR · The problem · How I framed it · Starting prompts (steal-these gallery, verbatim) · How the idea was refined · Research done · Decision ledger (choice · alternatives · rationale) · Artifact excerpts · Screenshots · Skills-used strip · Takeaway for PMs. Reader opens zero repo files. PM-language voice; no unexplained jargon.
FR-51Quality gate: a synthesized article MUST contain ≥3 real prompts AND ≥1 decision-ledger row with real alternatives; a draft missing either is rejected and regenerated (or the candidate flagged thin).
FR-52Screenshots (D9, D15): capture visual artifacts referenced in the thread via Playwright (serve local HTML on localhost; file:// is blocked). Capture only thread-referenced artifacts (author may add). When Playwright/browser is unavailable, degrade to an embedded text excerpt + a "screenshot unavailable" note — never hard-fail.
FR-53Tweet/thread (D13): emit a succinct tweet + a numbered tweet-thread version distilling the article. v1 emits HTML article + tweet/thread only; markdown/LinkedIn variants deferred.

FR — Anonymizer & human gate

FRRequirement
FR-60Detect-and-flag (D8), always-on: best-effort detection of proper nouns (NER), known repo/branch names, emails, $/metric patterns, and entries in ~/.pmos/playbook/sensitive.yaml. The skill NEVER auto-scrubs the concrete content and NEVER auto-marks "safe."
FR-61REVIEW-BEFORE-SHARING.md per playbook folder: lists every retained proper noun + every quantitative claim (with location), AND enumerates every captured screenshot as a mandatory manual-verification item (D10). The author signs off (a hard, un-skippable gate before "shareable").
FR-62Denylist seeding (D16): ~/.pmos/playbook/sensitive.yaml ships absent/empty; on first run offer a one-time optional prompt to seed known client/repo/strategy names; persist for reuse.

FR — Output layout

FRRequirement
FR-70Each playbook → its own folder {docs_path}/playbooks/{YYYY-MM-DD}_<slug>/ containing: index.html (article), screenshots/, tweet-thread.md, REVIEW-BEFORE-SHARING.md, plus index.sections.json + copied assets/ (HTML substrate, atomic write, cache-bust).
FR-71Slug derived from the thread topic (kebab, ≤4 words). Existing-folder → prompt overwrite / suffix / cancel.

Non-functional requirements

NFRRequirement
NFR-01Scale: scout must handle a repo with thousands of session files without LLM ingestion of bodies; the script reads only cheap fields and streams. Interactive-only filter is applied before clustering.
NFR-02Safety: no network/publishing side effects; the only writes are under {docs_path}/playbooks/ and (with consent) ~/.pmos/playbook/sensitive.yaml.
NFR-03Portability: scripts are zero-dependency Node (ESM, Node ≥18); no npm install. Bash glue follows repo Bash-portability invariants (${BASH_SOURCE[0]:-$0} fallback).
NFR-04Determinism: scout output is deterministic given the same logs + window (stable ordering; no wall-clock in ranking beyond mtime which is data).

Data contracts

Scout candidate JSON (scout.mjs stdout)

{
  "coverage": { "session_dirs": N, "via_worktree": M, "interactive": K, "headless_dropped": H, "candidates": C },
  "ambiguous": [ { "dir": "...", "reason": "branch-only|sibling-only", "gitBranch": "..." } ],
  "candidates": [
    {
      "slug": "kebab-topic",
      "title": "ai-title or derived",
      "why_teachable": "one line",
      "score": 0.0,
      "session_ids": ["uuid", ...],
      "branch": "feat/... | HEAD",
      "skills": ["/feature-sdlc", "/grill", ...],
      "decision_signals": N,
      "span_days": N,
      "merge_suggestion": { "with_slug": "...", "confidence": 0.0 } | null
    }
  ]
}

Session record fields consumed

type (user|assistant|permission-mode|mode|ai-title|…), message.content (string|blocks), cwd, gitBranch, userType, aiTitle; tool_use blocks with name:"AskUserQuestion"; <command-name>…</command-name> tags inside user message text.
