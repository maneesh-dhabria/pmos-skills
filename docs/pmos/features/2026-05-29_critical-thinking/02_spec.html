/critical-thinking — Technical Spec (Tier 3)

Overview

A standalone, manually-triggered pmos-learnkit skill that runs a time-banded critical-thinking practice session for PMs: a varied mix of PM-scoped exercise shapes, own-words reasoning graded on moves, a calibration/per-muscle scorecard that accumulates across sessions, and optional repo-contextual scenarios. Builds on requirements 01_requirements + grill decisions D1–D4.

  2. Phase 1 — Commitment band (AskUserQuestion, hard input): 4 bands → exercise count (FR-2).
  3. Phase 2 — Plan the mix: mix-engine selects the session's shapes (FR-4) weighted by under-practiced muscles from the scorecard; assigns each a PM domain (FR-5); marks which 1–2 are repo-contextual when eligible (FR-8).
  4. Phase 3 — Run exercises (loop): per exercise — generate scenario fresh at runtime from the assigned domain (or repo) → present with explicit answer instructions → capture own-words answer (or MC via AskUserQuestion) → grade on moves (FR-7) naming ≥1 gap → for calibration, log prediction/outcome.   5. Phase 4 — Session summary + scorecard update: muscles practiced, per-muscle takeaways, weakest muscle, calibration (Brier) trend, streak; persist via scripts/scorecard.js update (FR-10/11).
  6. Phase 5 — Capture Learnings (FR-13).
  7. Anti-Patterns (DO NOT) (FR-14).
  8. Platform Adaptation (FR-12).

Exercise-shape contract (reference/exercise-shapes.md)

Each of the 9 shapes is specified with a fixed two-part schema: Generate (how to construct the scenario; inputs: domain or repo context; output: prompt + explicit answer instructions + input mode free-form|MC) and Evaluate (the named moves to check; the probe/follow-up if a move is missing; what "good" looks like; the muscle(s) it scores). v1 shapes:

Move-based grading (reference/grading-rubrics.md)

The 8 named moves: surface assumptions · steelman alternatives · trace 2nd-order effects · separate evidence from inference · state falsifiability ("what would change your mind") · spot bias · reframe the question · causal reasoning. Each shape's Evaluate half maps to a subset. Grading rule (FR-7): score the presence/quality of moves, never the chosen option; output a short verdict that MUST name ≥1 missing/weak move and one concrete way to strengthen it. No pure praise.

Runtime scenario generation (D4)

Scenarios are generated fresh at runtime — there is no static template bank. For each exercise the skill invents a believable PM dilemma on the spot from the chosen shape + one of the six PM domains (product design, prioritization/tradeoffs, metrics/experimentation, influence/stakeholder, strategy-under-ambiguity, GTM), varying actors, constraint, and metric so no two sessions repeat. The mix engine rotates domains across the session (mirrors FR-4). (Decision D4 reversed the original "hand-written template bank" plan — the user wanted fresh runtime generation only.)

Repo-context detection — README-or-propose (FR-8, D3)

Framing source is README-or-propose (read-only): if a README* exists, derive the product-decision framing from it; if none exists, study the codebase and propose a reusable README.md (offer to write it) for this and future sessions. Hard rules: cap at 2 repo exercises/session; ≥1–2 generic always; never echo tokens/keys/credentials/.env contents; if no clean product-decision framing emerges (and no README can sensibly be proposed), all exercises are generic. (D3 reversed the original "README + docs/ headings + commit subjects" approach.)

Scorecard + scripts/scorecard.js (FR-9/10, D1)

File: ~/.pmos/learnkit/critical-thinking/scorecard.json (created on first run).

Schema (scorecard-schema.md):

{
  "version": 1,
  "sessions": [{ "date": "<ISO>", "band": "<band>", "shapes": ["..."], "muscles_practiced": {"<muscle>": <count>} }],
  "muscle_scores": { "<muscle>": { "seen": <n>, "strong": <n> } },
  "calibration": { "predictions": [{ "p": 0.7, "outcome": 0|1 }], "brier": <float> },
  "streak": { "last_session_date": "<ISO>", "count": <n> }
}

CLI contract (node stdlib only; §E):

  • node scorecard.js read → prints current scorecard JSON (seeds + prints empty if absent).
  • node scorecard.js update <session.json> → atomic temp-then-rename merge of a session delta; recomputes Brier = mean((p − outcome)²); updates streak; prints the updated rollup.
  • node scorecard.js summary → weakest muscle + Brier + streak (for Phase 4).
  • Degradation (§E): if node is unavailable, the skill body falls back to reading/writing the JSON itself and computing Brier inline, logging a one-line note. Missing/corrupt file → reseed with a warning, never crash.

Mix-engine selection (FR-4)

Deterministic-with-rotation: (1) rank muscles ascending by muscle_scores.strong/seen (under-practiced first; unseen = top priority); (2) pick shapes covering the top muscles for the band's count; (3) never repeat a shape until the pool is exhausted; (4) no two consecutive exercises share a shape; (5) tie-break by least-recently-used shape from sessions[].shapes. First-ever session (empty scorecard) → spread across all three groups.
