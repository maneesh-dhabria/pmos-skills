Approach

Three layers across the substrate and two skills. _shared/tracker-crudl.md §2 is the canonical home of the id format (T1) — every other site cites it. Then the allocation call-sites stop computing max+1 (T2), the define merge gains a loud id-uniqueness gate (T3), and INDEX.md becomes a regenerated artifact (T4). Ripple-audit every id consumer (T5), then tests + cross-plugin _shared sync (T6).

Risks & verification

  • Randomness ban. Skill scripts forbid Math.random()/Date.now() in resume-sensitive paths. The generator must source entropy from crypto or accept the id as an input — pin the exact mechanism in build; don't smuggle a banned call.
  • Canonical-home discipline (§K). The id format lives only in tracker-crudl §2; allocation sites cite it. A duplicated format string is a review fail.
  • _shared sync. tracker-crudl.md is shared substrate — the change must flow through scripts/sync-shared.sh --from=pmos-toolkit, never a hand-copy (the drift hook guards this).
  • Don't lexical-sort ids anywhere. Non-monotonic ids break any code assuming id ordering ≈ chronology — that's why INDEX sorts on created: and next must be audited (T5).
  • Back-compat. Legacy 4-digit ids (0001–0019, incl. book-summary's 0018/0019) must keep parsing and never be rewritten (AC2).

Final verification checklist

  • Two simulated parallel mints off one baseline → different ids (AC1)
  • Duplicate-id define merge → loud refusal listing the id (AC3)
  • Legacy + new id coexistence test green (AC2)
  • INDEX regenerated, sorted by created:, no hand-edit step remains (AC4)
  • sync-shared.sh clean; skill-eval rubric green (AC5/AC6)
