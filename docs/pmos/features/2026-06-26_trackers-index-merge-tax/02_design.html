Trackers — get the derived index off the merge path

Epic 260626-8pa · route: skill · pmos-toolkit · 2026-06-26
Cross-skill coherence contract for /backlog, /mytasks, /people + the shared _shared/tracker-crudl.md substrate.

TL;DR

Each of the three trackers commits an INDEX.md "regenerable cache" that every mutating handler rewrites — a structural merge-conflict magnet under parallel worktrees, with a per-mutation regeneration tax, for a file no script and no human reads. Fix: stop persisting a committed index; derive the at-a-glance view on read from the record files (which is what the scripts/web layer already does). Rewrite the substrate contract so the pattern cannot reappear.

This is the shaped problem in docs/pmos/shape/2026-06-26_backlog-index-merge-tax.html, widened from /backlog-only to all three trackers + substrate per the maintainer's scope decision.

Problem (recap + ground truth)

  • The committed cache buys nothing the merge path can't. The deterministic layer already derives everything from the record files: backlog/scripts/serve-web-lib.mjs globs items/*.md; mytasks/scripts/serve.js + lib.js load items/; the web viewers never read INDEX.md.
  • But the LLM render paths DO read the committed INDEX — and this is the real work, not just a git rm:
    • backlog/SKILL.md:237/243 — bare-/backlog dashboard uses INDEX's existence as the empty-state gate and prints the INDEX blob verbatim.
    • mytasks/SKILL.md:110–112 (Phase 1 echo) + :187 (list step 1: "INDEX.md answers most filters").
    • people/SKILL.md:83–85 — bare-/people echoes INDEX verbatim.
    These prose procedures must be rewritten to derive-and-render from the record files, exactly as the scripts already do.
  • Three regeneration mechanisms, not one. /backlog and /people regenerate via an inline LLM procedure (no script). /mytasks regenerates via a real JS function scripts/lib.js → regenerateIndex() that also folds an idempotent workstream:→project: frontmatter migration (Phase 12 step 0) — a load-bearing side effect to preserve elsewhere.
  • External blast radius is exactly one call. feature-sdlc/SKILL.md:448 (the define definition-merge) runs /backlog rebuild-index after the docs-only merge. Its corruption guard — check-id-uniqueness.mjs post-merge <root>/backlog/items — reads items/, not INDEX, so the safety property survives removal. /complete-dev has zero INDEX references.

The coherence contract (the invariant all stories cite)

INV-1 — No committed, mutation-rewritten derived file on the merge path. A tracker's at-a-glance index is a derived view, computed on read from the record files. It is never persisted as a committed artifact, never written by a mutating handler, and never a merge dependency. Staleness is impossible because nothing is stored.

INV-2 — Web-default view, render-on-read fallback (grill outcome). The bare command's default at-a-glance surface is the web viewer (which already derives from the record files). When the web viewer cannot run — --non-interactive, headless, no browser, no server — the bare command degrades to an inline render derived on read from the same record-file scan. Both paths derive from the record files, so terminal and web agree by construction. Empty-state = "zero record files," never "index file missing."

INV-3 — No static export. The committed INDEX.md is deleted outright; no on-demand .gitignored export replaces it (grill: "retire entirely; default to web, not inline"). The view exists only as the web render (default) or the inline derived render (fallback) — never as a persisted file.

INV-1 is the load-bearing change: it lives in the substrate (_shared/tracker-crudl.md), so a future tracker built on the substrate inherits the merge-safe behavior by construction — "doesn't happen again."
