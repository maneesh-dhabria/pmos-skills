Plan — /people NEW zero-dep web viewer

Story 260626-nq0 · Epic 260626-8pa · Route skill · Plugin pmos-toolkit · Deps 260626-5cq · Spec 02_design.html (#change-people-web).

Executable plan is tasks.yaml; this is the human-readable companion.

Scope

The only new build in the epic. After S3 (260626-5cq) makes /people derive-on-read, this story brings it to parity with /backlog and /mytasks by adding a zero-dependency local web viewer so bare /people defaults to the web UI (INV-2). Modeled on /backlog's serve-web.mjs + serve-web-lib.mjs + web/viewer.html trio. The server derives the people listing fresh per request directly from ~/.pmos/people/*.md — it never reads or writes an index (INV-3) — and exposes only whitelisted, non-PII-leaking fields.

Risks & decisions

  • PII safety (AC4). Person files can carry sensitive fields. The API must expose only a whitelist; the test asserts no field leaks beyond it. Whitelist is pinned in T1 before the server is built.
  • Reuse the S3 parser. The web server and the inline fallback must agree on the person listing — both read through the same derive-on-read path from 260626-5cq, not a second parser.
  • Dep ordering. Claims after 260626-5cq merges (the read path must be derive-on-read before a web-default surface is layered on it).
