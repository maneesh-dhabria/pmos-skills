Approach

Thin driver over the 260704-6rq engine: walk the 61 bundled primers, apply enrichPrimer() to each, commit the enriched artifacts. This is the network run (unlike rgt's backfill) — every added source is fetch-verified this run (INV-2). Safety is delegated to the engine's per-primer rubric gate and revert (INV-6); the driver's own contract is coverage (every one of the 61 accounted for in the report) and convergence (a second run adds nothing, INV-5). A --dry-run --limit N trial de-risks the full lights-out run.

Risks & decisions in play

  • Hard deps (260704-6rq + 260704-rgt). Needs the engine and the corpus References section present. Claim-time merge (D9) brings both branches into this worktree before build; not buildable until the rgt epic ships.
  • Lights-out risk (D5). No human sees each of the 61 before commit. Mitigations: the engine rubric-gates + reverts per primer (INV-6); the --dry-run --limit 3 trial (T1) previews behavior; the run report (T2) makes every change auditable post-hoc; additive-only diffs (INV-4) keep review tractable.
  • Network variance. ~30% fetch-verify yield and rate limits mean the run is slow and some primers add little; that is expected — coverage gate skips low-yield topics rather than blocking (a low-add or no-op primer is a valid, reported outcome, not a failure).
  • Index churn (rgt D8). sources_count grows honestly; word_count excludes References. Only a justified index regen — never a silent side effect.
