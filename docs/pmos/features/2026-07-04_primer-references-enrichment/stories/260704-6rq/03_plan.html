Approach

One engine module (enrich-references.mjs) is the story's center of gravity. It orchestrates the 7-step per-primer refine of design §4.1 but reuses, never forks, two existing mechanisms: the corpus relevance matcher (curated-references-match.mjs, INV-1) and the sourcing.md fetch-verify loop (INV-2). The only non-deterministic step is the prose weave, which is bounded on both ends — fetch-verify admits only real sources, and the rubric gate reverts any primer it breaks (INV-6). References is regenerated, not authored, by importing epic 260704-rgt's injectReferences (INV-3).

Risks & decisions in play

  • Dependency on 260704-rgt (hard). The engine imports injectReferences from references-section.mjs (rgt's story ytr) and assumes the corpus already carries ## References. Build only after rgt's branches are claim-time-merged into this worktree (D9).
  • Reuse boundary (INV-1/INV-2). The single biggest failure mode is re-implementing the matcher or the verify loop. T1 imports curated-references-match.mjs; T2 routes through the existing sourcing.md path — no second relevance or trust mechanism.
  • Non-deterministic weave, bounded. The LLM weave is not byte-idempotent; INV-5 requires convergence (dedup by url, no re-weave) not byte-equality. T3 dedups against the current sources.json; T4 reverts anything that breaks R1/R11.
  • Additive-only (INV-4/D7). The engine must never rewrite/delete existing prose or sources — only weave additions and append. Keeps 61-primer diffs reviewable and blast radius bounded.
  • Cap (D4). Unbounded additions would bloat primers; the per-primer cap (~4–6) keeps this a refine, not a re-write.
