Design — /primer curated-corpus enrichment

pmos-learnkit · epic 260704-dgq · route: skill · the cross-cutting coherence contract for stories 260704-6rq (enrichment engine) and 260704-e3b (corpus backfill). Stories cite this doc by anchor. Depends on epic 260704-rgt (References section) — sequenced after it. No epic /spec — this design doc is the skill-epic contract; per-skill spec folds into each story's /plan.

1. Overview & the actual gap

/primer's new-primer generation already factors in the ~1,800-record curated corpus: Phase 3 sourcing runs the curated-references overlay (_shared/topic-research/curated-references.md), which prefilters the corpus per topic via curated-references-match.mjs (a pure IDF tag-overlap matcher), reranks, and feeds survivors into the same fetch-verify loop as live search. So the forward problem — "are these resources wired in for new primers?" — is already solved (suppressible with --no-curated). This epic does not touch that path.

The gap is purely retrospective. The 61 bundled corpus primers (data/primers/*.html + *.sources.json) were generated before the overlay existed at today's corpus size, so their ~16-source-each evidence sets under-use the 1,797-record corpus. Many relevant, already-verified curated resources were never considered when those primers were built.

This epic closes that gap by enriching each bundled primer, primer-by-primer: run the existing overlay prefilter over the corpus for that primer's own topics, fetch-verify the top curated candidates this run, and — for survivors — weave them into the relevant teaching section's prose and into sources.json, then regenerate the primer's ## References section via epic 260704-rgt's shared injectReferences generator. A real refine pass, not a list append.

2. Relationship to epic 260704-rgt (hard dependency)

260704-rgt surfaces each primer's existing sources.json as a bottom ## References section — its INV-3 is explicit: nothing re-fetched, no new sources. This epic is the complement: it adds new curated sources to those same primers. The coupling is concrete and must be sequenced:

  • Same 61 files. Both rgt's backfill (story 3jt) and this epic rewrite the same data/primers/*.html and touch sources.json. Running both in-flight would produce commit clashes — the risk the maintainer flagged.
  • Sequencing (D3). rgt lands first. This epic dependencies: [260704-rgt]. After enrichment mutates a primer's sources.json and prose, it re-invokes rgt's injectReferences so the References section reflects the enriched, larger source set — reusing rgt's one generator (INV-1 of rgt), never re-implementing it.
  • Honesty preserved. Because new sources are woven into the teaching prose (not merely listed), rgt's "sources the primer was built on" semantics stay true — an enriched primer was rebuilt on the enriched set.

3. Invariants

  • INV-1 — Reuse, don't fork, the corpus matcher. Candidate selection reuses the existing curated-references-match.mjs prefilter + the curated-references.md overlay procedure. The enrichment engine does not invent a second relevance mechanism; new primers and backfilled primers select curated candidates the same way.
  • INV-2 — Every added source is fetch-verified this run (D2). A curated record is a candidate, not a verified source. It is admitted only after passing sourcing.md's existing hard-gate + fetch-verify (~30% yield). No corpus record is written into a primer on trust. Preserves /primer's core rule: every claim traces to a source fetched this run.
  • INV-3 — References is regenerated, never hand-authored. After enrichment mutates sources.json, the ## References section is regenerated by rgt's deterministic injectReferences (rgt INV-1/INV-2). This epic never hand-writes a References list.
  • INV-4 — Enrichment is additive & non-destructive. Existing verified sources, prose, and structure are preserved. Enrichment only adds woven sentences/pointers and appends to sources.json; it never rewrites or deletes existing teaching prose or drops existing sources.
  • INV-5 — Re-runnable & convergent. A second enrichment run over an already-enriched primer adds nothing new it cannot verify and does not re-weave already-present sources (dedup by verbatim url against the current sources.json). It is not required to be byte-idempotent (LLM prose weave is not deterministic — INV-2's fetch-verify and dedup bound it), but it must converge: no duplicate sources, no runaway growth.
  • INV-6 — Rubric-gated, per primer. Every enriched primer must still pass /primer's existing rubric.md — R1 (every <a href> ∈ sources.json[].url), R11 (References membership, from rgt), and the trust-tier checks. A primer that fails the rubric after enrichment is reverted, not shipped (D5).

4. Architecture — the enrichment engine (story 260704-6rq)

A new mode/CLI plugins/pmos-learnkit/skills/primer/scripts/enrich-references.mjs (zero-dep Node ESM for the deterministic parts) orchestrates a per-primer refine. It is engine + orchestration; the prose-weave step is LLM-driven (the only non-script part), gated by the rubric.

4.1 Per-primer procedure

StepWhatDeterminism
1. TopicsRecover the primer's topics from its sources.json[].topic set (the per-H2 evidence map already keys on topic).script (pure)
2. PrefilterFor each topic, pick query tags from the corpus tag-vocabulary and run curated-references-match.mjs (INV-1) → top-K candidates. Exclude any whose url is already a verbatim member of the primer's sources.json (dedup, INV-5).script + LLM tag-pick (as in the overlay)
3. Coverage gateReuse the overlay's per-topic coverage gate (default T=3, S=0): a topic with too few candidates is skipped and logged. A primer whose every topic is skipped is a no-op (logged, unchanged).script
4. Fetch-verifyRun surviving candidates through sourcing.md's hard-gate + fetch-verify (INV-2). Cap admitted new sources per primer (D4). Losers are dropped.fetch + gate
5. WeaveFor each verified new source, weave a grounded sentence/pointer into the relevant H2's prose (the H2 matching the source's topic), and append the entry to sources.json. Additive only (INV-4).LLM (rubric-gated)
6. Regenerate ReferencesCall rgt's injectReferences(html, sources) so ## References reflects the enriched set (INV-3).script (rgt)
7. Rubric gateRe-score the enriched primer against rubric.md (R1, R11, trust-tier). Pass → write atomically. Fail → revert this primer, log the failure (D5, INV-6).script + reviewer

4.2 SKILL.md wiring

A new verb /primer enrich (natural-language: "enrich the bundled primers with curated sources") documented in SKILL.md, dispatching the engine. Anti-patterns gains "don't add a curated source without fetch-verifying it this run" (mirrors the trust anti-pattern #1). No change to the forward generate path — the overlay already covers new primers.

4.3 Tests

Unit tests for the deterministic parts (topic recovery, dedup against existing sources.json, coverage-gate no-op, cap enforcement) + a fixture primer proving an added-and-verified source lands in both prose and References and passes R1/R11. LLM weave itself is exercised in the story's dogfood, not unit-asserted.

5. The corpus backfill (story 260704-e3b)

A CLI run that walks all 61 bundled primers and applies the engine (§4) to each, fully unattended, rubric-gated only (D6). It is the data change: enriched data/primers/*.html + *.sources.json committed together. Depends on 260704-6rq (engine) and 260704-rgt (References generator + section already present on the corpus). Emits a run report: per primer, sources considered / verified / added / skipped, and any primer reverted for a rubric failure. No index churn beyond sources_count (which grows honestly with the added sources); word_count excludes References per rgt D8.

6. Decisions (maintainer-settled)

  • D1 — Enrichment depth: full refine pass. Verify curated candidates and weave survivors into teaching prose and References — not a References-only append. Keeps rgt's "built on" semantics honest and delivers the real value. (Rejected: references-only append — lists sources the primer wasn't built on; hybrid prose-only-where-material — harder to gate deterministically.)
  • D2 — Verification: re-fetch-verify this run. Every added source passes sourcing.md's fetch-verify, respecting /primer's trust invariant and guarding against corpus link-rot. (Rejected: trust the corpus — violates fetch-this-run contract, risks stale links across 61 curated primers.)
  • D3 — Depend on 260704-rgt, sequence after. rgt owns the References format + injectReferences + rubric R11. This epic dependencies: [260704-rgt]; enrichment regenerates References via rgt's generator. Prevents the same-61-files commit clash. (Rejected: fold into rgt — contradicts rgt's INV-3; run independently — leaves the clash to manual discipline.)
  • D4 — Approach: primer-by-primer, capped. Iterate primers, not tags — maps 1:1 onto the overlay machinery and keeps relevance anchored to each primer's framing. Cap admitted new sources per primer (default small — e.g. up to the depth-floor delta / ~4–6 verified additions) so enrichment stays a refine, not a bloat. (Rejected: batch-by-tag — inverts the index; a tag-relevant source isn't necessarily relevant to a specific primer's framing.)
  • D5 — Review posture: fully unattended, rubric-gated only. 61 primers × full refine is infeasible to human-review per primer; the run is lights-out, gated by the existing rubric (R1/R11 + trust) and fetch-verify. A primer failing the rubric after enrichment is reverted (INV-6), never shipped. (Rejected: interactive per-primer — impractical at 61; sample-review — deferred; the rubric gate is the safety net and the run report surfaces every change for post-hoc audit.)
  • D6 — Two stories, engine → backfill. 260704-6rq ships the enrichment engine + SKILL.md verb + tests (deps: 260704-rgt for injectReferences). 260704-e3b runs it over all 61 primers as the data change (deps: 260704-6rq + 260704-rgt). One skill, one plugin (pmos-learnkit), one release unit.
  • D7 — Additive-only enrichment. Never rewrite or delete existing prose/sources; only weave additions and append. Bounds blast radius and keeps diffs reviewable (INV-4).
  • D8 — Forward path untouched. New-primer generation already runs the overlay; this epic adds no forward behavior and no new forward verb beyond /primer enrich for the retrospective pass. --no-curated semantics unchanged.
