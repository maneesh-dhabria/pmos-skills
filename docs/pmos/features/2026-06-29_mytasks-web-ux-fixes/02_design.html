Context

The recent /mytasks web overhaul (epic 260626-a8a, story tf4) shipped an inline-everything client-only SPA. A /design-crit of the running app surfaced a coherent set of UX gaps. They are not independent bugs — three cross-cutting themes drive almost all of them, and a few shared conventions fix many findings at once.

Surfaces touched (all under plugins/pmos-toolkit/skills/mytasks/scripts/):

  • webapp/app.js (536 L) — client SPA logic: views, quick-add parse, row render, editor, completion.
  • webapp/app.css (118 L) — colour tokens, contrast, due-date semantics, layout, hit areas.
  • webapp/index.html — shell (toast host, panel structure).
  • serve.js / people.js — only for the F-05 person auto-create (the POST /api/people endpoint already exists from story 71x; the work is wiring the client call, not new server capability).

The CLI / terminal /mytasks behaviour and the on-disk task/people schema are out of scope — this epic is web-UI-only (see Invariants).

Decisions (maintainer-approved)

#DecisionRationale
D1@ opens a filtered people picker (F-05, grill-refined). Typing @ opens a dropdown of existing people, filtered live as the user types; selecting one assigns it. The dropdown carries an explicit + Add "<handle>" button that mints a new person via POST /api/people — so a new person is created only on a deliberate click, never silently from a typo. The added/selected person then shows in the editor People field and the PEOPLE sidebar.Closes the silent-drop AND the typo-junk risk in one affordance; the picker doubles as the @ autocomplete (FR-1). Uses the existing people CRUD + endpoint — no schema change.
D2One friendly date format everywhere (F-04, grill-refined): relative for near dates — Today / Tomorrow / Yesterday (±1 day), weekday alone for the rest of this week (e.g. Fri) — and absolute beyond (Fri Jul 3; add year only when not current-year). Applied to lists, rows, editor display, and toasts. The native <input type="date"> stays for picking; its rendered display text is overridden to this form.Most scannable; unambiguous; kills the ISO-vs-DD/MM split. Relative words recomputed on render so "Today" never drifts.
D3Count badges + overdue-on-Today (N1/F-02): every smart view gets a trailing count badge (e.g. Overdue (1)); the Today view additionally surfaces overdue items so a cold open never reads empty while work is due. Today remains the landing view.Returning user immediately sees what needs attention; no surprising landing-view change.
D4Full scope: all 12 surfaced findings + the low-severity unsurfaced F-13 (hit areas / hover) + the 5 deterministic contrast failures ship in this one story.The fixes share the same 3 files; deferring the low item would only fragment a single colour/affordance pass.
D5One shared "optimistic-action + toast + undo" convention (cross-cutting CC1). A single helper renders a transient toast naming the result, with an Undo affordance where reversible. Undo lives on the toast window only (grill-refined) — once the toast dismisses, the action is final; no persistent undo stack. Reused by quick-add (F-03) and completion (F-11).One convention fixes the invisible-feedback findings; toast-window scope keeps the undo state trivial.
D6One colour-token pass (CC3): a single accessible gray (≥4.5:1 on white, e.g. #6b6e7a), red reserved for genuinely-overdue / destructive only (≥4.5:1), upcoming dates neutral/amber + icon (F-08).Centralizes the 5 contrast misses + the red-for-all-dates issue into the token layer.
D7One representation per data type (CC2): dates (D2), labels rendered as chips everywhere (F-10), task metadata (project/label/person) shown as inline chips on the row and in the editor (F-07).Stops representation drift between surfaces.

Cross-cutting conventions

  • CC1 — Invisible feedback (D5). Quick-add (FR-1), completion (FR-11), and assignment (FR-2) all succeed silently today. One optimistic-action + toast/undo helper drives the quick-add + completion feedback; assignment gets explicit feedback through the people picker (FR-2). Toast-window undo only.
  • CC2 — Representation drift (D7). Pick one representation per data type and reuse it: dates (FR-3), labels (FR-10), metadata chips (FR-7).
  • CC3 — Contrast & colour-semantics (D6). One accessible gray + red-reserved-for-overdue/destructive token pass (FR-14, FR-8).

Invariants

  • INV-1 — CLI unchanged. The terminal /mytasks command flow and all non-web SKILL.md phases behave byte-identically; this epic edits only the webapp client + the F-05 client→/api/people wiring.
  • INV-2 — No schema / data-format change. No new fields on the task or person records; person auto-create uses the existing person schema and endpoint. On-disk files written by prior versions still load.
  • INV-3 — Zero new dependencies. No npm packages; vanilla JS/CSS only (the webapp's standing constraint).
  • INV-4 — Tests green, no regressions. Existing scripts/tests suites (lib, people, serve, registry, import-parse, recur) stay green; new assertions cover the parse-preview, person auto-create, friendly-date formatter, and completion/undo paths.
  • INV-5 — Skill conformance. Conforms to skill-patterns.md §A–§L; passes skill-eval.md ([D]+[J]); the inline non-interactive block stays byte-identical; 4 hygiene lints + audit-recommended green. SKILL.md doc updates reflect any new web behaviour.

Out of scope

  • Terminal CLI rendering, inference heuristics, output formats, schema changes.
  • New task fields, new views beyond surfacing existing data, or import/export changes.
  • Server-side people model changes (auto-create reuses the existing endpoint + schema).
  • Release prerequisites (version bump, changelog, README row, manifest sync) — /complete-dev owns these per §G; they are listed under the plan's Release prerequisites, never in a build wave.

Story split

One story (260629-28w), route: skill. All 14 FRs revise the same 3–4 files (app.js, app.css, index.html, + the F-05 client wiring). The D24 litmus — would a task in one story depend on a task in another? — fails any multi-story split here: the toast helper (CC1), the colour tokens (CC3), and the chip renderer (CC2) are shared dependencies of multiple findings, so splitting would force cross-story coupling on the same lines. One /execute run, one PR. (Same shape as the recent singleton skill epic 260629-bm9.)
