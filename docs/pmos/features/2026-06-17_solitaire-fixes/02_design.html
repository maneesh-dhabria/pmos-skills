Design — /solitaire fixes

Epic 260617-bx0 · route: skill · plugin: pmos-gamekit · 2026-06-17 · This is the epic design_doc — the coherence contract; the single story's /plan cites its anchors.

Context

/solitaire is a launch-only pmos-gamekit skill: it serves one self-contained, offline Klondike file — plugins/pmos-gamekit/skills/solitaire/game/solitaire.html (~820 lines) — through the shared zero-dependency game launcher. All gameplay lives in that one file. The engine (window.SolitaireEngine, lines 1–387) is a pure, browser-agnostic module exposing isLegalMove(state, m), applyMove, clone, newGame, isWin, findFoundationMove, nearWinState, isMovableRun, etc.; the UI/render layer (lines 393–820, guarded by document) renders it and wires interaction. Tests live in tests/run.mjs (engine selftest) plus a Playwright e2e drive-to-win (?test=nearwin).

The maintainer reported five first-run rough edges. All five are confined to the single game file and ship as one /solitaire release, so this is a one-story skill epic (the D24 litmus: the items are tightly coupled in one file and cannot be independently skill-eval'd or shipped). Product decisions on the two genuinely open questions (card-back variety; draw-animation feel) were resolved with the maintainer up front — see D2 and D5.

Functional requirements

FR1 — Header wording

  • FR1.1 — The <h1> reads "Solitaire" with no "· Klondike" suffix span.
  • FR1.2 — <title> reads "Solitaire" (drop "— Klondike").
  • FR1.3 — The skill stays Klondike-rules solitaire; only the displayed/window wording changes. SKILL.md's internal "Klondike" references (rules description) may stay; the user-facing header must not say it.

FR2 — Per-game card backs

  • FR2.1 — Bundle ~6 visually distinct, self-contained back-cover designs (CSS gradients/patterns or inline SVG; no external assets, no network).
  • FR2.2 — On each new game, choose one back at random and apply it to every face-down card uniformly for that game.
  • FR2.3 — The chosen back is fixed for the duration of a game (re-renders, undo, draw, recycle do not change it). D3.
  • FR2.4 — Each back is legible as a "card back" (covers the card, good contrast against the table, readable at the mobile breakpoint).

FR3 — Idle auto-hint

  • FR3.1 — A shared finder returns the first productive legal move for a state (foundation moves, waste→tableau, tableau→tableau, tableau→foundation; excludes draw/recycle). The manual H hint reuses it (so H now hints non-foundation moves too).
  • FR3.2 — After ~8s of no pointer/keyboard activity in the game, outline the source card of that move using the existing .hint style; clear it on the next interaction or move.
  • FR3.3 — The idle timer re-arms on any interaction and is paused/cleared on win or when no productive move exists (the F4 banner takes over).
  • FR3.4 — Honor prefers-reduced-motion for any pulse; the outline itself is static.

FR4 — No-moves detection

  • FR4.1 — Engine helpers: hasProductiveMove(state) (any non-draw/recycle legal move now) and isDeadlocked(state) (no productive move appears at any point in one full stock-cycle simulation), both pure and exported.
  • FR4.2 — After each committed move, if !isWin(state) && isDeadlocked(state), show a non-blocking banner: "No moves left — deal a new game?" with a New-game button; dismissible.
  • FR4.3 — The banner must never appear on a winnable position (no false positives) — the cycle scan, not a single-position check, guarantees this.
  • FR4.4 — Starting a new game clears the banner.

FR5 — Draw animation

  • FR5.1 — On a draw, the newly-drawn card visually originates at the stock pile and moves to the top of the waste fan in one smooth motion.
  • FR5.2 — The previously-fanned waste cards ease to their new offsets rather than snapping/reflowing.
  • FR5.3 — Animation is interruptible and never blocks input; rapid draws degrade gracefully (no queue pile-up, no stuck cards). Final DOM state equals a plain re-render.
  • FR5.4 — Honor prefers-reduced-motion (instant placement when set).

Invariants (coherence contract)

  • Inv-1 (engine purity): the engine module stays browser-agnostic and side-effect-free; new logic (FR3.1 finder, FR4.1 helpers) is added to it without importing DOM. tests/run.mjs must pass headless.
  • Inv-2 (cosmetic isolation): card-back choice, hint outline, deadlock banner, and draw animation are presentation-only — none mutate state or enter history/undo.
  • Inv-3 (offline + self-contained): no new external assets, fonts, or network. Everything inlines into the one HTML file.
  • Inv-4 (no false deadlock): isDeadlocked uses the full stock-cycle scan; a position with any reachable productive move (including after exhausting draws) is not deadlocked.
  • Inv-5 (test harness intact): the ?test=nearwin / __solitaire hooks and the win animation keep working; new engine exports get selftest cases.

Story split

One story (D1): 260617-ks1 — "/solitaire first-run fixes (header, card backs, idle hint, no-moves, draw anim)", route: skill, plugin pmos-gamekit, no dependencies. Its ACs are the five FR groups above plus the invariants; skill-patterns.md §A–§L + the host CLAUDE.md are standing acceptance criteria. The per-story /plan cites this design's anchors.

Open questions / deferred

  • Card-back picker (deferred, D2): a user-facing control to choose/lock a favorite back was the runner-up option. Deferred — revisit if players ask for it. The random-per-game default ships first.
  • Idle delay tuning: 8s is the proposed default; the implementer may settle on 6–10s based on feel during the build self-check.
  • Back-cover count: "~6" is a target, not a contract; 5–8 polished designs is acceptable so long as variety is obvious.
