Design contract — /primer browse library + ship the 61-primer corpus

Epic 260615-565 · route: skill · plugin: pmos-learnkit · Loop 1 (define) · 2026-06-15 · This is the epic's design_doc (route:skill replaces /spec with a design contract). One fused build story: 260615-dhk.

1. Problem & intent

/primer (pmos-learnkit) produces verified-source, audience-shaped teachable HTML artifacts one topic at a time. Two gaps:

  1. No way to browse the collection. Generated primers land in {docs_path}/primer/ with only a thin filename-date listing (primers.html, Phase-5 step 8). There is no faceted, searchable library view.
  2. A pre-built corpus has nowhere to live. A 61-primer PM-curriculum corpus was generated by the feat/primer-bundle harness (61 *.html + 61 *.sources.json + a rich primers-index.json). It should ship with the skill so every install carries the curriculum out of the box.

Both are solved by one move: adopt the proven /frameworks browse model — a committed corpus plus an on-demand, gitignored, single-file, offline-capable filterable page generated by a zero-dependency build-library.mjs.

2. Model: /frameworks browse (the precedent we copy)

/frameworks already ships this exact shape; we mirror it deliberately so the two skills stay coherent:

  • Committed corpus under the skill (data/frameworks.json + owned SVGs).
  • scripts/build-library.mjs — zero-dep Node ESM, --out <path> + --selftest, atomic temp-then-rename, reads the corpus and emits a single self-contained index.html that works offline from file://.
  • The generated page is gitignored (committed corpus, regenerable output).
  • browse | list (or bare invocation with no problem text) → a #browse phase that builds the page and opens it.
  • No-browser / no-Playwright → still write the page; opening is best-effort.

4. build-library.mjs (new)

New file plugins/pmos-learnkit/skills/primer/scripts/build-library.mjs, modeled directly on the frameworks one. Zero external dependencies; Node ESM.

Inputs & invocation

node ${CLAUDE_SKILL_DIR}/scripts/build-library.mjs --out {docs_path}/primer/library.html
node ${CLAUDE_SKILL_DIR}/scripts/build-library.mjs --selftest
  • Reads its sibling ../data/primers-index.json (the committed corpus) — resolved relative to the script's own location, not CWD.
  • Also scans the output directory (dirname(--out)) for user-generated primers: *.html minus the library page itself, minus *.draft.html (rejected drafts per the recovery path are never listed). Title from <title>/<h1>; date from the YYYY-MM-DD_<slug> filename; audience/depth shown as — when unknown.

Output

  • A single self-contained library.html — inline CSS + JS, zero external requests, opens offline from file://. Atomic temp-then-rename write.
  • One card per primer: title, category, audience, depth, source-count, word-count, date; the card links to the primer's standalone HTML.
  • Link resolution: for curated primers the href is a relative path computed at build time from the --out directory to the skill's data/primers/ dir (works at file:// against the installed plugin cache or the repo). For user primers the href is the bare sibling basename.
  • Facets (client-side, no server): Collection (Curated | Yours), super_category, category, audience, depth; plus a free-text search box over title + category. An applied-filters bar with removable chips + "Clear all" (mirror frameworks' UX).

5. Reconciliation — one unified page (key decision)

The existing Phase-5 step-8 "Listing regen" produces primers.html for user-generated primers only. Rather than ship two competing listing pages, this design unifies them:

DecisionResolution
D-UNIFYOne page, {docs_path}/primer/library.html, renders both populations — the shipped 61-primer corpus (Collection=Curated) and any user-generated primers found in the output dir (Collection=Yours). Both reachable from one place.
D-RETIREThe bespoke primers.html generator (Phase-5 step 8) is retired. Phase 5 now invokes build-library.mjs so a freshly-written primer appears in library.html after a normal run. A stale primers.html on disk is best-effort removed.
D-GITIGNOREThe generated library.html is a regenerable artifact → gitignored (committed corpus, gitignored output — exactly the frameworks split). Dogfood runs in this repo do not dirty the tree.
D-VERBNew browse | list verb, and bare /primer with no topic, route to a new #browse phase: build + open library.html. argument-hint updated.
D-DEGRADENo browser / no Playwright → the page is still written; opening is best-effort; the path is printed. Never a hard failure.

6. Acceptance criteria (epic-level; inherited verbatim by story 260615-dhk)

  1. AC1 — The 61-primer corpus (61 *.html + 61 *.sources.json + primers-index.json) is committed under plugins/pmos-learnkit/skills/primer/data/; primers-index.json parses as a 61-entry array with the schema in §3 intact.
  2. AC2 — scripts/build-library.mjs emits a single self-contained library.html that opens offline from file:// with zero external requests; --selftest passes.
  3. AC3 — The page lists all 61 curated primers as cards (title, category, audience, depth, source-count, word-count, date); each card links to its primer HTML, which itself opens from file://.
  4. AC4 — Facet filters (Collection, super_category, category, audience, depth) + free-text search all work client-side; combining facets narrows results; "Clear all" resets.
  5. AC5 — User-generated primers in {docs_path}/primer/ appear under Collection=Yours; the shipped corpus under Curated; both reachable from the one page.
  6. AC6 — /primer browse, /primer list, and bare /primer (no topic) build + open library.html; argument-hint and a #browse phase document it.
  7. AC7 — Phase-5 step-8 listing regen now calls build-library.mjs (bespoke primers.html generator retired); after a normal /primer <topic> run the new primer appears in library.html.
  8. AC8 — With no browser / no Playwright the page is still written and its path printed; opening is best-effort and never fails the run.
  9. AC9 — Tests under skills/primer/tests/ cover build-library against the shipped index; library.html is gitignored; skill-eval [D] floor and the four lints (flags-vs-hints, phase-refs, non-interactive-inline, audit-recommended) pass.

7. Scope & non-goals

  • In scope: the corpus transplant, build-library.mjs, the browse/list verb + #browse phase, the Phase-5 reconciliation, tests, gitignore.
  • Out of scope: regenerating or editing any of the 61 primers' content; changing the /primer research/draft/eval pipeline (Phases 1–5 generation logic); any --json programmatic API (frameworks has one; not requested here — defer); per-primer inline rendering into the library page (we link to standalone files, we do not inline).

8. Story split (route:skill judgement)

One in-scope skill (/primer); the corpus, the generator script, the SKILL.md verb, and the tests are not independently skill-eval'able — they ship or fail together. The G3 default of one-story-per-skill holds → one fused story 260615-dhk, carrying ACs 1–9 as its change-set. Plan + tasks.yaml authored this loop; build via /feature-sdlc build --story 260615-dhk.

9. Build-time risks the plan must hold

  • Corpus base: the transplant git checkout feat/primer-bundle -- <data path> must run on the build branch (branched from clean main); never merge feat/primer-bundle itself.
  • Link paths at file://: the relative href from the emitted page to the skill data/primers/ dir must resolve both in this repo and in an installed plugin cache. Verified live in the dogfood task.
  • Single-file / offline: no external asset refs (CSS/JS/fonts/images) — enforced by a test + the live dogfood (0 network requests).
  • render token gotcha: if any rendering reuses the html-authoring template, strip its leading doc-comment before substitution (see repo memory reference_html_authoring_render_token_gotcha). build-library is self-contained, so likely N/A — but flagged.
