/primer library-viewer parity with /frameworks

Epic 260626-z2p · route: skill · plugin: pmos-learnkit · one story (0g6). Bring /primer's browse library to full feature parity with /frameworks under the shared _shared/library-viewer/ interface, fixing the one genuine contract break in passing.

This is the epic design_doc (the route: skill analogue of a /spec): it names the cross-cutting invariants and the substrate↔consumer boundary the story's /plan must honour. Per-task implementation detail lives in the story's /plan, not here.

1. Problem

Both /frameworks and /primer ship a browse/list verb that builds a self-contained, offline, faceted library page via a scripts/build-library.mjs that consumes the shared zero-dep engine plugins/pmos-learnkit/skills/_shared/library-viewer/lib.mjs (public surface frozen in library-viewer/guidelines.md: esc, renderMarkdown, parseBlocks, renderBody, extractFacets, buildIndex, filterEngine, sortGroups, emitHtml).

A code-level audit (2026-06-26) found that /primer is already a faithful thin consumer of the substrate — it is not forked, not reinventing the engine. But against /frameworks (the reference consumer) it is feature-poor and carries one genuine contract violation:

  • Real bug — hard-coded subtitle count. build-library.mjs builds const sub = `${curatedN} curated…` and passes it as subtitleTemplate with no {count} placeholder. The substrate's .replace('{count}', …) is therefore a no-op, no #subtitleCount element is emitted, and the runtime line subtitleCount.textContent = DATA.length never fires. This violates the hard constraint in guidelines.md ("count is dynamic … never a hard-coded number") that /frameworks and /learn-list both honour. Primer's own selftest cements the bug by asserting the static form (/· 1 of yours/).
  • Feature gap — link-out only. Primer sets cfg.card (link-out cards) and ships no in-page reader, one view only (detailed, with a .viewswitch{display:none} CSS hack over a config that already emits the switch), all-single-select facets, no valueLabels rename seam, and a dead summaryField:'summary' pointing at a field no card carries. /frameworks ships the in-page sidebar reader, three views (list/cards/detailed), multi-select dropdown facets, and the valueLabels seam.
  • Adapter shape drift. Primer builds cards inline across two loaders (loadCurated() + scanUserPrimers()) with no single named toCard adapter or ALLOWED whitelist constant — unlike /frameworks (toCard(r, svgs)) and /learn-list (toCard(r) + ALLOWED).

Decided scope (user, 2026-06-26): full feature parity with /frameworks — adopt the in-page reader, multi-select facets, three views, and valueLabels, plus the bug fix and adapter tidy-ups. The reader's content model is the one design fork (primers are standalone HTML documents, not corpus markdown blocks): the user chose a lazy-loaded sandboxed iframe of the primer's own HTML (D3).

3. Substrate ↔ consumer boundary

The single most important coherence rule: the iframe reader is a substrate capability, added generically — not bespoke primer code. The substrate's current reader renders reader.columns from corpus markdown; an iframe is a new reader mode. It is added to lib.mjs as an additive, default-off seam (e.g. config.reader.mode: 'iframe' + config.reader.iframeField) so any consumer can opt in. Rationale (D1): a primer-only iframe hack would re-fork what the substrate exists to prevent, and a generic seam is equally available to a future /learn-list or /frameworks need.

4. Cross-skill invariants (the design_doc contract)

  • INV-1 — substrate change is additive + default-off. Every existing consumer that does not set reader.mode (or sets 'columns') renders byte-identically to today. Proven by re-running the shipped, un-edited /frameworks and /learn-list build-library test suites green.
  • INV-2 — /frameworks and /learn-list are zero-regression. No edits to their build-library.mjs or corpora. Their selftests and *.test.sh pass unchanged.
  • INV-3 — primer stays a thin consumer. No engine logic forked into primer; new behaviour rides substrate seams only.
  • INV-4 — offline + self-contained preserved. The page (and the iframe-loaded primers) work from file:// with no external <link>/<script src>/<img>; the iframe is sandboxed and points only at sibling on-disk primer files.
  • INV-5 — substrate public surface stable. The 9 frozen exports in guidelines.md keep their signatures; the new reader mode is config-driven, not a new export. guidelines.md is updated to document the seam.
  • INV-6 — single plugin / single release unit. All edits are within pmos-learnkit; rides one pmos-learnkit release. The substrate is learnkit-only (no cross-plugin sync-shared).

5. Decisions

D1 — The iframe reader is a generic substrate seam, never primer-bespoke

Add the iframe reader mode to lib.mjs as an additive, default-off config path, not as primer-local code. Keeps INV-1/INV-3 and lets /frameworks//learn-list adopt it later for free. The alternative (a primer-only iframe injection) was rejected: it re-forks the engine the substrate exists to unify.

D2 — Fix the dynamic {count} subtitle (the real bug) and its selftest

Pass a subtitleTemplate that contains the literal {count} token so the substrate emits #subtitleCount and the runtime sets it from DATA.length. The curated/yours static breakdown may stay as surrounding prose, but the live total must be the dynamic token. The primer selftest assertion /· 1 of yours/ is updated to assert the {count}/#subtitleCount form (it currently locks in the defect).

D3 — In-page reader = lazy sandboxed iframe of the primer's standalone HTML (user decision)

Primers are full standalone HTML documents, so the reader pane lazy-loads the selected primer in a sandboxed <iframe> (closest to frameworks' read-without-leaving feel; works offline from file://). Only the opened primer's src is set (no eager load of all 60+). The iframe carries the primer's own toolbar/styling — acceptable. An "Open in new tab" control is retained in the reader header so the link-out affordance is not lost. Empty-state message when nothing is selected. Considered + rejected: metadata+excerpt+button (lighter, but not "read without leaving" — the user explicitly chose true in-page reading).

D4 — Multi-select facets: category + audience become multi-dropdown; valueLabels for super_category

Facets that are naturally multi-valued or wide become multi-dropdown (category; add + search if option count warrants), and audience multi-dropdown. collection (Curated/Yours), depth (brief/standard/deep), and super_category stay single-select. Adopt the valueLabels presentation-rename seam for super_category (parity with frameworks' SUPER_CATEGORY_LABELS). Exact per-facet kinds finalized in the story plan against live option counts.

D5 — Three views with list default; remove the CSS view-hide hack

Configure views:[list, cards, detailed] with list default (the substrate default), matching frameworks. Delete the .viewswitch{display:none} rule from PRIMER_CSS — it was a presentational workaround over the single-view config.

D6 — Single named toCard adapter + ALLOWED whitelist

Replace the inline card construction in loadCurated() and scanUserPrimers() with one named toCard(rec) adapter (explicit field-by-field, never spread) plus an ALLOWED constant, mirroring /learn-list. Behaviour-neutral; aligns the adapter seam and keeps the PII-safe explicit-whitelist posture.

D7 — Drop the dead summaryField:'summary' config

No primer card carries a summary field and it is absent from searchFields; the config is inert. Remove it so config and data agree. (If a future excerpt field is wanted it can be added deliberately with matching card data + searchFields.)

D8 — Thumbnails are out of scope for v1

Frameworks' thumbField + lazy IntersectionObserver thumbnails have no analogue in primer: primers carry no image asset to anchor a thumbnail. Generating cover tiles is scope creep beyond "match the viewer interface". Documented as an intentional parity gap, not a regression — revisit only if primers gain cover images.

D9 — Tests adopt frameworks-style reader-path assertions

Primer's build-library.test.sh + selftest gain assertions for the new contract: the iframe reader element + lazy-src plumbing, the three-view switch present (not CSS-hidden), the multi-dropdown facet controls, the valueLabels rename, and the dynamic {count}/#subtitleCount. The shipped /frameworks and /learn-list suites are re-run un-edited to prove INV-1/INV-2.

6. Story plan

One in-scope skill (/primer) and one cohesive change set spanning the substrate seam + the primer consumer + tests + the #browse SKILL.md prose — all touching the same files, sequential by nature. Per the route:skill split rule this is a singleton epic → one story; the substrate seam is fused into the same story because it is small, only meaningful with the consumer that exercises it, and cannot be independently skill-eval'd.
