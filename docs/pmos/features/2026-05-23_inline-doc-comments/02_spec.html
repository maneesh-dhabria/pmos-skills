1. Problem Statement

The pmos-toolkit pipeline emits 14 HTML artifacts per feature (12 skills + 2 orchestrator-emitted), but feedback on those artifacts lives only in free-form chat — costing 200–800 tokens of restated location context per revision round, and offering no cross-session audit trail. The locked direction (Finalist A-hybrid in requirements §Solution) attaches feedback to a specific element via a pre-baked overlay, persists threads to a committed sidecar JSON, and resolves them through a single /comments resolve command that dispatches per-thread subagents routed by the originating skill. Primary success metric: ≥80% of opened threads reach resolved status within their feature's review cycle, with ≥90% anchor resolution rate via id-first matching.

3. Non-Goals

  • NOT supporting file:// as a capture path — because FSA refuses file:// origins (research finding #3); the always-serve architecture (D1) makes http://localhost the only supported path.
  • NOT shipping a CI-side drift-hook mirror in v1 (D7) — local pre-commit hook only; CI mirror deferred to v1.1 if drift-bypass becomes a real problem. Accepts --no-verify bypass risk for solo-operator scope.
  • NOT building a browser-tab auto-refresh during live /comments resolve (D14, requirements OQ8) — operator manually reloads to see resolved threads; revisit if it causes confusion.
  • NOT producing a resolution-log artifact (D15, requirements OQ10) — per-run chat summary + git history of sidecars carries the durable trail; no comments-resolved-YYYY-MM-DD.log file.
  • NOT supporting automated sidecar follow-rename in v1 (D12, requirements OQ1) — operator runs git mv on both .html and .comments.json manually; drift hook catches mismatches.
  • Inherits all 9 non-goals from requirements §Non-Goals — mobile/tablet viewing, cross-artifact coherence checks, vendoring Annotator.js / Hypothes.is client, auto-migration of existing artifacts, official Safari/Firefox support for in-place writes, render-comments-into-MD path, thread labels/severity metadata, multi-author UI, clean-share mode that hides overlay for stakeholders.

5. User Personas & Journeys

5.1 Primary persona — solo PM-engineer operator

Per requirements §Problem-Who: single operator running /requirements, /spec, /plan, /wireframes, /prototype, /artifact, /diagram on a laptop. Context: 3–10 pipeline runs per week; every feature has 2–5 revision rounds; today each round costs 200–800 tokens of restated location context. The author field on every thread (requirements D17) reserves space for future multi-operator use without v1 build cost.

5.2 User Journeys

Six journeys documented in requirements §User Journeys: primary single-thread happy path, clarification flow, batch mode with conflict, orphan-after-rewrite, resolver-subagent-failure, Safari/Firefox fallback. This spec inherits all six unchanged. The architecture pivot (S1, always-serve) changes the launch step from "double-click .html → Chrome opens" to "double-click launcher → Chrome opens http://localhost/<artifact>" but does not change any downstream UX step. See requirements §journeys for the step-by-step flows.

5.3 New journey — first-time launcher discovery

  1. Operator cds into a feature folder and runs ls.
  2. Sees the artifact 02_spec.html alongside comments-open.command (macOS), comments-open.sh (Linux), and comments-open.bat (Windows).
  3. Double-clicks the launcher matching their OS (macOS: Finder → .command; Linux: chmod +x on first use then double-click; Windows: Explorer → .bat).
  4. Launcher script runs node assets/serve.js --port=0 --open=02_spec.html (zero argument means "pick first .html in folder" — documented in launcher script header comment).
  5. serve.js binds to an ephemeral port, starts an HTTP server scoped to the feature folder, and shells open/xdg-open/start against http://localhost:<port>/02_spec.html.
  6. Browser opens; overlay loads; FileSystemHandle requested on first interaction; granted handle stored in IndexedDB keyed by localhost:<port>/02_spec.html.
  7. Operator comments freely; saves go through FSA on Chrome/Edge.
  8. When operator closes the browser tab, serve.js stays running until its idle timeout (default 5 minutes); next launcher double-click on the same folder reuses the running server if it finds it via PID file.

7.1 Capture surface

IDRequirement
FR-01Every HTML artifact emitted by the 14 in-scope skills (12 emitters + 2 orchestrator artifacts; see FR-21 inventory) MUST include in its <head>: (a) <meta name="pmos:skill" content="<skill-slug>"> with the emitting skill's slug as the value, (b) <meta name="pmos:plugin-version" content="<current-version>">, (c) <link rel="stylesheet" href="assets/comments.css?v=<plugin-version>">, and in its <body> just before the closing tag: <script src="assets/comments.js?v=<plugin-version>" defer></script>.
FR-02On overlay load (browser fires DOMContentLoaded), comments.js MUST: (a) detect the artifact's origin — if file://, surface a banner with copy-to-clipboard text "Open via comments-open.command (macOS) / comments-open.sh (Linux) / comments-open.bat (Windows) in this folder" and exit (no capture, no save); (b) if http(s)://, attempt FileSystemHandle rehydration from IndexedDB keyed by window.location.pathname; (c) read the sibling <artifact-basename>.comments.json (best-effort via fetch against the serving origin); (d) render existing threads in the side panel.
FR-03Text-selection capture MUST: (a) listen on document for selectionchange; (b) if a non-empty selection lies within <main class="doc">, render a floating "Comment" button positioned next to the selection end-point; (c) on click, open the side-panel compose form with the selection's quoted text + 30-char prefix + 30-char suffix prefilled as the dual-anchor draft.
FR-04Heading capture: when the operator clicks a "Comment on this section" affordance on an <h2> or <h3>, the thread's anchor.id_anchor MUST be set to the heading's id attribute; quote_anchor MUST be the heading text + 30-char body-prefix.
FR-05SVG-region capture: when the operator clicks a <g data-anchor="…"> or top-level <rect|path data-anchor="…"> element inside an <svg>, the thread's anchor.diagram_anchor MUST be set to {svg_id: <parent svg id or null>, shape_id: <data-anchor value>, bbox: null}. For SVGs without data-anchor on any descendant (foreign SVGs from /artifact), capture MUST fall back to {svg_id, shape_id: null, bbox: [x, y, w, h]} where bbox is the click-point bounding-box ±20 px.
FR-06Stakeholder read-only posture: the overlay MUST read localStorage['pmos:reviewMode'] on load; if the value is 'off', the overlay MUST hide its UI surfaces (no floating button, no side panel, no banners). Comments persist in the sidecar regardless; threads remain readable in source.

7.2 Persistence

IDRequirement
FR-10Each artifact <artifact>.html has a sibling <artifact>.comments.json conforming to the schema in §10.1. The sidecar MUST be committed to git (D2).
FR-11Chrome/Edge writes MUST use the File System Access API: showDirectoryPicker() on first save → store handle in IndexedDB → getFileHandle(<artifact>.comments.json, {create: true}) → createWritable() → write(JSON.stringify(sidecar, null, 2)) → close(). Atomic via the FSA write-then-close semantics.
FR-12Safari/Firefox writes MUST mirror to localStorage per-thread as drafts during compose, and on explicit "Save sidecar" click MUST trigger a browser download of the assembled sidecar JSON with filename <artifact-basename>.comments.json. The operator manually replaces the prior file; localStorage drafts MUST clear on successful download.
FR-13FSA permission re-prompt mid-session: every save attempt MUST call handle.requestPermission({mode: 'readwrite'}) before createWritable(); if the returned status is not 'granted', the overlay MUST surface a non-blocking banner "Click to grant write access" without losing the pending draft (draft persists in localStorage until the next granted save).
FR-14Sidecar atomic write: writes MUST go via createWritable({keepExistingData: false}) + full write() + close() — never partial. On close() failure (quota / permission revoked), the prior sidecar is preserved by FSA semantics; the overlay surfaces the failure as a banner with a "Retry save" button.
FR-15The pre-commit drift hook (installed via scripts/install-comments-hooks.sh at .githooks/pre-commit-comments-drift) MUST: (a) detect any staged *.html file under docs/pmos/ for which the sibling *.comments.json exists on disk but is not staged, OR vice versa; (b) refuse the commit with stderr message "comments-drift: <path> is staged but its sibling <sibling> is not. Run 'git add <sibling>' or use --no-verify to bypass."; (c) be installable idempotently; (d) be bypassable via git commit --no-verify per S5.
FR-16Sidecar serialization contract (F5 elevation from §10.3): every write of <artifact>.comments.json MUST serialize with: (a) JSON.stringify(sidecar, null, 2) — 2-space indent, no tabs; (b) LF line endings (\n), never CRLF; (c) trailing newline at end of file. Both the overlay (comments.js) AND the resolver (/comments resolve) MUST honor this contract. Rationale: minimal git diffs (no whitespace churn between operator-edits and resolver-edits) — the format-on-write convention prevents alternating commits from a Chrome-on-macOS overlay and a Linux-CLI resolver introducing newline-mode differences.

FR-22Per-thread dispatch MUST: (a) read the artifact's <meta name="pmos:skill"> content; (b) map the slug to the originating skill's location (plugins/pmos-toolkit/skills/<slug>/SKILL.md); (c) dispatch a Task tool subagent (blocking) with prompt prefix [mode: <current-mode>]\n[output_format: html]\n + the contract-entrypoint call payload (§9.1); (d) reap the structured response.
FR-23Anchor resolution at runtime: (a) try anchor.id_anchor first — query the DOM for the matching id; (b) within that element's descendant text, run Bitap fuzzy search for anchor.quote_anchor.quote with prefix/suffix proximity bias; (c) if Bitap score < Match_Threshold (default 0.5, configurable via skill arg --anchor-threshold), mark thread orphan: true; (d) if anchor.diagram_anchor is present, resolve via SVG data-anchor=<shape_id> match, falling back to bbox overlap (within ±20 px center) if shape_id is null.
FR-24--confirm-each (default mode): for each thread, after the subagent returns a proposed edit, display the diff snippet to the operator via chat output, then issue one AskUserQuestion with options {Accept, Reject + refinement note, Modify proposal, Skip / leave open}. On Reject + refinement note, the operator's next-turn reply is appended to the thread as a user message and the subagent is re-dispatched. Re-dispatch is capped at 2 iterations per thread (S10); on the 3rd reject attempt the option set collapses to {Modify proposal, Skip / leave open}.
FR-25--batch mode: pre-flight computes the anchor-overlap relation per the rules below; wave-planner (reusing plugins/pmos-toolkit/skills/execute/SKILL.md Phase 1 algorithm) partitions threads into waves of pairwise-disjoint anchors; within each wave, dispatch subagents in parallel; between waves, sequential. Edits within a wave MUST be applied right-to-left in the document (S14) so earlier-batch offsets remain valid. Operator confirms the combined diff once per wave. Overlap rules (F4 resolution): (i) two text anchors overlap iff their resolved DOM ranges share any byte position in the rendered artifact's text content; (ii) two SVG anchors overlap iff their resolved bounding boxes intersect by ≥1 pixel (axis-aligned rect-intersect — !(a.x + a.w <= b.x \|\| b.x + b.w <= a.x \|\| a.y + a.h <= b.y \|\| b.y + b.h <= a.y)); (iii) a text anchor and an SVG anchor never overlap (different DOM trees — text edits go to <main class="doc"> prose, SVG edits go to <svg> shape attributes; no shared byte range).
FR-26--auto mode: applies every subagent-proposed edit without operator confirmation. AskUserQuestion-based clarification still surfaces — --auto applies confidently-resolved edits only, not ambiguous ones.
FR-27Per-thread output on success: append a system message {role: 'system', author: 'system', body: '<≤3-sentence prose summary> <diff_ref>', ts: <ISO-8601>}; set status: 'resolved'. The diff_ref is either a SHA when committed (post-S19 the resolver does NOT commit, so the in-run value is always a staged-region reference "staged: lines <start>–<end> in <artifact>").
FR-28Per-thread output on failure (subagent returns success: false): append a system message with body "<error_enum>: <rationale>. Next: <suggested action: rephrase / split / mark won't-fix>"; status remains open; resolver continues to next thread without aborting the batch (D22).
FR-29Clarification flow (interactive mode): when a subagent returns a clarification request (a structured {clarification: {question, options: [..2..4..]}} instead of an edit proposal), the resolver MUST issue a live AskUserQuestion in the chat with the question + options; the operator's pick is fed back to the subagent for a second-pass edit proposal. Capped at 1 clarification round per thread; subsequent ambiguity collapses to a system reply + status: open.
FR-30Apply-edit-at-anchor contract (referenced from _shared/apply-edit-at-anchor.md) — see §9.1 for the full input/output schema and §9.2 for the error taxonomy.
FR-31Resolver MUST stage modified .html + .comments.json via git add but MUST NOT commit (D19). End-of-run chat output includes the operator-facing instruction "Review with git diff --cached then commit."
FR-32--non-interactive mode: process every thread whose subagent does not return a clarification request; for clarification-returning threads, append a system message "deferred — operator input required (re-run interactively for this thread)" and leave status: open; resolver continues. End-of-run summary lists deferred threads.

9.1 Apply-edit-at-anchor entrypoint (S13 + FR-NEW-B)

Each of the 12 originating skills implements a function callable from /comments resolve's per-thread subagent. 9.2 Error taxonomy (FR-NEW-B closed set)

error_enumWhen emitted
anchor_orphanedThe thread's anchor failed to resolve at runtime (id-anchor missing AND quote-fallback Bitap score < threshold). Subagent emits without attempting edit.
edit_conflictedThe proposed edit would touch a region of the artifact that another concurrently-being-applied thread in the same wave has already modified. Wave planner should prevent this; if it fires, the planner has a bug — log + skip.
agent_judged_infeasibleThe subagent assessed the request and concluded it cannot be applied without violating a constraint (broken FR, incoherent with rest of doc, requires re-architecting). Operator sees the rationale + suggested next action.
agent_erroredThe subagent crashed, timed out, or returned malformed JSON. Generic catchall; system_reply contains the captured error excerpt.

Closed set: any new failure mode surfaced during build MUST trigger a back-to-requirements discussion before the enum is extended (FR-NEW-B).

9.3 Idempotency

Per FR-NEW-B: calling the entrypoint twice with the same {thread, anchor} MUST be a no-op if the requested edit is already present in the artifact. Implementation: subagent reads the anchored region, compares against the thread's request, and if the desired state is already present, returns {success: true, diff_ref: "no-op: edit already applied", system_reply: "Edit already present in artifact; marking resolved without changes."}. This protects against re-running /comments resolve after a partial-stage commit.

Semantic-match definition (F3 resolution): "already applied" means ≥80% of the keywords in the thread's last user message body (after stopword removal — using the standard English stoplist of ~150 words: the, a, an, is, are, of, to, in, …) appear as substrings in the artifact's currently-anchored region (or its post-edit DOM neighborhood, ±200 chars). The 80% threshold is a soft heuristic — if the score is between 60% and 80%, the subagent MUST proceed with the edit attempt (avoiding false positives that would skip genuine work) and let the second-pass diff naturally show "no change" if truly idempotent. If <60%, the edit is not yet applied and proceeds normally.

10.2 Schema evolution (S3)

  • Read-tolerant: the resolver and overlay MUST accept any schema_version ≤ current. Forward-incompatible reads emit a warning and downgrade gracefully (e.g., a v2 field unknown to a v1 reader is preserved on write — see "preserve-unknown" rule below).
  • Write-current: every write upgrades the file to the current schema_version. Migration is implicit in the read-then-write path; no separate migrator binary.
  • Refuse-load on newer: if schema_version > current tool, the resolver MUST refuse with stderr "comments.json schema_version=<N> is newer than /comments (current=<M>). Upgrade pmos-toolkit and retry." and exit 64.
  • Preserve-unknown: the read path MUST preserve any unknown top-level keys + unknown thread keys + unknown message keys verbatim on write, so a forward-running tool doesn't strip a backward-running tool's data. (Same posture as package-lock.json across npm versions.)
  • Lineage UUID: generated on first sidecar creation (uuid v4); preserved across all subsequent writes. If a sidecar is copied between artifacts, the lineage UUID stays — drift hook + a future /comments doctor command can detect duplicates by lineage collision.

10.3 Write protocol

  1. Read current sidecar (or initialize {schema_version: 1, lineage: <uuidv4>, threads: []} if absent).
  2. Mutate in-memory copy.
  3. Write via createWritable({keepExistingData: false}) + write(JSON.stringify(sidecar, null, 2)) + close(). JSON serialization uses 2-space indent + LF line endings + trailing newline (so git diffs are minimal).
  4. Failure handling: on close() rejection (quota / permission revoked), keep the draft in localStorage + surface retry banner; the prior sidecar on disk is preserved by FSA semantics.

One-shot test (built by /plan task): take 50 random text spans from docs/pmos/features/2026-04-*/ + 2026-05-0[1-7]_* artifacts (pre-feature corpus); for each, generate a synthetic thread with dual-anchor (id + quote+prefix+suffix); then regenerate the artifact with light AI-rewrite (prose preserved, wording shuffled); run anchor resolver against the regenerated artifact. 15. Rollout Strategy

15.1 Build order (vertical-slice → fanout)

  1. Spike (1 day, /plan task): id-coverage audit across all 14 surfaces (S8). Blocks downstream substrate work until pass.
  2. Substrate (1-2 days): build _shared/html-authoring/assets/comments.js + comments.css + vendor diff-match-patch.js; build the 3 launcher scripts; extend serve.js with --port=0 / --pid-file; write the substrate contract doc _shared/apply-edit-at-anchor.md.
  3. Vertical slice on /spec only (1 day): implement apply-edit-at-anchor in /spec; bake comments.js reference into /spec's emit; write /spec's contract test. End-to-end smoke: comment on this very spec doc, run /comments resolve, watch it apply.
  4. Fanout to remaining 11 skills (2-3 days): per-skill PR or single multi-skill PR — each skill gets the apply-edit-at-anchor entrypoint + bake-comments.js-into-emit + contract test. Per FR-NEW-A this is the 12/12 ship-blocker — no descope.
  5. Resolver CLI (1 day): build plugins/pmos-toolkit/skills/comments/ with SKILL.md + the integration test; the substrate dispatches the apply-edit-at-anchor entrypoint per-thread.
  6. SVG retrofit (1 day, S15): /diagram + /wireframes emit data-anchor on group + shape elements.
  7. Pre-commit drift hook (½ day): install script + the hook itself + its test.
  8. Calibration (½ day, S2): 50-span test set + Bitap threshold tune.
  9. /verify integration (½ day): add check-comments-coverage.sh to /verify's gate set.
  10. Manual smoke + docs (½ day): launcher scripts on macOS, Linux, Windows; download-to-replace on Safari; update repo CLAUDE.md.

Total estimate: 8-10 days of focused work. /plan will produce the exact task breakdown with TDD steps per task.

15.2 Feature flags + degradation

  • No feature flag for the overlay itself — once a skill's emit includes the comments.js reference, every freshly-emitted artifact has the overlay. No staged rollout because the overlay degrades silently when no sidecar exists (zero threads, zero UI).
  • The stakeholder hide-overlay flag (localStorage['pmos:reviewMode']) is per-machine, not a build-time flag. Operator toggles via keyboard shortcut.
  • Degradation paths:
    • file:// open → overlay surfaces banner, no capture (E1).
    • Unsupported browser → overlay surfaces banner, no capture (NFR-04).
    • FSA unavailable (Safari/Firefox) → download-to-replace fallback (FR-12).
    • FSA permission denied → drafts persist in localStorage, retry banner (FR-13).
    • Subagent dispatch fails → thread stays open with system reply, resolver continues (D22 + FR-28).

15.3 Rollback

  • Substrate-level rollback: revert the substrate commits — the launcher scripts + comments.js + comments.css disappear from _shared/html-authoring/assets/; existing sidecars stay on disk (no data loss) but no overlay loads.
  • Per-skill rollback: remove the comments.js reference from that skill's emit template; existing artifacts with the reference baked in keep working (the substrate file still resolves) until the operator regenerates.
  • Sidecar data is never destroyed by rollback — sidecars are committed JSON files, not generated state. git revert + manual cleanup if needed.

15.4 Migration of existing artifacts

Per requirements D24: zero-touch. Existing artifacts (pre-feature) keep their existing HTML; the overlay is NOT retrofitted. The first time an operator regenerates an existing artifact via its originating skill (e.g., /spec --refine), the new emit includes the comments.js reference and the overlay activates from that point forward. Archive artifacts that are never re-engaged stay clean.
