Section A — MSF Analysis (grounded)

J1 — Operator happy path (select → comment → resolve)

DimConsiderationFinding (grounded)
MWhy does the operator open the artifact at all?To review a freshly-emitted artifact before merge/share — already daily-habit per req §ux-motivation ("3–10 pipeline runs/week, 2–5 revision rounds per feature"). Zero CLI cost (D-A-hybrid) reinforces motivation; double-clicking .html works.
MDoes the UI make the next action obvious?Yes, but only after selection. On 01-empty the operator sees the doc + a small "No threads yet" launcher in the corner — the wf-anno "↑ Try selecting any text" is the only affordance hint. New operators may not realise selection is the entry verb. Friction-low.
FPhysical effort to capture a thoughtLow — select, click floating button (02), type, submit. Two clicks + typing. The keyboard equivalent ⌘M exists but is only visible on a dedicated state tab (02-keyboard), not surfaced on hover state (02-hover). Discoverability gap.
FDecision cost in the resolve loopThe 04-inprog Accept / Reject buttons are unambiguous, but the diff snippet is dark-themed (terminal-style) inside an otherwise light panel — context shift adds 0.5–1s of "where am I looking?" cognitive load.
FTrust gradientThree modes (--confirm-each / --batch / --auto) per D6 are not visible in wireframes — the operator must know to pass a flag. Surfacing mode in the launcher or side panel is missing.
SReward for opening a threadStrong on resolve — 04-resolved shows a green pill, a system reply with a diff pointer ("§4.2 first sentence; staged diff lines 124–128. Resolved."), and a clear close-out. Satisfies "no surprise edits" satisfaction signal from §ux-satisfaction.
SAudit trail visibilityThe "Show 3 resolved" toggle on 01-resolved-hidden is small (12px font, 2×8 padding) — keeps the operator from feeling like history is buried, but only marginally. Could be larger.

J2 — Operator orphan recovery

DimConsiderationFinding (grounded)
MWhy bother reattaching?The thread holds work-in-progress feedback that the operator already invested cognitive effort in (1 day ago, per 05-banner caption). Losing it = re-doing the analysis. Strong sunk-cost motivation.
FDetecting the orphanThe orange banner on 01-orphan-present + the pinned-top T5 card on 03-orphan are both clear. However, on 01-orphan the banner is full-width inside the doc body, not at the top of the viewport — if the operator scrolls past the first 200px without noticing, they could miss it.
FReattach mode signalling05-select uses a dashed blue banner + cursor-change implied. Selecting a new span is the same gesture as opening a new thread (selection). Risk: the operator could accidentally open a new thread instead of reattaching. Mode confusion possible.
FCost of the alternative actions"Mark resolved" and "Mark won't-fix" are equally weighted with "Reattach" on 05-view. Won't-fix is the destructive option (per D23, closure-reason lives in last message) — should be visually subordinate, not equal-weight.
SConfidence the reattach worked05-select has a hint "New dual-anchor (id + quote) saves automatically" — good. But no confirmation toast after click. The operator must trust silence = success.

J3 — Operator rejects a resolver proposal

DimConsiderationFinding (grounded)
MWhy reject vs. accept?Operator believes the AI misread intent. The 3-option reject panel (04-reject) is clear: edit / re-prompt / leave open. The "Re-prompt with refinement note (1 of 2)" pill telegraphs the OQ9 cap upfront — good.
FCognitive load of choosing among 3Three same-weight buttons stacked vertically with no recommended-default. Operator must read all three to pick. Should bias visually: "Edit the proposal inline → accept" is the most-used path per D25 rationale — make it primary, the other two secondary.
F"Edit the proposal" costThe wireframe shows the button but not the inline edit affordance. /spec will design it; today it reads like a black box.
SCapping the re-prompt loopThe "1 of 2" indicator is reassuring — operator knows they won't be stuck in a loop. Good.

J4 — Operator on Safari (FSA fallback)

DimConsiderationFinding (grounded)
MWhy use Safari at all?Borrowed laptop, family member's machine, default browser on macOS — incidental but real (D11 calls this out). Operator's motivation is to capture a thought RIGHT NOW; persisting can be one extra manual step.
FRecognising the fallback modeThe yellow info-banner on 06-banner is unambiguous: "This browser doesn't support File System Access. 2 drafts pending · localStorage mirror active." Good. The "Save sidecar (download)" button label spells out the mechanic — good.
FCost of moving the downloaded file06-saved tells the operator to "Move ~/Downloads/02_spec.comments.json next to the artifact, overwriting any prior." Three steps: find in Downloads, drag to right folder, accept overwrite. Sustained friction per session — but D11 accepted this trade-off.
FForgetting the manual move06-saved clears localStorage drafts after download — meaning if the operator forgets to move the file, they lose their drafts on next browser session. Missing safety net: should keep localStorage drafts until the file is verified at the artifact location (post-reload re-detect).
SConfidence after save"✓ 2 drafts persisted" toast on 06-saved is reassuring. But it confirms the download, not the file-placement step. Misleading.

J5 — Stakeholder first exposure to shared artifact

DimConsiderationFinding (grounded)
MWhy click on a highlighted span at all?Curiosity + the visual cue (underline + tint). But the stakeholder has no prior context — they don't know what threads ARE. Significant motivation gap.
FUnderstanding the read-only state08-others-thread surfaces a clear "🔒 Read-only — you are viewing as stakeholder@acme.com" banner. Identity attribution works. However, the stakeholder never set their email — where does it come from? (Likely from git config or a prompt). Wireframes don't show first-time onboarding for stakeholders.
FHiding the overlay08-hidden tells the user to run localStorage.setItem('pmos:reviewMode','on') in DevTools. Hard-fail for non-technical stakeholders: 95% of stakeholders don't open DevTools. Per req §goals "stakeholders viewing a shared artifact see the UI but threads they didn't author are effectively read-only" — this is the explicit posture, but the hide-overlay path is technical-only.
SReward for the stakeholderNone obvious. They can read threads but can't reply. The read-only banner explains why. Acceptable — the stakeholder isn't the primary persona (operator is). But /verify should re-check this when the feature lands.
