Problem

The pmos-toolkit pipeline emits 12+ HTML artifacts per feature (01_requirements, 02_spec, 03_plan, wireframes, prototypes, diagrams, surveys, artifacts, ideate briefs, grill reports, polish outputs, etc.). When the operator spots something wrong during review, the only feedback channel today is free-form chat — they describe where the problem is and what should change, the agent re-anchors against the doc, and the loop costs context every revision round. Threaded back-and-forth on a single concern is not auditable across sessions; "I asked about this last week" lives only in chat scrollback.

The core gap: there is no precise, persisted, AI-readable way to attach feedback to a specific element in a generated HTML artifact and have an AI session resolve it without re-prompting from scratch.

Who experiences this?

The operator of the pmos-toolkit pipeline (initially: a single PM-engineer running /requirements, /spec, /plan, /wireframes, /prototype, /artifact, /diagram in solo workflow) reviewing generated HTML artifacts before merge or stakeholder share. The author field on every thread (D17) leaves room for future multi-operator use without v1 build cost.

Why now?

Three converging conditions make this the right time:

  • HTML-primary artifacts shipped in v2.33.0. The _shared/html-authoring/ substrate (style.css, viewer.js, chrome-strip.js, build_sections_json.js) is mature and consistent across 12+ skills — the bundling-into-emit story is finally viable.
  • Stable section/heading id conventions exist. _shared/html-authoring/conventions.md §3 already mandates kebab-case ids on every <h2>/<h3>, and assert_heading_ids.sh (T22) blocks artifacts missing them. The id-first anchor strategy works because this groundwork is in place.
  • Subagent-driven execution shipped in v2.39.0 (/execute --subagent-driven). The wave-planner pattern (deps + file-conflict relation) is reusable for the --batch resolve mode's anchor-overlap planner (D15) without inventing new substrate.

Goals & Non-Goals

Goals

  • Precision review: the operator can attach feedback to an exact text span, heading, or diagram region in any HTML pipeline artifact — measured by anchor-resolution rate (id-first hit ≥90%; quote-fallback hit ≤5%; orphan ≤5%) at 30 days post-ship.
  • AI-driven edit application: a single command (/comments resolve <artifact>) consumes open threads, applies edits, and writes thread-level audit replies — measured by end-to-end latency <60s per thread in default --confirm-each mode, and by ≥80% of opened threads reaching resolved status within their feature's review cycle.
  • Replace chat as the primary revision channel: for ≥50% of revision rounds across /spec, /requirements, /wireframes, /artifact at 30 days — measured by sampling 10 random feature folders' commit histories.
  • Zero-command capture surface: opening any emitted .html file in Chrome/Edge presents the comment overlay without any prior CLI invocation — measured by binary check (does the bundle load when the file is double-clicked?).
  • Cross-session audit trail: every thread's full history (user messages + system replies + status transitions) is preserved in git via the committed sidecar — measured by spot-checking 5 resolved threads' git logs.

Non-Goals (explicit scope cuts)

  • NOT building a multi-author collaboration UI in v1 — because the operator is solo today; the author field is reserved on every message (D17) for cheap optionality later, but no presence, no conflict-resolution UI, no multi-cursor.
  • NOT supporting mobile / tablet viewing — because the operator works on a laptop; revisit if stakeholders begin reviewing on phones.
  • NOT running cross-artifact coherence checks at v1 (G7) — because directory-level resolve runs each artifact's threads independently; contradictions across 01_requirements + 02_spec are rare in solo flow and surface to the user post-resolve.
  • NOT vendoring Annotator.js or Hypothes.is client (D20) — because comment UIs are simple; the existing libs bring dependency surface and service-backend assumptions; purpose-built keeps the substrate lean.
  • NOT auto-migrating existing artifacts (D24) — because the artifacts the operator actually re-engages with get the bundle on regen; archive artifacts stay clean.
  • NOT enabling Safari/Firefox as officially supported browsers for capture in v1 — because File System Access API coverage is incomplete; those browsers get a download-to-replace fallback (D11), not full parity.
  • NOT building a render-comments-into-MD path (D12) — because .md sidecar is HTML-derived (FR-33); comments are HTML-primary; MD readers see clean docs.
  • NOT adding thread labels / severity / type metadata at v1 (D14) — because the body text + status carries enough signal for solo operator + LLM resolver.

User Experience Analysis

Motivation

  • Job to be done: "When I spot something wrong in a generated artifact, mark exactly which element is wrong and what should happen, so a follow-up Claude session resolves it without me re-supplying location context."
  • Importance/Urgency: Daily — the operator runs the pipeline 3–10 times per week; every feature has 2–5 revision rounds; today's free-form chat costs 200–800 tokens of restated context per round.
  • Alternatives: (a) Continue with free-form chat (current state — high context cost). (b) Maintain a separate "review notes" markdown file (loses anchoring, no AI handoff). (c) Use a hosted annotation service like Hypothes.is (introduces an external service, breaks the "self-contained repo" property). The chosen direction strictly dominates each: cheaper than (a), more precise than (b), self-contained vs. (c).

Satisfaction Signals

  • The operator runs /comments serve — oh wait, there is no /comments serve — just opens the file and starts commenting. No command barrier.
  • After /comments resolve, the operator sees a staged-changes diff and reads ≤3-sentence system replies in each thread explaining what changed. No surprise edits.
  • A thread re-opened a week later still anchors correctly to its target span — id stability + quote fallback proven by spot-check.
  • The operator stops opening chat tabs for "small fix" requests on generated artifacts; the comment overlay becomes the default revision surface.

Solution Direction

Finalist A-hybrid — the locked direction from the ideate brief, reinforced by the deep-grill (25 resolved Qs):

Capture surface (zero commands)

Every HTML artifact emitted by a pipeline skill ships with a <script src="assets/comments.js"> reference pre-baked into the rendered HTML. The substrate file lives at _shared/html-authoring/assets/comments.js (purpose-built, zero-dep, ≤20KB minified — D20, G5). Opening the .html file in any browser activates the overlay: text-selection produces a floating "Comment" button → opens a side-panel thread (Google Docs UX); SVG regions with data-anchor attributes become clickable comment targets; foreign SVGs without data-anchor get bounding-box pixel anchors as fallback (D4).

Persistence (committed sidecar)

Threads persist to <artifact>.comments.json, sibling to the HTML. Schema carries: schema_version, threads[] with id (8-char nanoid, G2), anchor (dual: stable id + quote/prefix-suffix on every thread — D3), status ∈ {open, in_progress, resolved} (D23), messages[] with {role: user|system, author: <user|system|git-config-name>, body, ts} (D17). Sidecar is committed to git (D2); resolved threads stay in the file with overlay default-hiding them (D8). Pre-commit hook refuses partial-pair commits (D21).

AI handoff (one command)

/comments resolve <artifact|directory> reads the sidecar(s), routes per-thread via <meta name="pmos:skill"> to the originating skill's "apply-edit-at-anchor" entrypoint (D1, D18 — all 12+ HTML-emitting skills gain this entrypoint at v1), and dispatches a fresh subagent per thread with the originating skill's context. Per-thread output: either (a) applied edit + ≤3-sentence system reply with diff pointer + status: resolved (D9), OR (b) clarification — live AskUserQuestion in interactive mode (D5) presenting 2–4 options inline; user picks; resolver continues. --confirm-each (default) confirms each proposed edit; --batch uses an anchor-overlap-aware planner mirroring v2.39.0's wave planner (D15); --auto applies all without confirmation. Resolver stages .html + .comments.json but does not commit (D19); user commits manually.

Re-render survival

When /spec --refine (or any skill's regenerate path) rewrites an artifact with open comments: regeneration proceeds without blocking (D3); post-regen re-anchor pass walks threads, trying id-first → quote/prefix-suffix fallback; threads that match neither surface as orphan with a banner in the overlay on next open. Diagram retrofit: /diagram + /wireframes SVG generators add data-anchor on shapes (D4 — 1-day bounded spike, prerequisite to v1 ship).

Stakeholder share posture

Overlay enabled by default for the operator; hidden by setting localStorage.setItem('pmos:reviewMode', 'off') per-machine (D7). Stakeholders viewing the file see the UI but threads they didn't author act as read-only.

Failure handling

Resolver subagent failure (anchor orphaned / edit conflicted / agent judged infeasible / agent errored — G4) keeps the thread open and appends a system reply with the failure taxonomy + suggested next action (rephrase / split / mark won't-fix); resolver continues to next thread (D22). Closure-reason (fix-applied vs. won't-fix) lives in the last system message — no separate status (D23).

User Journeys

Primary journey — single-thread happy path

  1. Operator runs /spec inline-doc-comments; pipeline emits 02_spec.html with bundled comments.js.
  2. Operator double-clicks 02_spec.html → Chrome opens it; overlay activates automatically.
  3. Operator selects "the resolver dispatches a generic subagent" in section §4.2.
  4. Floating "Comment" button appears next to selection → click opens side panel.
  5. Operator types: "this should specify originating-skill dispatch, not generic — see grill Q1." → Submit.
  6. Side panel shows thread #1 with status open; 02_spec.comments.json auto-saves via File System Access API.
  7. Operator commits 02_spec.html + 02_spec.comments.json (pre-commit hook passes — pair in sync).
  8. Operator runs /comments resolve 02_spec.html.
  9. Resolver reads thread #1, reads <meta name="pmos:skill" content="spec">, dispatches a subagent loaded with /spec's context.
  10. Subagent proposes the edit; --confirm-each mode displays the diff snippet to operator.
  11. Operator: accept → resolver applies edit, appends system reply "Changed §4.2 first sentence; see staged diff lines 124–128. Resolved.", flips status: resolved, stages both files.
  12. Operator runs git diff --cached to confirm, then commits.
  13. Next open of 02_spec.html: overlay shows zero open threads; "show resolved" toggle reveals thread #1.

Alternate journey — clarification flow

  1. Operator adds a thread: "the validation logic here looks wrong — fix it."
  2. Operator runs /comments resolve.
  3. Subagent reads the thread + the anchored region + the originating skill's conventions, determines the request is ambiguous (multiple plausible interpretations).
  4. Resolver invokes live AskUserQuestion in the chat session: "Validation issue at <quoted span>: (a) reject when X is null (default Per FR-12), (b) reject when X is empty-string, (c) coerce X to null on empty-string. Which?"
  5. Operator picks (b).
  6. Subagent applies the (b) edit; appends system reply citing the user's choice; flips resolved; resolver moves to next thread.

Alternate journey — batch mode with conflict

  1. Operator opens 02_spec.html with 6 open threads — three target distinct sections, two target overlapping paragraphs in §5, one targets a diagram region.
  2. Operator runs /comments resolve 02_spec.html --batch.
  3. Pre-flight planner groups threads by anchor overlap (G6, D15): wave 1 = {T1, T2, T3, T6 (diagram)} run in parallel; wave 2 = {T4, T5} run sequentially within their overlap group.
  4. Wave 1 dispatches 4 subagents in parallel; all return; resolver presents the combined diff (4 staged regions); operator accepts collectively.
  5. Wave 2 runs T4 sequentially, then T5 (using wave-1's post-edit state). Each surfaces its own diff.
  6. Operator commits.

Error journey — orphaned thread after rewrite

  1. Operator has thread #5 anchored to span "the legacy auth middleware stores tokens in cookies" in 01_requirements.html.
  2. Operator runs /requirements --update (the requirements skill's update path) which rewrites the entire prose.
  3. Post-regen re-anchor pass: thread #5's id anchor still hits the section, but the quote fallback no longer matches because the paragraph was rewritten.
  4. Anchor confidence falls below threshold (Levenshtein ≥0.85 required); thread marked orphan in sidecar metadata.
  5. Operator opens 01_requirements.html next time; overlay shows banner: "1 orphaned thread — content was rewritten. View thread → reattach to new selection / Mark resolved / Mark won't-fix."
  6. Operator chooses "reattach to new selection", selects the equivalent span in the rewritten doc; thread reanchors with new quote anchor.

Error journey — resolver subagent fails

  1. Operator runs /comments resolve 03_plan.html with 3 open threads.
  2. Thread #2's subagent returns "infeasible: the requested edit would violate FR-14 (T4 must not import from T7)."
  3. Resolver does NOT abort the batch (D22); appends system reply to thread #2 with failure type agent_judged_infeasible + the rationale + suggested next actions (rephrase the thread / split into two threads / mark won't-fix); leaves status: open; moves to thread #3.
  4. Threads #1 and #3 resolve normally.
  5. Final summary: "Resolved 2 of 3 threads. Thread #2 left open — see thread body for failure detail."

Error journey — Safari fallback path

  1. Operator opens 02_spec.html in Safari (e.g., on a borrowed laptop).
  2. Overlay activates; banner: "Browser does not support File System Access API. Save sidecar manually after commenting."
  3. Operator drops two comments; each is mirrored to localStorage.
  4. Operator clicks "Save sidecar" → browser downloads 02_spec.comments.json.
  5. Operator moves the downloaded file next to 02_spec.html (overwriting any prior); localStorage drafts cleared.
  6. Commit + resolve flow proceeds identically to Chrome path.
