/flappy-bird visual variety

epic 260624-fly · route: skill · plugin pmos-gamekit · define run 2026-06-24

1 · Context & scope

/flappy-bird is a launch-only pmos-gamekit skill that serves one self-contained file, game/flappy-bird.html (621 lines), through the shared zero-dependency game launcher. The file splits cleanly into a pure engine (window.FlappyEngine — physics, scoring, state transitions; DOM-decoupled, timer-free, seedable; Inv-1) and a render/UI layer that consumes it. Today it ships one teal sky, one yellow round bird, and no attribution.

This epic adds four cosmetic enhancements and one attribution mark — no gameplay change:

  1. Backgrounds — 7 procedural themes, random per game, no immediate repeat.
  2. Bird shapes — 4 procedural silhouettes, random per game.
  3. Bird color — fixed 5-color palette (color + accents), contrast-safe pick.
  4. pmos wordmark — subtle top-HUD link to the repo.

Out of scope: physics, gravity/flap impulse, pipe gaps, difficulty tiers, scoring, ceiling clamp, pause, game-over, session-best + new-best flourish, the launcher/serve contract, and the SKILL.md launch flow. These are behavior-preserved; only the cosmetic render + HUD layers move.

2 · Decisions

#DecisionRationale
D1Variant selection is a pure, seedable pickVariants(seed, prevBg) on the engine surface; render only consumes its output.Preserves Inv-1 — randomization is testable, not a render-only side effect. Real games pass a fresh seed; selftests pin one.
D2Contrast invariant is a hard selftest. For every emittable (bg,color) pair, bird-vs-bg AND pipe-vs-bg luminance deltas exceed the threshold.Protects playability — the bird and pipes are never lost against a background. The picker is constrained to never emit a failing pair.
D3No-immediate-repeat applies to background only. Bird shape & color may repeat."Every game a different background" is the explicit ask. The shape×color combination space (20) is large enough that forcing all-three non-repeat adds branch complexity for no felt benefit.
D4Single file, single story. All four enhancements + selftests in flappy-bird.html (+ tests/run.mjs).One /execute session; no cross-task split (D24 litmus passes) → a singleton epic with one story.
D5Per-theme pipe tint + ground spec allowed.Classic green pipes blend into the forest theme; space has no grassy ground. Each theme coordinates its own pipe/ground so contrast holds everywhere.
D6Variants picked at newGame; the initial ready-preview behind the start overlay stays the default sky theme.A calm, recognizable baseline before play; each game then re-rolls a fresh, different theme.

3 · Data model (pure registries)

3.1 Theme registry — 7 backgrounds

A pure array; each entry is data only (no DOM). The motif draw functions live in the render layer (§5) and are keyed by motif.

// index 0 is the default (sky); order is stable (selftests assert by id)
THEMES = [
  { id:'sky',        grad:[...], ground:{top,fill}, pipe:null,        lum:'light', motif:'clouds'   },
  { id:'ocean',      grad:[...], ground:{...},       pipe:null,        lum:'mid',   motif:'bubbles'  },
  { id:'space',      grad:[...], ground:{platform},  pipe:{...tint},   lum:'dark',  motif:'stars'    },
  { id:'sunset',     grad:[...], ground:{...},       pipe:{...tint},   lum:'mid',   motif:'sunhills' },
  { id:'night-city', grad:[...], ground:{...},       pipe:{...tint},   lum:'dark',  motif:'skyline'  },
  { id:'forest',     grad:[...], ground:{...},       pipe:{...tint},   lum:'mid',   motif:'hills'    },
  { id:'cave',       grad:[...], ground:{...},       pipe:{...tint},   lum:'dark',  motif:'stalac'   },
]
  • grad — sky gradient stops (top→bottom).
  • ground — ground band colors; space uses a thin dark platform, others a tinted strip.
  • pipe — optional {light,dark,shine} tint overriding the default green where needed (D5).
  • lum — coarse luminance class of the upper play area where the bird flies (light|mid|dark), used by the contrast rule. A numeric reference luminance backs it for the selftest delta math.
  • motif — render-layer draw key.

3.3 Bird shapes — 4

Ids: round (classic), sparrow (slim body, pointed beak), duck (broad body, flat bill), owl (round head, ear tufts, large eyes). Each is a pure draw function (§5.2) taking the chosen {body,accent,beak} + the existing wing-flap phase.

4 · The seedable picker (Inv-1)

// pure; lives on the engine surface (e.g. FlappyEngine.pickVariants)
pickVariants(seed, prevBg) -> { bg, bird, color }   // all integer indices
  rng   = mulberry32(seed)                 // the engine's existing seedable PRNG
  bg    = a theme index != prevBg          // D3: never the previous background
  color = a palette index whose lum delta vs THEMES[bg] >= THRESHOLD   // D2: contrast-safe
  bird  = any of the 4 shape indices       // free
  return { bg, bird, color }

Invariants the picker guarantees (asserted by §6 selftests):

  • I-repeat: bg !== prevBg for any seed (when ≥2 themes exist).
  • I-contrast: the emitted (bg,color) always satisfies the bird-vs-bg luminance threshold; and every theme's pipe (default or tint) satisfies pipe-vs-bg (a per-theme property, checked exhaustively).
  • I-determinism: same (seed, prevBg) ⇒ identical result.
  • I-coverage: across many seeds, all 7 backgrounds / 4 birds / 5 colors are reachable (no theme/color is unintentionally excluded for some prevBg).

Contrast feasibility is pre-checked at design time: each background admits at least two palette colors above the threshold, so the color selection never dead-ends regardless of bg. If a future theme/palette edit broke that, the I-coverage selftest fails loudly.

5 · Rendering integration

5.1 Background

The render draw() currently hardcodes a teal gradient + grass ground (lines ~461–476). Replace with: read the active theme (chosen at newGame), paint its grad, call its motif draw fn (subtle, optionally slow-parallax with the existing scroll clock), then paint its ground. Pipes use theme.pipe tint when present, else the default green (lines ~492–503). Motifs are lightweight (a few shapes); no new asset files, no new timers.

5.2 Bird

drawBird() (lines ~505–531) currently hardcodes the yellow round bird. Replace its single body with a switch(birdShape) over the 4 pure draw fns, each consuming the active {body,accent,beak} color and the existing birdFlap phase + eye. The collision radius (BIRD_R) and physics are unchanged — shapes are silhouettes within the same hitbox.

5.3 Selection wiring

newGame() picks a fresh seed (the engine already has mulberry32), calls pickVariants(seed, prevBgIndex), stores the active theme/bird/color + records the new bg index as prevBgIndex for the next game. Initial load paints the default sky theme behind the start overlay (D6).

7 · Selftests & verification

Extend tests/run.mjs (the headless selftest harness). New assertions:

  • Registry completeness: 7 themes each with all required fields; 4 bird shape ids; 5 palette colors each with {body,accent,beak,lum}.
  • I-contrast: iterate every (bg,color) the picker can emit → bird-vs-bg delta ≥ threshold; iterate every theme → pipe-vs-bg delta ≥ threshold.
  • I-repeat: for each prevBg in 0..6 and many seeds, pickVariants(seed,prevBg).bg !== prevBg.
  • I-determinism: pickVariants(s,p) equals itself on repeat.
  • I-coverage: union over many seeds reaches all 7 bg / 4 bird / 5 color indices.
  • Existing physics/scoring/transition selftests unchanged and green.

Plus a headless live dogfood: boot the game, drive a seeded game, assert a non-default theme + non-round/non-yellow bird render, the wordmark href equals the repo URL, a flap→score→game-over→ restart cycle works, and the console is error-free.
