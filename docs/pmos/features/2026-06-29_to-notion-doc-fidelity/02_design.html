Design — /to-notion-doc fidelity fixes

Epic 260629-7hm · Plugin pmos-utilities · Route skill · Status defining · Date 2026-06-29

This is the cross-cutting design contract for a single-story revision epic. It applies seven findings from the first real production run of /to-notion-doc (converting a 242-block POV HTML into a Notion page across two context windows). The conversion itself was structurally correct — verification passed 242/242 — but three post-conversion feedback sessions surfaced an honesty gap in the report, output-quality bugs in the image-stub pipeline, and several source structures the parser did not recognize as conversion candidates. Three architectural choices resolved with the maintainer during this define run are marked resolved.

1. Problem & charter fit

The skill's claimed contract is "convert a local .md/.html/.txt file into a Notion page that preserves structure, remembers preferences, never drops content, writes in resumable chunks, and verifies the result block-for-block." The production run held that contract on completeness — but fell short on three softer obligations the contract implies:

  1. Honest reporting. Two limitations are deterministic and always present (in-page anchor links degrade to plain text because block UUIDs are unknown at write time; fit-page-width="true" only renders full-width once the page-level "Full width" toggle is on). The Phase 5 report declared "clean · Open questions: 0", so the user discovered both with confusion. A faithful converter must disclose what it cannot do, not just what it did.
  2. Output quality of the media stubs. The image-stub pipeline has two owners writing the same block (map-script render + orchestrator injection) → duplicate markers and broken nesting; and the stub's shape surprised users.
  3. Structure recognition. Attribute/description sections (<dl> and label→description div subtrees) and sibling "Annexure"-prefixed sections carry obvious semantic shape the parser flattened to paragraphs / a flat heading run, forcing ~19 post-hoc update_content calls.

All three serve the same charter ("help me maintain a faithful conversion") — this epic closes the gap between "structurally complete" and "faithful + honest". No new skill, no new charter; a revision of the existing pmos-utilities skill.

4. Functional requirements

FR-1 — honest Phase 5 report (F1, D4)

  • FR-1.1 Phase 5 always emits a Post-conversion actions section (even when verification is clean), containing: (a) the count of in-page anchor links that were written as plain text, with the reason (block UUIDs unknown at write time) and the manual-fix instruction; (b) confirmation that fit-page-width="true" is set on all N tables, plus the note that the page-level "Full width" toggle must be enabled for visual full-width.
  • FR-1.2 The "Open questions" / "Run outcome: clean" line must not contradict the Post-conversion actions section — a clean run still lists its standing limitations. The counts (anchor-links-as-text N, tables M) are derived from the map plan, not hard-coded.

FR-2 — single-owner stub injection (F2, D1)

  • FR-2.1 map-to-notion.mjs emits exactly one stub callout per image node and per ambiguous-media node, at the correct list/toggle nesting depth, carrying a fillable caption slot. No second callout is rendered for the same source index.
  • FR-2.2 Phase 2 (SKILL.md) fills the resolved caption + copied-asset path into the map-emitted stub; it MUST NOT inject an additional callout or hand-edit nesting. The dual-write path is removed from the SKILL.md.
  • FR-2.3 A selftest in map-to-notion.mjs asserts that an image node and an ambiguous-svg node each produce exactly one callout in the NFM (no duplicate "Unresolved svg" + stub pair) and that the callout body is tab-indented at the node's depth.

FR-3 — stub shape + expectation-setting (F3, F4, D2)

  • FR-3.1 upload-image.mjs buildStub() returns a single callout block whose body is 🖼 [filename] · Caption: [original caption] followed by the copied relative path and a "drag into an image block to fill" hint. The separate empty image-placeholder block is removed. Caption defaults to alt text, then filename, when absent.
  • FR-3.2 Phase 2 prints an explicit up-front banner whenever ≥1 local image is headed to the stub rung: "Notion MCP cannot create a native image-upload field — local images become callout stubs naming the copied file path, with the caption inline; drag the file into an image block to fill." Shown before output, not after.
  • FR-3.3 notion-blocks.md §5 is updated to describe the single-callout stub shape (no placeholder pair) so the reference stays the one home for the fact.

FR-5 — attribute/description → table (F5, D3)

  • FR-5.1 parse-doc.mjs detects <dl>/<dt>/<dd> structures and <div> subtrees that alternate a short label element with a description element, and emits each as a 2-column table node (header Attribute | Description) instead of paragraphs. Conservative: a div subtree qualifies only when it is a clean alternating label/description pair list (no mixed content), so prose is never mis-tabled.
  • FR-5.2 The Phase 1 census reports detected candidates: "N sections detected as table candidates → converted to 2-column tables". Counts derived from the parser output.
  • FR-5.3 Detected tables flow through the existing §3 table-fidelity contract (cell padding to table_width, fit-page-width="true"), so they are never partially dropped.

FR-6 — annexure grouping (F6, D5)

  • FR-6.1 parse-doc.mjs detects ≥3 sibling top-level sections whose headings share a common leading token (case-insensitive, e.g. "Annexure", "Appendix") and flags them as a groupable cluster.
  • FR-6.2 Phase 1 asks once (Recommended = group): "Group all N <prefix> sections under a single parent toggle?" On accept, the sections are nested under one toggle heading; on decline, they stay flat. AUTO-PICK group under --non-interactive (carries a Recommended option). This is the only structural restructuring offered — it does not generalize.

FR-7 — per-phase status banner (F7, D6)

  • FR-7.1 Each SKILL.md phase ends by printing a single-line banner to chat: Phase N complete — <key counts for that phase>. Phase N+1 starting. (e.g. Phase 2: "2 stubs resolved, 0 user-skipped"). The banner is chat text (survives compaction), not a file write.
  • FR-7.2 The banner content is the phase's salient state, chosen so a fresh continuation can orient without re-reading tool output. It does not replace .write-cursor.json (the durable resume cursor) — the two coexist.

5. Invariants

  • INV-1 — conversion completeness is never weakened. Every source si still reaches a terminal disposition {mapped, stubbed, user-skipped}; Phase 4 verify still fails loudly on any unaccounted-for block. The new table/grouping nodes are accounted-for like any other.
  • INV-2 — every script keeps its embedded --selftest green. parse-doc, map-to-notion, upload-image, chunk-blocks, verify-page selftests pass after the edits; new behavior (single-owner stub, dl→table, annexure detect, caption-inline stub) gets new selftest assertions. Deterministic work stays in scripts (§H — never have the model do arithmetic/parsing a script can do).
  • INV-3 — notion-blocks.md stays the one home for every Notion-API / NFM fact; SKILL.md cites, never restates (§K). The stub-shape change lands in §5; the table-fidelity facts in §3 are unchanged.
  • INV-4 — no regression to the happy path. A source with no images, no dl/label-divs, and no annexure clusters converts byte-identically to before (the new detectors are additive and fire only on their signals).
  • INV-5 — non-interactive contract preserved. The new Phase 1 annexure ask carries a Recommended option (AUTO-PICK group); the Phase 2 banner is informational (no prompt). The frozen non-interactive-block stays byte-identical; allowed-tools stays Bash, Read (no AskUserQuestion added to the frontmatter list — the existing asks already degrade per Platform Adaptation).

6. Conformance

The produced skill must conform to skill-patterns.md §A–§L and pass skill-eval.md (both the [D] deterministic and [J] judge halves). Repo policy from CLAUDE.md applies: canonical skill path (plugins/pmos-utilities/skills/to-notion-doc/SKILL.md), the non-interactive inline-block byte match (tools/lint-non-interactive-inline.sh), the recommended-option audit (tools/audit-recommended.sh), and the flag/phase-ref lints (tools/lint-flags-vs-hints.sh, tools/lint-phase-refs.sh). Release prerequisites (version bump, changelog, manifest sync) are /complete-dev's job at Loop 3 — not tasks in this story's waves (§G).

7. Story

One story — 260629-sch (route: skill, plugin pmos-utilities, no dependencies): all seven findings in one /execute run over SKILL.md + parse-doc.mjs + map-to-notion.mjs + upload-image.mjs + notion-blocks.md. Its ACs are the seven findings' fixes (the FRs above), verbatim. See the story item and stories/260629-sch/03_plan.html.

8. Open questions

None blocking. The three architectural forks (stub owner, stub shape, dl→table policy) were resolved with the maintainer during this define run. The dl/label-div detector heuristic (FR-5.1) is deliberately conservative — if it proves too narrow or too broad in the next real run, tune the qualifying predicate, not the architecture.
