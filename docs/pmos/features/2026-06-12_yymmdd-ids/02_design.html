Year-prefixed backlog ids — <YYMMDD>-<rand3>

Epic 0612-jjs · route: skill · pmos-toolkit · singleton epic (D18) wrapping one build story (0612-d14). Define-stage design contract — the build story (/plan → /execute → /skill-eval → /verify) cites these anchors.

Context

Maintainer feedback (2026-06-12): "Can we include the year number as well in the numbering logic in /backlog rather than only mmdd-rnd → yymmdd-rnd is what I am proposing."

The id scheme shipped one bump ago in epic 0020 (pmos-toolkit/v2.68.0) as the coordination-free <MMDD>-<rand3> form (e.g. 0612-k3f) — see plugins/pmos-toolkit/skills/_shared/tracker-crudl.md §2 (the single home of the id format and validator). The MMDD date-hint omits the year, so a bare 0612 is ambiguous across years — last June and this June produce indistinguishable prefixes. Prepending a 2-digit year (260612-k3f) makes the date-hint complete and self-dating. This is a direct, mechanical extension of 0020, not a redesign.

Decisions captured at define

#DecisionRationale
D1 Additive triple-accept — no migration. Only newly-minted ids carry the year. The validator widens to accept all three permanently-valid forms: legacy 4-digit serial, current <MMDD>-<rand3>, and new <YYMMDD>-<rand3>. Existing ids, filenames, branches, claim locks, and parent:/dependencies: refs are never rewritten. Rewriting an existing id is a corruption (it is referenced by branches, claims, and released changelog entries — tracker-crudl §2.1). The 0020 scheme already grandfathers legacy serials; this adds a third grandfathered arm rather than a one-time migration. Low risk, ~6 code sites.
D2 Scope = both /backlog and /mytasks mint year-prefixed ids going forward (maintainer-chosen at define). The feedback names /backlog; the maintainer extended it to /mytasks for one uniform date-based scheme across both trackers. Both live in pmos-toolkit — one release unit, D17 satisfied.
D3 veto-able at build /mytasks adopts the same <YYMMDD>-<rand3> scheme (not a year+serial hybrid). /mytasks ids today are pure 4-digit serials with no date prefix to extend, so "add the year" necessarily means moving to the only date-based scheme in the system. Existing serials stay valid under the triple-accept validator. Inventing a third <YYMMDD>-<serial> hybrid would fork the scheme and re-introduce a counter. Trade-off: /mytasks ids lose their human-incrementable "task 42" quality (the rand3 suffix is collision-free-by-construction, harmless for a single store). If the maintainer would rather keep /mytasks on human serials, drop the mytasks task at build — it is isolated.
D4 Ids remain NOT a sort key. The year makes the date-hint unambiguous, but the rand3 suffix keeps ids non-monotonic within a day. The §2.2 invariant — "no consumer may lexical-sort ids as a chronology proxy; sort on created:" — is preserved verbatim. Decouples this change from any sort-order rework; created: stays the chronology source (§3/§5). No /backlog next ordering change.

The new scheme (exact)

Format of a newly-minted id — <YYMMDD>-<rand3>:

  • YY — 2-digit year (getFullYear() % 100, zero-padded).
  • MM DD — 2-digit month + day (unchanged).
  • rand3 — 3 lowercase Crockford-base32 chars, alphabet 0123456789abcdefghjkmnpqrstvwxyz (no i l o u) — unchanged.
  • Example: 260612-k3f.

Canonical triple-accepting validator — this exact string is duplicated byte-identically in two homes (mint-id.mjs and tracker-crudl.md §2.1):

^([0-9]{4}|[0-9]{4}-[0-9a-hj-km-np-tv-z]{3}|[0-9]{6}-[0-9a-hj-km-np-tv-z]{3})$

The three arms are unambiguous — exactly 4 digits (legacy serial); 4 digits + - + 3 base32 (current); 6 digits + - + 3 base32 (new). The - separator plus the digit count disambiguates all three.

Correctness trap (must-have regression test): the naive "swap [0-9]{4} → [0-9]{6}" widening drops the current <MMDD>-<rand3> arm and would silently invalidate every existing 0612-xxx id (epics 0612-h2j, 0612-w4e, 0612-jjs; stories 0612-ejq, 0612-2w7, 0612-d14; …). The validator MUST keep all three arms; the build adds an explicit test asserting a current 0612-k3f id still validates.

Out of scope (v1)

  • Migrating existing <MMDD>-<rand3> or legacy 4-digit ids (D1 — additive only).
  • Making ids a chronological sort key (D4 — created: stays canonical).
  • A <YYMMDD>-<serial> hybrid for /mytasks (D3 — one uniform scheme).
  • 4-digit-year (YYYY) prefixes — the maintainer proposed 2-digit YY.

Epic acceptance criteria

  • New ids minted as <YYMMDD>-<rand3> in /backlog (and /mytasks per D3); no max+1 reintroduced.
  • The dual→triple validator accepts all three forms; every existing id (4-digit, MMDD-rand3, new) parses and is never rewritten.
  • Format + validator changed only in the single canonical home (tracker-crudl.md §2), cited — not restated — elsewhere (§K).
  • All id-consuming sites stay scheme-agnostic; id-scheme.test.sh green including the new triple-accept regression check; /backlog add stays a frictionless single-tool-call flow.
  • Skill rubric green (skill-eval.md floor 43/47) for the two edited skills.
