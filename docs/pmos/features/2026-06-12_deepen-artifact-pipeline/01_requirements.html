Requirements — Deepen /artifact into a document pipeline

Tier: 3 · Mode: skill-feedback · Target: plugins/pmos-toolkit/skills/artifact · Manifest: 2.64.0 → 2.65.0 · Epic: #0010 · Source brief: docs/pmos/ideate/2026-06-12_deepen-artifact-pipeline.html

Standing acceptance criteria: the revised skill MUST conform to reference/skill-patterns.md §A–§L (frontmatter, triggering, structure/progressive-disclosure, gates §H, flags §I, phases §J, one-fact-one-home §K, subagent dispatch & model selection §L) and the repo CLAUDE.md (canonical skill path, manifest version-sync, /complete-dev as release entry point).

1. Problem

Producing a high-stakes document (PRD, strategy memo, RFC, postmortem) today means hand-chaining five skills in the right order: /artifact to draft, then /diagram, then /polish, then /grill — and remembering to research first and to get stakeholder critique. Most users skip the steps that matter (research, critique, stress-test) because they're separate, manual, and easy to forget. /artifact already owns drafting, templating, context-gathering, and a single reviewer loop; it is the natural home for the missing front-end (research), the missing critique depth (multi-stakeholder), and the orchestration of the finishing skills.

2. Goals & non-goals

Goals

  • One /artifact run can take a document from context → researched → drafted → multi-persona-critiqued → diagrammed → polished → grilled, scaled by a single --depth dial.
  • Preserve the fast-draft path: a quick draft must not pay the full pipeline tax.
  • Reuse existing machinery (template add research flow, the reviewer loop, the HTML substrate) rather than reimplementing.
  • Every new heavy stage degrades cleanly in --non-interactive so /artifact stays headless-safe and usable as a /feature-sdlc sub-step.

Non-goals

  • A separate /doc-pipeline skill (rejected in /ideate — deepen /artifact instead).
  • Folding /diagram, /polish, /grill source into /artifact — they stay standalone; /artifact orchestrates them.
  • Real prose rewriting in the comment-resolver shim (still deferred).
  • Re-grilling on every refine/update run by default (see OQ).

4. Per-capability requirements

F1 — Template proposal when none matches

FR-3 In the create flow, add a no-template-match branch: when the requested doc type resolves to no built-in or user template, offer to propose a template by invoking the existing template add research-grounded flow (T.1–T.6) inline (research subagent + section alignment + eval-gen).

FR-4 After a successful create with a proposed template, offer to save it to ~/.pmos/artifacts/templates/. Default = No (premortem: avoid a junk template library); the save offer is gated on the new-template best-practice checklist passing.

FR-5 Add reference/new-template-guidelines.md — generic best practices for template authoring (section design, eval-criteria design, persona set, length target) used by the proposal flow and the save-checklist.

F2 — Document-length target

FR-6 Templates carry an optional length_target in frontmatter (e.g. ~1500 words / 2–3 pages / tight). When proposing a NEW template, ask the user for one (gap-question). Pass it to generation as informational steering, never a hard gate (per /primer word-target learnings — word targets are routinely missed; don't pad).

F3 — Research phase (deep only)

FR-7 Before drafting, when --depth deep: first decide "is external research warranted?" — auto-skip for internal/template-backed docs with sufficient gathered context; log the decision. FR-8 If warranted, propose a research plan (3–6 questions/areas) and get user approval before any fan-out. FR-9 On approval, fan out general-purpose subagents (per the /ideate music-coach learning: stable idx per item, strict output contract, "omit rather than guess; mark low-confidence", validating+tolerant merge). FR-10 Save the synthesized research as a standalone doc under <feature_folder>/research/<slug>-research.html with source provenance — verifiable URLs only; no source, no claim. FR-11 The draft consumes the research doc as additional gathered_context.

F4 — Multi-persona critique (standard + deep)

FR-12 Add a persona-panel phase, distinct from the existing eval reviewer (which checks template eval.md criteria). FR-13 Personas are template-defined (a template's frontmatter may carry a personas: list, e.g. PRD → eng-lead, design, GTM, exec) OR recommended from context (3–4) when the template defines none. FR-14 Each persona critiques the draft in a parallel general-purpose subagent, returning grounded findings (≥40-char quote anchors per _shared/reviewer-protocol.md). FR-15 Findings are reconciled with the user (per _shared/findings-dispositions.md, ≤4/batch) before the draft is updated. NFR-1 The panel must demonstrably add findings the eval reviewer doesn't (assumption test in brief) — if it routinely duplicates, that's a signal to reconsider.

F5 — Diagram pass (deep only)

FR-16 When --depth deep, after the draft stabilizes, propose a short list of candidate diagrams (type + what it clarifies + target section). FR-17 On user approval, run each via a /diagram subagent in non-interactive mode with: mandated full-viewBox background <rect> (dark-mode safety, per /primer learning), a fixed insertion anchor. FR-18 Validate each SVG before inline insert — heading-id + sections.json + comments-coverage smoke must stay green; skip-on-failure with a logged note (never blind-insert, per premortem). FR-19 Remember "always run diagram pass" as a per-project preference (e.g. .pmos/settings.yaml :: artifact.diagram_pass).

F6 — /polish always (all depths)

FR-20 Run /polish as a finishing pass on every create. FR-21 Auto-apply mechanical findings only; route any voice-risk / high-risk finding through per-finding confirm (premortem: "always run" ≠ "always auto-apply"). FR-22 Invoke via the main-agent path (verify in /spec — /polish warns subagents can't invoke it; /artifact's create flow runs in the main agent, so this should hold, but the spec must confirm and document the constraint, including what happens when /artifact itself is dispatched as a subagent by /feature-sdlc).

F7 — /grill mandatory (all depths, scaled)

FR-23 Run /grill on every create, scaled by --depth (light/standard/deep). FR-24 Mode-aware: interactive turn-by-turn interrogation when a human is present; in --non-interactive, degrade to a written adversarial findings pass (no turn-by-turn) so /artifact stays headless-safe (premortem H-risk). Log the degradation explicitly.

5. Resolved design forks (from pressure-test)

  • D1 — Default depth = standard (user decision). The full battery is earned by --depth deep; brief is the explicit fast-path. Honors "grill+polish always" (both run even at brief) while protecting the fast-draft case.
  • D10 — Non-interactive composition is a hard requirement (FR-22, FR-24). The /spec must include a spike confirming each child skill (/grill, /polish, /diagram) can be invoked from /artifact and degrades headless; this is the #1 build risk.

6. User journeys

  1. Quick draft (#journey-fast): /artifact prd --depth brief → gathered context → draft → /polish (mechanical) → light /grill → save. Fast, today's speed preserved.
  2. Default doc (#journey-standard): /artifact prd → draft → persona panel (reconcile findings) → /polish → /grill → save.
  3. High-stakes doc (#journey-deep): /artifact "board strategy memo" --depth deep → "no template matches → propose one?" → research-warranted? → plan+approve → fan-out → research/ doc → draft → persona panel → diagram pass (propose+approve+validate+insert) → /polish → deep /grill → offer-save-template → save.
  4. Headless (#journey-headless): /artifact prd --non-interactive (e.g. from /feature-sdlc) → draft → persona panel (auto-reconcile, buffer OQ) → /polish (mechanical only) → /grill (written-findings) → save. No hard-block.

7. Acceptance criteria

  • All FR-1…FR-24 implemented; --depth default = standard; fast path (brief) demonstrably skips personas/research/diagram.
  • Each child-skill invocation (/grill, /polish, /diagram) works in --non-interactive with documented degradation; verified by a spike + a test.
  • Research saved to research/ with verifiable URLs; diagrams validated before insert (smoke green).
  • Persona panel produces grounded (quote-anchored) findings reconciled before draft update.
  • Revised SKILL.md conforms to skill-patterns.md §A–§L and passes skill-eval.md (the Phase 6a/7 gate).
  • No version-bump / changelog / README / manifest tasks in /execute waves — those are /complete-dev prerequisites (§G).
