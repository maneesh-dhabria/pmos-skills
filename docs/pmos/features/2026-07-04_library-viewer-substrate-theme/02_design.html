Design — library-viewer substrate re-theme

pmos-learnkit · epic 260704-vde · route: skill · the cross-cutting coherence contract for stories 260704-m7f (re-theme the shared library-viewer substrate) and 260704-v4a (refresh the 61 bundled primer corpus files). Stories cite this doc by anchor. Decisions D1–D8 maintainer-confirmed (three material forks resolved interactively at define time). No epic /spec — this design doc is the skill-epic contract; per-skill spec folds into each story's /plan.

1. Overview

Three pmos-learnkit skills — /frameworks, /primer, /learn-list — render an offline, filterable browse / library page through one shared substrate: plugins/pmos-learnkit/skills/_shared/library-viewer/lib.mjs. That substrate carries its own hand-written, hardcoded BASE_CSS — a dark, blue-accent, system-sans theme with a private token vocabulary (--bg:#0f1320, --accent:#6ea8fe). It looks nothing like the canonical pmos HTML substrate (_shared/html-authoring/assets/style.css, the "Editorial Technical" theme — warm-paper #f8f5ef, burnt-orange #b8431a accent, a serif/sans/mono three-voice type system, light-default with dark via prefers-color-scheme) through which every individual primer, spec, plan, and learn-list artifact already renders.

The result is a jarring seam: a user opens a warm-paper light primer, clicks "browse the library", and lands on a dark blue page that shares no visual DNA with it. This epic closes that seam. Two faces, one rule — every page a learnkit skill emits reads as one design system:

  • Browse pages — re-theme the shared library-viewer substrate so all three browse/library pages inherit the Editorial Technical look (story 260704-m7f).
  • Individual pages — the only off-theme individual artifacts are the 61 bundled primer corpus files, whose inlined CSS is a stale "Mono Minimal" snapshot (#fafafa stone / #c2410c) of the same token system, not the current warm-paper theme. Refresh them to the current style.css (story 260704-v4a).

Live-generated primers and learn-list research artifacts already render through the current substrate (they link to assets/style.css) and are therefore already correct — out of scope. /frameworks emits no individual per-item pages at all (in-page reader panes only).

Because the three consumers do not hand-author CSS, a re-theme of the substrate propagates to all three automatically — provided no consumer passes old-token CSS through the extraHead seam (audited in 260704-m7f, §4.3).

3. Invariants

Rules every story must preserve. A change that breaks one is wrong even if it "looks right".

  • INV-1 — offline / zero-request. The emitted page must remain fully self-contained and render from file:// with no network: no external <link>, no CDN, no @import. The canonical style.css is read at build time and inlined into the single output HTML — the offline invariant is unchanged (the consumers' tests already assert "no external stylesheet"; those assertions must still pass).
  • INV-2 — single source of truth. After this epic the browse-page palette, typography, and light/dark behavior come from the canonical style.css, not a hand-copied token block. The substrate reads ../html-authoring/assets/style.css (a same-plugin sibling — no cross-plugin dependency) at build time; the library-viewer's own CSS defines only the components style.css does not (browse layout, cards, facets, chips, reader pane) and references only --pmos-* tokens — never a hardcoded color. Drift is thereby structurally impossible: a future substrate theme change is picked up on the next browse-page build.
  • INV-3 — light default + prefers-color-scheme dark. Browse pages match the substrate exactly: warm-paper light by default, dark when the OS prefers it. Dark mode is inherited for free because every component color is a --pmos-* token that style.css already flips under @media (prefers-color-scheme: dark). No component may hardcode a dark-only color.
  • INV-4 — skill-agnostic substrate. lib.mjs stays skill-agnostic (its selftest.sh greps for this, invariant "D12"). Reading a sibling substrate file is not skill-specific and must keep the selftest green; the style.css path is resolved relative to the module's own location (import.meta.url), with a clear error if the file is absent.
  • INV-5 — idempotent corpus refresh. The bundled-primer re-theme (260704-v4a) replaces the inlined substrate CSS block(s) in place and is byte-idempotent: a second run is a no-op diff. It touches only the <style>/CSS region, never the primer body (<main>) — so it stays orthogonal to the in-flight primer content epics (§6).
  • INV-6 — no functional/interaction change. This is a re-skin, not a redesign. Search, faceting, view-switch (compact/detailed/list), the reader pane, and iframe/card-link modes behave exactly as before; only color, type, spacing, and light/dark change.

4. Story 260704-m7f — re-theme the library-viewer substrate

The core story. Rework lib.mjs's styling so emitHtml() produces Editorial-Technical browse pages.

4.1 — Inline the canonical stylesheet

At build time, read ../html-authoring/assets/style.css (resolved via import.meta.url) and inline it as the base layer of the emitted <style> block, ahead of the library-viewer's component CSS. This supplies the :root --pmos-* tokens, the prefers-color-scheme: dark overrides, the base element styling, and the bundled JetBrains-Mono @font-face. On a missing/unreadable file, fail loud with a clear error (do not silently fall back to the old dark CSS — a silent fallback would reintroduce the seam this epic removes).

4.2 — Rewrite component CSS onto --pmos-* tokens

Replace the dark BASE_CSS token block and every hardcoded color in the library-viewer's own component rules (masthead/header, .layout/.listing, cards + view modes, single-select + multi-dropdown facets, applied-filter chips, search box, the .reader sidebar + its animation, and IFRAME_READER_CSS) with references to --pmos-* tokens (--pmos-surface, --pmos-border, --pmos-rule, --pmos-text, --pmos-muted, --pmos-accent, --pmos-accent-bg, spacing/radius/font tokens). The three-voice type system applies: sans for controls/headings, serif for reader body prose, mono for structural/metadata chrome. Keep every existing class name and DOM structure (INV-6).

4.3 — Audit consumer extraHead CSS

Inspect all three build-library.mjs (/frameworks, /primer, /learn-list) for skill-specific CSS passed via the extraHead seam or via card/reader config that hardcodes old-theme colors (e.g. a blue accent, a dark card background). Reconcile any such CSS to --pmos-* tokens so the three pages are consistent. If none exists, record that finding — the audit is a required task, its outcome is not.

4.4 — Update the substrate's own guards

_shared/library-viewer/tests/lib.test.mjs (frozen public-API tests) and guidelines.md assert specifics of the old CSS (e.g. the dark --bg token, "no external stylesheet"). Update them to the new contract: inlines style.css, uses --pmos-* tokens, still emits no external <link> (offline preserved), still passes selftest.sh (skill-agnostic grep + node --test). The public API surface (emitHtml and the named exports) is unchanged — this is a styling change behind a stable interface.

5. Story 260704-v4a — refresh the 61 bundled primer corpus files

The bundled primers (plugins/pmos-learnkit/skills/primer/data/primers/*.html, 61 files) are self-contained: the substrate CSS is inlined (they cannot link to an assets/ dir and stay single-file-portable). Their inlined CSS is a stale "Mono Minimal" snapshot. No inline/freeze script exists today — this story builds one.

  • New idempotent CLI (e.g. primer/scripts/retheme-corpus.mjs, zero-dep Node ESM, mirroring the established backfill-CLI pattern) that, for each corpus file, replaces the inlined substrate <style> region with the current _shared/html-authoring/assets/style.css content (and comments.css if the corpus files inline it too — the transform must detect and replace exactly the substrate style block(s), matched by the substrate header-comment marker, not a naive "first <style>").
  • Run it over all 61 files; the diff must touch only the CSS region of each file (INV-5), never <main>. A second run is a no-op (idempotent).
  • A unit/integration test asserting: the marker-matched block is replaced, body bytes are unchanged, and re-running is byte-idempotent, on a small fixture.

Dependency: none on 260704-m7f — different code path (browse-page substrate vs. individual primer docs). The two stories are independent and may build in either order.

6. Coordination with in-flight primer epics

Two active epics already touch the same 61 bundled primer files: 260704-rgt (References section — injects a ## References block into <main>) and 260704-dgq (curated-corpus enrichment — rewrites source sets). Story 260704-v4a touches only the <style>/CSS region (INV-5), so it is structurally orthogonal to both (they edit body content). Build-time (Loop 2/3) sequencing note: whichever lands second re-touches the files, but because the regions are disjoint the transforms compose — the v4a CLI's idempotence + marker-scoped replacement means it can safely re-run after a References/enrichment pass. No hard dependencies: edge is declared; the release train orders them.

7. Decisions

  • D1 — reuse canonical style.css as single source of truth (maintainer-confirmed). The substrate reads + inlines the sibling style.css and layers component CSS on --pmos-* tokens, rather than hand-porting the theme values into BASE_CSS. Rationale: zero token drift — the browse pages follow future substrate theme changes automatically. Cost: a more invasive one-time component-class remap (§4.2). Rejected alt: hand-port values (simpler now, drifts later).
  • D2 — include the 61 bundled primers (maintainer-confirmed). The individual pages in scope are exactly the bundled corpus (live-generated pages are already correct). Rejected alt: library-viewer browse pages only.
  • D3 — light default + prefers-color-scheme dark (maintainer-confirmed). Match the substrate exactly; the browse page changes from always-dark to light-by-default. Rejected alt: keep dark-first.
  • D4 — same-plugin sibling read, not cross-plugin. pmos-learnkit carries its own synced _shared/html-authoring/ copy, so the substrate reads the learnkit-local style.css — no cross-plugin dependency, no dangling _shared/ cite. (The style.css is byte-identical to the toolkit canonical per the sync intersection.)
  • D5 — build-time inline, not view-time link. Reading + inlining at build keeps INV-1 (offline) intact while satisfying INV-2 (single source). The read happens once per browse-page build, not per view.
  • D6 — fail loud on a missing style.css. No silent fallback to the old dark CSS; a broken substrate path is a build error the author must fix.
  • D7 — corpus refresh is CSS-region-scoped + idempotent (INV-5), keeping it orthogonal to the in-flight primer body epics (§6).
  • D8 — re-skin only, no redesign (INV-6). Faceting, search, view modes, and reader behavior are untouched; this epic changes appearance, not interaction.

8. Risks

  • Hidden old-token CSS in a consumer extraHead would leave one page inconsistent. Mitigated by the §4.3 audit task (required).
  • Contrast/legibility regressions from mapping a dark layout onto light tokens (e.g. a card border that vanishes on warm paper). Mitigated by a live render-and-eyeball of all three browse pages in both color schemes during 260704-m7f's /verify (browser-evidence gate).
  • The corpus transform mis-matching the style block (replacing the wrong <style>, or corrupting body). Mitigated by marker-scoped replacement + the body-bytes-unchanged test (§5) and INV-5 idempotence.
