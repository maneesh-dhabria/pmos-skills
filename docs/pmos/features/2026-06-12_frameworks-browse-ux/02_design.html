Epic 0016 — /frameworks browse-UI fixes

Define-loop design contract · route: skill · pmos-learnkit · synthesized from --from-feedback · 2026-06-12

Context

The /frameworks skill (pmos-learnkit v0.19.0) ships a single self-contained index.html browse library, generated by scripts/build-library.mjs (the HTML/CSS/JS are emitted by buildHtml()). Five pieces of user feedback target the browse experience. All five live in one file's emit (plus its --selftest, the structure tests, and the SKILL.md prose) — a cohesive single-skill change set, wrapped here as a singleton epic with one story (D18 / G3).

This is a lean define (D-lean, mirroring epic 0015): no epic-level /requirements or /grill. Story 0017's acceptance criteria plus this design page are the contract; the multi-select filter work was grounded by dedicated UX research (NN/g, Baymard, GOV.UK, WAI-ARIA APG — summarized in § Filter research).

Coherence invariants (must hold after the change)

  • Zero-dependency, single-file, offline. No React/CDN/build step; the page must keep working from file://. No external asset refs (FR-ING-1).
  • Generator is the single source. All UI lives in build-library.mjs's emitted string; index.html is a build artifact, re-derivable via the build / apply-rederive.mjs path. No hand-edits to index.html.
  • Corpus stays semantically stable. The product-area rename is presentation-layer only — super_category values in frameworks.json, match.mjs, --json, and the situation taxonomy are untouched (D5).
  • Selftest is the gate. build-library.mjs --selftest + tests/structure.test.sh + tests/build-library.test.sh must stay green; assertions that encode the old UI (e.g. data-view="detailed" class="active", the More filters disclosure holding dtFilter, the inline tagrow cloud) are updated to encode the new contract, not deleted.

The five changes

1 — Selection highlight in all views + no page auto-scroll

Today only the detailed card gets a .open outline; list and compact rows show no selected state. And opening the reader re-runs render() (full groupsEl.innerHTML rebuild) and shifts the layout (sidebar takes ~42% width), which jumps the scroll position away from the clicked item.

D1. Apply a .selected class to the active item's element in all three view renderers (card already has .open; add an equivalent for the list <li>/anchor and the compact anchor), styled with the accent treatment. In openReader(), capture window.scrollY before the render() rebuild and restore it after, so selecting a deep item does not scroll the page. Keep preventDefault() on in-list anchor clicks (already present) so hash-nav never scrolls; the reader's own scrollTop=0 stays (it scrolls the panel, not the page). Deep-link load (#id on first paint) may still scroll-to — that is desired on initial load, distinct from in-page selection.

2 — List view is the default

D2. Initialize state.view='list' and move the active class from the detailed button to the list button in the emitted markup. No behavior change to the other views.

3 — Icons on the view-toggle buttons

D3. Add a small inline-SVG glyph to each of the three toggle buttons — a representative icon per view (compact = dense rows / comma-list, detailed = card grid, list = lines). Icons are inline SVG (offline-safe), paired with the existing text label and aria-label (icon is decorative aria-hidden; the button keeps its accessible name). "Avatar icons" is read as a per-view glyph in the toggle, not user avatars.

4 — Decision-type & tags as multi-select dropdown filters + applied-filters bar

Today: tags are an overwhelming inline chip cloud (tagrow, 100+ chips); decision type is a single-select <select> buried in a More filters disclosure. Replace both with dropdown multi-selects and a visible applied-filters bar. Design grounded in § Filter research.

D4b — applied-filters bar. An always-visible bar directly above the results lists every active filter as a removable chip labeled with its facet (Decision: prioritize ✕, Tag: pricing ✕, Area: Product Management ✕) plus a Clear all control. Removing a chip clears that one filter and unchecks the matching box. The chip bar and dropdown checkboxes stay in sync.
D4c — accessibility. Each dropdown trigger is a <button aria-expanded aria-controls>; options are native <input type=checkbox> (free keyboard + SR semantics — no role=listbox over checkboxes per APG). Escape closes the panel and returns focus to the trigger; the dropdown stays open across multiple selections (must NOT close per pick). Each ✕ chip is a real <button> with an accessible name (Remove filter: pricing). The result count updates an aria-live="polite" region.
D4d — per-option counts (should, not must). Showing pricing (34) in each option is a high-impact Baymard recommendation. Include it if cheap (counts computed from the corpus); acceptable to defer to a follow-up if it complicates the live-filter input. Tracked as a should AC, not a blocker.

Filter research (grounding for §4)

Authoritative-source synthesis (NN/g, Baymard, GOV.UK Design System, WAI-ARIA APG, Material/Carbon/Polaris):

  • Match control to cardinality. Low-card facet → show all checkboxes; high-card facet → dropdown panel with a type-to-filter search input. The inline 100+ chip cloud is the documented anti-pattern.
  • Applied-filters bar above results is the highest-leverage element — keep applied state visible; removable tokens labeled with the facet; always offer "Clear all" alongside per-chip removal.
  • OR within a facet, AND across facets is the conventional default; "select all that apply" implies OR-within.
  • A11y: disclosure button + native checkboxes (not role=listbox over checkboxes); combobox pattern for the type-to-filter input; Escape closes & restores focus; live region announces counts; never pre-select options.
  • Counts: "N of M" live (already present) + per-option counts are high-impact; prefer disabling dead options over silent zero-result pages.
  • Don't: close the dropdown on every pick; hide applied state; pre-select; dump the tag cloud; use vague facet labels ("Decision type"/"Tags" beat "Type").

Sources: nngroup.com/articles/applying-filters, nngroup.com/articles/filter-categories-values, baymard.com/learn/ecommerce-filter-ui, baymard.com/blog/have-filters-for-list-item-info, w3.org/WAI/ARIA/apg/patterns/combobox, w3.org/WAI/ARIA/apg/patterns/listbox, design-system.service.gov.uk/components/checkboxes.
