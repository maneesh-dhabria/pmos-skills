/mytasks goals & milestones — Design

Epic 260705-n2n · route: skill · the cross-story coherence contract (stories cite these anchors) · 2026-07-05

1. Overview & architecture

Add a durable goal entity (new file kind) to the existing /mytasks store, with embedded dated milestones. Tasks and projects attach to goals; two deterministic pace signals per goal (schedule adherence + attention allocation) plus derived progress are computed in lib.js pure functions and surfaced in a goals view, the at-a-glance index, and /morning-brief.

The design reuses the _shared/tracker-crudl.md substrate (id/slug, created/updated/schema_version, derived-on-read index, archive) — goals are a second tracker collection alongside items/, not a bolt-on. All new machinery lands in the existing skill at plugins/pmos-toolkit/skills/mytasks/.

3. Attachment & effective resolution (binds Story 2)

3.1 Fields & mapping

  • Tasks gain goal: (goal id | none sentinel | empty) and milestone: (milestone ref | empty). Empty = bare key (mytasks empty-optional binding). Task schema_version → 3 (additive; absent/1/2 stay valid via load-time normalization, mirroring migrateWorkstreamKey).
  • Project→goal mapping lives in the goal file (D8, grill-2): each goal carries attached_projects: [<project-slug>, …] in frontmatter — the durable source of truth. The resolver builds the in-memory projectGoals map by scanning goals' attached_projects; registry.json stays the optional, deletable visibility cache it is documented to be (never load-bearing for attachments). A project attached to two goals is a validation error (a project maps to at most one goal).

3.2 Effective-goal function (pure, unit-tested — INV-4)

effectiveGoal(task, projectGoals):
  if task.goal == 'none':          return null          # explicit detach wins (D5)
  if task.goal is a valid id:      return task.goal     # direct link wins
  if task.project in projectGoals: return projectGoals[task.project]   # inherit
  return null
effectiveMilestone(task): task.milestone if task.goal set directly; else null
                          (inherited goals attach at goal level, not milestone)

INV-5 A task counts toward a goal at most once — resolution is a single function, so direct+inherited can't double-count. INV-6 Attaching/detaching never mutates unrelated fields; attach to a non-existent goal/milestone errors and writes nothing.

4. Pace signals & progress (binds Story 3)

INV-7 (determinism, §H hard gate) every value below is computed by a lib.js pure function with a --selftest; the model never estimates a number or a band.

4.1 Schedule adherence — dated goals only (FR-10)

scheduleSignal(goal, today):
  if goal.type == 'open-ended': return null            # D2 — never 'behind'
  incomplete = milestones where not met and not derived-complete
  if none and all met:          return 'done'
  next = earliest-due incomplete milestone
  if next.due < today:          return 'behind'
  if next.due <= today + AT_RISK_DAYS and derivedProgress(next) < AT_RISK_PCT:
                                return 'at-risk'
  return 'on-track'

AT_RISK_DAYS / AT_RISK_PCT are named constants tuned in the Story 3 plan (default 7 days / 50%).

4.2 Attention allocation — all goals (FR-11, D1)

attentionSignal(goal, effectiveTasks, today):
  if effectiveTasks is empty:   return 'no-tasks-yet'   # grace state — never falsely 'starved'
  # D7 (grill-1): only REAL progress feeds a goal — a trivial `updated` bump does not.
  lastProgress = max over effectiveTasks of: completed date, created date
                 (a bare `updated` timestamp is NOT counted). NOTE: mytasks stamps
                 `created` + `completed` but not an in-progress transition; if a
                 status-change stamp is added later, include it here. Plan S3 confirms.
  window = cadenceWindow(goal.cadence)                  # 1/7/14/~30 days
  return 'starved' if (today - lastProgress) > window else 'fed'

4.3 Progress (FR-12, D3)

derivedProgress(scope) = completed(effectiveTasks in scope) / total(effectiveTasks in scope)
milestone.progress = 'met' (manual flag wins) OR derivedProgress(tasks whose milestone==ref)
goal.progress      = met-milestones + derived roll-up; 'no tasks yet' when total==0

INV-8 A manual milestone met flag overrides derived progress for that milestone (a milestone can be met without every task done). INV-9 A goal with zero effective tasks reads no-tasks-yet on both progress and attention — never 0% / starved / behind.

5. Surfacing (binds Story 3, D4)

  • /mytasks goals — derived-on-read view: one row/block per active goal with type, schedule band, attention band, progress, next milestone + due. Behind/starved sorted first.
  • Bare /mytasks index — a compact ## Goals summary section above the importance buckets, listing only goals that are behind/starved/at-risk (quiet when all healthy).
  • /morning-brief hook — the brief's pipeline reads the goals collection and emits a "goals needing attention" lane (behind + starved). Integration point, not a new build: mytasks exposes a goalsForBrief() lib function the brief consumes (mirrors how morning-brief reads other sources).
  • Web app (webapp/ + serve.js) — read-only goal display via a new GET /api/goals; full web CRUD is out of scope.

7. Decisions (D1–D6)

See 01_requirements.html#decisions — D1 per-goal cadence+recency attention; D2 dated vs open-ended type; D3 derived + manual milestone flag; D4 mytasks view + morning-brief; D5 inherit-by-default, per-task override; D6 new goal entity. D7 (grill-1): only real progress (completed/created, not a trivial updated bump) feeds a goal's attention signal. D8 (grill-2): the project→goal map lives in the goal file's attached_projects, not in the deletable registry.json. This design realizes them; the invariants above (INV-1…INV-9) are the coherence contract each story's /plan cites by anchor.
