Goals & milestones in /mytasks — Requirements

Epic 260705-n2n · route: skill · Tier 3 · 2026-07-05 · grounded in docs/pmos/shape/2026-07-05_goals-milestones-mytasks.html

Problem & goal

Maneesh's goals live outside /mytasks, so they go invisible and he steers by his reactive task list; and even when he recalls a goal he has no trustworthy signal of whether he is on schedule against its checkpoints or starving it of attention. Two co-equal jobs: A — capture & keep visible; B — sense the pace (dual signal).

Solution direction: add a durable goal entity to /mytasks with embedded dated milestones; let projects and tasks attach to goals (project attachment cascades to its tasks, per-task override wins); derive two trustworthy pace signals per goal — schedule adherence and attention allocation — and surface goals in the daily flow (a goals view + index summary + a /morning-brief hook).

Non-negotiable (ceiling-breaker): the signal must earn trust — open-ended goals never read "behind"; the attention signal is silence-based and per-goal-tunable so it doesn't cry wolf; attachment is low-friction (inheritance) so the data doesn't rot.

Design decisions (settled with the maintainer)

#Decision
D1Attention signal = per-goal cadence + recency. Each goal carries a target cadence (a check-in-style enum: daily/weekly/biweekly/monthly). Starved = no active or recently-completed attached task within that window. Silence-based; low upkeep; per-goal so a slow goal isn't nagged like a sprint.
D2Goal type: dated vs open-ended. dated goals have milestones/deadlines and get the schedule-adherence signal; open-ended (maintenance) goals have no deadline, get only the attention signal, and never read "behind". Kills the cry-wolf failure mode.
D3Progress = derived + manual milestone flag. Goal/milestone progress is derived from attached-task completion, AND a milestone can be manually marked met regardless of task state (for outcomes tasks only proxy). A milestone can be met without every attached task done.
D4Surfacing = mytasks view + morning-brief hook. A new /mytasks goals view + a goals summary atop the at-a-glance index; plus a /morning-brief integration that flags behind/starved goals.
D5Cascade = inherit-by-default, per-task override wins. Tasks in a goal-attached project inherit the goal; a task may set its own goal:/milestone: (or explicit none) to override. Inheritance is a default, not a cage.
D6Storage = new goal entity. Goals are a new file kind at ~/.pmos/tasks/goals/{id}-{slug}.md with milestones embedded in the goal file; tasks/projects link via new goal: (+ milestone:) fields. Reuses the _shared/tracker-crudl.md id/slug/index substrate; keeps goals distinct from project containers.

Functional requirements

Goal & milestone model (Story 1)

  • FR-1 A goal is a markdown file at ~/.pmos/tasks/goals/{id}-{slug}.md with frontmatter: schema_version, id (<YYMMDD>-<rand3> per tracker-crudl §2), title, type ∈ {dated, open-ended}, status ∈ {active, achieved, dropped}, cadence (check-in enum; drives the attention signal), optional target (ISO date, dated goals), created/updated. Milestones embedded as a body section.
  • FR-2 Milestones are an ordered list inside the goal file (a ## Milestones section or a frontmatter list): each has a stable local ref, a description, a due (ISO date), and a met flag (bool + optional met-date). A goal may have zero or more milestones; open-ended goals typically have none.
  • FR-3 Goal CRUD verbs: /mytasks goal add (rich-capture via _shared/interactive-prompts.md), /mytasks goal show <id>, /mytasks goals (list/index view), /mytasks goal edit/drop/achieve <id>, /mytasks milestone add <goal>, /mytasks milestone met <goal> <ref>. Natural-language forms per the skill's NL-first convention.
  • FR-4 Validation: closed enums for type/status/cadence; the skill never invents enum values. Milestone due dates parse via the existing date parser. Reuse the id/slug minting + archive convention from tracker-crudl.md.

Attachment & cascade (Story 2)

  • FR-5 Tasks gain optional goal: (goal id) and milestone: (milestone ref within that goal) frontmatter fields; empty by default (bare keys, per mytasks's empty-optional binding).
  • FR-6 A project may be attached to a goal via a project→goal mapping (in registry.json, the existing add-only visibility cache, or an equivalent durable map). Task markdown stays the source of truth for direct links.
  • FR-7 Effective goal resolution (D5): a task's effective goal = its own goal: if set (including the sentinel none → detached); else the goal mapped to its project:; else none. Deterministic, pure-function, unit-tested. The same rule yields the effective milestone.
  • FR-8 Verbs: /mytasks attach <task|project> <goal> [milestone], /mytasks detach <task|project>. Attaching a project sets the mapping; attaching a task sets its goal:. NL forms supported.
  • FR-9 Dedup / integrity: a task counts toward a goal once; resolution never double-counts a task that both directly attaches and inherits. Attaching to a non-existent goal/milestone errors clearly.

Pace signals, progress & surfacing (Story 3)

  • FR-10 Schedule-adherence signal (dated goals only): per goal, compute an at-a-glance read from milestone due dates vs. today and their met/derived state → {on-track, at-risk, behind, done}. Open-ended goals never produce this signal (D2). Deterministic; script-computed, never model-estimated (§H).
  • FR-11 Attention-allocation signal (all goals): fed vs starved from the most-recent attached-task activity (created/updated/completed) vs. the goal's cadence window (D1). "No attached activity within cadence → starved." Deterministic.
  • FR-12 Progress (D3): derived progress = completed ÷ total effective-attached tasks (and milestone-level roll-up); PLUS manual milestone met overrides. A goal with no attached tasks reads "no tasks yet", not 0%/behind.
  • FR-13 Surfacing (D4): (a) /mytasks goals renders each goal with type, both signals, progress, next milestone; (b) the bare /mytasks at-a-glance index gains a compact goals summary (behind/starved flagged first); (c) a /morning-brief hook surfaces behind/starved goals in the daily brief. Web app (webapp/) shows goals read-only (full web CRUD is out of scope — see below).
  • FR-14 All pace/progress computation lives in lib.js pure functions with a --selftest/unit-test surface; the SKILL body cites them and never re-derives arithmetic (§H hard gate).

Non-functional & constraints

  • Backwards compatible: existing tasks (no goal:/milestone:) stay valid; schema_version bumps additively with a load-time normalization like the prior workstream→project migration. No task file is rewritten on read except the additive normalization.
  • Trust-first (ceiling-breaker): open-ended goals never "behind"; attention signal silence-based + per-goal-tunable; inheritance keeps attachment low-friction; a goal with zero attached tasks is never falsely "starved" until it has had at least one attachment (grace state).
  • Determinism (§H): every signal/progress value is script-computed; the model authors prose, not numbers.
  • Single plugin, one release unit (pmos-toolkit). Conforms to skill-patterns.md §A–§L; passes skill-eval ([D]+[J]); 4 hygiene lints + audit-recommended green.

Out of scope (this epic)

  • Full web-app CRUD for goals/milestones (create/edit in the browser) — read-only web surfacing only; web authoring is a follow-up.
  • Goal hierarchies (sub-goals / OKR key-result trees) — a goal is flat with milestones.
  • Cross-goal analytics / historical burndown charts — the signals are point-in-time reads.
  • Notifications/reminders beyond the /morning-brief hook and the existing check-in surface.

Open questions (carried to design/plan)

  • Milestone storage: frontmatter list vs. a ## Milestones body table — resolved in 02_design (frontmatter list for machine-read, mirrored in body for humans).
  • Project→goal map home: extend registry.json vs. a new goals-side map — resolved in 02_design.
  • Exact "at-risk" threshold for the schedule signal (e.g. next milestone due within X days & <Y% derived progress) — tuned in Story 3 plan.
