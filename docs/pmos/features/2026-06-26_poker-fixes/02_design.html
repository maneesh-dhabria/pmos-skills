Design — /poker fixes

Epic 260626-fdh · route: skill · plugin: pmos-gamekit · 2026-06-26 · This is the epic design_doc — the coherence contract; the single story's /plan cites its anchors.

Context

/poker is a launch-only pmos-gamekit skill: it serves one self-contained, offline No-Limit Texas Hold'em file — plugins/pmos-gamekit/skills/poker/game/poker.html (~1575 lines) — through the shared zero-dependency game launcher. All gameplay lives in that one file. The engine (window.PokerEngine, the pure-logic <script> that assigns PokerEngine) is a browser-agnostic module exposing createHand, the 7-card evaluator (rank7/categoryOf), buildSidePots, decide/preflopStrength, personaFor, randomTable, and the static preflop reference data; the UI/render layer (guarded by document) renders the felt, seats, and action bar. Tests live in tests/run.mjs (engine selftest, node run.mjs --selftest).

The maintainer reported two first-run rough edges, both confined to that single game file. They are tiny and unrelated in mechanism (one is layout CSS, one is a one-line RNG floor), but both ship as one /poker release, so this is a one-story skill epic (the D24 litmus: tightly coupled in one file, cannot be independently skill-eval'd or shipped). The one genuinely open question — what "at least 2 players" means in poker terms, where the hero is always seated — was resolved with the maintainer up front (see D3).

Functional requirements

FR1 — Hero seat clearance (no action-bar overlap)

  • FR1.1 — On the table view, the hero ("You") seat block — holecards, nameplate, betchip, and lastact line — never visually overlaps .actionbar / .actionbar .msg ("Your move — …").
  • FR1.2 — Clearance holds across the viewport range: verified at desktop (1280px), a tablet width, and a narrow (~640px) width. The fix accounts for the overhang growing as the felt (a width-driven aspect-ratio box) shrinks.
  • FR1.3 — Apply both levers (D2): fixed clearance below the felt (margin-bottom on .table-wrap / the felt, or an increased .table-col gap) and a smaller hero slot t% (SLOTS[0].t 92 → ~88). Opponent seats and the rest of the table layout are unchanged (only the hero shows full-height face-up cards at the bottom edge).
  • FR1.4 — No new external assets; the felt's shape/aspect-ratio and the seat layout for all other positions stay visually intact.

FR2 — Random mode minimum opponents

  • FR2.1 — Selecting "Random" opponents deals at least 2 opponents (a 3-handed-or-larger table); a single-opponent heads-up game is never produced by Random. (D3)
  • FR2.2 — The random count stays bounded by the existing table maximum (5 opponents / 6 seats — there are 6 SLOTS); i.e. random ∈ {2,3,4,5} opponents.
  • FR2.3 — Explicit opponent-count picks (the numbered buttons) are unaffected — only the auto-random path gets the floor.
  • FR2.4 — The "New game — N opponent(s) (…)" log line and the singular/plural wording remain correct for the new range.

Invariants (coherence contract)

  • Inv-1 (engine purity): window.PokerEngine stays browser-agnostic and side-effect-free; no DOM enters it. If F2 adds a pure count helper it is exported and covered in tests/run.mjs; otherwise the engine is untouched. node tests/run.mjs --selftest stays green.
  • Inv-2 (cosmetic/UI isolation): the F1 CSS + SLOTS change and the F2 default are presentation/UI-config only — they never alter the deal, hand evaluation, betting, side-pots, or showdown.
  • Inv-3 (offline + self-contained): no new external assets, fonts, or network. Everything inlines into the one HTML file.
  • Inv-4 (no layout regression): the F1 clearance does not push the felt off-screen, shrink it below usability, or displace opponent seats / the references panel; the table reads correctly at all three test widths.
  • Inv-5 (test/launch harness intact): the engine selftest and the gamekit launch contract keep working; the existing card-deal animation and seat render are not broken by the SLOTS edit.

Story split

Its ACs are the two FR groups above plus the invariants; skill-patterns.md §A–§L + the host CLAUDE.md (canonical skill path, gamekit launch contract) are standing acceptance criteria. Open questions / deferred

  • F1 magnitudes (tunable, not contract): "margin-bottom ~30px" and "t: 92 → ~88" are the proposed starting values; the implementer may settle on nearby values during the 3-width self-check so long as FR1.1/FR1.2 hold with comfortable clearance and no felt clipping.
  • F2 max unchanged: only the floor moves (1→2 opponents); the upper bound stays 5 opponents. Raising the cap was not requested and is out of scope.
  • F2 helper vs one-liner (D5): implementer's choice; default to the minimal one-line change unless a pure helper is obviously cleaner and earns a selftest.
