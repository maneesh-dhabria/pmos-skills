Spec — /wireframes canvas-view enhancement

Created using pmos‑toolkit

Mode: skill-feedback · Tier: 2 · Source: 01_requirements.html

Overview

Add Phase 7.5 to /wireframes that aggregates the per-device wireframe HTML files into a single canvas.html viewer plus a canvas.json layout sidecar. New substrate at plugins/pmos-toolkit/skills/wireframes/assets/canvas/ holds the canvas template and the Node builder script. The builder is invoked from Phase 7.5 via node against the feature folder; it reads the per-device files + DESIGN.md, writes canvas.html (single-file, CDN deps) and canvas.json (auto-laid-out positions + DESIGN.md-derived arrows). canvas.html embeds the sidecar inline so it works in file:// without a fetch.

canvas.json schema (v1)

{
  "version": 1,
  "generated_at": "<ISO-8601>",
  "generator": "pmos-toolkit/wireframes canvas-aggregator v<plugin-version>",
  "viewport": { "default_zoom": 0.3, "default_x": 0, "default_y": 0 },
  "screens": [
    {
      "id": "<screen-id>-<device>",
      "screen_id": "<screen-id>",
      "device": "desktop-web",
      "source_file": "desktop-web.html",
      "anchor": "#<screen-id>",
      "title": "<screen title from h2/section heading>",
      "journey": "<journey-id or null>",
      "x": 0,
      "y": 0,
      "w": 1280,
      "h": 800
    }
  ],
  "arrows": [
    { "from": "<screen-id>-desktop-web", "to": "<next-screen-id>-desktop-web", "label": "submit", "journey": "<journey-id>" }
  ]
}

ID convention: {screen_id}-{device} — uniqueness across devices. screen_id = the data-screen attribute or the heading slug.

Functional requirements

  • FR-1 (always-on): Phase 7.5 runs unconditionally after Phase 7 (except in --bootstrap-design-only mode where Phase 7 itself is skipped — see D6). No flag, no gate.
  • FR-2 (screen extraction): for each per-device HTML file in the feature folder's wireframes/ dir, extract every top-level <section> with a data-screen attribute OR (fallback) every <section> with an id. The screen ID is data-screen if set, else the section id. The screen title is the first <h2> text inside the section.
  • FR-3 (auto-layout): on first run (no existing canvas.json at the output path), generate a default layout: each journey's screens lay out left-to-right (column = position-in-journey × (screen_width + 200px gutter)); journeys stack top-to-bottom (row = journey-index × (screen_height + 400px gutter)). Devices for the same logical screen stack vertically within the same column (mobile under desktop). Screens not assigned to any journey lay out in an "orphans" row at the bottom.
  • FR-4 (preserve existing layout): if canvas.json already exists at the output path, READ it, merge: keep all existing positions; add any new screens (post-regen) at the bottom in an "added" row; remove screens no longer present in the per-device files. Never silently drop user-curated positions.
  • FR-5 (DESIGN.md arrows): parse DESIGN.md's user-journey sections — for each journey, identify the ordered list of screen references; emit one arrow per consecutive pair within a journey. Journey reference: any link or list-item whose text or href contains a known screen_id. If DESIGN.md is missing, no arrows (warn in chat, do not fail).
  • FR-6 (single-file output): canvas.html is a single self-contained file. Inline the canvas.json as a <script type="application/json" id="canvas-data"> block so the viewer works under file:// without fetch. The companion canvas.json remains on disk as the canonical source for diff / merge.
  • FR-7 (CDN deps with SRI): panzoom@9.4.3 and leader-line@1.0.7 loaded via CDN with subresource-integrity hashes. Both small (<15KB combined gzipped).
  • FR-8 (drag + save): screens drag-positionable on the canvas (mousedown on screen wrapper → drag → mouseup commits position to in-memory state). A Save layout button serializes the current state to JSON and triggers a browser download named canvas.json. No fetch / no dev-server write.
  • FR-9 (pan/zoom): standard panzoom: scroll-wheel to zoom (0.1×–3×), drag empty canvas to pan, +/- buttons for keyboard accessibility. Default zoom 0.3× so the whole layout fits a 1440-px viewport.
  • FR-10 (iframe rendering): each screen rendered as <iframe src="<source_file>#<anchor>" sandbox="allow-same-origin" loading="lazy">. CSS transform: scale(1) at iframe level — the parent wrapper scales via panzoom's transform.
  • FR-11 (skill-eval pass): the modified SKILL.md must pass tools/skill-eval-check.sh --target claude-code with zero new [D] or [J] failures.
  • FR-12 (index.html link): Phase 7's index.html gains a top-row link to canvas.html labelled "Canvas view (all devices)".

Non-functional

  • NFR-1 (perf): Phase 7.5 adds <2s wall-clock for typical runs (≤20 screens × 3 devices). Single Node invocation; no subagents.
  • NFR-2 (no regressions): Phase 7 output (per-device files + index.html) unchanged in content; only the new top-row link added to index.html.
  • NFR-3 (offline-capable): after first load (CDN deps cached) canvas.html works offline. SRI hashes mean stale CDN is detected, not silently mis-rendered.
  • NFR-4 (idempotent): re-running /wireframes preserves user-curated canvas.json positions (FR-4).

Verification plan

  • V1 — Skill-eval: tools/skill-eval-check.sh --target claude-code plugins/pmos-toolkit/skills/wireframes/ exits 0.
  • V2 — End-to-end: Run /wireframes against an existing feature with desktop-web + mobile-web devices; verify canvas.html + canvas.json emitted; open canvas.html in browser; confirm all screens visible, draggable, arrows present from DESIGN.md.
  • V3 — Idempotency: Drag a screen; click Save; replace existing canvas.json with downloaded; re-run /wireframes; confirm the dragged position is preserved.
  • V4 — Bootstrap mode: Run /wireframes --bootstrap-design-only; confirm canvas.html / canvas.json are NOT emitted (Phase 7.5 doesn't run when Phase 7 doesn't run).
  • V5 — Missing DESIGN.md: Run on a feature with no DESIGN.md; confirm canvas.html still emits with screens but zero arrows; chat shows a warning.
