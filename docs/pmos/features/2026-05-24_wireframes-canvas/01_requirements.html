Requirements — /wireframes canvas-view enhancement

Created using pmos‑toolkit

Mode: skill-feedback (enhancement to existing /wireframes) · Tier: 2 · Target version: next minor of pmos-toolkit · Run Outcome: clean

Problem

Today /wireframes emits one stacked HTML file per device — usable for sequential review, but it hides cross-screen relationships. Stakeholders walking a design crit want to see the whole journey laid out spatially (Figma-like), with arrows showing flow, so they can compare states side-by-side and spot inconsistencies. They currently have to scroll back-and-forth through one file per device, mentally reconstructing the flow each time.

The acceptance criteria for any solution is the standing skill-authoring rubric at plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§F.

Solution direction

Add a canvas-view aggregator to /wireframes as a new Phase 7.5 that runs after Phase 7 (Index & Serve) and before Phase 8 (MSF). It produces a single self-contained canvas.html alongside the existing per-device files, plus a canvas.json sidecar holding screen positions and arrow definitions. Default behaviour: always emit (no flag, no gate). Single-file HTML, CDN-only deps, zero build — matches the repo's single-file-HTML invariant.

User journeys

J1 — Design crit walk-through

Designer opens canvas.html, sees every screen of every device laid out on an infinite pan/zoom surface with flow arrows derived from DESIGN.md journeys. Pans across the canvas to compare onboarding screens side-by-side. Zooms in to inspect one screen's details. Closes the tab — no state saved.

J2 — Persisted layout

Designer drags screens into a custom layout (groups onboarding top-left, dashboards centre, settings bottom-right). Clicks Save layout → browser downloads canvas.json with new positions. They drop it next to the existing canvas.json in the feature folder, commit. Next time anyone opens canvas.html they see the curated layout.

J3 — Auto-layout from DESIGN.md

On first run (no canvas.json exists), the aggregator derives a default layout from DESIGN.md journeys: screens in the same journey lay out left-to-right, journeys stack top-to-bottom. Arrows connect consecutive screens within a journey. User opens and the layout is already meaningful.

Decisions (captured from interactive Phase 2)

  • D1 — Trigger: always emit canvas.html alongside per-device files (no flag, no gate). Zero cognitive load; if users don't want it they just don't open it. Why: opt-in flags rot; gates add a prompt to every run.
  • D2 — Persistence: canvas.json sidecar in the feature folder. Save = browser-download (no dev-server write). User commits the file next to wireframes. Why: versionable, team-shareable, no daemon required.
  • D3 — Arrows: derived from DESIGN.md journey sections (parse the existing structure). No manual arrow-drawing in v1. Why: reuses authoring already done; smallest scope.
  • D4 — Rendering: each screen rendered via sandboxed <iframe> pointing at the per-device wireframe file with a hash anchor (e.g. desktop-web.html#login). No content duplication; iframes scale via CSS transform. Why: single source of truth — the canvas is a viewer, not a regenerator.
  • D5 — Deps: panzoom (~5KB) for pan/zoom, leader-line for arrows. Both via CDN with SRI hashes. No build step. Why: matches single-file-HTML invariant; no toolchain.
  • D6 — Bootstrap-only mode: in --bootstrap-design-only Phase 7.5 does NOT run (no wireframe HTML to aggregate). Why: bootstrap mode produces only DESIGN.md + COMPONENTS.md.

Out of scope (v1)

  • Real-time multi-user collaboration (no Figma-parity for live cursors).
  • Manual arrow drawing in the browser (D3 — DESIGN.md is the only arrow source).
  • Comments / annotations on the canvas surface (use the existing per-screen annotations layer).
  • Auto-save back to canvas.json (browser can't write files; explicit Save button → download).
  • Replacing the per-device files (canvas is additive — per-device files remain primary).

Constraints

  • Single-file HTML output (no build step, no module bundler, no Node dependency at view-time).
  • CDN-only deps with SRI hashes; works offline once cached.
  • Must run in file:// context (no localhost requirement for viewing — though dev-server still optional).
  • Iframe sandbox: sandbox="allow-same-origin" (needed for scroll/hash navigation; allow-scripts NOT set since wireframes are static).
  • canvas.json schema versioned (version: 1) for forward-compat.
