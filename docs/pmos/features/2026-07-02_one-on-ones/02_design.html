Design — /one-on-ones

pmos-managerkit · epic 260702-z77 · route: skill · the cross-cutting coherence contract for story 260702-6ks

1. Overview

/one-on-ones helps a manager run effective 1:1s — actively coaching, not a passive notepad. The unit is one rolling record per report. Verbs operate on that record: add creates it, note quick-captures into it, plan reads it into a coached prep agenda, log appends a session and closes loops, career runs the periodic deep-dive, and context verbs keep goals + feedback current. A bundled corpus makes the coaching grounded and attributable.

2. Invariants

INV-1 — identity in /people, record owned here. A report's identity (name, handle, role) is anchored in ~/.pmos/people. The 1:1 record is a separate file this skill owns. add reconciles the two; it never invents a person.

INV-2 — independent of /interview-feedback. No import, no _shared cite, no reading its scorecards or corpus. Performance/coaching feedback is manager-entered into the record. (Grep-clean AC.)

INV-3 — no /mytasks. The skill never reads or writes a report's task list. "Work context" comes from what the manager captures, not a task tracker they may not own.

INV-4 — privacy. Records live under ~/.pmos/one-on-ones/, outside the repo. The skill must never write a record inside the working tree; sensitive employee content is never committed.

3. Data model — the per-report record

One Markdown file per report at ~/.pmos/one-on-ones/<handle>.md, where <handle> is the /people handle. Three body zones under YAML frontmatter — the hybrid the shape brief settled on (persistent header + running-agenda inbox + reverse-chron session log):

---
schema_version: 1
handle: sarah-chen           # /people handle (identity anchor, INV-1)
name: Sarah Chen
role: Senior PM
cadence: weekly              # weekly | biweekly | ...
started: 2026-07-02
career_last_reviewed:        # date of last /one-on-ones career session
---

## Goals & growth focus
- Grow into staff-level scope by H2; wants more cross-org influence.

## Standing themes
- Recurring: feels under-recognized in leadership reviews.

## Operating manual
- Feedback: direct, in the moment. Recognition: private, written.
- What makes them grumpy: last-minute scope changes.

## Performance feedback        
- 2026-Q2: strong delivery; stretch = influencing without authority.

## Coaching feedback           
- Working on: running crisper design reviews.

## Open action items
- [ ] (me) intro Sarah to Platform lead — since 2026-06-20

## Inbox                       
- [growth] discuss the staff-eng ladder expectations
- [blocker] blocked on legal review for launch

## Sessions                    
### 2026-06-25
- Check-in: energized after the offsite.
- Topics: roadmap Q3; recognition.
- Decisions: Sarah leads the retro.
- Action items: (Sarah) draft retro agenda [open]; (me) raise recognition w/ VP [done]
- Questions surfaced: "what does staff scope look like here?"

A parser/serializer round-trips this structure byte-stably (frontmatter + the fixed section set). note appends to ## Inbox; log prepends a dated block under ## Sessions and moves discussed inbox lines out; context verbs edit the header sections. First write creates ~/.pmos/one-on-ones/.

5. plan — the coached prep agenda

Assembled deterministically from the record, then annotated with coaching from the corpus (§6). Sections, in order:

  1. Human first — a check-in opener ("How are you as a person?" / "What's top of mind?"), always first.
  2. Last time's open loops — open action items + questions from the newest session entry, surfaced for follow-through.
  3. Their agenda — inbox items (report's + manager's captures), grouped by intent tag.
  4. Growth & feedback in view — goals, standing themes, and the latest performance + coaching feedback, so the manager plans against them (structurally resists status-creep).
  5. Coached suggestions — intent-tagged question picks from the corpus + nudges: human-first, listen-heavy (Horowitz 10/90), solicit upward feedback, and flags:
    • status-creep flag — if the last N session entries are topic-only with no growth/human content.
    • stale-action flag — an open action item older than a threshold.
    • career-due flag — career_last_reviewed older than ~a quarter → suggest a career session.

Output: a commentable HTML prep artifact via the html-authoring substrate (so the manager can annotate it), plus a terminal summary. The flags are computed by script (deterministic), never by the model guessing — §H.

6. Coaching corpus (bundled reference)

Ships in the skill's reference/ (offline, no network). Feeds plan's suggestions and career. Contents:

  • 12 durable principles — report's meeting; not a status report; listen > talk; near-never cancel; high-leverage; go below the surface; solicit feedback on yourself then be silent; capture + close loops; coaching/growth are the point; learn how each person wants to be led early; match style to person/moment; protect the human layer.
  • Named models — Manager Tools 10/10/10; Rands' Update/Vent/Disaster; Laraway Career Conversations; GitLab agenda template; Hogan first-1:1 operating-manual questions.
  • Intent-tagged question bank — opener / growth-career / blockers / morale / feedback-up / relationship / workload.

Attribution caveats (must be honored in the corpus)

The 10/90 talk ratio is Horowitz, not Grove (Grove's version: "ask another question").
Laraway's plan is "long-term vision + short-term plan" (Sandberg's 18-month), not a "15-month plan".
Hogan's "Bias Toward [Action/Feelings]" two-column exercise is unverified (likely Resilient Management) — label as such, do not assert verbatim.
Manager Tools 10/10/10 and the Rands "you're doing it wrong" line are documented-canon / paraphrase — attribute accordingly.

7. Non-interactive contract

Inline the canonical _shared/non-interactive.md block byte-identically (pmos-managerkit has no _shared/ yet — the block is inlined per skill anyway). Per-verb posture: note and log AUTO-PICK Recommended (unattended-safe capture); add with a missing /people identity DEFERs (never fabricate a person); log/career with no supplied body DEFERs (never fabricate feedback or session content). Every AskUserQuestion carries a (Recommended) option or an adjacent <!-- defer-only: … --> tag.

8. §H / §I / §L notes

  • §H gates: the status-creep / stale-action / career-due flags and the "who's due" overview are deterministic → computed by script, never model arithmetic. Coaching suggestions are advisory (judgment).
  • §I flags: NL-first — every verb/flag has a natural-language form; --tag, --non-interactive are the only contract flags (typed value / headless determinism). Others stay parsed as silent aliases.
  • §L dispatch: no subagent fan-out needed; the skill is a sequential CRUD-plus-coaching flow.
