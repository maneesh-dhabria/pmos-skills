Design — /interview-feedback robustness

Epic 260629-bm9 · Plugin pmos-managerkit · Route skill · Status defining · Date 2026-06-29

The cross-cutting design contract for a single-story revision epic. It applies five findings from the first production run of /interview-feedback (scoring one candidate from a Google Drive recording, a Google Docs transcript, a local case-docs folder, and a post-interview PDF submission). The skill's core contract — two grounded artifacts with evidence-tier citations verified verbatim — was eventually delivered, but only after two external-URL workarounds, a manual transcript-rescue operation, and two rounds of user-prompted correction to how the written submission was assessed. This epic closes those gaps so the skill is robust on the inputs a real interview round actually carries. One architectural enrichment resolved with the maintainer during this define run is marked resolved.

1. Problem & charter fit

The claimed contract is "turn one round's inputs — recording or transcript, notes, scorecard, brief — into a filled scorecard and per-interviewer effectiveness notes, every subjective claim evidence-tier-tagged and transcript citations verified verbatim." The run held the grounding contract but fell short on three obligations the contract implies for real inputs:

  1. Input robustness. Two of the four inputs were Google URLs that 401'd. The skill emitted no detection and no fallback — it silently fell through to a phase that later failed, and the user had to notice and proactively supply a local copy. A skill that accepts an input must tell the user when it cannot read that input, before proceeding.
  2. Data safety. transcribe.sh overwrote a speaker-attributed 1020-line Gemini transcript with a 767-line timestamped-only whisper file — silent destruction of the better grounding source, requiring a manual mv + restore. A tool must never clobber a better artifact that already exists.
  3. Analytical completeness on the written submission. The post-interview PDF was not factored into the recommendation at all (a No-Hire produced without referencing it), and when added it was scored in isolation (3.60/4.00) rather than in the context of the interview that preceded it — forcing two full rewrites. A faithful assessment reads a written submission against what the interview seeded, and the right lens depends on whether the submission was produced after a live session or prepared before one.

All three serve the same charter ("help me produce a grounded, defensible scorecard") — this is a revision of the existing pmos-managerkit skill. No new skill, no new charter.

3. Decisions

DDecisionRationale
D1URL inputs are probed at the top of Phase [Resolve]; auth-walled or unreachable ones stop-and-ask, never fall through. Known auth-walled hosts (drive.google.com, docs.google.com) are recognized up front; any other URL is probed with a lightweight request. On 401/403/404/timeout (or a known auth-walled host), emit the exact line and stop that input.The skill is local-first; a URL it cannot read is a dead end. Detecting it at Resolve — before transcription/scoring — turns a late silent failure into an early, actionable message (F1).
D2transcribe.sh never clobbers an existing transcript by default. If transcript.refined.txt already exists, whisper output is written to transcript.whisper.txt and a signal names both files; Phase [Ground] selects the better tier (speaker-attributed > timestamped-only). A new --force-transcribe flag (and its NL form) restores overwrite.Silent destruction of the better grounding source is the worst failure mode for a grounding-first skill. Preserve-by-default + let Ground choose is safe and loses nothing; --force-transcribe is the explicit escape (F2).
D3 resolvedThe written submission is assessed by scenario, never in isolation. Two scenarios, classified once (inferred default + a --submission-type flag): (A) post-live — submitted after a live session; (B) pre-live-then-present — prepared first, then presented/defended live. Each drives a different assessment lens (§FR-4). The assessment block is mandatory in the scorecard whenever a submission is present and feeds the overall reco.A written submission's signal is only legible against the interview that preceded it. For (A), the question is whether the candidate used the live context to structure and complete the missing parts — a submission that merely re-presents what was discussed, however polished, is a weak signal. For (B), the question is the intrinsic quality of the first submission plus how the candidate defends it, reacts to probes, and adjusts live. Scoring the document alone answers neither (F3, F4; maintainer enrichment).
D4The citation gate becomes visible. On pass, check-citations.mjs prints one stdout line ✓ citations: N transcript, M notes and the skill appends a one-line HTML comment at the top of each output file. The gate stays a hard fail on a non-verbatim transcript-tier citation.A hard gate the user never sees is indistinguishable from no gate. A single stdout line + an in-artifact comment makes the verification auditable without cluttering the visible artifact (F5).
D5Written-submission claims are grounded like every other claim — a new submission evidence tier, verified verbatim (≥40 chars) by check-citations.mjs against the submission file (a third positional). The three-bucket / quality-plus-defense analysis cites the transcript for "what the interview seeded" and the submission for "what the candidate wrote".The fix must not open a grounding hole: the new analysis introduces a new source, so the gate must cover it. Reusing the existing verbatim mechanism keeps grounding enforced, not asserted (F3/F4 + INV-1).

4. Functional requirements

FR-1 — URL-input detection & fallback (F1, D1)

  • FR-1.1 At the start of Phase [Resolve], every input token that is an http(s) URL is classified before any other input handling. A known auth-walled host (drive.google.com, docs.google.com) is treated as inaccessible without probing; any other URL is probed with a lightweight request (HEAD/GET).
  • FR-1.2 On 401/403/404/timeout or a known auth-walled host, emit exactly: Can't access <url> — please provide a local file path or paste the content. and stop processing that input. Do not fall through to Transcribe/Ground/Score with a missing input. A reachable, fetchable URL is downloaded into inputs/ verbatim and proceeds normally.
  • FR-1.3 Under --non-interactive an inaccessible URL DEFERs (free-form: a local path cannot be guessed) — log the open question and continue with the inputs that did resolve; never fabricate or silently drop.

FR-2 — non-clobbering transcription (F2, D2)

  • FR-2.1 Before writing transcript.refined.txt, transcribe.sh checks whether that file already exists. If it does, whisper output is written to transcript.whisper.txt instead (the existing file is never overwritten), and the script emits a signal on stdout naming both files and which carries speaker attribution (a Name: line-prefix pattern).
  • FR-2.2 --force-transcribe (NL: "re-transcribe / overwrite the transcript") restores the overwrite-transcript.refined.txt behavior. Default is preserve.
  • FR-2.3 Phase [Ground] selects the grounding source: when both a speaker-attributed transcript and a timestamped-only one exist, prefer the speaker-attributed file and say so. The selection is surfaced to the user, not silent.

FR-3 — written submission factored into the reco (F3, D3)

  • FR-3.1 Phase [Score] detects a post-interview / prepared written submission among the inputs (by filename pattern — e.g. submission, writeup, case, assignment, or a non-transcript/non-scorecard document — or the explicit --written-submission <path> flag).
  • FR-3.2 When a submission is present, the filled scorecard MUST carry a written-submission assessment block and the overall recommendation MUST reference it. A checklist gate closes Phase [Score] only when all are true: "Written submission present in inputs? → classified by scenario? → assessed in context? → referenced in reco? — Y/N." A No-Hire/Hire produced without the submission block, when a submission was supplied, fails the gate.

FR-4 — scenario-aware contextual assessment (F4, D3)

  • FR-4.1 The submission is classified into one scenario, once: (A) post-live — produced after a live session — or (B) pre-live-then-present — prepared before a live session and then presented/defended. Default inferred from timing/filename/the round brief; overridable with --submission-type post-live|pre-live. Under --non-interactive the inferred default is AUTO-PICKed (carries a Recommended option); ambiguity is logged as an open question.
  • FR-4.2 — Scenario A (post-live). Cross-reference each scored dimension against the transcript and bucket the submission's content as: (a) discussed in the interview, (b) directed by the interviewer's closing brief, (c) independently reached by the candidate. The assessment judges whether the candidate used the live-discussion context to appropriately structure and complete the missing parts. A submission that mainly restates (a)/(b) — however well presented — is explicitly flagged as a weak signal, not a strong one.
  • FR-4.3 — Scenario B (pre-live-then-present). Assess (i) the intrinsic quality of the first submission and the clarity of thought it shows, and (ii) from the transcript, how well the candidate defends that thinking, reacts to the interviewer's probes, and adjusts on the fly during the live session. Both halves feed the dimension scores.
  • FR-4.4 The submission is never scored as a standalone number divorced from interview context. The assessment block surfaces the scenario, the buckets (A) or the quality+defense split (B), and how each shaped the contributing dimension scores and the overall reco.

FR-5 — visible citation gate + submission grounding (F5, D4, D5)

  • FR-5.1 On pass, check-citations.mjs prints exactly one stdout line: ✓ citations: N transcript, M notes (counts of verified citations by tier; the submission tier is included in the line when present, e.g. K submission). On fail it keeps its current loud non-zero exit + offending-citation report.
  • FR-5.2 The skill appends a one-line HTML comment at the top of each output file on pass: <!-- citations verified: N transcript-tier, M notes-tier[, K submission-tier], <YYYY-MM-DD> -->. Visible in source, invisible in render — no clutter.
  • FR-5.3 check-citations.mjs gains a submission known tier and an optional third positional (the submission file). A data-cite-tier="submission" citation is verified as a verbatim ≥40-char substring of the submission file (same normalization as the transcript tier); a submission-tier citation with no submission file provided, or not found verbatim, fails the gate. Selftest coverage added.

5. Invariants

  • INV-1 — grounding stays enforced, not asserted. Every subjective claim still carries an evidence-tier citation; check-citations.mjs still hard-fails a non-verbatim transcript-tier citation. The new submission tier is verified verbatim too — the written-submission analysis does not open a grounding hole (§FR-5.3 / D5).
  • INV-2 — non-interactive contract preserved. The new asks carry a Recommended option (submission-scenario classification) or are explicitly defer-only free-form (the URL-fallback "provide a local path"); the frozen non-interactive-block stays byte-identical; the existing scoped tier-3 questionnaire refusal marker is untouched. No new entry is added to a frontmatter allowed-tools list (the skill declares none; its asks already degrade per Platform Adaptation).
  • INV-3 — scripts keep their --selftest green. check-citations.mjs and fill-scorecard.mjs selftests pass after the edits with new assertions added (submission tier, written-submission block); transcribe.sh's preserve-guard is covered by a case in tests/run-tests.sh. Deterministic work stays in scripts (§H).
  • INV-4 — no regression to the happy path. A run with a local transcript, no written submission, and no URL inputs behaves byte-identically: the URL probe fires only on URL tokens, the preserve-guard only when a transcript pre-exists, and the submission block only when a submission is present.
  • INV-5 — confidentiality unchanged. Candidate data stays local; the gitignore guard is intact; the written submission and any downloaded URL content are copied into inputs/ verbatim and never committed, never sent to an external service (the URL probe is a read of a user-supplied URL only).

6. Conformance

The revised skill must conform to skill-patterns.md §A–§L and pass skill-eval.md (both the [D] deterministic and [J] judge halves). Repo policy from CLAUDE.md applies: canonical skill path (plugins/pmos-managerkit/skills/interview-feedback/SKILL.md), the non-interactive inline-block byte match (tools/lint-non-interactive-inline.sh), the recommended-option audit (tools/audit-recommended.sh), and the flag/phase-ref lints (tools/lint-flags-vs-hints.sh, tools/lint-phase-refs.sh — note the new contract flags --force-transcribe and --submission-type + --written-submission must be both in argument-hint and handled in the body, or NL-sugar-marked). Release prerequisites (version bump, changelog, manifest sync, README row) are /complete-dev's job at Loop 3 — not tasks in this story's waves (§G).

7. Story

One story — 260629-jfz (route: skill, plugin pmos-managerkit, no dependencies): all five findings in one /execute run over SKILL.md + transcribe.sh + check-citations.mjs + fill-scorecard.mjs. Its ACs are the five findings' fixes (the FRs above), with F3 and F4 fused into the written-submission assessment feature. See the story item and stories/260629-jfz/03_plan.html. 8. Open questions

None blocking. The submission-scenario enrichment (post-live vs pre-live-then-present) was resolved with the maintainer during this define run. Two heuristics are deliberately conservative and should be tuned on the next real run, not re-architected: the submission-detection filename predicate (FR-3.1) and the auth-walled-host list (FR-1.1) — if a real input is mis-classified, widen the predicate/list, not the design.
