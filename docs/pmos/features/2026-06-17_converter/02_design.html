/converter — file-format conversion via a single-file web UI

Epic 260617-vvd · route: skill · plugin pmos-utilities · 2026-06-17 · This is the epic design doc (the cross-skill coherence contract every story cites by anchor) — the role spec_doc: plays for feature epics.

Overview

/converter is a new launch-only skill that starts a tiny zero-dependency Node server and opens a single self-contained HTML page in the browser. The page lets a PM convert one file format into another. v1 ships four bidirectional pairs (8 directions): JSON↔YAML, CSV↔JSON, HTML↔MD, and PDF↔MD. The whole thing is architected around a converter registry so new conversions are added by dropping in one library module — no server or UI edits (Inv-1).

Architecturally it mirrors pmos-gamekit's launch pattern (single-file HTML + zero-dep loopback server, ephemeral port, Ctrl-C stops, Node-prerequisite hard error) but adds a /convert backend endpoint. It lives in pmos-utilities (D1).

Goals & non-goals

Goals

  • A browser UI a PM can use to convert files between the v1 formats, launched with one command, fully local.
  • Zero npm dependencies — Node built-ins only; every conversion's logic is a vendored, reusable lib module (D6, Inv-3).
  • Extensible — the registry is the single extension point; a new conversion is one new lib module (registry, Inv-1).
  • Pure converters are deterministic and headlessly unit-testable; the one LLM-backed converter is clearly isolated (D4).

Non-goals (v1)

  • OCR / scanned-image PDFs, multi-column layout reconstruction, complex tables in PDF (D11).
  • Full YAML 1.2 (anchors/aliases/tags/multi-doc) — a pragmatic subset only (D9).
  • Lossless Markdown round-trips for every construct (lossy edges documented) (D10).
  • Auth, multi-user, cloud, persistence, file history. It is a local single-user tool.

D1 — Plugin placement: pmos-utilities

/converter serves none of the charters cleanly (it doesn't ship a feature, teach a topic, or play a game). It is standalone meta-tooling, which is what pmos-utilities holds (alongside /mac-health, /reflect). Chosen over a new pmos-convertkit plugin to avoid new-plugin scaffolding and ship faster; extensibility is satisfied inside the skill via the registry, not at the plugin level. Runner-up (deferred): a dedicated pmos-convertkit if the transform toolbox grows beyond a single skill.

D2 — Conversion runs server-side in Node

The HTML page is a thin client: pick a conversion, drop/select a file or paste text, POST /convert, preview + download the result. All conversion logic runs in the Node server. Rationale: Node gives us built-in zlib (PDF FlateDecode), Buffer (binary), filesystem, and child_process (the claude backend); converter modules stay pure and unit-testable without a browser; no reliance on browser-only APIs. The user explicitly approved a backend (“run a small http server with a basic backend script… prefer node js”).

Server endpoints

EndpointReturns
GET /the single-file ui/converter.html
GET /conversionsJSON list of registry descriptors (id, from, to, label, kind, requires, inputMode, outputMode) — the UI builds its selector from this (Inv-1)
POST /convertruns the chosen converter on the request body; returns text (preview) or a binary download with the right Content-Type (Inv-6). Errors return a structured JSON error with an actionable message (Inv-5).

Server uses only Node built-ins: http, fs, path, url, zlib, child_process, crypto, os, buffer. Loopback only (127.0.0.1), ephemeral port, Ctrl-C stops.

D3 — The converter registry (the extensibility contract)

Every conversion is a descriptor:

{
  id: 'json-to-yaml',          // unique
  from: 'json', to: 'yaml',
  label: 'JSON → YAML',
  kind: 'pure' | 'llm',        // pure = deterministic vendored lib; llm = claude subprocess
  requires: [],                // e.g. ['claude-cli'] for an llm converter; UI badges it
  inputMode: 'text' | 'binary',
  outputMode: 'text' | 'binary',
  convert(input, ctx) -> output | Promise<output>   // input: string|Buffer ; ctx: { log, tmpdir }
}

The registry auto-discovers lib/converters/*.js at server start (each module registers one or more descriptors). GET /conversions exposes the list; the UI renders the dropdown from it. Adding a conversion = add one descriptor module — zero server/UI edits (Inv-1). “2-way” ships as two descriptors per pair (forward + inverse), keeping the registry a flat list of 8 in v1.

D4 — Backend types: pure vs llm

  • pure — deterministic vendored lib. Synchronous-friendly, offline, no network, unit-tested with golden + round-trip fixtures. Seven of v1's eight directions are pure.
  • llm — shells out to the host claude CLI (claude -p) as a subprocess. Async; needs the CLI installed + network + tokens. Only pdf→md is llm in v1. Marked kind:'llm', requires:['claude-cli'] so the UI badges it (“Uses Claude — needs the CLI + network”).

This split is itself the future-extension story: later hard conversions (OCR, docx→md, image→text) reuse the llm backend without touching the pure path.

D5 — PDF↔MD strategy (hybrid)

The two directions are asymmetric because Claude emits text tokens, not binary files:

DirectionBackendHow
pdf→mdllm + pure fallbackPrimary: a mockable seam runClaudePdfToMd(pdfBuffer) → md that writes the PDF to os.tmpdir() and invokes the claude CLI to extract clean Markdown (handles real-world layouts/scans far better than any hand-rolled parser). Fallback when the CLI is absent/errors: the vendored lib/pdf-text.js FlateDecode text extractor, returning best-effort Markdown with a quality caveat surfaced in the response (Inv-5). Never crash/hang.
md→pdfpureThe vendored lib/pdf-writer.js — a minimal PDF writer using the 14 standard base fonts (no font embedding needed): headings (by font size), paragraphs with word-wrap via embedded AFM metrics, lists, blockquotes, and code blocks (Courier). Consumes the block model from lib/markdown.js. No images / complex tables in v1.

D6 — “No dependencies” defined

“No dependencies” means no npm packages installed. Allowed: Node built-in modules, and the host claude CLI as an optional backend for the one llm converter. The skill runs with zero installed packages and no network for 7 of 8 directions; pdf→md degrades gracefully to a built-in fallback if claude isn't available. A test asserts the skill's dependencies are empty (Inv-3). Every third-party concern is re-implemented as a vendored lib/ module we own and can extend.

D7 — Launch contract (mirrors the game-launcher)

The SKILL.md body is launch-only (plus a short list of supported conversions). It: (1) resolves server.js; (2) asserts Node is present (node --version) and on failure emits a Node-prerequisite hard error and stops — no silent file:// fallback (Inv-5); (3) runs node server.js; (4) reports the printed http://127.0.0.1:<port>/ URL and that Ctrl-C stops it. A free port is auto-selected and a missing Node is a hard error, so the skill issues no AskUserQuestion. It still carries the canonical non-interactive inline block per lint-non-interactive-inline.sh (the lint is the arbiter, like the game skills).

D7-test — Testing strategy

  • Per-lib golden + round-trip (node tests/run.mjs): parse(stringify(x)) deepEqual x for json/yaml; csv↔json on fixtures; md→html→md stable over the canonical subset (lossy edges asserted as documented).
  • Server endpoint tests: GET /conversions returns the full registry; POST /convert round-trips a sample per pure converter (mirrors gamekit's serve.test.mjs).
  • LLM converter: tested with the runClaudePdfToMd seam mocked (canned MD) + a fallback-path test (claude absent → vendored extractor invoked). No live API call in tests.
  • pdf-writer: structural assertion — output begins %PDF-, has an xref/trailer, and opens.
  • Per the host dogf_ood norm, the build also drives the running server + UI once (Playwright) and converts a real sample per pair.

Invariants (cross-story coherence contract)

IdInvariant
Inv-1Registry is the only extension point. Neither server.js nor the UI hardcodes a conversion; both derive from the registry. A new converter = one lib module, zero server/UI edits.
Inv-2Pure converters are pure. lib/converters/* of kind:'pure' export convert with no network/fs/DOM/global state; deterministic; headlessly testable. The one llm converter is the sole, explicitly-marked exception.
Inv-3Zero npm deps. No runtime dependency is added; built-ins + optional host claude CLI only; a test asserts dependencies is empty.
Inv-4Single-file UI. The served HTML is one self-contained file — no external refs, CDN, or build step.
Inv-5Graceful degradation. An llm converter whose backend is unavailable degrades to a vendored fallback or a clear actionable error — never crash/hang. Node-missing is a hard launch error (no silent file:// fallback).
Inv-6Text/binary contract. Descriptors declare input/output mode; the server delivers binary outputs (PDF) as a download with the correct Content-Type, text as preview + download.
