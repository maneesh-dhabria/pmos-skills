Design — Design-slop engine

Epic 260624-3jp · route: skill · pmos-toolkit · Tier 3. This is the cross-skill coherence contract (the design_doc:) the stories cite by anchor. It is the design output of the /shape session held earlier in this conversation; no epic-level /spec exists for a route:skill epic (skill spec is per-skill, folds into each story's /plan).

TL;DR

Port impeccable's deterministic frontend design-slop detector into pmos as a both-faces capability, vendored once as shared substrate and threaded through skills we already own — no new slash command. One engine; three consumers (detect via /design-crit + /verify, prevent via a generated rules floor); a drift-lint keeps detection and prevention in sync. All code/UI identifiers are pmos-native; the only tie to impeccable is one Apache-2.0 NOTICE.

Context classification

internal-tool — this is pipeline tooling for pmos itself (it improves the quality of what pmos generates and verifies). The lens floor downshifts strategic-positioning probes; the load-bearing lenses are feasibility-as-constraint (the dependency wrinkle, D-DEPS), licensing (D-VENDOR), and coherence with existing critique skills (non-duplication).

Engine internals (the vendored substrate)

Faithfully ported from impeccable's documented structure (see plugins/pmos-toolkit/agents/anti-patterns.md, which is impeccable's own maintenance guide carried in this repo for reference):

  • Rule registry (registry.mjs) — declarative array; each rule { id, category: 'slop'|'quality', name, description, skillSection, skillGuideline }. ~44 rules. skillSection ∈ { Typography, Color & Contrast, Layout & Space, Visual Details, Motion, Interaction, Responsive, UX Writing }. skillGuideline = a 3–6-word substring that must appear verbatim in the matching DON'T line of the prevention reference (the drift-lint key).
  • Pure checks (checks.mjs) — checkXxx(opts) → [{ id, snippet }]; no DOM access, plain options in. Real CSS/contrast math (contrastRatio, relativeLuminance, gradient parsing). Logic copied verbatim (D-VENDOR).
  • Two adapters wrap the pure checks: a browser adapter (uses getComputedStyle + getBoundingClientRect; the injected detector + window.pmosDesignScan()), and a Node adapter (no layout — reads explicit pixel dims via parseFloat(style.width); gates vertical-position checks behind a browser-only guard).
  • Snippet convention — every finding's snippet wraps its identifying text in straight double-quotes so tests can extract it (/"([^"]+)"/).

D-DEPS — the dependency wrinkle (the one thing story A must settle first)

The Node/static path needs to parse HTML + computed CSS without a browser. This is the single open mechanical decision; story A's /plan resolves it before any other story can build. Options, with leaning:

OptionFit with repo no-npm/vendored posture
(a) Vendor impeccable's own parser stack — htmlparser2 + css-select + css-tree (all MIT), pre-bundled into slop-engine/ as vendored JS, no npm install at runtime. LEANINGBest — matches the repo's vendored-zero-dep pattern (cf. /converter vendored FlateDecode, /diagram vendored tidy-tree). Faithful to impeccable's actual static path.
(b) jsdom for the Node pathWeak — jsdom is a heavy npm dependency; would be the repo's first. Convenient (full DOM) but cuts against the grain. Acceptable only as a test/dev dependency for fixtures, not a runtime requirement.
(c) Reuse the vendored browser bundle inside a minimal DOM shimFragile — re-implements half of jsdom. Rejected unless (a) proves infeasible.

Decision: default (a) — vendor the minimal MIT parser stack so the runtime engine is offline + npm-free; jsdom, if used at all, is confined to test fixtures. Story A confirms (a) is feasible during planning; if a vendored parser can't reproduce a given check, that specific check degrades to browser-only (run via /design-crit) and is skipped on the Node path with a logged note — never silently dropped.

Consumer contracts

/design-crit — deterministic slop pre-pass (story B)

  • Before its LLM Nielsen/WCAG/PSYCH critique, /design-crit injects slop-engine/browser.js into its already-open Playwright page, calls window.pmosDesignScan(), and reads the findings from the DOM (per the anti-patterns agent: read .pmos-slop-* programmatically, don't screenshot).
  • Deterministic findings are presented first (objective, with rule id + snippet + fix guidance), then the LLM critique layers UX judgement on top — they are reported as distinct lanes so a reader sees "machine-flagged tells" vs "judged UX issues".
  • Additive + graceful: if the engine can't load, /design-crit behaves exactly as today (logged note).

/verify — frontend slop gate (story C)

  • In the frontend-QA hard-gate phase, /verify runs detect.mjs on the generated HTML via the Node path (no Playwright) — cheap, fast, gate-able.
  • Findings route through _shared/findings-dispositions.md (Fix / Modify / Skip / Defer; severity [Blocker]/[Should-fix]/[Nit]). quality-category contrast/a11y faults map to [Blocker]/[Should-fix] and can gate the release; slop tells map to [Should-fix]/[Nit] and are surfaced loudly but never hard-block (taste must not stop a ship). grill-confirmed 2026-06-24
  • Tiered: mandatory for UI features (frontend-detection positive), skipped-with-log for non-UI work. Degrades gracefully when the Node parser/jsdom is unavailable (logged, non-fatal — never flips a correct PASS to fail on tooling absence).

Prevention reference (story D)

  • _shared/design-slop-rules.md — DON'T lines grouped by the skillSection headings, generated from the registry's skillGuideline fields (a small generator script in slop-engine/, so the floor can't drift from the rules by hand).
  • Cited as a floor (one-line cite, per skill-patterns §K one-fact-one-home) by /wireframes, /prototype, and frontend /execute — alongside the existing DESIGN.md house style.

Drift-guard lint (story D)

  • tools/lint-slop-rules.sh (repo-root, bash-3.2-safe) — for every rule with a skillGuideline, assert that substring appears in _shared/design-slop-rules.md; fail loudly on drift. Ports impeccable's validateAntipatternRules() cross-validator.
  • Wired into .github/workflows/skill-hygiene.yml next to the existing lints.

Non-duplication vs existing critique skills

/design-crit and /msf-wf are LLM-judged (Nielsen, WCAG-by-reading, PSYCH, journey friction). The slop engine is deterministic, zero-LLM, taste/freshness-focused ("AI tells") plus arithmetic a11y (contrast ratios, line length). They are complementary lanes: the engine gives objective machine findings; the LLM gives judgement. Story B explicitly stacks them (engine first, then judgement) rather than replacing either.

Cross-skill invariants (coherence the stories must hold)

  1. Inv-1 — one engine. All four stories read the same _shared/slop-engine/; no consumer forks the rules. Story A is the sole author of the engine.
  2. Inv-2 — registry is the source of truth. The prevention reference is generated from it; the drift-lint enforces the link; skillGuideline substrings are the contract between rule and DON'T line.
  3. Inv-3 — zero impeccable strings. A grep for impeccable (case-insensitive) over the engine + consumers returns hits only in NOTICE and design-doc lineage prose. Story A and D each assert this.
  4. Inv-4 — deterministic + offline. No engine path calls an LLM or the network. The LLM lives only in /design-crit's existing critique, layered after.
  5. Inv-5 — graceful degradation. Engine-absent / parser-absent never hard-fails a consumer; it logs and falls back to prior behaviour.

Risks & open questions

  • D-DEPS feasibility — can the vendored MIT parser stack reproduce every check that impeccable's static path does? Story A planning must verify; checks that can't are browser-only (run via /design-crit) with a logged Node-path skip.
  • Rule false-positives on pmos's own artifacts — the engine must produce zero false positives on existing pmos HTML (the comment-overlay chrome, editorial template). Port impeccable's two-column flag/pass fixture discipline and add pmos-artifact pass-cases.
  • Browser-bundle size — impeccable's bundle is ~217 KB; injected into /design-crit's page only, not shipped to users. Acceptable, but note vs any bundle-size lints.
  • Upstream drift — we fork at a point in time; future impeccable rule updates are a manual re-sync, not automatic. Acceptable (this is a vendored port, not a dependency).

Out of scope (this epic)

A standalone /slop command; a browser extension; a public website/overlay; live-URL crawling beyond /design-crit's existing Playwright; authoring net-new rules beyond impeccable's set; an npm-published CLI.
