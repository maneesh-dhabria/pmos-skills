Problem

A product manager has accumulated a large, well-curated corpus of ~200 PM frameworks (22 category pages in Notion, each holding 9–16 frameworks with a consistent ### Name / Reference / Author / Overview / 💡-commentary sub-schema). Today that corpus is only reachable by scrolling prose pages in Notion: it is not searchable by the problem a PM is actually facing, has no per-framework structure to filter or build on, is online-only, and cannot be consulted programmatically by other tools. The missing layer is problem→framework retrieval plus a readable, filterable surface — not more content.

Who experiences this?

  • Primary — the PM (human reader). Mid-work, facing a concrete product problem (prioritize a roadmap, make a hard/reversible decision, choose a north-star metric, position, price) and wanting a proven lens rather than reinventing or googling.
  • Secondary — other pmos skills (machine consumer). /ideate, /requirements, /grill, /spec that could enrich their output by pulling the frameworks relevant to the work in hand.

Why now?

The corpus has reached a size (~200 frameworks) where retrieval, not authoring, is the bottleneck. The same plugin already ships /magazine — a proven offline, filterable, self-contained HTML library backed by a closed registry — so ~80% of the delivery shape is a clone, not new invention. The remaining surface (problem→framework matching + a cross-skill lookup) is the genuinely new, high-leverage work.

Goals & Non-Goals

Goals

  • A PM describes a problem in their own words and gets a capped, ranked top-3–5 best-fit frameworks, each with a ≤1-sentence "why it fits" — measured by the author rating ≥1 of the top-5 as "I'd actually use this" (precision@5 the author trusts).
  • A PM can browse and filter the whole corpus offline (by category, super-category, lifecycle stage, decision type, problem tag, free-text) in a single self-contained HTML file that works from file:// — measured by the library opening and filtering with no network and no broken assets.
  • Each framework's detail surfaces the body, steps, references and the author's 💡 commentary as a distinct "PM's take" block — measured by commentary being present and visually distinct for every framework that has one.
  • Another pmos skill can call /frameworks --query "<problem>" --json and receive ranked structured matches — measured by ≥1 real consumer skill producing measurably better output with the lookup than without.
  • The corpus is refreshable from Notion on demand without breaking offline use — measured by /frameworks sync re-ingesting and the skill continuing to run with zero runtime Notion dependency.

Non-Goals

  • NOT a runtime Notion dependency — because the skill must run fully offline and fast; Notion is upstream authoring only.
  • NOT authoring or editing framework content — because the corpus is curated in Notion; this skill is the retrieval + structure layer on top.
  • NOT hot-linking Notion's S3 image URLs — because they are signed and expire (~1 hr), which would 404 the detail views.
  • NOT a generic public framework wiki — because the value is this author's curated 200 + personal commentary, offline and skill-callable.
  • NOT writing the derived tags back to Notion in this iteration — because the situation-taxonomy is skill-owned for fast iteration (deferred to a possible follow-up).
  • NOT auto-syncing or nudging on staleness — because refresh is an explicit, user-driven action.

User Experience Analysis

Motivation

  • Job to be done: "When I'm facing a product problem, surface and apply the best-fit proven framework, so I reason better and faster without reinventing the wheel."
  • Importance/urgency: Decisions are time-pressured; a PM who can name the right lens fast reasons more credibly. If they can't find it, they default to gut or a generic web search.
  • Alternatives: Scroll Notion (slow, no problem-search); google "PM prioritization frameworks" (generic, uncurated, no personal take); rely on memory (incomplete).

Satisfaction Signals

  • The top result is one the PM recognizes as apt — and the "why it fits" line confirms the match reasoning.
  • The detail carries the author's 💡 take, which reads as a trusted colleague's opinion, not a generic encyclopedia entry.
  • The library feels instant and works on a plane (offline), reinforcing "this is mine, always available."

Solution Direction

A two-stage skill that mirrors /magazine's deterministic-then-LLM pipeline and offline-HTML spine:

  • Ingestion (build-time, run via /frameworks sync): walk the 22 Notion category pages through the Notion MCP, split each on ### framework headings, and produce a shipped data/frameworks.json corpus — lean fields extracted deterministically, rich match-fields LLM-derived once and cached, body preserved verbatim, commentary captured as a distinct field, images handled per D5.
  • Retrieval (runtime, offline): a PM picks a situation from the closed taxonomy or types a free-text problem → the skill ranks the corpus and returns a capped top-3–5 with a "why it fits" line. A --json mode returns the same ranked matches structured for other skills.
  • Browse (runtime, offline): a single self-contained index.html library — faceted filter rail (category · super-category · lifecycle stage · decision type · problem tag) + search box + result cards that expand inline to full detail, including the "PM's take" commentary block. Works from file://.

Surface map (what the user invokes):

/frameworks "<problem>"          → ranked top-5 in chat + opens library focused on them
/frameworks "<problem>" --json   → ranked {id,name,why,score} for skills (no UI)
/frameworks list | browse          → opens the full filterable library
/frameworks sync                   → re-ingest from Notion → data/frameworks.json
/frameworks situations             → list the closed PM-situation taxonomy

User Journeys

Primary journey (problem → framework)

  1. PM runs /frameworks "I need to decide whether to sunset a feature".
  2. Skill loads the shipped corpus (offline), ranks it, and returns the top-3–5: e.g. One-Way vs Two-Way Doors ("the sunset is reversible-ish — size the door first"), Regret Minimization, Pros/Cons with weights — each with a one-line why.
  3. Skill opens (or points to) the library index.html with those frameworks surfaced; PM clicks one to expand full detail + the author's 💡 take.
  4. PM applies the framework.

Machine journey (skill-to-skill)

  1. Another skill (e.g. /ideate Expand) calls /frameworks --query "<the work's problem>" --json.
  2. Skill returns ranked {id, name, why, score} — no UI, no chat noise.
  3. The caller weaves the relevant frameworks into its own output.

Browse journey

  1. PM runs /frameworks browse (or opens the library file directly, even offline).
  2. Filters by category "Decision Making" + decision type "irreversible"; types "pricing" in search.
  3. Expands cards inline to read detail + commentary; copies a reference link.

Sync / refresh journey

  1. PM updates a framework in Notion, then runs /frameworks sync.
  2. Skill re-ingests via Notion MCP, rebuilds data/frameworks.json (re-deriving cached match-fields), updates provenance; reports a diff summary (added/changed/removed).
