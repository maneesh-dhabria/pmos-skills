Architecture

Two-stage pipeline mirroring /magazine: deterministic Stage-A scripts (zero-dep Node .mjs) + in-session Stage-B agent steps (Notion MCP fetch, LLM match-field derivation, /diagram generation). Runtime retrieval + browse are fully offline over the shipped corpus.

Runtime vs build boundary

  • Build-time (/frameworks sync): Stage-B agent + Stage-A scripts produce data/frameworks.json, data/diagrams/*.svg, and index.html. Requires Notion MCP + LLM + /diagram.
  • Runtime (/frameworks "<problem>" / browse / --json): reads the shipped corpus only. Deterministic prefilter via match.mjs + an in-session LLM rank for free-text. No network.

Corpus schema (frameworks.json)

Array of records. Lean fields (deterministic from Notion) + cached match-fields (LLM-derived once at sync). Full contract in reference/corpus-schema.md.

{
  "id": "decision-making/regret-minimization",   // <category-slug>/<name-slug>, stable
  "name": "Regret Minimization Framework",
  "aliases": ["Regret Minimization"],
  "category": "Decision Making",
  "category_code": "2.4.4",
  "super_category": "People, Personal & Career",  // derived from 2.x (see map)
  "summary": "≤160-char one-liner (deterministic: first sentence of overview, or LLM-tightened)",
  "body_md": "<verbatim Overview markdown — nothing dropped>",
  "references": [{"type":"Article","url":"https://..."}],
  "author": "Jeff Bezos",            // or null
  "commentary": "<verbatim 💡 callout text>",   // or null  (the 'PM's take')
  "diagram": "data/diagrams/decision-making__regret-minimization.svg",  // or null
  "source_url": "https://notion.so/<page>#<framework>",
  "last_synced": "2026-06-07",
  // ---- cached match-fields (LLM-derived once; vocab-validated) ----
  "problem_tags": ["irreversible-decision","high-stakes"],   // ⊆ closed registry
  "when_to_use": "≤1 sentence",
  "when_not_to_use": "≤1 sentence",
  "decision_type": "judgment",       // enum: judgment|analysis|prioritization|framing|estimation|n/a
  "lifecycle_stage": ["any"],        // enum: discovery|definition|delivery|growth|any
  "related": ["decision-making/one-way-vs-two-way-doors"]
}

super_category map (from category_code 2.x): 2.1.x → Strategy & Business · 2.2.x → Product · 2.3.x → Analytics, Design & Finance · 2.4.x → People, Personal & Career.

FR-SCHEMA-1: body_md is preserved verbatim — extraction never drops prose. FR-SCHEMA-2: records with a sparse source still validate (only id, name, category, body_md are required; the rest may be null/empty). FR-SCHEMA-3: problem_tags must be a subset of the closed registry in situations.json; derive-fields.mjs rejects unknown tags.

Situation taxonomy & tag registry (situations.json)

Skill-owned (not written to Notion). Closed problem_tags registry + ~15–25 recognizable PM situations, each pre-mapped to framework ids. Flat tags; situations grouped under the 4 super-categories for display (resolves OQ3).

{
  "version": 1,
  "problem_tags": ["prioritization","north-star-metric","irreversible-decision",
                   "pricing","positioning","stakeholder-alignment","estimation", ...],
  "situations": [
    {"id":"prioritize-roadmap","label":"I need to prioritize a roadmap",
     "super_category":"Product","tags":["prioritization"],
     "frameworks":["product/rice","product/kano","product/moscow"]},
    {"id":"reversible-decision","label":"I'm stuck on a reversible decision",
     "super_category":"People, Personal & Career","tags":["irreversible-decision"],
     "frameworks":["decision-making/one-way-vs-two-way-doors","decision-making/regret-minimization"]}
  ]
}

FR-TAX-1: situations are authored from the 22 Notion categories; every situation's frameworks[] must reference existing corpus ids (validated). FR-TAX-2: /frameworks situations lists them grouped by super_category.

Ingestion pipeline (/frameworks sync)

  1. Fetch (Stage-B, MCP): for each of the 22 category pages, notion-fetch → raw enhanced-markdown. Large pages exceed the tool token cap → fetch saves to a temp file; the agent slices it.
  2. Split (Stage-A, split-corpus.mjs): split each category's markdown on ### headings → per-framework raw records. Deterministically extract name, body_md (the Overview bullets verbatim), references[] (parse the - Reference - [Type](url) line), author (the - Author - line), commentary (the trailing 💡 callout), category/category_code (from the page title), source_url.
  3. Derive match-fields (Stage-B, LLM + derive-fields.mjs): per framework, the agent emits problem_tags (⊆ registry), when_to_use, when_not_to_use, decision_type, lifecycle_stage, related, summary. derive-fields.mjs validates the enums/registry and merges into records. Bulk fan-out: parallel subagents, N frameworks each, strict output contract (per the bulk-lookup learning).
  4. Diagrams (Stage-B, /diagram batch): per framework, at full rigor: /diagram "<name>: <summary>" --source <body> --non-interactive --approach "<derived framing>" --theme technical --out data/diagrams/<id>.svg --on-failure ship-with-warning. Fan out in waves. --changed-only skips frameworks whose body_md content-hash is unchanged since last sync (cache in data/.diagram-hashes.json).
  5. Assemble + validate (Stage-A): write data/frameworks.json; validate-corpus.mjs emits a coverage report (count, field coverage %, diagram coverage %, dangling related/refs); exit 1 if <95% name+body+references or any invalid tag.
  6. Build library (Stage-A, build-library.mjs): frameworks.json + diagrams → self-contained index.html (diagrams inlined as <svg>; offline from file://).

FR-ING-1: sync never hot-links S3; diagrams are owned SVGs inlined into the library. FR-ING-2: one /diagram failure logs + keeps the prior SVG (or text-only) and continues the batch. FR-ING-3: Notion unreachable → sync fails cleanly; the existing shipped corpus is untouched.

Matching & --json contract

Two-stage rank (full contract in reference/matching.md):

  • Situation path: input is a known situation id (or fuzzy-matches a situation label) → return that situation's frameworks[], ranked by tag overlap.
  • Free-text path: (1) match.mjs deterministic prefilter — tokenize the problem, score each record by weighted overlap with problem_tags (×3), name/aliases (×2), when_to_use + summary keywords (×1) → top ~15 candidates + scores. (2) In-session LLM re-ranks the candidates against the problem and writes a ≤1-sentence "why it fits" per pick. Cap output at 5.
  • Confidence floor: if the top deterministic score < threshold (default 0.15, tunable via --floor), mark low_confidence and return ≤2 closest with an explicit caveat rather than padding to 5 (resolves OQ2).

--json shape (FR-JSON-1):

/frameworks --query "<problem>" --json
→ {"query":"<problem>","count":N,"low_confidence":false,
   "matches":[{"id","name","why","score","category","decision_type","diagram"}]}

FR-JSON-2: a contract test (tests/json-contract.test.mjs) asserts the shape (keys, types, cap≤5, score∈[0,1]) against match.mjs output over a fixture corpus — this is the "proven contract" that stands in for a live consumer (wiring /ideate is a backlog follow-up). FR-JSON-3: --json emits only JSON to stdout (no chat prose), single object, parseable.

Command surface & frontmatter (skill-patterns §A/§B)

/frameworks "<problem>"              → ranked top-5 in chat + opens library focused on them
/frameworks "<problem>" --json       → structured matches (no UI/prose)  [--floor N]
/frameworks browse | list            → open the full filterable library (index.html)
/frameworks situations               → list the closed situation taxonomy
/frameworks sync [--changed-only]    → re-ingest from Notion → corpus + diagrams + library

"/frameworks", "which framework should I use", "find a PM framework for", "frameworks for prioritization/decisions", "build my framework library". user-invocable: true. FR-FM-2: argument-hint enumerates every token/flag above + --json --floor --changed-only --format --non-interactive --interactive. FR-FM-3 (§F): codex manifest mirror carries the interface block.

SKILL.md body requirements (skill-patterns §C/§D)

  • FR-BODY-1 (§C): progressive disclosure — heavy detail lives in the 4 reference/ files; SKILL.md body stays lean and points to them.
  • FR-BODY-2 (§D): body includes ## Platform Adaptation, ## Track Progress, a ## When NOT to use section, the non-interactive contract block (byte-identical to _shared/non-interactive.md), and a final ## Phase N: Capture Learnings (literal — colon after N, capital "Learnings") that emits the one-line learning output.
  • FR-BODY-3 (§E): every script is zero-dep Node .mjs with a --selftest mode; BASH_SOURCE-style path fallbacks in any .sh.
  • FR-BODY-4: Anti-Patterns section (DO NOT hot-link S3; DO NOT block runtime on Notion; DO NOT pad matches past the confidence floor; DO NOT regenerate unchanged diagrams under --changed-only; DO NOT write derived tags back to Notion).
