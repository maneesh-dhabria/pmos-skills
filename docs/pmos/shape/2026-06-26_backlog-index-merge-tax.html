Backlog INDEX.md — a no-reader cache on the merge path

Created using pmos‑toolkit

The shaped problem

Every backlog mutation has to rewrite a committed status snapshot (backlog/INDEX.md) that no tool and no human ever reads — so the operator pays a hand-regeneration tax on each /backlog change and each /feature-sdlc define, and when parallel worktrees touch that one file at once they eat avoidable git merge conflicts — all to keep current a view whose every field is already derivable on demand from the item YAML.

Who / when / why-now: Whoever runs backlog mutations or /feature-sdlc define|build (today, the model acting for the operator). It bites on every mutating command, and acutely whenever concurrent worktrees / release-trains rewrite INDEX.md in parallel. Why now: parallel worktrees + release trains are this repo's default working model, and completed items already dominate the corpus (124 of 143), so the hand-regenerated file only grows larger and more collision-prone over time.

This is a problem brief. It deliberately proposes no solution — see Handoff below.

Felt problem (HMW + JTBD)

How might we: keep the backlog's at-a-glance state correct without maintaining a committed, hand-regenerated file that nothing reads and that collides across worktrees?

Who: the operator — and the model acting on their behalf — running the define → build → release loops across parallel git worktrees.

When / where: every /backlog add|refine|set|promote and every /feature-sdlc define merge — the moment a mutating handler ends by rewriting INDEX.md and that change lands on the commit / merge path.

Job-to-be-done: when I mutate the backlog, I want the canonical state to update with zero extra bookkeeping and zero merge friction, so I can run many features in parallel without collisions on a shared file.

Falsifiable hypothesis: If we solve the INDEX-on-the-merge-path coupling, then define/build commits carrying INDEX diffs, and cross-worktree merge conflicts on backlog state, observably drop to ~zero.

Abstraction ladder

Climbing why / descending how to find the right altitude — with a 5-Whys pass on the "merge conflict on INDEX.md" symptom.

  • Symptom (↓ how): two worktrees both edited backlog/INDEX.md → git merge conflict on the file.
  • Why? Every mutating handler ends by rewriting the whole INDEX.md, and worktrees run concurrently — so the same lines move in two branches.
  • Why rewrite it at all? INDEX.md is a committed cache that must be kept fresh to match the item YAML (the tracker-crudl "regenerable cache" contract).
  • Why keep that cache fresh — why does it exist? To offer a human-readable at-a-glance backlog without running any tooling.
  • Why does that justify a committed, mutated-on-every-write file? It doesn't — no human or tool actually reads it (confirmed write-only this session). Every programmatic consumer already re-derives from the YAML.
  • Root (↑ why): a derivable view is being persisted as committed mutable state on the merge path, with no consumer.

Chosen altitude: the mid-rung — "a derivable, no-reader view is persisted as committed mutable state on the concurrent-merge path." High enough to question whether INDEX.md should exist; low enough to act on without re-litigating the whole tracker design.

Decomposition

Sub-problems, with the real one marked ★.

  1. ★ The no-reader committed cache. INDEX.md is committed, mutable, hand-regenerated state that every mutation rewrites — incurring (a) a regeneration / maintenance tax and (b) cross-worktree merge conflicts — while no consumer reads it. This is the leverage point.
  2. Trust / freshness. Even taken as a human snapshot, it can be stale (the "Last regenerated" line can lag the YAML), so it fails at the only job it nominally has. (Subordinate to ★ — and moot if ★ is resolved, since nobody reads it anyway.)
  3. Anticipated parse cost. Re-deriving views from all 143 → N item files on every command could get slow at scale. Currently speculative — 143 small files parse fast; the operator raised it, but it is a distinct, lower-priority problem and possibly not yet real. "Move completed items to a folder" is a solution to this sub-problem, not a problem — out of scope for this brief.
  4. Pattern blast-radius. The same committed-derivable-cache shape exists in the sibling trackers (/mytasks, /people) built on _shared/tracker-crudl.md — so a fix scoped to backlog alone may leave the same pattern (and the same costs) in place elsewhere.

Open questions

  • Scope to siblings? Does the same fix apply to /mytasks and /people, which share the tracker-crudl substrate and their own INDEX files — or is this backlog-only?
  • Is SP3 real? Is the parse-performance concern real at the current/expected corpus size, or premature? Needs a measurement before any "move completed items to a folder" work is justified.
  • Where would a human view live, if ever wanted? If an at-a-glance view is desired later, where does it live so it's not on the merge path (generate-on-read? the /backlog web viewer only? a .gitignored build artifact)? A solution-space question — deferred to /ideate or /requirements.
  • Latent external consumers? Are there INDEX.md readers outside this repo's skills (external tooling, bookmarks, doc links) to grep for before removal?
