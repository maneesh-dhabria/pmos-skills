Inline-everything HTML artifacts

Created using pmos‑toolkit

TL;DR

Pmos-emitted HTML artifacts currently load CSS, JS, and comments overlay from a sibling assets/ directory. When opened standalone (Claude Code desktop preview, email, Dropbox, archived copies), the sibling fetches fail and the file renders unstyled with no overlay. Proposal: make every emitted artifact fully self-contained — inline CSS, inline JS, comments persisted as an embedded JSON block inside the HTML — and route the comments write path exclusively through a tiny serve.js running on http://localhost. file:// viewing becomes a perfectly-styled read-only view; commenting requires the existing comments-open.* launcher trio. Sidecar .comments.json files are retired; the drift hook, FSA fallback, and DMP (80KB) get deleted along with them. Per-artifact overhead lands around 50KB; the 14 emit surfaces inherit the change through one template substitution.

Validation gate before any implementation: a 10-minute spike confirming Claude Code desktop preview actually honors inline <style> blocks (the entire premise rests on this).

HMW

How might we make pmos HTML artifacts fully functional (styling + comments) when opened standalone — desktop app, email, Dropbox — without an assets/ sibling directory?

Target user & JTBD

User: the plugin author (sole user today) plus any stakeholder who receives a pmos artifact by link or attachment.

Job-to-be-done: when I open a pmos artifact anywhere — Claude Code desktop preview, Mail.app, a Dropbox preview, a downloaded copy — I want to see it properly styled and (when launched via the comments helper) be able to read and write annotations, so that I don't have to first remember to serve it from a local web server just to make the document usable.

Hypothesis

Inlining CSS + JS + initial comments JSON into the emitted HTML — and routing all comment writes through a local node http server that rewrites the embedded JSON block in place — yields a single self-contained file that styles correctly under file:// and supports full read/write under http://localhost, while eliminating the entire sidecar/drift-hook/FSA-fallback surface area.

Idea variants considered

Twelve variants were generated across four groups using First Principles + SCAMPER. Group A inlined everything; Group B split rendering vs. interaction; Group C eliminated dependencies (DMP, turndown); Group D substituted the packaging layer (post-emit bundler, dual emit, data-URIs).

After clarification, the chosen finalist is a composite that combines: full inline at emit time (Group A) + dependency trimming (drop DMP, drop turndown in favor of optional MD sidecar; Group C) + switch comments persistence from sidecar JSON to inline JSON block read/written by serve.js. The build-step bundler (#10) was rejected as needless indirection — inline-at-template-render is a 10-line glue change, not a build pipeline.

How it works

Emit shape. _shared/html-authoring/template.html grows three new tokens: {{inline_css}}, {{inline_js}}, {{inline_comments_json}}. The renderer reads style.css + comments.css from the authoring source, drops them into a single <style> block; reads viewer.js + a slimmed comments.js into a single <script> block; seeds the comments JSON with {"schema":1,"threads":[]} wrapped in <script id="pmos-comments" type="application/json"> delimited by sentinel comments (<!-- pmos-comments:start --> / <!-- pmos-comments:end -->) for unambiguous regex anchoring.

Read path. Under file:// (Claude desktop preview, email, Dropbox), the page loads with full styling; viewer.js renders existing comment threads from the inline JSON. Read-only mode is auto-detected by attempting a HEAD /save — if the fetch fails, the overlay shows threads but hides the compose UI.

Write path. Under http://localhost served by the existing comments-open.* launcher trio invoking serve.js, the overlay is fully interactive. On submit, the client POSTs to /save with the full updated JSON + expected_version. The server reads the artifact file, regex-replaces the content between the sentinel comments, writes to artifact.html.tmp, and atomically renames to artifact.html. JSON strings are escaped so that any embedded </script> sequences in comment bodies serialize as and never break the script tag.

What gets deleted. The .comments.json sidecar format; the pre-commit drift hook pairing artifact+sidecar; the FSA write path; the localStorage fallback; diff-match-patch.js (80KB, only needed because sidecar+artifact could drift — now they can't because they're one file); the per-skill assets/ sibling directory under {docs_path}/<skill>/. Launchers + serve.js live in one known location (e.g. ~/.pmos/serve/) invoked with the artifact path as argv.

Re-emit handling. When /spec --update (or any other re-emit) regenerates the artifact, the renderer first reads the existing file's <script id="pmos-comments"> block, preserves it, and writes it back into the new emission. The existing quote-hash anchor resolver in comments.js ports forward unchanged: matched → preserve; near-match → status: needs-review; lost → status: orphaned with a visible bottom-of-page panel.

Size profile. Inline CSS (style 15KB + comments 6KB ≈ 21KB) + inline JS (viewer 13KB + slimmed comments-js ~15KB ≈ 28KB) → ~50KB overhead per artifact. A typical /spec HTML goes from ~50KB to ~100KB. DMP and turndown (~110KB combined) are not inlined.

Alternatives & prior art

  • Post-emit bundler (#10). Keep authoring modular; add a bundle-artifact.js pass that inlines after emit. Rejected: adds a build step where template substitution suffices.
  • Inline-CSS-only (#4/#5). Honest cheap fix — styling works everywhere, comments overlay silently no-ops without sibling assets. Worth keeping as a fallback if the desktop-preview spike fails for the JS path but passes for CSS.
  • Dual emit (#12). Ship both current external-asset version and a .standalone.html sibling. Rejected: doubles disk; no advantage over making the standalone shape the only shape.
  • Data-URI assets. Equivalent in effect to inline <style>/<script> but harder to debug. Rejected as obscure.
  • FSA-only write path. Already partially implemented (T22). Browser-specific (Chrome), requires user-gesture file picker on every save, and doesn't work in Safari/Firefox at all. Local-http server is more uniform.
  • Prior art: Jupyter notebooks (.ipynb) store everything (cells, outputs, metadata) in a single JSON file rendered to HTML on view. Observable notebooks ship as self-contained HTML for the same portability reason. Single-file SVG with embedded scripts is a well-established pattern.

Success signals

  • Primary: opening any pmos-emitted HTML artifact in Claude Code desktop preview shows the toolbar, properly-styled prose, and existing comment threads (read-only) — no missing-asset visual artifacts.
  • Secondary: launching via comments-open.sh <artifact> opens the same file at http://localhost:PORT with full read+write comments.
  • Tertiary: total asset footprint under docs/pmos/ drops to zero sibling assets/ directories; per-artifact size increase ≤ 60KB; no .comments.json files remain after migration.
  • Quality: drift hook removed from scripts/install-comments-hooks.sh; diff-match-patch.js deleted from authoring source; coverage gate in /verify shrinks accordingly.
  • Negative signal to watch for: regression of comment anchor resolution accuracy after DMP removal — should be measurable against the existing calibration corpus at plugins/pmos-toolkit/skills/comments/tests/fixtures/calibration-spans-2026.json.

Next steps

  1. 10-minute desktop-preview spike against assumption #1. This is the precondition for everything else. If it fails, pivot to "always-serve via http" as the only path and skip the file:// inlining optimization.
  2. If spike passes: kick off /feature-sdlc in a separate session, seeding it with this artifact path. The pipeline will produce a Tier-2 or Tier-3 requirements doc → spec → plan → execute, given the multi-surface blast radius (14 emit surfaces + serve.js + migration script + drift-hook removal + DMP/turndown deletion).
  3. Before /spec runs, grep audit for DMP call sites (assumption #4 test) so the spec can either pin the server-side resolver design or descope DMP removal.
  4. Decide the launcher home: keep comments-open.* per-feature folder, or relocate to a single ~/.pmos/serve/ install. (Recommend the latter — one fewer thing to copy per emit.)

Open questions

  • Does Claude Code desktop preview honor inline <style> AND inline <script>, or just <style>? (Determines whether the overlay renders threads under file:// or only styling works.)
  • Should serve.js be one-artifact-per-port or a persistent daemon serving the whole docs/pmos/ root? Persistent daemon is friendlier; one-shot is simpler.
  • Is output_format=both an acceptable new default (resolves Copy-MD degradation cleanly), or should the button just disappear when turndown is removed?
  • What's the migration story for already-published / already-shared artifacts that still have sidecars on someone else's disk? (For solo-user today: not a concern. Flagging for future.)
  • Does the embedded JSON approach interact awkwardly with any HTML viewer that strips unknown <script> tags (e.g. email clients sanitizing the doc)? — likely yes, but those viewers also strip the overlay JS, so comments-in-email was never on the table anyway.
