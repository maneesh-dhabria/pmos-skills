/frameworks — PM framework finder skill

Created using pmos‑toolkit

TL;DR

A new pmos-learnkit skill, /frameworks, that turns the user's ~200-framework PM corpus (currently 22 prose category pages in Notion) into a structured, offline, filterable HTML library where a PM describes a problem and gets 3–5 ranked, best-fit frameworks with a one-line "why it fits" — and exposes a --json lookup so other pmos skills can pull relevant frameworks programmatically.

Recommendation: Proceed — but build it in two waves. Wave 1 (the spine): extract-once to a local offline corpus + lean schema + free-text matching + single-file filterable HTML that preserves the author's commentary. Wave 2 (the differentiators): LLM-cached match fields, a closed situation-taxonomy, the cross-skill --json API with one real consumer, and /frameworks sync. The two highest risks — lossy extraction and matching that erodes trust — are both retired in Wave 1, so the spine is shippable on its own.

How Might We

How might we let a PM (or another pmos skill) describe a problem they're facing and surface the right proven framework(s) from a curated ~200-framework corpus — in a self-contained, filterable artifact they can read and act on offline?

Target user & job-to-be-done

Who: A product manager (primary), and other pmos skills as a secondary "machine" consumer (/ideate, /requirements, /grill, /spec).

When / where: Mid-work, facing a concrete product problem — prioritizing a roadmap, making a hard/reversible decision, choosing a north-star metric, positioning, pricing — and wanting a proven lens rather than reinventing or googling.

JTBD: When I'm facing a product problem, I want to surface and apply the best-fit proven framework, so I can reason better and faster without reinventing the wheel.

Hypothesis

We believe extracting the corpus into a structured local index with problem-oriented matching and a filterable offline UI will produce faster, higher-trust framework selection for PMs (and reusable framework lookups for other skills) because the frameworks are already consistently structured and authoritative; the missing layer is problem→framework retrieval and a readable, filterable surface — not more content.

Idea variants considered

Generated via First Principles + Analogous Inspiration (cloning /magazine's offline-HTML library + closed registry, and /learn-list's verified-source/annotated/ranked pattern). The chosen composite finalist is flagged with ★.

  1. ★ Extract-once → local corpus. A script + one-time LLM pass converts the 22 Notion pages into a checked-in frameworks.json (+ markdown bodies). Notion becomes upstream authoring, not a runtime dependency; the skill runs fully offline.
  2. Live Notion-backed at runtime — query Notion MCP on every call. (Foil — kills offline / portability / speed; rejected.)
  3. ★ /frameworks sync re-ingest. A command re-pulls from Notion to refresh the local corpus; each record carries source_url + last_synced provenance.
  4. Rich per-framework record (full analytic schema: when_NOT_to_use, inputs_required, decision_type, related[] …) built at extraction.
  5. Lean per-framework record only (id, name, category, tags[], summary, body_md, references[]) — defer rich fields.
  6. ★ Two-layer schema. Lean fields extracted deterministically; the rich match fields (problem_tags, when_to_use, decision_type) are LLM-derived once at ingestion and cached, so match quality costs zero runtime tokens.
  7. Closed tag/facet filter only (category · lifecycle-stage · decision-type · tag), like /magazine's registry.
  8. Free-text semantic match — PM describes the problem in prose; in-session LLM ranks frameworks with a ≤1-sentence "why it fits."
  9. ★ Closed "situation" taxonomy + free-text fallback. A curated set of recognizable PM situations ("prioritize a roadmap", "stuck on a reversible decision", "choose a north-star metric") pre-mapped to frameworks; free-text falls back to variant 8.
  10. ★ Single self-contained index.html library. Faceted filter rail + search box + result cards; offline from file://. Clones /magazine.
  11. Master–detail SPA in one file — filter+list left, detail pane right; no reloads.
  12. ★ Inline expandable detail + "PM's take" block. Cards expand in place; detail renders body, steps, references and the author's 💡 commentary as a distinct block (the author's voice is the differentiator vs. a generic framework wiki).
  13. ★ /frameworks --query "<problem>" --json returns ranked structured matches (id, name, why, score) consumable by other skills — frameworks-as-a-lookup-service.
  14. Named skill integrations — /ideate Expand pulls decision/creativity frameworks; /requirements pulls prioritization frameworks; /grill pulls decision frameworks. (Folded into Wave 2 as the "one real consumer" proof.)

Chosen finalist (composite): 1 + 3 + 6 + 9 + 10 + 12 + 13, sequenced into two waves (see How it works).

How it works (chosen finalist)

  • Wave 1 — the spine (ships standalone):
    • Ingest once. A script walks the 22 Notion category pages via the Notion MCP, splits each on ### framework headings, and writes a local frameworks.json + per-framework body_md. The 2-level category taxonomy is derived for free from the 2.x.y numbering.
    • Lean schema, deterministic. Each record: id, name, aliases[], category, super_category, summary, body_md, references[{type,url}], author, commentary. The author's 💡 callouts are captured verbatim as commentary — nothing prose is discarded.
    • Vendor or drop images. Notion S3 image URLs are signed and expire (~1 hr) — download-and-vendor locally at ingestion, or fall back to text-only. Never hot-link S3.
    • Free-text matching. PM describes a problem → in-session LLM ranks the corpus, returns top 3–5 with a ≤1-sentence "why it fits." Hard cap at 5; show the reasoning.
    • Single-file filterable HTML. One index.html library: faceted filters (category · super-category · tag) + search box + result cards that expand inline to full detail with the "PM's take" commentary block. Offline from file://, cloning /magazine.
  • Wave 2 — the differentiators (build once the spine is trusted):
    • LLM-cached match fields. Add problem_tags[], when_to_use, when_NOT_to_use, decision_type, lifecycle_stage[], related[] — derived once at ingestion, cached in the record so matching is sharper at zero runtime cost.
    • Closed situation-taxonomy. ~15–25 recognizable PM situations, each pre-mapped to frameworks; free-text falls back to Wave-1 matching. (This is designed, not imported — the Notion Tags field is empty today.)
    • Cross-skill API + one real consumer. /frameworks --query "<problem>" --json returns ranked {id, name, why, score}. Ship one consumer (e.g. /ideate Expand pulling decision frameworks) to prove the contract before generalizing.
    • /frameworks sync. Re-ingest from Notion; provenance + last_synced drive a staleness nudge.

Premortem: failure modes

It is 2027-06-07. /frameworks shipped 12 months ago and failed. Why?

Failure modeLikelihoodMitigation
Lossy / inconsistent extraction. The 200 frameworks vary (some have steps, some don't; images are expiring S3 links) — the corpus ended up sparse and matches felt random.HKeep body_md verbatim so nothing is lost even when structured fields are sparse; build an extraction validator; spot-check a sample per category; treat images as optional.
Matching erodes trust. "I need to prioritize" returned 30 frameworks or the wrong ones; PMs gave up after 2–3 bad results.HHard-cap at 3–5 ranked results with a visible "why it fits" line; invest the Wave-2 cached problem_tags/when_to_use fields exactly to lift precision; let users see the reasoning.
Expiring S3 images break detail pages. Notion's signed image URLs (the ones sampled expire in ~1 hr) 404 in the artifact.HDownload-and-vendor images locally at ingestion, or go text-only. Never hot-link S3. (Wave 1.)
Cross-skill API with no consumers. Built --json, no skill adopted it (integration cost > perceived benefit).MDefer the API to Wave 2 and ship it with one real consumer; don't generalize a contract nobody calls.
Corpus drifts from Notion. The book kept getting edited; the local copy went stale; sync was never run.MProvenance + last_synced banner + a cheap staleness nudge; make sync idempotent and one-command.
Scope creep. Tried to ship rich schema + taxonomy + API + sync + great UX in v1 and never shipped.MThe two-wave plan is the mitigation — Wave 1 is independently shippable and retires both H-risks.

Munger inversion — choices that would guarantee failure

  • Hot-link Notion's expiring S3 image URLs and depend on the Notion MCP at runtime → broken, offline-hostile artifact.
  • Re-derive problem_tags fresh on every query → slow, token-expensive, non-deterministic matches.
  • Extract only structured fields and discard the prose body + 💡 commentary → throw away the actual differentiator (the author's voice).
  • Ship the --json API with zero real consumers → dead weight nobody validates.
  • Return everything weakly-related instead of a capped, ranked top-5 → instant trust erosion.
  • Invent a giant tag taxonomy up front (the empty Tags field) ungrounded in real PM situations → unused filters.

Success signals

Leading: A PM types a real problem and rates ≥1 of the top-5 results as "I'd actually use this" (precision@5 the author trusts); the filterable library is opened and browsed offline; image/detail rendering holds up with no broken assets.

Lagging: The author reaches for /frameworks instead of scrolling Notion; at least one other pmos skill adopts the --json lookup and its output is measurably better with it; sync keeps the corpus current across book edits.

Open questions

  • Matching engine in Wave 1: pure in-session LLM reasoning over the corpus (no index) is simplest, but does precision@5 hold at ~200 records, or do we need the Wave-2 cached match fields sooner than planned?
  • Corpus location: does frameworks.json live inside the plugin (versioned, shared with anyone who installs pmos-learnkit) or under ~/.pmos/ (user-private)? The author's commentary is personal voice — is this corpus shippable to other users, or author-only?
  • Image strategy: vendor images into the artifact (bigger files, fully offline) vs. text-only (smaller, loses diagrams). How many of the ~200 frameworks are genuinely image-dependent?
  • Situation-taxonomy ownership: the Notion Tags field is empty — who curates the ~15–25 situations, and do they get written back to Notion as the canonical tag set, or live only in the skill?
  • Refresh trigger: is sync manual-only, or should the skill detect a stale corpus and nudge on invocation?
