Deepen /artifact into a Document Pipeline

Ideation brief · 2026-06-12 · idea-type: extend · slug deepen-artifact-pipeline

TL;DR

Rather than build a new /doc-pipeline skill, deepen /artifact so one run can take any document — template-backed or proposed-from-scratch — from context to a researched, multi-stakeholder-critiqued, diagrammed, polished, and grilled final. This honours the repo's anti-reimplementation norm: /artifact already owns context-gathering, templating, drafting, and a reviewer loop; the net-new surface is research, multi-persona critique, a from-scratch template path, and orchestration of /diagram + /polish + /grill.

Pressure-test verdict: build — conditionally. Every "always / mandatory" in the spec is a premortem trigger. Make --depth the master dial and convert "always do" into "always offer / run-in-safe-mode."

How might we…

How might we let a PM produce any document — template-backed or from-scratch — that arrives already researched, critiqued through real stakeholder lenses, diagrammed, polished, and stress-tested, in one guided pass instead of hand-chaining five skills?

Target user & JTBD

User: a PM (or eng lead) authoring a high-stakes document — PRD, strategy memo, proposal, RFC, postmortem.

JTBD: "When I have a high-stakes doc to write, I want a workflow that takes me from context → defensible draft → hardened final, so I ship something that survives scrutiny without remembering to run /artifact, then /diagram, then /polish, then /grill in the right order."

Hypothesis

If /artifact absorbs the missing front-end (research), the missing critique depth (multi-persona), the missing from-scratch path (propose+save template), and orchestrates the existing finishing skills, then users will stop manually chaining and the output will survive real stakeholder review with fewer surprises — provided the heavy machinery is earned by stakes (via --depth) rather than imposed on every run.

Idea variants considered

15 shape-variants were generated across five families: orchestrator-shaped (thin conductor, stage-gated, reuse /feature-sdlc), config-driven (template-defines-pipeline, doc-type registry, personas-ride-template), differentiator-first (build only research / persona-sim / from-scratch), effort-scaled (--depth dial, review-room loop), and minimal-footprint (handoff-chain, MVP cut).

Selected: Deepen /artifact (variants #2 + #9 + #15 + #5) — enrich the existing skill, tier-scale the effort, keep the smallest footprint, and add a self-growing from-scratch template path. Rejected a standalone /doc-pipeline orchestrator: a whole new surface to maintain plus skill-coupling/release-cycle cost, when only ~4 capabilities are genuinely new.

Amplify: 11-star ladder

Skipped — the user converged on a concrete design directly; no UX ceiling to raise via amplification.

How it works (the deepened flow)

Layered onto /artifact's existing 8-step Create flow, gated by --depth brief|standard|deep:

  1. Template proposal (when none match). Propose a new template from context; if the template warrants external research, do it alongside eval generation; offer to save as a reusable template (default No, checklist-gated). Ship new-template best-practice guidelines (section design, eval criteria, persona set, length target).
  2. Document-length target. Templates carry a length target; when proposing a new template, ask the user for one.
  3. Research phase (approval-gated). First decide "is external research warranted?" (auto-skip for internal/template-backed docs). If yes: propose a research plan → user approves → fan out subagents → save output as a separate doc under <feature_folder>/research/ with source provenance (verifiable URLs only).
  4. Draft (existing Step 8).
  5. Multi-persona critique. 3–4 stakeholder personas (template-defined, or recommended from context) read the draft in parallel subagents; findings reconciled with the user before the draft updates.
  6. Diagram pass (recommended). Propose a diagram list → user approves → subagents run /diagram (non-interactive, mandated background-rect, fixed insertion anchor) → validate each SVG before inline insert. Remember "always run diagram pass" as a per-project preference.
  7. Polish (always-on). Run /polish at the end — auto-apply mechanical findings, route voice-risk findings through per-finding confirm.
  8. Grill (mandatory, depth-scaled). Run /grill on every artifact, scaled by --depth; interactive interrogation when a human is present, written-findings degradation when headless.

Alternatives & prior art

  • /feature-sdlc — the in-repo precedent for an orchestrator that sequences existing skills without reimplementing them. The rejected "thin /doc-pipeline" variant is structurally this, for docs.
  • /artifact reviewer subagent — already does single-reviewer eval-criteria critique; multi-persona critique must prove it adds value beyond this (see assumptions).
  • /ideate gating — precedent for "most runs don't earn the full battery"; the --depth dial here mirrors it.
  • /primer word-floor learnings — caution that document-length targets are routinely missed; treat the length target as informational steering, not a hard gate.

Munger inversion — choices that would guarantee failure

  • Make the full pipeline mandatory on every run regardless of --depth.
  • Run /grill's turn-by-turn interrogation inline with no non-interactive degradation.
  • Auto-apply /polish rewrites without the per-finding voice-conflict gate.
  • Insert diagram-subagent SVGs inline without validating them.
  • Default "save as template" to Yes, and persist before the doc is finished.

Success signals

  • Users run the deepened /artifact once and stop manually chaining /diagram / /polish / /grill.
  • Default/lite runs stay fast (no regression in time-to-first-draft for quick docs).
  • Artifacts survive real stakeholder review with fewer surprises (persona critique catches what the single reviewer missed).
  • The template library grows in quality, not junk — saved templates pass the best-practice checklist.
