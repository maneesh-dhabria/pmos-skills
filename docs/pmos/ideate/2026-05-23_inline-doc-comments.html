Inline doc comments for HTML pipeline artifacts

/ideate brief · 2026-05-23 · slug inline-doc-comments · idea-type extend

TL;DR

A cross-cutting feature for every pmos-toolkit skill that emits HTML (12+ skills today): the user opens a generated artifact, selects a text span or diagram region, drops a comment thread, and a follow-up Claude invocation (/comments resolve <artifact> or pipeline-aware --resume) consumes open threads — either applying edits and posting a system reply marking the thread resolved, or asking clarification via threaded options.

Three finalists pressure-tested below differ on operating shape: (A) served overlay + Docs-style UX + stable-id JSON sidecar, (B) static-file overlay + Figma-style pins + quote-anchored sidecar, (C) Notion-style inline @claude markers embedded in the artifact. Cross-cutting decision table at § Decision table locks in Finalist A-hybrid as the chosen direction: A's Docs-style UX + stable-id JSON sidecar, with the comment-overlay JS pre-baked into every emitted artifact (B's static-bundle trick) so the user can add comments anytime by just opening the .html file — no serve step. Quote+prefix/suffix anchoring (B's robustness layer) is written on every thread as a mandatory fallback. The single command in the loop is /comments resolve <artifact>, which triggers an agent to read the sidecar + the artifact and resolve each open thread.

How might we

How might we let users annotate any HTML pipeline artifact (text spans + diagram regions) inline, so those annotations drive AI-applied edits with multi-turn threads, system replies, and resolution states — uniformly across every skill that emits HTML?

Target user & JTBD

User: the operator of the pmos-toolkit pipeline (currently: Maneesh, primary user of /requirements, /spec, /plan, /wireframes, /prototype, /artifact, /diagram, etc.) reviewing generated HTML artifacts before merge.

Job: When I spot something wrong in a generated artifact, I want to mark exactly which element is wrong and what should change, so a follow-up Claude session resolves it without me re-supplying location context or re-prompting from scratch.

Hypothesis

If review feedback can be attached to a specific anchor in the artifact (vs. a free-form chat message), the AI's edit precision goes up and the user's framing cost goes down. Anchors are the load-bearing primitive. Multi-turn threads + system replies + resolved state are the lightweight workflow shell that makes the captured anchors auditable across sessions.

Idea variants considered

12 variants generated across three axes; user selected three finalists for pressure-testing.

Axis 1 — persistence & identity model

  1. Sidecar JSON, anchor by stable element id — comments in <artifact>.comments.json; anchors point at <h2 id> / <section id> / explicit [data-anchor]. Re-render preserves ids → threads survive.
  2. Sidecar JSON, anchor by text quote + prefix/suffix (Hypothes.is style) — robust to id churn; fuzzy-match breaks when prose is rewritten.
  3. Embedded in HTML as <aside data-comment-thread> siblings — single-file portability; every re-render must merge.
  4. Git-tracked .comments/ directory — one MD file per thread; diff-able and code-review-able.
  5. SQLite under ~/.pmos/comments.db — central store; survives {docs_path} moves; harder to share.

Axis 2 — capture & AI handoff

  1. In-browser overlay served by /comments serve <artifact> — local static server injects comment UI; persists to sidecar JSON; /comments resolve dispatches open threads to Claude. [Finalist A]
  2. Static HTML enhancement, no server — comment UI is plain JS shipped in _shared/html-authoring/assets/comments.js; uses localStorage + an "export" button that downloads the sidecar JSON the user commits. [Finalist B]
  3. Pure-CLI capture — /comment add <artifact> --anchor <id-or-quote> --body "...".

Axis 3 — interaction model

  1. Google Docs clone — selection → floating "Comment" button → side-panel thread → reply/resolve. [Finalist A]
  2. GitHub PR review clone — comments queued as a "review batch", submitted together; Claude processes the whole batch as one pass.
  3. Notion-style inline @claude mentions — typed directly into the doc as inline markers; comments are first-class content. [Finalist C]
  4. Figma-style numbered pins — pins on text + diagrams; resolved pins persist as ghost markers. [Finalist B]

How it works (per finalist)

Finalist A-hybrid — static-bundled overlay + Docs UX + stable-id JSON sidecar + quote fallback (CHOSEN)

  • Capture (zero commands): every HTML artifact emitted by a pipeline skill ships with a <script src="assets/comments.js"> pre-baked into the rendered HTML (a substrate-level addition to _shared/html-authoring/). User opens the .html file in any browser — file URL or otherwise — selects text → floating "Comment" button → side-panel thread (Google Docs UX). Diagram regions become commentable when the SVG carries data-anchor attributes on shapes.
  • Lightweight state: comments.js auto-persists to a sidecar <artifact>.comments.json next to the file. Schema: { threads: [ { id, anchor: {kind: "id"|"data-anchor"|"quote", ref, quote?, prefix?, suffix?}, status: "open"|"in_progress"|"resolved", messages: [ {role: "user"|"system", body, ts} ] } ] }. Every thread writes both a stable-id anchor AND a quote+prefix/suffix anchor — resolve prefers id, falls back to quote, surfaces orphan otherwise. Writes go through the File System Access API where available; otherwise the bundle prompts a one-click "save sidecar" (or falls back to localStorage + an export button as a last resort for restricted browsers).
  • Resolve (one command): /comments resolve <artifact> reads the sidecar + the artifact, identifies the originating skill via <meta name="pmos:skill">, and for each open thread either (a) applies the edit, appends a system reply summarising what changed + diff pointer, flips status: resolved, OR (b) appends a system reply containing 2–4 clarification options for the user to pick by replying in the thread on next open. Threads in clarification state stay in_progress until the user replies.
  • Re-render survival: stable section/heading ids are already mandated by _shared/html-authoring/conventions.md §3; quote fallback covers prose churn. Orphaned threads surface a banner in the overlay UI on next open.
  • Stakeholder-share posture: the comment overlay is gated behind a "?review=1" URL param or a body-class toggle, so the artifact shared with stakeholders renders clean by default; reviewers append the param to enable commenting.

Finalist B — static-file overlay + Figma pins + quote-anchored sidecar

  • Capture: the existing viewer.js bundle ships a comments.js sibling that activates with no server. Click anywhere → drops a numbered pin; selection-then-pin attaches a quote anchor; pins are visible badges over text + diagrams.
  • Persistence: localStorage for in-progress drafts; an "Export comments" button downloads <artifact>.comments.json for the user to commit. Resolved pins become ghost (greyed) badges still visible in the rendered doc.
  • Anchor strategy: quote + ±32-char prefix/suffix (Hypothes.is W3C Web Annotations fuzzy-match); falls back to "orphan" status when the quote no longer matches, surfaced as a banner.
  • Resolve: same /comments resolve <artifact> but the JSON is the canonical input — no server needed for the AI pass.

Finalist C — Notion-style inline @claude markers

  • Capture: the rendered HTML has a "review mode" toggle that switches the body to contenteditable; the user types @claude <instruction> inline next to the offending content. Markers render as styled inline pills.
  • Persistence: markers are embedded as <mark data-claude-instruction="…" data-thread-id="…"> in the artifact itself. No sidecar. Multi-turn happens via Claude appending sibling <mark role="system"> nodes inline.
  • Resolve: /comments resolve walks the DOM for [data-claude-instruction], applies edits, replaces the marker with a system-reply marker or removes it on resolution.
  • Trade: lowest-ceremony capture for prose; weak for diagrams (you can't type inside an SVG); muddles authored content vs. annotations in the file the user reads.

Alternatives & prior art

  • Hypothes.is — open-source web annotations; canonical reference for quote + prefix/suffix anchoring (W3C Web Annotations Data Model). Direct steal target for Finalist B.
  • Google Docs comments — selection → floating button → side panel → reply/resolve. UX template for Finalist A.
  • GitHub PR inline review — batched comments, single-shot AI resolution. Considered as variant #10 but rejected: pipeline artifacts are not PRs and batching delays feedback loops.
  • Figma comment pins — numbered, persistent-after-resolve. UX template for Finalist B's diagram handling.
  • Notion inline mentions — comments-as-content. UX template for Finalist C.
  • VS Code / Cursor inline-chat — closest existing pattern in the AI-coding space; differs in that it operates on source files, not rendered HTML.
  • Internal prior art — the existing viewer.js chrome-strip + serve.js substrate in _shared/html-authoring/ is the foundation; any finalist extends it rather than replacing it.

Munger inversion — how to guarantee failure

Per finalist: choices that, if made, would guarantee this idea fails.

Finalist A

  • Ship without quote-anchor fallback — every /spec --refine orphans every thread.
  • Make /comments serve the only entry point and require it to be running for /comments resolve to work (couples capture and resolve unnecessarily).
  • Skip the pre-commit sidecar-sync hook — sidecar drifts; users blame the feature when threads point at stale content.
  • Build it as a single skill instead of substrate — every HTML-emitting skill has to be updated by hand, inconsistencies proliferate, retrofitting is abandoned at skill #5.

Finalist B

  • Trust quote-anchor fuzzy match silently — comments re-attach to wrong spans, user loses trust within a week.
  • Rely on user-clicked "Export" — drafts get lost; users learn to distrust the feature.
  • Skip the "orphan" surfacing UI — users don't know their comments stopped pointing anywhere.

Finalist C

  • Persist threads inline only — sharing the artifact requires a "strip markers" pass that may forget some.
  • Treat diagrams as out-of-scope — the original requirement explicitly included diagrams.
  • Ship without restricted-contenteditable enforcement — users accidentally rewrite their own spec while trying to comment.

Chosen direction: Finalist A-hybrid — static-bundled comment overlay shipped with every emitted artifact (zero commands to capture), JSON sidecar persistence with dual-anchor (stable-id + quote/prefix/suffix), and a single /comments resolve <artifact> command that triggers an agent to read the sidecar + the artifact and resolve each open thread (either applying edits with a system-reply summary, or posting a system-reply with 2–4 clarification options). Finalist C rejected; A-strict dominated by A-hybrid.

Success signals

  • 30 days post-ship: ≥10 threads opened across ≥3 distinct HTML-emitting skills; ≥80% of opened threads reach resolved status (vs. abandoned).
  • Anchor robustness: ≤5% orphan rate across all threads (target: stable-id resolves before quote-fallback in ≥90% of cases).
  • Latency: /comments resolve end-to-end (read sidecar → dispatch → apply edits → write system reply) under 60s for single-thread batches.
  • Friction proxy: the user starts using /comments serve as the default review surface for /spec, /wireframes, /artifact outputs — replacing free-form chat feedback for ≥50% of revision rounds.
  • Negative signal (would-trigger-rethink): users open threads then resolve via free-form chat instead of /comments resolve — means the AI handoff is not landing.

Open questions

  • Q1: Should /comments resolve be a standalone skill, or a flag on each originating skill (e.g. /spec --resolve-comments)? Standalone is cleaner; per-skill leverages each skill's existing context-loading. Lean: standalone, with the skill dispatching internally.
  • Q2: When a thread's anchor orphans, should resolve auto-skip it (and surface in the system reply), or block the whole batch until the user re-anchors? Lean: auto-skip with summary.
  • Q3: Multi-author future-proofing — should thread messages carry an author field now, even though the immediate use is solo? Lean: yes, cheap optionality.
  • Q4: Markdown sidecars (output_format=both) — do comments apply to the .md twin, or only to the HTML primary? Lean: HTML-primary only; .md regenerates from HTML on resolve.
  • Q5: Does this need its own plugin (separate from pmos-toolkit)? Lean: no — it's substrate-shaped, lives under _shared/comments/ and ships with the toolkit.
  • Q6: Should the comment-overlay JS bundle be vendored (like turndown.umd.js) or built from a tiny purpose-built script? Lean: purpose-built — comment UIs are simple; vendored libraries (e.g. Annotator.js) are heavy and over-shaped for this.
