Integrating /architecture into /feature-sdlc

Ideate · 2026-05-28 · slug architecture-in-feature-sdlc

TL;DR

Wire /architecture into /feature-sdlc in two slots, but ship a new --from-spec mode in the same release so the pre-spec slot has something to actually run. Today, /architecture is a static-analysis tool over a filesystem of code — it has no signal when pointed at /spec prose. Without the new mode, the pre-spec slot is theatre.

  • Slot 1 — Phase 4.5 (after /spec, before /plan): runs /architecture --from-spec <spec-path>. LLM judge reads the principles document + the spec; renders findings.
  • Slot 2 — Phase 7 sub-step (inside /verify): existing static-analysis tools (dependency-cruiser, ruff, AST harness) run against changed modules and produce mechanical evidence; the same LLM judge reads principles + code + that evidence and renders findings.
  • Tier-gated from day 1: T1 (bug fix) skips both. T2 (enhancement) runs post-impl only. T3 (feature) runs both.
  • One principles document, one judge. No dual-implementation rule set. Static tools are evidence inputs, not parallel evaluators. Same judge prompt template, different inputs for spec vs. code review.

How might we…

How might we wire /architecture into /feature-sdlc so it informs design and catches regressions, without ballooning pipeline time on small features?

Target user & JTBD

User: The repo maintainer (Maneesh) running /feature-sdlc on features in pmos-toolkit and adjacent repos.

Job: When I drive a feature end-to-end, surface architecture violations at the moments where I can still cheaply act on them — at spec time (cheap to re-design) and at verify time (cheap to add a cleanup task before merge) — without paying the cost on bug fixes or small enhancements that don't introduce structural change.

Hypothesis

Two-slot integration (pre-spec LLM-judge + post-impl static-analysis), sharing one rule definition, will catch ≥1 architectural issue per Tier 3 feature that the current pipeline misses, with <90s added latency in Phase 4.5 and zero added cost on Tier 1.

Idea variants considered

12 variants generated in Phase 2, grouped by placement count:

Single-placement

  1. Phase 4 (/spec) pre-step — audit current state, feed findings as constraints.
  2. Phase 5 (/plan) sub-step — surface concerns when tasks are being defined.
  3. Phase 7 (/verify) only — post-impl regression check.
  4. Phase 8a (/reflect) only — post-merge audit, log to learnings.
  5. Phase 0 baseline only — snapshot for the record.
  6. Phase 4.5 between /spec and /plan — validate proposed design.
  7. Standalone, handoff suggestion only.

Two-placement (bookend)

  1. Phase 4 + Phase 7 — design + verify.
  2. Phase 0 + Phase 7 — baseline + diff.
  3. Phase 4 + Phase 8a — both soft, never blocks.
  4. Phase 5 + Phase 7 — plan-tasks include arch fixes, verify they landed.

Tier-gated

  1. Tier-3 only, Phase 4 + Phase 7 — bookend, but only at full tier.

The user pushed back on the pre-spec slot, asking the right question: "does /architecture only audit code, or can it audit the proposed architecture from /spec?" That collapsed variants 1, 6, 8, 10, 12 into a single dependency: they require a new --from-spec mode to exist. Otherwise they're no-ops on prose. Decision: ship the new mode in the same release rather than as a backlog item.

How it works

Foundation: one principles document, one LLM judge

  • Principles document — prose file capturing L1 (universal, ≤15 rules) + L2 (stack-specific TS/Python) + L3 (per-repo overrides). Lives at plugins/pmos-toolkit/skills/architecture/principles.md (or similar). Single source of truth.
  • Static-analysis tools (dependency-cruiser, ruff, the Python AST harness) are evidence-gathering tools, not parallel evaluators. They detect mechanical violations (cycles, fan-in/out, unused exports) and emit structured findings.
  • One LLM judge subagent, parameterized by input artifact type. Reads: (a) principles doc, (b) the artifact under review, (c) static-tool output when reviewing code. Renders findings citing specific principles.

New invocation: /architecture --from-spec <spec-path>

  • Reads the spec HTML/MD; extracts the proposed module list, dependencies, and architectural assertions.
  • Dispatches the judge subagent with principles + spec as inputs. No static tools run (no code to scan).
  • Emits the same HTML+MD+JSON triplet shape as code-audit mode, written under {docs_path}/architecture/{date}__from-spec.{html,md,json}.
  • Fails loud if the spec lacks module/dependency structure (no silent no-op).

Existing code-audit mode (unchanged interface)

  • Static tools scan changed modules → mechanical evidence.
  • Same judge subagent runs with principles + code + static-tool output as inputs → findings.
  • Today the static-tool output is the report. The change is wrapping it in the judge pass so the same evaluator handles both modes.

Slot 1 — /feature-sdlc Phase 4.5 (between /spec and /plan)

  • Soft gate (Recommended-Skip on T2, Recommended-Run on T3, suppressed on T1).
  • Runs /architecture --from-spec against the just-emitted spec.
  • Findings surface in chat; user decides whether to revise spec or proceed.
  • State key: phase_4_5_arch_spec_findings_path in state schema v5.

Slot 2 — /feature-sdlc Phase 7 (inside /verify)

  • Existing static-analysis /architecture mode.
  • Scoped with --since <merge-base> so only changed modules are audited (avoids whole-repo noise).
  • Findings rolled into /verify's existing report alongside lint/test/multi-agent code-review output.
  • State key: phase_7_arch_code_findings_path.

Alternatives & prior art

  • Single-slot post-impl only (variant #3). Cheaper to build (no new mode), narrower value. Rejected because it leaves the spec-design phase blind to architectural constraints — exactly the slot where it's cheapest to act.
  • Standalone-only, handoff suggestion (variant #7). Lowest cost, lowest signal — users don't run optional tools. Rejected.
  • Bookend at all tiers (variant #8). Pays the cost on bug fixes that don't need it. Rejected via tier-gating.
  • Phase 5 (/plan) integration (variant #2). Considered; /plan already includes deep code study, and surfacing arch concerns there is too late — the spec is already committed.
  • Prior art: /simulate-spec is the existing optional gate between /spec and /plan. New Phase 4.5 should mirror its shape (soft gate, separate artifact, opt-in confirmation).

One assumption sits at the heart of this release — the LLM judge reliably applies prose principles to spec prose. Testable cheaply before committing to integration: run the prototype against 5 historical specs, measure precision/recall. (The original design had a second H/H risk — keeping static and LLM rule forms aligned — that's been eliminated by collapsing to a single principles document + single judge.)

Success signals

  • ≥1 architectural issue caught per Tier 3 feature that the prior pipeline missed (measured over 5 features post-launch).
  • <90s added latency in Phase 4.5 (P50).
  • Zero added latency on Tier 1 runs.
  • False-positive rate on --from-spec below 20% (measured by user accept/reject ratio on findings).
  • Principles document remains the single source of truth — no rule appears in code that isn't traceable to a principle in the doc.

Open questions

  • Should Phase 4.5 emit a separate artifact in {docs_path}/features/<feature>/ or fold into the existing spec file as a sibling section?
  • Where does the principles document live? plugins/pmos-toolkit/skills/architecture/principles.md (versioned with the skill) vs. {repo}/.pmos/architecture/principles.md (per-repo override, layered on top of skill defaults). Likely both — L1/L2 in the skill, L3 in the repo.
  • Does the judge subagent need a separate prompt template per input type (spec vs. code), or one template that branches on input?
  • If a Phase 4.5 finding causes the user to revise the spec, does the pipeline re-run --from-spec automatically, or wait for explicit re-invocation?
  • Tier-gating shape — is "T2 post-only" actually right, or should T2 also get pre-spec when the spec introduces a new module?
