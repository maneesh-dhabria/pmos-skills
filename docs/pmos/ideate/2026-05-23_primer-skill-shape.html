/primer skill shape for pmos-learnkit

Created using pmos‑toolkit

TL;DR

Ship /primer as a lean 5-phase pipeline (intake → research → outline → draft → eval) inside a new pmos-learnkit plugin, reusing the pmos-toolkit _shared/ substrate, with no source-eval registry in v1, interactive-by-default, and a --depth tier flag. Defer the curator-lens machinery from /rewrite (author YAML registry, diagram-pass, persona-pass, pinned-tag resume) until v1 usage proves they're needed.

Recommendation: Proceed — but cut the --depth tier from v1 too (ship mid only), so the v1 surface is one mode, one path, one output. The biggest risk is shipping nothing because v1 grew back into /rewrite.

How Might We

How might we ship a /primer skill that produces a deep, source-grounded PM-topic primer with materially less machinery than the existing /rewrite pipeline, while still earning a credible "curator-lens" quality bar — and do it inside a new pmos-learnkit plugin that fits naturally next to the pmos-toolkit pipeline skills?

Target user & job-to-be-done

Who: Maneesh (the author) and PM-adjacent practitioners who want to ramp on a topic they don't yet own.

When / where: Before joining a meeting / scoping a feature / reviewing a doc on an unfamiliar topic; inside Claude Code with the pmos-skills marketplace installed.

JTBD: When I want to ramp on a PM topic I don't yet own, I want a primer that synthesizes vetted sources into a teachable artifact, so I can move from "I've heard of this" to "I can lead a conversation about it" in one read.

Hypothesis

We believe shipping a 5-phase, single-file, substrate-reusing /primer skill without the source-eval registry will produce a primer that's 70% of /rewrite's quality at <30% of its build cost and surface area because (a) the load-bearing quality comes from the outline-principles + writing-style reference docs, not the YAML author registry; (b) prompt-level curator-lens framing captures most of the trust signal; (c) the pmos-toolkit _shared/ substrate (html-authoring, interactive-prompts, pipeline-setup, non-interactive) eliminates 60%+ of the orchestrator boilerplate; and (d) interactive-by-default catches hallucinated facts at the outline gate before drafting amplifies them.

Idea variants considered

Generated via First Principles + SCAMPER (Eliminate / Combine emphasis). Finalist composition flagged with ★.

  1. Single-file minimalist — one SKILL.md, three inline phases (research → outline+draft one-shot → eval). No subagents. Closest to /diagram / /ideate shape.
  2. ★ /rewrite-lite (5 phases) — intake → research → outline → draft → eval. Drop blind-primer, assembly-persona, diagram-pass, learnings-as-phase.
  3. Three-subagent fan-out — single SKILL.md dispatches 3 parallel subagents (source-hunter, outliner, drafter); controller stitches.
  4. Reviewer-loop, no standalone eval — fold eval into one reviewer-subagent + auto-apply iteration (max 2 loops), matching /polish / /wireframes.
  5. Topic-first only — /primer <topic> [--audience X]. No URL / source-grounded mode.
  6. Topic-or-URL auto-detect — one surface, two entry shapes.
  7. Topic + audience + "what I already know" — three-input intake; the third input lets the primer skip basics.
  8. Bring-your-own-sources — --sources <urls,…> seeds research; skill still discovers more but anchors on seeds.
  9. ★ Depth-tier flag — --depth shallow|mid|deep (1k / 3k / 6k words). One skill, three doses.
  10. Topic-tree (recursive) — parent + 4-6 sub-topics; user picks which sub-primers to also expand.
  11. ★ Reuse pmos-toolkit _shared/ substrate — no port of pmos-content's _pipeline/; lean on html-authoring, interactive-prompts, pipeline-setup, non-interactive that already exist.
  12. HTML-primary output — primer is an HTML artifact at {docs_path}/primer/<date>_<slug>.html; markdown sidecar via output_format=both. (Implicit in #11 substrate choice.)
  13. Source-eval as a separate skill — extract author-trust YAML registry as its own /source-eval in pmos-learnkit; /primer reads from it but doesn't manage it.
  14. ★ No source-eval in v1 — strip curator-positioning + author-trust machinery entirely; ship "topic → research → draft" without an opinion about author quality. Earn the registry in v2.
  15. ★ Interactive-by-default, --autonomous flag — invert /rewrite's default; for a learning artifact the user reads cover-to-cover, gating is the value.

Composite finalist (★): #2 + #9 + #11 + #14 + #15 — a lean rewrite-style pipeline, native to pmos-skills substrate, depth-configurable, no source-eval burden, interactive-by-default.

How it works (chosen finalist)

  • Plugin home: plugins/pmos-learnkit/ in the pmos-skills marketplace repo, registered in both .claude-plugin/marketplace.json and .codex-plugin/marketplace.json (per the four-file version-sync invariant in CLAUDE.md).
  • Skill path: plugins/pmos-learnkit/skills/primer/SKILL.md (lowercase-hyphenated, matches frontmatter name: primer — the canonical skill-path invariant).
  • Invocation: /primer <topic> [--audience <senior-pms|all-pms|exec>] [--depth shallow|mid|deep] [--autonomous]. Interactive-by-default; gates at outline (Phase 3) and eval (Phase 5). One-shot consolidated AskUserQuestion intake when fields are unspecified.
  • Phase 1 — Intake: resolve topic, audience, depth, output_format from .pmos/settings.yaml. Single consolidated AskUserQuestion for any missing fields. Derive slug + path: {docs_path}/primer/{YYYY-MM-DD}_<slug>.html.
  • Phase 2 — Research: WebFetch on 6–10 candidate sources (not WebSearch alone; verify each URL resolves before quoting); context7 for any library-style docs; pull a "scope of the topic" outline. Persist a sources.json companion with URL + 1-line takeaway each. Prompt-level curator-lens framing replaces the YAML registry in v1.
  • Phase 3 — Outline: render a section-by-section outline (10–14 H2s for --depth mid); interactive gate by default — user can edit / cut / reorder. Re-prompts allowed.
  • Phase 4 — Draft: single-pass drafting against the approved outline, inlining citations to sources.json URLs. ~3000 words for --depth mid.
  • Phase 5 — Eval: reviewer subagent reads draft against a 10-check binary rubric (covers ground / cites real URLs / no hand-wavy claims / right reading level / structurally complete / etc.); auto-apply fixes, max 1 iteration. Interactive gate by default to confirm before write.
  • Output: HTML primary at {docs_path}/primer/<date>_<slug>.html with sections.json companion, MD sidecar via output_format=both. Asset substrate copied from _shared/html-authoring/assets/. Polish-compatible — /polish <path> works out of the box.
  • What's NOT in v1: source-eval YAML registry, diagram-pass, persona-pass, pinned-tag resume, autonomous-decisions log, blind-primer, learnings-as-phase, worked-examples phase, topic-tree, BYO-sources flag. All deferred to v2+ based on real usage signal.

Success signals

Leading: (a) A1 side-by-side rates the no-registry primer at ≥70% of /rewrite output on 2 known topics. (b) A complete v1 SKILL.md fits in <500 lines (vs /rewrite's ~120 lines orchestrator + ~9 phase files totalling thousands). (c) First end-to-end /primer run completes in <6 minutes wall-clock for --depth mid. (d) Reviewer subagent catches ≥1 bad citation or hand-wavy claim per run on average.

Lagging: (e) You run /primer a second, then third time without grimacing at the cost. (f) At least 1 generated primer is shared externally (Notion / link / DM) within 2 weeks of v1. (g) At least 1 follow-up pmos-learnkit skill (/flashcards, /quiz, /source-eval) becomes the natural next request — meaning /primer created the appetite for the rest of the kit.
