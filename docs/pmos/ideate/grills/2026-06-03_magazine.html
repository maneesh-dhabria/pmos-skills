Summary

Target: docs/pmos/ideate/2026-06-03_magazine.html · Depth: deep · Questions asked: 11 · Author overrides of the recommended option: 0 (but three answers materially reframed the design beyond the offered options).

The grill shifted /magazine from a "fast, lightweight, on-demand digest skill" into a deterministic, resumable, parallelized local pipeline with an in-session agent curation/summarization phase. Two threads did the heavy lifting: (1) the execution-architecture fork (who summarizes, and which phases can run headless) and (2) the "full body" source-of-truth question, which became "always crawl the article via its link." A third reframe — making Top picks agent-curated rather than engagement-derived — dissolved a latent file://-write-back contradiction in the chosen 7★ scope.

Resolved (supersedes the brief)

#BranchResolution
1Execution architectureHybrid. Deterministic crawler builds the work-list; one subagent per post fans out summarization. Not agent-does-all; not a pure API script.
2Speed criterion"Fast weekly run" dropped. It's a deterministic, resumable, parallelized local pipeline; the issue is rendered incrementally as items complete.
3Process boundaryHeadless + resumable covers fetch → audio-download → whisper-transcribe. Summarize + tag + render runs in-session via subagents.
4Source of truthAlways follow each item's <link> and readability-extract the full article; RSS-embedded body is fallback-only (crawl failure). Paywalled pages → preview-only flag.
5Top picks mechanismAgent-curated, not engagement-derived. A subagent ranks items vs. declared interests at render time → zero browser write-back; "no server" holds. User-starred "to-revisit" shelf → deferred to v2.
6Run / issue / resume unitOne issue per invocation; item set snapshotted at start; per-item lifecycle in state; "since last run" cursor advances only on full completion (resume never drops/double-counts).
7Audio / transcript lifecycleCache transcripts forever by GUID (resume + re-summarize never re-transcribe); delete audio immediately after a successful transcript.
8Plugin homepmos-learnkit (beside /primer, /learn-list, /critical-thinking).
9Interest signal seedingFirst-run interview captures declared topics/feeds; empty/sparse → curation falls back to item-intrinsic importance so Top-picks is never random.
10Per-item failure handlingRender a degraded, flagged card (reason badge: paywalled / transcript-failed / not-summarized); nothing silently dropped; failures logged for retry.
11Cost postureUnconstrained — quality over cost. Strongest model, full context; no cost-driven downgrade/chunking. (--max-per-feed kept as a sanity knob.)

Gaps surfaced (push to /requirements or /plan)

  • Richer state schema than the brief's "timestamp + seen-set" — needs a per-item lifecycle (discovered → downloaded → transcribed → summarized → rendered → failed:<reason>). Specify explicitly in the spec.
  • Incremental render on file:// won't auto-refresh an already-open issue (manual reload). Decide if acceptable or add a meta-refresh while a run is active.
  • Cross-feed cross-posting (same article in two feeds = two GUIDs) isn't deduped by GUID alone — consider URL-canonicalization dedup.
  • OPML bulk import for initial feed setup (still open in the brief).
  • Whisper model choice (base/small/medium) + long-episode time cap (still open).
  • Crawler politeness: always-crawl means more outbound requests — needs rate-limiting + per-GUID crawl cache to avoid re-fetching.
