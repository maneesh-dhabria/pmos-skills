/playbook

ideation · new skill · pmos-learnkit · grilled (deep) · 2026-06-03 Created using pmos‑toolkit

TL;DR

/playbook [--repo <path>] [--days N | --sessions N | --since <date>] [--include-headless] is a repo-scoped skill (home: pmos-learnkit) that mines your own Claude Code session history and the repo's generated docs to produce focused, self-sufficient case-study articles that teach fellow PMs exactly how you used AI to solve a specific problem. It resolves all session dirs for a repo via a multi-signal worktree resolver, filters to interactive sessions by default (dropping Claude-as-subprocess/headless runs), clusters the remaining sessions into problem threads (the unit of a playbook), does a cheap deterministic scouting pass, and proposes a ranked top-N of candidate playbooks (one per coherent thread — not one generalized playbook). On the ones you pick it deep-reads only that cluster and writes a per-playbook folder containing an HTML article + a tweet / tweet-thread + screenshots. Each article captures your starting prompts, how ideas were refined, research done, load-bearing decisions & trade-offs (extracted prose-first from the thread, with AskUserQuestion blocks as structured anchors), a steal-these-prompts gallery, and a skills-used provenance strip. A detect-and-flag anonymizer + an always-on human review gate (REVIEW-BEFORE-SHARING.md) precede any sharing; the skill never auto-marks "safe."

Recommendation: GO — pressure-tested via a deep grill (see grill report; 12 decisions resolved). Two commitments are load-bearing: (1) the resolver must be multi-signal (nested-prefix ∪ sibling-token-strip ∪ gitBranch∈merge-history) — pm-docs's main dir has 2 sessions while the real work (345 artifacts) lived in worktrees, and merged-and-deleted sibling worktrees are only recoverable via merge history; (2) interactive-only filtering via the permission-mode record discriminator — it collapses book-companion from 3,839 inflated headless sessions to a handful of real ones and is what makes the scale tractable.

How Might We

How might we turn a PM's buried, worktree-scattered Claude Code session history into short, focused, self-sufficient case studies — one per real problem they solved — that show other PMs concretely how to apply AI to their own day-to-day work?

Target user & job-to-be-done

Who (author): An AI-native PM who has solved real problems across many repos/sessions and wants to teach peers — and build a reputation — without manually trawling logs or re-narrating from memory.

Who (reader): A fellow PM, often non-coding, who wants concrete, replicable examples of AI applied to PM work (research, concept notes, specs, surveys, features) — not abstract "how I use AI" advice.

JTBD: "When I've spent weeks solving real problems with AI inside a repo, give me a few focused, shareable articles — each anchored to one specific problem, showing the prompts I started with, how I refined, the trade-offs I decided, and the skills I used — so a peer can read one and replicate the approach on their own problem."

Hypothesis

We believe that mining a single repo's full session history (worktrees included) + its generated docs, and synthesizing focused per-problem articles that surface the actual prompts, refinements, and load-bearing decisions will produce case studies a peer PM can replicate on their own work because the teachable value isn't a generalized playbook — it's the concrete decision trail and real prompts behind one specific problem, which neither memory-based recall nor a generic "AI tips" post can reconstruct.

Idea variants considered

Generated via First Principles + Analogous Inspiration. Finalist ingredients flagged with ★. The composite was reshaped by user feedback away from a generalized cross-repo playbook toward focused, per-repo/per-folder articles.

  1. ★ Index-first scout (ephemeral) — read only session titles + first prompts + command-tags + repo folder structure to propose candidates; deep-read only what's picked. spine (index is internal/ephemeral, not persisted — its only benefit is "propose well + don't burn tokens").
  2. Cross-repo theme clustering — group by problem theme across repos. (Rejected — user wants focused per-repo articles, not generalized themes.)
  3. ★ Repo-scoped, folder-aware candidates — run from a repo; large/multi-project repos propose multiple playbooks, one per coherent folder/project/research (e.g. each pm-docs folder → a candidate). spine
  4. ★ Worktree-complete session resolution — multi-signal: nested-prefix ∪ sibling-token-strip ∪ gitBranch∈merge-history (recovers merged-and-deleted siblings). load-bearing
  5. ★ Interactive-only filter (grilled-in) — drop Claude-as-subprocess/headless runs via the permission-mode record discriminator; --include-headless escape. spine · scale
  6. ★ Problem-thread clustering (grilled-in) — the playbook unit is a session-thread (branch-then-topic), not a folder. Folders/docs are supporting evidence. anchor
  7. ★ Decision-ledger extraction — mine AskUserQuestion proposals + your responses + your pushbacks into a "trade-offs I decided" table. The highest-value payload for PMs.
  8. ★ Sample-prompt gallery — the actual starting prompts + refinement prompts + skill guidance you received, as a "steal these" box.
  9. ★ Skills-used provenance strip — which skills/tools, in what order, with one-line "why this skill here," from command-name tags.
  10. ★ Generated-docs mining — mine the repo's pmos-generated docs/pmos/** artifacts (and other docs) for interesting load-bearing choices/snippets, beyond the session logs.
  11. ★ Additive scope window — --repo <path> combined with --days N / --sessions N / --since <date>; dry-run count before any heavy read.
  12. ★ Ranked top-N candidate proposal — score candidates by instructiveness (decision density, # skills, session span, struggle→resolution arc); propose ranked top ~5 with "why teachable," user picks. (User-chosen default.)
  13. ★ Per-playbook output folder — each playbook → {docs_path}/playbooks/{YYYY-MM-DD}_<slug>/ with HTML article + tweet/tweet-thread.
  14. ★ Always-on anonymizer + review file — scrub repo/client names; emit REVIEW-BEFORE-SHARING.md listing redactions + remaining proper nouns for manual sign-off. Never auto-marks "safe."
  15. For-non-coding-PMs translation framing — render engineering specifics into PM-language takeaways. folded into article voice

How it works (chosen finalist)

A repo-scoped, propose-then-deep-read pipeline. Scope is additive; the scouting pass is a deterministic script (the LLM never ingests raw logs at scout time). Design grilled deep — see the grill report.

  • 0 · Resolve scope (multi-signal worktree resolver) — default --repo = cwd; combine additively with --days N / --sessions N / --since <date> (default when none given: last 30 days). Session logs carry cwd + gitBranch per record. Attribute a session dir to the repo when ANY signal fires: (a) nested-prefix — decoded cwd starts with repo + "/.claude/worktrees/" (pm-docs style); (b) sibling token-strip — cwd is a sibling dir <repo>-<slug> whose name, with trailing slug tokens stripped, resolves to this repo (poker-coach style); (c) branch-in-merge-history — the session's gitBranch appears in the repo's git log --merges / reflog (recovers worktrees already merged AND deleted from disk). Confirm ambiguous attributions before any heavy read.
  • 1 · Filter to interactive sessions (default) — drop Claude-as-subprocess/headless runs using the permission-mode record-type discriminator (interactive TUI emits it; SDK/claude -p runs don't), corroborated by secondary signals (very short + templated system-style opening + no command-name). --include-headless re-admits them. This is what makes scale tractable (book-companion 3,839 → handful).
  • 2 · Deterministic script scout — a script (not the LLM) reads cheap fields (ai-title + first user prompt + command-name tags + gitBranch + mtime) across kept sessions, clusters into problem threads (branch-then-topic: group by non-HEAD branch, else topic-similarity + temporal gap; merge compaction continuations), enumerates the repo's folders + generated docs, scores each thread, and emits a compact ranked-candidate summary. Print found N dirs (M via worktree/sibling/merged), K interactive sessions (H headless dropped), C candidate threads — a dry-run confirm before deep-read.
  • 3 · Propose ranked candidates — present ranked top ~5 threads with a one-line "why this is teachable," gated by the instructiveness floor (≥1 non-trivial decision AND ≥1 skill/tool beyond /exit — suppresses thin candidates like a 1-session nidhi). User picks which to build, and can merge/split.
  • 4 · Deep-read picked clusters (subagent fan-out) — one subagent per chosen playbook, strict output contract; reads the full sessions + relevant docs only for that cluster. Extracts prose-first: the starting prompt (first genuine human message of the earliest session, skipping compaction summaries + command-name wrappers), refinement arc, research done, load-bearing decisions + your pushbacks/redirects (semantic extraction, with AskUserQuestion blocks as high-confidence structured anchors), skills-used sequence, interesting doc snippets.
  • 5 · Synthesize the article (self-sufficient) — focused case study embedding everything inline so the reader opens zero repo files: problem → how you framed it → verbatim starting prompts (steal-these gallery) → refinement → research → decision-ledger table (choice · alternatives weighed · rationale) → text artifact excerpts → screenshots of visual artifacts (Playwright capture) → skills-used provenance strip → PM-language takeaway.
  • 6 · Anonymizer detect + human gate — best-effort detection (NER, known repo/branch names, email/$/metric patterns, configurable ~/.pmos/playbook/sensitive.yaml denylist); flag every retained proper noun + every quantitative claim in REVIEW-BEFORE-SHARING.md, and list every captured screenshot for mandatory manual verification. The skill never auto-marks "safe" — the author is the gate (hard, un-skippable).
  • 7 · Emit per-playbook folder — {docs_path}/playbooks/{YYYY-MM-DD}_<slug>/ containing index.html (article), screenshots, the tweet / tweet-thread, and REVIEW-BEFORE-SHARING.md. Existing-folder prompt (overwrite / suffix / cancel).

Success signals

Leading: A focused article is produced in <10 min from /playbook --days 30 without hand-curating logs; each post carries ≥3 real prompts + ≥1 decision table; worktree coverage line shows it found the scattered sessions, not just the 2 in the repo dir.

Lagging: A peer PM reads one post and replicates the approach on their own problem; you publish ≥3 playbooks from one repo run; /playbook becomes your default "write it up" step after finishing a body of work.
