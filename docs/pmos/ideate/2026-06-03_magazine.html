TL;DR

⟳ Updated 2026-06-03 after a deep /grill pass — see grill report. The architecture below supersedes the original "fast lightweight utility" framing.

/magazine turns your scattered RSS subscriptions (newsletters + podcasts) into a self-contained, filterable HTML digest, built by a deterministic, resumable, parallelized local pipeline. Sources live in ~/.pmos/magazine/feeds.yaml (managed via /magazine add|remove|list); each invocation opens one issue for its window (since last run; --days N / --feed <name> scope it) and fills it incrementally as items process. A background-capable stage does fetch → audio-download → whisper-transcribe (resumable); an in-session stage fans out one subagent per post to summarize + tag + render. Bodies are always crawled from each item's <link> (not the RSS stub). Each item → 3–5 bullet takeaways + a read/listen link, auto-tagged from a closed central tag registry; the issue opens with an agent-curated "Top picks" lane. One HTML file (card grid; filters: feed multi-select + date range + tag chips; works from file://, no server) + a cross-issue searchable library index.html.

Recommendation: GO — v1 = public feeds only, in pmos-learnkit. Speed is explicitly not a goal: the pipeline can run long, resume, and update the issue as topics complete; cost is unconstrained (quality over cost). The remaining risks are source data quality (always-crawl + whisper mitigate them) — a 10-minute pre-build fetch test settles them. The one irony to accept: v1 public-only leaves Lenny's paid posts out, and even his podcast needs the deferred v2 auth-feed support.

How Might We

How might we turn my scattered RSS subscriptions — newsletters and podcasts — into one skimmable, filterable HTML digest of what's new since last time, so I stay current without opening 20 apps, and actually keep doing it weekly?

Target user & job-to-be-done

Who: Me — an operator/PM subscribed to a dozen-plus newsletters and podcasts across RSS, drowning in a cross-app backlog.

When / where: Weekly (or whenever the backlog piles up), at the point where opening Substack + a podcast app + an RSS reader separately is enough friction that I just… don't, and fall behind.

JTBD: "When my cross-feed backlog piles up, give me a concise, tagged, link-out summary of what's new since last time — so I can decide in minutes what's worth my full attention, and feel caught up without the app-hopping tax."

Hypothesis

If a single command produces a trustworthy 3–5-bullet summary per item with a one-click path to the full thing — windowed to only what's new, tagged for filtering, and saved as a durable HTML issue — then I'll triage my subscriptions in minutes instead of hours, and the habit will actually stick (re-run weekly) because the per-run cost is near-zero and the output compounds into a searchable library.

Load-bearing belief: the value is trust + windowing, not raw aggregation. An RSS reader already aggregates; what it doesn't do is tell me what's worth my time since last time in language I trust.

How it works

⟳ This section was rewritten after the /grill pass. The design is a two-stage pipeline: a headless, resumable, deterministic prep stage feeds an in-session agent stage.

Config & state (under ~/.pmos/magazine/)

  • feeds.yaml — each feed: name, url, type (newsletter|podcast), optional default_tags, priority. Mutated by /magazine add <url> · remove <name> · list.
  • tags.yaml — closed central tag registry; LLM-seeded on first run, batch-approved; always includes an uncategorized bucket.
  • interest.yaml — declared topics/feeds you care about (captured via a first-run interview), feeding the Top-picks curation subagent. Not an engagement-learned signal in v1.
  • state.json — per-feed last-run cursor + a per-item lifecycle record (discovered → downloaded → transcribed → summarized → rendered → failed:<reason>) keyed by GUID. This is the resume + dedup ledger.
  • transcripts/ — whisper transcripts cached forever by GUID (resume + re-summarize never re-transcribe). Audio is downloaded to a temp path and deleted immediately after a successful transcript.
  • crawl-cache/ — extracted article text by GUID, so a link is never re-crawled.

Stage A — deterministic prep (headless-capable, parallel, resumable; /magazine = since last run · --days N · --feed <name> · --max-per-feed N):

  1. Discover. Fetch each feed XML in isolation (timeout; a dead/malformed feed is skipped + reported, never aborts the issue). Window to items newer than the cursor / inside --days, minus already-rendered GUIDs. Snapshot this item set — it defines the issue.
  2. Crawl. For every item, follow its <link> and readability-extract the full article (cache by GUID). RSS body is fallback-only on crawl failure; paywalled pages → preview-only flag.
  3. Transcribe. For podcast items, download the enclosure audio and transcribe via whisper/whisper.cpp if on PATH (cache transcript, delete audio). No whisper → use show-notes + an "install whisper" hint.

Stage A is resumable from state.json at per-item granularity and can run long / in parallel — speed is not a goal.

Stage B — in-session agent (subagent fan-out; incremental):

  1. Summarize + tag. One subagent per ready item → 3–5 bullets + read/listen link (from the crawled article / transcript), tagged against the closed registry (only from the set; flags "no fit → suggest add"). Unconstrained cost — strongest model, full context.
  2. Curate Top picks. A subagent ranks the summarized items against interest.yaml (falls back to item-intrinsic importance when interests are sparse).
  3. Render incrementally. Re-emit the self-contained issue HTML as items complete — "Top picks" lane + card grid (title, source badge, date, tag chips, bullets, link); client-side filters (feed multi-select + date + tags). Failed items still render as degraded, reason-flagged cards (never silently dropped). Update the searchable library index.html.
  4. Commit on completion. The "since last run" cursor advances only when the issue fully completes, so interrupt+resume never drops or double-counts.

Home: pmos-learnkit (a staying-current / learning surface beside /primer, /learn-list, /critical-thinking; no workstream context), reusing the _shared/html-authoring substrate.

Success signals

Leading: I open an issue and skim 3–5 bullets per item; I click through ~10–20% of items; <1 untrustworthy summary per 20; "Top picks" actually matches what I'd have chosen. (Speed is explicitly not a success criterion — the pipeline may run long; it just has to resume cleanly and update the issue as it goes.)

Lagging: I re-run it regularly without being nagged; I stop opening Substack/podcast apps to "catch up"; the searchable library becomes where I go to recall "what did I read about X?"
