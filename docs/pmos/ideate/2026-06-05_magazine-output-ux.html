Magazine issue output UX

Created using pmos‑toolkit

TL;DR

A portfolio of 11 UX upgrades to each /magazine issue, sorted by effort × value × offline-risk. The decisive insight: 8 of the 11 are pure render-issue.js changes — no pipeline edits, no new data, zero risk to the offline/file:// promise — so they should ship as one fast release. The other three (thumbnails, per-post diagrams, transcription cron) each carry a real cost or a real tension with the "self-contained, trustworthy" core and should be gated behind cheap probes, not bundled in.

Recommendation: GO now on the 8 render-only wins as Wave 1. GATE thumbnails & the cron behind two 10-minute probes (Wave 2). VALIDATE per-post diagrams on a single hand-made sample before building anything (Wave 3). Do not big-bang all 11.

How Might We

How might we make each magazine issue more visually engaging and faster to skim and navigate — without breaking its core promise of a trustworthy, self-contained digest that works offline from file://?

Target user & job-to-be-done

Who: The magazine owner (you) — a PM with a dozen-plus newsletter + podcast subscriptions, catching up on a backlog.

When / where: A dedicated catch-up session, opening the latest issue from disk (often offline), wanting to triage and read in one sitting rather than app-hopping.

JTBD: When I sit down to clear my feed backlog, I want to skim and move through an issue in a way that is pleasant and fast, so I actually finish it and trust what I read — instead of bouncing between five apps.

Hypothesis

We believe raising the visual hierarchy and adding lightweight reading affordances (layout choice, read-state, color-coding, dropdown filters, a catch-up budget) will produce more items actually read per issue and a stronger pull to use /magazine over a raw RSS app because the current single-grid-of-text issue is trustworthy but undifferentiated to scan — the eye has no pre-attentive cues and no sense of progress, so long issues feel like a wall.

How it works (chosen finalist)

North-star reframe (the "8★" issue): opening an issue feels like opening a well-designed digital magazine, not a text dump. The header tells you the catch-up cost ("~46 min skim · 4h audio"). You pick a reading shape — a calm grid, an In-shorts card-by-card swipe, or a dense listicle. Newsletters and podcasts are instantly distinguishable; tags carry stable colors so themes pop; filtering is two dropdowns and a result count. As you go, cards mark themselves read and the counter ticks down, so a half-finished issue welcomes you back. Images and the occasional top-pick diagram add texture — but only where they earn their place, and never at the cost of opening the file offline.

The load-bearing constraint: every issue must stay self-contained and work from file:// with inline CSS/JS (NFR-3). This cleanly splits the portfolio: the render-only wins respect it for free; the enrichment ideas (images, diagrams) are exactly the ones that strain it. Hence the sequencing below.

Premortem: failure modes

It is 2027-06-05. This redesign shipped 12 months ago and underwhelmed. Why?

Failure modeLikelihoodMitigation
The 8 cheap wins got held hostage by the 3 hard ones — bundled into one feature, nothing shipped for months.HighShip Wave 1 (render-only) as its own release; treat 9/10/15 as independent follow-ups.
Offline promise quietly broke — thumbnails inlined as base64 ballooned issues to multi-MB, or a sibling assets/ dir broke "email the file."Med–HighDefault images off; cap dimensions/size; make the single-file-vs-folder call explicitly before building (9).
Auto-diagrams from 4-bullet summaries look AI-generated and erode the "trust" the digest sells.MedTop-picks only, opt-in flag, glanceable-quality bar, trivially disableable; validate on one sample first.
Three layouts + read-state + keyboard + dropdowns tangled into inline JS; one bug blanks the whole issue.MedKeep no-JS grid as the fallback; isolate state in one small module; progressive enhancement.
Transcription cron failed silently (whisper broke / disk filled) or raced state.json with an interactive run — an issue came up mysteriously empty.MedFile lock around state writes; a log file; transcribe.sh --selftest precheck; surface last-run status in the issue footer.
Color + images + diagrams + chrome compounded into a busy page that skims slower than the plain grid — defeating the actual goal.MedOne restrained accent system; measure skim-time before/after on a real issue; restraint over decoration.

Munger inversion — choices that would guarantee this fails:

  • Ship all 11 in one big-bang release, so nothing lands until everything is done.
  • Hotlink remote images directly, so issues break the moment you're offline or a CDN rotates the URL.
  • Generate a diagram for all 32 items regardless of whether it helps — maximizing token cost and visual noise.
  • Build the layouts, keyboard nav, and read-state as tangled global JS with no no-JS fallback.
  • Run the cron with no lock, no log, and no health surfacing, so failures are invisible until an issue is empty.

Success signals

Leading: skim-time per issue drops; you open more read/listen links per issue; you actually use mark-as-read and the "N left" counter; you switch layouts at least once (proves the choice was wanted).

Lagging: you reach for /magazine over your RSS app across consecutive weeks; issues feel "designed" enough to share; the backlog gets finished, not abandoned half-read.

Open questions

  • Single self-contained HTML file vs. an issue-folder with assets/ — which matters more to you: portability (email/AirDrop) or rich local images?
  • Do your actual feeds carry per-item images, or only per-show art? (Determines whether thumbnails are worth it at all.)
  • Is a per-post diagram signal or noise for a 4-bullet summary — and would it apply to podcasts too, or newsletters only?
  • Is device-local localStorage read-state acceptable, or do you want read-state to survive across machines (which would need a sidecar / sync, breaking single-file)?
  • launchd (macOS-native) vs cron for the transcription job — and how should silent failures reach you?
  • Carousel (In-shorts) default on mobile only, or a global default? Which layout should a fresh issue open in?
