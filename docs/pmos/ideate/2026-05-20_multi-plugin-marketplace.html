Multi-plugin marketplace migration

Created using pmos‑toolkit

TL;DR

Chosen topology — confirmed after surfacing the discovery requirement (each plugin is an independent product needing its own GitHub presence, stars, issues) and confirming Claude Code's marketplace.json supports external git sources:

Umbrella repo + per-plugin repos remotely, monorepo locally, split-on-release.

  • Remote: a tiny new umbrella repo github.com/maneesh-dhabria/pmos-agent-skills holds only .claude-plugin/marketplace.json + README. Each plugin (pmos-toolkit, pmos-learnkit, future pmos-*) lives at its own GitHub repo and is referenced via {"source": "github", "repo": "..."}.
  • Local: keep the existing agent-skills/ monorepo with one .git. Cross-plugin substrate refactors (_shared/html-authoring, _shared/interactive-prompts) stay atomic — single commit, single grep surface.
  • Release: git subtree split --prefix=plugins/<name> pushes filtered history to the standalone GitHub repo per plugin on tag. Wrapped in a ~150 LOC release script with --dry-run.
  • Codex: wait-and-see — Codex's external-source schema isn't documented at parity yet; ship Claude Code umbrella first, add Codex umbrella when their spec stabilizes. Acceptable 3–6 month gap given Codex is the smaller surface.

This shape gives each plugin independent discoverability (its own stars, issues, README, SEO) without paying the cross-plugin refactor tax of a full local split.

Recommendation: #3 — umbrella + monorepo-local + split-on-release. The discovery axis flipped the analysis: #2 (namespaced tags) fails on independent-product discoverability, and #1 (full local split) gives up the _shared/ refactor ergonomics that pmos-toolkit already depends on heavily. #3 is the only variant that satisfies both axes simultaneously, at the cost of one one-time release-tooling investment.

How Might We

How might we restructure agent-skills so it cleanly hosts multiple plugins (pmos-toolkit, pmos-learnkit, future ones) with independent release cycles and remotes, while keeping the marketplace manifest cohesive and CLAUDE.md invariants intact?

Target user & job-to-be-done

Who: Maneesh — sole maintainer, currently shipping pmos-toolkit (v2.49.0) and about to scaffold pmos-learnkit.

When / where: Pre-scaffold of pmos-learnkit; the next 1–4 weeks before any second-plugin code lands.

JTBD: When I'm ready to ship a second plugin alongside pmos-toolkit, I want a repo topology that lets me tag/release/push one without dragging the other, so I can iterate independently without paying ongoing tax on every commit.

Hypothesis

We believe adopting an umbrella marketplace repo that references each plugin's standalone GitHub repo, while keeping a local monorepo and splitting on release will produce per-plugin discoverability + independent release cycles + preserved cross-plugin substrate refactor ergonomics because Claude Code's marketplace.json supports external GitHub sources with ref/sha pinning, git subtree split can extract clean filtered history per release, and the umbrella repo itself is near-static (changes only when adding a plugin or tuning a manifest field).

Idea variants considered

Generated via SCAMPER + Analogous Inspiration. Finalists flagged with ★.

  1. ★ Separate git repo per plugin — the doc's current plan; each plugin has own .git + GitHub remotes; root is thin registry.
  2. ★ Monorepo with namespaced tags — keep single .git; release via pmos-toolkit/v2.50.0 + pmos-learnkit/v0.1.0 tags.
  3. ★ Subtree-split on release — monorepo is source of truth; git subtree split --prefix=plugins/<name> pushes filtered history to per-plugin GitHub mirrors at release time.
  4. Git submodules — root references each plugin as a submodule.
  5. Sparse-checkout distribution — single repo; CI builds per-plugin distribution tarballs/branches.
  6. Hybrid: monorepo dev, auto-mirror each plugins/* to standalone GitHub repos via subtree push on tag (close relative of #3).
  7. Workspace pattern: shared tooling via symlink, each plugin has own .git.
  8. Orphan branches per plugin in one repo (never merged).
  9. Marketplace-as-pure-manifest — root holds only marketplace.json + docs; plugin source lives entirely in external repos.
  10. Release-script monorepo — keep single .git; add release.sh <plugin> handling tagging via on-the-fly subtree-split.
  11. Bare mirror repos alongside monorepo, updated post-commit.
  12. Two-tier split: agent-marketplace (manifest only) + agent-plugins (all plugin source).

How it works (chosen finalist: umbrella + monorepo-local + split-on-release)

Remote topology

github.com/maneesh-dhabria/
├─ pmos-agent-skills/      ← umbrella: .claude-plugin/marketplace.json + README + docs
├─ pmos-toolkit/           ← standalone plugin (already exists; stays as-is)
├─ pmos-learnkit/          ← standalone plugin (new)
└─ pmos-<future>/          ← each future plugin gets its own repo

Each plugin owns its own GitHub presence: stars, issues, discussions, README, SEO. The umbrella is install-time discovery (/plugin marketplace add maneesh-dhabria/pmos-agent-skills).

Umbrella marketplace.json

{
  "name": "pmos-agent-skills",
  "owner": { "name": "Maneesh Dhabria", "url": "https://github.com/maneesh-dhabria" },
  "plugins": [
    {
      "name": "pmos-toolkit",
      "description": "PM + software delivery pipeline.",
      "source": { "source": "github", "repo": "maneesh-dhabria/pmos-toolkit", "ref": "main" },
      "category": "productivity",
      "homepage": "https://github.com/maneesh-dhabria/pmos-toolkit"
    },
    {
      "name": "pmos-learnkit",
      "description": "...",
      "source": { "source": "github", "repo": "maneesh-dhabria/pmos-learnkit", "ref": "main" },
      "category": "productivity",
      "homepage": "https://github.com/maneesh-dhabria/pmos-learnkit"
    }
  ]
}

Pin via "ref": "v2.50.0" or "sha": "..." if you want frozen install points; "ref": "main" tracks latest.

Local topology

Unchanged from today — single .git at agent-skills/, plugins under plugins/<name>/. Shared substrate (plugins/pmos-toolkit/skills/_shared/ and any future top-level shared modules) edits remain single-commit, atomic, greppable across all plugins.

Release flow (per plugin)

  1. Work locally in agent-skills/; commit normally.
  2. /complete-dev --plugin <name> bumps the targeted plugin's .claude-plugin/plugin.json + .codex-plugin/plugin.json + CHANGELOG.md.
  3. Tag locally with namespaced format: git tag pmos-learnkit/v0.1.0 (still useful as a monorepo cross-reference).
  4. Release script invokes git subtree split --prefix=plugins/<name> -b release-<name>, then pushes release-<name> → <plugin-repo>:main and the tag v0.1.0 (unprefixed in the standalone repo).
  5. Umbrella repo is touched only if a new plugin is being added or the manifest is being re-pinned.

Release script sketch

#!/usr/bin/env bash
# scripts/release-plugin.sh <plugin-name> <version>
set -euo pipefail
plugin="$1"; version="$2"
prefix="plugins/$plugin"
remote="$plugin"   # git remote pointing at github.com/maneesh-dhabria/$plugin

[[ -d "$prefix" ]] || { echo "no such plugin: $prefix"; exit 1; }
git remote get-url "$remote" >/dev/null || { echo "no remote: $remote"; exit 1; }

split_branch="release-$plugin"
git subtree split --prefix="$prefix" -b "$split_branch"
git push "$remote" "$split_branch:main"
git push "$remote" "$plugin/v$version:refs/tags/v$version"
git branch -D "$split_branch"

Add --dry-run, pin git version in CI, integration-test against a throwaway plugin before going live.

Codex strategy

  • Phase 1 (now): ship Claude Code umbrella. Each plugin's .codex-plugin/plugin.json still exists in its standalone repo — Codex users install each plugin individually (current Codex norm anyway).
  • Phase 2 (later): when Codex documents external-source schema at parity with Claude Code, add .codex-plugin/marketplace.json to the umbrella. No structural change required.

Reversibility

If this topology ever stops fitting (e.g., one plugin becomes so dominant that the umbrella feels like overhead, or vice versa), the escape hatches are cheap: collapse umbrella into a plugin repo's README; or stop using subtree-split and let plugin repos diverge to be source-of-truth. Local monorepo is preserved either way.

Munger inversion — what would guarantee failure?

  • For #1: Force-push without auditing skill scripts; cp (not symlink) the pre-push hook; leave plugins/pmos-toolkit/ tracked in root .git alongside its own .git (double-tracking → conflicting commits); skip git remote -v verification before each force-push; forget .gitignore after git rm --cached.
  • For #2: Tag without prefix (v2.50.0 alone); let /complete-dev bump both plugins when only one changed; mix plugin commits in a single PR/merge; skip extending the pre-push hook; reuse main without per-plugin release branches when scoping demands it.
  • For #3: Allow direct commits to mirror repos; run subtree-split on every push instead of on tag; skip pinning git version; let the recipe live in one developer's head; use deprecated filter-branch instead of subtree split or filter-repo.

Success signals

Leading (within 1 release of pmos-learnkit v0.1.0): pmos-agent-skills umbrella repo exists with valid marketplace.json; end-user can run /plugin marketplace add maneesh-dhabria/pmos-agent-skills and see both plugins listed; release-plugin.sh pmos-learnkit 0.1.0 pushes a clean history to the standalone pmos-learnkit repo without manual intervention; pmos-toolkit's release cadence is untouched.

Lagging (3–6 months in): Each plugin has its own non-zero GitHub stars/issues count (i.e., people are finding them by their own names, not just via the umbrella); cross-plugin substrate refactors (_shared/html-authoring updates) remain single-commit on the local monorepo and split cleanly to both standalone repos on next release; release script LOC has stayed under 200; no commits have leaked directly to the standalone plugin repos (branch protection holds); Codex umbrella manifest has been added (or a conscious decision made not to).

Open questions

  • Umbrella ref strategy: pin to "ref": "main" (always latest) or to specific tagged versions (e.g., "ref": "v2.50.0")? Latest = simpler maintenance, no umbrella commits per release. Pinned = reproducible installs, but every plugin release needs an umbrella update.
  • Cutover risk on pmos-toolkit: first git subtree split push rewrites the standalone repo's history. Are there forks/clones/external PR references that need coordinated cutover, or is the repo solo enough that a clean replace is fine?
  • Should _shared/ become its own plugin / package eventually? If pmos-learnkit consumes >30% of it, factoring it out (and having both plugins depend on it) becomes more honest than copy-by-subtree. Defer the decision until learnkit is real.
  • Codex umbrella schema: when (not if) Codex documents external-source support, do we mirror Claude Code's umbrella structure, or does Codex's design force a different shape?
  • GitLab remote: current root pushes to GitLab as origin. Does GitLab keep getting the monorepo (yes, that's the source of truth), or do we also create per-plugin GitLab mirrors via subtree-split? (Suggest: monorepo only on GitLab; per-plugin only on GitHub.)
