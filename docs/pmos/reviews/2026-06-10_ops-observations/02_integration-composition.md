# Ops review — integration, composition & orchestration between skills

**Date:** 2026-06-10 · **Scope:** the seams BETWEEN skills (cross-skill invocation, contracts, orchestration mechanics) across pmos-toolkit (32 skills), pmos-learnkit (6), pmos-utilities (2) + `_shared/` substrates. Orthogonal to the [2026-06-10 skill-design review](../2026-06-10_skill-design-review/report.md) — that report's settled findings (verbosity, markdown duplication-tax extraction, flag policy) are not re-litigated here; where a seam overlaps a prior finding it is cited, not re-counted.

## Verdict in one paragraph

The seam *architecture* is unusually deliberate — there is a real taxonomy of mechanisms (typed file contracts, shared contract docs, blocking subagents, thin aliases, folded substrates), and the typed seams are genuinely robust (closed enums, schema docs as single source, coverage gates, per-seam tests). The prior audit's question — "is the seam mechanism itself the root cause of phantom contracts?" — has a clear answer: **yes**. Every phantom/stale contract found (prior audit's `grill --deep`, readme's Phase 7.6 hook; this review's new `--from-reflect` phantom and the 10-file `.comments.json` sidecar staleness) lives at a **prose-mechanism seam** where skill A narrates skill B's behavior; **zero** phantoms were found at typed seams (state-schema.md-pointed state.yaml, apply-edit JSON contract, `resolve-input.md`, `execution_mode` frontmatter). The cost profile is inverted, though: the typed seams pay a heavy duplication tax in *code* (14 forked `apply-edit-at-anchor.js` shims, ~2,700 LOC; the FR-50/51/52 reviewer-validation harness restated at ≥3 call sites) that the prior report's markdown-dedup plan does not cover. And one advertised composition surface — `/frameworks --json` "lets other skills ask programmatically" — has **zero consumers**.

## Orchestration mechanics — how /feature-sdlc actually runs a stage

**Stages run in the SAME context, accumulating.** The orchestrator "invokes" each child skill (`Invoke /pmos-toolkit:requirements …`) — a Skill-tool/inline invocation that loads the child's SKILL.md and runs it in the parent conversation. Subagents are reserved for *roles*, not stages: reviewers (grill FR-50/51/52, msf-wf per-wireframe, skill-eval `[J]` judge), `/diagram` rendering, `/comments` per-thread resolvers, and (opt-in) `/execute --subagent-driven` implementers.

The repo knows this and has built compensating machinery rather than changing the mechanism:

1. **Compact checkpoint** (feature-sdlc:490–499) — fires before heavy phases (wireframes, prototype, execute, verify) precisely because context accumulates. "Skills cannot trigger `/compact` directly — only the user can… 'Pause' exits cleanly so the user can `/compact` and re-run with `--resume`."
2. **Pause/resume as the real handoff protocol** — `state.yaml` `phases[]` cursor (feature-sdlc:336), never-overwrite `started_at` as `/execute`'s git-log resume cursor (feature-sdlc:467–476), and the EnterWorktree failure handoff block (feature-sdlc:288–300) which is literally a *manual fresh-context mechanism*: "cd <path>; claude --resume; then /feature-sdlc --resume".
3. **Self-contained child briefs** — Anti-pattern #3 (feature-sdlc:858) forbids "see the state file" dispatch; every child gets artifact paths + marker lines only.

**Implication:** a long Tier-3 run is a *checkpointed batch process where the user is the scheduler*. That's a defensible design, but note the architectural headroom: because Anti-pattern #3 already forces self-contained briefs and the inter-stage interface is already files-on-disk (`resolve-input.md` contract), the pipeline is **one step away from fresh-context-per-stage dispatch** (each stage a subagent receiving only artifact paths). The genuine blocker is not state — it's **interactivity**: child skills issue `AskUserQuestion` prompts mid-stage, and subagents can't prompt the user. The existing `--non-interactive` contract (AUTO-PICK + OQ buffer + `[mode:]` marker) is, in effect, the missing half of a subagent-stage design already built. If long-run context exhaustion becomes the dominant operational pain, the cheapest evolution is: stages whose mode resolves to non-interactive (or whose gates were pre-confirmed via Phase 0e soft-gate defaults) dispatch as fresh subagents; interactive stages stay inline.

## Composition opportunities (capabilities in 2+ skills that should be one invocable unit)

Ranked by value ÷ cost. Items already scheduled by the prior report's text-dedup plan are flagged, not re-counted.

1. **apply-edit-at-anchor resolver → one shared module + per-skill feasibility config.** 14 forked Node shims (175–260 LOC each, 2,699 LOC total — `wc -l skills/*/scripts/apply-edit-at-anchor.js`) all implement the same id-first + ≥40-char substring resolver, same closed error_enum, same idempotency key; they differ almost only in per-artifact *feasibility rules* (e.g., feature-sdlc:887–888's two-surface table). The repo already documents the tax: "Adding a fifth [error_enum value] requires amending the contract doc first, then fanning out to the 14 skills" (comments:164). A bug in the anchor resolver is a 14-file fix today. Proposal: one `_shared/html-authoring/assets/anchor-resolver.js` (it ships with the asset substrate already copied everywhere) + each shim collapses to a ~30-line feasibility table requiring it. The P6 "minimal in-shim ≤200 LOC" decision optimized per-shim auditability; at 14 copies the calculus has flipped. Tests exist per-shim, so the refactor is regression-guarded. *NOT covered by the prior report (it counted markdown dedup only).* **Value: high. Cost: medium (one /skill-sdlc run + 14 mechanical collapses).**

2. **Reviewer-validation harness → one script + one substrate doc.** The FR-50/51/52 pattern (chrome-strip → subagent with canonical prompt → `sections_found` set-equality vs sections.json → ≥40-char quote substring-grep → hard-fail strings) is restated nearly verbatim at feature-sdlc:561–563 (grill), wireframes:501 (msf-wf), and inside spec/plan/artifact reviewer phases. The prior report flagged the *Reviewer Input Contract* (~50 lines) as the cleanest text extraction; go further per the adopted gates policy (*deterministic = script*): set-equality + quote-grep are pure arithmetic — make them one `_shared/html-authoring/assets/validate-reviewer-return.js` invoked everywhere, with the per-skill hard-fail prefix as an arg. Then a change to the grounding rule (e.g., 40→60 chars) is one file. **Value: high (this is the repo's core anti-hallucination invariant). Cost: low.**

3. **Wire /frameworks `--json` to its advertised consumers — or stop advertising.** `match.mjs --query "<problem>" --json` is offline, deterministic, zero-dep — a genuinely good composition surface with zero callers. Natural consumers: `/ideate` Pressure-test phase ("which framework fits this decision" → cite 1–2 with why), `/critical-thinking` scenario generation (framework-grounded drills), `/requirements` research synthesis. Each is a one-line Bash dispatch + a "weave at most N" rule. Alternatively, delete the "lets other skills ask programmatically" claim from the description — an advertised-but-unconsumed API is a phantom contract in the making. **Value: medium (user-visible quality lift). Cost: very low.**

4. **Converge the two fold patterns on fold-by-invocation.** The repo has two ways to fold a capability into a host skill: (a) **fold-by-invocation** — wireframes Phase 6 invokes `/msf-wf --apply-edits`; verify Phase 4b invokes `/architecture --since` — the folded skill stays the single implementation; (b) **fold-by-substrate** — requirements Phase 5a and spec Phase 6a inline `_shared/{msf,sim-spec}-heuristics.md` while the standalone skills (`/msf-req`, `/simulate-spec`) keep parallel "standalone phrasing" bodies (simulate-spec:143 acknowledges the dual source). Pattern (b) leaves two narrations of one capability per seam — exactly the disease the prior audit diagnosed. Recommendation: refactor `/msf-req` and `/simulate-spec` to be thin wrappers over their substrates (intent + pointer + deltas, the skill-sdlc shape), or invert and have requirements/spec invoke them. Either way, one body per capability. **Value: medium. Cost: low-medium (2 skills).**

5. **design-crit's PSYCH/MSF pass → consume, don't mirror.** design-crit:291–295 "format mirrors /wireframes and /prototype" + the "if you have access; otherwise:" fallback restates msf-wf's PSYCH output format. The prior report's P2 (`_shared/psych-scoring.md`) fixes the text; the *seam* fix is for design-crit to either invoke `/msf-wf` against captured journeys or cite the shared doc unconditionally (it's the same plugin — "if you have access" is never false; the hedge only licenses drift). **Value: low-medium. Cost: trivial once P2 lands.**

6. **readme → polish: replace filesystem-reach theater with the real seam.** voice-diff.sh's `_polish_substrate_check` finds polish's `voice*` files and *uses none of them* (voice-diff.sh:23 "We do not source it") — delete the check (it's pure noise) and keep the built-in tokenizer as the only path. The actual delegation ("Suggest: /polish") is fine as user-mediated, but readme `--update` mode could offer to *invoke* /polish directly (mechanism A) and run the FR-V-3 drift gate in the same session, closing the round-trip that currently depends on the user re-driving two skills in the right order with a manually-preserved `.pre-polish` baseline. **Value: medium (readme is the D-graded skill; this is part of its rewrite). Cost: low, fold into the planned readme rewrite.**

## Decomposition opportunities (skills doing 2+ jobs)

Assessed skeptically — each split adds a seam, and prose-mechanism seams are this repo's defect factory.

1. **feature-sdlc: decompose by *mode*, not by stage — via progressive disclosure, not new skills.** At 909 lines + 16 reference files, the bloat is not the stage dispatches (those are already thin); it's that **four run modes' conditional logic is interleaved through every phase** — each of the ~20 phase sections carries `feature`/`skill-new`/`skill-feedback`/`prototype` carve-outs (the mode×phase matrix at lines 36–50, prototype reordering shim at 578–595, skill-mode directives at 537/671/688/704/766). Every run loads all four modes' rules into context. Proposal: keep one orchestrator skill, but move mode-specific phase *deltas* to `reference/skill-mode.md` and `reference/prototype-mode.md`, loaded only when that mode resolves at Phase 0 — the same progressive-disclosure pattern the repo already executes well (wireframes' 52-file tree, per prior report). Do NOT split into separate orchestrator skills — the thin-alias entry points (skill-sdlc, prototype-sdlc) already give users mode-shaped front doors, and a shared resume/state/worktree core is the whole point. **Value: high (context per run, edit isolation). Cost: low-medium.**

2. **complete-dev: leave it together.** 24 phases / 777 lines looks decomposable (merge / learnings / readme freshness / changelog / bump / tag / push / cleanup), but: it already delegates the two parts with independent lives (/changelog, /verify); the prior report explicitly steelmanned release mechanics as non-ballast; and the phases are a *transaction* — splitting tag/push from bump creates partial-release states that the current monolith's ordering prevents. No part is independently useful enough to pay for the new seam. **Recommendation: no split.**

3. **verify: no split, but finish the folded-phase conversion.** Its 10 phases are genuinely one job (the post-implementation gate) and it's non-skippable — a fragile seam inside a hard gate is the worst trade available. Phase 4b (invoke `/architecture --since`) is the template; Phase 3's multi-agent code review is the only candidate with independent reuse (a standalone `/code-review` shape), and even there the host-side aggregation is most of the value. **Recommendation: keep; convert any future verify capability additions to fold-by-invocation rather than inline growth.**

4. **wireframes: already decomposed where it matters.** Phase 6 delegated to /msf-wf; DESIGN.md resolution is a reference file consumed by 3 skills (prototype:114, verify drift check); the canvas viewer has no independent use. The prior report's verdict (the 52-file tree IS progressive disclosure) holds at the seam level too. **Recommendation: no split.**

5. **magazine: the model citizen — mine it for the pattern.** Stage-A deterministic scripts / Stage-B in-session agent / ledger-as-queue / optional watch worker is exactly the decomposition the other big skills lack: parts are scripts with CLIs, not skills, so no new skill-to-skill seams were created. Worth codifying in skill-patterns.md: *"decompose into scripts before decomposing into skills."*

## Recommendations (ranked)

1. **Adopt a seam-mechanism policy in skill-patterns.md** (the durable fix; mirrors the gates policy the maintainer already adopted): *a cross-skill fact may live in prose only if it is a pointer to a typed source* — every seam must be one of: typed file contract (E), owned contract doc cited by path (F), invocation with flags that exist (A/B, lint-checkable), or thin alias (C). "Skill A narrates skill B's behavior" is banned; the two new defects this review found (`--from-reflect`, `.comments.json` ×10) would both have been impossible under it.
2. **Fix the `--from-reflect` phantom** (P0-class): either /reflect gains an opt-in `--save [path]` writing a discoverable artifact (and feature-sdlc:90 cites that path), or feature-sdlc drops `--from-reflect` and keeps only text/path sources. Decide once; don't patch the prose.
3. **Scrub the `.comments.json` sidecar sentence from the 10 stale "Apply comment-resolver edit" sections** (mechanical, one commit) — and while there, shrink the fanned boilerplate to the 2-line pointer the contract doc already mandates ("MUST cite that file rather than restate").
4. **Extract the shared anchor-resolver module** (Composition #1) and **script the reviewer-return validator** (Composition #2) — the two code-level dedups the markdown-dedup plan misses; both are regression-guarded by existing tests.
5. **Wire or un-advertise `/frameworks --json`** (Composition #3).
6. **Refactor /msf-req and /simulate-spec to thin-wrapper shape** over their substrates (Composition #4), making fold-by-invocation / single-body the only fold pattern.
7. **Mode-delta progressive disclosure for feature-sdlc** (Decomposition #1).
8. **Longer-term:** when context exhaustion on Tier-3 runs becomes the top complaint, implement fresh-subagent stage dispatch for non-interactive/pre-confirmed stages (see "Orchestration mechanics") — the contracts (self-contained briefs, resolve-input, `[mode:]` markers, OQ buffer) already exist; this is an assembly job, not a redesign.
9. **Do not** split complete-dev, verify, or wireframes; **do** add "decompose into scripts before skills" (the magazine pattern) to skill-patterns.md.
