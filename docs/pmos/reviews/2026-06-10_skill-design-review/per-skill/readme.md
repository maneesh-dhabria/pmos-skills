# readme — review

**Grade:** D
**Size:** SKILL.md 487 lines (458 excluding non-interactive block); references 7 files / 1,016 lines; scripts 7 files / ~1,996 lines; target ~200 lines for SKILL.md.

## TL;DR

- **Biggest win available:** one of the skill's three advertised modes (`--update`) is unreachable — §8's dual gate requires `.pmos/complete-dev.lastrun.yaml :: readme_update_hook`, which is set by a complete-dev "Phase 7.6 hook" that complete-dev never shipped. Repair or excise it, and the ~95 lines + 466-line script chain it drags along either come alive or come out.
- **Biggest risk in current design:** the SKILL.md is a transplanted spec, not an instruction doc — FR-IDs, truth tables, T-task numbers, "TBD" placeholder sections, and counts ("15-check", "3-persona") that contradict the actual reference files. The model is told to dispatch "the 3-persona pass" against a reference file that defines 4 personas, and to report "rubric X/15" against an 18-check rubric.yaml. The structure actively fights the model at runtime, not just the reader.
- **Worth keeping:** the deterministic substrate is genuinely excellent — `rubric.sh` (fixture-calibrated, ≥85% agreement gate, bash-3.2-portable), `workspace-discovery.sh` (21 workspace fixtures incl. kubernetes/babel-style edge cases), and the persona *content* in `reference/simulated-reader.md` (readers-who-can-leave framing, bounce-trigger examples, verbatim-quote grounding). The skill's assets are in `scripts/` and `reference/`; the liability is the SKILL.md wrapped around them.

## Findings

1. **[G][X] Phantom contract: §8's dual gate references a complete-dev Phase 7.6 that does not exist — `--update` mode is dead on its documented path.** Verified: `grep -rn "readme_update_hook\|Phase 7.6"` across `plugins/pmos-toolkit/skills/complete-dev/` returns zero hits; complete-dev's phases run 0→17 with no 7.6 (its Phase 7 is an unrelated skill-inventory drift check that never invokes /readme); `reference/lastrun-schema.md` has no readme key; Phase 0a's run-defaults list has no readme hook entry. Yet SKILL.md §8 requires **both** `~/.pmos/readme/config.yaml :: phase_7_6_hook_enabled` AND `.pmos/complete-dev.lastrun.yaml :: readme_update_hook` to be true before §7 fires, and states the latter "is per-run, set by /complete-dev's Phase 0a ask … It is NOT set manually." Nothing sets it → the FR-UP-4 truth table always lands on "No-op + warn", **including for manual `/readme --update main..HEAD`**. Origin found: `docs/pmos/features/2026-05-13_readme-skill/02_spec.html` planned "Phase 7.6 hook orchestration lives in /complete-dev; /readme just exposes" it, "opt-in / default-off until ≥3 real-repo runs" — the /readme half shipped, the /complete-dev half never did. **Fix:** the dual gate's own rationale ("the global flag without the per-run confirmation would surprise users") only applies to the *hook* path; a user typing `--update <range>` has already consented. Make manual `--update` gated only by the existing §7 per-section Apply/Skip prompts + rubric patch-fail guard; keep the dual gate solely for a future complete-dev hook (or delete §8 until that hook ships). This also revives `commit-classifier.sh` (255 lines) and the FR-V-3 voice-drift path (`voice-diff.sh`, 211 lines), which are currently reachable only through dead code.

2. **[R] Stale counts misdirect the runtime: "15-check rubric" vs 18 checks; "3-persona" vs 4 personas.** `rubric.yaml` declares 18 checks (16 `[D]` including `cross-cutting-capabilities-surfaced`, plus 2 `[J]` reviewer checks). The 4th persona (`returning-user-navigator`) and the `[J]` checks were added by `2026-05-15_readme-audit-fixes` (its spec: "15-check rubric + 3-persona simulated-reader pass is … blind to value-prop accuracy and information-architecture fit" → "4 personas + 1 reviewer"), but the counts were never propagated: frontmatter description, the title line, Core Pattern, §6 steps 6/8 ("rubric <X>/15", "<12/15"), §3 ("skip the 3 Task dispatches" — there are 5), Anti-Patterns ("the 3-persona pass"), `rubric.sh`'s header comment ("Runs 15 binary checks"), and even `reference/simulated-reader.md`'s own title ("3 persona prompts" — the file defines 4). **Fix:** remove numeric counts from prose entirely — say "the checks declared in `reference/rubric.yaml`" and "the personas in `reference/simulated-reader.md`". Counts in prose are exactly the kind of time-bombed detail the durability principle (north-star #4) warns about; the YAML is already the single source of truth and `rubric.sh` already reads it.

3. **[R] Construction scaffolding shipped in the final artifact.** Lines 122–128 are four literal placeholder headings: "### Subsection 2 — TBD (rubric runner)" … "### Subsection 5 — TBD (atomic write)". ~15 plan-task references remain: "lands in T14 / T18" (§ Single-file audit flow), "T16 lands §6", "(forward-cite — dangling until T20 merges)" in §9 — T20 merged, `reference/cross-file-rules.md` exists — "ships in T23, parallel Wave 1" (§11), "(T8 deliverable … T9 will fill in … T10 will refine)" in workspace-discovery.sh's header (the stub it describes is long since filled). A colleague cannot tell what is shipped vs. planned. **Fix:** delete the TBD headings, strip every T-number, and rewrite forward-cites as plain links. Pure quick-win — zero behavior change.

4. **[V][P] SKILL.md is a spec transplant; realistic target ~200 lines.** ~66 FR-ID citations ("Cite FR-V-2", "per FR-OUT-4", "(FR-MODE-1.)"), exact error-message strings for every failure branch, the full repo-miner JSON return shape inline (§5 — duplicating spec §9.2.2), a 7-row mode-resolution truth table where 3 sentences would do, and the §10 monorepo section restating §1/§6/§7's contracts a second time "as overlay". The domain *is* complex (3 modes × monorepo × subagents), but most of the length is not domain — it's restating what the scripts already enforce deterministically (`rubric.sh` exits 0/1/2; `workspace-discovery.sh` owns precedence; `_reviewer_validate.sh` owns reviewer validation) and what any capable model does anyway ("parse the JSON output", "capture stdout and exit code"). Compare Pocock's diagnose: 160 lines for a 6-phase process because it spends words only on the ONE thing models get wrong. Here the one thing models get wrong is *inventing README facts* — "never invent commands; use TODO markers" — and that load-bearing rule is buried at §6 step 3 and Anti-Patterns. **Fix (structural):** rewrite the body as: mode resolution (5 lines), per-mode intent + the one anti-pattern each (audit = read-only, never prompt; scaffold = never invent, ≤6 questions then stub; update = patch-only, rubric guard, never commit), pointers to `reference/` for persona/reviewer dispatch contracts and to scripts for everything deterministic. Move the §5 return shape and §7/§8 mechanics into a `reference/modes.md`.

5. **[R] Dependencies section contradicts the scripts it describes — three ways.** SKILL.md claims "**bash ≥ 4** — `scripts/_lib.sh` and all `scripts/*.sh` use bash-4 features (associative arrays, `${var,,}`)"; every script header says the opposite ("Bash 3.2-safe (macOS default). No associative arrays, no `${var^^}`" — rubric.sh:6, workspace-discovery.sh:19, voice-diff.sh, commit-classifier.sh, run-dogfood.sh) and grep confirms zero bash-4 constructs. Meanwhile `_lib.sh`'s own header says "Bash ≥ 4 required" while implementing 3.2-portable code. SKILL.md also claims "**node ≥ 18** — `voice-diff.sh` shells into a tiny node helper for unified-diff rendering"; voice-diff.sh contains zero node references — it is pure awk. The scripts went portable (good — repo "Bash portability" invariant) and the doc never followed. **Fix:** quick-win — correct the Dependencies block and _lib.sh comment; drop the node claim or scope it to the HTML substrate only.

6. **[G] The 4-persona simulated reader: insight generator at its core, ceremony at its edges.** The core earns its keep: the personas test four *distinct jobs* (60s evaluate / 5-min first-success / 30-min contribute / 2-min intent-mapping) the deterministic rubric cannot see; the "you are NOT a reviewer, you are a reader who can leave" anti-script plus concrete bounce-triggers is real prompt engineering against LLM helpfulness bias; the ≥40-char verbatim-quote + parent substring-grep contract prevents hallucinated findings; the theater-check (empty friction while rubric found ≥3 → one re-dispatch with "what makes you bounce?") targets a documented sycophancy failure mode with a hard 1-retry cap (D6). Keep all of that. The ceremony: **hard-fail-and-pause** on a 39-char quote or a persona-label typo turns a subagent formatting slip into a dead run — the proportionate response is *drop the entry with a warn* (the grep grounding still guarantees nothing fabricated merges); the line±2 + section-spine dedupe algorithm and the 120s-per-call wall-budget arithmetic are over-specified HOW; the two stub env vars (`READMER_PERSONA_STUB`/`READMER_REVIEWER_STUB`) are documented twice each (§2 and §3) with "DO NOT use in production" both times. **Fix:** soften persona-validation failures to drop-with-warn (keep hard-fail only for the reviewer's check_id set-equality, which feeds scored rubric rows); state timeouts once as advisory; document the stubs once, in the reference file.

7. **[S] The quote-grounding reviewer contract is a shared pattern restated four times locally — extract to `_shared/`.** `reference/simulated-reader.md` §4 says it "mirrors `/grill`'s FR-50/51/52 pattern exactly" and quotes /grill's SKILL.md at length. The same parent-side contract (inline un-stripped source → subagent returns JSON with ≥40-char verbatim quotes → parent substring-greps → reject misses) appears in SKILL.md §2 (twice: personas + reviewer), simulated-reader.md §2 and §4, and reviewer.md §3 — and lives independently in /grill, /artifact, and the other HTML-pipeline skills. There is no `_shared/` doc for it (checked: `_shared/` has `apply-edit-at-anchor.md`, `interactive-prompts.md`, etc., but no reviewer-quote contract). **Fix (structural, cross-skill):** create `_shared/subagent-quote-contract.md`; each skill cites it the way the comment-resolver phase already correctly cites `_shared/apply-edit-at-anchor.md` ("MUST cite that file rather than restate" per NFR-08 — the right pattern, applied inconsistently within the same file).

8. **[G] `tests/integration/update_hook_dry_run.sh` tests documentation, not behavior.** Its own header admits it: "/readme itself is un-mockable from bash; we substitute SKILL.md contract greps (FR-UP-3/4) for the runtime patch-drop and dual-gate paths." Cases (b) and (c) grep SKILL.md prose for the dual-flag table — they pass while the feature is dead (finding 1) and would fail on a wording change. Only case (a), which runs `commit-classifier.sh` directly, tests real behavior. **Fix:** keep (a); convert (b)/(c) into either real fixture-driven classifier/guard tests or delete them — prose-grep tests create false confidence and wording-brittleness simultaneously.

9. **[R] Stale `.comments.json` sidecar reference.** The comment-resolver phase says `/comments resolve` walks "open threads in a readme audit artifact's `.comments.json` sidecar" — the sidecar contract was retired in v2.58.0 (2026-05-28 inline-html-artifacts; comments now live in the inline `pmos-comments` block per repo CLAUDE.md). Quick-win: reword to "the artifact's inline pmos-comments block".

10. **[Ph] Section numbering is plan-shaped, not reader-shaped.** Phase 0 → Phase 0b → "Core Pattern" → "Implementation" containing an unnumbered "Single-file audit flow" that later text calls "§1", four TBD stubs, then §2–§11 (each § maps 1:1 to a /plan T-task), then "Phase N", then an unnumbered comment-resolver phase. §10 "supersedes their per-file flows for the duration of the run" — a section that overrides three earlier sections conditionally is the classic insertion-without-rethinking smell the criteria flag. **Fix:** folds into finding 4's rewrite — one mode-resolution section, three mode sections, one monorepo section, one delegation section.

11. **[F] Flag-register confusion and one orphan.** §3 says `--skip-simulated-reader` "is parsed by §1's argv loop alongside `--variant`, `--auto-apply`, etc." — but `--variant`/`--auto-apply`/`--selftest` are `rubric.sh` flags, not skill flags; the skill and its bundled script share one argv namespace in prose only. §10 declares `--scope` "mutex with the per-file `--path` flag" — `--path` is defined nowhere in the file or the argument-hint. Quick-win: delete the script-flag mentions and the `--path` clause (or define it).

12. **[X] Parallel-dispatch mandate has no degradation path.** §2 requires "5 `Task` tool calls in ONE assistant response" and "Sequential dispatch is forbidden (P3)" — justified only by wall-clock budget. On a platform without parallel subagent dispatch this makes the default path un-runnable; the Platform Adaptation section covers only the prompting contract. Quick-win: "dispatch in parallel where the platform supports it; otherwise sequentially" — the 120s-per-call budget already bounds the cost.
