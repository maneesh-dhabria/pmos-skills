# complete-dev — review

**Grade:** C+ (would a thoughtful reviewer call this well-designed for its job?)
**Size:** SKILL.md 774 lines (745 excluding non-interactive block); references 7 files / 494 lines (+ 28-line `diff_router.sh`); target ~450 lines.

## TL;DR

- **Biggest win available:** fix the staleness debt from the v2.41.0 (worktree-move) and 2026-06-05 (phase-renumber) refactors — the Track Progress line still enumerates fractional phases that no longer exist, `rollback-recipes.md` tells the model the worktree "was already removed in Phase 4" (now false, and it's a *destructive-recovery* doc), and two sibling skills (`/readme`, `/feature-sdlc`) reference complete-dev contracts (`Phase 7.6 readme hook`, `state.base_drift` surfacing) that complete-dev never implemented.
- **Biggest risk in current design:** stale instructions in the destructive paths. A model following rollback Recipe 3/5 verbatim would tell the user their worktree is gone when it isn't, and would tag/delete unprefixed `v<version>` tags that violate the FR-58 `${plugin}/v<version>` convention the SKILL.md itself mandates.
- **Done well, worth keeping:** the release mechanics themselves are genuinely load-bearing and well-reasoned — the Phase 9 stale-bump decision table, the shared-branch guard, the tag→push→push-tag ordering with no-auto-rollback, and the 14-item Anti-patterns section are exactly the "prescribe HOW only where deviation is a known failure mode, and say why" style the north star asks for. Most numeric rules trace to documented incidents (`docs/pmos/features/2026-05-13_complete-dev-run-defaults/DESIGN.md`, `2026-05-20_multi-plugin-marketplace`).

## Findings

1. **[R] Phase-name drift inside the file — Track Progress contradicts the headers.** Line 25 says "This skill has 23 phases (Phase 0, 0.5, 1, … 7.5, … 15.5, 16, 16.5, 17, 18)" but the actual headers are `0a`, `7a`, `15a`, `16a` (and the same sentence then says "the substantive worktree-cleanup body lives at Phase 16a", mixing both schemes). The 2026-06-05 renumber commit (`a76a5da`, "standardize phase numbering") converted headers tree-wide but missed this line and `lastrun-schema.md`'s field table (rows for `worktree_disposition` still say "16.5" while line 19 of the same file says "Phase 16a"). Why it matters: this is the first thing the model reads to build its task list; it creates 4 phantom phase names. Fix: mechanical rename in both files. **Renumber safety:** the lettered scheme is now the canonical cross-reference surface — `/feature-sdlc` (`SKILL.md:408`, `soft-gate-lastrun-schema.md:11`) and `/readme` (`SKILL.md:374`) reference "Phase 0a"/"Phase 16a" — so finishing the lettered rename is safe; a *positional* renumber (e.g., deleting the Phase 4 stub and shifting 5→4) is NOT safe without a tree-wide grep.
2. **[R] `rollback-recipes.md` was not updated for v2.41.0 and is now wrong in a destructive path.** Recipe 3's confirmation text says "Worktree at <path> was already removed in Phase 4 — it does NOT come back" and Recipe 5 says "Phase 4 deletes the worktree BEFORE push by design" — the exact behavior v2.41.0 reversed (cleanup moved to Phase 16a *after* push-tag; SKILL.md Anti-pattern #4 explicitly says "do not move the cleanup body back"). The recipes also use unprefixed tags (`git tag -d v<version>`, `git push --force origin v<version>`), contradicting FR-58's mandatory `${plugin_name}/v<version>`. Why it matters: these recipes are loaded precisely when something has already gone wrong; following them produces wrong user messaging and wrong tag operations. Fix: rewrite Recipes 3/5 for the post-v2.41.0 ordering (worktree is normally still present on push failure — that was the point of the move) and prefix all tag commands.
3. **[R] Two sibling skills document complete-dev behavior that doesn't exist.** (a) `/readme` `SKILL.md:374-376` describes "/complete-dev's Phase 0a ask" setting a per-run `readme_update_hook` and a "Phase 7.6 hook" — complete-dev has no such Phase 0a field, no Phase 7.6, and no mention of `readme_update_hook` anywhere (`git log -S readme_update_hook -- skills/complete-dev/` is empty: never landed). The documented `/readme --update` post-release hook can therefore never fire. (b) `/feature-sdlc` `SKILL.md:326` says "`/complete-dev` reads `state.base_drift` and surfaces the gap at merge time" — complete-dev never mentions `base_drift`. Why it matters: phantom contracts rot trust in all three skills; a model executing /readme will hunt for a hook that isn't there. Fix: either land the two small integrations in complete-dev (Phase 3 reads `state.base_drift` and surfaces it before the merge prompt; Phase 0a gains the opt-in row) or delete the references in the sibling skills. Flagging the coupling per the calibration rule — don't fix unilaterally.
4. **[V] The Phase-4-moved-to-16a story is told four times.** Phase 4 stub (¶ + log line), Phase 16a's closing **Note**, Anti-pattern #4, and `lastrun-schema.md` all narrate the same v2.41.0 rationale (~35 lines total). The anti-pattern entry is the right home (it states the failure mode). Fix: keep Anti-pattern #4 in full; reduce Phase 4 to one line ("stub — see Phase 16a / Anti-pattern #4") and drop the 16a Note. Saves ~25 lines with zero behavior change.
5. **[V/P] Per-phase short-circuit boilerplate repeats one rule seven times.** Phases 1, 3, 5, 8, 9 step 5, 14, and 16a each carry a bolded "Short-circuit when Phase 0a confirmed AND …: log `Phase N: <thing> auto-selected via Phase 0a`" paragraph with a bespoke mandated log line. The rule is uniform and already fully specified by `lastrun-schema.md`'s field-reference table + destructive allowlist. Fix: state once in Phase 0a — "When confirmed, every non-destructive prompt below consults `run_defaults` per `reference/lastrun-schema.md` and logs which default it applied; the destructive allowlist always re-prompts" — and delete the seven paragraphs (keep only the two non-obvious conditions inline: Phase 5's signals-changed re-prompt and Phase 3's guard-FAIL override). Saves ~45 lines and removes exact-wording log mandates the model doesn't need.
6. **[P] Phase 0's diff_router branch table re-specifies the script in prose.** ~30 lines (lines 101–123) enumerate four exit-code/stdout patterns that `diff_router.sh` (28 lines) already encodes in its own human-readable output ("Auto-detected --plugin X… Proceed?", "Diff spans… exit 64", "Substrate-only change detected…"). Fix: "Run `diff_router.sh`; relay its message — confirm on auto-detect, abort on the exit-64 refusal, ask which plugin rides a substrate-only change, ask the user to pick when it's silent." The script is the contract; the prose duplication will drift (the FR numbers already live in both).
7. **[G] diff_router's diff source may never fire in the canonical flow.** The script reads `git diff --cached` + `git diff HEAD` — staged/working-tree changes only. At Phase 0 of a normal run (post-/verify, all work committed on the feature branch) both are empty → exit 0 silent → FR-54 fallback prompt. Auto-detect (FR-51, built in `2026-05-20_multi-plugin-marketplace` T5) thus degrades to "ask the user" exactly when it's most useful. Not unsafe (the fallback asks rather than guesses) but worth verifying intent: branch-scope detection probably wants `git diff main...HEAD --name-only` unioned with the working tree.
8. **[Dur] Hardcoded model identity in the commit trailer.** SKILL.md Phase 11 and `commit-style.md` (twice) bake in `Co-Authored-By: Claude Opus 4.7 (1M context)` — already stale (the host environment now mandates a different identity), and exactly the "model-specific, time-sensitive" content the north star says makes skills rot. `commit-style.md` even hedges "If the user specifies a different model identity…use that". Fix: "append the co-author trailer your host environment specifies; fall back to the user's configured identity" — no literal model name anywhere.
9. **[R] Orphaned reference to the retired `/push` skill.** Phase 7: "Detect skill inventory drift (per /push Phase 1a logic)". `/push` is retired (the description itself says "Supersedes the legacy /push"); a reader can't follow this pointer. Fix: delete the parenthetical — the two detection commands that follow are self-contained.
10. **[V] `release-recipes.md` (Phase 7a) is half pipeline-trivia, not release mechanics.** Recipes 1–2 (git-log filters for auto-apply noise) earn their keep; Recipe 3 (no mid-pipeline rebase) is a real anti-pattern but belongs to /feature-sdlc's resume contract, and Recipe 4 is a version-pinned "--help quick reference for the v2.34.0 flag surface" listing flags of *five other skills* — pure staleness fuel with no release-notes function. Fix: delete Recipe 4; consider moving Recipe 3 to feature-sdlc; fold 1–2 into two sentences inline and delete the file + the one-paragraph Phase 7a (a phase that exists only to point at a reference is a smell).
11. **[Ph] 23 tracked phases is procedure-for-procedure's-sake at the task-tracking layer.** The phase *headers* mostly map to real, order-sensitive release steps (the Phase 13→15→16 tag/push ordering is explicitly load-bearing — Anti-pattern #6), so the spine should stay. But creating 23 todo items, including a no-op stub (4) and three conditional sub-phases (15a, 16a, 0a), is ceremony. Fix: track ~8 groups (preflight+defaults / merge / deploy-detect+learnings / changelog+bump / commit / tag+summary / push(+retry) / cleanup+finalize+lastrun) while keeping the named phases as headings.
12. **[S] The lastrun pattern is duplicated, not shared.** `reference/lastrun-schema.md` and feature-sdlc's `reference/soft-gate-lastrun-schema.md` (which says "Mirrors `/complete-dev` Phase 0a") independently specify the same machinery: read contract (absent/malformed/CLI-override), atomic temp-then-rename write, "failed run never writes", consolidated Confirm-all/Edit-loop/Cancel prompt, destructive allowlist. Fix (structural): extract a `_shared/lastrun-confirm.md` protocol the two schemas instantiate with their field tables. Two copies will drift the same way the phase names did. Elsewhere substrate use is good: Phase 18 → `_shared/learnings-capture.md`, Phase 16a → `_shared/canonical-path.md`, inlined non-interactive block per the W14 contract.
13. **[R] Two phases named "Capture learnings".** Phase 6 (propose repo learnings from the branch diff into CLAUDE.md/AGENTS.md) and Phase 18 (skill-improvement learnings via `_shared/learnings-capture.md`) share a title but do different things. Fix: rename Phase 6 to "Propose repo learnings" — one-word change, removes a real reader stumble.
14. **[X] `${CLAUDE_PLUGIN_ROOT}` has no cross-platform fallback.** The Platform Adaptation section is otherwise above-average for this repo (explicit non-prompt defaults, `ExitWorktree` no-op fallback FR-CD05, TaskCreate degradation), but the only scripted step — `bash ${CLAUDE_PLUGIN_ROOT}/skills/complete-dev/scripts/diff_router.sh` — assumes a Claude-Code env var. Fix: add one line ("resolve the script relative to this SKILL.md when the env var is unset") or note the script's logic is reproducible inline.
15. **[F] `--reset-defaults` is barely earning a flag.** Its interactive effect is reachable via Phase 0a's "Edit one or more" path; its non-interactive use ("ignore drifted lastrun") is the only real case. Keep if non-interactive automation needs it; otherwise natural language ("ignore my last run's defaults") covers it.
