# feature-sdlc — review

**Grade:** C (the architecture is right and much machinery is earned, but incremental edits have left the file contradicting itself in at least six places, and the prose has drifted from "instructions" to "spec transcription" — a meaningful rewrite would pay off)
**Size:** SKILL.md 909 lines (~880 excluding non-interactive block); references 15 files / 1,976 lines; target ~400 lines for SKILL.md (+ references trimmed to ~1,400).

## TL;DR

- **Biggest win available:** fix the internal contradictions (schema v4-vs-v5, 39-vs-41 checks, `both`-sidecar retired-vs-emitted, fuzzy-detection rule-vs-example, failure-dialog stale phase lists) and strip the FR/D/G/T/W/E spec-ID citations — they reference spec HTML in this repo's `docs/`, which installed plugin users don't even have. That alone recovers ~150 lines and most of the readability.
- **Biggest risk in current design:** the resume/state contract — the skill's crown jewel — is the thing most damaged by drift. SKILL.md Phase 0b says `schema_version > 4 → abort` while `state-schema.md` says v5 is current and prototype mode *requires* v5. Per the letter, resuming a prototype-mode run aborts. The contract that "two files drifting apart is the one failure this pair is built to prevent" (skill-patterns.md closing line) has already failed inside its own skill.
- **Done well, keep:** orchestrator-with-thin-aliases is the right architecture (see Finding 1); the hardness-driven dialog construction (`failure-dialog.md`'s construction algorithm — option lists derived from one `hardness` tag, "never per-phase if/else") is genuinely elegant; the EnterWorktree handoff fallback is a model cross-platform degradation; the [D]/[J] split with quote-grounded fails (unverifiable fail → pass) is a real anti-hallucination invention.

## Findings

1. **[Ph/R] Architecture verdict: orchestrator-with-thin-aliases is right; the mode-conditionals woven through every phase are not.** The four modes share ~80% of the machinery (worktree, slug, state, resume, gates, child dispatch, summary) — separate skills would quadruplicate it, and `state-schema.md`'s mode-conditional `phases[]` already gives resume a single cursor. Keep the architecture. The cost is that *every* phase section now carries 1–4 mode-conditional paragraphs ("In `prototype` mode: skip wholesale. Log …"), and prototype-mode ordering is specified **three times**: per-phase skip directives, the `## Prototype-mode phase ordering` shim, and `state-schema.md`'s v5 `phases[]` list. Fix: make the mode × phase table (lines 38–51) + `state-schema.md`'s per-mode `phases[]` the *only* statement of membership and order; delete the per-phase "In prototype mode: skip wholesale" paragraphs (8 of them) and the ordering shim's 4-step restatement. One source, ~80 lines saved, and the next mode added doesn't require touching every phase section.

2. **[R] Resume contract self-contradiction — schema v4 vs v5.** SKILL.md Phase 0b: "`state.schema_version > 4` → abort … `< 4` → run the v1→v2→v3→v4 migration chain … `== 4` → no migration"; Phase 1: "Write `schema_version: 4` (FR-10)". But `reference/state-schema.md` line 9: "`schema_version: 5` is the current version (added with the `/feature-sdlc prototype` mode)", and SKILL.md's own prototype-ordering section cites "`state-schema.md` schema v5 prototype-mode block". A prototype-mode state file (v5) read by Phase 0b as written aborts as "from a newer version". Fix: SKILL.md should not state version numbers at all — "current version, migration policy, and chain live in `reference/state-schema.md`" — so the next bump can't drift. (This is the prototype-sdlc feature, `docs/pmos/features/2026-05-24_prototype-sdlc-skill/`, updating the reference but not the orchestrator body.)

3. **[R] `output_format=both` is simultaneously retired and live.** Line 486: "Mixed-format sidecar (FR-12.1): **retired** — `output_format=both` is treated as `html` until a future feature re-introduces MD export." Yet lines 378, 506, 824, 826, 832, and Anti-pattern 6 still instruct emitting `.md` sidecars "when `output_format=both`", and the `--format` flag still advertises `both` as valid. An agent reading top-to-bottom gets opposite instructions 20 lines apart. Fix: pick one (retire fully → drop `both` from the flag and the six sidecar clauses, or un-retire); either way it's a 10-minute mechanical edit.

4. **[P] The token-1 disambiguation grammar (FR-02, line 83) is over-prescribed and breaks the skill's own aliases.** The grammar says token 1 is a subcommand selector only when (a) sole token, (b) next token is a recognised flag, or (c) remainder is exactly one quoted argument. `prototype-sdlc/SKILL.md` documents forwarding `/prototype-sdlc list` → `/feature-sdlc prototype list` — which satisfies none of (a)/(b)/(c), so per the letter it dispatches as **feature mode with seed "prototype list"** (and even quoted, (c) yields a prototype run with seed "list", not the list view — `list` is a token-1-only selector). Unquoted multi-word seeds via `/skill-sdlc` mis-dispatch the same way. The grammar also omits `--reset-defaults` from its recognised-flag list while the `argument-hint` advertises it. A capable model handles "is `skill` a subcommand here?" correctly from intent; the rigid grammar is what introduces the bugs. Fix: replace the three-condition grammar with the principle ("`skill`/`prototype`/`list` as the first token select a mode; never infer mode from seed *content* — a bare description is always feature mode") plus the two genuinely ambiguous worked examples already present. Keep Anti-pattern 13 (no mode-sniffing) — that's the real failure mode, per the skill-mode spec (`docs/pmos/features/2026-05-11_feature-sdlc-skill-mode/02_spec.html`).

5. **[R] `reference/failure-dialog.md` phase lists are stale.** "Soft phases: `grill`, `msf-req`, `creativity`, `wireframes`, `prototype`, `simulate-spec`" — `msf-req`/`simulate-spec` were folded away in v2.34.0; `ideate` and `retro` (both soft) are missing; the hard list omits `feedback-triage` and `skill-eval`; the infra list omits `skill-tier-resolve`. The file's own line 13 states the right design — "Option lists are constructed from the phase's `hardness` tag (per `state-schema.md`) … never per-phase if/else" — then enumerates a contradictory hand-maintained list anyway. Fix: delete both enumerations; the hardness column in `state-schema.md` is the single source.

6. **[G] skill-eval.md has count-drift in its own header — the exact failure the bijection machinery exists to prevent.** Line 3: "39 binary pass/fail checks … the 20 `[D]` … the 19 `[J]`"; line 11: "MUST keep the total ≥35"; Totals section line 201: "41 checks — 21 `[D]` … 20 `[J]` … must keep the total ≥37". The §G addition (release-prereqs scope) updated the bottom and the script header (which says 21/20) but not the top. The skill's own frontmatter description and the repo CLAUDE.md still say "39". Fix: state counts exactly once (in Totals), make line 3 say "the binary skill-quality rubric — counts in `## Totals`", and have `--selftest` assert the prose count matches the table count so this class of drift becomes deterministic-fail.

7. **[G] The rubric: deterministic half earns its keep; about a third of the judge half encodes taste as binary law.** Against Pocock's `write-a-skill` (117 lines, 6-item advisory checklist): the comparison is not 41-vs-6 — it's *which checks catch failures a model actually commits*. The 21 `[D]` checks are nearly free (one bash run), catch real loader/install failures (`a-name-matches-dir` is a loader requirement; `c-portable-paths` catches the `/Users/alice` install-breaker; `g-plan-grep-clean` backstops a documented merge-damage failure mode with origin in §G's why), and are repeatable. Keep them hard. The 20 `[J]` checks split: ~12 are judgeable against evidence (`b-desc-has-when`, `b-desc-no-step-list`, `f-cc-argument-hint-matches`, `c-progressive-disclosure`) — keep; ~6 are vibes forced into pass/fail (`b-desc-pushy`, `d-imperative-form`, `d-explain-why`, `c-context-economy`, `d-examples-quality`, `a-name-verb-or-gerund`) — at temperature 0 these still flip across model versions, and a binary fail on "not pushy enough" triggers a full remediation iteration (plan addendum + /execute re-run + re-score) for a wording nit. Fix: keep them in skill-patterns.md as guidance, demote them in skill-eval.md to advisory findings (reported, never iteration-triggering) — the 2-iteration loop then spends its budget on checks with stable ground truth. The quote-grounding rule (unverifiable fail → pass) and the writer/reviewer separation (D10) are excellent — keep verbatim. The 2-iteration cap has documented origin (FR-45–47 + net-worse guard + "Restore iteration 1", `docs/pmos/features/2026-05-11_feature-sdlc-skill-mode/02_spec.html`) and prevents an unbounded fix-regress loop; keep.

8. **[Ph] Phase naming is insertion history, not structure: 0, 0a, 0b, 0c, 0d, 0e, 1, 1a, 2, 2a, 3, 3a, 3b, 3c, 4–6, 6a, 7, 8, 8a, 9, 10.** Six sub-lettered phase-0 stages, and "Phase 1a" is logged as "phase 1.5" in its own log lines (line 515: `[orchestrator] phase 1.5 ideate:` under the heading "Phase 1a") — the rename didn't reach the log strings. The lettering also forces prose like "Phase 6a (skill-eval) is light … no checkpoint fires before 6a". Fix at next structural edit: renumber to plain 1..N per mode (the state-schema `id` strings — `feedback-triage`, `skill-eval` — are already the stable identifiers; the display numbers are free to change), and group phase-0's six stages as one "Phase 0: dispatch + workspace" section with subsections.

9. **[V] The FR/NFR/D/G/T/W/E citation lattice is the single largest verbosity and durability tax.** Nearly every paragraph carries spec IDs from at least five different feature specs (FR-W04, FR-PA05, FR-R06, FR-L01–L08, FR-PSDLC-04, FR-TIER-SCOPE, spec §15 G1–G15, D10–D34, T12b, W3/W17, E7–E20). These point at HTML files under `docs/pmos/features/` in *this repo* — an installed plugin user has no such files, so the citations are dangling for the skill's actual audience, and they freeze the prose against rewriting (every edit must preserve ID references). Pocock principle 4 (durable, no doc-coupling) is the right call here. Fix: keep IDs inside `docs/pmos/features/` artifacts where they belong; in SKILL.md keep only the *why* sentences (which are good — e.g., base-drift's "a worktree branched off a stale local main produces silent damage at merge time" survives any spec renumbering). Estimated saving: 100–150 lines and most of the legal-contract register.

10. **[S] FR-50/51/52 reviewer-validation logic is duplicated, not substrate.** The chrome-strip → inline-HTML → `sections_found` set-equality → ≥40-char quote substring-grep contract appears in Phase 2a (grill), Phase 6a step 3 (FR-44), and — per repo memory — again in `/artifact`, `/spec`, `/wireframes`, `/prototype` reviewer loops. `_shared/` already hosts the html-authoring assets it depends on (`chrome-strip.js`, `sections.json`). Fix: extract `_shared/reviewer-validation.md` (prompt template + the two hard-fail checks); each call site becomes 3 lines. Same story for the soft-gate-lastrun pattern, which the file itself says "mirrors `/complete-dev` Phase 0a" — two parallel implementations of remember-my-gate-answers.

11. **[F] Three overlapping mechanisms exist for "stop asking me about optional gates":** `--minimal` (force-skips 4 gates), Phase 0e lastrun defaults (remembers 5 gates), and `--no-ideate` (pre-skips 1 gate that *also* has a deterministic auto-skip classifier and a Skip option at the gate itself). Each was added in a different release (W17, W3, 2.52.0) without consolidating. Fix: lastrun defaults subsume the other two — `--minimal` becomes "answer all soft gates skip this run" sugar within 0e; `--no-ideate` deletes (the classifier + gate already cover it).

12. **[R] Stale `.comments.json` sidecar reference.** "Apply comment-resolver edit" (line 875) describes `/comments resolve` "walking open threads in either orchestrator artifact's `.comments.json` sidecar" — the sidecar contract was retired in v2.58.0 for the inline `pmos-comments` block (repo CLAUDE.md, `2026-05-28_inline-html-artifacts`). One-line fix.

13. **[R] `fuzzy-idea-detection.md` worked example contradicts its own rules.** Rules apply in order, first hit wins; rule 4: "wordcount < 20 → fuzzy". Worked-example row 5: `"Fix the BSD-awk \b portability bug in scripts/rubric.sh"` — 8 words, no markers, no doc — is tabulated as "rule 5 → **formed**". Rule 4 fires first; per the rules it's fuzzy. Either the rules need a "concrete-artifact-path → formed" carve-out (probably the intent — the example annotation says "concrete bug") or the example row is wrong. As written, an agent gets license to pick whichever outcome it prefers.

14. **[X] Cross-platform posture is genuinely good — keep it.** EnterWorktree failure degrades to a byte-precise handoff block + grep-able `Status: handoff-required` line (any wrapper can automate it); AskUserQuestion degrades per `_shared/interactive-prompts.md`; TaskCreate degradation names the `00_pipeline` artifact as fallback; the worktree itself falls back to `--no-worktree`. The one gap: `realpath` is assumed in three places with the canonical-path fallback cited only once (Phase 0a Step 3); Phase 0b's drift check should cite the same fallback.

15. **[V] The `list` subcommand spec (FR-L01–L07, lines 98–118) is 20 numbered lines for "show a table of feat/* worktrees sorted by last_updated, degrade gracefully on missing state."** The legacy-version annotation and missing-path handling are worth one line each; the rest (column order, tie-breaking by slug, exact empty-state string) is the model's job. Target: 6 lines.
