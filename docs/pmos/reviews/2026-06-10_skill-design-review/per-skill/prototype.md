# prototype — review

**Grade:** C (would pay off meaningfully from a rewrite — the bones are good, the body triple-states its contracts and has lost sight of what a prototype is for)
**Size:** SKILL.md 677 lines (~648 excluding non-interactive block); references 9 files / 1,344 lines (142 of which are a superseded file still shipping) + SMOKE-TEST.md 49 lines + prototype.css 297 lines; target ~280–320 lines SKILL.md, 8 reference files / ~1,100 lines.

## TL;DR

- **Biggest win available:** the `x-interaction` contract, the Babel/IIFE failure-mode explanation, and the load-order rules are each stated 2–3 times across SKILL.md + reference files. Stating each once (in a reference, cited from phases) plus pruning the 27-item Anti-Patterns list (which restates phase rules) takes SKILL.md from 677 to ~300 lines with zero behavior change.
- **Biggest risk in current design:** the skill has drifted from "prototype answers a question" to "artifact passes its gates." It never asks what the prototype is supposed to validate, mandates full screen coverage as the contract, runs a production-grade a11y/visual rubric on throwaway code, and ends with file inventory rather than a captured verdict. Pocock's framing — the question decides the shape — is absent; scope, review depth, and friction journeys could all be derived from a stated question instead of being fixed-maximum.
- **Done well, keep:** Phase 5d runtime smoke is the single highest-value gate in the skill (a structurally-perfect file that throws `Identifier 'Button' has already been declared` on load is exactly the failure greps can't catch — the Babel shared-global-scope rationale is the model Pocock case of prescribing HOW because deviation is a known failure mode). Reference-file factoring (templates out of SKILL.md), honest degraded-mode banners, and the cite-don't-restate posture for `_shared/apply-edit-at-anchor.md` are all genuinely good.

## Findings

1. **[P/V] The skill doesn't know what question its prototypes answer — it generates a maximal artifact regardless.** Pocock's prototype skill opens with "A prototype is **throwaway code that answers a question**. The question decides the shape" and forces stating the question, building the minimum, and capturing the answer. This skill's implicit question ("will stakeholders buy this flow before we commit to /spec?") is legitimate and *different* from Pocock's engineer-probe — fidelity and a working demo are load-bearing here. But the question is never made explicit anywhere: Phase 1's confirmation asks *which screens and devices*, never *what we're trying to learn*; "Do NOT cherry-pick screens — full coverage is the contract" (Anti-Patterns) inverts the prototype logic (coverage should follow the question, e.g. "is the 4-step onboarding too heavy?" needs 4 screens, not all 14); Phase 10's handoff captures file inventory (folder, devices, entity counts) but no verdict. Phase 8 findings-routing and the friction pass are the question-shaped 20% — build around them. **Fix:** add 3 lines to Phase 1's confirm gate: "State what this prototype must validate (1–2 sentences: which journeys/decisions are at stake). Scope, review depth, and friction journeys follow from this." And add one line to Phase 10's handoff template: `Verdict: <what the prototype validated / invalidated; what changed upstream>`. No machinery needed.
2. **[V] Contracts stated in triplicate.** (a) The `x-interaction` mapping appears as a 9-bullet blockquote in Phase 4c (lines 267–279), again as an 8-item checklist in Phase 6 (lines 411–419), and again in the Anti-Patterns list (line 610) — three paraphrases of one contract that can drift apart independently. (b) The Babel IIFE rationale appears in Phase 5d prose (line 358), `device-html-template.md` §0 + two inline code comments, and `components-template.md` "File structure". (c) CSS/JS load order appears in Phase 5b, `device-html-template.md` rule 1, Phase 4d, and Anti-Patterns (line 613). **Fix:** move the x-interaction contract table to `reference/` (or into `eval-rubric.md`, which the reviewer already loads) and have 4c/6 cite it; keep the IIFE rationale only in `device-html-template.md`; state load order once there.
3. **[V/R] The Anti-Patterns section (27 items, lines 588–613) is mostly an index of rules already stated in the phases** — ≤2 loops, ≤5 journeys, tier gate, Lorem ipsum, x-interaction-is-a-contract, load order, overlay reuse, no-bootstrap each appear both in their phase and here. A reader can't tell which copy is normative. **Fix:** cut to the ~6 items that are genuinely cross-phase judgment calls (no build steps; no external resources/`@import`; don't re-run PSYCH/MSF; don't write design-system facts to the workstream; mock data must be domain-real; don't silently self-fix high-severity findings) and delete the restatements.
4. **[S] Tier detection/gating logic (Phase 2a) is duplicated across 9 skills** — requirements, spec, plan, wireframes, prototype, simulate-spec, msf-req, creativity, feature-sdlc each carry their own "read tier from req doc / ask / gate" prose (grep "Tier 1\|Tier 2\|Tier 3" across `skills/*/SKILL.md`). Each restates the tier definitions slightly differently. **Fix (structural, cross-skill):** extract `_shared/tier-gate.md` (definitions + read-from-doc + ask-fallback); per-skill text shrinks to the gating policy ("Tier 1 stop, Tier 2 ask, Tier 3 mandatory" — 3 lines here).
5. **[S] The reviewer-subagent ≤2-loop refinement structure is the same machine in `/wireframes` Phase 4, `/prototype` Phase 6, and `/diagram`** — dispatch reviewer(s) with rubric, severity-routed fixes, review-log comment, hard cap 2, stop-early-on-clean. Each skill restates the loop mechanics. Notably `/wireframes` evolved a rigor ladder (high/medium/low — single cross-file reviewer, no second loop for ≤6 files) that `/prototype` never inherited, so a 2-screen Tier-2 prototype pays the same full per-file 2-loop cost as a 14-screen Tier-3 one. The cap itself is original design, not a scar (present in `docs/pmos/features/2026-04-30_prototype-skill/02_spec.md` line 152 with "diminishing returns" rationale). **Fix:** extract `_shared/reviewer-loop.md` (loop mechanics + cap rationale); adopt wireframes' rigor tiers here, keyed off the Phase 1 question/scope.
6. **[R] Orphaned references — direct evidence of insertion-without-rethink.** (a) Line 297: "4d runs inline (it's small) after **1.5** confirms the overlay" — there is no Phase 1.5; it means Phase 1a (pre-rename residue). (b) Line 175: "Re-run with `--force` to override" — `--force` appears nowhere else: not in `argument-hint`, no parsing, no defined behavior. (c) Line 33: "This skill has 14 phases" counts 0, 1, 1a, 2–12, but the unnumbered "Apply comment-resolver edit" section (line 617) calls itself "this phase," making the count ambiguous. **Fix:** s/1.5/1a/; either document `--force` in argument-hint + Phase 2 or drop the mention; retitle the resolver section "entrypoint," not phase.
7. **[X] The Phase 4a copy command is broken as written.** Line 249: `cp "${CLAUDE_PLUGIN_ROOT:-$HOME/.claude-personal/plugins/cache/pmos-toolkit/pmos-toolkit/*/}skills/prototype/assets/prototype.css" …` — the glob `*` is inside double quotes and will never expand; the fallback default also hardcodes a Claude-Code-personal cache layout, contradicting the Platform Adaptation section. Only the "if the copy fails, Read and Write it" escape hatch saves the phase. **Fix:** make Read-then-Write the primary instruction (it's portable and always works); drop the shell incantation entirely.
8. **[Ph] Phase 1a (decimal) is a self-confessed insertion smell** — the file itself apologizes: "Decimal phase number is intentional — Phase 2 onward keeps existing numbering so external references … still resolve" (line 108). The stated reason (external reference stability) is real but weak — the only hard external coupling found is `/feature-sdlc`'s stage orchestration and prose mentions; a one-time renumber (1a→2, shifting 2–12 to 3–13) costs one grep across the repo. Combined with finding 6's stale "1.5", decimal numbering is already causing the drift it was meant to prevent. **Fix (quick):** renumber in the next touch; until then, at minimum fix the internal references.
9. **[P] The `.deviations.md` ledger (Phase 3/4b/4c, lines 216–225) over-engineers a process-purity concern.** A mandated 3-field markdown schema, "Silent deviation is a contract violation," and an index-footer renderer — all to record that the model generated mock data inline instead of via subagent. The user-visible failure it guards (claiming subagent rigor that didn't happen) is real but is fully served by one line: "If you bypass a prescribed subagent, say so in the index footer with a one-line reason." The schema, the per-phase repetition of the exception criteria (stated identically in 3, 4b, 4c), and the "contract violation" framing are defensive over-specification. **Fix:** collapse to one sentence in a single place; keep the footer surfacing.
10. **[G] Phase 5c structural greps are ~80% redundant once Phase 5d runtime smoke runs.** Script-tag presence, screen-count mismatch, and missing inline data all manifest as render failures or console errors that 5d catches live (and 5d is mandatory). Only the forbidden-pattern grep (Lorem ipsum / `console.log` / TODO) catches things a successful render won't. The 5c block also has a latent bug: `grep -c 'react@18\|babel/standalone\|…'` counts matching *lines*, not distinct matches, so "Expected: ≥6" passes/fails incorrectly when tags share lines. **Fix:** fold 5c down to the forbidden-pattern grep + screen-count check; let 5d own runtime integrity. When Playwright is unavailable, the full 5c grep set is the fallback — say that in 5d's degraded branch.
11. **[V/P] Phase 9b's serve recipe is 30 lines of bash ritual for "start a static server and verify it serves the right directory."** The subshell-cwd gotcha is a genuine scar worth keeping — as a 2-line warning ("bind cwd in the same subshell as the server; a separate `cd` then `npx http-server` serves `~`"). The hardcoded `PORT=8765`, the kill/retry procedure, and the second optional curl are micromanagement of a task any capable model does. **Fix:** keep the scar + "smoke the URL before announcing it"; delete the rest (~20 lines).
12. **[R/G] `reference/styles-derivation.md` (142 lines) is superseded but still ships and is still linked.** The banner is honest, but SKILL.md references it twice (Phase 4d, Anti-Patterns line 612) only to say "don't use it." Dead weight in the loaded skill, and the legacy `house-style.json` content below the banner can still be retrieved and acted on by a confused model. **Fix:** delete the file (history lives in git and in `docs/pmos/features/2026-05-02_design-md-prototype-integration/`); drop both mentions.
13. **[X] Version-history annotations violate durability** — "Phase 1a was added in v2.8.0 … Phase 5d runtime smoke added in v2.9.0" (line 33), "superseded as of v2.8.0" (line 295), "added in v2.9.0 after a retro" (`eval-rubric.md` F-group, line 87 — the retro citation is valuable, the version number isn't). North-star #4: no time-sensitive info in skill bodies; changelog facts belong in the changelog. **Fix:** strip version numbers; keep the F-group origin story (it explains *why*, which is durable).
14. **[G] The Phase 6 production-grade rubric mismatches the artifact's throwaway nature in two groups.** A (Accessibility: contrast ≥4.5:1, aria-labels, tab order) and parts of V (atom-usage policing) audit qualities stakeholders walking a demo will never exercise, at the cost of 2 review loops × N devices. Counterpoint kept in view: A2/A5 (focus visible, keyboard CTAs) overlap the x-interaction contract, and F (Field-Earns-Its-Place) is the rubric's best group — genuinely question-shaped, with a cited retro origin. **Fix:** demote A1/A3/A4 to advisory (reviewer may note, never drives a loop); keep I/J/M/F/R loop-driving. This shortens loops without losing demo-fatal coverage.
15. **[S] DESIGN.md machinery lives in `/wireframes`' reference dir but is consumed by three skills.** `design-artifact-resolver.md` reaches into `wireframes/reference/design-md-resolver.md` and `design-md-to-css.md`; `/verify` also consumes the same artifacts. Cross-skill reference reaching works but couples `/prototype` to `/wireframes`' internal layout — a rename there silently breaks here, and the repo's own substrate convention (`_shared/` holds cross-skill logic) says this is substrate. **Fix (structural):** move the DESIGN.md resolver/generator family to `_shared/design-md/`; leave pointer stubs for one release.
