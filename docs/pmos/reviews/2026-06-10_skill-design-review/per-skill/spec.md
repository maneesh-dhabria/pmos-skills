# spec — review

**Grade:** C (would a thoughtful reviewer call this well-designed for its job? The workflow design is genuinely good; the file carrying it is ~60% restated contracts, feature-spec residue, and prompt choreography — a meaningful rewrite pays off without changing user-visible behavior.)
**Size:** SKILL.md 705 lines (676 excluding non-interactive block); references 1 file / 339 lines (`reference/spec-templates.md`); plus 1 script (`scripts/apply-edit-at-anchor.js`, 179 lines) and 4 test files. Target ~280 lines (≈250 excluding the non-interactive block), with folded-phase mechanics moved to a second reference file.

## TL;DR

- **Biggest win available:** ~400 of 676 lines can go without losing any enforced behavior — strip feature-spec residue (57 FR-refs plus D/T/W/M decision IDs), collapse the Phase 5 emit block and anchor algorithm to substrate citations, move Phase 6a/6b orchestration internals to a reference file, and replace the diagram dispatch state machine and AskUserQuestion choreography with intent statements.
- **Biggest risk in current design:** Phase 6a still operates on `02_spec.md` — a file Phase 5 no longer writes (HTML-primary since v2.58.0). The FR-66 clobber guard (`git status --porcelain 02_spec.md`) is a silent no-op today, and the auto-apply loop is told to edit the wrong artifact. Incremental edits have left the skill contradicting both itself (heading-`id` vs `{#anchor}` token) and its substrate (`<img>` vs inline-SVG-only).
- **Done well, keep:** Phase 4's verification-plan sketch is this skill's "This is the skill" moment — a cheap chat-visible gate that catches under-thought verification before the spec is written. The tier system, the role-ordering rationale in Phase 3 ("architecture constrains schema constrains APIs…"), and the substrate citizenship for resolve-input / sim-spec-heuristics / structured-ask-edge-cases are exactly the right shape.

### Special attention: vs Pocock's to-prd

`to-prd` produces a comparable artifact (PRD with decision log, testing decisions, out-of-scope) in 75 lines: 3 process steps + inline template + one durable rule ("no file paths — they outdate"). What /spec legitimately carries that to-prd doesn't: pipeline plumbing (settings, feature folders, backlog bridge ~30 lines), tiering (~28), the non-interactive contract (29, lint-enforced), the HTML/comments emit contract (worth ~25 lines of citations), folded-phase gates (~30 trimmed), and an interview+review workflow that is real added value (~75 as intent). That justifies **~3.5–4× to-prd, i.e. ~280 lines. The file is 9.4×.** The ~400-line delta decomposes as: restated substrate contracts (~90), Phase 6a/6b orchestration internals + FR/D/T/W ID residue (~110), AskUserQuestion choreography (~60), triple-stated rules incl. the 18-bullet anti-pattern list (~50), the diagram dispatch state machine (~31), misc defensive over-specification (~60). Verdict: **~40% domain-necessary, ~60% distrust-of-model plus leaked feature-spec prose.** Notably, to-prd's one durability rule ("don't include file paths, they outdate") is the exact failure mode /spec's own body exhibits.

## Findings

1. **[R] Phase 6a targets a retired artifact — `02_spec.md`, 4 references (lines 487–521).** The substrate delegation, the FR-66 pre-apply guard (`git status --porcelain 02_spec.md`), and the auto-apply instructions all name `02_spec.md`; Phase 5 writes `02_spec.html` and the MD sidecar was retired in v2.58.0 (`2026-05-28_inline-html-artifacts`). Phase 6a was authored against the pre-HTML world (`2026-05-10_pipeline-consolidation`) and never migrated. Today the clobber guard always passes (file absent) and a literal-minded run would attempt edits to a nonexistent file. **Fix:** replace with `02_spec.html` or, better, "the artifact resolved by `_shared/resolve-input.md` phase=spec". One-line-per-site quick win.

2. **[V][P] Feature-spec residue makes the skill read like its own changelog.** 57 `FR-`/`NFR-` citations plus opaque IDs — D2, D5, D11, D14, D15, D16, D35, M1, T5, T6, T10, T12b, W3, P3 — from at least four different feature folders, indistinguishable from each other in-file. Lines like "Replaces the obsolete `/feature-sdlc` Phase 6 gate (W3)", "(resolves spec Open Question #1; recorded in /plan plan §Decision Log P3)", "real prose rewriting is deferred to T12+" are commit-message material, not instructions; a reader (or a non-Claude agent) cannot act on them. This is the Pocock durability rule inverted. **Fix:** strip the IDs; where provenance matters, one citation per section to the feature folder (e.g., "origin: `2026-05-09_html-artifacts`"). Saves ~60 lines and most of the legal-contract register.

3. **[S][V] Phase 5's emit contract restates the substrate — and the same ~46-line block is duplicated across 9 skills.** The asset-copy table (`cp -n`/`install -m 0755`, lines 263–275), cache-bust, atomic write, meta-tag, index-regen prose appears near-identically in artifact, diagram, ideate, plan, prototype, requirements, survey-design, wireframes (grep `install -m 0755`). The contract is enforced by `_shared/html-authoring/tests/fanout.test.sh` — which exercises `render.js`, not SKILL.md prose — and `scripts/check-comments-coverage.sh` only greps each skill tree for a `comments.js` mention. So collapsing to ~6 lines ("Emit per `_shared/html-authoring/README.md` + `conventions.md`: atomic write of `02_spec.html` + `.sections.json`, copy `assets/*` incl. `comments.js` idempotently, set `pmos_skill=spec`, regenerate `index.html`") is safe against both gates. **Fix:** add an emit-checklist section to the html-authoring substrate once; 9 skills cite it. ~40 lines saved here, ~350 repo-wide.

4. **[R] The heading-anchor algorithm is stated three times and self-contradicts.** "Heading IDs" (line 304) says compute HTML `id`s per `conventions.md` §3; "Anchor Emission Rule" (lines 342–363) restates the identical 4-step algorithm but instructs emitting a **trailing `{#kebab-anchor}` markdown token** — a pre-HTML mechanism from the `2026-05-08_plan-skill-redesign` era; `conventions.md` §3 is the canonical copy. Two more substrate conflicts: SKILL.md line 308 says the spec references diagram SVGs "via `<img>` (or inlines it)" while `conventions.md` §4 forbids `<img>` for diagrams; SKILL.md line 283 says no on-disk `_index.json` (FR-41) while `conventions.md` §10 still mandates emitting one. **Fix:** delete the Anchor Emission Rule section entirely (cite §3 once), align the `<img>` sentence to "inline the SVG", and update `conventions.md` §10 — the drift is evidence for finding 3's consolidation.

5. **[P][G] The diagram dispatch (lines 306–336) is a 31-line state machine the model can't faithfully execute.** `diagram_subagent_state {elapsed_s, attempts, cap_hit}` is a wall-clock accumulator an LLM cannot reliably maintain; 300s timeouts and the 1800s cap (origin: `2026-05-09_html-artifacts`) encode a real failure mode — runs stalling on mermaid-cli/network — but the encoding is pseudo-code, not instruction. Worse, the closing claim "This pattern is canonical across `/spec` and `/plan`" is stale: `/plan`'s SKILL.md contains zero diagram mentions post-redesign. **Fix:** 8 lines of intent — "dispatch `/pmos-toolkit:diagram` as a blocking subagent; up to 3 attempts per diagram; if it keeps failing or diagram generation is dominating the session, author the SVG inline and say so in the figcaption (provenance matters to reviewers)". Keep the figcaption-provenance requirement; drop the accumulator.

6. **[P] Phase 6's AskUserQuestion choreography (~60 of 103 lines) micromanages presentation.** Exact severity-tag wording with two worked examples, ≤4-per-batch arithmetic, exact option labels and their descriptions, a numeric definition of "structural" ("invalidates three or more sections"). The load-bearing ideas — tag findings `[Blocker]/[Should-fix]/[Nit]`, offer Fix/Modify/Skip/Defer dispositions, never dump findings as a prose wall, escalate re-architecting findings out of the inline-edit flow — survive in ~10 lines plus the existing `_shared/structured-ask-edge-cases.md` citation. The model knows how to batch questions.

7. **[V] Rules are triple-stated.** Phase 6's 7-item structural checklist ≈ exit-checklist items 1–9 ≈ "Document Guidelines" bullets ≈ half of the 18-bullet Anti-Patterns section (e.g., "one diagram per flow" appears 3×; "don't promote without user confirmation" 3×; "Open Questions forbidden" 3×). **Fix:** the universal exit checklist is the single rubric; the structural checklist becomes "check the exit checklist each loop"; anti-pattern bullets that merely negate an already-stated rule are deleted (keep the 3–4 that name non-obvious failure modes, e.g. the Tier-2 industry-research skip and over-specification-is-an-anti-pattern). ~45 lines.

8. **[F] Flag drift.** `--skip-folded-arch` and `--msf-auto-apply-threshold` exist only in phase bodies, absent from the frontmatter `argument-hint` — undiscoverable. `--format` still advertises `both`, which line 285 declares retired ("treated as `html`"). `--msf-auto-apply-threshold` is a misnomer in /spec — here it tunes the *sim-spec* apply threshold; the "msf" name is shared with /requirements and /wireframes (cross-skill coupling — rename carefully or alias). See Flags inventory.

9. **[Ph] Phases 0a/6a/6b are insertion smells, and 6a/6b (~98 lines) are orchestration internals, not skill instructions.** `state.yaml.phases.spec.folded_phase_failures[]` dedup rules, per-finding commit-body trailer formats (`Depends-on: P<M>`), "commits-as-state is the resume cursor" — this is /feature-sdlc plumbing that leaked in from the originating feature plans (`2026-05-10_pipeline-consolidation`, `2026-05-28_architecture-in-feature-sdlc`). **Fix:** keep the tier gates, escape flags, and the one principle that matters (folded-phase failures are advisory — warn and continue, never halt /spec) in ~15 lines each; move dispatch/commit/state mechanics to `reference/folded-phases.md`, loaded only when the phase actually runs. Also fixes the incoherence that Phase 6a's heading says "Tier 1/2 optional" while its body specifies only T1 (skip unless opted in) and T3 (default-on) — T2 is unstated.

10. **[S] The comment-resolver section cites the contract, then restates it anyway.** Lines 668: "this phase MUST cite that file rather than restate the contract" (NFR-08) — followed by ~35 lines restating resolution order, the closed error enum, and the idempotency JSON shape, all of which live in `_shared/apply-edit-at-anchor.md` and the shim. **Fix:** keep "Contract: `_shared/apply-edit-at-anchor.md`; shim: `scripts/apply-edit-at-anchor.js`; tests: `tests/apply-edit-at-anchor.test.js`" (~6 lines). The dispatch coupling with `/comments resolve` is preserved — only prose goes.

11. **[V] Phase 3's silent-roles protocol over-polices a good idea.** The role table, ordering rationale, and "zero questions is fine — five is fine if all five matter" are excellent. The ~20 lines of silent-roles block format + anti-pattern + "the Skip-if column is the ONLY valid reason" compress to: "At the end of Phase 3, list every role you didn't interview with a one-line reason — never skip silently." The Data Flow Trace (origin: `2026-05-08_spec-skill-grill-updates`, retro-driven) earns its place, but states its trigger twice — the numbered procedure and then a "Trigger (property-based)" paragraph with a 12-item enumeration; keep the property, one example, and "when in doubt, run it — it's cheap".

12. **[X] Cross-platform posture is actually good.** The Platform Adaptation section covers the three real dependencies (interactive prompts, subagents, Playwright) with degradation paths that match what the body does, and Phase 6 carries its own platform fallback for the findings protocol. Tool names appear as "e.g., `TaskCreate` in Claude Code". The stderr-print contracts (mode/output_format lines) assume a CLI runtime — accepted as the global non-interactive baseline, not a per-skill defect.
