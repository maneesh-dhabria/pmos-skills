PMOS Skills v3 — Session-Log Evidence

What 30 days of real transcripts say about where the marketplace actually hurts. Mined from both config dirs (~/.claude-personal + ~/.claude-workmax) across every repo, by 10 parallel Sonnet miners working from raw JSONL. Companion to the revised v3 proposal (which responds to the original proposal) — this document is the empirical ground the tier verdicts should stand on.

Window: 2026-06-11 → 2026-07-11 Corpus: 234 sessions enumerated (202 personal, 32 work) 169 transcripts with pmos-skill activity · 340 skill×session pairs 10 Sonnet miner groups · 16 reports ~90 deduped findings · 30 blocker-class Blockers re-verified vs main @ d46b7ae6: 15 fixed · 7 partial · 8 open

Method & Tagging

Every session directory under both config dirs was enumerated; transcripts were grepped for skill invocations (both Skill-tool calls and <command-name> wrappers), then split into 10 groups (loop/define sessions × 4 chunks, release trains × 2, pipeline-in-other-repos, authoring/thinking skills, learnkit, managerkit/utilities/gamekit, workmax). Each group went to a Sonnet subagent with a shared extraction protocol: jq/grep only (never whole-file reads), classify each session, and report findings only with a verbatim evidence quote. Findings below are deduped across miners; when the same defect appeared in multiple independent groups, the recurrence is called out — those are the strongest signals in this document.

blocker broke the deliverable or required user rescue   friction cost retries, turns, or trust   nit noise   positive pattern worth codifying  |  interactive human at the wheel   autonomous cron/loop/lights-out  |  personal work config dir  |  FIXED verified fixed in current repo   PARTIAL partially addressed   OPEN verified still present

Staleness caveat — these findings span 30 days of an actively self-improving repo. Many of the worst incidents spawned feedback epics that already shipped inside the same window, so a finding's presence here does not mean it is still live. Every blocker-class finding below was therefore re-verified against the current repo state (main @ d46b7ae6, 2026-07-11) — see Blocker Verification for the per-finding verdicts. Result: of 30 blocker-class findings, 15 are verified fixed, 7 partially addressed, 8 still open. The v3 plan should draw only on the open/partial set — and treat the fixed set as evidence that the feedback loop works, plus a source of regression-test candidates. Friction/nit findings were not individually re-verified; treat them as directional and re-check before acting on any one of them.

Interactive vs autonomous findings are different animals, per the mining brief. Interactive findings are mostly trust and ceremony: users pushing back, re-litigating, or rescuing. Autonomous findings are mostly economics and contracts: retry loops, gate flakes, starved crons, and silent degradation that nobody was watching. v3 fixes should be routed accordingly — trust problems need hard gates and evidence artifacts; autonomy problems need scripts, state files, and machine-readable exit signals.

Themes describe what the 30-day window showed, including incidents whose specific fixes have since shipped — e.g. the worst instances of themes 1, 2, and 6 are already remediated (see Blocker Verification). A theme stays listed because its class recurred across independent skills; the per-finding status tables govern what v3 still needs to do.

What This Changes in the v3 Plan

Mapping the evidence onto the revised proposal: the transcripts confirm five of its positions, sharpen two, and add four recommendations neither document had. Per the verification pass, the recommendations below draw only on the open and partial blocker sets — the 15 verified-fixed findings feed v3 solely as regression-test candidates (their fixes are exactly the behaviors golden-scenario tests should pin).

Confirmed by field evidence

  • Golden-scenario contract tests (revised proposal, "More Aggressive #6") — the /backlog viewer's @null/misclassification defects, verify's PASS-on-broken-UI, and /magazine's silent --days no-op are exactly the class a behavioral safety net catches and no SKILL.md prose ever will.
  • R3→R1 re-tier of /backlog and /mytasks — the script cores behaved (zero claim-lock misfires across ~90 build ticks); the failures were UX shells and doc drift, not architecture.
  • complete-dev as the first true R3 — it accounts for more distinct high-severity findings than any other skill (train state, >4-epic crash, mirror pushes, residual re-litigation, cross-epic pre-flight, [J] ladder), and every one of them is a structure problem prose patches won't fix.
  • Loaded-token budget over line counts — the compaction-loss theme (#6) shows the binding constraint is what survives a context window, not how long the file is.
  • §K one-fact-one-home enforcement via machine-resolvable cites — claim-holder drift, AC-format mismatch, and the dangling learnkit substrate cite are all §K violations that only release-time judgment gates caught, months of runtime later.

Sharpened

  • State extraction from feature-sdlc isn't just decomposition hygiene — it's the fix for themes 6 and 8. The state file must carry: per-phase checklist, dirty-file list (for read-state recovery), claim heartbeat, and needs_creds/buildability stamps.
  • The §H "arithmetic → script" rule generalizes to "harness choreography → script." The dominant autonomous failure mode is prose driving mechanical sequences (claim/write-back, worktree lifecycle, ToolSearch loads, TaskCreate shapes). A scripted build-tick helper eliminates the whole class the way mint-id.mjs eliminated id collisions.

New recommendations (not in either proposal)

  1. Accepted-residuals baseline — a committed per-skill/per-check exemption file consumed by skill-eval-check; the single highest ceremony-to-value fix available, and it removes the "EXIT 1 is normal" reflex that enables silent gate-skips.
  2. Confidentiality hard gate — tool-level publish prohibition for skills that ingest sensitive inputs, plus the inherit-the-source's-constraints rule that already worked once in /requirements. One near-miss is one too many.
  3. Unattended-mode contract as a first-class substrate: pre-claim buildability gate, needs_creds stamps, machine-readable queue-exhausted exits, reconcile-in-flight step 0, and the CronDelete self-stop as the reference pattern. Autonomous operation is now ~40% of pipeline volume; it deserves its own contract, not interactive mode with prompts removed.
  4. Gate evidence lines — every hard gate prints resolved script path + exit code + count before the next phase may start. This is the cheap, universal antidote to theme 2 and costs one line per gate.

What's genuinely healthy: claim-lock CRUD, changelog drafting (zero pushback in ~29 trains), /ideate (~16 invocations, no friction), /grill (users answer through 12–20 rounds), /reflect's transcript locator, /logos resume, the kb-guard confidentiality catch, and one cron that correctly self-stopped on an empty backlog. v3 should treat these as load-bearing patterns to preserve, not surfaces to rewrite.
