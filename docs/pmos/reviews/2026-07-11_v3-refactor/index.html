PMOS Skills v3 Refactor Proposal

A skill-by-skill proposal for rewriting the marketplace from intent outward: less generated ceremony, smaller SKILL.md files, more script-owned mechanics, clearer product boundaries, and a portable runtime contract across Claude/Codex-style agents.

Date: 2026-07-11 Scope: 64 user-facing skills Posture: rewrite allowed; preserve main intent Companion audit: docs/pmos/reviews/2026-07-11_repo-audit/

Executive Recommendation

Do not do a line-by-line cleanup as v3. Treat v3 as a product reset. Keep the strong substrates and hard-earned gates, but rewrite the user-facing skills around a much smaller operating contract: one job, one default path, explicit escape hatches, scripts for deterministic work, and plugin-aware shared runtime text.

The v2 repo is capable but heavy. Its core failure mode is that the SKILL.md file often tries to be a requirements doc, runbook, policy manual, CI contract, and agent prompt at once. v3 should make SKILL.md a job card: what this skill is for, what to read, what phases to run, what must never happen, and what script/shared contract owns the mechanics.

Most skillsTarget 60-120 SKILL.md lines, not 200-700.
Complex workflowsKeep orchestration small; move state and validation into scripts.
FlagsOnly machine-coupled, destructive, typed, or headless flags stay visible.
GatesDeterministic = hard; judgment = advisory; arithmetic/parsing = script.

The v3 House Style

  1. Start with a one-sentence job. If the sentence needs "and" three times, split the skill or hide sub-modes behind reference files.
  2. Use 3-6 phases by default. Setup, gather/ingest, generate/analyze, verify, emit, learn. Nested phase numbers are a smell.
  3. Remove copied non-interactive prose from skill bodies. Replace the frozen 27-line block with a plugin-aware runtime include plus a linter that resolves the include. If the platform requires inline text, generate it at packaging time.
  4. Use a shared adapter layer for platform concepts. Agent prompts should say "ask", "dispatch", "track progress", "stage", not `AskUserQuestion`, `Task`, `TodoWrite`, or Claude-specific transcript homes unless the adapter says so.
  5. Keep SKILL.md declarative. The body should name scripts and contracts, not restate script internals, JSON schemas, or exact shell command sequences except where the command is the product.
  6. Make every claim testable or obviously advisory. A skill saying "never drops citations" needs a script/test. A skill saying "be concise" is guidance, not a gate.
  7. Separate product quality from release ceremony. The release pipeline should verify artifacts; it should not make every authoring skill narrate release-history FR/D/T lineage.

Core Workflow Refactor

Area v3 proposal Why
Feature pipeline Make `/feature-sdlc` a router and progress ledger, not the owner of feature, skill, prototype, define, build, worktree, and release logic. Extract v3 entrypoints: `/define-feature`, `/build-feature`, `/skill-sdlc`, `/prototype-sdlc` as real first-class workflows with shared state schema. Today it is coherent but over-compressed. Users experience it as one giant switchboard.
Execute/verify Introduce explicit completion modes: `evidence-only`, `commit-per-task`, `squash-friendly`, `no-commit`. `/verify` emits `PASS`, `PASS-WITH-GAPS`, or `FAIL` with machine-readable evidence, not a long narrative gate. Current inline execution assumes commit cadence and hard gates that do not fit every repo.
Comments Redefine `/comments resolve` around two honest result types: `applied-visible-edit` and `annotated-for-author`. Do not mark resolved unless visible content changed or the operator explicitly accepts annotation-only resolution. This is the most important trust fix in the system.
Runtime portability Create a shared `runtime-contract.md` with plugin parameterization and platform adapters for prompt, dispatch, transcript, browser, filesystem, staging, and task tracking. The current skills still smell like Claude Code-era generated prompts.
Testing/CI Move to plugin-level CI matrices: hygiene, substrate selftests, skill-specific deterministic tests, browser smoke where UI exists, localhost tests in a named loopback job. Many good tests exist but are not consistently release-blocking.

Rewrite Tiers

TierMeaningUse when
R0Keep intent; light v3 trimThe skill is already small and mostly clear. Remove boilerplate, fix plugin runtime text, keep structure.
R1Rewrite SKILL.md onlyThe substrate is sound, but the instruction body is too ceremonial or over-detailed.
R2Rewrite skill + extract mechanicsThe skill has deterministic parsing/state/gates in prose, missing tests, or too many flags/phases.
R3Product-level redesignThe skill boundary itself is wrong, overloaded, or needs a new workflow contract.

Suggested Migration Order

  1. v3 runtime substrate first. Parameterized non-interactive contract, citation resolver lint, plugin-aware shared file policy, platform adapter vocabulary.
  2. Fix trust bugs. `/comments`, managerkit path drift, plugin identity line, dangling shared citations.
  3. Script the prose state machines. `/reflect`, `/book-summary`, `/complete-dev`, `/execute`, `/verify` evidence ledger.
  4. Rewrite thin wrappers. Gamekit, `skill-sdlc`, `prototype-sdlc`, `changelog`, `session-log`, `wayrttd` to prove the v3 style.
  5. Rewrite core workflow skills. `feature-sdlc`, `requirements`, `spec`, `plan`, `execute`, `verify`, `complete-dev` as a coherent v3 pipeline.
  6. Then polish authoring products. Landing page, prototype, diagram, logo, summary/video, wiki, magazine, Notion.

Definition of Done for a v3 Skill

  • SKILL.md has a single job sentence and a target line count justified by complexity.
  • Every deterministic promise has a script/test or is explicitly advisory.
  • Every relative citation resolves under a linter.
  • No hardcoded plugin identity appears in shared boilerplate.
  • All platform-specific tool names live in a Platform Adaptation or shared runtime adapter, not scattered through phases.
  • Flags in `argument-hint` pass the four-test: machine coupling, destructive opt-in, typed value, or headless determinism.
  • CI runs the skill's deterministic tests when that skill or its shared substrate changes.
