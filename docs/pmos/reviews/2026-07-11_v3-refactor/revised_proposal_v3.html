PMOS Skills v3 — Adopted Plan (Third Revision)

The adopted direction for v3, superseding the second-revision synthesis. Produced by a deep adversarial grill of that document (14 decisions, 2026-07-11; report at .pmos/grills/2026-07-11_revised-proposal-v2.html). The central change: v3 is no longer an in-place refactor of this repository. It is a new repository, built from scratch around a v3 rubric, that ports the proven engines and rewrites all instruction prose, cutting over one plugin at a time.

Date: 2026-07-12 Status: adopted direction Supersedes: revised_proposal_v2.html Scope: 65 user-facing skills across 5 plugins (43 toolkit · 8 learnkit · 7 gamekit · 4 utilities · 3 managerkit) Posture: greenfield repo, per-plugin cutover, sole-user window

Executive Verdict

Build v3 as a new repository: port the engines, rewrite every instruction file from scratch against a v3-first rubric, and cut over plugin by plugin — gamekit first, pmos-toolkit last.

The second revision correctly identified what to preserve (script-backed engines, earned safeguards, concrete platform instructions) and what to remove (duplicated narration, model-executed mechanics, line-count theater). It was wrong about where the work happens. An in-place migration forces every v3 skill through gates that encode the v2 house style, entangles the rewrite with a live release train, and makes each "wave" pay a compatibility tax that a sole-user product does not owe. A greenfield repo dissolves all three problems and flips the cost of the previously deferred architecture: a shared CLI and corpus separation are nearly free on day 0 and expensive later.

CARRIED FORWARDJob-card skill bodies, script-owned mechanics, concrete tool names, engines preserved, trust contracts (/comments, /verify, /execute), no line-count compliance, description diet.
CHANGEDIn-place migration → new repo with per-plugin cutover; transitional rubric → rubric-first (structure blocking, thresholds advisory); cull deferred → cull at port time; Phase 0–5 program → substrate epic + plugin waves.

Governing Principle

Port what is proven, rewrite what is prose, and make the new repository's own gates enforce the v3 contract from the first merge.

  • Engines port. Scripts, tests, corpora, and the HTML/render substrate move largely verbatim (refactored onto the shared CLI where they duplicate, and freely improved where clearly better). The evidence base is unambiguous: scripted cores are the reliable part — zero claim-lock misfires across ~90 autonomous build ticks, zero ID collisions since mint-id.
  • Prose is rewritten from scratch. Every SKILL.md, the rubric, and all shared protocol text is authored fresh in the v3 house style. No v2 instruction text is carried forward by copy-paste.
  • Data outlives both repos. Ported engines must keep reading the existing ~/.pmos stores (tasks, people, learnings, interview records) unchanged. Schema compatibility is a hard requirement of the engine port, not a nice-to-have.
  • Sole-user window. There is currently one user. No backward-compatibility shims, no dual-landing of fixes, no staged deprecations. This is a temporary license, not a permanent posture — it expires the day a second user installs.

v3 Architecture

1. A new repository, seeded by a substrate epic

The first unit of work is not a skill. It is the foundation the first skill needs in order to merge honestly:

  • The v3 rubric and its checker. Written fresh; carries forward the universal v2 rules that need no recalibration (deterministic-vs-judgment gates, flag four-test, stable phase anchors, citation resolution, one fact one home, explicit platform adaptation) and adds the v3 measurements (declared-reads budget, description size, prose-owned mechanics, behavioral coverage).
  • The shared pmos CLI. One tool owning serve, render, anchor, store, and mint — replacing the four-plus duplicated localhost servers, launcher trios, and repeated render/storage code scattered across today's 125 scripts. Every ported engine re-points at it during its port.
  • The test harness. Deterministic script/CLI tests wired into CI from the first commit, plus the headless model-smoke runner used at plugin-flip time.
  • The scripted release engine. Diff routing, version bump, tag construction and validation, push planning, dry-run contract, and release evidence — built as scripts from day one. This is the v3 complete-dev core; the second revision's #1 extraction target is satisfied by construction rather than surgery.

2. Skill bodies are thin behavioral interfaces (unchanged)

Each SKILL.md states the single job and trigger boundary, the default journey and legitimate modes, the scripts and shared contracts that own mechanics, the safety and trust invariants, and the evidence required before reporting success. No universal line cap; the budget is declared-reads tokens, measured and linted.

3. Deterministic state belongs in deterministic code (unchanged, strengthened)

Routing tables, state transitions, ID allocation, parsing, arithmetic, schema validation, version manipulation, tag construction, and evidence aggregation are scripts or tests. The evidence report extends this: harness choreography (claim → build → write-back ticks) is also script territory. And every hard gate must emit evidence — the resolved script path, exit code, and finding count — so a skipped gate is visible rather than narratable.

4. Platform instructions stay concrete; the Codex mirror rides along

Primary-platform tool names appear in the path where reliable behavior depends on them; alternative-platform behavior lives in one adaptation section. The dual .claude-plugin/.codex-plugin manifests port to the new repo with the same lockstep-version rule.

5. Corpora are versioned data artifacts

The bundled corpora (665 case studies, 346 frameworks, 61 primers with 1,306 sources, the archetype corpus) move as data artifacts with their own schemas and validation gates, referenced by skill releases rather than embedded in them. Separation happens at port time, when each corpus is being touched anyway.

6. Unattended operation is a first-class contract

Roughly 40% of pipeline volume runs autonomously. Every skill's rewrite states its unattended posture explicitly: what it does without a human (proceed, degrade, or refuse), what it defers, and what evidence it leaves behind. The non-interactive contract is redesigned as part of the substrate epic — as a real contract with a linted single home, not a 27-line block re-pasted into 65 files.

The v3 Rubric

Structure blocks from day 0; numbers earn their teeth. Structural checks are binary and carry over from hard-won v2 policy — they block immediately. Numeric thresholds (declared-reads budget, description size, phase count) run advisory through the gamekit and utilities waves, then are set from the observed distribution of real v3 skills and promoted to blocking before managerkit migrates. A threshold is promoted only with a documented failure mode and a low false-positive record — never because the number looks attractive.

  1. Blocking from day 0: no deterministic mechanic lives only in prose; every citation resolves; flags pass the four-test; one fact, one home; platform adaptation is explicit; hard gates emit evidence lines; unattended posture is declared.
  2. Advisory until calibrated: declared-reads token budget; frontmatter description size; phase-count and structure-shape guidance.
  3. Never checks: line counts, prose style compliance, or anything a reasonable rewrite would satisfy by formatting rather than by improving reliability.

Trust Contracts

Carried from the second revision and extended with the session-evidence findings. All are built into the fresh rewrites, not patched into the old repo:

  • /comments honest outcomes: visible edits, author annotations, skipped work, and unresolved threads are distinct terminal states; resolution is never implied without one.
  • /verify structured verdicts: PASS, PASS-WITH-GAPS, FAIL with machine-readable evidence. Deterministic failures stay hard; judgment gaps are explicit.
  • PASS-WITH-GAPS consumption: blocks unattended release; interactive release requires explicit user acknowledgement, recorded with the gap list in the release record (accepted-residuals baseline).
  • /execute modes: evidence-only, commit-per-task, squash-friendly, and no-commit, backed by an execution ledger.
  • Gate-evidence lines: every hard gate prints the resolved script path, exit code, and finding count — the antidote to the "gate silently skipped, then rationalized" failure mode observed five independent times.
  • Unattended-mode contract: every skill declares proceed/degrade/refuse behavior for autonomous invocation.
  • Confidentiality hard gate: skills handling sensitive material (interview records, people data) enforce redaction/scoping as a gated check, not guidance.

Migration Program

Replaces the second revision's Phases 0–5 entirely. Each wave ends with a plugin flip: the plugin becomes installable from the new repo and is archived in the old one.

Wave 0 — Substrate epic (new repo, before any skill)

  • Scaffold the repository, marketplace entry, and CI.
  • Build the v3 rubric + checker (structure blocking, thresholds advisory), the pmos CLI skeleton, the test harness, and the scripted release engine.
  • Redesign the non-interactive/unattended contract as substrate.
  • Freeze the old repo: no new feature epics; stop cron automation; run the one-time backlog triage (re-mint keepers as v3-native items).

Exit gate: the new repo can gate, test, version, tag, and release a trivial skill end-to-end using only its own machinery.

Wave 1 — pmos-gamekit (7 skills; the house-style pilot)

  • Rewrite the seven launchers as minimal job cards over the shared launcher engine.
  • First real pass of the rubric, CLI, and release engine; collect threshold observations.

Exit gate: model-smoke suite green; plugin flips; gamekit archived in the old repo.

Wave 2 — pmos-utilities (4 skills) + threshold calibration

  • Port engines (converter, mac-health snapshot, reflect extractors, Notion mapper) onto the CLI where they duplicate; rewrite bodies.
  • After this wave: set numeric rubric thresholds from the ~11 migrated skills and promote them to blocking.

Wave 3 — pmos-managerkit (3 skills)

  • Confidentiality hard gate ships here first (interview records, people data).
  • ~/.pmos record-store compatibility proven by test: existing interview and 1:1 data reads unchanged.

Wave 4 — pmos-learnkit (8 skills) + corpus separation

  • Corpora (case studies, frameworks, primers) move to versioned data artifacts with schema gates; skills reference corpus versions.
  • Shared library-viewer substrate consolidates the duplicated browse/list machinery.

Wave 5 — pmos-toolkit (43 skills, ported in dependency order) + cull

  • Substrate consumers first (pipeline skills over the render/comments engines), then the pipeline itself (requirements → spec → plan → execute → verify), then feature-sdlc as a thin router over script-owned state, with complete-dev's body landing as a wrapper over the Wave-0 release engine.
  • Cull at port: each skill faces a port/merge/don't-port decision using transcript evidence + user judgment. The thinking-tools cluster (shape, ideate, grill, wayrttd, creativity, ripple-effects) and the msf pair get their merge design here, as part of their rewrite.
  • Trust contracts (/comments outcomes, /verify verdicts, /execute modes) are built into these rewrites.

Exit gate: toolkit flips; old repo fully archived (read-only), exactly like the previous cutover.

Build-Step Decision (unchanged: deferred, with sharper triggers)

Still deferred. The new repo ships as plain source — the installed artifact is the reviewed file. Reconsider when at least two of these triggers fire: (1) the Codex manifest lockstep becomes a recurring source of drift (now the most likely first trigger, since the mirror ports); (2) generated counts/catalogs ("N bundled") drift despite the one-fact-one-home rule; (3) the shared-contract single-home policy proves unenforceable without compilation. Adoption additionally requires CI proof that generated artifacts are current and behaviorally equivalent to source.

Definition of Done for a v3 Skill

  • One clear job sentence and a trigger boundary distinguishable from adjacent skills.
  • Declared-reads manifest present; token sum within the calibrated budget; lint confirms every unconditional read is declared.
  • Frontmatter description is concise (~2 sentences + trigger phrases) while retaining distinctive triggers.
  • No deterministic mechanic lives only in prose; every deterministic promise is backed by a script, test, or schema, or is explicitly advisory.
  • Hard gates emit evidence lines (script path, exit code, finding count).
  • Unattended posture declared: proceed / degrade / refuse, with evidence left behind.
  • Platform-critical instructions use concrete tool names; alternatives isolated in one adaptation section; Codex manifest in lockstep.
  • Citations and shared-substrate references resolve under lint.
  • Flags pass the four-test and have natural-language equivalents where appropriate.
  • Deterministic CI tests exist for its engine surface; the skill passes the model-smoke fixture before its plugin flips.
  • If ported: reads existing ~/.pmos data unchanged. If culled or merged: the decision and its evidence are recorded.

Success Measures and Stop Conditions

Context efficiencyTotal description bytes and per-skill declared-reads fall versus the v2 baseline (40,586 description bytes; measured per-skill loads) without loss of correct triggering in daily use.
ReliabilityDeterministic decisions execute in scripts with evidence lines; the "gate skipped and rationalized" failure mode stops appearing in transcripts.
TrustEvery run ends in an honest terminal state; gaps ship only with recorded acknowledgement; unattended runs refuse rather than improvise.
MaintainabilityOne CLI where there were four servers; one canonical home per fact; corpora versioned separately from code.
Migration safetyEach wave flips a whole plugin behind green smoke tests; the old repo remains installable per-plugin until its flip; nothing requires a cross-repo flag day.
Product focusThe ported catalog is smaller than 65; every surviving skill has invocation evidence or an explicit keep rationale.

Stop or revise the program if: the substrate epic (Wave 0) doesn't converge on a releasable foundation quickly — that signals the greenfield scope is wrong, not that gates should be skipped; ported engines start failing against existing ~/.pmos data; the advisory thresholds, once calibrated, generate frequent judgment disputes; model-smoke tests become prose snapshots; or a second user arrives mid-migration, which immediately revokes the sole-user license and reinstates compatibility duties on whatever remains unflipped.

Open Questions

  • New repo identity: name, marketplace entry, and remote topology (retain the GitLab mirror?) are undecided.
  • Thinking-tools merge shape: which 2–3 skills with which modes — a design pass owed at Wave 5, informed by invocation evidence.
  • Rubric authorship loop: the Wave-0 rubric gates skills, but what reviews the rubric itself? Proposal: the grill + the calibration checkpoint after Wave 2 serve as its review, recorded in the repo.

Decision Summary

Stand up the new repository with the Wave-0 substrate epic: v3 rubric + checker, shared pmos CLI, test harness, scripted release engine, and the redesigned unattended contract. Freeze the old repo and triage its backlog once. Then migrate plugin by plugin — gamekit, utilities (calibrate thresholds), managerkit, learnkit (separate corpora), toolkit (cull and merge as part of the port) — porting engines, rewriting all prose, and flipping each plugin only behind green deterministic tests and model smoke.

Do not rebuild the engines. Do not carry v2 instruction text forward. Do not point the old pipeline at the new repo. Do not compile the repository until the named triggers fire.

v3 succeeds when the new repository's own gates make dishonest completion harder than honest completion — and the old repository can be archived without anything of proven value left behind.
