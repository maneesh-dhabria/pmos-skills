PMOS Skills v3 — Recommended Proposal v2

A standalone recommendation for simplifying the PMOS skills marketplace while preserving proven engines, earned safeguards, and reliable platform behavior. This proposal synthesizes and resolves the major disagreements between the original v3 proposal and the critical revision.

Date: 2026-07-11 Status: superseded by revised_proposal_v3.html (2026-07-12, post-grill) Scope: 64 user-facing skills across 5 plugins Posture: staged refactor, source-compatible

Executive Verdict

Adopt the revised proposal as the strategic baseline, but do not introduce a source/ → dist/ build system yet.

v3 should be a staged simplification of the repository that preserves working engines and hard-earned safeguards. It should not be a wholesale rewrite, a line-count exercise, or a packaging-architecture migration disguised as an instruction cleanup.

The original proposal identifies the correct destination: smaller job-card skills, script-owned deterministic mechanics, clearer product boundaries, structured verification evidence, and less ceremony. The revised proposal supplies the better diagnosis: several supposed rebuild candidates are already script-backed products; the existing skill rubric enforces v2 conventions; and frontmatter descriptions impose a marketplace-wide context cost.

This proposal combines those strengths while rejecting two forms of premature architecture: a new compiled distribution layer and indiscriminate skill consolidation.

ADOPTJob-card posture, description diet, comments trust contract, verify verdicts, execute modes, and extraction of genuine prose state machines.
MODIFYIntroduce the v3 rubric transitionally; use behavioral tests according to risk; keep one feature-sdlc entrypoint plus thin aliases.
DEFERBuild system, shared PMOS CLI, corpus separation, and broad skill mergers until measured duplication or usage justifies them.
REJECTLine-count compliance, vague platform-neutral tool vocabulary, and rebuilding engines that already own their mechanics.

Governing Principle

Preserve proven engines and earned safeguards; remove duplicated narration and model-executed mechanics; redesign only where the product contract—not merely the instruction file—is wrong.

This distinguishes four kinds of work that the earlier proposals sometimes blur:

  • Instruction diet: reduce prose while keeping behavior and engine contracts intact.
  • Mechanics extraction: move deterministic state, parsing, routing, and validation from model instructions into scripts.
  • Contract correction: change a user-visible promise or outcome taxonomy because the current behavior is misleading.
  • Product redesign: change the skill boundary, workflow, or persistence model. Reserve this for cases where the product itself is wrong.

Recommended v3 Architecture

1. Source remains the installed artifact

Do not create a compiled dist/ tree in the first v3 program. Contributors, reviewers, and users should continue to inspect the same files the agent loads. This preserves debuggability and avoids dist drift, compiler failures, and release ambiguity.

Use lightweight generation only for clearly derived metadata, where the generated output cannot obscure executable behavior. Retain the inline non-interactive contract and its drift lint until a build system solves at least two otherwise-intractable problems and has an explicit source/dist verification contract.

2. Skill bodies become thin behavioral interfaces

Each SKILL.md should explain:

  • the single job and trigger boundary;
  • the default user journey and legitimate modes;
  • the scripts, references, or shared contracts that own mechanics;
  • the safety and trust invariants that must remain visible to the model;
  • the evidence required before reporting success.

There is no universal line cap. The budget is the context actually loaded for the normal path, including always-read references.

3. Deterministic state belongs in deterministic code

Routing tables, state transitions, ID allocation, parsing, arithmetic, schema validation, version manipulation, tag construction, and evidence aggregation must be scripts or tests. The model may choose among valid options and interpret evidence; it should not simulate a state machine embedded in prose.

4. Platform instructions stay concrete

Use the real primary-platform tool names in the path where reliable behavior depends on them. Put alternative-platform behavior in one concise adaptation section. Avoid generic verbs where they make tool invocation less predictable.

5. One primary workflow entrypoint, thin aliases where useful

/feature-sdlc remains the main router. Existing aliases such as /skill-sdlc and /prototype-sdlc remain thin user-facing conveniences. The state machine and resume logic move behind the interface; new top-level workflows are added only when user behavior shows routing ambiguity that aliases cannot solve.

Transitional v3 Rubric

Do not replace the v2 rubric in a flag day. A new rubric must judge both unmigrated skills and v3 pilots without forcing either false failures or mass exemptions.

  1. Preserve universal rules: deterministic versus judgment gates, flag classification, stable phase anchors, citation resolution, one fact per canonical home, and explicit platform adaptation.
  2. Add v3 measurements—description size, normal-path loaded context, prose-owned deterministic mechanics, and behavioral-contract coverage—as advisory reports.
  3. Pilot the rules on a thin wrapper, an existing engine wrapper, and a prose-heavy orchestrator.
  4. Calibrate thresholds from pilot evidence rather than choosing attractive round numbers.
  5. Promote a check to blocking only when it detects an actual quality or reliability failure with a low false-positive rate.

Recommended Migration Program

Phase 0 — Baseline and transitional rules

  • Record description bytes and normal-path loaded context by skill.
  • Add advisory v3 rubric checks without invalidating existing v2 skills.
  • Classify skills by behavioral-test risk.
  • Freeze new duplicated runtime prose during the migration.

Exit gate: existing gates remain green, and the v3 advisory report runs across all plugins.

Phase 1 — Low-risk marketplace cleanup

  • Shorten frontmatter descriptions while preserving distinctive trigger phrases.
  • Fix dangling citations, stale paths, plugin identity drift, and incorrect catalog counts.
  • Rewrite thin aliases, game launchers, changelog, session-log, and similar wrappers in the candidate v3 style.

Exit gate: routing smoke tests show no material loss of triggering, and pilot wrappers satisfy the advisory rubric without hidden reference bloat.

Phase 2 — Trust contracts

  • Implement honest /comments outcomes.
  • Implement structured /verify verdicts and evidence.
  • Define /execute completion and commit modes.
  • Demote judgment-only gaps from false hard failures.

Exit gate: contract tests cover terminal states, backward compatibility, and failure behavior.

Phase 3 — Extract genuine prose state machines

  1. complete-dev
  2. feature-sdlc
  3. execute
  4. verify
  5. other measured cases such as transcript discovery in reflect

Exit gate: old and new fixtures produce compatible plans or artifacts; dry runs demonstrate that deterministic decisions no longer depend on prose execution.

Phase 4 — Promote the v3 house style

  • Evaluate advisory checks against the completed pilots.
  • Remove checks that reward formatting without reliability.
  • Promote proven checks to blocking.
  • Migrate remaining R1/R2 skills in plugin-sized waves.

Exit gate: the rubric has a documented failure mode for every hard gate and a low false-positive record on migrated skills.

Phase 5 — Reassess deferred architecture

  • Use invocation evidence to archive or merge confusing or unused skills.
  • Measure duplicated server, storage, render, and launcher code before creating a shared CLI.
  • Evaluate corpus separation based on repository size and independent release needs.
  • Reconsider source/ → dist/ only if repeated required transformations justify compilation.

Build-Step Decision

What compilation could eventually solveWhy it should be deferred now
True includes, platform-specific emits, generated manifests and catalogs, and compile-time token budgets.The installed artifact would diverge from reviewed source, introducing dist drift, contributor friction, compiler debugging, and a second release surface.

Decision: defer, with explicit reconsideration criteria. Adopt a build step only when at least two recurring transformations cannot be safely handled through source validation or narrowly scoped metadata generation, and when CI can prove that generated artifacts are current and behaviorally equivalent.

Definition of Done for a v3 Skill

  • The skill has one clear job sentence and a trigger boundary distinguishable from adjacent skills.
  • The normal path fits a measured loaded-context budget; reference files are counted when always read.
  • Frontmatter is concise enough for discovery while retaining high-signal trigger phrases.
  • No deterministic mechanic lives only in prose.
  • Every deterministic promise is backed by a script, test, schema, or explicitly advisory status.
  • Platform-critical instructions use concrete tools, with alternatives isolated in one adaptation section.
  • Relative citations and shared-substrate references resolve automatically.
  • Machine-facing flags satisfy the existing four-test and have natural-language equivalents where appropriate.
  • Behavioral protection matches risk: hard contracts for mutation, routing, artifacts, release, and machine output; targeted evaluation for open-ended reasoning.
  • CI runs affected deterministic and behavioral tests when the skill or its shared substrate changes.
  • The migration preserves existing user-visible behavior unless an approved contract change explicitly says otherwise.

Success Measures and Stop Conditions

Context efficiencyMarketplace description bytes and normal-path loaded context fall without measurable trigger regression.
ReliabilityFewer deterministic decisions are performed by the model; contract tests cover critical state and terminal outcomes.
TrustUsers can distinguish success, success with gaps, annotation, skip, and failure without interpreting narrative caveats.
MaintainabilityFacts have one canonical home, references resolve, and critical workflows no longer require editing several prose copies.
DiscoverabilityDescription trimming does not materially reduce correct skill selection or increase ambiguous routing.
Migration safetyEach wave is independently releasable and reversible; no phase requires a repository-wide flag day.

Stop or revise the program if: concise descriptions reduce correct triggering; new rubric checks generate frequent judgment disputes; behavioral tests become prose snapshots; source and runtime artifacts diverge; or a migration requires widespread exemptions to remain releasable.

Decision Summary

Proceed with v3 as a source-compatible, evidence-led simplification program. Start with the transitional rubric and description diet, prove the house style on thin wrappers, correct trust contracts, then extract the genuine prose state machines beginning with complete-dev.

Do not rebuild comments, mytasks, or backlog. Do not multiply top-level workflow entrypoints. Do not weaken platform instructions in the name of abstraction. Do not compile the repository until repeated, measured problems justify a build system.

The v3 program succeeds when the repository becomes easier for an agent to execute correctly—not merely shorter for a human to skim.
