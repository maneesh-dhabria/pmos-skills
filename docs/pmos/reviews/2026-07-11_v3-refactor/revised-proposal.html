A critical response to the original v3 refactor proposal, grounded in measurements of the actual repo rather than the proposal's own framing. Executive Summary

The original proposal's direction is right, but it was written from reading SKILL.md files — not the directories around them. Three structural defects: (1) several R3 verdicts are wrong on the facts — the skills it wants "rebuilt around scripts" already are script-backed engines; (2) the migration order misses step 0 — this repo's own gate system (skill-eval.md mirrors skill-patterns.md 1:1, 53 checks, 4 lints) enforces the v2 house style, so every v3 rewrite run through the pipeline will be rejected by the repo's own gates unless the rubric ships first; (3) it optimizes the wrong token budget — the always-loaded cost of this marketplace is ~40KB (~10k tokens) of frontmatter descriptions in every session, which the original never mentions.

Rebuttals to the Original

Rebuttal 1 — Several R3 verdicts don't survive contact with the repo

mytasks, backlog, and comments are tagged R3 ("product-level redesign") on the theory that their mechanics live in prose. The measurements above show the opposite: their mechanics already live in substantial, tested script engines. These are R1 body-trims — plus, for comments, the product-contract fix, which is real but is a contract change, not a rebuild. Mislabeling them R3 inflates the migration by months and risks breaking working, well-tested engines for cosmetic gain.

Rebuttal 2 — The non-interactive include contradicts a documented decision, and half-does the real move

The original's house-style rule #3 says: replace the frozen inline non-interactive block with "a plugin-aware runtime include," and "if the platform requires inline text, generate it at packaging time." Two problems:

  • CLAUDE.md records why the block is hand-maintained inline (W14 posture): skills platforms have no include mechanism at load time, and the repo deliberately rejected an auto-propagating body-rewriter as riskier than the re-paste tax. The lint detects drift; fixes are manual by design.
  • "Generate at packaging time" assumes a packaging step that doesn't exist. Claude Code installs plugins straight from the repo — the source is the artifact.

So this recommendation is either (a) already settled — keep inline + lint — or (b) an argument for a real build step, which is a much bigger and more interesting decision the original should have made explicitly. See "The Build-Step Decision" below. Do not adopt as written.

Rebuttal 3 — "Say 'ask', not AskUserQuestion" will degrade behavior on the platform that matters

Concrete tool names are load-bearing. Told to use AskUserQuestion, the model reliably calls it; told to "ask the user," it drifts into free-text questions that violate the very non-interactive contract this repo works so hard to enforce. The repo's existing pattern — Claude-native body plus a Platform Adaptation section with per-platform degradations — is the better trade. Portability abstraction belongs in a build step (emit per-platform variants from one source), not in vaguer prompts.

Rebuttal 4 — Line-count targets are the wrong metric

"60–120 SKILL.md lines" as a Definition-of-Done bullet will produce compliance theater: prose pushed into reference files that get loaded anyway. The right rules are (a) the one the repo already half-has — no deterministic mechanic may live in prose (§H) — and (b) a loaded-token budget per invocation (SKILL.md + always-read references combined). Measure what the model actually ingests, not what one file's wc -l says.

Rebuttal 5 — The migration order has a missing step 0

This repo is self-referentially gated: /skill-sdlc scores every skill against skill-eval.md (53 checks: 24 [D] + 23 [J] gated, 6 advisory), which mirrors skill-patterns.md 1:1, plus four hygiene lints. Those artifacts encode the v2 house style. Any v3 rewrite run through the repo's own pipeline will be rejected by its own gates — and bypassing the gates for 64 skills defeats the point of having them. Step 0 must be: rewrite skill-patterns.md + skill-eval.md + the lints to the v3 house style, ship that as its own release, then migrate skills through the updated pipeline. The original's "v3 runtime substrate first" gestures at this but names only the non-interactive contract and adapters, not the rubric.

Rebuttal 6 — Splitting /feature-sdlc into four top-level entrypoints has a hidden cost

Every new user-facing skill adds an always-loaded description (see the 40KB problem). The routing UX isn't the problem — /skill-sdlc and /prototype-sdlc aliases already exist at 22–25 lines each. The problem is the 715-line body. Different surgery: extract the state machine and mode dispatch into scripts, keep one entrypoint plus the existing thin aliases. Half-agree on decomposition, disagree on multiplying the top-level surface.

Rebuttal 7 — No behavioral safety net for the rewrite itself

All the repo's gates are static (lints, rubric checks, structure tests). The original green-lights rewriting 64 skills with no mechanism to prove behavior was preserved. Before any R2/R3 lands, build golden-scenario contract tests: headless runs of each skill against fixture inputs, asserting on the artifacts / JSON / exit codes produced. The repo has already proved this works locally — comments has resolver-mode tests; frameworks has a --json contract test. Rewrite confidence should come from "same fixtures, same outputs," not from re-reading prose.

The Build-Step Decision (the question hiding inside the original)

Commit to a build step (source/ → dist/), or explicitly refuse it — but decide. The original's include proposal, adapter vocabulary, per-platform text, and half its Definition-of-Done all quietly presuppose one. Make it a first-class decision.

A build step buysIt costs
  • Real includes — the non-interactive block and shared protocol text compiled in, killing the sync-shared.sh intersection-only bootstrap bug documented in CLAUDE.md.
  • Per-platform emits — claude/codex variants from one source, replacing the dual-manifest duplication.
  • Generated counts and catalogs — README rows, marketplace entries, "N bundled" claims become generated; the "7 bundled vs 8" class of [J]-gate bug becomes impossible rather than caught-at-release.
  • Enforced token budgets at compile time.
  • Dist drift and a second artifact to trust.
  • Contributor friction (edit source, build, verify dist).
  • "The installed plugin is not the source" — debugging indirection.

Recommendation: take the build step. At 5 plugins × 64 skills and growing, the generated-consistency wins subsume the original's proposals #3 and #4 and half its DoD list. If refused, then keep the inline block + lint exactly as-is and drop those items from v3 scope.

More Aggressive Changes (beyond the original)

1. Description-token dietCap every frontmatter description at ~2 sentences + trigger phrases; move the rest into the body. ~10k tokens refunded in every session. One day of work; do it before anything else.
2. One pmos CLIAt least four independent localhost servers (comments, mytasks, backlog, converter/gamekit launchers), multiple .command/.sh/.bat trios, repeated anchor/render/storage code across 125 scripts. A shared tools/pmos-cli (serve, render, anchor, store, mint) cuts more maintenance surface than any SKILL.md diet.
3. Cull and merge skills64 skills is a product problem. The thinking-tools cluster (shape, ideate, grill, wayrttd, creativity, ripple-effects) is six skills for one job family — collapse to 2–3 with modes. msf-req/msf-wf are one evaluator with two target types. Use /reflect + transcript history to archive zero-invocation skills (the archive/skills/ pattern already exists). Every cut is a permanent description-token refund and one less thing the [J] gate must keep coherent.
4. Separate data from codeCorpora (665 case studies @ 1.47MB, 346 frameworks, 61 primers × 1,306 sources, archetype corpus) ship inside skill releases. Version corpora as their own artifacts with schema + validation gates (the case-studies corpus story already built exactly this shape); skill releases reference a corpus version.
5. Machine-readable skill manifestsA small skill.json per skill (flags + four-test class, phases + slugs, scripts, tests, non-interactive posture: inline/refused/delegated) turns four bespoke bash lints into one schema validation, and makes catalogs, README rows, and argument-hints generated rather than hand-synced.
6. Golden-scenario contract testsHeadless fixture-in → artifact/JSON/exit-code-out assertions per skill, run in plugin-level CI when the skill or its substrate changes. This is the enabler that makes an aggressive rewrite safe at all.

Revised Migration Order

  1. Step 0 — rewrite the rubric first. skill-patterns.md + skill-eval.md + the four hygiene lints updated to the v3 house style (loaded-token budget replaces line counts; §H/§I/§J/§K carried forward). Ship as its own release. Nothing else can merge through the pipeline until this lands.
  2. Description-token diet across all 64 skills. Highest leverage, lowest risk, zero behavior change.
  3. Make the build-step decision. If yes: scaffold source/→dist/, move the non-interactive block and shared protocols to includes, generate manifests/counts. If no: strike the include and adapter-vocabulary items from scope permanently.
  4. Golden-scenario contract tests for every skill slated R2/R3, before its rewrite starts.
  5. Fix trust bugs (unchanged from original): /comments contract, managerkit path drift, plugin identity line, dangling shared citations.
  6. Script the prose state machines, starting with complete-dev (778:36 ratio), then /execute, /verify evidence ledger, /reflect, /book-summary.
  7. Rewrite thin wrappers (gamekit, changelog, session-log, wayrttd, aliases) to prove the v3 style — unchanged from original.
  8. Rewrite core workflow skills as a coherent v3 pipeline — unchanged from original, but feature-sdlc keeps one entrypoint per Rebuttal 6.
  9. Cull/merge pass + polish authoring products. Thinking-tools consolidation, msf merge, corpus extraction, then landing-page / prototype / diagram / logo / summary / wiki / magazine / Notion.

Revised Definition of Done for a v3 Skill

  • SKILL.md has a single job sentence. Target line count justified by complexity → a loaded-token budget (SKILL.md + always-read references) enforced by lint or build step.
  • No deterministic mechanic lives in prose (§H); every deterministic promise has a script/test or is explicitly advisory.
  • A golden-scenario contract test exists and passes: fixture in → asserted artifact/JSON/exit code out.
  • Frontmatter description ≤ ~2 sentences + trigger phrases.
  • Every relative citation resolves under a linter (unchanged from original).
  • Platform-specific tool names stay concrete and live in the body's primary flow for the primary platform, with degradations in one Platform Adaptation section — not abstracted to vague verbs.
  • Flags in argument-hint pass the four-test (unchanged — this is existing §I policy, not new).
  • CI runs the skill's deterministic tests when the skill or its shared substrate changes (unchanged from original).

Bottom line. Adopt the original's product fixes (/comments contract, /verify verdicts, /execute modes, complete-dev/feature-sdlc decomposition, gamekit wrappers, job-card posture). Reject its include-without-a-build-step, adapter vocabulary, line-count DoD, and the R3 labels on mytasks/backlog/comments. Add rubric-first migration, the description diet, contract tests — and make the build-step decision consciously, because it is the actual architecture question hiding inside the original proposal.
