1. Thesis

v3 should be a product architecture refactor, not a cleanup pass. The current repo has strong substrates and serious accumulated value, but too much of the operating model lives as copied prose inside SKILL.md files. The v3 goal is to preserve intent while separating user-facing skills from shared engines and deterministic mechanics.

Updated premise: this skill marketplace is currently for one maintainer. Backwards compatibility and old muscle memory are not requirements. v3 should prefer clean names and a smaller command surface over aliases that preserve legacy process language.

The main product problem is not that the skills are long. The problem is that many skills are simultaneously doing four jobs:

  • They are user-facing commands.
  • They are operational runbooks for the agent.
  • They are implementation specs for scripts and artifacts.
  • They are policy documents for portability, release, security, and review gates.

v3 should make the command surface simpler while making the internals more modular and testable.

2. v3 Principles

Small front doorsMost user-facing SKILL.md files target 60-120 lines.
Shared enginesRepeated mechanics move to Layer 2 with scripts/tests/templates.
One default pathEvery skill has a clear normal run and explicit escapes.
Evidence over ceremonyHard gates are deterministic or script-backed.

Design Rules

  1. SKILL.md is a job card. It should tell the agent what the skill is for, when to use it, what to load, what phases to run, and what must never happen.
  2. Scripts own state, parsing, arithmetic, schemas, and deterministic verification. If a skill describes a state machine in prose, that is a refactor candidate.
  3. References own reusable policy. Source trust, runtime portability, artifact authoring, reviewer protocol, UX scoring, and release rules should not be restated per skill.
  4. Judgment stays judgment. Reviewer opinions, dogfood satisfaction, PM taste, and UX critique can block a clean PASS, but should usually surface as PASS-WITH-GAPS rather than pretending to be deterministic failures.
  5. User-facing names reflect the user's intent. Internal SDLC nouns can remain behind the scenes; the command should match what the user thinks they are doing.

3. Three-Layer Architecture

LayerPurposeExamples
Layer 1: User-facing skills Memorable commands users invoke directly. They should feel like products or workflows, not implementation modules. /feature, /backlog, /verify, /primer, /interview-feedback, /converter
Layer 2: Shared engines Reusable mechanics, policy, schemas, scripts, templates, and evaluators loaded by Layer 1 skills. source trust, artifact authoring, review protocol, UX evaluation, thinking lenses, tracker engine, browser verification
Layer 3: Plugin infrastructure Marketplace packaging, runtime portability, CI, release/versioning, plugin metadata, and install/dependency contracts. manifest sync, plugin CI matrix, marketplace metadata, runtime contract expansion, skill linting

Important: moving a current skill to Layer 2 does not mean deleting its value. It means its behavior becomes a reusable engine rather than a command the user has to remember.

Recommendation

Use simple, intent-based names for Layer 1. Because backwards compatibility is not required, old process-heavy names should be retired instead of preserved as public aliases. The finished v3 marketplace should expose only the clean names; any temporary redirects are implementation scaffolding, not product surface.

Command Naming Policy

  • Prefer a verb for utilities: convert, verify, release, polish.

5. Feature Orchestrator

Decision proposal: keep one user-facing front door for product-building and make it /feature. Do not preserve /feature-sdlc as a public command after v3 lands.

The current define/build separation is directionally right. v3 should preserve that mental model under the cleaner /feature command. The split should happen behind the scenes, not by forcing users to remember many implementation commands.

6. Tiering

The tier system should stay. It is one of the strongest abstractions in the repo because it scales rigor without multiplying commands.

TierMeaningBehavior
Tier 1Lightweight, low-risk, localMinimal ceremony, fewer reviewers, local tests/checks, no broad folded phases unless obviously relevant.
Tier 2Standard product-quality workNormal requirements/spec/plan depth, targeted reviewer passes, standard verification, backlog/state updates.
Tier 3High-risk or release-grade workDeep verification, broader review, architecture/spec simulation, integration/browser checks, stricter release readiness.

Tiering Should Control Depth, Not Identity

The user should not need to know a different skill name for deep work. They should call the same command with a tier or natural language rigor hint.

/feature define "idea..." --tier 2
/feature build --next --tier 1
/feature build 0012 with deep verification

v3 Tier Contract

  • Tier determines reviewer count, folded evaluators, verification breadth, artifact depth, and release gates.
  • Tier does not change the user's next command.
  • Tier defaults should be conservative: Tier 2 for normal feature work, Tier 1 for explicitly lightweight work, Tier 3 for security/data/release/high-blast-radius work.
  • Every skill that honors tiering should cite one shared tier matrix instead of restating tier semantics.

How Skills Reference Layer 2

A v3 skill cites the shared contract and states only skill-specific deltas.

Use `_shared/source-trust/README.md` for source admission, citation ledger,
and thin-source degradation. Do not restate those rules here.

Delta for /book-brief:
- source identity must match the resolved book title + author
- social-only takeaways must be flagged or dropped

This avoids the current failure mode where every skill carries a slightly different copy of the same policy.

10. Missing Skills and Opportunities

The earlier instinct to add many product-management skills is too broad. v3 should reduce the command surface. Most product-leader outputs are artifact types, not skills. The default should be: add a template to /artifact first; extract a Layer 2 reasoning module when multiple templates need the same method; create a standalone skill only when the workflow has distinct intake, state, integrations, validation, or repeated operating cadence.

Decision Rule

If the difference is...Put it in...Examples
Mostly output structure/artifact templateRoadmap, strategy doc, opportunity-solution tree, exec brief, launch plan, experiment design, metric tree, pricing/packaging brief.
A reusable reasoning methodLayer 2 modulePrioritization model, opportunity mapping, metric decomposition, decision framing, evidence ledger, experiment-readiness checks.
A distinct operational workflowStandalone skillMessy-source ingestion, multi-session capture, backlog updates, contact/person context, connector-heavy workflows, deterministic import/export, or recurring maintenance.

11. Migration Plan

Phase 0: Align on Product Architecture

  • Agree on Layer 1 / Layer 2 / Layer 3 boundaries.
  • Agree on clean v3 command names and which legacy public names to retire.
  • Agree on tier semantics and default tier behavior.
  • Classify every skill as primary, Layer 2, renamed, or archived.

Phase 1: Build v3 Infrastructure

  • Create plugin-aware runtime contract.
  • Add relative citation resolver lint.
  • Create Layer 2 module skeletons and reference pattern.
  • Add plugin-level CI matrix for existing tests.

Phase 2: Fix Trust and Drift

  • Fix current /comments apply-vs-annotate semantics while renaming the public workflow to /artifact-comments.
  • Fix non-toolkit pmos-toolkit: status lines.
  • Fix stale managerkit paths and dangling shared citations.
  • Script /reflect transcript parsing and /book-summary source ledger checks.

Phase 3: Prove v3 on Small Skills

  • Rewrite gamekit launchers, changelog, wayrttd, and the first set of renamed thin commands.
  • Use these as v3 exemplars before rewriting core pipeline skills.

Phase 4: Rewrite Core Workflow

  • Introduce /feature as the new front door.
  • Refactor definition, planning, build, verify, and release modules behind it.
  • Retire /feature-sdlc after /feature define and /feature build cover the workflow.

Phase 5: Productize Remaining Skills

  • Learning/research skills.
  • Visual/product-builder skills.
  • Managerkit skills.
  • Connector-backed utilities.

External Pattern Notes

The direction above is consistent with recent public analysis of agent skills as engineered artifacts: skills work best when they use progressive disclosure, separate skill instructions from reusable tools/resources, and manage trust/lifecycle explicitly. Maintenance studies also show that reused skills tend to accumulate additive inline local knowledge, which is exactly the drift v3 should avoid.

Bottom Line

The v3 refactor should preserve the workflow you already use: backlog-backed definition and build, tiered rigor, and one clear front door. The change is both naming and internal architecture: make /feature the product-builder command, retire legacy process-heavy names, move repeated methods into Layer 2, keep deterministic work script-backed, and make direct engine commands optional rather than required memory.

The first implementation milestone should not rewrite a big skill. It should establish the runtime/Layer 2 standard and prove it on small skills. Then use that standard to refactor the core product-building workflow.
