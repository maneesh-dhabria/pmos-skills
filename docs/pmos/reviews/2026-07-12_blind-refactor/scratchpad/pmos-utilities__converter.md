### converter-F1 [Blocker] SKILL.md mandates a false privacy claim while pdf→md ships text to the Claude API
- Where: plugins/pmos-utilities/skills/converter/SKILL.md:96
- Quote: "Nothing is uploaded or persisted — it is a local single-user tool."
- Problem: Phase 0 step 4 instructs the launching agent to tell the user verbatim that nothing is uploaded, and the intro (SKILL.md:15-16, "nothing is / ever uploaded off your machine") and the frontmatter description ("everything runs offline on your machine, no npm install, no account, no upload") repeat it. But the shipped registry includes `pdf→md` with `kind: 'llm'` (lib/converters/pdf-md.js:26), whose primary backend `runClaudePdfToMd` hands the PDF's extracted text to the host `claude` CLI — i.e., the document's content leaves the machine for Anthropic's API whenever the CLI is available. The UI badges it "Uses Claude" (converter.html:126), so the UI is honest, but the skill body — the thing the agent speaks from — mandates misinformation about exactly the property (privacy of the user's data) the skill markets. The claim must be scoped: "pure conversions never leave your machine; the Claude-badged PDF→Markdown pair sends extracted text to the Claude API (local fallback if the CLI is absent)."

### converter-F2 [Should-fix] SKILL.md's conversion inventory is stale — 4 pairs shipped, 2 described
- Where: plugins/pmos-utilities/skills/converter/SKILL.md:18
- Quote: "v1 ships two bidirectional pure pairs — **JSON ↔ YAML** and **CSV ↔ JSON** — built on a"
- Problem: `lib/converters/` contains four pairs (8 conversions): json-yaml, csv-json, html-md, pdf-md — the latter two fully built, tested (html-descriptors, pdf-descriptors, pdf-writer, pdf-text, claude-seam suites), and served. Yet the body says "two bidirectional pure pairs", Phase 0 step 4 tells the agent to report "the supported conversions (JSON↔YAML, CSV↔JSON)" (SKILL.md:93), and the description hedges HTML↔MD / PDF↔MD as future ("and, as the skill grows"). A user who asks the agent what the converter can do gets told half the truth, and a user searching "convert HTML to markdown" may not trigger the skill confidently. Classic enumerated-set count-claim drift: prose was written for the v1 story and never updated when the second story landed. Either enumerate all four pairs or (better, since the registry is the single source of truth) stop enumerating in prose and say "report the conversion list the server prints at startup" — the server already emits `Conversions: <ids>` (server.js:181) precisely so nothing else needs the hardcoded list.

### converter-F3 [Should-fix] UI file/conversion state desynchronizes when the user switches conversion after loading a file
- Where: plugins/pmos-utilities/skills/converter/ui/converter.html:147
- Quote: "convSel.addEventListener('change', function () { renderBadges(byId(convSel.value)); });"
- Problem: `loadFile` decides binary-vs-text by the conversion selected *at load time* and the change handler only re-renders badges — it never clears `fileBuffer`, re-enables the textarea, or re-reads the file. Two concrete failure paths: (1) load a PDF with pdf→md selected (textarea disabled, shows "[binary file loaded: x.pdf]"), switch to md→html, click Convert → the placeholder string "[binary file loaded: x.pdf]" is silently converted as if it were the document; (2) select pdf→md and paste text instead of choosing a file → `fileBuffer` is null so the client sends the JSON envelope `{id, input}` as the request body, and the server's binary path feeds those envelope bytes to the PDF extractor (server.js:93-96), yielding the "no extractable text layer" caveat for a nonsense reason. Both are ordinary interactions, not adversarial ones. Fix: on conversion change, reset file state (or re-ingest the chosen file per the new descriptor's inputMode), and refuse Convert when a binary-input descriptor has no fileBuffer.

### converter-F4 [Should-fix] POST /convert has no Origin/Host check — a drive-by web page can invoke the user's Claude CLI
- Where: plugins/pmos-utilities/skills/converter/scripts/server.js:161
- Quote: "if (req.method === 'POST' && pathname === '/convert') {"
- Problem: the server binds loopback (good) but accepts any cross-origin POST — there is no Origin or Host header validation anywhere in the request handler. Browsers happily fire cross-origin POSTs to 127.0.0.1 (the response is opaque without CORS, but the side effect runs). Because `pdf→md` shells out to the user's authenticated `claude` CLI (spending their API credits per request), any web page open while the converter runs can port-scan localhost and repeatedly trigger LLM calls with attacker-chosen text. The prompt is well-sandboxed (`--allowedTools ''`, no paths — lib/claude-pdf.js:69), so the blast radius is credit burn plus CPU, not data exfiltration, which keeps this at Should-fix rather than Blocker. A ~5-line guard (reject POSTs whose Origin header is present and not `http://127.0.0.1:<port>`, and whose Host isn't loopback) closes it; this is the standard hardening for localhost tool servers.
