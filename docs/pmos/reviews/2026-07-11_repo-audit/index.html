PMOS Skills Repo Audit

A dispassionate review of the five-plugin skills marketplace: how the skills are engineered, where the architecture is holding, and where generated instruction accretion is now creating correctness, portability, and maintenance risk.

Date: 2026-07-11 Scope: 64 user-facing skills across 5 plugins Method: repo-audit frame + local checks + parallel review passes Output: recommendations only; no source fixes applied

Verdict

The repo is much healthier than the June audit mechanically, but the marketplace is still overfit to the history of how it was generated. The deterministic hygiene layer is now strong: flag docs and phase refs pass repo-wide, non-interactive blocks are byte-stable where supported, core manager/game/converter tests mostly pass, and the plugin manifests are coherent. The weak layer is the human-facing skill contract: too much toolkit-specific boilerplate leaked into other plugins; several skills encode parsing, aggregation, or trust gates in prose instead of scripts; and at least one high-risk contract says comments are "applied" when many shims only insert annotation comments.

The biggest architectural mistake is no longer simple verbosity. It is contract drift at boundaries: prose claims, copied runtime blocks, shared-substrate citations, and generated feature-design language have spread faster than the checks that make those claims true.

What Is Working

  • The post-June hygiene campaign paid off. `lint-flags-vs-hints.sh` and `lint-phase-refs.sh` passed across 64 skills. The old class of dead flags and ghost phase references is much less visible.
  • Several substrates are genuinely good engineering. Gamekit's pure engine globals and Node selftests, toolkit's HTML authoring/comments contracts, learnkit's topic-research substrate, managerkit's scorecard validators, and utilities' converter parsers are the right direction.
  • The plugin split mostly makes product sense. Toolkit = feature delivery, learnkit = learning artifacts, gamekit = browser games, managerkit = manager work, utilities = environment/document utilities. The charters are understandable.
  • Tests are strong where they exist. Seven game engine selftests passed. Managerkit `interview-feedback` and `one-on-one` suites passed. Converter's pure conversion suites passed. Comments coverage passed.
  • The repo is better at deterministic gates than most skill collections. The best skills use scripts for arithmetic, schema checks, citations, and parsing, then leave judgment to the model.

Top Findings

P01. `/comments resolve` can mark annotation-only shims as successful applied edits

`/comments` tells the operator that Accept "applies the edit to disk" and marks the thread resolved. Many per-skill `apply-edit-at-anchor.js` shims return `success: true` with an `applied_artifact`, but the artifact mutation is only an HTML annotation comment inserted before the anchor, not the requested content edit.

Evidence: `comments/SKILL.md` Phase 3/4 lines 103-119; annotation-only shims in `requirements`, `plan`, `artifact`, `spec`, `polish`, `shape`, `wireframes`, `diagram`, `prototype`, `survey-design`, `survey-analyse`, `architecture`, `readme`, `ideate`, `wayrttd`.

Why it matters: this is a trust bug. A reviewer can believe a stakeholder comment was resolved while the only actual artifact change is a hidden implementation note. That is worse than a visible failure.

Recommendation: either make shims perform real text/HTML edits, or change the contract and UI language from "applied/resolved" to "annotated/deferred" until real edits exist. Add a test that fails when `success: true` only adds `` without modifying visible content.

P02. Non-toolkit plugins inherit a hard-coded `pmos-toolkit:` runtime identity

The canonical non-interactive block prints `pmos-toolkit: /<skill> finished...` inside learnkit, managerkit, utilities, and gamekit skills. The lint enforces byte identity, so the wrong identity is mechanically preserved.

Evidence: `book-summary/SKILL.md:77`, `to-notion-doc/SKILL.md:66`, `interview-feedback/SKILL.md:84`, `snake/SKILL.md:71`, and similar matches across non-toolkit plugins.

Recommendation: make the canonical block parameterized (`{{plugin_name}}`) or define the line as `pmos: /<plugin>:<skill> finished...`. Then update the lint to compare the invariant body while allowing the plugin token.

P03. Some skills cite shared files or authoring guides that do not exist at that relative path

Managerkit's `interview-feedback` and `interview-guide` cite `../feature-sdlc/reference/skill-patterns.md`, which would resolve inside managerkit and does not exist. Gamekit, managerkit, and utilities cite `_shared/non-interactive.md` through the copied block, but those plugins do not ship that local file.

Evidence: `interview-feedback/SKILL.md:19`, `interview-guide/SKILL.md:20`; correct contrast in `one-on-one/SKILL.md:23`; dangling `_shared/non-interactive.md` citations in non-toolkit plugins.

Recommendation: add a `lint-shared-citations.sh` that resolves every relative markdown/code citation in SKILL.md bodies. Fix the managerkit paths immediately. Either ship local `non-interactive.md` copies in every plugin that cites it, or change the citation to an explicit canonical toolkit path.

P14. `/reflect` is a prose state machine with no script/test backing

`/reflect` defines transcript discovery, multi-session selectors, candidate caps, subagent concurrency, timeout behavior, aggregation hashes, and output shaping entirely in SKILL.md prose. Its folder contains only `SKILL.md`.

Evidence: `reflect/SKILL.md` lines 29-50, 82-122, 159-194; no scripts/tests under `plugins/pmos-utilities/skills/reflect/`.

Recommendation: extract transcript discovery, selection, and aggregation into zero-dependency scripts with fixtures for Claude personal/work homes and Codex homes. Keep the skill body focused on interpretation and recommendations.

P15. `/book-summary` has trust-critical gates but no local scripts or tests

The skill requires book identity resolution, fetched-this-run source verification, source ledger integrity, reviewer quote validation, and atomic trio writes. Unlike `primer`, `frameworks`, and `magazine`, it has no local scripts/tests; only reference docs.

Evidence: `book-summary/SKILL.md` lines 80-128; folder contains `SKILL.md` and `reference/*` only.

Recommendation: add structure tests and deterministic checks for source-ledger membership, citation-to-ledger consistency, `sections.json` parity, and "no emitted source unless fetched/verified". Mirror the stronger learnkit substrate skills.

P16. CI does not cover all plugin-specific test assets

Repo hygiene CI is broad, but gamekit engine tests, converter tests, managerkit tests, and many learnkit substrate tests are not obviously wired into plugin-level CI. `audit-recommended.yml` only triggers on toolkit paths, even though the script now scans supported skills across plugins.

Evidence: `.github/workflows/audit-recommended.yml` paths only include `plugins/pmos-toolkit/...`; `.github/workflows/skill-hygiene.yml` is broader but does not run plugin functional suites.

Recommendation: add one workflow per plugin, or a matrix workflow keyed by plugin. Include localhost tests in a clearly named job so sandbox failures are not confused with code failures.

P17. Marketplace packaging is healthy, but repo instructions still mention stale `.Codex-plugin` paths

The current repo has `.claude-plugin` and `.codex-plugin` marketplace directories, not `.Codex-plugin`. The provided repo instructions still describe `.Codex-plugin` as canonical. That mismatch creates false audit findings and can mislead future release work.

Evidence: filesystem inventory shows `./.claude-plugin` and `./.codex-plugin`; no `./.Codex-plugin` directory.

Recommendation: update repo invariants to current casing and naming. If backwards compatibility with `.Codex-plugin` is intentionally retired, say that explicitly.

P28. Some toolkit orchestrators remain too broad for a single skill body

`feature-sdlc`, `complete-dev`, `verify`, `plan`, `execute`, `artifact`, and `mytasks` still dominate the line-count table. This is partly earned complexity, but `feature-sdlc` in particular now carries classic feature delivery, skill creation, prototype mode, define/build backlog loops, worktree lifecycle, skill-eval, and release handoff.

Recommendation: do not split just to shrink. Split where the user mental model is already separate: skill authoring, prototype discovery, and backlog define/build should become thin entrypoints or reference-backed modules under a smaller orchestrator.

P29. Gamekit has strong engine tests but an incomplete product-level contract

The engine tests passed, and the shared launcher passed once localhost binding was allowed. But two game bundles include outbound GitHub links despite "offline/no network" positioning, browser rendering is not smoke-tested, and gamekit tests are not wired into CI.

Recommendation: add a static scan for outbound `href/src/action` URLs and active network APIs. Add browser smoke tests that load each game, assert no console errors, and check a visible primary UI. Keep the current engine tests.

Fix Program

P0 - Correctness and trust

  1. Resolve the `/comments` apply-vs-annotate contract. This is the highest leverage fix because it affects every HTML artifact review loop.
  2. Parameterize or replace the `pmos-toolkit:` non-interactive summary line across plugins, then update the lint.
  3. Fix managerkit stale `skill-patterns.md` paths and add a relative-citation resolver lint.
  4. Decide how non-toolkit plugins should cite `non-interactive.md`: local copy, canonical toolkit path, or parameterized shared package.

P1 - Mechanical coverage

  1. Add plugin CI matrix jobs for gamekit, managerkit, learnkit, utilities, and toolkit functional suites.
  2. Widen `audit-recommended.yml` path triggers to all plugin skill files and shared non-interactive surfaces.
  3. Add browser smoke checks for gamekit and server-backed UI tools.
  4. Add `/book-summary` deterministic tests for ledger, citations, sections, and source verification claims.
  5. Extract `/reflect` transcript parsing and aggregation into scripts with fixtures.

P2 - Design and product polish

  1. Split or thin the broadest orchestrators only where the user-facing product boundary is real: skill authoring, prototype discovery, backlog define/build.
  2. Move remaining judgment-heavy "must" gates into PASS-WITH-GAPS where deterministic failure is not present.
  3. Make shared substrates plugin-aware instead of maintaining intentionally divergent copies, especially HTML authoring renderer defaults.
  4. Clarify connector seams for Notion, future Drive/Slack/Gmail-style integrations, and Codex transcript access.
  5. Remove outbound links from game bundles or explicitly loosen the "offline/no network" claim.

Bottom Line

The repo is not a throwaway pile of generated skills. It has a real architecture, and the deterministic substrate work is paying off. But the marketplace is now large enough that copied prose is a liability. The next maturity step is to stop treating SKILL.md as the place where every contract lives. Put parseable state, identity, citation, test, and routing contracts in scripts or resolvable shared files; leave SKILL.md for intent, branching logic, and the few invariants the model must carry in working memory.

In practical order: fix comments resolver truthfulness, fix copied plugin identity/path drift, wire plugin tests into CI, then thin the large orchestrators.
